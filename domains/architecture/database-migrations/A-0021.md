# Summary

Defines the database migration architecture for Ledgius using Flyway-style versioned and repeatable SQL scripts. Replaces the legacy approach of bootstrapping from a LedgerSMB `pg_dump` clone with explicit, auditable, version-controlled SQL scripts that create the complete schema from scratch. All unused LedgerSMB tables, functions, and stored procedures are removed.

# Requirement Link

Foundational architecture document governing how the Ledgius database schema is created, evolved, and seeded. Enables reproducible deployments without any LedgerSMB dependency.

# Technical Context

The original Ledgius database was bootstrapped by:
1. Running `ledgersmb-admin create` to produce a full LedgerSMB schema (~220 tables, ~500 stored procedures)
2. Loading the Australian COA via `ledgersmb-admin setup load`
3. `pg_dump` the LedgerSMB database and restoring into the Ledgius tenant database
4. Applying 16 incremental tenant migration scripts on top

This approach had several problems:
- Schema contained ~200 unused tables and hundreds of dead stored procedures
- No visibility into what the schema actually contained
- LedgerSMB tooling required for database bootstrap (Perl dependencies)
- No Flyway-compatible migration history
- Seed data scattered across migration scripts and external tooling

# Proposed Design

## Migration Strategy

### Two Databases, Two Migration Tracks

| Database | Purpose | Migration Directory |
|----------|---------|-------------------|
| `ledgius_platform` | Multi-tenant control plane (users, tenants, RBAC) | `migrations/platform/` |
| `ledgius` (per-tenant) | Accounting data (GL, AR, AP, payroll, etc.) | `migrations/tenant/` |

### Script Types

**Versioned scripts** (`V<major>.<minor>__<description>.sql`):
- Applied exactly once, in version order
- Never modified after deployment
- Used for: schema creation, schema alterations, one-time data migrations

**Repeatable scripts** (`R__<description>.sql`):
- Re-applied whenever their checksum changes
- Applied after all versioned scripts
- Used for: seed data, reference data, views, functions
- Must be idempotent (use `INSERT ... ON CONFLICT DO UPDATE`, `CREATE OR REPLACE`, etc.)

### Version Numbering

- `V1.00__` through `V1.16__` — initial schema (extracted from LedgerSMB + existing migrations)
- `V1.17__` onward — new schema changes
- Major version bump (`V2.00__`) reserved for breaking schema changes

### Migration History Table

A `flyway_schema_history` table tracks which scripts have been applied:

```sql
CREATE TABLE IF NOT EXISTS flyway_schema_history (
    installed_rank  SERIAL PRIMARY KEY,
    version         TEXT,
    description     TEXT NOT NULL,
    type            TEXT NOT NULL CHECK (type IN ('SQL', 'REPEATABLE')),
    script          TEXT NOT NULL,
    checksum        TEXT,
    installed_by    TEXT NOT NULL DEFAULT current_user,
    installed_on    TIMESTAMPTZ NOT NULL DEFAULT now(),
    execution_time  INT NOT NULL,
    success         BOOLEAN NOT NULL
);
```

## Schema Extraction

### Tables Retained (Actively Used by Go API)

**LedgerSMB-inherited (16 tables):**
`transactions`, `acc_trans`, `gl`, `ar`, `ap`, `account`, `account_heading`, `account_link`, `entity`, `entity_credit_account`, `company`, `open_item`, `invoice`, `currency`, `cr_report`, `cr_report_line`

**Ledgius-added (via migrations 001-016):**
All tables from the existing tenant migrations are retained.

### Tables Removed

All other LedgerSMB tables (~200) are excluded from the new migration scripts. This includes:
- Inventory/parts system
- Manufacturing/projects
- Fixed assets
- Legacy payroll (replaced by Ledgius payroll)
- Legacy tax modules (replaced by `tax_code`)
- LedgerSMB user/session management
- Batch/queue/draft systems
- Custom field systems
- Business unit hierarchies

### Stored Procedures and Functions Removed

All LedgerSMB PL/pgSQL stored procedures are excluded. The Go API implements all business logic directly (with CEL/OPA/GoRules for rules). Only Ledgius-created functions are retained (e.g., `prevent_audit_modification()` trigger function).

## Seed Data Strategy

Seed data is managed via repeatable scripts:

| Script | Content |
|--------|---------|
| `R__seed_chart_of_accounts_au.sql` | Australian General COA (~100 accounts + headings + account links) |
| `R__seed_tax_codes.sql` | GST, FRE, INP, EXP, N-T, CAP tax codes |
| `R__seed_currencies.sql` | AUD, USD, GBP, EUR, NZD with initial exchange rates |
| `R__seed_taxonomy.sql` | Australian tax taxonomy tree |
| `R__seed_entity_schemas.sql` | Entity schema registry and fields |
| `R__seed_payg_rates.sql` | PAYG withholding coefficients FY2020-2026 |
| `R__seed_super_rates.sql` | Super guarantee rates FY2020-2026 |

## Migration Runner

A shell script (`flyway.sh`) orchestrates migration execution:

1. Reads connection parameters from environment
2. Creates `flyway_schema_history` table if not exists
3. Applies versioned scripts in order (skipping already-applied)
4. Applies repeatable scripts (re-applying if checksum changed)
5. Records each execution in history table

### Makefile Targets

```makefile
db-migrate          # Apply all pending migrations to tenant DB
db-migrate-platform # Apply all pending migrations to platform DB
db-seed             # Apply repeatable seed scripts
db-status           # Show migration status
db-reset            # Drop and recreate databases, apply all migrations
db-validate         # Verify all migrations applied successfully
```

## Deployment Flow

### Local Development
```bash
cd ledgius-infra
make up          # Start Docker containers
make db-init     # Create databases + apply all migrations + seed
```

### Fly.io Production
```bash
cd ledgius-db
fly ssh console -C "/app/flyway.sh migrate"
```

## Anti-Patterns

- Never modify a versioned script after it has been applied to any environment
- Never use GORM AutoMigrate — all schema changes go through SQL scripts
- Never seed data inline in versioned scripts — use repeatable scripts
- Never include LedgerSMB stored procedures — all logic in Go + rules engine
- Never hardcode connection strings — always read from environment

# Out of Scope

- Automated rollback scripts (manual rollback via new versioned script)
- Multi-region database replication
- Schema diff tooling (future consideration)
- Tenant database provisioning automation (covered by A-0016)
