# Summary

Cross-cutting standard for storing sensitive personal-information fields encrypted at rest. Field-level (column-value) encryption using AES-256-GCM, with envelope encryption: a per-tenant Data Encryption Key (DEK) wraps the cleartext, and the DEK itself is wrapped by a single platform-wide Master Key Encryption Key (KEK) held in Fly secrets. Wrapped DEKs are stored in the platform DB; cleartext DEKs are decrypted on first use and held in an in-process cache for the lifetime of the API process.

The first concrete user of this design is the STP Phase 2 TFN field (R-0005 PAY-STP-039). The same primitive will later cover bank account numbers, super-fund member numbers, and any other field flagged as sensitive PII in subsequent requirements. Field-level encryption is the substantive control; full-volume encryption (Fly volume + Postgres TDE) is layered underneath as defense-in-depth, not as a substitute.

# Terms

| Term | Definition |
|---|---|
| **Field-level encryption** | Ciphertext stored in a single column; decryption happens in the application at point of use. The DB sees only ciphertext bytes. Distinct from disk-level / volume-level encryption. |
| **Envelope encryption** | Two-layer key hierarchy: a *data key* (DEK) actually encrypts the data; a *key-encryption key* (KEK) encrypts the DEK. Reduces blast radius (compromise of one DEK ≠ compromise of all data) and simplifies rotation (rotate the KEK without re-encrypting data). |
| **Master KEK** | The single top-of-tree key for the Ledgius platform. Held in Fly secrets / external KMS, never in the database. Loaded into the API process at startup. |
| **Tenant DEK** | A per-tenant 256-bit key that encrypts that tenant's sensitive field values. Generated at tenant provisioning, stored in the platform DB wrapped by the Master KEK. Decrypted on demand by the API; cached in-process. |
| **DEK Version** | Monotonically-increasing integer per tenant. Each ciphertext records the DEK version that encrypted it; rotating a tenant's DEK creates a new version while old ciphertexts remain decryptable until re-encrypted in a background sweep. |
| **Cleartext window** | The narrow region of the request flow where a sensitive field exists in plaintext: from `Decrypt(ciphertext)` to the function-return / log-line that uses it. The architecture aims to keep this window as small as possible — typically a single function call. |
| **Sensitive field** | A column flagged as PII for the purposes of this spec. The canonical inventory is the table in §Inventory. New sensitive fields are added there as requirements identify them. |

# Technical Context

What already exists:

- **Multi-tenant connection model** (A-0016) — each tenant has its own DB; one platform DB holds tenant metadata and provisioning state.
- **Tenant manager** (`pkg/tenant.Manager.Resolve`) — already returns a per-tenant `*gorm.DB` plus the tenant's row from the platform DB. Natural place to attach a per-tenant DEK accessor.
- **Fly secrets** — already used for `JWT_SECRET`, `RESEND_API_KEY`, `OPENAI_API_KEY`. Adding `LEDGIUS_TFN_MASTER_KEK` is mechanically identical.
- **Tenant provisioning pipeline** (A-0047) — has hooks for "do this on first provisioning of a new tenant"; ideal place to generate the initial DEK.
- **`pgcrypto` Postgres extension** — already present in the dev DB image.

What does not exist and is in scope:

- A platform-DB table to hold tenant DEKs (wrapped).
- A Go package `pkg/cryptostore/` providing `Encrypt(tenantID, plaintext) → ciphertext` and `Decrypt(tenantID, ciphertext) → plaintext` with the envelope semantics.
- A startup hook that loads the Master KEK from `LEDGIUS_TFN_MASTER_KEK` and panics if absent.
- A DEK rotation tool (one-shot operator command; not in v1's scope but the design must accommodate it).
- Audit-log integration so every decrypt records actor + reason.

# Threat Models Considered

The choice of field-level over volume-level encryption is driven by which attacks each defends against. The right answer depends on which threats are realistic for Ledgius.

| Threat | Volume / TDE | Field-level | Notes |
|---|---|---|---|
| Stolen physical disk / cloud-provider raw storage exfiltration | ✅ Defends | ✅ Defends | Either layer alone is sufficient. |
| Stolen Postgres backup file (e.g. `pg_dump` artifact in S3) | ❌ No | ✅ Defends | Backups inherit the file format, not the volume encryption. |
| SQL injection / app-layer breach with valid DB credentials | ❌ No | ✅ Defends | Attacker has a connection — sees plaintext on TDE; sees ciphertext on field-level (no DEK). |
| Read replica / streaming dump consumed by analytics | ❌ No | ✅ Defends | Replicas inherit ciphertext; readers without the DEK can't decrypt. |
| Developer pulls a prod snapshot to dev | ❌ No | ✅ Defends | Snapshot is plaintext under TDE; ciphertext under field-level. |
| Accidental `SELECT *` in an error log | ❌ No | ✅ Defends | Log carries whatever's in the row; ciphertext is harmless. |
| Cross-tenant accidental disclosure (one tenant's row in another's view) | ❌ No | ✅ Defends per-tenant | Per-tenant DEKs mean a leaked row is unreadable to anyone without that tenant's DEK. |
| Malicious insider with platform-DB access | ❌ No | ⚠ Partial | Insider sees wrapped DEKs but not the Master KEK. Can't decrypt without breaking out of the DB. |
| Malicious insider with API-host access | ❌ No | ❌ No | Insider can read the in-process DEK cache. Mitigation is access control + audit, not encryption. |
| KEK leak | n/a | ❌ Defeats | Whole-system compromise; mitigated by KEK rotation + alerting on Fly-secret access. |

The realistic threats for Ledgius (cloud-hosted, multi-tenant, small team) cluster heavily on the rows where field-level is the only defense. TDE alone is insufficient.

**Conclusion**: field-level encryption for the substantive control; volume encryption layered underneath as defense-in-depth.

# Proposed Design

## Key Hierarchy

```
            ┌──────────────────────────────────────┐
            │ Master KEK (32 bytes, AES-256)       │
            │ Held in Fly secrets:                 │
            │   LEDGIUS_TFN_MASTER_KEK (base64)    │
            │ Loaded once at API startup           │
            │ Rotates ~yearly or after incident    │
            └────────────────┬─────────────────────┘
                             │ wraps (AES-256-GCM)
                             ▼
            ┌──────────────────────────────────────┐
            │ Tenant DEK (32 bytes, AES-256)       │
            │ Stored wrapped in platform.tenant_   │
            │   data_key                           │
            │ One per tenant per version           │
            │ Rotates ~yearly or on tenant request │
            └────────────────┬─────────────────────┘
                             │ encrypts (AES-256-GCM)
                             ▼
            ┌──────────────────────────────────────┐
            │ Sensitive field ciphertext           │
            │ Stored in the tenant DB as bytea     │
            │ Format: nonce(12) || ciphertext      │
            │ Each row records dek_version         │
            └──────────────────────────────────────┘
```

## Algorithm Choice

- **Cipher**: AES-256-GCM. Authenticated encryption (AEAD); resists ciphertext tampering. The Go standard library `crypto/cipher.NewGCM` provides it.
- **Key size**: 256 bits.
- **Nonce**: 96 bits (the GCM standard), sourced from `crypto/rand`. **Never reused** under the same key. With ~10^28 possible nonces and the realistic per-tenant write rate, collision probability is negligible.
- **Associated Data (AAD)**: the column identifier (e.g. `"employee.tfn"`). Binds ciphertext to the column it lives in; copying ciphertext between columns invalidates the auth tag. Optional but recommended.

GCM is preferred over CBC + HMAC because it is one primitive with one set of failure modes, and the standard library implementation is well-vetted.

**Rejected**: ChaCha20-Poly1305 — equally fine cryptographically, but Postgres / pgcrypto / industry tooling round AES-GCM is denser. AES-NI is available on Fly's hardware. AES-256-GCM is the better default.

## Storage Format

Ciphertext on disk is a single bytea column with this layout:

```
+---------+----------------+--------+
| nonce   | ciphertext     | tag    |
| 12 B    | len(plaintext) | 16 B   |
+---------+----------------+--------+
```

The Go GCM API returns `ciphertext || tag` as one slice, so the on-disk format is `nonce || aead.Seal(...)`. `Open` consumes the same layout.

The `dek_version` is stored in a sibling column on the same row. This lets the decrypt path know which DEK version to use without reading the platform DB on every decrypt (the lookup happens once per process per tenant).

```sql
-- Example schema fragment for an encrypted column.
ALTER TABLE employee
    ADD COLUMN tfn_ciphertext  BYTEA  NULL,
    ADD COLUMN tfn_dek_version SMALLINT NULL;
```

`tfn_provided` (existing boolean) remains for back-compat and quick-existence checks; it is set true when `tfn_ciphertext IS NOT NULL`.

## Platform-DB Tenant Key Table

```sql
-- migrations/platform/V<NN>__tenant_data_key.sql
CREATE TABLE tenant_data_key (
    tenant_id      uuid       NOT NULL REFERENCES tenants(id),
    dek_version    smallint   NOT NULL,
    wrapped_dek    bytea      NOT NULL,           -- DEK encrypted with Master KEK
    kek_id         text       NOT NULL,           -- which Master KEK wrapped this row
    created_at     timestamptz NOT NULL DEFAULT now(),
    rotated_from   smallint   NULL,               -- prior version this replaces
    PRIMARY KEY (tenant_id, dek_version)
);

CREATE INDEX tenant_data_key_active_idx
    ON tenant_data_key (tenant_id)
    WHERE rotated_from IS NULL;  -- the active row per tenant
```

The "active" DEK per tenant is the one with the highest `dek_version` and `rotated_from IS NULL`. New encryption uses the active DEK; decryption uses whatever version the row records.

`kek_id` lets the platform run more than one Master KEK at a time during rotation — old DEKs stay wrapped under the old KEK until re-wrapped.

## Go API Surface

`pkg/cryptostore/` is the only place AES / GCM primitives are called.

```go
package cryptostore

// Store provides envelope-encrypted storage of sensitive field values.
// All operations are scoped to a tenant; cross-tenant decryption is
// structurally impossible.
type Store interface {
    // Encrypt the plaintext under the tenant's active DEK. Returns the
    // ciphertext blob (nonce || ciphertext || tag) and the dek_version
    // that encrypted it. AAD binds the ciphertext to a column identifier.
    Encrypt(ctx context.Context, tenantID string, aad []byte, plaintext []byte) (ciphertext []byte, dekVersion int16, err error)

    // Decrypt the ciphertext using the recorded DEK version. Records
    // an audit row stamped with actor (from JWT claims on ctx) and
    // reason. Decryption is the only operation that produces plaintext;
    // callers must keep the cleartext window as narrow as possible.
    Decrypt(ctx context.Context, tenantID string, dekVersion int16, aad []byte, ciphertext []byte, reason string) (plaintext []byte, err error)

    // Rotate generates a new DEK version for the tenant. Existing
    // ciphertext stays valid (decryptable under prior versions) until
    // the rewrap sweeper runs. Returns the new version number.
    Rotate(ctx context.Context, tenantID string) (newVersion int16, err error)
}

// New constructs a Store backed by the platform DB for DEK storage and
// an in-process LRU cache. Master KEK is read from the LEDGIUS_TFN_MASTER_KEK
// env var; absence is a fatal startup error.
func New(platformDB *sql.DB, opts ...Option) (Store, error)
```

Decryption requires a `reason` string — this becomes the audit-log entry's metadata. Conventional values:

- `"stp.payload.generate"` — STP submission generation
- `"export.payee_csv"` — payroll export of payee data
- `"admin.tfn_lookup"` — operator-driven explicit lookup (rare, gated by role)

There is no `Decrypt(...)` overload that omits `reason`.

## Cleartext Window Discipline

Decrypted values must:

1. **Live in a `[]byte` or `string` for as short as possible.** Pass to the consumer (e.g. payload generator) and let the value go out of scope immediately after.
2. **Never be logged.** No `slog.Info("decrypted tfn=%s", tfn)` — caught by `golangci-lint` rule (custom linter check).
3. **Never round-trip through JSON serialisation of an internal type.** API responses that include sensitive fields must explicitly mark them and either redact (e.g. `***234`) or include only when the caller has the right role.
4. **Never appear in error messages.** Wrap underlying errors with `cryptostore: decrypt failed for tenant %s field %s` — never include the value being decrypted.

The consumer is responsible for the cleartext window length. The architecture provides the primitives; review enforces discipline.

## Caching

Per-tenant cleartext DEKs are cached in-process (LRU, 512 entries, ~16KB total). Cache hit avoids the round-trip to the platform DB and the AES-decrypt of the wrapped DEK. Cache miss costs ~one DB query + one AES operation, ~1ms.

Cache is process-local: no Redis. A tenant's DEK is loaded on first use after API restart and stays warm for the process lifetime. Eviction happens only under memory pressure (LRU); rotated DEKs are inserted alongside the prior version, not replacing it.

DEK cache is **never persisted** — it lives only in process memory. Process restart re-fetches from the platform DB.

## Decryption Audit Log

Every successful `Decrypt` writes a row to the existing `audit_log` table:

```
action      = "field.decrypt"
entity_type = aad column identifier (e.g. "employee.tfn")
entity_id   = the row's primary key
metadata    = {tenant_id, dek_version, reason, actor_user_id, ip}
```

This produces a queryable trail: "who has decrypted this employee's TFN, when, and why". Required for compliance reviews and incident forensics.

Failed decryptions log a separate `field.decrypt.failed` action with the error class (auth-tag failure, unknown DEK version, KEK unavailable). Auth-tag failures are alarming — they indicate ciphertext tampering or DEK / row mismatch.

## Initialisation and Provisioning Flow

### API startup

1. Read `LEDGIUS_TFN_MASTER_KEK` from environment. Expect base64-encoded 32 bytes. Absence ⇒ fatal log + os.Exit(1) (no silent-skip).
2. Validate the KEK: derive a constant ("KEK_HEALTHCHECK") with HMAC-SHA-256, compare against a fingerprint stored in `tenant_data_key.kek_id` of any existing row. Mismatch ⇒ likely wrong env var; fatal.
3. Initialise the cryptostore with the KEK and a handle to the platform DB.

### New tenant provisioning (per A-0047)

1. After tenant DB creation, the provisioning pipeline calls `cryptostore.InitialiseTenant(tenantID)`.
2. Generate a fresh 32-byte DEK from `crypto/rand`.
3. Wrap it with the Master KEK via AES-256-GCM (with `tenantID` as AAD).
4. Insert into `tenant_data_key` with `dek_version=1`, `kek_id=<active-kek-fingerprint>`.
5. Record audit: `tenant.dek_initialised`.

### Existing-tenant retrofit (one-shot)

When this design lands, all existing tenants need a v1 DEK before any sensitive-field column is populated. The retrofit migration:

1. For each tenant in `tenants` without a `tenant_data_key` row, generate + wrap + insert a DEK at `dek_version=1`.
2. Idempotent — re-running is a no-op (ON CONFLICT DO NOTHING on `(tenant_id, dek_version)`).
3. Runs as part of the deploy step that introduces the cryptostore, before the first encrypted-column writer is enabled.

## Key Rotation

Two distinct rotation paths:

### DEK rotation (per tenant)

- Triggered: yearly schedule, suspected compromise, tenant request.
- Operator runs `cryptostore.Rotate(tenantID)` → new `dek_version` row inserted, marked active.
- Existing ciphertexts continue to decrypt under their recorded prior version.
- Background sweeper re-encrypts ciphertexts to the new version over time. Sweeper is idempotent and resumable.
- Once all ciphertexts of a column are at the new version, the prior DEK row can be deleted (post-v1 hygiene; not in v1 scope).

### KEK rotation (global)

- Triggered: yearly, suspected master-secret compromise, key-management policy.
- Procedure (planned, not in v1 scope but design must support):
  1. Generate new KEK; install in Fly secrets alongside the old.
  2. Re-wrap every `tenant_data_key.wrapped_dek` under the new KEK; update `kek_id`.
  3. Verify; remove the old KEK from secrets.
- The `kek_id` column on `tenant_data_key` lets the system run with both KEKs simultaneously during the transition.

## Defense-in-Depth Layers

Field-level encryption is **layered with**, not **instead of**:

| Layer | Provided by | Defends against |
|---|---|---|
| Field-level (this spec) | `pkg/cryptostore/` | App-breach, log leak, dump, replica, dev-snapshot, cross-tenant, partial insider |
| Database-level (TLS in transit) | Postgres + GORM client | Network sniffing |
| Volume-level (Fly volumes) | Fly infrastructure | Stolen physical disk |
| OS-level (LUKS / dm-crypt on Fly's underlying hosts) | Fly + Linux | Same as above |
| Backup-level (encrypted dump targets) | ledgius-infra backup pipeline | Backup-file theft |

Every layer is cheap to enable and adds an independent control. Failing one layer does not collapse the next.

# API / Interface Changes

No HTTP-API surface changes from this spec directly — it is library + storage. Endpoints that previously returned a sensitive field (e.g. `GET /employees/:id`) continue to do so, but the value flows through the cryptostore rather than coming straight from the column.

What changes by convention:
- Response bodies redact sensitive fields by default. A `?reveal=true` query param on a per-endpoint basis (gated by role) returns the cleartext and triggers the decrypt audit row.
- New JWT claim or middleware-injected role check around any decrypt: only roles in `{Bookkeeper, Accountant, Owner, PlatformAdmin}` can request reveal.

# Risks / Trade-offs

| Risk | Impact | Mitigation |
|---|---|---|
| Master KEK leak | Total compromise of all tenant TFNs | Fly secrets access is logged + alerted; KEK rotation procedure documented; alarm on unexpected secret reads. |
| Operator forgets to set `LEDGIUS_TFN_MASTER_KEK` in production | API fails to start | Fail-fast at startup is the correct behaviour — no silent fallback to plaintext. Document in deploy runbook. |
| Lost DEK (DB row deleted in error) | Affected ciphertexts unreadable forever | Platform DB is backed up; `tenant_data_key` is append-only in normal operation. Add a CI check that the migration never adds DELETE on `tenant_data_key`. |
| Application bug round-trips ciphertext as JSON | Looks like garbled text in a UI; no data loss | Linter rule + tests; the `bytea` column type is distinct from `text` in GORM. |
| Decrypt audit-log volume | Performance / storage | Per-decrypt log row is ~200 bytes; payload generation decrypts N TFNs once per submission. ~negligible at v1 scale. |
| Cleartext leak via accidental `fmt.Sprintf("%+v", employee)` | TFN appears in logs | The `Employee` struct holds ciphertext, not plaintext. The cleartext window is by-design narrow; printing the struct reveals only ciphertext. |
| GCM nonce collision | Catastrophic confidentiality break | 96-bit random nonce + write rate make collision probability ~0 for any plausible Ledgius scale. Rotate DEK if approaching 2^32 writes per DEK (we won't, but document the bound). |
| Per-tenant DEK overhead at very high tenant count | Memory / DB load | LRU 512 entries + on-demand fetch handles 10k+ tenants. Cache stats exposed as metrics for capacity-planning. |

# Alternatives Considered

## Whole-DB encryption only (TDE / volume encryption)

**Rejected** as the primary control. Defends only the stolen-disk threat. App breach, log leak, dump, replica, dev-snapshot — all bypass it. TFN-class data under the Privacy Act 1988 NDB scheme would not survive a competent compliance audit on this control alone.

## Postgres `pgcrypto` column-level (`pgp_sym_encrypt` / `pgp_sym_decrypt`)

**Rejected** as the primary control. Three reasons:

1. **Key has to come from somewhere**: passed via SQL parameter, session GUC, or constant. All paths leak the key into Postgres logs / `pg_stat_activity`. Keeping the key in the app and never sending it to Postgres is structurally safer.
2. **Re-implementing AAD and metadata**: `pgcrypto` doesn't expose the column-binding semantics cleanly; we would re-implement them at the app layer anyway.
3. **Diversity of failure modes**: cryptographic primitives in two layers (app + Postgres) means twice the surface area to keep current.

`pgcrypto` is fine for one-off internal data-at-rest tooling; it is not the right home for the substantive control on PII.

## ChaCha20-Poly1305

**Rejected** for AES-256-GCM. Equally secure cryptographically. AES-256-GCM has hardware acceleration on the API host, broader interop with operational tooling, and is what every other piece of infra in the stack already uses. No reason to introduce a second AEAD.

## Per-row DEKs (one DEK per encrypted value)

**Rejected** as too costly. Storage per row ≈ 50 bytes of metadata, two AES operations per read, plus a re-key burden every rotation. Per-tenant DEKs strike the right balance between blast-radius limitation and operational cost.

## Tink / age / libsodium / cloud-KMS-only

**Considered, kept available for future**. The Go standard library provides everything v1 needs without the dependency. The `Store` interface is small enough that a future swap-in of Tink or a cloud KMS (AWS KMS, GCP KMS) is a one-package change. v1 keeps the stack lean.

## Storing the Master KEK in the platform DB

**Rejected**. A single platform-DB compromise = all tenant data decryptable. The KEK *must* live somewhere the platform DB cannot decrypt itself. Fly secrets is the simplest answer; an external KMS is the natural future migration.

# Test Strategy

## Unit (`pkg/cryptostore/`)

- Round-trip: `Encrypt → Decrypt` returns the original plaintext for representative inputs (empty, ASCII, UTF-8, exactly nonce-size, large).
- Wrong AAD: `Decrypt` with mismatched AAD returns auth-tag-failure error.
- Wrong DEK version: `Decrypt` with a non-existent version returns "unknown DEK version".
- KEK rotation: re-wrap of a DEK under a new KEK preserves decryptability; old ciphertexts still decrypt.
- Nonce uniqueness: 100k encryptions under the same DEK produce 100k distinct nonces.

## Integration

- `cryptostore.New` reads `LEDGIUS_TFN_MASTER_KEK` and refuses to start without it.
- `InitialiseTenant` followed by `Encrypt + Decrypt` works end-to-end.
- Decrypt produces an `audit_log` row with the recorded `reason`.
- A simulated app crash mid-rotation (DEK v2 inserted but ciphertexts still v1): verifies decryption still works.

## Property-based (optional v1)

- For random plaintext, `Decrypt(Encrypt(p))` ≡ p.
- For random ciphertext + AAD, `Decrypt` either returns plaintext or an auth-tag failure (no third state).

## Manual / runbook-verified

- KEK rotation runbook: step-by-step procedure exercised in a staging environment before being applied to production.

# Rollout / Migration Notes

1. **Land platform-DB migration** (`tenant_data_key`) in a release that does not yet write any sensitive fields.
2. **Land `pkg/cryptostore/`** with `LEDGIUS_TFN_MASTER_KEK` plumbing. Deploy; verify startup succeeds.
3. **Run the retrofit script** to seed `dek_version=1` for every existing tenant.
4. **Land the first concrete user** (R-0005 TFN field — V1.61 + payroll service writes). Prior to this point, no encrypted columns exist; after, TFN writes go through the cryptostore.
5. **Document the rotation runbook** before the first real prod TFN is stored.
6. **Subsequent sensitive-field rollouts** follow the same shape: column migration adds `<field>_ciphertext` + `<field>_dek_version`; service code routes through `cryptostore.Encrypt` / `Decrypt`.

The Master KEK is generated once and stored in Fly secrets at step 2. The procedure:

```bash
# Generate a fresh 256-bit KEK.
openssl rand -base64 32
# Set in Fly:
fly secrets set LEDGIUS_TFN_MASTER_KEK="<base64-output>" -a ledgius-api
```

Document the generated KEK in the offline operator-secrets vault. Loss of the KEK and absence from Fly secrets means all tenant DEKs become permanently unwrappable — recoverable only from offline backup.
