# Summary

Embed tenant plan features and user role function permissions
directly in the JWT access token. Resolve features and permissions
via an in-memory TTL cache (`jellydator/ttlcache/v3`) to eliminate
per-request database queries. The middleware reads claims from the
token instead of querying the DB, achieving zero-cost permission
checks on every request.

# Terms

| **Term** | Definition. |
|---|---|
| **Feature** | A named capability gated by the tenant's pricing plan (e.g. `payroll`, `bank_feeds`). Determines whether the tenant can access a feature at all. |
| **Function** | A granular action within a feature, gated by the user's role (e.g. `payroll:create_employee`). Determines what the user can do within an allowed feature. |
| **Access token** | Short-lived JWT (30 min) containing user identity + tenant + role + features + functions. Used on every API request. |
| **Refresh token** | Long-lived opaque token (7 days) stored in DB. Used to obtain a new access token with fresh claims. |
| **Feature cache** | In-memory TTL cache keyed by `tenant_id`, returning the list of enabled feature slugs for that tenant's plan. |
| **Permission cache** | In-memory TTL cache keyed by `role`, returning the list of allowed `feature:function` slugs for that role. |

# Problem

The current implementation runs two database queries on every
authenticated API request:

1. `LoadFeatures` — `SELECT f.slug FROM plan_features JOIN ...
   WHERE t.id = ?` (per tenant)
2. `LoadRolePermissions` — `SELECT f.slug, ff.slug FROM
   role_function_permissions JOIN ... WHERE rfp.role = ?` (per role)

At 100 requests/second this is 200 unnecessary DB queries/second.
Features and permissions change rarely (plan changes, admin
permission edits, new deployments), so caching is safe.

# Design

## Token Structure

### Current Claims

```json
{
  "uid": "user-uuid",
  "tid": "tenant-uuid",
  "role": "bookkeeper",
  "exp": 1750000000
}
```

### Proposed Claims

```json
{
  "uid": "user-uuid",
  "tid": "tenant-uuid",
  "role": "bookkeeper",
  "features": ["invoicing", "bills", "bank_reconciliation", "gst_bas", "financial_reports"],
  "functions": ["invoicing:view_invoices", "invoicing:create_invoice", "bills:view_bills", "bills:create_bill"],
  "exp": 1750000000
}
```

## Token Lifecycle

| Event | Access TTL | Refresh TTL |
|---|---|---|
| Login | 30 min | 7 days |
| Switch tenant | 30 min | 7 days |
| Refresh | 30 min | unchanged |

The frontend auto-refreshes every **25 minutes** (5 min before
expiry) via a background timer. On refresh, the API re-resolves
features + permissions (from cache or DB) and issues a new access
token with updated claims.

## Cache Architecture

### Feature Cache

```
Key:    tenant_id (UUID string)
Value:  []string (feature slugs)
TTL:    15 minutes
Source: SELECT f.slug FROM plan_features pf
        JOIN features f ON f.id = pf.feature_id
        JOIN tenants t ON t.plan_id = pf.plan_id
        WHERE t.id = ? AND pf.enabled = true
```

### Permission Cache

```
Key:    role (string: "owner", "bookkeeper", etc.)
Value:  []string (feature:function slugs)
TTL:    15 minutes
Source: SELECT f.slug || ':' || ff.slug
        FROM role_function_permissions rfp
        JOIN feature_functions ff ON ff.id = rfp.function_id
        JOIN features f ON f.id = ff.feature_id
        WHERE rfp.role = ? AND rfp.allowed = true
```

### Cache Behaviour

- Cache is in-memory only (`jellydator/ttlcache/v3`)
- Thread-safe with concurrent reads from HTTP middleware
- Background goroutine evicts expired entries
- No explicit invalidation needed:
  - Plan feature changes → deployment → cache cleared (process restart)
  - Role permission changes → deployment → cache cleared
  - New features/functions → deployment → cache cleared
  - Tenant changes plan (admin action) → 15 min TTL handles it
- Maximum staleness: 15 min (cache) + 30 min (token) = 45 min worst case
- At steady state: **zero DB queries** for permission checks

### Cache Package

**`github.com/jellydator/ttlcache/v3`** — chosen for:
- Go generics (`Cache[string, []string]`)
- Native per-entry TTL
- Built-in Loader function (auto-populate on miss, deduplicated)
- Zero external dependencies
- Actively maintained

## Middleware Changes

### Before (current)

```
Request → TenantAuth → LoadFeatures(DB query) → LoadRolePermissions(DB query) → Handler
```

### After (proposed)

```
Request → TenantAuth → read features from JWT claims → read functions from JWT claims → Handler
                        (zero DB queries)

Login/Refresh → check cache(tenant_id) → HIT → embed in JWT (zero DB queries)
                                        → MISS → query DB → cache 15 min → embed in JWT
```

### LoadFeatures middleware (updated)

```go
func LoadFeatures(...) gin.HandlerFunc {
    return func(c *gin.Context) {
        claims := UserFromContext(c.Request.Context())
        if claims != nil && len(claims.Features) > 0 {
            // Read from JWT — zero DB query
            ctx := ContextWithFeatures(c.Request.Context(),
                NewFeatureSet(claims.Features))
            c.Request = c.Request.WithContext(ctx)
        }
        c.Next()
    }

```

### Cache initialisation (main.go)

```go
import "github.com/jellydator/ttlcache/v3"

featureCache := ttlcache.New[string, []string](

)

permissionCache := ttlcache.New[string, []string](
    ttlcache.WithTTL[string, []string](15 * time.Minute),

```

### Token refresh with cache

```go
func (s *AuthService) resolveFeatures(ctx context.Context, tenantID string) []string {
    if item := s.featureCache.Get(tenantID); item != nil {
        return item.Value() // cache hit
    }
    // Cache miss — query DB
    slugs := queryFeaturesFromDB(ctx, tenantID)
    s.featureCache.Set(tenantID, slugs, ttlcache.DefaultTTL)
    return slugs
}
```

## Frontend Changes

### Auto-refresh timer

```typescript
// In auth provider — refresh token 5 min before expiry
useEffect(() => {
  const interval = setInterval(() => {
    refreshAccessToken()
  }, 25 * 60 * 1000) // 25 minutes
  return () => clearInterval(interval)
}, [])
```

### useFeatures hook (updated)

```typescript
// Read features directly from JWT claims instead of API call
export function useFeatures() {
  const { claims } = useAuth()
  const featureSet = new Set(claims?.features ?? [])
  return {
    hasFeature: (slug: string) => featureSet.has(slug),
    allEnabled: !claims?.features || claims.features.length === 0,
  }
}
```

This eliminates the `GET /features` API call entirely — the
frontend reads permissions from the token it already has.

## Cross-Machine Cache Invalidation

When a tenant upgrades their plan, they expect access to new
features immediately — not in 45 minutes. The cache must be
flushable across all running API machines.

### Approach: DB-signalled cache version

A single integer row in the platform DB acts as a cache version
counter. When the cache needs flushing, the version is incremented.
Each machine polls the version periodically and clears its local
cache when a change is detected.

### Schema

```sql
CREATE TABLE IF NOT EXISTS cache_version (
    id      INT PRIMARY KEY DEFAULT 1 CHECK (id = 1),
    version BIGINT NOT NULL DEFAULT 1,
    updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
INSERT INTO cache_version (id, version) VALUES (1, 1)
ON CONFLICT DO NOTHING;
```

Single row, single integer. Increment to invalidate all caches.

### API endpoint

```
POST /internal/cache/flush   (platform admin only)
  → UPDATE cache_version SET version = version + 1, updated_at = now()
  → returns { "version": N, "message": "cache flush signalled" }
```

### Machine-side polling

Each API machine runs a background ticker (every 60 seconds):

```go
func (s *CacheWatcher) Start(ctx context.Context) {
    ticker := time.NewTicker(60 * time.Second)
    for {
        select {
        case <-ticker.C:
            var current int64
            s.db.Raw("SELECT version FROM cache_version WHERE id = 1").Scan(&current)
            if current != s.lastVersion {
                s.featureCache.DeleteAll()
                s.permissionCache.DeleteAll()
                s.lastVersion = current
                s.logger.Info("cache flushed by version signal", "version", current)
            }
        case <-ctx.Done():
            ticker.Stop()
            return
        }
    }
}
```

### When flush is triggered

| Event | Who triggers | Propagation |
|---|---|---|
| Tenant upgrades plan | Platform admin via `ChangeTenantPlan()` | Auto: calls `/internal/cache/flush` |
| Plan features edited | Platform admin via `SetPlanFeatures()` | Auto: calls `/internal/cache/flush` |
| Role permissions edited | Platform admin via `SetRolePermissions()` | Auto: calls `/internal/cache/flush` |
| Manual flush | Sysops via API or admin UI | Manual: `POST /internal/cache/flush` |

### Propagation timeline

```
Admin changes plan
  → cache_version incremented (instant)
  → machines detect within 60s (polling)
  → local caches cleared
  → next login/refresh rebuilds from DB
  → new JWT has updated features
  → tenant has access within ~90s (60s poll + 30s token refresh)
```

For the upgrading tenant specifically: after the plan change, the
frontend can trigger an immediate token refresh (`POST /auth/refresh`)
which forces the API to rebuild the JWT with fresh features from
the DB (cache miss after flush). This gives the tenant access to
new features **within seconds** of the upgrade.

### Frontend: force refresh on plan change

When the tenant's plan is changed (either by the admin or by the
tenant upgrading via self-service), the frontend should call
`refreshAccessToken()` immediately rather than waiting for the
25-minute timer. This ensures the user sees their new features
without delay.

## Performance with Cache Invalidation

| Metric | Value |
|---|---|
| DB queries per request | 0 |
| DB queries on login/refresh | 0-1 (cache hit/miss) |
| DB queries for cache polling | 1 per 60s per machine (lightweight) |
| Cache flush propagation | ~60s across all machines |
| Tenant upgrade to access | ~seconds (force refresh) |

With 4 machines polling every 60s, that's 4 queries/minute —
negligible. The `cache_version` query is a single-row primary
key lookup, essentially free.

# Out of Scope

- Redis/external cache (not needed at current scale)
- Per-user permission overrides (future — owners granting extra
  access to individual bookkeepers)
- JWT encryption (signing is sufficient for integrity)
