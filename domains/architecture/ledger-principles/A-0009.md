# Summary

Defines 24 non-negotiable backend architecture principles for Ledgius as a deterministic, auditable, idempotent, and recoverable accounting system built in Go. These principles govern all financial data handling across the ledger, posting engine, domain services, and API layer.

# Requirement Link

N/A — this is a foundational architecture document that all domain requirements inherit from.

# Technical Context

Ledgius is migrating from LedgerSMB (Perl + PL/pgSQL) to a Go API + React frontend. Business logic is being extracted from PostgreSQL stored procedures into Go domain services. These principles ensure accounting correctness is enforced through invariants and architecture rather than trust or ad-hoc implementation.

# Proposed Design

## Core Objective

Design a deterministic, auditable, idempotent, and recoverable accounting system.

## Principle 1: Ledger as Source of Truth

All financial state must derive from immutable journal entries.

## Principle 2: Double-Entry Enforcement

Enforced at:
- Application layer
- Database constraints
- Posting engine

## Principle 3: Domain-Driven Commands

APIs accept business intent, not raw journal lines.

## Principle 4: Atomic Transactions

All accounting mutations occur in a single DB transaction.

## Principle 5: Idempotency

All write endpoints must support idempotency keys.

## Principle 6: Immutability of Posted Data

No edits. Only reversals or adjustments.

## Principle 7: Audit Trail

Append-only logs with full traceability.

## Principle 8: Period Controls

Strict enforcement of open/closed periods.

## Principle 9: Monetary Precision

Use decimal types. Explicit rounding rules.

## Principle 10: Centralized Posting Engine

Single service for all accounting logic.

## Principle 11: Permissions & Approvals

Role-based with separation of duties.

## Principle 12: CQRS Separation

Strict write model, optimized read model.

## Principle 13: Reconciliation

Continuous integrity validation.

## Principle 14: Import Staging

Validate before posting.

## Principle 15: Concurrency Control

Use locks, constraints, idempotency.

## Principle 16: Business Identifiers

Stable unique references.

## Principle 17: Observability

Correlation IDs and structured logs.

## Principle 18: Backup & Recovery

Backups + tested restore procedures.

## Principle 19: Domain Invariants

Explicit, enforced, testable.

## Principle 20: Testing Strategy

Unit, integration, replay, failure.

## Principle 21: State Machines

Explicit lifecycle states.

## Principle 22: Logical Operation Capture

Append-only domain events.

## Principle 23: Minimize Hidden DB Logic

Avoid opaque triggers.

## Principle 24: Accounting Truth Boundary

Separate workflow from ledger.

## Key Principle

Accounting correctness is achieved through invariants, not trust.

# Affected Components

- All Go domain packages under `api/internal/`
- Posting engine (`api/internal/journal/`)
- All financial write endpoints
- Database schema constraints
- Audit subsystem

# Data Flow

```
Client Request --> Gin Handler --> Domain Service (business logic + tx boundary)
  --> Repository (DB read/write within tx) --> PostgreSQL (constraints + immutability)
  --> Audit Logger (append-only event)
```

# API / Interface Changes

All write endpoints must:
- Accept idempotency keys
- Return business-intent responses (not raw journal lines)
- Enforce CQRS separation between write and read paths

# Storage / Schema Changes

- All monetary columns use PostgreSQL `NUMERIC` / Go `shopspring/decimal`
- Journal entries are append-only (no UPDATE/DELETE on posted records)
- Period controls enforced via database constraints and service-layer checks
- Audit trail stored as append-only event log

# Risks / Trade-offs

- **Immutability increases storage** — append-only journals and audit logs grow; mitigated by archival and partitioning strategies.
- **CQRS complexity** — separate read/write models add implementation overhead; justified by correctness and performance benefits.
- **Centralized posting engine** — single point of complexity; mitigated by comprehensive testing (Principle 20).

# Alternatives Considered

- **Trust-based correctness**: Relying on developer discipline rather than architectural invariants. Rejected — accounting systems require enforced guarantees.
- **Mutable ledger entries**: Allowing direct edits to posted data. Rejected — violates accounting standards and auditability requirements.

# Test Strategy

- Unit tests for all business logic in domain services
- Integration tests verifying double-entry enforcement at the database level
- Replay tests for idempotency verification
- Failure injection tests for transaction atomicity
- Parity tests comparing Go-driven output against legacy stored procedure output

# Rollout / Migration Notes

These principles are enforced from day one of the Go implementation. They apply to all new domain code and all business logic extracted from legacy stored procedures. See A-0010 (Transaction Management) and A-0015 (Migration Safety) for extraction and migration details.
