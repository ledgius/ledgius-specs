# Summary

Exact double-entry posting rules for every fixed-asset event: acquisition (cash or bill-linked, with/without GST), disposal (gain/loss), depreciation (per-asset in a per-period parent transaction), depreciation reversal, disposal reversal, and correction. Each rule specifies accounts, amounts, BAS tax labels, deductibility flags, and the parent-child transaction / acc_trans shape.

The posting engine described here is the **only** path that mutates ledger rows for assets — the asset service never writes `acc_trans` directly. It calls `journal.Post(ctx, cmd)` with a domain command, and the journal service applies this contract.

# Requirement Link

Implements the GL-posting portion of R-0062 per the ledger principles in A-0009 (double-entry, atomicity, idempotency, immutability). Consumed by T-0030 (acquisition), T-0031 (disposal), T-0032 (depreciation).

# Scope boundary — three layers

This document specifies the **GL accounting layer only**. Fixed-asset events produce other outputs that are **not** in scope here:

1. **GL accounting layer** (this doc) — double-entry journal rows produced per event under AASB 116/108/136 accounting rules. Source of truth for the financial statements.
2. **BAS / GST reporting layer** (forthcoming **A-0042**) — derives G1, G10, G11, 1A, 1B figures for the quarterly BAS. BAS reports **read** the GL rows plus the source tax documents (bills, invoices, tax codes, creditable %, reporting basis) — they are *not* driven by labels baked onto `acc_trans`.
3. **Income-tax depreciation &amp; disposal layer** (forthcoming **R-0073**) — computes Div 40 balancing adjustment on disposal, CGT component for the non-taxable-use portion, simplified depreciation / small business pool where applicable. Tax outcomes are computed in a separate layer at year-end and ITR time; they do **not** equal the accounting gain/loss from this doc.

Corollary — and a reversal from the previous draft of this spec: `acc_trans` rows do **not** carry `bas_label` tags. BAS labels are a reporting concept, not a GL concept. Tagging them on rows would (a) double-count on bill-reclassification journals where the original bill already sits in BAS-scope and (b) misrepresent BAS-cash-vs-accrual method differences. The BAS layer will join to the source tax document instead.

# Technical Context

Three reasons the contract lives in its own document:

1. **Compliance review**: accountants review a single focused doc to verify GST posting and gain/loss mapping are correct, without diving into task-level detail.
2. **Test surface**: each rule maps to a unit test with fixed inputs and expected journal rows — the contract *is* the test oracle.
3. **Reuse**: mobile and future integrations post assets through the same service; the contract pins behaviour regardless of caller.

Existing ledger tables (`account`, `acc_trans`, `transactions`) are unchanged. Asset-specific tags (`asset_id`, `run_id`, `correction_id`, `deductible`, `reclass_from_bill`) are added as metadata columns on `acc_trans` per T-0029's migration. `bas_label` is **intentionally not** in that tag set — see the scope-boundary note above.

# Proposed Design

## 1. Account Resolution

Each asset carries three nominal accounts resolved at acquisition time and cached on the `asset_register` row:

- `capital_account_id` — where cost sits on the balance sheet (category A).
- `accum_depreciation_account_id` — contra-asset accumulated dep (category A, contra).
- `depreciation_expense_account_id` — P&L expense for periodic depreciation (category E).

Resolution order:

1. Explicit override on the asset row (user picked specific accounts during Buy).
2. Defaults from the asset's `asset_category` row.
3. Tenant-level system defaults (seeded per COA template).

Disposal-time accounts (Gain on Disposal, Loss on Disposal) are resolved at disposal time from tenant COA defaults — not stored on the asset.

**AASB 116 §68 requires gain/loss on derecognition of PPE to be recognised in profit or loss, but gains are *not* classified as revenue.** The COA mapping therefore must route these to an "Other Income / Other Expense" reporting group, not to operating revenue. To enforce this, the `account` rows used for disposal gain/loss carry:

- `reporting_category = 'other_income'` (for Gain on Disposal) or `'other_expense'` (for Loss on Disposal)
- COA-level category is still I (Income) or E (Expense) for P&amp;L placement, but P&amp;L subtotalling treats `reporting_category` as the presentation driver — operating revenue roll-ups exclude rows flagged `other_income` / `other_expense`.

Typical accounts:

- Gain on Disposal of Assets — category I, `reporting_category='other_income'`.
- Loss on Disposal of Assets — category E, `reporting_category='other_expense'`.
- Some tenants use a single "Gain/Loss on Disposal" account — the contract supports either shape; sign of the amount disambiguates.

COA seed templates (ledgius-db `coa/au`) must include these two accounts (or the single combined variant) with `reporting_category` populated. T-0029 migration validates the tenant COA has a disposal gain/loss account resolvable before any disposal command can run.

Bank / AP accounts are caller-supplied at acquisition/disposal time (which bank, which AP).

GST input credit (on acquisition) uses the tenant's standard GST Paid account (category A — asset, since it's recoverable).
GST output liability (on disposal) uses the tenant's standard GST Collected account (category L).

## 2. Acquisition — Cash (Standalone)

**Inputs**: asset, `cost_ex_gst`, `gst_amount` (≥ 0), `source_bank_account_id`, `purchase_date`, `gst_applies` (bool).

Parent `transactions`: `type='asset_acquisition'`, `reference='ASSET-{asset_id}-ACQ'`, `posted_at={now}`, `period={month of purchase_date}`.

Child `acc_trans` rows:

| # | Account | Side | Amount | Tags |
|---|---|---|---|---|
| 1 | `asset.capital_account_id` | Dr | `cost_ex_gst` | `asset_id={asset.id}` |
| 2 | GST Paid (asset) | Dr | `gst_amount` (if `gst_applies && gst_amount > 0`, else row omitted) | `asset_id={asset.id}` |
| 3 | `source_bank_account_id` | Cr | `cost_ex_gst + gst_amount` | `asset_id={asset.id}` |

**Invariants**: sum Dr = sum Cr. Test: `cost_ex_gst + gst_amount == credit_total`.

**BAS layer** (per scope boundary, derived elsewhere): A-0042 identifies this acquisition as eligible for G10 (if `cost_ex_gst > $1,000`) or G11 (if `cost_ex_gst ≤ $1,000`) by joining the source tax document (or, for cash mode, the tax code attached to the capital line) and the tenant's BAS reporting basis (cash vs accrual). The `acc_trans` row itself carries no BAS label.

## 3. Acquisition — Bill-Linked

**Inputs**: asset, `bill_id` (existing AP bill), optional `bill_line_ids` (if asset is one line among many).

Preconditions: bill is in state `Open` or `Approved` (not yet paid does not matter — the acquisition is posted when the asset is capitalised, not when the bill is paid).

Parent `transactions`: `type='asset_acquisition'`, `reference='ASSET-{asset_id}-ACQ'`, `source_transaction_id={bill.transaction_id}`.

Child `acc_trans` rows: the bill's line(s) are **moved** from the tenant's default expense account to the asset's capital account via a reclassification journal:

| # | Account | Side | Amount | Tags |
|---|---|---|---|---|
| 1 | Tenant default expense (from bill line) | Cr | `cost_ex_gst` | `asset_id`, `reclass_from_bill={bill_id}` |
| 2 | `asset.capital_account_id` | Dr | `cost_ex_gst` | `asset_id`, `reclass_from_bill={bill_id}` |

GST treatment on the bill is unchanged (already posted when the bill was created). No new GST rows.

**BAS layer note**: the reclassification journal carries **no** BAS tags. The original bill is already in the BAS extract for the period it was posted in; re-tagging the reclassification would double-count G10. A-0042 BAS extraction joins to the source bill for BAS purposes, not to the reclassification journal. The `reclass_from_bill` tag on the reclassification rows lets A-0042 identify them as "reclass only — ignore for BAS".

**Alternative** (simpler for users creating the bill at the same time as the asset): the Buy Asset flow creates a brand-new bill with the capital account directly on the line — no reclassification needed. This is the default for new bills initiated from the asset flow; reclassification covers the case of capitalising an existing bill line that was misposted.

## 4. Depreciation — Periodic Run

**Inputs**: `period_end`, list of eligible assets with their calculated per-asset amounts.

Parent `transactions`: `type='depreciation_run'`, `reference='DEP-{period_end:YYYY-MM}'`, one parent per run.

Child `acc_trans` rows: two per eligible asset.

For each asset `a`:

| # | Account | Side | Amount | Tags |
|---|---|---|---|---|
| 1 | `a.depreciation_expense_account_id` | Dr | `a.calculated_amount` | `asset_id={a.id}`, `run_id={run.id}`, `deductible={true ∨ business_use_pct for vehicles}` |
| 2 | `a.accum_depreciation_account_id` | Cr | `a.calculated_amount` | `asset_id={a.id}`, `run_id={run.id}` |

For Motor Vehicles with `business_use_pct < 100`, the expense is split across two rows on the Dr side: deductible portion (tagged `deductible=true`) and non-deductible portion (tagged `deductible=false`). Credit side is one row for the full amount.

The `deductible` tag is ledger-level presentation metadata — it splits the expense row for reporting clarity. It is **not** a tax determination. The income-tax-deductible amount (Div 40, FBT interactions, entity-type-specific rules) is computed in R-0073's tax layer using the asset's category, business-use %, entity type, and ownership structure — not by reading `deductible` alone.

**Invariants**: per-asset Dr total = per-asset Cr total. Run-level Dr total = Run-level Cr total.

## 5. Depreciation — Reversal

**Inputs**: `run_id`, optional `asset_ids` (subset for per-asset reversal; empty = whole run).

Parent `transactions`: `type='depreciation_reversal'`, `source_transaction_id={original run.transaction_id}`.

Child `acc_trans` rows: the exact opposite sides of the original run's rows, for the affected assets only. No amount change — the reversal offsets, it does not recompute.

On the `depreciation_run` row, `status` flips to `reversed` (or `partially_reversed` when asset-scoped). The original rows are NOT deleted.

## 6. Disposal

**Inputs**: `asset_id`, `disposal_date`, `sale_proceeds_inc_gst`, `gst_on_sale`, `reason`, optional `buyer_entity_id`, optional `destination_bank_account_id` (for cash sale), optional `ar_entity_credit_account_id` (for credit sale).

Preconditions: asset in `Active` or `FullyDepreciated`. If `disposal_date` > last dep run's `period_end` for this asset, a pro-rata depreciation entry is posted first (see §4, same transaction, one extra asset-scoped child parent? — no: same parent). Simpler: the disposal parent transaction includes both the pro-rata dep child rows AND the disposal child rows.

### 6.1 Derived values

```
cost              = asset.cost_ex_gst
accumulated       = asset.accumulated_depreciation + pro_rata_added_this_disposal
book_value        = cost − accumulated
sale_ex_gst       = sale_proceeds_inc_gst − gst_on_sale
gain_loss         = sale_ex_gst − book_value        (positive = gain, negative = loss)
```

### 6.2 Child acc_trans rows

Parent `transactions`: `type='asset_disposal'`, `reference='ASSET-{asset_id}-DISPOSAL'`.

Pro-rata dep rows (if any) — see §4.

Disposal rows:

| # | Account | Side | Amount | Tags |
|---|---|---|---|---|
| D1 | Bank or AR (caller) | Dr | `sale_ex_gst` | `asset_id`, `disposal=true` |
| D2 | GST Collected (liability) | Cr | `gst_on_sale` (if > 0, else omitted) | `asset_id`, `disposal=true` |
| D3 | `asset.accum_depreciation_account_id` | Dr | `accumulated` (clear it) | `asset_id`, `disposal=true` |
| D4 | `asset.capital_account_id` | Cr | `cost` | `asset_id`, `disposal=true` |
| D5 | Gain on Disposal (`reporting_category='other_income'`) | Cr | `gain_loss` **if > 0**, else omitted | `asset_id`, `disposal=true` |
| D6 | Loss on Disposal (`reporting_category='other_expense'`) | Dr | `−gain_loss` **if < 0**, else omitted | `asset_id`, `disposal=true` |

**BAS layer note**: A-0042 identifies the disposal for BAS reporting (G1 — total sales, 1A — GST on sales) by joining the parent `transactions` (type=`asset_disposal`) plus the tax code on the sale side. The `acc_trans` rows carry no BAS tags.

**Tax layer note**: the accounting gain/loss computed here is **not** the Div 40 balancing adjustment and **not** the CGT outcome. R-0073 computes:
- Balancing adjustment under Div 40 = `termination_value − adjustable_value` for tax purposes (private-use portions excluded from the Div 40 path).
- CGT component under s118-24 for the non-taxable-use proportion (if any) and for pre-CGT assets.
These tax calculations run at year-end / ITR time against the GL rows posted here plus asset-level metadata (entity type, business-use %, acquisition date for pre-CGT test).

**Invariant**: `sale_ex_gst + gst_on_sale + accumulated (Dr) == cost + gst_on_sale (Cr) + |gain_loss|` where the sign of gain_loss determines which side carries the difference. Concretely in both gain and loss scenarios:

```
Dr total = sale_ex_gst + accumulated [+ loss_amount if loss]
Cr total = cost [+ gst_on_sale if GST] [+ gain_amount if gain]
Dr total == Cr total
```

### 6.3 Scrap / donation (zero proceeds)

`sale_proceeds_inc_gst = 0`, `gst_on_sale = 0`. Result: D1 and D2 omitted. `gain_loss = −book_value` (a loss). D6 = book_value. Invariant holds.

## 7. Disposal Reversal

Rare — only when a disposal was wrong. Admin-level. Parent `transactions`: `type='asset_disposal_reversal'`, `source_transaction_id={disposal.transaction_id}`. Rows are exact inverses of §6 rows, including any pro-rata dep rows.

## 8. Correction — four distinct paths

AASB 108 distinguishes between **error correction** (retrospective or current-period) and **change in accounting estimate** (prospective). This contract mirrors that distinction with four separate commands — each with its own GL shape. A-0040 §3 is the state-machine side of the same split.

### 8.1 Clerical edit (no GL)

Non-posting-field edit — asset name, description, notes. No `acc_trans` rows. Audit row only with diff. Handled by A-0040 §3 `EditAssetNonPosting` command. No journal here; this section is the cross-reference.

### 8.2 Same-period reclassification (reverse + repost, same period)

Error caught before the period closes. Examples: wrong capital account, wrong supplier, wrong GST treatment within an open period.

Two journals in one DB transaction:

1. Reversal: inverses of the acquisition (or acquisition + any dep posted this period). Parent `transactions`: `type='asset_reclass_reversal'`, `correction_id={uuid}`.
2. Fresh acquisition per §2 or §3. Parent `transactions`: `type='asset_acquisition'`, `correction_id={same uuid}`, `source_transaction_id={reversal id}`.

Period lock check: original period must be OPEN and not BAS-lodged (§10a). If lodged/locked, route to §8.4 instead.

### 8.3 Change in accounting estimate (prospective only, no GL impact)

Change to useful life or residual value based on new information (AASB 116 §61, AASB 108 §32–40). **This is NOT an error** — it's a revised estimate, applied prospectively.

No reversal journal. No change to already-posted depreciation. The asset's `useful_life_years` and/or `residual_value` are updated via a dedicated command; future depreciation runs use the new values. Asset carries an `estimate_changes[]` log so prior values are auditable.

Audit row: `estimate_changed` with `{field, old, new, effective_from, reason}`. No `acc_trans` rows.

Recalc formula post-change (both methods):
- Remaining book value = `cost − accumulated_depreciation_to_date` (unchanged at the moment of change).
- Remaining periods for straight line = `new_useful_life − periods_elapsed`.
- Going-forward annual = `(remaining_book_value − new_residual) / remaining_periods` for straight line; diminishing value uses the standard formula with the new life and the current book value.

### 8.4 Prior-period restatement (AASB 108 retrospective)

Material error affecting prior-period financial statements. AASB 108 §41–49 requires retrospective restatement.

Constraints:
- Requires `assets:correct` + `admin:restate_prior_period` function permissions (tightest role gate).
- Must carry a documented reason and reviewer sign-off (captured in audit payload).
- Prior period is CLOSED/LODGED — we do **not** rewrite its rows. We post the correction to the **current** open period.

GL shape: a single current-period parent `transactions` of `type='asset_prior_period_restatement'`. Child rows post the net correction to:

- `capital_account_id` / `accum_depreciation_account_id` — the balance-sheet difference rolled into the current period.
- **Retained Earnings (Opening)** — the P&amp;L portion of prior-period error goes here, never to current-period P&amp;L. This preserves the comparatives treatment per AASB 108.

Every row carries `restatement_id` and `restatement_period={YYYY-MM}` pointing to the period being restated.

Financial-statement presentation (re-issued comparatives with prior-period figures restated) is handled in the reporting layer, which reads these rows and adjusts the comparative column — out of scope for this contract.

### 8.5 Depreciation NOT auto-replayed

Across §8.2 and §8.4, depreciation is not automatically recomputed. Rationale:
- §8.2 same-period reclass: next periodic run picks up the corrected asset from the current period forward. Mid-period catch-up is posted as a normal dep run line.
- §8.4 prior-period restatement: the restatement amount already reflects the depreciation differential; replaying would double-book.

### 8.6 Command → contract mapping

| A-0040 command | This doc section | Period lock behaviour |
|---|---|---|
| `EditAssetNonPosting` | §8.1 (no GL) | Allowed in any state |
| `ReclassifyCurrentPeriod` | §8.2 | Rejected if target period locked |
| `ChangeDepreciationEstimate` | §8.3 | Allowed — prospective, no past-period writes |
| `RestatePriorPeriod` | §8.4 | Required path when target period is locked |

## 9. Idempotency

Every asset journal carries an idempotency key:

- Acquisition: `asset:acquire:{asset_id}`.
- Depreciation run: `dep_run:{period_end}:{run_uuid}` — new UUID per run, blocks double-post of the same run.
- Per-asset dep line: `dep_line:{run_id}:{asset_id}` — uniqueness constraint prevents duplicates.
- Disposal: `asset:dispose:{asset_id}`.
- Reversal: `asset:reverse:{source_transaction_id}`.

`journal.Post` rejects duplicate idempotency keys with `ErrDuplicateJournal` per A-0009 §Principle 5.

## 10. Tag schema

Every asset-originated `acc_trans` row MUST carry:

- `asset_id` — the asset it belongs to.

Optional tags per event:

- `deductible` — `true`/`false` on dep expense rows (vehicles with business-use-%). Ledger-level presentation only; **not** a tax determination (R-0073 owns tax deductibility).
- `run_id` — on per-run dep rows.
- `disposal` — `true` on disposal rows.
- `correction_id` — on correction reversal + fresh rows (see A-0040 §3 for the four correction command variants).
- `restatement_id` — on prior-period restatement rows (AASB 108 retrospective correction).
- `reclass_from_bill` — on bill-linked acquisition reclassification rows. Signals to A-0042 BAS extractor: "ignore for BAS" (original bill already counted).

**Not in the tag set**:
- `bas_label` — BAS labels (G1/G10/G11/1A/1B) are derived by A-0042 from source tax documents + tax codes + reporting basis, not tagged on GL rows. See scope-boundary note at top of this doc.

These tags drive: period close (run_id), audit queries (correction_id, restatement_id), BAS extractor filtering (reclass_from_bill), and expense-split reporting (deductible).

## 10a. Closed-period and BAS-lodgement locks

A GL command produced by this contract MUST check period locks before writing:

- `period_lock(tenant_id, period).locked = true` → reject with `ErrPeriodLocked`. Caller must route through a current-period reclassification or prior-period restatement command (see A-0040 §3).
- `bas_lodgement(tenant_id, period).lodged = true` → same rejection rule. BAS-lodged periods cannot be silently rewritten.

This applies to acquisition (if `purchase_date` falls in a locked period), depreciation runs (if `period_end` falls in a locked period), disposals (if `disposal_date` falls in a locked period), and reversals (if the target original row is in a locked period).

The prior-period restatement command (A-0040 §3) is the **only** path that legally writes GL rows referencing a locked period — and it does so by posting current-period correction entries with `restatement_id` tags, not by retroactively editing the locked period's rows.

## 11. Testing Requirements

Per A-0009 §Principle 22 (ledger must balance). Table-driven tests for:

- Acquisition cash with GST, without GST, GST-free supplier, cost ≤ $1,000 (future G11 case — no BAS tag on the row, verified).
- Acquisition bill-linked with reclassification — verifies the reclassification journal carries `reclass_from_bill` and no BAS-label (A-0042 will treat as "ignore for BAS").
- Acquisition bill-linked with new bill created same tx.
- Depreciation run — straight line, diminishing value, instant write-off cases in one run.
- Depreciation reversal — whole run, single asset.
- Disposal — gain, loss, scrap, credit sale, cash sale, motor vehicle with business-use %.
- Disposal with pro-rata dep trigger (disposal date between runs).
- Disposal reversal.
- Gain row maps to an account with `reporting_category='other_income'`; P&amp;L subtotal excludes it from operating revenue (assertion in reporting-layer test; referenced here for traceability).
- §8.2 `ReclassifyCurrentPeriod` — same-period reverse+repost in open period.
- §8.2 rejected when target period is locked.
- §8.3 `ChangeDepreciationEstimate` — prospective update; no acc_trans rows written; audit `estimate_changed` row only; next run uses new values.
- §8.4 `RestatePriorPeriod` — current-period correction with `restatement_id` tag; P&amp;L portion routes to Retained Earnings (Opening), not current P&amp;L.
- §8.4 requires both `assets:correct` and `admin:restate_prior_period` permissions.
- §10a — every command (acquire, run, reverse, dispose) rejects when the target period is locked, except `RestatePriorPeriod`.

Each test asserts:
- Exact accounts, sides, amounts, tags.
- `sum(Dr) == sum(Cr)` for the whole transaction.
- `asset_register.accumulated_depreciation` post-run matches ledger sum.
- `book_value = cost − accumulated` post-event.
- Audit row exists for the event with correct actor + diff.

## 12. Observability

Per A-0015 (not yet written — noting hook), each asset command emits a structured log with:
- `asset_id`, `tenant_id`, `actor_user_id`.
- `command`, `transaction_id`, `idempotency_key`.
- `amounts` (summary: cost, gst, book_value, gain_loss).
- `duration_ms`.

Metrics: `asset.acquisitions_total`, `asset.depreciation_runs_total`, `asset.disposals_total`, `asset.corrections_total`.
