# Summary

Defines a three-pillar migration safety framework guaranteeing that the new Ledgius system produces identical accounting outcomes to the legacy LedgerSMB system at every step of migration. The three pillars are: parity assertions (Go test helpers for row-level data equivalence), dual-post gateway (routing every write to both systems), and transaction atomicity (all-or-nothing database operations).

# Requirement Link

N/A — foundational architecture document governing migration safety from LedgerSMB to Ledgius.

# Technical Context

Ledgius is progressively replacing LedgerSMB's PL/pgSQL stored procedures with Go domain services. During migration, both systems must operate in parallel with automated verification ensuring no accounting divergence. This framework treats migration safety as an architectural concern, not merely a testing concern.

# Proposed Design

## Core Objective

Guarantee that the new Ledgius system produces **identical accounting outcomes** to the legacy LedgerSMB system at every step of the migration, with automated verification and atomic transaction control.

## Pillar 1: Parity Assertions (Go Test Helpers)

Reusable Go test helper functions that verify row-level data equivalence between the legacy and Ledgius databases.

### Design

```go
// pkg/parity/assert.go

// AssertRowParity checks that a specific row in a table matches across both databases.
// It compares all columns (or a specified subset) by a given identity key.
func AssertRowParity(t *testing.T, opts ParityCheck) {
    t.Helper()
    // Query both DBs, compare, report differences
}

type ParityCheck struct {
    Table       string            // e.g. "gl" or "acc_trans"
    IdentityKey string            // e.g. "id" or "trans_id"
    IdentityVal any               // the row's key value
    Columns     []string          // nil = all columns; else specific subset
    IgnoreCols  []string          // columns to skip (e.g. internal IDs, timestamps)
    ColMapping  map[string]string // maps Ledgius column names to legacy names if schemas diverge
}
```

### Capabilities

- **Single row**: assert one row matches by primary key
- **Multi-row / set**: assert all rows for a foreign key match (e.g. all `acc_trans` rows for a given `trans_id`)
- **Aggregate**: assert computed values match (e.g. trial balance totals, account balances)
- **Schema-aware**: column name mapping when Ledgius schema diverges from legacy
- **Diff output**: on failure, produce a clear column-by-column diff showing expected vs actual

### Usage in Tests

```go
func TestPostInvoice_Parity(t *testing.T) {
    // 1. Post invoice via dual-post gateway
    inv := postInvoice(t, invoiceFixture)

    // 2. Assert GL entries match
    parity.AssertSetParity(t, parity.SetCheck{
        Table:       "acc_trans",
        ForeignKey:  "trans_id",
        ForeignVal:  inv.TransID,
        OrderBy:     "chart_id, amount",
        IgnoreCols:  []string{"entry_id", "approved"},
    })

    // 3. Assert invoice header matches
    parity.AssertRowParity(t, parity.ParityCheck{
        Table:       "ar",
        IdentityKey: "invnumber",
        IdentityVal: inv.InvNumber,
        IgnoreCols:  []string{"id"},
    })

    // 4. Assert trial balance still in sync
    parity.AssertTrialBalance(t)
}
```

### Table Coverage Progression

| Phase | Tables Under Parity Check |
|-------|--------------------------|
| Early | `gl`, `acc_trans`, `ar`, `ap` |
| Mid   | `+ invoice`, `+ payment`, `+ entity_credit_account` |
| Late  | All financially significant tables |

## Pillar 2: Dual-Post Gateway

A mechanism that routes every write operation to both the Ledgius system and the legacy LedgerSMB system, ensuring both receive identical business intent.

### Architecture

```
                         +---------------------+
   Client Request ------>|  Gin Handler         |
                         |  (domain package)    |
                         +----------+-----------+
                                    |
                                    v
                         +---------------------+
                         |  Domain Service      |
                         |  (business logic)    |
                         +----------+-----------+
                                    |
                                    v
                         +---------------------+
                         |  Dual-Post Gateway   |
                         |  (pkg/dualpost)      |
                         +------+-------+------+
                                |       |
                    +-----------+       +-----------+
                    v                               v
          +------------------+            +------------------+
          |  Ledgius Writer  |            |  Legacy Writer   |
          |  (GORM)          |            |  (stored proc    |
          |                  |            |   via raw SQL)    |
          +------------------+            +------------------+
```

### Gateway Contract

```go
// pkg/dualpost/gateway.go

type DualPostGateway struct {
    ledgiusDB *gorm.DB
    legacyDB  *sql.DB  // raw SQL for stored procedure calls
    mode      PostMode
}

type PostMode int

const (
    ModeLedgiusOnly   PostMode = iota // Ledgius DB only (production, post-migration)
    ModeLegacyOnly                    // Legacy DB only (fallback)
    ModeDualPost                      // Both DBs (migration period)
    ModeDualPostVerify                // Both DBs + automatic parity assertion
)
```

### Operation Flow (ModeDualPost)

1. Begin Ledgius transaction
2. Execute Ledgius write (GORM)
3. Begin legacy transaction
4. Translate operation to legacy stored procedure call
5. Execute legacy write
6. If both succeed: commit both
7. If either fails: rollback both, return error with context on which system failed

### Legacy Translators

Each domain provides a translator that converts the Ledgius operation into the equivalent LedgerSMB stored procedure call:

```go
// internal/journal/legacy_translator.go

type LegacyTranslator interface {
    // Translate converts a Ledgius domain operation into the equivalent
    // legacy stored procedure call(s).
    Translate(ctx context.Context, op Operation) ([]LegacyCall, error)
}

type LegacyCall struct {
    Procedure string // e.g. "journal__add"
    Args      []any
}
```

### Mode Configuration

- **Development**: `ModeDualPostVerify` — both systems + automatic parity checks after each operation
- **CI**: `ModeDualPostVerify` — fail the build on any divergence
- **Staging**: `ModeDualPost` — both systems, parity checked via scheduled jobs
- **Production (during migration)**: `ModeDualPost` — both systems, monitoring for divergence
- **Production (post-migration)**: `ModeLedgiusOnly` — legacy decommissioned

## Pillar 3: Transaction Atomicity

All logically joined operations execute as a single database transaction. This is a first principle of the new system, not an optimisation.

### Rules

1. **One business operation = one transaction.** Posting an invoice creates GL entries, updates AR, and adjusts balances — all in one `BEGIN ... COMMIT`.
2. **No partial state.** If any part of a compound operation fails, the entire operation rolls back. There is no "invoice created but GL entries missing" state.
3. **Transaction boundary lives in the service layer.** Domain services own the transaction lifecycle, not handlers and not repositories.
4. **Repositories receive the transaction, they don't create it.**

### Service Layer Example

```go
// internal/journal/service.go

func (s *Service) PostJournalEntry(ctx context.Context, cmd PostJournalCmd) (*JournalEntry, error) {
    var result *JournalEntry

    err := s.db.WithinTransaction(ctx, func(tx *gorm.DB) error {
        // All of these use the same tx
        entry, err := s.repo.CreateEntry(ctx, tx, cmd.Header)
        if err != nil {
            return fmt.Errorf("create entry: %w", err)
        }

        if err := s.repo.CreateLines(ctx, tx, entry.ID, cmd.Lines); err != nil {
            return fmt.Errorf("create lines: %w", err)
        }

        if err := s.repo.UpdateBalances(ctx, tx, cmd.Lines); err != nil {
            return fmt.Errorf("update balances: %w", err)
        }

        if err := s.audit.Record(ctx, tx, AuditEvent{...}); err != nil {
            return fmt.Errorf("record audit: %w", err)
        }

        result = entry
        return nil
    })

    return result, err
}
```

// pkg/db/transaction.go

type DB struct {
    ledgius *gorm.DB
    legacy  *sql.DB
}

// WithinTransaction executes fn inside a single Ledgius DB transaction.
func (db *DB) WithinTransaction(ctx context.Context, fn func(tx *gorm.DB) error) error {
    tx := db.ledgius.WithContext(ctx).Begin()
    if tx.Error != nil {
        return fmt.Errorf("begin transaction: %w", tx.Error)
    }

    if err := fn(tx); err != nil {
        tx.Rollback()
        return err
    }

    return tx.Commit().Error
}

### Dual-Transaction Atomicity

During the migration period when dual-posting is active:
- Both databases participate in the operation
- If the Ledgius write succeeds but the legacy write fails: rollback Ledgius, log the divergence, return error
- If the legacy write succeeds but the Ledgius write fails: rollback legacy, log the divergence, return error
- True distributed 2PC is not used (too complex for this use case). Instead: Ledgius commits first (source of truth), legacy failure is logged and retried or flagged for manual reconciliation

### What This Prevents

| Scenario | Without atomicity | With atomicity |
|----------|------------------|----------------|
| Invoice posted, GL entries fail | Orphaned invoice, broken books | Entire operation rolled back |
| Payment allocated, balance update fails | Incorrect outstanding balance | Entire operation rolled back |
| Bank import succeeds, matching fails | Unmatched transactions in limbo | Import rolled back, retry cleanly |
| Dual-post: Ledgius succeeds, legacy fails | Silent divergence | Logged, alerted, recoverable |

## Integration with Existing Principles

This framework implements several principles from A-0009:

- **#1 Ledger as Source of Truth** — Parity assertions prove the new ledger matches the legacy source of truth
- **#4 Atomic Transactions** — Transaction atomicity is a first principle, not optional
- **#5 Idempotency** — Dual-post gateway supports replay of failed operations
- **#7 Audit Trail** — All dual-post operations are logged with both system outcomes
- **#13 Reconciliation** — Parity checks are continuous reconciliation between systems
- **#15 Concurrency Control** — Transaction boundaries prevent interleaving
- **#19 Domain Invariants** — Parity assertions are executable invariant checks

## Key Principle

**Migration safety is not a testing concern — it is an architectural concern.** The dual-post gateway and parity assertions are production infrastructure, not just test utilities.

# Affected Components

- `api/pkg/parity/` — parity assertion test helpers
- `api/pkg/dualpost/` — dual-post gateway
- `api/pkg/db/` — transaction helpers (single and dual)
- `api/internal/*/legacy_translator.go` — per-domain legacy translators
- All domain service layers (transaction boundary owners)
- CI pipeline (parity tests gate the build)

# Data Flow

```
Client Request --> Domain Service --> Dual-Post Gateway
  --> Ledgius Writer (GORM, within tx) --> Ledgius DB
  --> Legacy Writer (stored proc, within tx) --> Legacy DB
  --> Parity Check (in verify mode) --> Pass/Fail
  --> Commit both or Rollback both
```

# API / Interface Changes

No external API changes. The dual-post gateway is internal infrastructure. API consumers are unaware of whether single-post or dual-post mode is active.

# Storage / Schema Changes

No new schema changes for the framework itself. The framework operates on existing tables in both databases. Parity assertions compare table state across both databases using configurable column mappings.

# Background Processing

- Scheduled parity reconciliation jobs in staging/production (compare aggregate financial state between databases)
- Divergence alerting when dual-post mode detects mismatches

# Risks / Trade-offs

- **Dual-database write latency** — Writing to two databases doubles write time during migration; acceptable given the safety benefit and temporary nature.
- **Legacy translator maintenance** — Each domain needs a translator to convert Go operations to stored procedure calls; maintenance burden decreases as domains are fully extracted.
- **No true 2PC** — Distributed two-phase commit is not used; instead Ledgius commits first with legacy failure logged for retry/reconciliation. Risk of brief inconsistency window.
- **CI build time** — Parity tests add time to every build; justified as they are the primary safety mechanism.

# Alternatives Considered

- **Big-bang migration without parallel verification**: Switch from legacy to new system in one step. Rejected — unacceptable risk given decades of encoded edge cases in stored procedures.
- **Read-only shadow comparison**: Run new system as read-only shadow and compare outputs. Rejected — does not verify write correctness.
- **True distributed 2PC**: Use XA transactions across both databases. Rejected — too complex for this use case; Ledgius-first commit with retry is sufficient.

# Test Strategy

- Parity assertion unit tests for every extracted stored procedure
- Dual-post integration tests verifying both databases receive identical data
- Trial balance reconciliation tests at aggregate level
- Divergence detection tests verifying alerting fires on mismatches
- Mode switching tests (LedgiusOnly, LegacyOnly, DualPost, DualPostVerify)

# Rollout / Migration Notes

The framework operates in phases:
1. **Phase 1 (Mirror)**: Both databases structurally identical, `ModeDualPostVerify` in dev/CI
2. **Phase 2 (Diverge)**: Ledgius schema evolves, legacy translators handle mapping, `ModeDualPost` in staging/production
3. **Phase 3 (Consolidate)**: Legacy DB becomes read-only reference, shift to regression suite, eventually decommission to `ModeLedgiusOnly`
