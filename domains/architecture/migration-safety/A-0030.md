# Terms

# Inventory format

The canonical inventory lives at
`ledgius-specs/domains/architecture/migration-safety/lsmb_inventory.md`.
It is a single Markdown table per artifact kind (one table for triggers,
one for functions, etc.) so that diffs in PRs are reviewable line-by-line.

```md
## Triggers

| name | schema.table | purpose | status | risk_tier | replaced_by | dropped_in_migration |
|---|---|---|---|---|---|---|
| ap_audit_trail | public.ap | LSMB-era audit trail — writes to gl_audit_trail on every INSERT/UPDATE/DELETE on ap | in-use | high | (none yet — see T-####) | — |
| ap_track_deleted_transaction | public.ap | Maintains track_global_sequence for deleted ap rows | in-use | medium | (none yet) | — |
| audit_log_immutable | public.audit_log | Ledgius-shape: enforces append-only on audit_log per R-0040 | retained | low | n/a (purposefully retained) | — |

## Functions

| name | purpose | status | risk_tier | replaced_by | dropped_in_migration |
|---|---|---|---|---|---|
| gl_audit_trail_append() | Trigger function called by ap_audit_trail / ar_audit_trail / gl_audit_trail. Appends to public.audittrail. | in-use | high | (none yet) | — |
| track_global_sequence() | Trigger function — writes deletion records to transactions_history | in-use | medium | (none yet) | — |
```

`status` values are the canonical four (`unmapped` / `in-use` / `replaced`
/ `dropped`) plus one Ledgius-only value: **`retained`** for our own
deliberate triggers (e.g. `audit_log_immutable`) — these are exempt from
the migration-status completeness percentage calculation.

# Audit query

`ledgius-db/scripts/lsmb_inventory_audit.sql` runs three SQL queries
against the live DB and prints three sections.

## Section 1 — actual artifacts present

```sql
-- Section 1: actual triggers + functions in the live DB
SELECT 'trigger' AS kind, tgname AS name, c.relname AS table_name
FROM pg_trigger t
JOIN pg_class c ON c.oid = t.tgrelid
WHERE NOT t.tgisinternal
  AND c.relnamespace = (SELECT oid FROM pg_namespace WHERE nspname = 'public')
ORDER BY c.relname, t.tgname;

SELECT 'function' AS kind, p.proname AS name, NULL AS table_name
FROM pg_proc p
JOIN pg_namespace n ON n.oid = p.pronamespace
WHERE n.nspname = 'public'
  AND p.prokind = 'f'
ORDER BY p.proname;
```

## Section 2 — drift report

The script imports the inventory (a small Go helper or a `psql \copy`
loaded into a temp table) and computes:

- artifacts present in DB but not in inventory → **drift, needs investigation**
- artifacts in inventory marked `dropped` but still present → **broken drop migration**
- artifacts in inventory marked `in-use` and present → **work outstanding**

Implementation: one `WITH inventory AS (...) SELECT ...` query whose
`inventory` CTE is materialised from the parsed `lsmb_inventory.md`.

## Section 3 — completeness summary

```
Artifacts inventoried:        N
  unmapped:                   N
  in-use:                     N
  replaced (not yet dropped): N
  dropped:                    N
  retained (Ledgius own):     N

Completeness:    (replaced + dropped) / (in-use + replaced + dropped)
                 [excludes 'retained' and 'unmapped']
```

# CI guardrail

A small Go program at `ledgius-db/cmd/lsmb-lint/main.go` runs in CI:

```go
// Pseudo-code
for _, sqlFile := range newOrModifiedMigrations {
    body := readFile(sqlFile)
    if !startsWithExemptComment(body) {
        if regexp.MustCompile(`(?im)^\s*CREATE\s+(TRIGGER|FUNCTION|OR\s+REPLACE\s+FUNCTION|PROCEDURE)`).MatchString(body) {
            log.Fatalf("%s: introduces server-side logic without `-- lsmb-exempt: <reason>` marker on first line. See R-0054 LMT-008.", sqlFile)
        }
    } else {
        // Exempt — verify the inventory has a corresponding `retained` entry
        if !inventoryHasRetainedFor(sqlFile) {
            log.Fatalf("%s: marked lsmb-exempt but no `retained` inventory entry. See R-0054 LMT-009.", sqlFile)
        }
    }
}
```

Wired into the `ledgius-db` CI workflow as a required check on every
PR. Runs fast (file scan only — no DB connection needed).

# Reporting endpoint

`internal/admin/migration/handler.go` (new package):

```go
// Spec references: R-0054, A-0030.

package migration

type StatusReport struct {
    Total           int            `json:"total"`
    ByStatus        map[string]int `json:"by_status"`
    ByRiskTier      map[string]int `json:"by_risk_tier"`
    CompletenessPct int            `json:"completeness_pct"`
    InUseHighRisk   []Artifact     `json:"in_use_high_risk"`  // drill-down for the admin UI
}

func (h *Handler) GetStatus(c *gin.Context) {
    // 1. Parse lsmb_inventory.md (cached for 5min)
    // 2. Tally per status + risk_tier
    // 3. Compute completeness percentage:
    //    (replaced + dropped) / (in-use + replaced + dropped)
    // 4. Return JSON
}
```

Route: `GET /api/v1/admin/migration-status`. Gated by
`middleware.RequireRole("platform_owner")` (R-0052).

The inventory file is shipped into the API container at build time (it's
in the `ledgius-specs` repo, copied into the API image via the
`Dockerfile`'s build context). Cached parse on a 5-minute TTL.

# Replacement pattern guidance

For each artifact moving from `in-use` → `replaced`:

1. Identify the trigger/function's behaviour (read its source in
   `ledgius/sql/` or extract it via `pg_get_functiondef`)
2. Build the equivalent in Go inside the appropriate domain package
   (e.g. AP audit triggers → write to `audit_log` per R-0040 from
   `internal/payable/service.go` on every state-changing call)
3. Apply A-0015's parallel-verification framework to confirm the Go
   path produces equivalent output to the trigger
4. Update inventory: status `in-use` → `replaced`, populate `replaced_by`
5. (Time delay — let parallel-verification accumulate evidence in
   production)
6. Ship a drop migration that removes the artifact and updates inventory
   to `dropped`

This pattern is the per-artifact unit of work tracked by T-0028 and
spawned tasks.

# Migration sequencing

Drop legacy artifacts in **risk-ascending** order so the highest-risk
ones (audit trails on financial tables) are dropped last, after the
longest period of parallel verification:

1. **Phase 1 (low risk)**: cosmetic / unused functions that no Go code
   calls and no trigger references
2. **Phase 2 (medium risk)**: read-side helper functions used by reports
   (verify the report Go-side replacement first)
3. **Phase 3 (high risk)**: audit-trail triggers on `ap`, `ar`, `gl`,
   `acc_trans` — only after Go-side audit pattern (R-0040 +
   `audit.WriteFromCtx` per R-0052) is verified covering every write
   path

# Risks / Trade-offs

| Risk | Mitigation |
|---|---|
| Inventory drifts from reality | Audit query in CI catches drift; PR review process catches inventory edits without corresponding code |
| Engineer adds a trigger by accident | CI guardrail (LMT-008) blocks the PR |
| Drop migration fires before Go replacement is solid | `replaced` → `dropped` requires explicit PR with verification reference per LMT-012; mandatory inventory update gates the drop |
| `retained` triggers (Ledgius own) accidentally count against completeness | Excluded from the completeness numerator + denominator by design |
| Inventory file becomes unwieldy as it grows | Tables are partitioned by kind (triggers / functions / views / procedures); within each, sorted by status for diffability |
