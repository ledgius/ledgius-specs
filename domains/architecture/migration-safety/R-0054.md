# Summary

Make Ledgius's migration away from inherited LedgerSMB (LSMB) database
artifacts (triggers, functions, views, stored procedures) **visible,
controlled, and measurable**. Establish an inventory, a status-tracking
discipline, an audit query, and a CI guardrail so that the platform can
prove "all business logic lives in Go services" rather than rely on it as
an aspiration.

# Terms

| Term | Definition |
|---|---|
| **LSMB legacy artifact** | Any database object — trigger, PL/pgSQL function, view, stored procedure, sequence-coupled rule — that was inherited from the original LedgerSMB Perl codebase and has not been deliberately recreated under Ledgius's Go-first architecture. |
| **Inventory** | The authoritative enumerated list of every LSMB legacy artifact currently present in any tenant or platform database, with metadata: name, table/schema, purpose, replaced-by Go service (if any), status, risk tier. |
| **Status** | The migration state of an artifact: `unmapped` (not yet inventoried), `in-use` (still relied upon by application logic), `replaced` (a Go service performs the equivalent function but the artifact has not yet been dropped), `dropped` (removed from the live DB by a versioned migration). |
| **Risk tier** | A grading of the artifact's blast radius if removed prematurely: `low` (cosmetic / unused), `medium` (used by reports or read paths), `high` (writes to financial tables, fires on every transaction). |
| **Drop migration** | A versioned SQL migration in `ledgius-db/migrations/tenant/` (or `platform/`) that removes one or more legacy artifacts and updates the inventory's status to `dropped`. |
| **CI guardrail** | An automated check in CI that fails the build if a new migration introduces a `CREATE TRIGGER`, `CREATE FUNCTION`, `CREATE OR REPLACE FUNCTION`, or `CREATE PROCEDURE` statement, unless the migration carries an explicit exemption marker. |
| **Audit query** | A read-only SQL script that, run against any live database, reports every trigger / function / view / procedure currently present, joined against the inventory to flag artifacts not in the inventory or marked `dropped` but still present. |

# Problem / Business Context

The original Ledgius database was bootstrapped from the LedgerSMB Perl
codebase (`ledgius/sql/`). That import brought in a substantial body of
PL/pgSQL: audit-trail triggers, sequence-tracking triggers, balance-helper
functions, account-management procedures, and so on. None of these are
documented in the Ledgius `ledgius-db/migrations/` tree because they were
loaded at bootstrap by the legacy `Dockerfile.lsmb` image, not by the new
migration framework.

A-0015 (Migration Safety) describes the **parallel-verification framework**
used to validate individual migrations of behaviour from PL/pgSQL into Go
services — an excellent technique for the *act* of migrating one
procedure. But A-0015 does not address:

- Knowing **what is left to migrate** (no inventory exists)
- Tracking **per-artifact migration status** (no status field, no spec)
- Preventing **regression** (nothing stops a future migration adding new
  triggers or functions)
- Verifying **completeness** (no audit query confirms a tenant DB is
  free of inherited artifacts)

The result is the platform is in a "we know roughly what we want, we
don't measure how far we are" state. The four LSMB triggers we observed
in the live `ap` table (`ap_audit_trail`, `ap_track_deleted_transaction`)
are typical: present, undocumented, silently firing on every write, with
no record of whether their behaviour is replaced by Go code.

This requirement establishes the discipline to fix that.

# Scope

- The **inventory** of all LSMB legacy artifacts present in a freshly
  bootstrapped Ledgius tenant database (and platform database)
- The **per-artifact status tracking** model
- The **audit query** that reports actual-vs-inventory deltas
- The **CI guardrail** preventing new triggers/functions/procedures from
  entering the codebase without explicit exemption
- The **drop-migration discipline**: every legacy artifact removal is a
  versioned migration that updates the inventory
- **Reporting** to the platform owner population (R-0052) on overall
  migration completeness percentage

# Out of Scope

- The actual replacement of any specific trigger or function — that work
  is per-domain and tracked outside this requirement (e.g. a future task
  spec replaces the AP audit-trail triggers with the Ledgius `audit_log`
  pattern from R-0040)
- The parallel-verification framework itself — owned by A-0015
- Migrating the legacy Perl application code (separate concern)
- Full removal of the `ledgius` legacy repository

# Actors / Users

- **Platform owner** (per R-0052) — reads the migration completeness
  report, prioritises further migration work
- **Engineer adding a new Ledgius migration** — must respect the CI
  guardrail, must update the inventory when a drop migration ships
- **CI / deployment pipeline** — runs the audit query against any DB
  before promotion to production, blocks deploys with unexpected
  artifacts present
- **System auditor / accountant** — relies on the inventory's risk-tier
  classification to assess whether financial-impact triggers are still
  active

# Preconditions

- A-0015 (Migration Safety parallel-verification framework) is in place
- R-0052 `platform_owner` population exists for the report endpoint
- `ledgius-db` migration framework can run versioned and repeatable
  migrations against tenant + platform databases

# Functional Requirements

## Inventory

- **LMT-001**: System SHALL maintain an authoritative inventory of every LSMB legacy artifact present in a freshly bootstrapped Ledgius tenant DB and platform DB. The inventory lives in `ledgius-specs/domains/architecture/migration-safety/lsmb_inventory.md` (or equivalent canonical file) and is the single source of truth.
- **LMT-002**: Each inventory entry SHALL record at minimum: `name`, `kind` (trigger / function / view / procedure / sequence-coupled-rule), `schema.table` (where applicable), `purpose` (one-line plain English), `replaced_by` (Go package + function path, or NULL), `status` (`unmapped` / `in-use` / `replaced` / `dropped`), `risk_tier` (`low` / `medium` / `high`), `dropped_in_migration` (file name, NULL until dropped).
- **LMT-003**: Adding a new entry to the inventory SHALL only happen when (a) a freshly bootstrapped DB is profiled and a previously-unknown artifact is found, or (b) a Go-based replacement is in place and a new entry is added pre-emptively. Random discovery is not the intended path.

## Status tracking

- **LMT-004**: An artifact's status SHALL only progress in the order `unmapped → in-use → replaced → dropped`. Reverse transitions are forbidden — once an artifact is `dropped`, it must not reappear.
- **LMT-005**: Transitions to `replaced` SHALL reference the spec ID of the work that built the Go replacement (e.g. R-0054, T-0028) and the test coverage demonstrating equivalence. Transitions to `dropped` SHALL reference the migration file that removed the artifact.

## Audit query

- **LMT-006**: System SHALL provide an audit query script (`ledgius-db/scripts/lsmb_inventory_audit.sql`) that, run against any live tenant or platform DB, reports:
    - artifacts present in the DB but not in the inventory (drift — needs investigation)
    - artifacts in the inventory marked `dropped` but still present in the DB (broken drop migration — must fix)
    - artifacts in the inventory marked `in-use` (work still to do — informational)
    - count totals per status
- **LMT-007**: The audit query SHALL be runnable in non-mutating read-only mode and SHALL be safe to run in production.

## CI guardrail

- **LMT-008**: CI SHALL fail any new migration in `ledgius-db/migrations/` (or `ledgius-api/migrations/`) that contains `CREATE TRIGGER`, `CREATE FUNCTION`, `CREATE OR REPLACE FUNCTION`, or `CREATE PROCEDURE` statements, unless the migration file's first comment line carries the marker `-- lsmb-exempt: <reason>`.
- **LMT-009**: A small allow-list of *Ledgius-shape* triggers/functions remain permitted with the exemption marker — specifically: triggers that enforce append-only on `audit_log` (per R-0040, see migration `007_recurring_templates_audit_currency.sql`). Each exempt entry must be documented in the inventory with `kind=ledgius-trigger` and `replaced_by=NULL` (purposefully retained).
- **LMT-010**: The CI check SHALL also flag `CREATE OR REPLACE` repeatable migrations (`R__*.sql` Flyway-style) if they introduce server-side logic.

## Drop migrations

- **LMT-011**: Every artifact removal SHALL be a versioned SQL migration in `ledgius-db/migrations/tenant/` (or `platform/`) using `DROP TRIGGER IF EXISTS` / `DROP FUNCTION IF EXISTS` patterns. Migrations SHALL be re-entrant per existing CLAUDE.md migration conventions.
- **LMT-012**: A drop migration SHALL NOT ship until the inventory entry's `replaced_by` field references a verified Go service. Drop migrations and inventory updates SHALL ship in the same PR.

## Reporting

- **LMT-013**: A new endpoint `GET /api/v1/admin/migration-status` SHALL return a JSON summary: `{total: N, by_status: {unmapped: N, in_use: N, replaced: N, dropped: N}, by_risk_tier: {...}, completeness_pct: N}`. Gated on `active_role=platform_owner` (R-0052).
- **LMT-014**: Endpoint SHALL be backed by parsing the inventory file at request time (no DB table required).

# Non-Functional Requirements

- **LMT-NFR-001**: The audit query SHALL complete within 5 seconds on any production-scale tenant DB.
- **LMT-NFR-002**: The CI guardrail SHALL add no more than 30 seconds to the build pipeline.
- **LMT-NFR-003**: The inventory file SHALL remain human-readable Markdown — no binary or schema-coupled storage — so that change history lives in git diff.

# Data / Entities / Fields

The inventory is a Markdown table, not a database table. See A-0030 for
the canonical row format and the audit query design.

# UX / UI Behaviour

- **Platform-owner admin page** (extends pricing-plans admin from R-0053)
  gains a "LSMB Migration Status" tile showing completeness %, broken
  out by risk tier, with a drill-down list of `in-use` artifacts.

# Validation Rules

- A drop migration whose target artifact is still `in-use` in the
  inventory SHALL fail CI with a clear error pointing the engineer at
  this requirement.
- A new inventory entry with `status=dropped` but `dropped_in_migration=NULL`
  SHALL fail the inventory parser used by the reporting endpoint.

# Error Handling

- Audit query failures (e.g. permissions issue on `pg_proc`) SHALL be
  reported but SHALL NOT block deployment — a degraded report is better
  than no report
- CI guardrail false positives (e.g. a Ledgius-shape function that's
  legitimately needed) SHALL be unblockable via the explicit exemption
  marker, with mandatory inventory documentation

# Security / Permissions

- Audit query: read-only, runnable by any DB user with `SELECT` on
  `pg_catalog.*` (the standard postgres role)
- Reporting endpoint: gated on `active_role=platform_owner` (R-0052)
- Inventory file: stored in `ledgius-specs`, normal PR review process
  applies

# Acceptance Criteria

- [ ] `lsmb_inventory.md` exists with at minimum the four currently-known
      `ap`-table triggers documented (`ap_audit_trail`,
      `ap_track_deleted_transaction`, plus their `gl_audit_trail_append`
      and `track_global_sequence` function dependencies)
- [ ] Audit query script runs against the live tenant DB and reports a
      delta count of zero for "in DB but not in inventory" — i.e. the
      inventory accounts for every observed artifact
- [ ] CI lint job fails a deliberate test migration containing
      `CREATE FUNCTION foo()`
- [ ] CI lint job passes a test migration containing
      `-- lsmb-exempt: enforces append-only on audit_log` followed by a
      `CREATE TRIGGER ... ON audit_log`
- [ ] `GET /api/v1/admin/migration-status` returns valid JSON with
      non-zero `total` and reasonable `completeness_pct`
- [ ] Endpoint returns 403 for non-platform-owner roles

# Open Questions

1. **Inventory location**: Markdown file in `ledgius-specs` (proposed) or
   YAML in `ledgius-db/inventory/` (alternative)? Markdown is more
   spec-aligned; YAML is more machine-parseable. Recommendation:
   Markdown for the canonical doc, with a generated YAML cache for the
   reporting endpoint to parse.
2. **Allow-list scope for Ledgius-shape triggers**: Right now we have one
   (`audit_log_immutable`). As the four-tier identity work (R-0052) and
   future work add more (e.g. append-only triggers on `bank_feed_event_log`),
   keep them explicitly enumerated rather than blanket-permitting any
   trigger on tables matching `*_log`.
3. **Inventory of Perl-application views and ledger-side functions**:
   the legacy `ledgius/sql/` directory contains hundreds of stored
   procedures (the LedgerSMB API surface). Most are not used by the
   Ledgius Go API. Should we inventory them at all, or only inventory
   what's actually present in the live DB? Recommendation: inventory
   what's *present*, not what's *defined* in source — keeps the scope
   tractable.
