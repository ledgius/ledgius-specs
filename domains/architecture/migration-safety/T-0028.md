# Summary

Phased plan to deliver the LSMB migration tracking infrastructure
(R-0054 / A-0030) and lay the groundwork for systematic per-artifact
removal of LedgerSMB legacy database objects. Per-artifact replacement
work is *not* in this plan — that work spawns task-by-task as each
domain's Go services mature.

# Terms

Vocabulary used in this task plan (LSMB legacy artifact, inventory,
status, risk tier, drop migration, CI guardrail, audit query) is defined
canonically in **R-0054 §Terms**.

# Branch

`feat/lsmb-migration-tracking` on `ledgius-specs`, `ledgius-db`,
`ledgius-api`.

Every source file created or modified under this task carries the
`// Spec references: R-0054, A-0030, T-0028.` provenance header per
the project Provenance rule.

# Phase 1 — Inventory bootstrap

## T-0028.1: Profile the live tenant DB
- Connect to a representative tenant DB and run the Section-1 audit
  queries from A-0030 to enumerate every trigger and function present
- Capture output to `lsmb_observed.txt` (gitignored, used as input only)
- **Verification**: count of triggers and functions, manually compared
  against the live `\dft+` and `\df` outputs

## T-0028.2: Author `lsmb_inventory.md`
- Located at `ledgius-specs/domains/architecture/migration-safety/lsmb_inventory.md`
- Schema per A-0030 §"Inventory format"
- Initial entries: every artifact observed in T-0028.1, all marked
  `status=in-use` (replaced/dropped happens later)
- Risk tier assigned per artifact based on what tables it fires on:
  - `low` — sequence trackers, cosmetic
  - `medium` — read-side helpers, report support
  - `high` — financial-table triggers (ap, ar, gl, acc_trans audit)
- Ledgius-own triggers (e.g. `audit_log_immutable`) marked `status=retained`
- **Verification**: every artifact in `lsmb_observed.txt` has a row in
  `lsmb_inventory.md`; risk-tier assignments PR-reviewed

# Phase 2 — Audit query

## T-0028.3: `ledgius-db/scripts/lsmb_inventory_audit.sql`
- Implements all three sections per A-0030 §"Audit query"
- Inventory CTE constructed from `lsmb_inventory.md` parsed by a small
  helper (`ledgius-db/cmd/inventory-parser/main.go`) that emits a SQL
  values list
- **Verification**: run against the live tenant DB; Section-2 drift
  count is zero (inventory accounts for everything observed)

# Phase 3 — CI guardrail

## T-0028.4: `ledgius-db/cmd/lsmb-lint/main.go`
- Pure file-scan Go program per A-0030 §"CI guardrail"
- Walks `migrations/tenant/`, `migrations/platform/`, and equivalent
  paths in `ledgius-api`
- For each `.sql` file modified in the current PR (use `git diff
  --name-only origin/master...HEAD`), checks for prohibited statements
- Exempts files whose first non-blank line matches `-- lsmb-exempt: <reason>`
- For exempted files, asserts the inventory has a `retained` entry
  citing the file path
- **Verification**: passes a deliberate `CREATE TRIGGER` test fixture
  with the exempt marker; fails the same fixture without the marker

## T-0028.5: Wire `lsmb-lint` into CI
- New job in `ledgius-db/.github/workflows/ci.yml`: `lsmb-migration-lint`
- Runs `go run ./cmd/lsmb-lint`
- Required check on PR merges to master
- **Verification**: opening a deliberate test PR adding
  `CREATE FUNCTION foo()` is blocked by CI

# Out of scope for T-0028

- Per-artifact Go-side replacement of any specific trigger or function
  — each is its own work item, spawned as warranted, tracked by its
  own task spec
- Drop migrations themselves — see "Per-artifact pattern" below; T-0028
  ships only the framework
- Auto-generation of the inventory file from `pg_catalog` (future
  enhancement per A-0030)
- Slack/PagerDuty notifications on completeness milestones (future)

# Per-artifact pattern (for reference, not in this plan)

Once T-0028 is shipped, each LSMB artifact removal follows this loop:

1. Pick an artifact from inventory (`status=in-use`, low-risk first)
2. Spawn a small task spec: T-#### "Replace `<artifact>` with Go service"
3. Implement the Go-side replacement in the appropriate domain package
4. Apply A-0015 parallel verification to confirm equivalence
5. Update inventory: `in-use` → `replaced`, populate `replaced_by`
6. Soak in production for an agreed window (e.g. one BAS quarter)
7. Ship a drop migration removing the artifact + inventory update to
   `dropped`
8. Audit query confirms artifact gone

The completeness percentage rises monotonically as drops land.
