# Summary

Defines mandatory representation, precision, scale, and rounding rules for all
monetary values across Ledgius — PostgreSQL persistence, Go calculation, JSON
transport, and TypeScript/React consumption. Cross-cutting standard binding on
payroll, AR/AP, tax (GST/PAYG/BAS), assets, banking, and any future financial
domain. Closes the gap that allowed the QA-Golden fixtures to drift on
`Hours × Rate ≠ Amount` rounding by making decimal-precision contracts
explicit at every layer.

# Technical Context

Ledgius is a Go API + React/TypeScript frontend on PostgreSQL, migrating from
LedgerSMB (Perl + PL/pgSQL). Financial domains span payroll, AR, AP, tax,
assets, liabilities, banking, and BAS. Calculations cross language boundaries
(SQL → Go → JSON → TypeScript) and must remain bit-exact end to end so that:

1. Books balance to the cent.
2. Statutory artefacts (BAS, STP Phase 2, payment summaries) reproduce the
   same value as the underlying ledger.
3. Replays of historical pay runs produce identical results, even after
   calculation rules change.
4. Frontend previews and backend authoritative calculations never diverge on
   floating-point error.

The QA-07 fixture review surfaced multiple `Hours × Rate ≠ Amount` rounding
discrepancies traceable to ad-hoc precision choices. Without an enforced
standard, drift recurs every time a new domain or developer introduces money
handling.

# Proposed Design

## Core Rule

> **Use exact decimal arithmetic everywhere. Binary floating-point is
> prohibited for any monetary, rate, hours, or percentage value.**

## End-to-End Representation

```
PostgreSQL NUMERIC(p, s)
        ↓
Go decimal.Decimal (shopspring/decimal)
        ↓
JSON string values
        ↓
TypeScript branded decimal strings + decimal.js
```

Final posted/payable values may additionally be stored as integer minor
units (`BIGINT` cents) for deterministic banking, payment file generation,
and cross-system reconciliation.

## Layer-by-Layer Standard

| Layer | Required representation |
|---|---|
| PostgreSQL calculation columns | `NUMERIC(p, s)` |
| PostgreSQL final payment / ledger amount | `BIGINT` cents + `currency_code CHAR(3)` |
| Go calculation values | `decimal.Decimal` |
| Go final cents | `int64` |
| API JSON decimal values | strings, e.g. `"123.4500"` |
| API JSON cents values | strings, e.g. `"12345"` |
| TypeScript calculation values | `Decimal` from `decimal.js` |
| TypeScript API boundary | branded decimal-string types |

### Prohibited

- PostgreSQL `money`, `real`, `double precision`
- Go `float32`, `float64` for any monetary, rate, hours, or percentage value
- JavaScript / TypeScript `number` for payroll, tax, ledger, or
  reconciliation arithmetic
- `decimal.NewFromFloat(...)` in Go (binary error has already happened
  before `Decimal` sees the value)

## Precision and Scale Catalogue

These are **mandatory** for new schema; existing tables migrate at the next
schema-touch opportunity per A-0009 migration safety rules.

| Value class | PostgreSQL type | Rationale |
|---|---|---|
| Gross / tax / deductions / net (intermediate) | `NUMERIC(19,4)` | Sub-cent precision pre-rounding |
| Reimbursements, allowances | `NUMERIC(19,4)` | Same as money calculations |
| Hourly rate, salary rate | `NUMERIC(19,6)` | Pro-rata and award penalty multipliers |
| Hours worked, leave hours | `NUMERIC(9,4)` | Quarter-hours and finer increments |
| Tax rate / percentage / multiplier | `NUMERIC(12,8)` | Stored as decimal fraction (e.g. `0.11500000`) |
| Final payment amount | `BIGINT` cents | Deterministic posted/payment value |
| Currency code | `CHAR(3)` | ISO 4217; never implicit |

## Rounding Principles

```
Use higher precision during calculation.
Round only at legally or contractually defined boundaries.
Store the final rounded value AND the unrounded calculation inputs.
Store enough calculation detail to explain the result later.
```

### Rules

1. **No premature rounding.** Do not round to cents and then to dollars.
   Apply the statutory rule directly to the unrounded calculation result
   (e.g. PAYG NAT 1004 rounds raw weekly tax to the nearest dollar — not
   to cents first).
2. **Rounding mode is explicit.** Default to `ROUND_HALF_UP` for money;
   any other mode (banker's rounding, statutory truncation) MUST be
   recorded in the calculation envelope.
3. **Rounding scale is explicit.** `internalScale = 4`, `moneyScale = 2`
   are the defaults; deviations require sign-off and a calculation
   envelope entry.
4. **Round once.** Each statutory boundary rounds exactly once. Down-stream
   consumers re-use the rounded value rather than re-rounding.

## Calculation Envelope (Auditability)

Every authoritative calculation MUST persist enough detail to reproduce the
result years later, even after rule changes. Minimum payload (JSON in a
`calculation_detail` JSONB column or equivalent):

```json
{
  "calculationVersion": "payroll-au-v2026.04",
  "rounding": { "moneyScale": 2, "internalScale": 4, "mode": "ROUND_HALF_UP" },
  "inputs":   { "hoursWorked": "7.5000", "hourlyRate": "52.375000", "taxRate": "0.32500000" },
  "outputs":  { "grossAmount": "392.8125", "taxAmount": "127.6641",
                "netAmount": "265.15", "netAmountCents": "26515" }
}
```

Mandatory fields: `calculationVersion`, `rounding`, raw `inputs`, final
`outputs` including the integer-cents posted value, currency code,
created-at timestamp, actor (user/system).

## Database Contract — Reference Schema

```sql
CREATE TABLE payroll_line (
    payroll_line_id BIGSERIAL PRIMARY KEY,
    employee_id     BIGINT NOT NULL,
    pay_run_id      BIGINT NOT NULL,
    currency_code   CHAR(3) NOT NULL DEFAULT 'AUD',

    hours_worked      NUMERIC(9,4)  NOT NULL,
    hourly_rate       NUMERIC(19,6) NOT NULL,

    gross_amount      NUMERIC(19,4) NOT NULL,
    tax_amount        NUMERIC(19,4) NOT NULL,
    deduction_amount  NUMERIC(19,4) NOT NULL,
    net_amount        NUMERIC(19,4) NOT NULL,

    net_amount_cents  BIGINT NOT NULL,

    calculation_version TEXT  NOT NULL,
    calculation_detail  JSONB,
    created_at          TIMESTAMPTZ NOT NULL DEFAULT now(),

    CONSTRAINT payroll_line_currency_code_check
        CHECK (currency_code ~ '^[A-Z]{3}$'),
    CONSTRAINT payroll_line_non_negative_check
        CHECK (gross_amount >= 0 AND tax_amount >= 0
               AND deduction_amount >= 0 AND net_amount_cents >= 0),
    CONSTRAINT payroll_line_hours_check
        CHECK (hours_worked >= 0)
);
```

For higher-volume domains (payroll, banking) split calculation rows
(`*_calculation_line`) from posted ledger rows (`*_ledger_entry`) so the
ledger remains append-only and re-calculation does not mutate posted truth.

## Go Standard

- Import `github.com/shopspring/decimal` for every monetary value.
- GORM tag the column type explicitly: `gorm:"type:numeric(19,4);not null"`.
- Parse decimals from strings: `decimal.RequireFromString("52.375000")`.
- Do **not** use `decimal.NewFromFloat`.
- For `pgx`, register `github.com/jackc/pgx-shopspring-decimal` (or
  equivalent) so `numeric` arrives as `decimal.Decimal`, not `float64`.
- Convert to cents only at the posting boundary:
  `amount.Mul(decimal.NewFromInt(100)).IntPart()`.

## API JSON Standard

Decimal values are **strings**:

```json
{
  "currencyCode": "AUD",
  "hoursWorked": "7.5000",
  "hourlyRate": "52.375000",
  "grossAmount": "392.8125",
  "taxAmount": "127.6641",
  "netAmount": "265.15",
  "netAmountCents": "26515"
}
```

JSON numbers are prohibited for monetary values because most clients parse
them as IEEE-754 floats, losing precision before validation can catch it.

## TypeScript / React Standard

- Branded types at the API boundary:
  ```ts
  export type DecimalString = string & { readonly __brand: "DecimalString" };
  export type CentsString   = string & { readonly __brand: "CentsString" };
  ```
- Use `decimal.js` for any in-browser arithmetic (calculator-style
  previews, line totals, GST splits). Configure
  `Decimal.set({ precision: 28, rounding: Decimal.ROUND_HALF_UP })`.
- Runtime-validate decimal strings on ingest (not just at compile time).
- `bigint` may be used for cents in client logic; never serialise raw
  `bigint` to JSON — convert to string.
- Frontend previews are non-authoritative. Backend calculation is the
  source of truth; the UI re-renders backend values rather than trusting
  its own arithmetic for posted amounts.

## House Rules (binding)

1. No floating-point types for monetary, rate, hours, or percentage values.
2. PostgreSQL money columns use `NUMERIC(p, s)`; `money` is prohibited.
3. Final posted amounts are stored as integer cents alongside the
   `NUMERIC` calculation amount.
4. Go uses `decimal.Decimal` for all monetary arithmetic.
5. API decimal values are serialised as strings.
6. React/TypeScript treats API monetary values as branded decimal strings.
7. Frontend previews are allowed; backend calculations are authoritative.
8. Rounding happens only at explicit business/legal boundaries.
9. Every authoritative calculation persists a calculation envelope
   (version, inputs, rounding, outputs).
10. Currency code is always stored explicitly — never implicit.

## Testing Requirements

Per A-0009 + the global accounting-compliance-first rule, every financial
test MUST assert:

- Debits = credits at the journal level.
- `round(hours × rate, 2) == amount` for any line that derives amount
  from the pair.
- `sum(line.net_amount_cents) == header.total_cents` for any aggregate.
- Calculation envelope is present and replayable for sampled rows.
- Currency code is non-null and matches ISO 4217 on every monetary row.

The QA-Golden tenants under `ledgius-db/fixtures/datasets/qa-golden/` are
the canonical regression set and MUST pass these assertions on every
schema or rule change.

## Migration Plan

This standard is binding on **all new** schema, services, and components.
Existing tables/services migrate opportunistically:

1. Schema touch (Flyway versioned script) — bring the column to spec.
2. Go service touch — replace `float64` with `decimal.Decimal` and update
   GORM tags.
3. API touch — switch JSON numeric fields to strings (versioned where the
   API contract is public; in-place where consumers are internal only).
4. Frontend touch — switch to branded decimal strings + `decimal.js`.

Backwards-compat shims are not permitted past one release cycle per the
project's no-permanent-shims rule.

# Out of Scope

- FX conversion, multi-currency revaluation, and historical FX rate
  snapshots — covered separately (future spec).
- Cryptocurrency or non-ISO-4217 currency representation.
- UI display formatting beyond stating that `Intl.NumberFormat` is
  acceptable for display only, never as an intermediate calculation type.

# Open Questions

1. Should `calculation_detail` JSONB be normalised into a dedicated
   `calculation_envelope` table for cross-domain query? Defer until a
   second domain (AR/AP) needs envelope replay.
2. Does the engine need a dedicated `Money` value type wrapping
   `(amount decimal.Decimal, currency string)` to make currency mismatch
   a compile-time error? Lean yes; track separately.
