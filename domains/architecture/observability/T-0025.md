# Summary

Improve error visibility and observability across the platform so that API failures are immediately surfaced to users and developers, not hidden behind empty states or buried in logs.

# Problem

A SQL syntax error (SQLSTATE 42601) in the reconciliation queue endpoint went undetected because:
- The frontend showed "All reconciled" instead of an error message
- The error was only visible by manually grepping Fly.io logs
- No structured error logging with request context existed
- No admin-visible error dashboard existed

# Task Breakdown

## T-0025.1: Frontend — show API errors (DONE)

DataTable and key components now show red error banners when API calls fail, instead of silently showing empty state. Implemented 2026-04-12.

## T-0025.2: API — structured error logging with request context

Enhance the existing `middleware.Logger` to:
- Log all 4xx/5xx responses with: request path, method, status code, user email, tenant ID, duration, error message
- For 500s: log the full error chain including SQL statements that failed
- Use structured slog fields for machine-parseable log output
- Include request ID for correlation

**Files:** `pkg/middleware/middleware.go`

## T-0025.3: API — admin recent-errors endpoint

`GET /api/v1/admin/errors?limit=50` (platform admin only):
- Returns last N error events from an in-memory ring buffer
- Each entry: timestamp, path, method, status, user, error message, request ID
- Ring buffer size: 200 entries (no database needed)
- Clears on API restart (acceptable — not a permanent audit trail)

**Files:** `pkg/middleware/error_log.go`, `pkg/middleware/features_handler.go`

## T-0025.4: Development — fail loudly on first run

When running locally (`SINGLE_TENANT_MODE=true`):
- Any 500 response also prints a prominent `=== API ERROR ===` banner to stderr
- SQL errors include the failing query text
- Panic recovery middleware logs the full stack trace (already exists, verify it's visible)

**Files:** `pkg/middleware/middleware.go`

## T-0025.5: Fly.io — log alerting (future)

Integrate with a log aggregation service or set up Fly.io log drains to catch 500s proactively. Options:
- Fly.io log shipping to Grafana Cloud / Datadog
- Simple webhook on 500 count threshold
- PagerDuty/Slack integration for critical errors

This is a future enhancement — T-0025.2 through T-0025.4 are sufficient for development phase.
