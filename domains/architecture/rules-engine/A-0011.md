# Summary

Defines a three-engine rules architecture (CEL, OPA, GoRules) with hard scope boundaries, versioned immutable bundles, deterministic execution, audit-defensible journaling, and replay capability. Each engine has a narrow, enforceable scope: CEL handles arithmetic formulas, OPA handles policy conformance, and GoRules handles accounting/tax decision options.

# Requirement Link

N/A — foundational architecture document governing how business rules, policy evaluation, and accounting treatment decisions are executed across the system.

# Technical Context

Accounting systems require audit-defensible decision-making where every calculation, policy check, and treatment selection can be reproduced exactly at a later date. This architecture separates concerns across three rule engines to prevent scope creep and ensure each decision is traceable, replayable, and explainable.

# Proposed Design

## 1. Objectives

- Audit defensible with full traceability and reproducibility
- Deterministic: same input plus same versions must yield the same result
- Strict separation of concerns
- Versioned, immutable configuration bundles

## 2. Hard Boundaries (Non-Negotiable)

| Layer | Owns | Must Not Do |
|---|---|---|
| CEL | Arithmetic formulas, validation, simple derived values | Policy decisions, accounting decision selection, external side effects |
| OPA | Policy conformance, allow/deny, obligations, reason reporting | Tax math, accounting calculations, ledger postings |
| GoRules | Accounting and tax decision options, classification, rule-driven recommendations | Authorization, permissions, direct persistence |
| Go Domain | Decimal math enforcement, ledger posting, persistence, transactionality | Ad hoc rule authoring inside code paths |

**Boundary rule:** engines do not call each other directly. Application orchestration calls them in a controlled sequence.

## 3. Canonical Input Model

All engines consume a canonical, versioned decision input shape:

```json
{
  "actor": {},
  "tenant": {},
  "transaction": {},
  "document": {},
  "jurisdiction": {},
  "effective_date": "YYYY-MM-DD",
  "context": {}
}
```

## 4. Versioned Bundles

Each execution references immutable bundles:

| Layer | Bundle Identifier |
|---|---|
| CEL | `cel_bundle_vX.Y.Z` |
| OPA | `opa_bundle_vX.Y.Z` |
| GoRules | `gorules_bundle_vX.Y.Z` |

Each recorded execution must store:
- bundle version
- engine version
- canonical input hash (SHA-256)
- execution timestamp
- deterministic replay key

## 5. CEL Layer

### Intended Use
- GST formula calculations
- depreciation formulas
- validation predicates
- simple threshold checks

### Example YAML
```yaml
version: 1.0.0
formulas:
  gst_standard:
    expression: "amount * 0.10"
    scale: 2
    rounding_mode: "half_up"
  depreciation_diminishing_value:
    expression: "base_value * rate"
    scale: 2
    rounding_mode: "half_up"
```

### CEL Rules
- no wall-clock time access
- no randomness
- no network or database access
- no hidden mutable state
- decimal-safe inputs only
- parse and type-check expressions before activation

## 6. OPA Layer

### Intended Use
- determine whether a business situation conforms to policy
- explain why it conforms
- explain which policy clauses failed when it does not conform

### Standard Output
```json
{
  "allow": false,
  "reasons": ["Purchase exceeds delegated approval threshold"],
  "failed_conditions": ["approval.threshold.max_amount"],
  "obligations": [],
  "policy_refs": ["FIN-APPROVAL-2026-01:clause-4.2"]
}
```

### OPA Rules
- policy only
- no accounting math
- no posting logic
- decisions must include human-readable reasons and machine-readable references
- every evaluation must be journaled

## 7. GoRules Layer

### Intended Use
- accounting treatment selection
- tax treatment options
- recommendation generation where multiple lawful options exist
- traceable user choice capture

### Standard Output
```json
{
  "options": [
    {
      "code": "instant_writeoff",
      "label": "Immediate asset write-off",
      "reasoning": ["Asset value under configured threshold", "Eligible entity class matched"]
    },
    {
      "code": "standard_depreciation",
      "label": "Standard depreciation schedule",
      "reasoning": ["General depreciation pathway available"]
    }
  ],
  "recommended_option": "instant_writeoff"
}
```

### GoRules Rules
- no auth decisions
- no direct ledger write
- must emit option set, recommendation, and explanation
- if user selects a non-default valid option, selection must be journaled with actor and reason

## 8. Determinism Requirements

The platform must enforce:
- decimal math only
- explicit rounding mode
- explicit scale
- effective date passed as input
- no implicit "now"
- no environment-dependent behavior
- no mutable production bundles

## 9. Journaling and Traceability

### Minimum decision log fields
```text
id
timestamp_utc
tenant_id
actor_id
engine_name
engine_version
bundle_version
input_hash_sha256
input_json
output_json
explanation_json
user_selection_json
correlation_id
replay_key
```

### Audit principles
- append-only storage
- immutable event records
- redaction policy for sensitive fields
- linked correlation across CEL, OPA, and GoRules executions
- ability to reconstruct the exact execution path

## 10. Replay Requirements

A replay must use:
- the original canonical input
- the exact engine versions
- the exact immutable bundle versions
- the same decimal and rounding configuration

Expected outcome:
- identical outputs
- identical explanation structure, except for replay metadata

## 11. Effective Dating and Jurisdiction

All formulas, policies, and decision bundles must support:
- `effective_start`
- `effective_end`
- `jurisdiction_code`
- optional tenant override scope

No rule becomes active without explicit effective dating.

## 13. Failure Policy

| Engine | Failure Behavior |
|---|---|
| CEL | Reject the request as non-deterministic or invalid for execution |
| OPA | Deny by default |
| GoRules | Block accounting transaction processing until resolved |

## 14. Summary

This architecture provides:
- hard scope boundaries
- versioned and replayable behavior
- explainable decisions
- audit-defensible journaling
- deterministic accounting rule execution

# Affected Components

- Rules engine orchestration layer (application-level sequencing of CEL, OPA, GoRules)
- CEL formula evaluation module
- OPA policy evaluation module
- GoRules decision engine module
- Decision journal / audit log storage
- All domain services that invoke rule evaluation (tax, posting, approval)

# Data Flow

```
Domain Service --> Rules Orchestrator
  --> CEL Engine (formulas, validation) --> decision log
  --> OPA Engine (policy conformance) --> decision log
  --> GoRules Engine (treatment selection) --> decision log
  --> Return aggregated result to Domain Service
```

All three engines consume the same canonical input model. Application orchestration controls the sequence. Engines do not call each other directly.

# API / Interface Changes

- All rule engine invocations accept the canonical input model shape
- All rule engine outputs follow standardized response contracts (CEL: computed values, OPA: allow/deny with reasons, GoRules: option sets with recommendations)
- Decision log entries exposed via audit API

# Storage / Schema Changes

Decision journal table with minimum fields:
- `id`, `timestamp_utc`, `tenant_id`, `actor_id`
- `engine_name`, `engine_version`, `bundle_version`
- `input_hash_sha256`, `input_json`, `output_json`
- `explanation_json`, `user_selection_json`
- `correlation_id`, `replay_key`

Append-only storage with immutable event records.

# Background Processing

N/A — all rule evaluations are synchronous within the transaction boundary.

# Risks / Trade-offs

- **Three-engine complexity** — More engines means more integration surface; mitigated by hard boundaries and standardized input/output contracts.
- **Bundle versioning overhead** — Every bundle change requires a new immutable version; justified by audit defensibility and replay requirements.
- **Determinism enforcement** — Restricting engines from wall-clock time, randomness, and network access limits flexibility; this is intentional for accounting correctness.

# Alternatives Considered

- **Single rules engine**: Using only one engine for all concerns. Rejected — conflates policy, arithmetic, and decision selection, making audit traceability harder.
- **Inline business rules in Go code**: Embedding rules directly in domain services. Rejected — prevents non-developer rule authoring, versioning, and replay.
- **Database-stored rules**: Storing rules in PostgreSQL. Rejected — opaque to version control and harder to replay deterministically.

# Test Strategy

- Determinism tests: same input + same bundle versions must produce identical output across runs
- Boundary enforcement tests: verify each engine rejects out-of-scope operations
- Replay tests: historical inputs replayed against pinned bundle/engine versions
- Journaling tests: verify all decision log fields are populated correctly
- Effective dating tests: verify rules activate/deactivate based on date ranges

# Rollout / Migration Notes

Rule bundles are deployed as immutable versioned artifacts. New bundle versions are activated via effective dating — no in-place mutation of production bundles. Historical decisions remain replayable against their original bundle versions indefinitely.

# Appendix: v3 Changelog (Changes from v2)

## 1. Hard Boundaries Upgraded
v2 described layers at a high level. v3 converts those into explicit ownership and prohibition rules so each engine has a narrow, enforceable scope.

## 2. Canonical Input Model Added
v3 introduces a single canonical decision input shape across CEL, OPA, and GoRules. This reduces drift, simplifies audits, and supports deterministic replay.

## 3. Determinism Requirements Made Explicit
v3 states clear deterministic controls:
- no floating-point dependency
- no implicit current time
- no randomness
- no mutable bundles

## 4. Versioned Immutable Bundles Strengthened
v3 formalizes immutable bundle identifiers per engine and requires bundle version plus engine version to be recorded for every execution.

## 5. Replayability Defined
v2 referenced reproducibility. v3 defines replay prerequisites and expected replay behavior so historical results can be reproduced defensibly.

## 6. Journaling Schema Strengthened
v3 adds a concrete minimum event record structure with hashes, correlation identifiers, user selections, and replay keys.

## 7. OPA Explainability Standardized
v3 standardizes OPA output into: allow, reasons, failed conditions, obligations, and policy references. This makes policy conformance evidence more defensible.

## 8. GoRules Output Contract Improved
v3 requires GoRules to emit: all valid options, recommended option, structured reasoning, and user choice traceability.

## 9. Effective Dating and Jurisdiction Added
v3 explicitly requires effective start, effective end, and jurisdiction support for all bundles.

## 10. Licensing Controls Tightened
v3 adds operational controls around SBOM, CI scanning, and distribution-aware license review.

## 11. Failure Policies Clarified
v3 defines default failure behavior per engine to avoid ambiguous runtime behavior.

## 12. Audit Focus Elevated
v3 is explicitly designed as an audit-defensible architecture rather than just a functional rules integration design.
