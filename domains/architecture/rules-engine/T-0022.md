# Summary

Task plan for creating a comprehensive test suite that captures every expression, policy, and rule as explicit regression tests before migrating them to CEL/OPA/GoRules in Phase 5. Every test must pass against the current Go service implementation. After Phase 5 migration, the same tests must pass against the CEL/OPA/GoRules engines, serving as the regression gate.

# Requirement and Design References

- Phase 5 rules engine migration (CEL, OPA, GoRules)
- Existing parity tests (20 checks) verify aggregate outcomes but not individual expression/policy/rule assertions
- `api/pkg/rules/bundles/gst_v1.0.0.yaml` -- GST CEL formulas
- `api/pkg/rules/bundles/payroll_v1.0.0.yaml` -- Payroll CEL formulas
- `api/pkg/rules/bundles/accounting_v1.0.0.yaml` -- Accounting CEL formulas
- `api/pkg/rules/` -- OPA policies and go-rules definitions

# Assumptions

- Current Go service implementations are the source of truth for expected behaviour
- All tests must be engine-agnostic: same inputs and expected outputs regardless of whether the implementation is Go code or CEL/OPA/GoRules
- Effective dates are always explicit parameters (no dependency on wall-clock time)
- Decimal precision is preserved across all operations
- Test framework supports table-driven tests (Go testing conventions)

# Preconditions

- All existing parity tests (20 checks) pass
- Current Go services build cleanly with no failures
- Rules engine bundles are deployed and operational

# Recommended Execution Order

1. **Slice 1 (Expression Tests)** -- highest risk area, most formulas to capture. Start with GST and BAS calculations.
2. **Slice 2 (Policy Tests)** -- second priority, captures allow/deny decisions that are hardest to debug after migration.
3. **Slice 4 (Effective Date Boundary Tests)** -- run in parallel with Slice 2, as these are orthogonal.
4. **Slice 3 (Rule Tests)** -- classification rules, depends on understanding from Slices 1-2.
5. **Slice 5 (Determinism Tests)** -- final validation, can reuse test data from all previous slices.

# Testing Tasks

- Create test file structure under `api/internal/rules_migration_test/` or equivalent
- Implement table-driven tests for each slice using Go `testing` package
- Ensure all tests are parameterised with explicit `effectiveDate` input
- Ensure all monetary values use `shopspring/decimal` in test assertions
- Add tolerance-based assertions for rounding (within 0.01)
- Document test count baseline when complete
- Integrate into CI pipeline as a required gate

# Rollout Tasks

- Run full test suite against current Go services -- all must pass (baseline)
- Document total test count and coverage map
- After Phase 5 migration: re-run same tests against CEL/OPA/GoRules engines
- Tests remain in CI permanently -- they gate every commit during and after Phase 5

# Done Criteria

- All expression tests pass against current Go services
- All policy tests pass against current Go services
- All rule tests pass against current Go services
- All effective date boundary tests pass
- All determinism/replay tests pass
- Test count documented and baselined
- Tests integrated into CI pipeline
