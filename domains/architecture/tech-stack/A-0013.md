# Summary

Defines the complete technology stack for Ledgius across backend, frontend, and infrastructure layers. Includes the domain-based package organisation for both Go and React, code organisation rules, and decision records for technology choices worth revisiting.

# Requirement Link

N/A — foundational architecture document defining technology choices for the entire system.

# Technical Context

Ledgius is a modernised fork of LedgerSMB, progressively migrating from Perl/PL/pgSQL to a Go API + React frontend. The tech stack must support dual-database operation during migration, domain-based code organisation, Australian tax compliance, and eventual SaaS deployment.

# Proposed Design

## Backend

| Layer | Technology | Notes |
|-------|-----------|-------|
| Core Ledger | LedgerSMB (Perl + PL/pgSQL) | Legacy system; business logic progressively extracted to Go |
| API Layer | Go (Gin + GORM) | REST API with all business logic in Go domain services |
| Database | PostgreSQL | Two instances: legacy (LedgerSMB schema) and Ledgius (evolved schema) |
| BAS Reporting | Go service | Dedicated service for Australian tax calculations and ATO submission |
| Bank Ingestion | Go service | Pipeline for importing bank feeds (OFX/CSV/API) and auto-matching |

## Frontend

| Concern | Technology | Rationale |
|---------|-----------|-----------|
| Build / Dev | Vite | Fast HMR, native ESM, superior DX over Webpack |
| UI Framework | React + TypeScript | Type safety, ecosystem, hiring pool |
| Routing | React Router | Standard, well-maintained |
| Data Fetching | TanStack Query | Caching, background refetch, optimistic updates |
| Forms | React Hook Form | Performant, minimal re-renders |
| Validation | Zod | Schema-first validation, shared with API contracts |
| Data Grids | AG Grid | Spreadsheet-like: grouping, pivoting, heavy filters, large datasets |
| Component Library | shadcn/ui | Accessible, composable, full styling control |
| Styling | Tailwind CSS | Utility-first, pairs well with shadcn/ui |

## Code Organisation - Domain Packages (Go)

The Go API uses domain/topic-based package organisation, not layered (store/model/service/route). Each domain owns its models, repository, service logic, and HTTP handlers together.

```
api/
  cmd/server/           # Entrypoint, wires domains together
  internal/
    account/          # Chart of accounts domain
      model.go      # GORM models + Zod-equivalent validation
      repository.go # DB queries (GORM)
      service.go    # Business logic
      handler.go    # Gin route handlers
      routes.go     # Route registration
      account_test.go
    journal/          # General ledger / journal entries
    receivable/       # AR: invoices, payments received
    payable/          # AP: bills, payments made
    contact/          # Customers and vendors
    banking/          # Bank feeds, reconciliation, ingestion
    tax/              # GST, BAS reporting, PAYG
    reporting/        # Trial balance, P&L, balance sheet
    tenant/           # Multi-tenancy (Phase 4)
    auth/             # Auth middleware, JWT, roles
  pkg/
    db/               # GORM connection setup, dual-DB manager, tx-on-context helpers
    middleware/        # Shared Gin middleware (CORS, request ID, logging)
    response/         # Consistent API error/success response helpers
    shadow/           # Shadow write + parallel verification engine
```

**Why domain packages:**
- Each domain is self-contained — easy to reason about, test, and modify independently.
- Avoids circular dependencies that plague layered architectures.
- New domains are added without touching existing code.
- `pkg/` is reserved for genuinely shared infrastructure (DB connections, middleware, response helpers).

**Rules:**
- Domains import from `pkg/` but never from each other directly. Cross-domain communication goes through interfaces or the service wire-up in `cmd/server/`.
- Each domain registers its own routes via a `RegisterRoutes(rg *gin.RouterGroup)` function.
- Tests live alongside the code they test.
- Transaction boundaries are owned by the service layer via tx-on-context (see A-0010).
- Repositories check context for an existing tx; if absent, create a local one that is NOT propagated back to context (top-down only).
- All writes go through the dual-post gateway during migration (see `pkg/dualpost/` and `pkg/parity/`).
- Business logic lives in Go services, not in SQL stored procedures (see extraction strategy in A-0010).

## Code Organisation - Domain Modules (React)

The React frontend mirrors the Go backend's domain structure:

```
ui/
  src/
    domains/
      account/      # Chart of accounts
        components/
        hooks/    # TanStack Query hooks, React Hook Form configs
        schemas/  # Zod schemas
        pages/
        api.ts    # API client for this domain
      journal/
      receivable/
      payable/
      contact/
      banking/
      tax/
      reporting/
    shared/           # Shared UI components, layout, utils
    router.tsx
    main.tsx
```

## Decision Records

Decisions worth revisiting later:

- **AG Grid vs MUI X Data Grid**: Starting with AG Grid for its superior pivot/grouping. Revisit if licensing costs become a concern for SaaS (AG Grid Enterprise is per-developer licensed).
- **Auth provider**: Deferred until Phase 4. Keycloak for self-hosted, Clerk/Auth0 for SaaS simplicity.
- **Tailwind**: Adopted alongside shadcn/ui which depends on it. If the team finds utility-first cumbersome, CSS modules are a fallback.

```
React UI (Vite + TanStack Query)
  --> Go API (Gin handlers)
    --> Domain Services (business logic + tx boundary)
      --> Repositories (GORM queries)
        --> PostgreSQL (Ledgius DB + Legacy DB during migration)
```

# API / Interface Changes

- REST API served by Go Gin framework
- Domain-based route registration
- Consistent response helpers via `pkg/response/`
- API contracts validated with Zod schemas shared between frontend and backend

# Storage / Schema Changes

- Two PostgreSQL 16 instances during migration (legacy + Ledgius)
- Schema migrations managed by Flyway or goose for Ledgius DB
- LedgerSMB's built-in migrations for legacy DB

# Risks / Trade-offs

- **AG Grid licensing** — Enterprise license is per-developer; may become costly for SaaS. Fallback: MUI X Data Grid.
- **Dual-database overhead** — Running two PostgreSQL instances adds infrastructure cost during migration. This is temporary.
- **Auth provider deferred** — Authentication is deferred to Phase 4; early development uses simplified auth. Risk of rework if auth requirements are underestimated.

# Alternatives Considered

- **Webpack instead of Vite**: Rejected — slower HMR, heavier configuration.
- **Next.js instead of React + Vite**: Rejected — SSR not needed for an accounting application; adds complexity without benefit.
- **Layered Go architecture (store/model/service/route)**: Rejected — leads to circular dependencies and cross-cutting changes for new features.
- **MUI X Data Grid**: Considered but AG Grid offers superior pivot/grouping for financial data grids. May revisit for licensing reasons.

# Test Strategy

- Go: unit tests per domain, integration tests against PostgreSQL, parity tests against legacy DB
- React: component tests with React Testing Library, E2E tests with Playwright
- CI: parallel test execution via GitHub Actions

# Rollout / Migration Notes

Tech stack is established from project inception. The dual-database architecture supports progressive migration from legacy to new system. See A-0016 (Dual Database Strategy) for database migration phases.
