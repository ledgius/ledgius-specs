# Summary

Defines the strategy for extracting all business logic from LedgerSMB PL/pgSQL stored procedures into Go domain services, using a transaction-on-context pattern with strict top-down ownership. Establishes the extraction order, parity verification requirements, and the boundary between business logic (Go) and data integrity (PostgreSQL).

# Requirement Link

N/A — foundational architecture document governing how business logic is migrated from legacy stored procedures to Go.

# Technical Context

LedgerSMB encodes significant accounting logic inside 500+ PostgreSQL stored procedures. This logic must move to Go for testability, dual-post feasibility, code visibility, and domain ownership. The extraction must be methodical because these stored procedures are battle-tested and encode decades of accounting edge cases.

# Proposed Design

## 1. Why Extract SQL Business Logic

LedgerSMB encodes significant accounting logic inside PostgreSQL stored procedures. This must move to Go because:

- **Testability** — Go service logic is unit-testable without a running database
- **Dual-post feasibility** — business logic in stored procs is locked to one database; Go-side logic can drive both
- **Visibility** — stored procedure logic is opaque to code review, linting, and static analysis
- **Domain ownership** — each domain package should own its business rules alongside its models and handlers
- **Aligns with principle #23** (Minimize Hidden DB Logic) from A-0009

### What stays in the database

- Schema constraints (NOT NULL, UNIQUE, CHECK, foreign keys)
- Indexes
- Simple views for read optimisation
- Triggers only where referential integrity requires them (e.g. cascading soft deletes)

### What moves to Go

- All accounting calculations (posting, balancing, allocation)
- Validation beyond simple constraints
- Workflow logic (state transitions, approval flows)
- Tax calculations (GST, PAYG)
- Business rules (period controls, duplicate detection)

## 2. Transaction-on-Context Pattern

Transactions are propagated via `context.Context` with strict top-down ownership.

### Rules

1. **Service layer creates transactions.** Only service methods (the business logic layer) may begin a transaction and place it on the context.
2. **Repositories look for an existing transaction.** Store/repository methods check the context for a `*gorm.DB` transaction. If one exists, use it. If not, create a local one for that single operation.
3. **Local transactions are not propagated.** When a repository creates its own transaction (because none was on the context), it does NOT place it back onto the context. This prevents callees from accidentally widening a caller's transaction scope.
4. **Top-down only.** Transaction scope flows downward through the call stack, never upward. A child function cannot escalate the caller into a transaction.
5. **Always pass `context.Context`.** Every function in the chain receives and forwards context.

### Implementation

```go
// pkg/db/txcontext.go

type txKey struct{}

// TxFromContext extracts the current transaction from context, if any.
func TxFromContext(ctx context.Context) *gorm.DB {
    if tx, ok := ctx.Value(txKey{}).(*gorm.DB); ok {
        return tx
    }
    return nil
}

// ContextWithTx returns a new context carrying the given transaction.
func ContextWithTx(ctx context.Context, tx *gorm.DB) context.Context {
    return context.WithValue(ctx, txKey{}, tx)
}
```

### Service Layer (creates and owns the transaction)

```go
// internal/receivable/service.go

func (s *Service) PostInvoice(ctx context.Context, cmd PostInvoiceCmd) (*Invoice, error) {
    var result *Invoice

    err := s.db.Transaction(func(tx *gorm.DB) error {
        // Place tx on context — all downstream calls use it
        txCtx := db.ContextWithTx(ctx, tx)

        // 1. Create invoice header
        inv, err := s.repo.CreateInvoice(txCtx, cmd.Header)
        if err != nil {
            return fmt.Errorf("create invoice: %w", err)
        }

        // 2. Create line items
        if err := s.repo.CreateLineItems(txCtx, inv.ID, cmd.Lines); err != nil {
            return fmt.Errorf("create line items: %w", err)
        }

        // 3. Post GL entries (cross-domain via interface)
        if err := s.journalPoster.PostEntries(txCtx, inv.ToJournalLines()); err != nil {
            return fmt.Errorf("post GL entries: %w", err)
        }

        // 4. Update AR balance
        if err := s.repo.UpdateOutstanding(txCtx, inv.ContactID, inv.Total); err != nil {
            return fmt.Errorf("update outstanding: %w", err)
        }

        // 5. Record audit event
        if err := s.audit.Record(txCtx, audit.InvoicePosted, inv); err != nil {
            return fmt.Errorf("record audit: %w", err)
        }

        result = inv
        return nil
    })

    return result, err
}
```

### Repository Layer (receives transaction from context, or creates local)

```go
// internal/receivable/repository.go

func (r *Repository) CreateInvoice(ctx context.Context, header InvoiceHeader) (*Invoice, error) {
    conn := r.getConn(ctx)

    inv := &Invoice{...}
    if err := conn.WithContext(ctx).Create(inv).Error; err != nil {
        return nil, fmt.Errorf("insert invoice: %w", err)
    }
    return inv, nil
}

// getConn returns the transaction from context if available,
// otherwise returns the default DB connection.
// It does NOT create or propagate a transaction.
func (r *Repository) getConn(ctx context.Context) *gorm.DB {
    if tx := db.TxFromContext(ctx); tx != nil {
        return tx
    }
    return r.db
}
```

### What This Prevents

| Scenario | Without this pattern | With this pattern |
|----------|---------------------|-------------------|
| Repository accidentally starts a wider transaction | Nested transactions, unexpected commits/rollbacks | Repository uses caller's tx or its own isolated one |
| Service forgets to wrap in transaction | Partial writes across tables | Service layer is the only place transactions begin — code review catches missing tx easily |
| Child function widens caller's transaction scope | Subtle bugs where unrelated operations share a tx | Local transactions are never propagated upward |
| Cross-domain call needs to participate in caller's transaction | Complex distributed transaction management | Transaction flows down via context — the journal poster uses the same tx as the invoice poster |

## 3. Business Logic Extraction Strategy

### Extraction Order

Extract domain by domain, simplest to most complex, with parity verification at each step:

| Order | Domain | Key Stored Procedures | Complexity |
|-------|--------|----------------------|------------|
| 1 | Chart of Accounts | `account__save`, `account__delete`, `account_get` | Low |
| 2 | Contacts | `entity__save`, `entity_credit_account__save` | Low-Medium |
| 3 | General Ledger | `journal__add`, `journal__approve`, `gl__*` | Medium |
| 4 | Accounts Receivable | `invoice__new`, `invoice__post`, `payment__post` | High |
| 5 | Accounts Payable | `ap__*`, mirrors AR patterns | High |
| 6 | Reconciliation | `reconciliation__*` | Medium-High |
| 7 | Reporting | Various report functions | Medium (read-only) |

### Extraction Process Per Domain

For each domain, follow this sequence:

1. **Catalogue** — List all stored procedures in the domain, document their inputs/outputs/side effects
2. **Map** — Identify which logic is business rules (moves to Go) vs constraints (stays in DB)
3. **Implement** — Write the Go service with the tx-on-context pattern
4. **Test** — Unit tests for the Go logic (mocked repository), integration tests against the Ledgius DB
5. **Parity** — Run parity assertions comparing Go-driven Ledgius DB output against stored-proc-driven legacy DB output
6. **Retire** — Once parity is proven, the stored procedure is no longer called by the new system (but remains in the legacy DB for shadow comparison)

### Critical Risk: Decades of Encoded Edge Cases

LedgerSMB's stored procedures are battle-tested and encode decades of accounting edge cases. The extraction must be methodical — domain by domain — with the parity framework proving equivalence at each step. There is no shortcut here. A stored procedure that has handled thousands of real-world transactions over many years cannot be assumed to be simple, even if it looks short.

**Mitigation: mandatory parity unit tests.** Every extracted domain must have direct unit tests that execute the same business operation against both systems and compare the final values stored in the database tables. These are not optional integration tests — they are the primary safety mechanism.

```go
// internal/receivable/parity_test.go

func TestPostInvoice_ParityWithLegacy(t *testing.T) {
    // Setup: identical starting state in both databases
    fixtures.SeedBothDatabases(t, testCompany)

    // Act: post invoice via Go service (writes to Ledgius DB)
    inv, err := ledgiusService.PostInvoice(ctx, invoiceCmd)
    require.NoError(t, err)

    // Act: post same invoice via legacy stored procedure (writes to legacy DB)
    legacyTransID := legacy.CallStoredProc(t, "invoice__post", invoiceCmd.ToLegacyArgs())

    // Assert: every affected table has identical final values
    parity.AssertSetParity(t, parity.SetCheck{
        Table:      "acc_trans",
        ForeignKey: "trans_id",
        ForeignVal: inv.TransID,
        LegacyForeignVal: legacyTransID,
        OrderBy:    "chart_id, amount",
        IgnoreCols: []string{"entry_id"},  // internal sequence IDs will differ
    })

    parity.AssertRowParity(t, parity.ParityCheck{
        Table:       "ar",
        IdentityKey: "invnumber",
        IdentityVal: inv.InvNumber,
        IgnoreCols:  []string{"id"},
    })

    // Assert: aggregate financial state matches
    parity.AssertTrialBalance(t)
    parity.AssertAccountBalance(t, inv.ARAccountID)
    parity.AssertAccountBalance(t, inv.RevenueAccountID)
    parity.AssertAccountBalance(t, inv.GSTAccountID)
}
```

### Rules for parity tests

1. **One parity test per extracted stored procedure** — no exceptions. If a stored proc is being replaced by Go logic, there must be a test that runs both and compares.
2. **Compare final table state, not intermediate steps** — the Go implementation may use different internal steps, but the rows in the database must be equivalent.
3. **Test edge cases explicitly** — zero-amount lines, negative amounts (credit notes), multi-currency, rounding boundaries, period boundaries. These are where legacy procs encode the hard-won edge cases.
4. **Parity tests run in CI on every commit** — they are not slow-running optional tests. They gate the build.
5. **A parity test failure blocks the extraction** — if the Go service produces different table state from the legacy stored proc, the extraction is not complete.

### What to Watch For

- **Implicit side effects** — stored procedures may update multiple tables or fire triggers; catalogue all side effects before extracting
- **Sequence dependencies** — some procedures assume a specific calling order; the Go service must enforce this explicitly
- **Error handling differences** — PL/pgSQL `RAISE EXCEPTION` vs Go error returns; ensure all error paths are covered
- **Decimal precision** — Go must use `shopspring/decimal` or equivalent, never `float64`; match PostgreSQL's `NUMERIC` precision exactly

## 4. Integration with Migration Safety

This pattern integrates directly with the three pillars from A-0015 (Migration Safety):

- **Parity assertions** verify that Go-driven writes produce identical results to stored-proc-driven writes
- **Dual-post gateway** executes the Go service logic against Ledgius DB, then calls the legacy stored procedures against the legacy DB, then compares
- **Transaction atomicity** is enforced by the tx-on-context pattern — the service layer guarantees all-or-nothing
- **Parity unit tests** are the primary safety mechanism — they execute both code paths and compare final database state row-by-row

## Key Principle

**Business logic lives in Go, data integrity lives in PostgreSQL.** The database enforces what data *can* exist (constraints). The Go service layer decides what data *should* exist (business rules).

# Risks / Trade-offs

- **Extraction complexity** — Stored procedures encode decades of edge cases; methodical domain-by-domain extraction with mandatory parity tests mitigates this.
- **Dual-database overhead during migration** — Running two databases adds infrastructure cost and operational complexity; this is temporary and justified by migration safety.
- **Context-propagated transactions** — Developers must understand the pattern; a misuse (e.g. creating a transaction in a repository) can break atomicity guarantees. Mitigated by code review conventions and lint rules.

# Alternatives Considered

- **Keep business logic in stored procedures**: Rejected — prevents testability, dual-post, and code visibility.
- **Big-bang rewrite without parity testing**: Rejected — too risky given decades of encoded edge cases in legacy stored procedures.
- **ORM-generated transactions (auto-commit per operation)**: Rejected — violates atomic transaction principle; compound operations need a single transaction boundary.

# Test Strategy

- **Parity tests**: One per extracted stored procedure, comparing Go-driven and stored-proc-driven final database state
- **Unit tests**: Business logic in domain services tested with mocked repositories
- **Integration tests**: Full database round-trip tests against the Ledgius database
- **Edge case coverage**: Zero-amount lines, negative amounts, multi-currency, rounding boundaries, period boundaries
- **CI gating**: All parity tests run on every commit and block the build on failure

# Rollout / Migration Notes

Extraction follows the domain order specified above (Chart of Accounts first, Reporting last). Each domain extraction is complete only when all parity tests pass. The dual-post gateway operates in `ModeDualPostVerify` during development and CI, ensuring any divergence is caught immediately.
