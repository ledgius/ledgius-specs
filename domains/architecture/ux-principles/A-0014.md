# Summary

Defines the complete product UX model for Ledgius as a workflow-centric, decision-oriented accounting system. Covers product positioning, core UX principles, user roles, navigation model, attention hierarchy, cognitive load budgeting, interaction contracts, keyboard ergonomics, density modes, state machine rendering, and page-by-page specifications for all major screens (dashboard, invoices, bills, journals, bank reconciliation, chart of accounts, reports, audit, approvals, period close, imports). Consolidates content from v3, v4, and the original 12 UX/UI principles.

# Technical Context

Ledgius replaces LedgerSMB's legacy Vue.js frontend with a React 19 + TypeScript application. The UX must serve four user modes (business owner, bookkeeper, finance admin, auditor) across workflows that prioritize correctness, auditability, and expert-speed operation. This specification drives page architecture, component behaviour, action/state design, API affordances, and audit model exposure.

# Proposed Design

## 1. Product Positioning

### 1.1 What Ledgius is optimizing for

Ledgius optimizes for this core accounting loop:

```text
Identify work -> Review evidence -> Decide -> Verify impact -> Commit -> Audit later
```

Every major screen should support this loop.

### 1.2 What Ledgius is not

Ledgius is not primarily a form-filling app, a dashboard vanity metrics app, a CRUD admin panel, or a bookkeeping toy optimized for beginners only. Ledgius is a financial operations workspace.

### 1.3 Primary product promise

The system should make users feel:
- "I know what needs attention"
- "I can see the financial effect before I commit"
- "I can move quickly without losing control"
- "I can explain and defend what happened later"

## 2. Core UX Principles

### 2.1 Decision before data entry

The UI should frame most workflows as decisions, not fields. Examples: reconciliation is "match / split / create / defer"; bill review is "is this valid, coded correctly, and postable?"; invoice review is "is this accurate, due, and ready to send?"

### 2.2 Verification before commit

Before any financial commit, the user should be able to verify: accounts affected, tax effect, totals, balance effect, period/date effect, locking implications.

### 2.3 Exception-first attention design

The UI should prioritize what is wrong, blocked, unmatched, overdue, or risky. The system should not make users hunt for the next problem.

### 2.4 Audit visible by default

The audit trail should not live in a hidden admin-only screen. At entity level, users should always see: created by, last changed by, current state, posting status, locks/period constraints, reversal/adjustment chain where relevant.

### 2.5 Safe speed

The fastest workflow must still be safe: keyboard-first support for experts, strong inline validation, reversible patterns where possible, destructive actions replaced by accounting-safe alternatives.

### Foundational Principles (from v1/v2)

1. **Task-Oriented Design** — Users act via clear tasks: Create Invoice, Receive Payment, Reconcile Bank
2. **Consistency** — Uniform layout, actions, terminology
3. **Visibility of State** — Always show Draft/Posted/Reversed, outstanding balances
4. **Error Prevention** — Confirm destructive actions, validate inputs early
5. **Audit Visibility** — Users can trace who did what, when, why
6. **Minimal Cognitive Load** — Avoid clutter, show only relevant info
7. **Progressive Disclosure** — Advanced details hidden until needed
8. **Feedback** — Immediate success/failure indicators
9. **Reconciliation UX** — Clear matching workflows
10. **Accessibility** — Readable, keyboard navigable
11. **Encapsulation** — Each screen handles one responsibility
12. **Predictable Navigation** — No hidden flows

**Key Principle:** Users must understand financial consequences before committing actions.

## 3. User Roles and UX Expectations

### 3.1 Roles

- Business owner / approver
- Bookkeeper / accountant
- Operations / finance admin
- Auditor / reviewer

### 3.2 Role expectations

**Business owner / approver**: Concise queue of decisions, confidence signals, clear status and impact, minimal accounting jargon where avoidable.

**Bookkeeper / accountant**: Dense fast workflows, strong keyboard support, easy drilldown, bulk actions, complete audit visibility.

**Finance admin / operations user**: Consistent templates and defaults, controlled mutation rules, clear exception queues, import/review/post workflows.

**Auditor / reviewer**: Immutable history, searchable event chains, reason codes and notes, side-by-side before/after views.

## 4. Mental Model and Navigation Model

### 4.1 Primary navigation structure

Top-level information architecture maps to accounting work, not database entities:

- Dashboard
- Sales
- Purchases
- Banking
- Ledger
- Reports
- Contacts
- Settings
- Audit

### 4.2 Section meaning

- **Dashboard**: Operational work queue, not vanity reporting
- **Sales**: Invoices, credit notes, receipts, customer balances, customer actions
- **Purchases**: Bills, vendor credits, payment scheduling, expense coding
- **Banking**: Bank feeds, reconciliation, transfers, cash allocation, statement review
- **Ledger**: Journal entries, chart of accounts, periods, tax journals, adjustments
- **Reports**: P&L, balance sheet, aged receivables, aged payables, GST/tax, cash movement
- **Contacts**: Customers, vendors, contact defaults, terms, tax treatment
- **Settings**: Org settings, tax settings, numbering, approval rules, closing periods, import rules
- **Audit**: System-wide immutable event view and change intelligence

### 4.3 Avoid entity-fragmented navigation

Do not create a top nav that mirrors tables (invoices, bills, journals, bank transactions, audit logs). That is an implementation view, not a user workflow view.

## 5. Attention Hierarchy

### Tier 1: Immediate attention
Always highest contrast / closest to action: out-of-balance states, unmatched bank lines, overdue invoices/bills, blocked posting states, exception counts, primary commit actions.

### Tier 2: Working context
Needed for decisions: line items, accounts, tax breakdowns, references, due dates, match evidence.

### Tier 3: Metadata and background context
Needed sometimes but should not dominate layout: IDs, timestamps, import source details, internal system keys, technical notes.

**Rule:** A screen should expose only one Tier 1 decision area at a time. Do not create screens with multiple competing primary actions.

## 6. Cognitive Load Budget

The UI should assume most users can actively reason about only 3 to 5 important things at once. Therefore: each panel should answer one question, each section should support one action cluster, supporting context should be collapsible or secondary.

Good section framing: "Transactions to reconcile", "Invoice totals and tax", "Posting impact", "Exceptions requiring review".

Bad section framing: "General Information", "Details", "Miscellaneous".

## 7. Global Interaction Contracts

### 7.1 No silent mutation
Posted accounting data should never be silently overwritten. Use: draft edit before posting, reverse, void, credit note, adjustment journal, superseding revision with explicit audit trail.

### 7.2 Confirm high-impact actions by impact
Bad: "Are you sure?" Good: "This will post to Accounts Receivable for $4,250.00, create GST payable of $386.36, and lock the invoice against direct editing."

### 7.3 Every commit action needs a visible post-condition
After commit, users should immediately see: new state, who performed it, timestamp, resulting totals/status, next valid actions.

### 7.4 Undo model
For drafts: direct undo is acceptable. For posted items: no destructive undo; use reverse/void/credit/adjustment.

## 8. Keyboard and Expert Mode

### 8.1 Baseline shortcuts

| Action | Shortcut |
| --- | --- |
| Save draft | Ctrl/Cmd + S |
| Post / Commit | Ctrl/Cmd + Enter |
| Cancel edit | Esc |
| New line | Enter |
| Delete current draft line | Ctrl/Cmd + Backspace |
| Search global | / or Ctrl/Cmd + K |
| Open audit panel | Shift + A |
| Focus exceptions | Shift + E |

### 8.2 Grid navigation contract

In editable tables: arrow keys move cell focus, Enter enters edit mode, Enter again commits cell edit, Tab moves horizontally, Shift+Tab moves backward, Esc abandons in-cell edit, paste of row/column data should be supported where practical.

### 8.3 Expert affordances

Denser spacing, shortcut hints inline on primary actions, optional sticky totals bar, batch actions without hiding review context.

## 9. Density Modes

### 9.1 Comfortable mode
For onboarding, occasional users, approvals, mobile or narrower widths. Larger spacing, more visible labels, stronger helper text, fewer columns at once.

### 9.2 Compact mode
For bookkeepers, reconciliation, journal entry, review queues. Tighter rows, persistent keyboard focus states, high information density, reduced helper text after competence is established.

## 10. Status and State Machine Principles

Entity states should be explicit, named, and enforced consistently across UI and backend. Every record should visibly show: current state, lock status, due/overdue status if relevant, reconciliation status if relevant, approval status if relevant.

State diagrams for Invoice, Bill, Journal Entry, and Bank Transaction Review are defined in A-0012 (Entity State Machine Specification). UI rendering must follow those state machines for status pills, action gating, and disabled action explanations.

## 11. Dashboard Specification

### 11.1 Purpose
The dashboard is a work queue and risk surface. It is not a KPI billboard.

### 11.2 Primary sections
Today/This week work queue, Reconciliation queue, Approval queue, Overdue receivables/payables, Exceptions panel, Recent commits and reversals, Key deadlines.

### 11.3 Behaviour rules
Hide all-zero queue cards by default. Sort work cards by urgency, not alphabetically. All cards should deep-link into filtered views. Counts must reflect actionable items only.

### 11.4 Card content rules
Each queue card should show: issue count, financial magnitude if meaningful, age or urgency, one direct action. Example: "14 bank lines unmatched - oldest 5 days - Review now". Bad example: "Banking".

## 12. Global Search and Command Surface

Search should find: invoices, bills, contacts, journal references, accounts, bank references, audit events. Results grouped by entity type. From command search, support: open record, create invoice/bill/journal, go to reconciliation queue, open chart of accounts, open audit event search.

## 13. Invoice Entry Screen

### Goal
Create, review, approve, send, and later audit an invoice with minimal friction and clear financial consequences.

### Zones
- **Header zone**: customer, invoice number, issue date, due date, state, payment status
- **Line items grid**: description, quantity, unit amount, account/category, tax code, line total (inline editable, Enter on final cell adds next row)
- **Totals panel**: live subtotal, tax total, grand total, payments applied, outstanding amount
- **Verification panel**: accounts affected, tax treatment summary, warnings, posting state, next action validity
- **Activity panel**: created by/when, edited by/when, sent by/when, payment events, credits/reversals

### Validation rules
Block commit when: missing customer, invalid dates, invalid tax code, negative total where invoice type disallows it, no lines.

### Warnings vs blockers
Blockers: impossible accounting state. Warnings: due date earlier than issue date, unusually large amount vs contact history, duplicate reference risk.

## 14. Bill Entry Screen

### Goal
Process supplier spend quickly while preserving coding accuracy and audit traceability.

### UX differences from invoice screen
Bills emphasize: vendor defaults, expense account coding, tax treatment, due date scheduling, approval and payment readiness.

### Additional panels
Vendor defaults and recent coding suggestions, duplicate bill detection evidence, payment scheduling summary.

### Duplicate risk detection
Surface potential duplicates using: supplier name similarity, invoice number/reference similarity, amount match, date proximity. Do not hard-block without strong certainty; present review evidence instead.

## 15. Journal Entry Screen

### Goal
Support deliberate, accurate, high-control ledger adjustments.

### Core UX requirements
Running debit/credit totals always visible, balance delta pinned while editing, post button disabled when imbalance != 0, explanation/memo strongly encouraged, attachment support optional but valuable.

### Line grid columns
Account, description, debit, credit, tax code if applicable, tracking dimensions if supported.

### Posting confirmation must show
Posting date, affected period, number of lines, whether period is near locked/closed boundaries.

## 16. Bank Reconciliation Screen

### Importance
This is one of the highest-value workflows in the system. It must be first-class.

### Primary goal
Move users from imported bank activity to a reconciled, evidenced ledger position with minimal ambiguity.

### Main screen zones

**Bank lines queue**: amount, date, description/reference, source account, match confidence state.

**Match workspace**: candidate matches, confidence reasons, selected action, balance consequences.

**Exception rail**: unmatched items, duplicates, unusual amounts, stale deferred items, suspicious splits.

### Evidence-based matching rules
Each candidate should show evidence chips: exact amount match, date within 2 days, reference similarity, contact match, previous pattern match. Never show a single confidence score without evidence explanation.

### Core actions
Match, Split, Create transaction, Transfer, Defer, Exclude if business rules allow.

### Reconciliation speed requirements
Full keyboard operability, one-key accept next suggestion in expert mode, bulk accept only for low-risk exact matches, persistent context after resolve.

### Split UX rules
Split workflow must visibly maintain: remaining amount, number of allocations, tax treatment per split line, unresolved residual if any.

## 17. Chart of Accounts Screen

### Goal
Support fast discovery, safe maintenance, and controlled evolution of the ledger structure.

### UX requirements
Search-first interface, tree and flat view toggle, type and usage filters, visible indicators for archived/locked/system accounts.

### Account detail view must show
Account type, tax defaults, parent grouping, active/inactive state, recent usage examples, audit history.

### Mutation controls
Potentially risky actions like account type changes must: require explicit warning, show downstream effect, require reason note, maybe require approval depending on org settings.

## 18. Reports UX

### Principle
Reports are not dead documents. They are interactive investigation surfaces.

### Requirements
Every report should support: date range controls, basis toggle if applicable, compare periods, drilldown from every material number, export with visible filters included.

### Drilldown contract
Clicking a number should reveal: underlying transactions, source record links, filter context preserved, subtotal context retained.

### Report clarity requirements
Explicit sign conventions, visible basis and period, sticky subtotal/header rows on large reports, highlight material changes in comparison mode.

## 19. Audit UX

### Principle
Audit is a core user journey, not a hidden compliance feature.

### Entity-level audit exposure
Every major record should have an activity panel showing: create event, field changes, status changes, send/post/pay actions, reversals and linked records, actor identity and timestamp.

### Global audit screen
Must support filtering by: date range, user, entity type, entity ID/reference, action type, risk level.

### Audit event design
Each event should show: action summary, before/after diff where relevant, actor, time, linked record references, note/reason code if provided.

### Immutability expectations
The audit log must be append-only in user experience and technical architecture.

## 20. Approvals UX

### Approval panel must show
What is being approved, financial amount, tax effect, due date/period effect, attached evidence if relevant, submitter, key warnings.

### Approval actions
Approve, Reject with reason, Request changes, Escalate. Rejecting must require a reason. That reason becomes part of the activity trail.

## 21. Period Close UX

Users should always know: current open period, soft-close status, hard-close status, what will happen if they post into a prior period. When posting into sensitive periods: warn before commit, show posting period explicitly, route into adjustment flow if period is closed.

## 22. Imports UX

### Import workflow stages
```text
Upload -> Parse -> Validate -> Map -> Review exceptions -> Commit import -> Audit
```

### Import screen requirements
Show: source file, detected schema, mapping preview, row counts, validation errors, warnings, sample transformed rows.

### Import commit rules
Users must know before commit: how many rows will be created, how many rows will be skipped, whether any existing entities will be linked, whether posting will occur immediately or draft records will be created.

## 23. Empty, Loading, and Error States

**Empty states**: Guide next action, not just announce absence. Bad: "No invoices". Good: "No invoices in Draft. Create a new invoice or clear filters."

**Loading states**: Preserve layout skeletons, avoid full-screen spinners, keep table headers visible during refresh, indicate whether data is stale or refreshing.

**Error states**: State what failed, whether the action committed or not, what user can do next, whether local changes were preserved.

## 24. Mobile and Responsive Rules

Core heavy accounting workflows are desktop-first. On mobile, prioritize: dashboard queue review, approvals, invoice/bill status checks, contact lookup, lightweight creation and comments. Do not force dense journal or reconciliation workflows into poor mobile experiences.

## 25. Design System Guidance

### Color semantics
- primary: commit/action emphasis
- success: resolved/posted/paid
- warning: requires attention soon
- danger: destructive / non-routine / accounting-safe reversal warnings
- neutral: context and structure

### Typography hierarchy

| Level | Purpose |
| --- | --- |
| XL | balances, totals, deltas |
| L | page and section headers |
| M | field labels, table headers |
| S | metadata, helper text, secondary info |

### Spacing
Use an 8px spacing system. Financial UIs benefit from consistency more than novelty.

### Component priorities
Most critical reusable components: editable financial grid, totals summary panel, status pill system, evidence chips, exception list item, activity timeline, confirmation panel with impact summary.

## 26. Anti-Patterns

Do not:
- use modals for core accounting workflows that need review context
- hide financial impact until after commit
- overload dashboards with vanity metrics
- allow posted data to be edited like drafts
- require mouse-only interaction for high-frequency workflows
- collapse audit into a separate admin-only universe
- use generic success states that hide actual resulting state

## 27. Inline Creation Pattern

### 27.1 Principle

For entity list pages with simple creation forms (flat fields, no line items), the "New" action must **expand an inline creation section above the list** rather than navigating to a separate page. This keeps the existing list visible while the user creates new items, serving two critical purposes:

1. **Duplicate prevention** — users can see existing entries while adding, avoiding accidental duplicates
2. **Batch entry workflow** — users can work through a list of new items without navigating back and forth

### 27.2 When to use inline creation

Use inline creation when the form:
- Has flat fields only (no dynamic line items or sub-tables)
- Has fewer than ~15 fields
- Does not require complex relational lookups (e.g. cascaded Combobox pickers)
- Creates a single entity per submission

### 27.3 When to use separate page creation

Use a dedicated creation page when:
- The form has dynamic line items (invoices, bills)
- The form has cascaded relational dependencies (credit notes → invoice lookup)
- The vertical space required is unbounded

### 27.4 Inline creation UX rules

1. **Toggle via "New" button** — the same header button that previously navigated to `/new` now toggles the inline form open/closed
2. **Position** — the form appears directly above the data table, below the page header
3. **Visual separation** — use a bordered card with subtle background (`bg-gray-50`) and a collapse/expand affordance
4. **List remains visible** — the data table stays below the form at all times; page scrolls if needed
5. **Focus on open** — first input field receives focus when the form expands
6. **Auto-close on success** — after successful creation, the form collapses, the list refreshes, and a success notification appears
7. **Escape to close** — pressing Escape collapses the form (with confirmation if fields are dirty)
8. **Persist form state** — if the user scrolls down to check the list and scrolls back, form data is preserved
9. **Cancel button** — secondary button to discard and collapse the form
10. **Multiple entries** — after successful creation, the form resets but stays open for the next entry (user closes explicitly when done)

### 27.5 Pages using inline creation

| Page | Entity | Fields |
|------|--------|--------|
| Chart of Accounts | Account | Code, Description, Category, Heading, Contra, Tax |
| Customers | Customer | Name, Legal Name, ABN, Currency, Terms, Credit Limit |
| Vendors | Vendor | Name, Legal Name, ABN, Currency, Terms, Credit Limit |
| Products & Services | Product | Name, SKU, Type, Unit, Sell/Buy Price, Tax Codes, Accounts |
| Employees | Employee | Name, Email, Phone, Start Date, Type, Pay Cycle, Residency, Super, Tax |

### 27.6 Pages keeping separate creation

| Page | Entity | Reason |
|------|--------|--------|
| Invoices | Invoice | Dynamic line items with running totals |
| Bills | Bill | Dynamic line items with running totals |
| Credit Notes | Credit Note | Cascaded invoice lookup with validation |

## 28. Implementation-Level UX Requirements (formerly 27)

### Frontend implications
Optimistic draft saves only where safe, explicit mutation result states, sticky side panels for totals/verification/audit on desktop, entity activity streams as first-class components, reusable state pill and evidence-chip systems.

### Backend implications
State transitions, commit preflight/validation summaries, audit event feeds, confidence evidence for reconciliation suggestions, duplicate detection evidence, lock/period constraints.

### Data model implications
Immutable event history, linked reversal chains, explicit entity states, soft vs hard locks, reason codes/notes for critical mutations.

## 28. Acceptance Criteria for UX Reviews

A screen should fail UX review if the reviewer cannot answer these questions in under 10 seconds:
1. What is the primary job of this screen?
2. What requires attention first?
3. What happens if I commit this action?
4. Can I see the financial impact before committing?
5. Can I find what changed later?

A workflow should fail UX review if: it requires excessive context switching, it hides key totals or balance effects, it obscures why a record is blocked, it prevents experts from moving quickly, it lets users create ambiguous financial states.

## 29. Final Product Principle

Users must always be able to answer:
1. What am I doing?
2. Why am I doing it?
3. What financial impact will this have?
4. What state will this record be in after I commit?
5. How do I review or explain this later?

If the UI fails any of those questions, it is not done.

# Data Flow

```
User Action --> React Component --> TanStack Query Hook --> Go API
  --> Preflight validation response (blockers, warnings, impact summary)
  --> User confirms --> Commit mutation --> Updated state + audit event
  --> UI reflects new state, shows post-condition feedback
```

# Risks / Trade-offs

- **Complexity of workflow-centric UX** — More design effort than simple CRUD forms; justified by accounting correctness requirements.
- **Dual density modes** — Maintaining comfortable and compact modes doubles some UI work; essential for serving both novice and expert users.
- **Keyboard-first design** — Requires thorough shortcut management and focus handling; critical for bookkeeper productivity.

# Alternatives Considered

- **Forms-first CRUD approach**: Simple form-per-entity design. Rejected — fails the core accounting loop of decision-verification-commit-audit.
- **Dashboard-first vanity metrics**: KPI-heavy dashboard as landing page. Rejected — accountants need work queues, not charts.
- **Mobile-first responsive**: Designing primarily for mobile. Rejected — core accounting workflows are desktop-first; mobile gets reduced capability.

# Test Strategy

- Component tests for all shared UI components (StatusPill, StatusStepper, editable grids, evidence chips)
- Page-level tests verifying correct state rendering, action gating, and disabled action explanations
- Keyboard navigation tests for all critical workflows
- UX review acceptance criteria applied to every screen before release

# Rollout / Migration Notes

UX principles are applied to all new frontend pages as they are built. Existing pages are progressively updated to comply. The UX review acceptance criteria (Section 28) gate all page releases.
