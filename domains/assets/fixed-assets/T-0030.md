# Summary

Enhancement to T-0029's cash-mode acquisition — adds the two bill-linked entry modes so the user can either (a) create a new AP bill at the same time as acquiring the asset or (b) reclassify an existing bill's expense line(s) into the asset's capital account.

T-0029 delivers **cash mode** (bank → capital + GST input + audit, in one tx). This task layers on the two remaining entry modes per R-0062 AST-004a and A-0041 §3:

- **Create-new-bill mode** — the Buy Asset flow initiates a new `ap_invoice` with the capital account coded directly on the line. Bill + asset are created in one DB transaction.
- **Link-existing-bill mode** — the user picks an already-posted, unpaid bill that was (incorrectly or provisionally) coded to a generic expense account. A reclassification journal moves the line value from the expense account to the asset's capital account.

The previous draft of T-0030 owned all three modes including cash. That was wrong — cash mode is the foundation and has now moved into T-0029 so the feature is usable immediately after T-0029 merges. T-0030 is the enhancement.

# Terms

- **Cash mode** (T-0029) — direct bank→capital posting with no AP bill involved.
- **Create-new-bill mode** (this task) — bill + asset created together; bill's line is coded to the capital account.
- **Reclassify mode** (this task) — a reclassification journal moves a bill line from expense to capital; the original bill is unchanged.

# Branch

`feat/fixed-assets-bill-linked-acquisition` — rebased on T-0029's Phase-2 (Acquire) branch once that merges, so bill-linked modes layer on top of cash mode.

# Phase 1 — Backend

## T-0030.1: Extend AcquireAssetCmd

Extend the command from T-0029.12 with an explicit entry mode:

```go
type EntryMode string
const (
    EntryModeCash        EntryMode = "cash"           // T-0029 — bank_account_id
    EntryModeNewBill     EntryMode = "create_new_bill"
    EntryModeLinkBill    EntryMode = "link_existing_bill"
)

type AcquireAssetCmd struct {
    // ... T-0029 fields ...
    EntryMode     EntryMode

    // Cash mode (T-0029):
    BankAccountID *int

    // Create-new-bill mode (this task):
    NewBillDraft  *BillDraft // bill reference, date, due date, line(s)

    // Link-existing-bill mode (this task):
    LinkBillID       *int       // transactions.id of the source bill
    LinkBillLineIDs  []int      // which line(s) to reclassify (unmodified lines stay on the expense account)
}

type BillDraft struct {
    Reference     string
    BillDate      time.Time
    DueDate       time.Time
    SupplierEntityCreditAccountID int
    // The asset is line 1; additional lines are not in scope for v1.
    TaxCode       string
}
```

Validation: exactly one of `BankAccountID` / `NewBillDraft` / `LinkBillID` must be set, consistent with `EntryMode`.

## T-0030.2: Create-new-bill flow

1. Begin tx (T-0029's RunInTx).
2. INSERT `asset_register` (same as T-0029.12 step 6).
3. Delegate to `bill.Service.CreateBill` with a one-line bill where the line's account is `asset.capital_account_id` and the tax code drives GST. `CreateBill` posts its normal AP journal (Dr capital + Dr GST Paid + Cr AP) and returns `bill_id` + `transaction_id`.
4. Set `asset.acquisition_transaction_id = bill.transaction_id` and `asset.linked_bill_id = bill.id`.
5. Audit row with `payload.entry_mode = 'create_new_bill'`.
6. COMMIT.

**No new journal command** needed — the bill service already posts the correct GL per the standard AP flow; the capital account on the line is what makes it an asset acquisition. The asset row's `linked_bill_id` + `acquisition_transaction_id` give the two directions of traversal.

## T-0030.3: Reclassify mode

1. Load source bill + the picked line(s). Validate: bill not in locked period, line account is a non-capital expense account (typical case — user coded generically then realised it should be capitalised).
2. Begin tx.
3. INSERT `asset_register`.
4. Call `journal.Post(ctx, journal.AssetAcquisitionReclassifyCmd{asset, bill_line_ids, target_capital_account_id})` applying A-0041 §3:
   - Cr the tenant default expense account (from bill line) for `cost_ex_gst` — tagged `asset_id`, `reclass_from_bill={bill_id}`.
   - Dr `asset.capital_account_id` for `cost_ex_gst` — tagged `asset_id`, `reclass_from_bill={bill_id}`.
   - GST on the bill is unchanged — no new GST rows (the original bill is already in BAS scope).
5. Set `asset.acquisition_transaction_id` = reclassification transaction id; `asset.linked_bill_id = bill.id`.
6. Audit `acquired` with `payload.entry_mode = 'link_existing_bill'`, `source_bill_id=bill.id`.
7. COMMIT.

## T-0030.4: Journal service extension

Add `AssetAcquisitionReclassifyCmd` to `internal/journal/` per A-0041 §3. Rows carry `asset_id` + `reclass_from_bill` tags. Idempotency key: `asset:acquire:{asset.id}` (same key as cash mode — one asset, one acquisition path).

## T-0030.5: Tests

- Create-new-bill happy path: verify bill is created with line on capital account, asset's `linked_bill_id` + `acquisition_transaction_id` set, audit row correct.
- Reclassify happy path: verify reclassification rows match A-0041 §3, original bill untouched, no new GST rows.
- Reclassify rejects when source bill's line isn't an expense account (e.g. already capital, or a GST account).
- Reclassify rejects when source bill is in a locked period.
- Error branches: multiple entry modes set; no entry mode set; bill_id doesn't exist; line_id doesn't belong to bill.
- Ledger invariant per case.

# Phase 2 — Frontend

## T-0030.6: BuyAssetPage enhancement

Building on T-0029's cash-mode BuyAssetPage:

- Add an entry-mode segmented control at the top: `[Cash / Bank]` `[Create new bill]` `[Link existing bill]`.
- **Cash mode** (T-0029): unchanged.
- **Create-new-bill mode**: reveal bill-header fields (reference, bill date, due date) below the common acquisition fields. Supplier dropdown drives the bill's supplier. Submit posts both journals atomically.
