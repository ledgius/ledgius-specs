# Summary

Deliver the asset disposal flow end-to-end. Backend: `POST /api/v1/assets/:id/dispose` that atomically posts any pro-rata depreciation, then the **accounting** disposal journal per A-0041 §6, and transitions the asset to `Disposed`. Frontend: wire up `SellDisposePage` to the endpoint, support pre-selected asset via `?assetId=` query param (launched from the detail page), and render the live accounting gain/loss calc.

Depends on T-0029 (schema + register/detail) and T-0030 (there must be something to dispose). Does NOT depend on T-0032 (pro-rata depreciation for partial period is computed locally in this task — but the formulas live in a shared `depreciation/calculate.go` module so T-0032 can reuse).

## Hard dependency on R-0073

**This task does not merge until R-0073 (Tax Depreciation &amp; Disposal Layer) is approved and its implementation stubbed.** Disposal triggers both:

1. An **accounting** event (AASB 116) — the gain/loss posted by this task.
2. A **tax** event — Div 40 balancing adjustment per s40-285 ITAA 1997, plus a CGT component per s118-24 for any non-taxable-use portion, plus small-business pool treatment per R-0074.

Shipping the accounting event without a tax-layer call site would give users a journal that looks right on the P&amp;L but leaves tax outcomes silently wrong until year-end. The reviewer's compliance check on PR #28 called this out explicitly.

Acceptable gate conditions before T-0031 merges:

- R-0073 spec is **Approved** status.
- R-0073 has a parallel implementation task (T-00xx) with at least a stub endpoint + DB schema landed.
- T-0031's dispose command calls the tax layer (or a named stub function) for each disposal, so the integration point is live from day one even if the tax computation is incomplete.
- If R-0073 is not yet ready: T-0031 is held. This task may be rebranched as `feat/fixed-assets-dispose-accounting-only` for a preview-only (read-only disposal summary, no POST) experience, but the POST endpoint does not ship without R-0073.

# Terms

- **Sale proceeds** — cash received (or receivable) for the asset.
- **Gain / loss on disposal** — `sale_ex_gst − book_value`. Positive = gain (income). Negative = loss (expense).
- **Pro-rata depreciation** — depreciation for the partial period between the last run and the disposal date, posted immediately before the disposal in the same DB transaction.

# Phase 1 — Shared depreciation calculator

## T-0031.1: extract calculator

Create `internal/asset/depreciation/calculate.go` with pure functions (no DB, no side effects):

```go
type Method string
const (
    StraightLine     Method = "straight_line"
    DiminishingValue Method = "diminishing_value"
    InstantWriteoff  Method = "instant_writeoff"
)

// PeriodDepreciation returns the depreciation for [from, to] for a single asset.
// Floors at residual — never returns a value that would push book below residual.
func PeriodDepreciation(
    method Method,
    costExGST, residual, accumulated decimal.Decimal,
    usefulLifeYears int,
    from, to time.Time,
) decimal.Decimal
```

Rules per R-0062 AST-007/008:

- StraightLine: `annual = (cost − residual) / life`; `period = annual × days / 365`.
- DiminishingValue: `period = book_value × days/365 × 200% / life`; book_value = cost − accumulated.
- InstantWriteoff: returns `cost − residual` if `accumulated == 0`, else 0 (already written off).

Unit tests cover boundaries: zero days, a single day, full year, partial year crossing FY end, residual floor, full depreciation stop.

Both T-0031 and T-0032 depend on this module.

# Phase 2 — Backend disposal

## T-0031.2: service command

Add to `internal/asset/service.go`:

```go
type DisposeAssetCmd struct {
    TenantID                    uuid.UUID
    ActorUserID                 uuid.UUID
    AssetID                     uuid.UUID
    DisposalDate                time.Time
    SaleProceedsIncGST          decimal.Decimal
    GSTOnSale                   decimal.Decimal
    Reason                      string // sold | scrapped | donated | traded_in
    BuyerEntityID               *int64
    DestinationBankAccountID    *int64   // exactly one of these two must be set
    ARBuyerEntityCreditAccountID *int64   // credit-sale mode
}

func (s *Service) DisposeAsset(ctx context.Context, cmd DisposeAssetCmd) (*Asset, error)
```

Implementation outline:

1. Load asset. Validate: state ∈ {Active, FullyDepreciated}, `disposal_date >= purchase_date`, `sale_proceeds >= 0`.
2. **Period-lock check** (AST-016 / A-0040 I9): reject with `ErrPeriodLocked` if `disposal_date` falls in a locked or BAS-lodged period. This is a hard gate — disposal cannot silently rewrite a closed period.
3. Resolve gain/loss accounts from tenant COA defaults. Verify both carry `reporting_category='other_income'` or `'other_expense'` — reject with `ErrGainLossAccountMisconfigured` otherwise.
4. Begin tx.
5. Compute pro-rata depreciation for `(last_run_end or purchase_date, disposal_date]` using the shared calculator. If > 0, build per-asset dep rows per A-0041 §4 and include them in the parent transaction.
6. Compute accounting `gain_loss = sale_ex_gst − (cost − (accumulated + pro_rata))`.
7. Call `journal.Post(ctx, journal.AssetDisposalCmd{...})` which applies A-0041 §6 — note the command takes the pro-rata dep rows AND the disposal rows and posts them under a single `transactions` parent of type `asset_disposal`. Rows carry `asset_id` and `disposal=true` only — **no `bas_label` tags** (A-0042 derives BAS).
8. Update asset: `status='disposed'`, `disposal_date`, `disposal_transaction_id`, `accumulated_depreciation += pro_rata`.
9. **Invoke the tax layer** (R-0073) to compute and record the Div 40 balancing adjustment and any CGT component. This is a cross-layer call, NOT a write to `acc_trans` — R-0073's output persists in its own tables. If R-0073 is not yet implemented, call a named stub function that logs the disposal + returns a "tax deferred" marker; do **not** no-op silently.
10. `audit.Write(entity='asset', action='disposed', payload={sale_ex_gst, accounting_gain_loss, reason, tax_layer_invoked: bool})`.
11. COMMIT.

Errors: `ErrAssetNotDisposable`, `ErrDisposalDateInvalid`, `ErrPeriodLocked`, `ErrGainLossAccountUnresolvable`, `ErrGainLossAccountMisconfigured`.

## T-0031.3: journal service extension

Add `AssetDisposalCmd` to `internal/journal/` — the only code path that writes the disposal rows. Takes the asset, disposal inputs, resolved accounts, and pre-computed pro-rata dep rows. Produces rows per A-0041 §6 (pro-rata first, then disposal) in one `transactions` parent.

Unit tests cover every row shape in A-0041 §6:
- Gain scenario (proceeds > book).
- Loss scenario (proceeds < book).
- Scrap (proceeds = 0, GST = 0).
- Cash sale vs credit sale (bank vs AR).
- Motor vehicle with `business_use_pct` — verifies pro-rata dep Dr expense is split into deductible + non-deductible per A-0041 §4 vehicle rule.
- Asset fully depreciated already (book_value = residual) — no pro-rata dep rows added; disposal rows handle the residual book value.
- Ledger invariant `sum(Dr) == sum(Cr)` asserted per case.

## T-0031.4: handler

`POST /api/v1/assets/:id/dispose` → `handler.Dispose` → `service.DisposeAsset`.

- Permission: `assets:dispose`.
- Feature gate: `fixed_assets`.
- Request body + URL param validated per §T-0031.2 step 1.
- Response: 200 with the updated asset + a summary of posted journal (parent transaction id, totals, gain_loss, pro_rata_dep_amount).

# Phase 4 — Verification

## T-0031.10: manual QA script

- Start from a seeded Active asset with non-zero accumulated dep and a last-run date earlier than today.
- Open detail page, click Dispose → form opens with asset pre-selected and book value populated.
- Enter sale price > book value → gain displayed in green.
- Enter sale price < book value → loss displayed in red.
- Enter sale price = 0 (scrap) → loss = book value.
- Submit → preview modal shows pro-rata dep rows + disposal rows → confirm.
- Verify back on detail: status = Disposed, disposal_date set, timeline has new `dep_posted` + `disposed` entries.
- Verify GL report for the tenant: one new `asset_disposal` parent txn with correct row set.
- Verify BAS-1A GST on sale line is tagged and will appear on the next BAS report.
