# Summary

Deliver the **accounting** depreciation engine end-to-end: preview, idempotent run, runs history, reversal, vehicle-logbook deductibility split at ledger level. Backend adds four endpoints over the `depreciation_run` / `depreciation_run_line` tables seeded in T-0029. Frontend replaces the DepreciationPage stub with a working control plane.

Scope note — **this task writes accounting depreciation only** per AASB 116. Income-tax depreciation (Div 40 per-asset method or small-business pool per R-0074) is owned by R-0073's tax layer. The two methods can produce different per-period amounts; they coexist and are reconciled at year-end. The `deductible` ledger tag set by this task is presentation metadata only, not a tax outcome.

Depends on T-0029 (schema + calculator module is created by T-0031 but may be inlined here if T-0031 lands later). If T-0032 lands before T-0031, move the calculator module extraction (T-0031.1) into this task and make T-0031 depend on it.

# Terms

- **Period** — monthly, calendar month-end in tenant timezone. Configurable later; out of scope here.
- **Eligible asset** — `status='active'`, `purchase_date <= period_end`, `book_value > residual`, not in a period already covered by a posted run for this period_end.
- **Preview** — calculate what the period run would post, show to user, do NOT write anything.
- **Run** — post the period journal atomically; idempotent per `(tenant_id, period_end)` while `status='posted'`.
- **Reversal** — compensating journal for a previously posted run; flips run status to `reversed`.

# Phase 3 — Backend engine

## T-0032.4: service commands

Add to `internal/asset/service.go`:

```go
type PreviewDepreciationCmd struct {
    TenantID  uuid.UUID
    PeriodEnd time.Time
}

type RunDepreciationCmd struct {
    TenantID    uuid.UUID
    ActorUserID uuid.UUID
    PeriodEnd   time.Time
}

type ReverseDepreciationRunCmd struct {
    TenantID    uuid.UUID
    ActorUserID uuid.UUID
    RunID       uuid.UUID
    Reason      string
}

type ListDepreciationRunsCmd struct {
    TenantID  uuid.UUID
    Status    *string
    FromDate  *time.Time
    ToDate    *time.Time
    Page      int
    PageSize  int
}

func (s *Service) PreviewDepreciation(ctx context.Context, cmd PreviewDepreciationCmd) (*DepreciationPreview, error)
func (s *Service) RunDepreciation(ctx context.Context, cmd RunDepreciationCmd) (*DepreciationRun, error)
func (s *Service) ReverseDepreciationRun(ctx context.Context, cmd ReverseDepreciationRunCmd) (*DepreciationRun, error)
func (s *Service) ListDepreciationRuns(ctx context.Context, cmd ListDepreciationRunsCmd) ([]*DepreciationRun, error)
```

### Preview

1. Determine period_start = first-of-month of `period_end` (last-day-of-month).
2. Load eligible assets (see §Terms).
3. For each, compute `amount` via calculator; `days_in_period` = count of days the asset was active in the period (typically days_in_period = days_in_month, but prorates for assets acquired mid-period).
4. Apply vehicle business-use split per §T-0032.3.
5. Return `DepreciationPreview { period_start, period_end, total_amount, lines: [{asset_id, name, method, amount, deductible, non_deductible, book_value_before, book_value_after}] }`.
6. No DB writes. Cacheable read-model.

### Run

1. Begin tx.
2. Attempt to INSERT `depreciation_run` with `status='posted'`. The unique partial index `(tenant_id, period_end) where status='posted'` rejects duplicate runs for the same period → map to `ErrDepreciationAlreadyRun`.
3. Recompute the preview inside the tx (don't trust a client-side cached preview — period state may have changed).
4. If zero eligible assets, still create the run with `total_amount=0` and `posted_at=now()` for the audit trail, skip journal posting. No empty journals.
5. For each line: INSERT `depreciation_run_line`.
6. Call `journal.Post(ctx, journal.DepreciationRunCmd{run, lines})` which applies A-0041 §4 — one parent `transactions` of type `depreciation_run`, two `acc_trans` rows per asset (or three for vehicles with split).
7. Update each asset's `accumulated_depreciation` and check `book_value <= residual` → flip to `fully_depreciated`.
8. `audit.Write(entity='depreciation_run', entity_id=run.id, action='posted', payload={period, total, asset_count})`.
9. COMMIT.
10. Return the persisted run.

### Reverse

1. Load run. Validate `status='posted'`.
2. Begin tx.
3. Call `journal.Post(ctx, journal.DepreciationReversalCmd{source_run_id})` which produces exact inverses per A-0041 §5.
4. Update each affected asset's `accumulated_depreciation` (subtract the line amount). If an asset was in `fully_depreciated` and this reversal brings book_value > residual, flip back to `active`.
5. UPDATE `depreciation_run` SET `status='reversed'`, `reversed_by`, `reversed_at`, `reversal_transaction_id`.
6. `audit.Write(entity='depreciation_run', action='reversed', payload={reason, original_total})`.
7. COMMIT.

### List

Straightforward paginated list with filters. Returns run rows + summary (no line detail unless a per-run endpoint is called).

## T-0032.5: journal service extension

- `DepreciationRunCmd` — input: run + lines. Produces rows per A-0041 §4 (with vehicle split).
- `DepreciationReversalCmd` — input: source run id. Loads the original rows via `run_id` tag on `acc_trans`, produces inverses in a new parent transaction.

Idempotency keys:
- `dep_run:{period_end}:{run.id}` — prevents double-post of the same run UUID (the `(tenant_id, period_end)` unique index on the run row is the primary guard; the journal key is a belt-and-braces check).
- `dep_line:{run.id}:{asset.id}` — unique per run line (already enforced at the run_line table level; mirrored in the journal tag).

## T-0032.6: endpoints

- `GET /api/v1/assets/depreciation/preview?period_end=YYYY-MM-DD` → `handler.Preview`.
- `POST /api/v1/assets/depreciation/run` body `{ period_end }` → `handler.Run`. `assets:run_depreciation`.
- `POST /api/v1/assets/depreciation/runs/:run_id/reverse` body `{ reason }` → `handler.Reverse`. `assets:reverse_depreciation`.
- `GET /api/v1/assets/depreciation/runs` → list.
- `GET /api/v1/assets/depreciation/runs/:run_id` → detail including lines.
- `GET /api/v1/assets/:id/depreciation-history` → per-asset history (lines across all runs).

## T-0032.7: tests

- Calculator module tests (from §T-0032.2 if not in T-0031, plus vehicle split).
- Service-level tests:
  - Preview returns correct lines for a mix of straight-line, diminishing-value, instant-writeoff, fully-depreciated (excluded), vehicle-with-80%-use.
  - Run succeeds once; second Run for the same period returns `ErrDepreciationAlreadyRun`; preview still works after posting (shows empty because no eligible).
  - Run with zero eligible assets creates a run with `total_amount=0`, no journal rows, audit row written.
  - Asset flips to `fully_depreciated` when book_value reaches residual post-run.
  - Reverse succeeds; asset `fully_depreciated` flips back to `active` if reversal pushes book above residual.
  - Reverse on already-reversed run returns error.
- Journal tests:
  - One run, five assets, including a Motor Vehicle with 60% business use — verify expense split rows + tag correctness.
  - Reversal produces inverse rows correctly for every case above.
- Handler tests via httptest — permissions, 409 on duplicate, 200 on success.
