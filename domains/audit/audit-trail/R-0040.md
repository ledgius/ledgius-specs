# Summary

Provide an immutable, append-only audit log that records every state-changing operation across all domain entities, capturing before/after snapshots, the acting user, and source IP address. The audit trail supports both global querying and per-entity activity feeds.

# Problem / Business Context

Australian accounting standards and the Privacy Act 1988 require businesses to maintain a verifiable record of all changes to financial data. Auditors, accountants, and business owners need to trace who changed what, when, and why. An immutable audit log is essential for compliance, dispute resolution, and internal controls.

# Scope

- Immutable audit log entry creation (append-only)
- Global audit log listing with optional entity type filtering
- Per-entity activity feed (e.g. all changes to a specific invoice)
- Before/after JSON snapshots for change diffing
- User and IP address attribution
- Configurable result limits with sensible defaults

# Out of Scope

- Audit log purging or archival policies
- Audit log export (CSV, PDF)
- Real-time audit event streaming / webhooks
- Tamper-proof cryptographic chaining (blockchain-style)
- Audit log search by user or date range (only entity_type filter implemented)

# Actors / Users

- Business Owner -- reviews audit trail for oversight
- Accountant -- verifies changes for compliance and reconciliation
- Bookkeeper -- checks recent activity on entities they manage
- Auditor (external) -- reviews change history during audit engagements

# Preconditions

- User is authenticated with a valid JWT
- Tenant context is established (audit entries are per-tenant database)

# Functional Requirements

- **AUD-001**: System shall store audit log entries as immutable records in the `audit_log` table
- **AUD-002**: Each audit entry shall capture: action performed, entity type, entity ID, before JSON snapshot, after JSON snapshot, acting user ID, IP address, and timestamp
- **AUD-002b**: (Extended by R-0052 IDR-014) Each audit entry shall additionally capture the active user population (`active_role`), the scope id within that population (`active_role_scope_id` — tenant_id / region_id / `*`), the originating client surface (`app_surface` — `web` / `mobile-ios` / `mobile-android` / `api` / `system`), the request correlation id (`request_id`), and an optional `actor_label` for system actors
- **AUD-003**: System shall provide a global audit log listing endpoint (`GET /audit-log`) with optional `entity_type` query filter
- **AUD-004**: System shall provide a per-entity activity endpoint (`GET /:entity/:id/activity`) that returns audit entries for a specific entity type and ID
- **AUD-005**: Global audit log shall default to returning the most recent 100 entries, configurable via `limit` query parameter
- **AUD-006**: Per-entity activity shall default to returning the most recent 50 entries, configurable via `limit` query parameter
- **AUD-007**: Audit entries shall be ordered by `created_at` descending (newest first)
- **AUD-008**: Before and after snapshots shall be stored as JSONB for structured querying capability
- **AUD-009**: The audit handler shall expose a reusable `EntityActivityHandler(entityType)` factory method so any domain can mount per-entity activity routes without code duplication

# Data / Entities / Fields

Primary entity: `Entry` (table: `audit_log`)

| Field | Type | Notes |
|-------|------|-------|
| id | int64 (PK) | Auto-increment |
| user_id | string (nullable) | UUID of the acting user |
| action | string | Required. e.g. "create", "update", "delete", "post" |
| entity_type | string | Required. e.g. "invoice", "contact", "account" |
| entity_id | string (nullable) | ID of the affected entity |
| before_json | jsonb (nullable) | JSON snapshot of entity state before the change |
| after_json | jsonb (nullable) | JSON snapshot of entity state after the change |
| ip_address | string (nullable) | Source IP of the request |
| created_at | timestamp | Required. Immutable creation timestamp |

# API Endpoints

| Method | Path | Description |
|--------|------|-------------|
| GET | `/audit-log` | List audit log entries. Query params: `entity_type` (optional filter), `limit` (default 100) |
| GET | `/:entity/:id/activity` | List audit entries for a specific entity. Mounted per-domain. Query params: `limit` (default 50) |

# Validation Rules

- Entity ID is required for per-entity activity queries (returns 400 if missing)
- Limit defaults to 100 (global) or 50 (per-entity) when not provided or when a non-positive value is supplied

# Error Handling

- Missing entity ID on per-entity activity endpoint returns 400 Bad Request with message "entity ID required"
- Database errors return 500 Internal Server Error
- Audit log write failures are wrapped with context for structured logging

# Security / Permissions

- All audit log endpoints require authentication
- Audit entries are scoped to the tenant database (multi-tenant isolation)
- Audit log is append-only; no update or delete endpoints are exposed

# Acceptance Criteria

- [x] Audit log entries are created with before/after JSON snapshots
- [x] Global audit log endpoint returns entries filtered by entity_type
- [x] Per-entity activity endpoint returns entries for a specific entity
- [x] Results are ordered newest-first with configurable limits
- [x] EntityActivityHandler factory allows any domain to mount activity routes
- [x] Audit entries include user_id, ip_address, and created_at

# Open Questions

1. **Date range filtering**: The global endpoint currently only filters by entity_type. Date range and user filters may be needed for auditor workflows.
2. **Retention policy**: No archival or purging strategy is implemented. High-volume tenants may need partitioning or offloading.
3. **Write-path integration**: The repository exposes a `Log()` method, but integration points in each domain (who calls Log and when) are not defined in this package -- each domain is responsible for calling the audit repository.
