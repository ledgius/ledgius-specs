# Summary

Direct bank account connections via the Basiq CDR-accredited aggregator that
automatically import transactions — eliminating manual CSV/OFX uploads.
Transactions flow into the existing reconciliation pipeline (R-0019) without
user intervention. Every event in the live-feed lifecycle is recorded with a
begin and end timestamp, status (success/error) and full error context to the
`bank_feed_event_log` (see R-0040 audit trail patterns).

# Business Context

Manual bank statement uploads are the single biggest friction point for
bookkeepers. Competitors (MYOB, Xero, QuickBooks) all offer "bank feeds" —
direct connections to AU banks that automatically pull transactions daily.
This is the most-requested feature for any AU accounting product.

The Australian market has a single sensible aggregator path:
**Basiq** — an Australian-founded company operating as an accredited CDR
intermediary / CDR Representative under the regime. It covers 170+ AU
financial institutions including all Big Four (CBA, NAB, Westpac, ANZ),
plus credit unions and neobanks.

# Compliance Posture

Ledgius operates as a downstream consumer of Basiq's CDR-accredited service.
Basiq holds the ACCC accreditation; Ledgius is not itself an Accredited Data
Recipient (ADR) and therefore does not need to seek ACCC accreditation for
this initial release. Notwithstanding, Ledgius MUST honour all consumer
rights flowing through from CDR:

- Maximum sharing duration: **12 months** (per CDR rules — see DSB standards). Business Consumer Consent caps at 12 months (per Basiq CDR docs).
- Right to revoke: consumers may revoke consent at any time via the Basiq Manage Consent flow; Ledgius MUST stop using the data immediately and trigger deletion/de-identification.
- Data minimisation: only fetch transaction fields required for reconciliation (R-0019).
- Deletion on revocation: per Basiq policy, "all data on Basiq is deleted upon expiration or revocation of consent". Ledgius MUST mirror this — purge or de-identify per-tenant transaction data sourced from a revoked consent, except where Australian law (e.g. tax record retention, AML) requires retention.

# Acceptance Criteria

## AC-1: Bank account connection

Users can connect their bank accounts from within Ledgius:
- Settings page or Banking section: "Connect Bank Account" button
- Opens the Basiq Consent UI (hosted by Basiq at `https://consent.basiq.io/home?token={client_token}&action=null`) — bank-grade, handles MFA, fully CDR-compliant
- User selects their bank, logs in at the bank, grants consent
- On success: bank account appears in Ledgius linked to a GL account
- Multiple bank accounts can be connected per tenant under a single Basiq user

## AC-2: Automatic transaction import

Once connected:
- Transactions are fetched on a configurable cadence (default 4 hours; see A-0025 §"Sync scheduling")
- New transactions are inserted into `bank_transaction` table with `source = 'feed'` (column added in this feature)
- Deduplication: transactions have stable Basiq-provided IDs (`bank_transaction.external_id`); duplicates are rejected via the existing `(account_id, external_id)` unique index
- Historical backfill: on first connection, fetch the last 90 days of transactions
- Transactions flow directly into the reconciliation pipeline (R-0019)
- A status badge in the header/sidebar shows "Bank feed: synced N hours ago"

## AC-3: Transaction data quality

Basiq returns CDR-conformant transaction objects. Per the DSB Banking standard
(`BankingTransaction` schema), each transaction provides three related
date-time fields with the invariant
`executionDateTime ≤ valueDateTime ≤ postingDateTime`. The implementation
MUST persist `posting_date` (from `postingDateTime`) as the canonical
`bank_transaction.trans_date`, and retain `transaction_date` (from
`executionDateTime`) where supplied.

The full set of fields mapped from Basiq → `bank_transaction` is specified in
A-0025 §"Transaction mapping".

## AC-4: Connection management

Users can manage their bank feed connections from `/settings/bank-feeds`:
- View connected accounts with last sync time and status
- Disconnect an account (stops future syncs, preserves imported data unless tenant elects deletion)
- Reconnect when consent expires (CDR maximum 12 months; consumer may pick shorter durations)
- Force a manual sync ("Refresh now" button)
- See sync history (drawn from `bank_feed_event_log`) and any errors

## AC-5: Consent and security

- Consumer credentials are NEVER seen or stored by Ledgius — Basiq Consent UI captures them at the bank
- CDR-compliant — consumer can revoke at any time via Basiq Manage Consent (`action=manage`)
- Server-to-server auth: Ledgius API → Basiq API uses Basiq's OAuth 2.0 client credentials grant
- Basiq API key + webhook signing secret stored as Fly.io secrets, never in code or VCS
- Consent expiry tracked per connection on `bank_feed_connection.consent_expires_at`

## AC-6: Error handling and notifications

- Sync failures (bank outage, consent expired, rate limit) MUST update connection status and surface in the UI — never fail silently
- 7 days before consent expiry: warning banner "Bank connection expires in N days — reconnect to continue automatic imports" + email notification (Resend)
- On consent expiry: connection moves to `status='expired'`; UI prompts reconnect via Basiq's `action=reauthorise` flow
- All failures recorded in `bank_feed_event_log` with full error payload

## AC-7: Fallback to manual upload

Bank feeds complement, not replace, manual uploads:
- Tenants without feeds continue using Import Transactions (CSV/OFX/QIF)
- Tenants with feeds may still upload files (e.g. for institutions not supported by Basiq)
- Deduplication works across both sources via `(account_id, external_id)` uniqueness — manual imports must populate `external_id` to participate

## AC-8: Pricing tier integration

Bank feeds are gated by the feature flag system (R-0041):
- Starter plan: manual upload only — feature flag `bank_feeds` disabled
- Essential+: 1–3 bank feed connections (per plan limits)
- Business: unlimited connections
- Feature gate: `bank_feeds` slug enforced via `middleware.RequireFeature("bank_feeds")` on all `/api/v1/bank-feeds/*` routes (excluding the public webhook endpoint)

## AC-9: Per-event audit logging (NEW — Daisy 2026-04-14)

Every live-feed event MUST be recorded in `bank_feed_event_log` with both a
**begin** and **end** record (single row with `started_at` + `completed_at`),
plus `status` (`running` → `success` | `failed`) and `error_message` /
`error_payload` populated on failure. The event types covered include:

| event_type            | Begin trigger                           | End trigger                                 |
|-----------------------|-----------------------------------------|---------------------------------------------|
| `connection.create`   | User initiates Connect Bank Account     | Basiq job complete OR user abandons consent |
| `connection.refresh`  | Manual sync / cron tick                 | Basiq job all-steps-complete or failed      |
| `consent.extend`      | Extension flow initiated                | Basiq returns new expiry                    |
| `consent.reauthorise` | Reauthorise flow initiated              | Connection re-active or failed              |
| `consent.revoked`     | Webhook receipt or manual disconnect    | Local cleanup complete                      |
| `webhook.received`    | Inbound POST received                   | Handler processing complete                 |
| `transactions.fetch`  | GET /transactions started               | Fetch complete (with counts)                |
| `transactions.dedup`  | Insert batch started                    | Insert complete (`new`, `duplicate` counts) |
| `expiry.check`        | Daily expiry-check job started          | Check complete (warnings sent count)        |

This requirement supersedes A-0025's earlier `bank_feed_sync_log` design —
the table is generalised to `bank_feed_event_log` per A-0025 §"Event journal".

# Technical Context

## Why Basiq

- AU-founded, CDR-accredited intermediary / CDR Representative
- 170+ AU institution coverage (per Basiq marketing, verify against
  https://www.basiq.io/financial-institutions for current list)
- CDR sharing in real-time-ish (CDR connections poll the data holder; latency
  is minutes, not seconds — claims of "real-time" should be tempered)
- Hosted Consent UI removes Ledgius from the bank-credential blast radius

## Key API flows (authoritative recipes from Basiq docs)

### Connection setup
1. Ledgius creates (or reuses) a Basiq user for the tenant — see A-0025 §"Provider user lifecycle"
2. Mint a client token bound to the Basiq user
3. Generate consent URL with `action=null` for first-time consent or `action=connect` for adding institutions
4. Redirect the consumer to the Basiq Consent UI
5. On return, poll the resulting Basiq job (steps: `verify-credentials` → `retrieve-accounts` → `retrieve-transactions`) per https://api.basiq.io/v2.1/docs/handling-jobs
6. Surface the user immediately after `verify-credentials` succeeds; complete the remaining steps in the background
7. Once `retrieve-accounts` succeeds, present the GL account mapping UI

### Transaction sync
1. Cron tick (or manual trigger) calls `GET /users/{userId}/transactions?filter=account.id.eq('{accountId}'),transaction.postDate.gteq('{lastSync}')` — paginated, max 500 per page
2. Map each Basiq transaction to `bank_transaction` per A-0025 §"Transaction mapping"
3. Upsert with `ON CONFLICT (account_id, external_id) DO NOTHING`
4. Record begin+end in `bank_feed_event_log`

### Consent management
- Basiq emits webhooks on connection lifecycle changes — verify per https://api.basiq.io/docs/webhooks-security (HMAC-SHA256 over `{webhook-id}.{webhook-timestamp}.{raw-body}`, headers `webhook-id`, `webhook-timestamp`, `webhook-signature`, 5-minute replay window, IP allowlist 13.238.192.210, 3.24.252.173, 3.104.17.39)
- Daily `expiry.check` job sends T-7 reminders and flips connection status

# Out of Scope (Current Release)

- Non-AU banks (Basiq is AU-only; international via Plaid/MX is future)
- Real-time push notifications from bank (CDR doesn't support this in 2026)
- PSD2 / UK / EU Open Banking
- Direct bank API integrations (bypassing Basiq)
- Transaction enrichment beyond what Basiq returns
- Use of Basiq's ML auto-categorisation (Ledgius uses its own rules engine — see R-0019)
- Ledgius applying for direct ACCC ADR accreditation

# Open Questions

1. **Tenant ↔ Basiq-user mapping**: one Basiq user per tenant or one per consumer-end-user within a tenant? Default is one-per-tenant for a single-bookkeeper SMB; multi-user tenants may require finer-grained consents (decide before Phase 1 ships).
2. **Sync cadence vs. cost**: Basiq charges per refresh + per-API-call. 4-hour cadence × 100 tenants × 30 days = 18,000 refreshes/month. Validate against current Basiq pricing before defaulting interval.
3. **Deletion vs. de-identification on revocation**: Australian Tax Office record-keeping rules require business records be retained for 5 years (TAA s262A). Need legal sign-off on whether revoked-consent transactions remain queryable for tax purposes (likely yes — they pre-date revocation and reflect a true business event).
