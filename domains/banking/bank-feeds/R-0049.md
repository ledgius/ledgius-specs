# Summary

Direct bank account connections via aggregator API (Basiq) that automatically import transactions — eliminating manual CSV/OFX file uploads. Transactions flow into the existing reconciliation pipeline (R-0019) without user intervention.

# Business Context

Manual bank statement uploads are the biggest friction point for bookkeepers. Competitors (MYOB, Xero) offer "bank feeds" — direct connections to AU banks that automatically pull transactions daily. This is the single most requested feature for any accounting product.

The Australian market has a clear aggregator: **Basiq** — AU-founded, CDR-compliant (Consumer Data Right), covers 170+ AU financial institutions including all Big Four (CBA, NAB, Westpac, ANZ), plus credit unions and neobanks.

# Acceptance Criteria

## AC-1: Bank account connection

Users can connect their bank accounts from within Ledgius:
- Settings page or Banking section: "Connect Bank Account" button
- Opens Basiq's hosted consent UI (secure, bank-grade, handles MFA)
- User selects their bank, logs in, grants access
- On success: bank account appears in Ledgius linked to a GL account
- Multiple bank accounts can be connected per tenant

## AC-2: Automatic transaction import

Once connected:
- Transactions are fetched automatically every 4-6 hours (configurable)
- New transactions are inserted into `bank_transaction` table with `source = 'feed'`
- Deduplication: transactions have bank-provided unique IDs — no duplicates
- Historical backfill: on first connection, import last 90 days of transactions
- Transactions flow directly into the reconciliation pipeline (R-0019)
- Status badge in header or sidebar shows "Bank feed: synced 2 hours ago"

## AC-3: Transaction data quality

Basiq provides structured data per transaction:
- Transaction ID (unique, stable)
- Date (posting date and transaction date)
- Amount (signed — negative for debits)
- Description (bank-provided narrative)
- Category (Basiq's auto-categorisation — optional, can inform our matching)
- Balance (running balance after transaction)
- Reference / cheque number (when available)
- Counterparty name (when available from NPP/OSKO)

This structured data maps directly to our canonical `bank_transaction` model (R-0019 Section 1), significantly improving matching accuracy vs CSV uploads where data is unstructured.

## AC-4: Connection management

Users can manage their bank feed connections:
- View connected accounts with last sync time and status
- Disconnect an account (stops future syncs, preserves imported data)
- Reconnect if consent expires (banks require re-consent periodically, typically 90 days for CDR)
- Force a manual sync ("Refresh now" button)
- See sync history and any errors

## AC-5: Consent and security

- User credentials are NEVER stored by Ledgius — Basiq handles authentication
- CDR (Consumer Data Right) compliant — user can revoke access at any time
- OAuth 2.0 token exchange between Ledgius API and Basiq
- Basiq API keys stored as Fly.io secrets, never in code
- Connection status and consent expiry tracked per tenant

## AC-6: Error handling and notifications

- If a sync fails (bank outage, consent expired, rate limit): show status in the app, don't silently skip
- If consent is about to expire: show a warning banner "Bank connection expires in 7 days — reconnect to continue automatic imports"
- If consent has expired: show error state on the connection, prompt to reconnect
- Email notification when consent expires (via Resend)

## AC-7: Fallback to manual upload

Bank feeds don't replace manual uploads — they complement them:
- Users without bank feeds continue using Import Transactions (CSV/OFX/QIF)
- Users with bank feeds can still upload files (e.g. for accounts not supported by Basiq)
- Deduplication works across both sources (feed + manual upload)

## AC-8: Pricing tier integration

Bank feeds may be a premium feature:
- Starter plan: manual upload only
- Essential+: 1-3 bank feed connections (per plan limits)
- Business: unlimited connections
- Feature gate: `bank_feeds` slug in the feature system (R-0041)

# Technical Context

## Basiq API

- **Website**: https://basiq.io
- **API Docs**: https://api.basiq.io/reference
- **Coverage**: 170+ AU institutions, CDR-accredited
- **Pricing**: per-connection or per-API-call model
- **Data refresh**: real-time for CDR connections, daily for screen-scrape fallback
- **Consent UI**: hosted widget (Basiq handles bank login, MFA, consent capture)

## Key API flows

### Connection setup
1. Ledgius API creates a Basiq user for the tenant
2. Generates a consent link (Basiq hosted UI)
3. User completes bank login in Basiq widget
4. Basiq notifies Ledgius via webhook: connection established
5. Ledgius fetches account list, user maps bank accounts to GL accounts

### Transaction sync
1. Cron job or webhook trigger: fetch new transactions since last sync
2. `GET /users/{userId}/transactions?filter[account.id]={accountId}&filter[transaction.postDate.from]={lastSync}`
3. Map Basiq transaction to `bank_transaction` model
4. Deduplicate by `external_id` (Basiq transaction ID)
5. Run normalisation pipeline (R-0019 Section 1)
6. Optionally trigger reconciliation pipeline

### Consent management
1. Basiq sends webhook when consent is about to expire
2. Ledgius marks connection as "expiring"
3. User prompted to reconnect via Basiq consent UI

# Out of Scope (Current Release)

- Non-AU banks (Basiq is AU-only; international via Plaid/MX is future)
- Real-time transaction notifications (push from bank — CDR doesn't support this yet)
- PSD2/Open Banking for UK/EU
- Direct bank API integrations (bypassing aggregator — too complex per-bank)
- Transaction enrichment beyond what Basiq provides
- Automatic categorisation using Basiq's ML categories (use our own rules engine)
