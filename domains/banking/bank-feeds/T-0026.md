# Phase 1 — Foundation: schema, Basiq client, provider user lifecycle

**Goal**: A tenant can be associated with a Basiq user; we can authenticate
to Basiq server-to-server.

## T-0026.1: DB migration — `migrations/tenant/018_bank_feeds.sql`

- Tables: `bank_feed_provider_user`, `bank_feed_connection`, `bank_feed_event_log`
- Column: `bank_transaction.source` with default `'manual'`, check constraint
- All schema verbatim from A-0025 §"Migration: 018_bank_feeds.sql"
- Re-entrant (`IF NOT EXISTS`); `COMMENT ON` for every new column and table
- **Verification**: `\d+` shows tables; running migration twice is idempotent

## T-0026.2: Basiq client — `internal/bankfeed/basiq_client.go`

- OAuth2 client_credentials token caching (server token, scope=SERVER_ACCESS)
- Methods: `CreateUser`, `GetUser`, `MintClientToken`, `ConsentURL` (URL builder), `ListAccounts`, `ListConnections`, `RefreshConnection`, `DeleteConnection`, `GetJob`, `GetTransactions`
- `BASIQ_API_KEY` + `BASIQ_BASE_URL` from env
- Structured error type `BasiqError` carrying HTTP status + body for event_log
- File header comment cites `https://api.basiq.io/reference`, `https://api.basiq.io/v2.1/docs/handling-jobs`
- **Verification**: integration test against Basiq sandbox account exercises CreateUser + MintClientToken

## T-0026.3: Event log helpers — `internal/bankfeed/event_log.go`

- `Begin(ctx, eventType, connectionID, payload) (eventID, error)` writes start row
- `Succeed(ctx, eventID, result)` updates end fields
- `Fail(ctx, eventID, err)` updates end fields with error_message + error_payload
- `Partial(ctx, eventID, result, err)` for partial-success cases
- All writes are independent of the caller's transaction (event log must record failed transactions too)
- **Verification**: unit tests cover each lifecycle path; orphan-detection query `WHERE status='running' AND started_at < now() - interval '1 hour'` returns nothing in steady-state

## T-0026.4: Provider user service — `internal/bankfeed/service.go` (initial)

- `EnsureProviderUser(ctx, tenantContactEmail) (*ProviderUser, error)`
  - Look up existing row in `bank_feed_provider_user`
  - If absent: call `BasiqClient.CreateUser`, persist row
- Wired into `cmd/server/main.go`
- **Verification**: API endpoint `POST /api/v1/bank-feeds/connect` returns a Consent URL when called with no existing provider user

# Phase 2 — Connection lifecycle: consent → accounts → mapping

**Goal**: A user can connect their bank, return from the Basiq Consent UI,
have accounts populated, and map them to GL accounts.

## T-0026.5: Connect endpoint — `POST /api/v1/bank-feeds/connect`

- Ensures provider user, mints client token, returns `{consent_url, return_to}`
- Begins `connection.create` event
- Front end redirects to `consent_url`
- **Verification**: `curl POST /connect → 200 with consent_url`

## T-0026.6: Webhook handler — `internal/bankfeed/webhook.go` + signature verification

- HMAC-SHA256 verification per A-0025 §"Webhook handler" — exact recipe
- IP allowlist enforced (config: `BASIQ_WEBHOOK_ALLOWLIST` defaults to documented IPs)
- Begins `webhook.received` event for every inbound call
- Dispatches to handler per Basiq event type
- **Verification**: unit tests with golden signatures; replay rejected; tampered body rejected; correct body verified

## T-0026.7: connections.created handler

- Fetches accounts via `BasiqClient.ListAccounts`
- Inserts `bank_feed_connection` rows with `status='pending'` and `bank_account_id=NULL`
- Persists `consent_expires_at` from connection metadata
- **Verification**: webhook test fixture creates connection rows; mapping endpoint can find them

## T-0026.8: Account mapping endpoint — `POST /api/v1/bank-feeds/connections/:id/map`

- Body: `{bank_account_id}`
- Sets `bank_account_id` on the connection, transitions to `status='active'`
- **Verification**: API + frontend mapping flow works end-to-end against sandbox

# Phase 3 — Sync engine, dedup, scheduler

**Goal**: Transactions flow automatically into `bank_transaction` and into the
reconciliation pipeline (R-0019).

## T-0026.9: Sync core — `internal/bankfeed/sync.go`

- `syncConnection(ctx, conn)`:
  - Begin `transactions.fetch` event
  - Determine `from`: `last_sync_at` or `now() - BANK_FEED_BACKFILL_DAYS`
  - Page through `BasiqClient.GetTransactions` (limit 500)
  - Map per A-0025 §"Transaction mapping"
  - Insert with `ON CONFLICT (account_id, external_id) DO NOTHING` returning insert/conflict counts
  - Update `bank_feed_connection.last_sync_at`, `last_sync_status`, `transactions_synced`
  - End event with counts
- **Verification**: integration test against sandbox imports ≥1 transaction with `source='feed'`; replay imports 0 new (all duplicate)

## T-0026.10: Manual sync endpoint — `POST /connections/:id/sync`

- Triggers `syncConnection` synchronously (with timeout) for the requested connection
- **Verification**: button in UI works

## T-0026.11: Single-leader scheduler — `internal/bankfeed/scheduler.go`

- Background goroutine started in `cmd/server/main.go`
- Acquires `pg_try_advisory_lock(BANK_FEED_SCHEDULER_KEY)` — only the leader runs ticks
- Tick cadence: `BANK_FEED_SYNC_INTERVAL` (default 4h)
- Per-tick: select due connections, dispatch to bounded worker pool
- Releases lock cleanly on shutdown signal
- **Verification**: starting two API instances locally — only one runs ticks (logs confirm); killing leader, surviving instance acquires lock on next tick

# Phase 4 — Consent expiry + notifications

**Goal**: Users are warned 7 days before consent expires and prompted to
reauthorise; expired connections stop syncing.

## T-0026.12: Expiry-check job — `internal/bankfeed/expiry.go`

- Daily cron (single-leader)
- Begins `expiry.check` event
- For each connection with `consent_expires_at < now() + interval '7 days' AND status = 'active'`:
  - Update status → `expiring`
  - Send email via existing `email.Service` (Resend) using a new template
  - Sub-event row per email sent
- For each connection with `consent_expires_at < now() AND status IN ('active', 'expiring')`:
  - Update status → `expired`
  - Send "expired" email
- End event with counts
- **Verification**: time-travel unit test (override `now` injection) confirms transitions; emails landed in Resend test inbox

## T-0026.13: Reauthorise endpoint — `POST /connections/:id/reauthorise`

- Mints client token, returns Consent URL with `action=reauthorise&connectionId={providerConnectionId}`
- Begins `consent.reauthorise` event
- **Verification**: end-to-end flow restores expired connection to active
