# Summary

Comprehensive technical architecture for the bank reconciliation workflow. Covers the canonical data model, five-pass staged matching pipeline, CEL-based scoring model, text normalisation service, precedent memory system, full API contract, three-pane frontend workstation, close-readiness dashboard, and tolerance configuration via the rules engine.

# Requirement Link

- R-0019 (Reconciliation Workflow Requirements)

# Technical Context

Bank reconciliation is a high-throughput, evidence-driven workflow. The architecture must support: (1) a staged multi-pass matching pipeline with explainable scoring; (2) immutable match records with full audit trail; (3) an exception queue with priority and ownership; (4) a precedent memory system that learns from approved matches; (5) a three-pane keyboard-driven workstation enabling a bookkeeper to clear hundreds of lines per session; and (6) period close tracking. All business logic scoring rules are defined in the rules engine (CEL/GoRules per A-0011), not hardcoded in Go.

# Data Model

## Canonical `bank_transaction` Table (Extended)

The following columns are added to the existing `bank_transaction` table. All additions are additive and non-breaking.

| Column | Type | Description |
|--------|------|-------------|
| `value_date` | date | Value date as reported by the bank (may differ from transaction date) |
| `bank_transaction_code` | text | Structured bank transaction code (ISO 20022 where available) |
| `counterparty_name` | text | Raw counterparty name from bank feed |
| `end_to_end_id` | text | End-to-end reference from bank (e.g. BECS reference, SWIFT end-to-end ID) |
| `check_number` | text | Check/cheque number if applicable |
| `deposit_number` | text | Deposit batch number if applicable |
| `channel` | text | Payment channel (bpay, eft, card, direct_debit, etc.) |
| `normalized_description` | text | Output of normalisation pipeline — uppercased, stripped, collapsed |
| `extracted_entities` | jsonb | Invoice numbers, check numbers, order IDs, tax codes, customer IDs extracted by normaliser |
| `duplicate_fingerprint` | text | Hash used for deduplication detection (amount + date + counterparty + channel) |
| `reconciliation_status` | text | State machine: `imported`, `auto_matched`, `needs_review`, `exception`, `resolved`, `closed` |
| `deferred_reason` | text | Reason code if status is deferred |
| `deferred_until` | date | Snooze date for deferred items |
| `exclude_reason` | text | Reason code if excluded |
| `excluded_by` | text | Email of user who excluded the line |
| `confidence_score` | numeric | Overall confidence score from last pipeline run (0–100) |
| `match_pass` | int | Pipeline pass number that produced the match (1–5) |
| `match_explanation` | jsonb | Score component breakdown from matching engine |

### `reconciliation_status` State Machine

```
imported -> auto_matched (pass 1–4 exact match)
imported -> needs_review (pass 5 probabilistic suggestion)
imported -> exception   (no candidate found, duplicate risk, etc.)
needs_review -> resolved (manual match confirmed)
needs_review -> exception (escalated)
exception -> resolved    (exception cleared)
any -> closed            (period lock applied)
resolved -> closed       (period lock applied)
```

## New Tables

### `reconciliation_match`

Immutable record of each match decision. Rows are never updated — a reversal creates a new row with status `reversed` and marks the original as `superseded`.

```sql
CREATE TABLE reconciliation_match (
  id                  bigserial PRIMARY KEY,
  bank_transaction_id bigint       NOT NULL REFERENCES bank_transaction(id),
  match_type          text         NOT NULL CHECK (match_type IN ('one_to_one','one_to_many','many_to_one','generated','split')),
  matched_entry_ids   int[]        NOT NULL,           -- ledger entry IDs linked by this match
  match_pass          int          NOT NULL,           -- which pipeline pass matched (1–5)
  confidence_score    numeric      NOT NULL,
  score_components    jsonb        NOT NULL DEFAULT '{}',
  explanation         text,
  rule_id             text,                            -- rules engine rule ID if rule-based
  precedent_id        bigint       REFERENCES reconciliation_precedent(id),
  matched_by          text,                            -- email; null if auto-matched
  matched_at          timestamptz  NOT NULL DEFAULT now(),
  auto_approved       boolean      NOT NULL DEFAULT false,
  status              text         NOT NULL DEFAULT 'active' CHECK (status IN ('active','reversed','superseded'))
);

CREATE INDEX idx_recon_match_bank_tx ON reconciliation_match(bank_transaction_id);
CREATE INDEX idx_recon_match_status  ON reconciliation_match(status);
```

### `reconciliation_exception`

Managed exception queue with priority, ownership, and comment trail.

```sql
CREATE TABLE reconciliation_exception (
  id                  bigserial PRIMARY KEY,
  bank_transaction_id bigint       NOT NULL REFERENCES bank_transaction(id),
  reason_code         text         NOT NULL CHECK (reason_code IN (
                        'timing_difference','missing_ledger','duplicate_bank',
                        'duplicate_ledger','bank_fee_unposted','transfer_unmatched',
                        'fx_variance','ambiguous_candidate','suspected_fraud')),
  priority            text         NOT NULL CHECK (priority IN ('critical','high','medium','low')),
  owner               text,                            -- assigned user email
  status              text         NOT NULL DEFAULT 'open' CHECK (status IN ('open','investigating','resolved','escalated')),
  comment_trail       jsonb        NOT NULL DEFAULT '[]', -- array of {actor, at, comment}
  evidence_attachments jsonb       NOT NULL DEFAULT '[]',
  snooze_until        timestamptz,
  materiality_amount  numeric,                         -- AUD amount, used to drive priority
  created_at          timestamptz  NOT NULL DEFAULT now(),
  resolved_at         timestamptz,
  resolved_by         text
);

CREATE INDEX idx_recon_exc_status   ON reconciliation_exception(status);
CREATE INDEX idx_recon_exc_priority ON reconciliation_exception(priority);
CREATE INDEX idx_recon_exc_bank_tx  ON reconciliation_exception(bank_transaction_id);
```

**Priority derivation** (evaluated by rules engine on insert/update):

| Reason Code | Base Priority | Escalation Triggers |
|-------------|--------------|---------------------|
| `suspected_fraud` | critical | always |
| `duplicate_bank` | high | amount > materiality threshold |
| `duplicate_ledger` | high | amount > materiality threshold |
| `missing_ledger` | medium | age > 7 days → high |
| `ambiguous_candidate` | medium | amount > threshold → high |
| `bank_fee_unposted` | low | age > 14 days → medium |
| `timing_difference` | low | age > 5 days → medium |
| `transfer_unmatched` | medium | age > 3 days → high |
| `fx_variance` | medium | delta > threshold → high |

# Staged Matching Pipeline Engine

Go service at `internal/banking/pipeline.go`. Executes five passes in sequence. Each pass operates only on bank transactions with `reconciliation_status = 'imported'` or `'needs_review'` and skips items already matched by earlier passes.

## Pass Structure (Common to All Passes)

Before scoring, each pass pre-filters candidates:

1. Same bank account
2. Same currency
3. Compatible sign (debit/credit)
4. Date within configurable window for the pass
5. Amount within configurable band (exact or tolerance)
6. Ledger entry not already reconciled (`reconciliation_status` not `resolved`/`closed`)

After a successful match, the pass:

- Inserts a `reconciliation_match` row with `match_pass = N`
- Updates `bank_transaction.reconciliation_status`
- Updates `bank_transaction.confidence_score` and `match_explanation`
- Emits a `reconciliation_audit_event`

---

## Pass 1 — ExactDeterministicMatcher

Matches on structural identifiers first, then falls back to triple-exact.

```
Step 1a — Structural ID match:
  bank_transaction.external_id         = ledger_entry.source_reference
  OR bank_transaction.end_to_end_id    = ledger_entry.payment_reference
  OR bank_transaction.check_number     = ledger_entry.check_number
  OR bank_transaction.deposit_number   = ledger_entry.deposit_number

Step 1b — Triple exact (fallback within pass 1):
  amount exact
  AND date exact (transaction_date = value_date)
  AND counterparty_id exact
```

Confidence for structural ID match: 100. Confidence for triple exact: 95.
Auto-approved: true. No human review required.

---

## Pass 2 — ToleranceMatcher

Matches on amount + relaxed date + reference similarity.

```
Step 2a — Amount exact + date within tolerance window + counterparty fingerprint match
Step 2b — Amount exact + normalised reference similarity > configured threshold
```

Date tolerance window is per-account, per-channel (see Tolerance Configuration section).
Normalised reference similarity uses a trigram similarity function (`pg_trgm`).

Confidence: computed by scoring model (see CEL Scoring section).
Auto-approved: true only if confidence >= 95 and no competing candidates.

---

## Pass 3 — CompositionMatcher

Handles one-to-many and many-to-one scenarios.

```
One-to-many:
  Find a set of ledger entries where SUM(amount) = bank_transaction.amount
  All entries within date tolerance window
  Maximum composition size: configurable (default 10 entries)

Many-to-one:
  Find a set of bank transactions where SUM(amount) = single ledger entry amount
  All transactions within date tolerance window
  Maximum composition size: configurable (default 5 transactions)
```

Composition search uses a bounded subset-sum algorithm with early exit. Confidence derived from component signals. Always queued for human review unless all component scores are >= 90.

---

## Pass 4 — RuleBasedGenerator

Creates ledger entries automatically from bank line patterns. Rules are defined in the rules engine (GoRules decision tables).

```
Bank fee templates:
  Pattern match on normalized_description (e.g. "ACCOUNT FEE", "MONTHLY FEE")
  -> Auto-create expense journal entry to configured bank fee account
  -> match_type = 'generated'

Interest income templates:
  Pattern match on normalized_description (e.g. "INTEREST CREDIT", "INTEREST PAID")
  -> Auto-create income journal entry to configured interest income account

Internal transfer detection:
  Same organisation owner group
  Opposite sign
  Near date (within 2 days)
  Amount exact or within fx tolerance
  -> Link both sides as transfer pair
  -> match_type = 'generated' on both sides

Card settlement matching:
  Merchant ID or settlement reference present
  -> Match against card expense claims or petty cash entries
```

All generated entries are auto-approved with `match_type = 'generated'`. A separate `generated_journal_ids` array in the audit event records what was created.

---

## Pass 5 — ProbabilisticSuggester

Weighted scoring model defined in CEL rules. Returns ranked candidates with a full explanation payload. **Never auto-posts.** All results are queued for human review with `reconciliation_status = 'needs_review'`.

```
For each unmatched bank_transaction:
  Fetch candidate ledger entries (pre-filtered)
  Score each candidate using CEL scoring formula
  Rank by score descending
  If top candidate score >= review_threshold AND gap to second >= gap_threshold:
    Set status = needs_review, attach top candidate explanation
  Else:
    Set status = exception, reason_code = ambiguous_candidate
  Store score_components in match_explanation JSONB
```

---

# Scoring Model (CEL Rules)

All scoring rules are defined in the rules engine (A-0011), not hardcoded in Go. The following are the canonical initial definitions.

```cel
// Score component weights (additive)
transaction_id_exact:       100
check_number_exact:          95
invoice_reference_exact:     90
exact_amount:                50
date_within_1_day:           20
date_within_3_days:          10
counterparty_fingerprint:    30
memo_similarity_strong:      20   // trigram similarity > 0.8
recurring_template:          15   // matches a known recurring pattern
precedent_boost:             var  // loaded from reconciliation_precedent.confidence_boost

// Hard rejection gates (evaluated before scoring)
sign_mismatch:               REJECT
currency_mismatch:           REJECT
already_reconciled:          REJECT
multiple_tied_candidates:    REQUIRE_REVIEW   // top two candidates within 5 points of each other

// Thresholds
auto_approve_threshold:      95
review_threshold:            60
gap_threshold:               15   // minimum gap between top and second candidate to avoid ambiguity
```

Score components are stored in `score_components` JSONB as `{ "component_name": score_value, ... }` for full explainability. The UI renders these as evidence chips.

# Text Normalisation Service

`internal/banking/normalizer.go`

The normaliser processes `bank_transaction.description` on import and updates `normalized_description` and `extracted_entities`.

## Normalisation Steps (in order)

1. Uppercase all characters
2. Remove punctuation (except hyphens and forward slashes used as separators)
3. Collapse multiple whitespace to single space
4. Trim leading and trailing whitespace
5. Map common abbreviations:

   | Abbreviation | Expansion |
   |-------------|-----------|
   | PYMT | PAYMENT |
   | PMT | PAYMENT |
   | TFR | TRANSFER |
   | XFER | TRANSFER |
   | REC | RECEIPT |
   | INV | INVOICE |
   | DD | DIRECT DEBIT |
   | SA | SAVINGS ACCOUNT |
   | CR | CREDIT |
   | DR | DEBIT |
   | CHQ | CHEQUE |
   | DEP | DEPOSIT |
   | INT | INTEREST |
   | FEE | FEE |
   | BPAY | BPAY |

6. Strip known bank prefixes/suffixes (configurable list per bank feed provider):
   - Prefixes: `VISA PURCHASE`, `EFTPOS`, `DIRECT CREDIT`, `DIRECT DEBIT`, `BANK TRANSFER`, `OSKO`, `NPP`
   - Suffixes: terminal IDs, batch numbers, date stamps appended by bank
7. Apply vendor alias dictionary sourced from `reconciliation_precedent` rows with `pattern_type = 'counterparty_mapping'`

## Extraction Pipeline

After normalisation, run regex patterns to populate `extracted_entities` JSONB:

```go
// Extracted entity schema (JSONB)
{
  "invoice_numbers": ["INV-0042", "INV-1234"],
  "check_numbers":   ["000123"],
  "order_ids":       ["ORD-9876"],
  "tax_codes":       ["ABN12345678901"],
  "customer_ids":    ["CUST-001"],
  "settlement_ids":  ["SETTLE-2026-04-07-001"]
}
```

Patterns:
- Invoice: `INV[-/]?\d+`, `INVOICE[-/]?\d+`, `TAX INV \d+`
- Check: `CHQ\s*\d+`, `CHEQUE\s*\d+`, `CHK\s*\d+`
- Order: `ORD[-/]?\d+`, `ORDER[-/]?\d+`, `PO[-/]?\d+`
- ABN: `\bABN\s*\d{2}\s*\d{3}\s*\d{3}\s*\d{3}\b`
- Customer: `CUST[-/]?\d+`, `CLIENT[-/]?\d+`

All extracted values are stored as arrays to handle multiple occurrences in a single description. Extracted values feed directly into Pass 1 and Pass 2 candidate matching.

# Precedent Memory Service

`internal/banking/precedent.go`

## On Manual Match Approval

When a user confirms a manual match:

1. Compute `normalized_pattern` from the bank transaction's `normalized_description`
2. Determine `pattern_type`:
   - If the match linked to a specific account with no entity: `classification`
   - If the match linked to a specific contact/vendor: `counterparty_mapping`
   - If the match was rule-triggered: `match_rule`
3. Check if a matching row exists in `reconciliation_precedent` (by `pattern_type` + `normalized_pattern`):
   - If yes: increment `approval_count`, update `last_approved_at`, recalculate `confidence_boost`
   - If no: insert new precedent row
4. Confidence boost formula: `min(30, approval_count * 3)` — capped at 30 points

## Precedent Visibility

Precedents are surfaced in the UI in the `DetailInspector` pane under "Historical Matches". The API endpoint `GET /reconciliation/precedents?account_id=` returns the precedent table for transparency, allowing bookkeepers to audit what the system has learned.

## Precedent Application in Pass 5

Before scoring in the probabilistic pass, load applicable precedents for the bank account. For each candidate, check if the normalised description matches a precedent pattern and apply `confidence_boost` to the score.

# API Endpoints

All endpoints are under `/api/v1` prefix, require authenticated session, and enforce RBAC per A-0011.

```
GET  /reconciliation/queue
     ?account_id=<id>
     &sort=risk|aged|amount
     &filter=unmatched|needs_review|exception|all
     &date_from=&date_to=
     &amount_min=&amount_max=
     Paginated. Returns bank_transaction rows with status, confidence, match_explanation.

GET  /reconciliation/lines/:id/candidates
     Returns ranked candidate ledger entries with score_components and evidence chips.

POST /reconciliation/lines/:id/match
     Body: { match_type, matched_entry_ids[], override_reason? }
     Commits a one-to-one or one-to-many match.

POST /reconciliation/lines/:id/split
     Body: { allocations: [{ account_id, amount, tax_code_id, entity_id?, notes? }] }
     Invariant: sum(allocations.amount) == bank_transaction.amount
     Creates ledger entries per allocation and links to bank line.

POST /reconciliation/lines/:id/create
     Body: { transaction_subtype, account_id, tax_code_id, contact_id?, reference?, notes? }
     Creates a new ledger entry from the bank line and links it.

POST /reconciliation/lines/:id/transfer
     Body: { from_account_id, to_account_id, notes? }
     Detects and warns on duplicate transfers. Creates transfer pair.

POST /reconciliation/lines/:id/defer
     Body: { reason_code, snooze_until?, notes? }
     Sets status to exception with deferred reason. Creates exception queue entry.

POST /reconciliation/lines/:id/exclude
     Body: { reason_code, notes }
     Requires elevated RBAC role. Sets status to excluded with audit trail.

POST /reconciliation/lines/:id/undo
     Reverses the last action. Sets previous reconciliation_match to status=reversed.
     Creates a new audit event with event_type=undo.

POST /reconciliation/bulk/accept
     Body: { bank_transaction_ids[] }
     Preview mode: returns list of matches that would be accepted.
     Confirm mode: bulk-accepts exact matches only (confidence >= 95, no conflicts).
     Each accepted match emits its own audit event with event_type=bulk_accept.

GET  /reconciliation/exceptions
     ?account_id=&status=open|investigating|escalated&priority=critical|high|medium|low
     Returns reconciliation_exception queue with linked bank_transaction data.

PATCH /reconciliation/exceptions/:id
     Body: { status?, owner?, comment?, snooze_until? }
     Updates exception status, owner, or appends to comment_trail.

GET  /reconciliation/lines/:id/audit
     Returns reconciliation_audit_event rows for a bank transaction, newest first.

GET  /reconciliation/summary
     ?account_id=&period_start=&period_end=
     Returns close-readiness metrics (see Close-Readiness Dashboard section).

GET  /reconciliation/precedents
     ?account_id=&pattern_type=
     Returns reconciliation_precedent rows visible to bookkeeper.

POST /reconciliation/run-pipeline
     ?account_id=
     Triggers five-pass pipeline for the account. Returns a job ID.
     Pipeline runs asynchronously; progress observable via summary endpoint.
```

## Close Conditions

A period can be closed only when all of the following are true:

- `unreconciled_count == 0`
- `balance_difference == 0` (or within configured rounding tolerance)
- No `reconciliation_exception` rows with `status IN ('open', 'investigating')` and `priority IN ('critical', 'high')`
- No `bank_transaction` rows with `reconciliation_status IN ('imported', 'needs_review', 'exception')` for the period date range

Closing a period sets `reconciliation_period.status = 'closed'` and transitions all resolved bank transactions to `reconciliation_status = 'closed'`.

# Tolerance Configuration

Stored in the rules engine as GoRules decision tables (per A-0011). Tolerance rules are per bank account, per payment method, and per transaction class. The pipeline reads tolerance rules at startup and refreshes on configuration change events.

## Decision Table Structure

| Bank Account | Payment Method | Transaction Class | Date Tolerance (days) | Amount Tolerance | Notes |
|-------------|---------------|-------------------|----------------------|-----------------|-------|
| * | payroll | * | 0 | exact | Payroll must match exactly |
| * | eft | receipt | 1 | exact | EFT receipts: 1 day tolerance |
| * | card | settlement | 3 | configurable_threshold | Card settlements: 3 day tolerance |
| * | bpay | * | 1 | exact | BPAY: 1 day, exact amount |
| * | direct_debit | * | 1 | exact | Direct debit: 1 day, exact |
| * | * | tax_payment | 0 | exact | Tax payments: always exact |
| * | * | * | 3 | exact | Default: 3 day tolerance, exact amount |

`configurable_threshold` is a per-organisation setting stored in organisation preferences, defaulting to 0.50 AUD.

# Data Flow

```
Bank statement imported (via import pipeline A-0017 or manual upload)
  -> bank_transaction rows created with reconciliation_status = 'imported'
  -> Normaliser runs on each row: populates normalized_description, extracted_entities
  -> Duplicate fingerprint computed and checked against existing rows

Pipeline triggered (POST /reconciliation/run-pipeline or on-import hook):
  -> Pass 1 ExactDeterministicMatcher runs: structural ID and triple-exact
  -> Pass 2 ToleranceMatcher runs: amount + date window + reference similarity
  -> Pass 3 CompositionMatcher runs: subset-sum one-to-many / many-to-one
  -> Pass 4 RuleBasedGenerator runs: fee templates, interest, transfers, card settlements
  -> Pass 5 ProbabilisticSuggester runs: scored candidates queued for review
  -> reconciliation_match rows inserted for all auto-matches
  -> reconciliation_exception rows inserted for unresolved lines
  -> reconciliation_audit_event rows emitted for every decision

User opens reconciliation workstation:
  -> BankLineQueue loads (paginated, filtered, risk-sorted)
  -> User selects a bank line (keyboard j/k or click)
  -> CandidatePanel fetches GET /reconciliation/lines/:id/candidates
  -> User reviews evidence chips and confidence scores
  -> User presses a (accept), s (split), c (create), t (transfer), d (defer), x (exclude)
  -> Action configured in CandidatePanel / ActionPanel
  -> User presses Enter to confirm
  -> POST to relevant endpoint
  -> reconciliation_audit_event emitted
  -> If manual match: precedent service checks for new/reinforced pattern
  -> Queue advances to next line (j key)

Period close:
  -> User reviews close-readiness summary card
  -> Resolves all blockers (unreconciled items, exceptions, balance difference)
  -> Clicks "Close Period"
  -> Server validates close conditions
  -> reconciliation_period.status set to 'closed'
  -> All resolved bank_transactions transitioned to 'closed'
```

---

# Background Processing

- **On-import normalisation**: Normaliser runs synchronously as part of the import pipeline for each bank_transaction row. Acceptable latency at import volumes.
- **Pipeline execution**: The five-pass pipeline runs asynchronously when triggered. Progress is observable via the summary endpoint.
- **Nightly re-scoring job**: A scheduled job re-evaluates stale deferred items (snooze_until has passed), escalates aged exceptions by priority rules, and refreshes `reconciliation_period` aggregate counts.
- **Duplicate fingerprint check**: Checked synchronously on import. Duplicate rows are flagged with `reconciliation_status = 'exception'` and `reason_code = 'duplicate_bank'` immediately.

# Risks and Trade-offs

| Risk | Severity | Mitigation |
|------|----------|-----------|
| Candidate scoring performance at scale | Medium | Pre-filter candidates by date window and amount band before scoring. Index on `(bank_account_id, reconciliation_status, transaction_date)`. Consider pre-computation for large accounts. |
| Subset-sum composition search at scale | Medium | Bound composition size (default 10 entries). Apply early-exit when partial sum exceeds target. Cap search window by date tolerance. |
| Stale candidates after partial reconciliation | Low | Pre-filter excludes already-reconciled entries. Pass 5 re-evaluates only `imported`/`needs_review` items. |
| Precedent overfitting | Low | Confidence boost capped at 30 points. Precedents visible to user for transparency and deletion. |
| Bulk accept misuse | Medium | Strict eligibility: confidence >= 95, no competing candidates, no conflict flags. Each bulk-accepted match emits its own audit event. Preview required before confirm. |
| Period lock conflict | Low | Close conditions enforced server-side. UI surfaces all blockers in close-readiness summary before allowing close action. |

---

# Alternatives Considered

1. **Automatic reconciliation without human review**: Rejected. Accounting reconciliation requires defensible evidence. Auto-matching (passes 1–4) is permitted for high-confidence exact matches, but pass 5 always requires human confirmation.
2. **Separate pages per action mode**: Rejected. The three-pane workstation with keyboard navigation is significantly faster for high-volume reconciliation.
3. **Background candidate pre-computation**: Considered for performance but deferred. Adds staleness risk as other lines are reconciled. Will revisit if performance profiling shows it is necessary.
4. **Hardcoded scoring weights**: Rejected. All scoring rules defined in CEL via the rules engine (A-0011) so weights can be tuned per organisation without code changes.
5. **Single-pass matching**: Rejected. A single scoring pass cannot cleanly separate deterministic structural matches (which should auto-approve) from probabilistic suggestions (which require review). Five passes with increasing uncertainty maps cleanly to the auto-approval vs review boundary.

# Test Strategy

- Unit tests for each pipeline pass with representative fixtures (exact match, tolerance match, composition, generated, probabilistic)
- Unit tests for normaliser: abbreviation mapping, prefix stripping, entity extraction regexes
- Unit tests for precedent service: new precedent creation, approval count increment, confidence boost formula
- Unit tests for CEL scoring model: verify each component fires correctly, rejection gates reject correctly
- Integration tests for each API endpoint: valid commit, invalid body, RBAC enforcement, audit event emission
- Integration tests for composition search: one-to-many with 3 entries, many-to-one with 2 bank lines
- Integration tests for bulk accept: preview mode, confirm mode, ineligible items excluded
- Integration tests for period close: all close conditions met, blocked when exceptions open
- End-to-end test: import bank statement → run pipeline → reconcile via all action modes → close period → verify audit trail
- Performance test: queue loading with 1,000+ unreconciled lines; pipeline run on 500 transactions

# Rollout and Migration Notes

New feature, additive only. Required database changes:

1. Extend `bank_transaction` with new columns (all nullable or with defaults, non-breaking)
2. Create `reconciliation_match`, `reconciliation_exception`, `reconciliation_precedent`, `reconciliation_audit_event`, `reconciliation_period` tables
3. Existing `bank_transaction` rows default to `reconciliation_status = 'imported'`
4. No migration of existing reconciliation data required — the pipeline will process existing unreconciled rows on first run

Feature flag `FEATURE_RECONCILIATION_V2` gates the new three-pane workstation and pipeline API until stable.
