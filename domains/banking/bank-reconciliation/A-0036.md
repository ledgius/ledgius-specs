# Summary

Architecture for the v2 bank reconciliation system: a full-width
transaction list with in-situ expansion, split allocations, rule
engine, and AI-assisted smart reconciliation. Replaces the current
compressed layout with a MYOB-exceeding workflow.

# Requirement Link

- R-0065 (Bank Reconciliation v2)

# Technical Context

Bank reconciliation is the most frequently used page in any accounting
system. The current Ledgius implementation was built as an MVP and
compresses transaction details, uses a side panel, and lacks rule-based
matching. Competitors (MYOB 7/10, Xero 8/10) have mature reconciliation
workflows with inline editing, split allocations, and smart matching.
This architecture brings Ledgius to 9/10.

## Data Model

### Existing tables (used as-is)
- `bank_feed_line` — raw bank transactions from feed/import
- `account` — chart of accounts (bank accounts have category A)
- `acc_trans` — GL entries (reconciliation creates these)
- `transactions` — parent transaction records

### New tables

#### `recon_rule`
```sql
CREATE TABLE recon_rule (
    id              SERIAL PRIMARY KEY,
    tenant_id       INT NOT NULL,           -- tenant scope
    name            TEXT NOT NULL,
    match_pattern   TEXT NOT NULL,       -- regex or LIKE pattern
    match_type      TEXT NOT NULL DEFAULT 'contains',  -- contains, exact, regex
    default_account_id INT REFERENCES account(id),
    default_tax_code_id INT REFERENCES tax_code(id),
    default_contact_id INT,
    auto_apply      BOOLEAN NOT NULL DEFAULT false,
    enabled         BOOLEAN NOT NULL DEFAULT true,
    priority        INT NOT NULL DEFAULT 0, -- lower = higher precedence
    direction       TEXT NOT NULL DEFAULT 'any',  -- withdrawal, deposit, any
    amount_match_type TEXT NOT NULL DEFAULT 'any', -- any, exact, set, range
    amount_match_value JSONB,               -- {value: 100} or {values: [38,43,68]} or {min: 40, max: 500}
    is_split        BOOLEAN NOT NULL DEFAULT false,
    split_lines     JSONB,              -- [{account_id, percentage, description, tax_code_id}]
    confidence      NUMERIC(3,2) DEFAULT 1.00,
    source          TEXT DEFAULT 'manual', -- manual, smart
    use_count       INT DEFAULT 0,
    rejection_count INT DEFAULT 0,
    name_override   TEXT,                   -- user-corrected meaningful name
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    created_by      TEXT,
    updated_by      TEXT
);
```

#### `recon_allocation`
```sql
CREATE TABLE recon_allocation (
    id                  SERIAL PRIMARY KEY,
    bank_feed_line_id   INT NOT NULL,
    allocation_method   TEXT NOT NULL,       -- rule_match, manual_allocation, linked_auto, linked_manual, excluded, deferred
    workflow_status     TEXT NOT NULL DEFAULT 'proposed',  -- proposed, approved
    rule_id             INT REFERENCES recon_rule(id),
    total_amount        NUMERIC NOT NULL,
    transfer_target_account_id INT REFERENCES account(id),
    counterpart_bank_feed_line_id INT,      -- for transfers: the other side
    transfer_status     TEXT,               -- pending_counterpart, matched, approved
    allocated_by        TEXT,
    allocated_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
    approved_by         TEXT,
    approved_at         TIMESTAMPTZ,
    reversed_by         TEXT,
    reversed_at         TIMESTAMPTZ
);

-- Unique: only one active (proposed/approved) allocation per bank_feed_line
CREATE UNIQUE INDEX idx_recon_allocation_active 
    ON recon_allocation (bank_feed_line_id) 
    WHERE workflow_status IN ('proposed', 'approved');
```

CREATE TABLE smart_recon_rejection (

#### `smart_recon_stats` (view)
```sql
-- Reporting view: smart reconciliation effectiveness
CREATE VIEW smart_recon_stats AS
SELECT
    r.id AS rule_id,
    r.name,
    r.source,
    r.use_count AS confirmations,
    r.rejection_count AS rejections,
    CASE WHEN (r.use_count + r.rejection_count) > 0
         THEN ROUND(r.use_count::numeric / (r.use_count + r.rejection_count), 2)
         ELSE NULL END AS acceptance_rate,
    r.confidence
FROM recon_rule r
WHERE r.source = 'smart';
```

## Booking Sandbox Architecture

The reconciliation system operates as a staging sandbox — all allocation
operations are metadata-only until explicitly approved by the user.

### Propose Phase (No GL Impact)
- User creates allocations (manual, rule, link, transfer)
- `recon_allocation` rows created with `workflow_status = 'proposed'`
- `bank_feed_line.workflow_status` updated to `'proposed'`
- `bank_feed_line.allocation_method` set accordingly
- NO `acc_trans` lines created
- NO `transactions` records created
- NO `gl` entries created
- User can freely modify, delete, and redo allocations

### Approve Phase (GL Commit)
- User reviews proposed allocations in Review mode
- On approval (individual or bulk):
  1. For each approved allocation, create journal entries:
     - `transactions` record (type = 'gl', approved = true)
     - `gl` entry header
     - `acc_trans` debit line(s) (category/expense/income accounts)
     - `acc_trans` credit line (bank account)
     - Tax component lines (if applicable, per REC-055)
  2. Update `recon_allocation.workflow_status` = 'approved'
  3. Update `recon_allocation.approved_by` and `approved_at`
  4. Update `bank_feed_line.workflow_status` = 'approved'
  5. Emit audit event with full allocation details
- Once approved, GL entries are locked (REC-047)

### Reversal
- Accountant-only action (REC-047 security)
- Creates reversal journal entries (opposite debit/credit)
- Sets `bank_feed_line.workflow_status` = 'unallocated'
- Clears `bank_feed_line.allocation_method`
- Transaction available for re-allocation

### Accounting Execution Matrix

Approval branches by `allocation_method`:

| Method | On Approve | Tables Written |
|---|---|---|
| `manual_allocation` | Create new journal (transactions + gl + acc_trans) | New rows |
| `rule_match` | Create new journal (same as manual) | New rows |
| `linked_auto` / `linked_manual` | Reconcile existing acc_trans — set source reference, no new journal | Update existing |
| `transfer` | Create transfer journal (debit target bank, credit source bank), approve both sides | New rows |
| `excluded` | Finalise state only | None |

This mirrors the import staging pipeline: validate → stage → review → commit.

## Smart Reconciliation Engine

### Architecture

```
┌────────────────────────────────────────────┐
│         Smart Reconciliation Engine         │
│                                            │
│  1. Meaningful Name Extractor              │
│     - Tokenize transaction description     │
│     - Remove noise (dates, refs, ~card)    │
│     - Identify merchant/business tokens    │
│     - Produce normalized "meaningful name" │
│                                            │
│  2. Historical Pattern Matcher             │
│     - Query past recon_allocation rows     │
│     - Group by meaningful name             │
│     - Score by frequency + recency         │
│     - Weight manual allocations higher     │
│                                            │
│  3. Rule Generator                         │
│     - From high-confidence patterns:       │
│       create suggested recon_rule          │
│     - Trial-match against unmatched txns   │
│     - Tag as source='smart'                │
│                                            │
│  4. Suggestion Presenter                   │
│     - Return confidence-scored matches     │
│     - High (≥0.85): green badge            │
│     - Medium (0.60-0.84): amber badge      │
│     - Low (<0.60): gray badge              │
│     - All require human confirmation       │
└────────────────────────────────────────────┘
```

### Meaningful Name Extraction Algorithm

```
Input: "Reddy Express 6808 Clayton 15/02 ~774924"

Step 1: Tokenize
  ["Reddy", "Express", "6808", "Clayton", "15/02", "~774924"]

Step 2: Remove noise tokens
  - Dates (DD/MM pattern): remove "15/02"
  - Card suffix (~NNNNNN): extract "~774924" as secondary signal, remove
  - Pure numbers > 4 digits: remove "6808" (likely reference)
    
Step 3: Remaining merchant tokens
  ["Reddy", "Express", "Clayton"]

Step 4: Normalize
  meaningful_name = "reddy express clayton"
  card_suffix = "774924" (secondary signal)

Step 5: Lookup
  - Search recon_allocation history for meaningful_name
  - If found with same allocation → high confidence suggestion
```

### Confidence Scoring

| Signal | Weight |
|---|---|
| Exact meaningful name match to prior allocation | +0.40 |
| Same amount as prior allocation | +0.20 |
| Same category/account in prior allocation | +0.15 |
| Multiple prior allocations (frequency) | +0.10 per occurrence (max +0.20) |
| Same card suffix across transactions | +0.05 |
| Recency (allocation within 90 days) | +0.05 |
| **Maximum possible** | **1.00** |

Threshold for auto-suggestion: ≥ 0.60
Threshold for "high confidence" badge: ≥ 0.85

| Method | Path | Purpose |
|---|---|---|
| POST | `/bank-reconciliation/smart-reconcile` | Run smart reconciliation on unmatched transactions |

| POST | `/bank-reconciliation/smart-reconcile/reject` | Reject a suggestion |

### Batch Response Contract

All batch endpoints (`/bulk/allocate`, `/bulk/approve`) return a
structured result:

```json
{
  "succeeded": 72,
  "failed": 3,
  "skipped": 1,
  "items": [
    {"bank_feed_line_id": 123, "status": "succeeded"},
    {"bank_feed_line_id": 456, "status": "failed", "reason": "already approved by another user"},
    {"bank_feed_line_id": 789, "status": "skipped", "reason": "workflow_status is not 'proposed'"}
  ]
}
```

Each line is processed in its own DB transaction (partial-success
semantics per REC-070).

- Confidence scoring
- Suggestion UI with confirm/reject

# Risks / Trade-offs

- **Smart reconciliation accuracy**: Pattern matching without ML has
  limits. Mitigation: conservative confidence thresholds + mandatory
  human confirmation. Accuracy improves over time as more allocations
  are confirmed.
- **Performance on large datasets**: Some businesses have 1000+ bank
  lines per month. Mitigation: pagination, virtual scrolling, server-
  side filtering.
- **Rule conflicts**: Multiple rules may match the same transaction.
  Mitigation: highest-confidence rule wins; user can override.

# Test Strategy

- Integration tests for split allocation (amounts must balance)
