# Summary

A complete redesign of the bank reconciliation workflow to match or
exceed competitor capabilities (MYOB scores 7/10; current Ledgius
scores 2/10). The new design introduces a full-width transaction
list with in-situ expansion, split allocations, sortable columns,
bank account labelling, rule management, and an AI-assisted smart
reconciliation engine that suggests matches from historical patterns.

# Problem / Business Context

Bank reconciliation is the highest-frequency bookkeeping task ŌĆö every
bank transaction must be matched to a ledger entry or categorised.
The current Ledgius reconciliation page has critical UX issues:

1. Transaction details are compressed into 3 rows on the left, wasting
   horizontal space
2. No inline expansion ŌĆö users must navigate to a separate detail panel
3. Numbers are bolded (visually heavy)
4. The right-side detail panel is unnecessary when details can expand
   in-situ
5. Columns are not sortable
6. Bank accounts lack descriptive labels
7. No split allocation capability
8. No rule-based auto-matching
9. No smart reconciliation / AI-assisted suggestions

Competitors (MYOB, Xero) provide inline expansion, split allocations,
rule creation from transactions, and smart/auto-matching. Ledgius must
match and exceed this standard.

# Scope

## Transaction List (UX Overhaul)

- **REC-001**: Transaction lines shall span the full page width and
  only wrap text if the browser window is narrower than the content.
- **REC-002**: Each transaction line shall be a single row displaying:
  Date, Bank Transaction (description), Withdrawal ($), Deposit ($),
  Match status, Tax code.
- **REC-003**: Numbers shall NOT be bold ŌĆö use `font-normal tabular-nums`.
- **REC-004**: All columns shall be sortable ascending/descending by
  clicking the column header. Sort indicator (Ō¢▓/Ō¢╝) shall be visible.
- **REC-005**: Clicking a transaction row shall expand it in-situ
  (below the row, pushing subsequent rows down) revealing three
  collapsible sections: Rule/Match, Actions, Audit.

## In-Situ Expansion Panel

- **REC-006**: The expanded area shall be a single unified panel that
  opens directly below the selected row. Inside this panel there are
  three sections arranged vertically:
  - **Rule/Match** (open by default): shows matched rule details or
    "No rule matched" with an option to search/create a rule. Includes
    the colour-coded meaningful name extraction (REC-028a) when smart
    reconciliation is active.
  - **Actions**: the available operations for this transaction:
    - Match to existing record (invoice, bill, journal)
    - Select categories (create new GL entry with category/account)
    - Transfer money (between bank accounts)
  - **Audit** (collapsed by default, expandable): shows who allocated,
    when, revision history, any rule applied.
  All three sections are always visible within the one panel (not
  separate tabs ŌĆö they stack vertically with section headers).
- **REC-007**: The right-side detail panel shall be removed. All
  detail is in the expanded panel.

## Cross-Transaction Rule Match Badges

- **REC-007a**: When a rule is selected or matched for the currently
  expanded transaction, the system shall immediately scan ALL other
  visible transactions in the list for matches against the same rule.
- **REC-007b**: Transactions that also match the rule shall display
  a **"pending match" badge** ŌĆö visually distinct from the existing
  "already matched" badge:
  - **Already matched** (confirmed): solid green badge with check Ō£ō
  - **Pending match** (will be matched if rule is confirmed): pulsing
    amber/orange badge with a dot ŌŚÅ, indicating "this transaction
    will also be matched when you confirm this rule"
- **REC-007c**: Clicking a pending-match badge shall expand that row
  and show the pre-filled allocation from the rule, ready for
  confirmation.
- **REC-007d**: When the user confirms/saves the rule on the original
  transaction, all pending-match transactions shall be presented in a
  confirmation dialog: "This rule also matches N other transactions.
  Apply to all? [Review each] [Apply all] [Skip others]".

## Bank Account Header

- **REC-008**: The bank account selector shall display the account
  name with a descriptive label (e.g. "Nursery Bank Account" with
  the bank logo/icon and BSB/account number if available).
- **REC-009**: The header shall show: bank balance, book balance,
  and the difference (variance) prominently.
- **REC-010**: The header shall show when the bank data was last
  updated (from bank feed or last import).

## Filter Tabs

- **REC-011**: Transaction list shall have filter tabs with counts:
  "All transactions", "Not matched (N)", "Matched".
- **REC-012**: Date range filter shall be available alongside the tabs.
- **REC-013**: A search field shall filter transactions by description
  text (case-insensitive, partial match).

## Batch Operations

- **REC-014**: Each transaction row shall have a checkbox for
  multi-select.
- **REC-015**: A batch action bar shall appear when transactions are
  selected showing the count and available actions.
- **REC-016**: Maximum batch size: 50 transactions per operation.

### Batch-Safe Actions

- **REC-066**: The following actions are batch-safe (can be applied
  to multiple selected transactions at once):
  - **Approve selected**: transitions all selected `allocated`
    transactions to `approved`. Only `allocated` lines are eligible;
    lines in other states are silently skipped.
  - **Confirm suggestions**: transitions all selected `suggested`
    transactions to `allocated` using their suggested rule. Only
    high-confidence suggestions (Ōēź0.85) are eligible for batch confirm.
  - **Clear selection**: deselects all checkboxes (no data change).
- **REC-067**: The following actions are NOT batch-safe and must be
  performed one transaction at a time:
  - **Categorise** (requires per-transaction account selection)
  - **Match to existing record** (requires per-transaction GL match)
  - **Transfer** (requires per-transaction target account selection)
  - **Split allocation** (requires per-transaction line entry)

### Batch Execution Semantics

- **REC-068**: Batch operations are **partial-success** ŌĆö each
  transaction in the batch is processed independently. If one
  transaction fails (e.g. an `allocated` line was reversed by another
  user between selection and approval), the remaining transactions
  continue processing.
- **REC-069**: After a batch operation completes, the UI shall show
  a result summary:
  - Ō£ō N transactions approved successfully
  - Ō£Ś M transactions failed (with per-row reason)
  - ŌŚŗ K transactions skipped (wrong state)
  Each failed/skipped row shall be highlighted in the transaction
  list with a brief inline error message visible without expansion.
- **REC-070**: Batch operations shall execute within a single HTTP
  request. The backend processes each line in its own DB transaction
  (not one giant transaction) so that partial success is possible
  and a single failure doesn't roll back the entire batch.

- **REC-044**: Each bank feed line shall carry a `recon_status` field
  with the following states and transitions:

| State | Meaning | Permitted transitions |
|---|---|---|
| `unmatched` | Imported from bank, no allocation exists | ŌåÆ `suggested`, ŌåÆ `allocated` |
| `suggested` | Smart reconciliation proposed a match (not yet confirmed by user) | ŌåÆ `allocated` (user confirms), ŌåÆ `unmatched` (user rejects) |
| `allocated` | User has created an allocation (single or split). GL entries created but not yet approved/finalised. | ŌåÆ `approved`, ŌåÆ `unmatched` (user reverses) |
| `approved` | Allocation reviewed and finalised. GL entries are committed and locked. | ŌåÆ `reversed` |
| `reversed` | A previously approved allocation was undone. Reversal GL entries created. | ŌåÆ `unmatched` (available for re-allocation) |

- **REC-045**: The filter tabs shall map to these states:
  - "Not matched" = `unmatched` + `suggested`
  - "Matched" = `allocated` + `approved`
  - "All" = all states except `reversed` (reversed are hidden by
    default, visible via a "Show reversed" toggle)
- **REC-046**: A `suggested` line shall NOT have GL entries. GL entries
  are only created when transitioning to `allocated`.
- **REC-047**: An `approved` line's GL entries are locked and cannot
  be edited. To change an approved allocation, the user must first
  reverse it (creating reversal entries) which returns the line to
  `unmatched`.

## Action Accounting Semantics

The three actions in the expanded row produce different accounting
results. Each is defined precisely:

### Match to Existing Record

- **REC-048**: "Match to existing record" links the bank line to a
  pre-existing unreconciled GL entry (e.g. an invoice payment, a bill
  payment, or a journal entry that is already in `acc_trans`).
- **REC-049**: Matching does NOT create new GL entries. It marks the
  existing `acc_trans` row(s) as reconciled by setting a reconciliation
  reference on the `acc_trans.source` field linking it to the bank
  feed line.
- **REC-050**: The match requires that the bank line amount equals
  the GL entry amount (or the sum of multiple selected GL entries for
  a composite match). A tolerance of ┬▒$0.01 is permitted for rounding.
- **REC-051**: Matching is only available when there are unreconciled
  GL entries on the same bank account that match the amount and
  approximate date range.

### Select Categories (Create New GL Entry)

- **REC-052**: "Select categories" creates a NEW GL journal entry
  to categorise the bank transaction. This is for transactions that
  have no pre-existing ledger entry (e.g. a bank fee, a cash purchase,
  an ad-hoc expense).
- **REC-053**: The GL entry created shall be:
  - **Withdrawal**: Debit the selected expense/asset account, Credit
    the bank account.
  - **Deposit**: Debit the bank account, Credit the selected income/
    liability account.
- **REC-054**: Each allocation line in the GL entry carries:
  - Account (required ŌĆö must be a non-bank account)
  - Amount (required)
  - Tax code (required ŌĆö drives BAS reporting)
  - Description (optional)
  - Contact (optional ŌĆö links to a customer/vendor)
- **REC-055**: Tax behaviour: amounts entered are **tax-inclusive**
  by default. The system calculates the tax component from the tax
  code rate and creates the tax GL entry automatically:
  - GST (10%): $110 entered ŌåÆ $100 to expense + $10 to GST account
  - FRE / N-T: $110 entered ŌåÆ $110 to expense, $0 tax

### Transfer Money

- **REC-056**: "Transfer money" records a movement between two bank
  accounts (e.g. transfer from cheque to savings, credit card payment).
- **REC-057**: Transfer requires selecting a **target bank account**
  (must be another account with `category = 'A'` and a bank account
  type). P&L accounts (income/expense) are NOT selectable as transfer
  targets.
- **REC-058**: The GL entry created shall be:
  - Debit the target bank account, Credit the source bank account
    (for a withdrawal from source).
  - Or: Debit the source bank account, Credit the target bank account
    (for a deposit into source).
- **REC-059**: No tax code is required for transfers (bank-to-bank
  movements are not GST-reportable).

## Split Allocations

- **REC-017**: When categorising a transaction, the user shall be able
  to split the amount across multiple GL accounts (split allocation).
- **REC-018**: Each split line shall have: Category (account), Amount
  ($), Amount (%), Line description, Tax code.
- **REC-019**: The split view shall show Subtotal and Out of Balance
  amounts. The total of all split lines must equal the transaction
  amount before saving.
- **REC-020**: Splits shall support 2ŌĆō20 lines.

### Split Allocation Behaviour Rules

- **REC-060**: Splits are **amount-driven by default**. The user enters
  dollar amounts on each line. Percentages are calculated and displayed
  automatically as a read-only convenience (`line_amount / total * 100`).
- **REC-061**: The user may switch to **percentage-driven** mode by
  clicking a toggle on the split grid header. In percentage mode, the
  user enters percentages and dollar amounts are calculated from the
  transaction total. Only one mode is active at a time ŌĆö mixed entry
  (some lines by amount, some by percentage) is NOT permitted.
- **REC-062**: **Rounding**: when calculating amounts from percentages,
  the system uses banker's rounding (round half to even) to 2 decimal
  places. Any rounding residual (┬▒$0.01) is applied to the last line
  to ensure the total balances exactly.
- **REC-063**: **Tax handling in splits**: each split line has its own
  tax code. Tax is **inclusive** ŌĆö the entered amount includes tax.
  The system extracts the tax component per line using the line's tax
  code rate. The sum of (net amounts + tax amounts) across all lines
  must equal the transaction total.
- **REC-064**: **Negative transactions** (credits/refunds): the
  transaction total is negative. Split line amounts must also be
  negative (or the system auto-negates positive entries). The same
  balancing rule applies: sum of lines = transaction total. The GL
  entries reverse direction (credit expense, debit bank).
- **REC-065**: **Validation before save**: the Save button is disabled
  until:
  - All lines have a category selected
  - All lines have a tax code selected
  - Sum of line amounts = transaction total (out of balance = $0.00)
  - At least 2 lines exist (a single line is not a split ŌĆö use the
    non-split path instead)

## Rule Management

- **REC-021**: Users shall be able to create a rule from any
  allocation. The "Create rule" button shall appear alongside
  Save/Cancel in the expanded row.
- **REC-022**: A rule shall define: match pattern (text pattern on
  transaction description), default category/account, default tax code,
  default contact, and whether to auto-apply or suggest only.
- **REC-023**: Rules shall be listed and managed in a Rules section
  accessible from the reconciliation page.
- **REC-024**: When a new transaction matches an existing rule, the
  rule shall be auto-applied (if the rule is set to auto-apply) or
  shown as a suggestion in the Match column.

## Smart Reconciliation (Assisted Workflow)

- **REC-025**: The reconciliation page shall offer a "Smart
  Reconciliation" toggle (off by default). When enabled, the system
  analyses unmatched transactions and proposes likely matches.
- **REC-026**: Smart reconciliation is an **assisted workflow, not
  automatic commit**. All suggested matches require explicit human
  confirmation before being applied.

### 8.a Meaningful Name Extraction

- **REC-027**: The system shall extract a "meaningful name value"
  from each bank transaction description by identifying:
  - Business / supplier / merchant name
  - Product or service description
  - Stable merchant identifier embedded in the text

- **REC-028**: The extraction shall ignore or de-prioritise noise:
  - Transaction identifiers / settlement numbers
  - Dates embedded in descriptions
  - Formatting artifacts
  - Abbreviated card numbers (purely incidental)

- **REC-028a**: When smart reconciliation is active, the meaningful
  name extraction shall be **visually presented** with colour-coded
  highlighting on **ALL visible transaction descriptions** in the
  list ŌĆö not just the currently expanded row:
  - **Gold/yellow background** (`bg-amber-100`): the "gold" ŌĆö meaningful
    business/service/product tokens that form the matching identity.
  - **Pink background** (`bg-red-100`): noise tokens being discarded
    (identifiers, dates, card suffixes, reference numbers).
  - Unclassified tokens (ambiguous): no highlight (default text).
  - This gives the user an immediate bird's-eye view of what the
    system considers meaningful across the entire transaction list.
  - In the expanded row's Rule/Match section, the user can correct
    the extraction by clicking a token to toggle its classification
    (gold Ōåö pink Ōåö none). Changes to token classification feed back
    into the smart reconciliation engine to improve future extractions.

### 8.a.i Card Suffix Handling

- **REC-029**: Transaction descriptions containing abbreviated card
  values beginning with `~` (e.g. `~774924`) shall NOT treat those
  values as the primary merchant identity.
- **REC-030**: The `~xxxxxx` suffix MAY be used as a secondary
  linking signal ŌĆö if the same suffix appears across multiple
  transactions, it helps group transactions made on the same card.
- **REC-031**: Card suffix signals shall support confidence scoring
  but shall NOT override merchant/business-name detection.

### 8.b Historical Allocation Inference

- **REC-032**: Smart reconciliation shall examine past transaction
  allocations and identify candidate rules when there is strong
  historical similarity.
- **REC-033**: High-confidence suggestions shall be generated when
  prior transactions share:
  - The same or highly similar meaningful business name
  - The same amount (where applicable)
  - The same allocation outcome (category, account, tax code)
  - The same or very similar pattern over time
- **REC-034**: The system shall place particular weight on:
  - Previous manual allocations (user-confirmed intent)
  - Previous confirmed matches
  - Repeated exact-amount transactions with the same merchant

### 8.c Suggest, Preview, Confirm

- **REC-035**: Smart reconciliation shall generate suggested rules
  and perform a trial match against current unmatched transactions.
- **REC-036**: Suggested matches shall be presented to the user in
  the "Match" column with a confidence indicator (high/medium/low)
  and the matched rule/pattern.
- **REC-037**: The user must explicitly confirm each suggestion before
  it is committed. Bulk-confirm is allowed for high-confidence matches.
- **REC-038**: Suggestion quality shall improve deterministically based
  on confirmed history. Specifically:
  - **REC-038a**: Each confirmed suggestion increments the matched
    rule's `use_count`. The confidence scoring algorithm (A-0036 ┬¦
    Confidence Scoring) adds +0.10 per historical occurrence (capped
    at +0.20). Therefore a rule confirmed twice scores 0.20 higher
    than one confirmed once.
  - **REC-038b**: Each rejected suggestion is logged with the reason.
    If a rule accumulates 3+ rejections without intervening
    confirmations, its `confidence` floor is reduced by 0.10 (minimum
    0.00), effectively suppressing it from future suggestions.
  - **REC-038c**: When a user corrects a smart-extracted meaningful
    name (REC-028a token reclassification), the correction is stored
    as a `name_override` on the rule. Future extractions for the same
    raw description pattern use the override instead of re-extracting.
  - **REC-038d**: Testable metric: for any tenant with Ōēź50 confirmed
    allocations, the system shall achieve Ōēź70% suggestion acceptance
    rate (confirmed / (confirmed + rejected)) on the next batch of
    unmatched transactions. This is measured via the `smart_recon_stats`
    view (to be created in the reporting schema).
