# Summary

A complete redesign of the bank reconciliation workflow to match or
exceed competitor capabilities (MYOB scores 7/10; current Ledgius
scores 2/10). The new design introduces a full-width transaction
list with in-situ expansion, split allocations, sortable columns,
bank account labelling, rule management, and an AI-assisted smart
reconciliation engine that suggests matches from historical patterns.

# Problem / Business Context

Bank reconciliation is the highest-frequency bookkeeping task — every
bank transaction must be matched to a ledger entry or categorised.
The current Ledgius reconciliation page has critical UX issues:

1. Transaction details are compressed into 3 rows on the left, wasting
   horizontal space
2. No inline expansion — users must navigate to a separate detail panel
3. Numbers are bolded (visually heavy)
4. The right-side detail panel is unnecessary when details can expand
   in-situ
5. Columns are not sortable
6. Bank accounts lack descriptive labels
7. No split allocation capability
8. No rule-based auto-matching
9. No smart reconciliation / AI-assisted suggestions

Competitors (MYOB, Xero) provide inline expansion, split allocations,
rule creation from transactions, and smart/auto-matching. Ledgius must
match and exceed this standard.

# Scope

## Transaction List (UX Overhaul)

- **REC-001**: Transaction lines shall span the full page width and
  only wrap text if the browser window is narrower than the content.
- **REC-002**: Each transaction line shall be a single row displaying: Date, Bank Transaction (description), Withdrawal ($), Deposit ($), Match status, Tax code.
- **REC-003**: Numbers shall NOT be bold — use `font-normal tabular-nums`.
- **REC-004**: All columns shall be sortable ascending/descending by
  clicking the column header. Sort indicator (▲/▼) shall be visible.
- **REC-005**: Clicking a transaction row shall expand it in-situ
  (below the row, pushing subsequent rows down) revealing three
  collapsible sections: Rule/Match, Actions, Audit.

## In-Situ Expansion Panel

- **REC-006**: The expanded area shall be a single unified panel that
  opens directly below the selected row. Inside this panel there are
  three sections arranged vertically:
  - **Rule/Match** (open by default): shows matched rule details or
    "No rule matched" with an option to search/create a rule. Includes
    the colour-coded meaningful name extraction (REC-028a) when smart
    reconciliation is active.
  - **Actions**: the available operations for this transaction:
    - Match to existing record (invoice, bill, journal)
    - Select categories (create new GL entry with category/account)
    - Transfer money (between bank accounts)
  - **Audit** (collapsed by default, expandable): shows who allocated,
    when, revision history, any rule applied.
  All three sections are always visible within the one panel (not
  separate tabs — they stack vertically with section headers).
- **REC-007**: The right-side detail panel shall be removed. All
  detail is in the expanded panel.

## Cross-Transaction Rule Match Badges

- **REC-007a**: When a rule is selected or matched for the currently
  expanded transaction, the system shall immediately scan all other
  **visible** transactions in the current page/viewport and display
  pending-match badges on those that also match the rule. This is a
  **visual preview only** — its purpose is to teach the user what
  types of transactions the rule will match. The actual rule, once
  confirmed, applies to **all unmatched transactions** in the current
  filtered date range (including those not currently visible due to
  pagination), not just the visible subset.
- **REC-007b**: Transactions that match the rule being edited shall
  display **positive visual feedback** — matches are a good thing
  (the rule is working), not a warning:
  - **Row highlight**: light blue/green tint (`bg-primary-50` or
    `bg-green-50`) with a left border accent — NOT amber/yellow
    (which implies warning).
  - **Matching text**: the portion of the description matching the
    pattern shall be highlighted with a `<mark>` tag using a
    blue/green background (`bg-primary-100`).
  - **Allocated column**: temporarily changes from `N` to `~N~ Y`
    (strikethrough N with Y beside it) to preview the effect of
    saving the rule.
  - **Status column**: temporarily changes from `Unallocated` to
    `New Allocation` to preview the state transition.
  - **Match count**: displayed inline with the pattern field label:
    "Match pattern — matches N of X unallocated".
- **REC-007c**: Clicking a previewed match row shall expand that row
  and show the pre-filled allocation from the rule, ready for
  confirmation.
- **REC-007d**: When the user confirms/saves the rule on the original
  transaction, all matching transactions are immediately allocated
  with `allocation_method = 'rule_match'` and
  `workflow_status = 'proposed'`. The Save button reflects the
  count: "Save & Allocate (N)".

## Bank Account Header

- **REC-008**: The bank account selector shall display the account
  name with a descriptive label (e.g. "Nursery Bank Account" with
  the bank logo/icon and BSB/account number if available).
- **REC-009**: The header shall show: bank balance, book balance,
  and the difference (variance) prominently.
- **REC-010**: The header shall show when the bank data was last
  updated (from bank feed or last import).

## Filter Tabs

- **REC-011**: Transaction list shall have filter tabs with counts:
  "All transactions", "Unallocated (N)", "Proposed (N)", "Approved".
- **REC-012**: Date range filter shall be available alongside the tabs.
- **REC-013**: A search field shall filter transactions by description
  text (case-insensitive, partial match).

## Batch Operations

- **REC-014**: Each transaction row shall have a checkbox for
  multi-select.
- **REC-015**: A batch action bar shall appear when transactions are
  selected showing the count and available actions.
- **REC-016**: Maximum batch size: 50 transactions per operation.

### Batch-Safe Actions

- **REC-066**: The following actions are batch-safe (can be applied
  to multiple selected transactions at once):
  - **Approve selected**: transitions all selected `proposed`
    transactions to `approved`. Only `proposed` lines are eligible;
    lines in other statuses are silently skipped.
  - **Confirm suggestions**: transitions all selected smart
    reconciliation suggestions to `proposed` using their suggested
    rule. Only high-confidence suggestions (≥0.85) are eligible for
    batch confirm.
  - **Clear selection**: deselects all checkboxes (no data change).
- **REC-067**: The following actions are NOT batch-safe and must be
  performed one transaction at a time:
  - **Categorise** (requires per-transaction account selection)
  - **Match to existing record** (requires per-transaction GL match)
  - **Transfer** (requires per-transaction target account selection)
  - **Split allocation** (requires per-transaction line entry)

### Batch Execution Semantics

- **REC-068**: Batch operations are **partial-success** — each
  transaction in the batch is processed independently. If one
  transaction fails (e.g. a `proposed` line was cleared by another
  user between selection and approval), the remaining transactions
  continue processing.
- **REC-069**: After a batch operation completes, the UI shall show
  a result summary:
  - ✓ N transactions approved successfully
  - ✗ M transactions failed (with per-row reason)
  - ○ K transactions skipped (wrong state)
  Each failed/skipped row shall be highlighted in the transaction
  list with a brief inline error message visible without expansion.
- **REC-070**: Batch operations shall execute within a single HTTP
  request. The backend processes each line in its own DB transaction
  (not one giant transaction) so that partial success is possible
  and a single failure doesn't roll back the entire batch.

## Bank Line Lifecycle (State Model)

A bank feed line is tracked by two independent fields that together
describe its full lifecycle position:

- **`allocation_method`** — HOW the transaction was matched/resolved
- **`workflow_status`** — WHERE the transaction is in the approval lifecycle

This two-field model separates the *mechanism* of allocation from the
*approval state*, enabling richer reporting (e.g. "what percentage of
transactions are auto-matched?") and a clean "Booking Sandbox" pattern
where all allocations are staging data until explicitly approved.

### Booking Sandbox Principle

No GL entries (`acc_trans`, `transactions`, `gl`), no journal entries,
and no audit trail pollution occur until `workflow_status = 'approved'`.
All allocations — whether from rules, manual categorisation, or
auto-linking — are staging data that the user can freely create, modify,
delete, and redo with zero accounting impact. This matches the import
staging pipeline pattern used elsewhere in Ledgius.

### `allocation_method` Values

| Value | Meaning |
|---|---|
| `null` | No allocation exists (initial state on import) |
| `rule_match` | Matched by a reconciliation rule (auto-applied or user-confirmed suggestion) |
| `manual_allocation` | User manually categorised, split, or selected a category |
| `linked_auto` | System auto-linked via invoice/bill reference detection (REC-079) |
| `linked_manual` | User manually matched to an existing record (invoice, bill, journal) |
| `excluded` | User explicitly excluded this transaction from reconciliation |
| `deferred` | User deferred this transaction for later review |

### `workflow_status` Values

| Value | Meaning |
|---|---|
| `imported` | Raw bank feed line, no action taken |
| `unallocated` | Previously allocated but allocation was cleared (e.g. after reversal) |
| `proposed` | An allocation exists but has NOT been approved — staging data only, no GL impact |
| `approved` | Allocation reviewed and approved — GL entries created and locked |

### State Diagram

```
                    ┌───────────────────┐
                    │     imported      │  ← initial state on import
                    │  (method = null)  │     workflow_status = 'imported'
                    └────────┬──────────┘
                             │
          ┌──────────────────┼──────────────────┐
          ▼                  ▼                  ▼
  ┌───────────────┐  ┌──────────────┐  ┌───────────────┐
  │   proposed    │  │   proposed   │  │   proposed    │
  │ (rule_match)  │  │ (manual/     │  │ (linked_auto  │
  │               │  │  linked_     │  │  or excluded  │
  │               │  │  manual)     │  │  or deferred) │
  └───────┬───────┘  └──────┬───────┘  └───────┬───────┘
          │                 │                   │
          └─────────────────┼───────────────────┘
                            ▼
                    ┌───────────────────┐
                    │     approved      │  ← GL entries created here
                    │  (method kept)    │     and ONLY here
                    └────────┬──────────┘
                             │
                             ▼  (reversal)
                    ┌───────────────────┐
                    │   unallocated     │  ← GL reversal entries created
                    │  (method = null)  │     available for re-allocation
                    └───────────────────┘
```

- **REC-044**: Each bank feed line shall carry two fields —
  `allocation_method` and `workflow_status` — with the following
  permitted combinations and transitions:

| workflow_status | allocation_method | Meaning | Permitted transitions |
|---|---|---|---|
| `imported` | `null` | Raw bank feed line, no action taken | → `proposed` (with any non-null method) |
| `unallocated` | `null` | Was allocated/approved but reversed; available for re-allocation | → `proposed` (with any non-null method) |
| `proposed` | any non-null | Allocation exists as staging data — no GL entries, freely editable | → `approved`, → `imported`/`unallocated` (user clears allocation, method reset to null) |
| `approved` | any non-null | GL entries created and locked | → `unallocated` (reversal: GL reversal entries created, method cleared to null) |

- **REC-045**: The filter tabs shall map to these fields:
  - "Unallocated" = `workflow_status IN ('imported', 'unallocated')`
  - "Proposed" = `workflow_status = 'proposed'`
  - "Approved" = `workflow_status = 'approved'`
  - "All" = all statuses
  - Optional sub-filters by `allocation_method` within each tab
- **REC-046**: GL entries are ONLY created when `workflow_status`
  transitions to `approved` — never at `proposed`. A `proposed`
  allocation is staging data with zero accounting impact. The user
  can freely modify, delete, or redo any proposed allocation without
  creating GL noise.
- **REC-047**: An `approved` allocation's GL entries are locked and
  cannot be edited. To change an approved allocation, the user must
  first reverse it (creating reversal GL entries) which transitions
  the line to `workflow_status = 'unallocated'` with
  `allocation_method = null`, making it available for re-allocation.

### Review & Approve Workflow

- **REC-044a**: The reconciliation page operates in two modes:
  - **Working mode**: the default view for allocating unmatched
    transactions. The user works through `imported`/`unallocated`
    transactions, categorising, matching, or linking them. Each action
    sets the `allocation_method` and transitions `workflow_status` to
    `proposed`.
  - **Review mode**: a dedicated view for reviewing and approving
    proposed allocations. Transactions are grouped by resolution type
    (allocation_method and, for rule matches, by the specific rule).
    The user reviews each group and approves or rejects.
- **REC-044b**: Review mode groups proposed transactions by resolution
  type:
  - **Rule matches** — grouped under their rule, showing: rule name,
    match pattern, match count, total amount. Each rule group is
    collapsible.
  - **Manual allocations** — transactions categorised by hand.
  - **Auto-linked** — transactions matched via invoice/bill reference
    detection (REC-079/080).
  - **Manual links** — transactions manually matched to existing
    records.
  - **Deferred** — transactions the user marked for later review.
  - **Excluded** — transactions explicitly excluded from
    reconciliation.
  Each group has an "Approve All" button and individual approve/reject
  controls per transaction.
- **REC-044c**: A **bucket distribution bar** shall be displayed at
  the top of the reconciliation page showing live counts and
  percentages for each `allocation_method`. The bar updates in
  real-time as the user works. The goal state is: unallocated = 0,
  everything approved.
- **REC-044d**: **End-of-period reporting** shall provide a percentage
  breakdown by `allocation_method`: A% rule-matched, M% manual, L%
  linked (auto + manual), D% deferred, E% excluded. A higher
  auto-match percentage indicates less bookkeeper time required and
  better rule coverage.

## Action Accounting Semantics

The three actions in the expanded row produce different accounting
results. Each is defined precisely.

> **Sandbox Principle**: All allocation actions (category, match,
> transfer, link) create `recon_allocation` metadata with
> `workflow_status = 'proposed'`. NO GL entries (`acc_trans`,
> `transactions`, `gl`) are created until the user explicitly
> approves. This allows the user to freely experiment, change
> allocations, delete and redo — with zero accounting impact until
> approval.

### Match to Existing Record

- **REC-048**: "Match to existing record" links the bank line to a
  pre-existing unreconciled GL entry (e.g. an invoice payment, a bill
  payment, or a journal entry that is already in `acc_trans`).
- **REC-049**: Matching does NOT create new GL entries. It marks the
  existing `acc_trans` row(s) as reconciled by setting a reconciliation
  reference on the `acc_trans.source` field linking it to the bank
  feed line.
- **REC-050**: The match requires that the bank line amount equals
  the GL entry amount (or the sum of multiple selected GL entries for
  a composite match). A tolerance of ±$0.01 is permitted for rounding.
- **REC-051**: Matching is only available when there are unreconciled
  GL entries on the same bank account that match the amount and
  approximate date range.

### Select Categories (Create New GL Entry)

- **REC-052**: "Select categories" stages a NEW GL journal entry
  to categorise the bank transaction. This is for transactions that
  have no pre-existing ledger entry (e.g. a bank fee, a cash purchase,
  an ad-hoc expense). The GL entry SHALL be created when the
  allocation transitions from `proposed` to `approved`, NOT when the
  user clicks Save. At Save time, only the `recon_allocation` staging
  record is persisted.
- **REC-053**: The GL entry created (upon approval) shall be:
  - **Withdrawal**: Debit the selected expense/asset account, Credit
    the bank account.
  - **Deposit**: Debit the bank account, Credit the selected income/
    liability account.
- **REC-054**: Each allocation line in the GL entry carries:
  - Account (required — must be a non-bank account)
  - Amount (required)
  - Tax code (required — drives BAS reporting)
  - Description (optional)
  - Contact (optional — links to a customer/vendor)
- **REC-055**: Tax behaviour: amounts entered are **tax-inclusive**
  by default. The system calculates the tax component from the tax
  code rate and creates the tax GL entry automatically:
  - GST (10%): $110 entered → $100 to expense + $10 to GST account
  - FRE / N-T: $110 entered → $110 to expense, $0 tax

### Transfer Money

- **REC-056**: "Transfer money" records a movement between two bank
  accounts (e.g. transfer from cheque to savings, credit card payment).
- **REC-057**: Transfer requires selecting a **target bank account**
  (must be another account with `category = 'A'` and a bank account
  type). P&L accounts (income/expense) are NOT selectable as transfer
  targets.
- **REC-058**: The GL entry created shall be:
  - Debit the target bank account, Credit the source bank account
    (for a withdrawal from source).
  - Or: Debit the source bank account, Credit the target bank account
    (for a deposit into source).
- **REC-059**: No tax code is required for transfers (bank-to-bank
  movements are not GST-reportable).

### Transfer Two-Sided Matching

- **REC-075**: When the user selects "Transfer money" on a bank line,
  the system shall search the other bank account's unmatched lines for
  a corresponding entry (opposite direction, same or similar amount,
  within ±2 business days). If a match is found, it shall be
  pre-selected as the other half of the transfer.
- **REC-076**: When both halves of a transfer are matched, both bank
  lines transition to `proposed` (and subsequently `approved`)
  together. Neither line remains unallocated.
- **REC-077**: If the other half has not yet arrived from the bank
  feed (e.g. withdrawal posted today, deposit appears tomorrow), the
  system shall create the transfer GL entry from the available side
  and leave a "pending counterpart" marker on the target account.
  When the matching deposit/withdrawal arrives on the other account,
  the system auto-suggests matching it to the pending transfer.
- **REC-078**: A transfer is NOT a payment to a supplier, a receipt
  into income, or a match to an existing invoice/bill/journal. The
  "Transfer money" action is exclusively for inter-account balance-
  sheet movements between accounts the business controls (cheque →
  savings, cheque → credit card, bank → petty cash).

### Auto-Linking (Invoice/Bill Reference Detection)

- **REC-079**: The system shall scan transaction descriptions for
  invoice numbers (INV-NNNN, #NNNN) and bill numbers matching open
  receivables/payables.
- **REC-080**: When a reference match is found with compatible amount
  (within a tolerance of ±$0.01), the transaction is auto-linked with
  `allocation_method = 'linked_auto'` and
  `workflow_status = 'proposed'`.
- **REC-081**: Auto-linked transactions appear in Review mode under
  the "Auto-Linked" grouping for user confirmation before approval.

## Split Allocations

- **REC-017**: When categorising a transaction, the user shall be able
  to split the amount across multiple GL accounts (split allocation).
- **REC-018**: Each split line shall have: Category (account), Amount
  ($), Amount (%), Line description, Tax code.
- **REC-019**: The split view shall show Subtotal and Out of Balance
  amounts. The total of all split lines must equal the transaction
  amount before saving.
- **REC-020**: Splits shall support 2–20 lines.

### Split Allocation Behaviour Rules

- **REC-060**: Splits are **amount-driven by default**. The user enters
  dollar amounts on each line. Percentages are calculated and displayed
  automatically as a read-only convenience (`line_amount / total * 100`).
- **REC-061**: The user may switch to **percentage-driven** mode by
  clicking a toggle on the split grid header. In percentage mode, the
  user enters percentages and dollar amounts are calculated from the
  transaction total. Only one mode is active at a time — mixed entry
  (some lines by amount, some by percentage) is NOT permitted.
- **REC-062**: **Rounding**: when calculating amounts from percentages,
  the system uses banker's rounding (round half to even) to 2 decimal
  places. Any rounding residual (±$0.01) is applied to the last line
  to ensure the total balances exactly.
- **REC-063**: **Tax handling in splits**: each split line has its own
  tax code. Tax is **inclusive** — the entered amount includes tax.
  The system extracts the tax component per line using the line's tax
  code rate. The sum of (net amounts + tax amounts) across all lines
  must equal the transaction total.
- **REC-064**: **Negative transactions** (credits/refunds): the
  transaction total is negative. Split line amounts must also be
  negative (or the system auto-negates positive entries). The same
  balancing rule applies: sum of lines = transaction total. The GL
  entries reverse direction (credit expense, debit bank).
- **REC-065**: **Validation before save**: the Save button is disabled
  until:
  - All lines have a category selected
  - All lines have a tax code selected
  - Sum of line amounts = transaction total (out of balance = $0.00)
  - At least 2 lines exist (a single line is not a split — use the
    non-split path instead)

## Rule Management

- **REC-021**: Users shall be able to create a rule from any
  allocation. The "Create rule" button shall appear alongside
  Save/Cancel in the expanded row.
- **REC-022**: A rule shall define: match pattern (text pattern on
  transaction description), default category/account, default tax code,
  default contact, and whether to auto-apply or suggest only. Match
  patterns support `*` as wildcard in Contains mode (e.g.
  `DEPOSIT*Square` matches any description containing "DEPOSIT"
  followed by "Square" with any text between). Amount matching
  supports: Any, Exact, Set (comma-separated values), Range
  (min-max).
- **REC-023**: Rules shall be listed and managed in a Rules section
  accessible from the reconciliation page.
- **REC-024**: When a new transaction matches an existing rule, the
  rule shall be auto-applied (if the rule is set to auto-apply) or
  shown as a suggestion in the Match column.
- **REC-024a**: When a rule is created or saved, it SHALL be
  immediately applied to ALL unallocated transactions
  (`workflow_status IN ('imported', 'unallocated')`) matching the
  pattern. Matched transactions transition to
  `workflow_status = 'proposed'` with
  `allocation_method = 'rule_match'`. The user sees the match count
  update in real-time.
- **REC-024b**: Rules management shall be accessible as a slide-out
  drawer from the reconciliation page. The drawer shows all rules in
  priority order with: name, pattern, match type, default
  account/tax/contact, confidence, use count, enable/disable toggle,
  drag-to-reorder priority.

### Rule Scope and Precedence

- **REC-071**: Rules are **tenant-wide** — a rule created on one bank
  account applies to transactions on all bank accounts for the same
  tenant. Rules are not bank-account-specific because the same
  merchant (e.g. "BP Warrenheip") appears across accounts.
- **REC-072**: A single rule pool serves both debit (withdrawal) and
  credit (deposit) transactions. The rule matches on description text
  regardless of transaction direction. However, the default account on
  the rule must be compatible with the direction:
  - Withdrawal rule → default account should be expense/asset category
  - Deposit rule → default account should be income/liability category
  - If a rule matches a transaction of incompatible direction, the
    match is shown as a suggestion (never auto-applied) so the user
    can override the account.
- **REC-073**: When **multiple rules match** a single transaction, the
  system resolves precedence using this ordered tiebreak:
  1. **Manual rules** (source='manual') outrank **smart-generated
     rules** (source='smart') — user intent takes priority over
     machine suggestion.
  2. Among rules of the same source, **higher confidence** wins.
  3. Among rules with equal confidence, **more specific match pattern**
     wins (exact > contains > regex).
  4. Among rules still tied, **higher use_count** wins (more
     historically confirmed = more trusted).
  5. If still tied, the most recently created rule wins.
  Only the winning rule is applied or suggested. Losing rules are not
  shown to avoid confusion.
- **REC-074**: A rule may be **disabled** without deletion. Disabled
  rules are excluded from matching but retained for audit history.

## Smart Reconciliation (Assisted Workflow)

- **REC-025**: The reconciliation page shall offer a "Smart
  Reconciliation" toggle (off by default). When enabled, the system
  analyses unmatched transactions and proposes likely matches.
- **REC-026**: Smart reconciliation is an **assisted workflow, not
  automatic commit**. All suggested matches require explicit human
  confirmation before being applied.

### 8.a Meaningful Name Extraction

- **REC-027**: The system shall extract a "meaningful name value"
  from each bank transaction description by identifying:
  - Business / supplier / merchant name
  - Product or service description
  - Stable merchant identifier embedded in the text

- **REC-028**: The extraction shall ignore or de-prioritise noise:
  - Transaction identifiers / settlement numbers
  - Dates embedded in descriptions
  - Formatting artifacts
  - Abbreviated card numbers (purely incidental)

- **REC-028a**: When smart reconciliation is active, the meaningful
  name extraction shall be **visually presented** with colour-coded
  highlighting on **ALL visible transaction descriptions** in the
  list — not just the currently expanded row:
  - **Gold/yellow background** (`bg-amber-100`): the "gold" — meaningful
    business/service/product tokens that form the matching identity.
  - **Pink background** (`bg-red-100`): noise tokens being discarded
    (identifiers, dates, card suffixes, reference numbers).
  - Unclassified tokens (ambiguous): no highlight (default text).
  - This gives the user an immediate bird's-eye view of what the
    system considers meaningful across the entire transaction list.
  - In the expanded row's Rule/Match section, the user can correct
    the extraction by clicking a token to toggle its classification
    (gold ↔ pink ↔ none). Changes to token classification feed back
    into the smart reconciliation engine to improve future extractions.

### 8.a.i Card Suffix Handling

- **REC-029**: Transaction descriptions containing abbreviated card
  values beginning with `~` (e.g. `~774924`) shall NOT treat those
  values as the primary merchant identity.
- **REC-030**: The `~xxxxxx` suffix MAY be used as a secondary
  linking signal — if the same suffix appears across multiple
  transactions, it helps group transactions made on the same card.
- **REC-031**: Card suffix signals shall support confidence scoring
  but shall NOT override merchant/business-name detection.

### 8.b Historical Allocation Inference

- **REC-032**: Smart reconciliation shall examine past transaction
  allocations and identify candidate rules when there is strong
  historical similarity.
- **REC-033**: High-confidence suggestions shall be generated when
  prior transactions share:
  - The same or highly similar meaningful business name
  - The same amount (where applicable)
  - The same allocation outcome (category, account, tax code)
  - The same or very similar pattern over time
- **REC-034**: The system shall place particular weight on:
  - Previous manual allocations (user-confirmed intent)
  - Previous confirmed matches
  - Repeated exact-amount transactions with the same merchant

### 8.c Suggest, Preview, Confirm

- **REC-035**: Smart reconciliation shall generate suggested rules
  and perform a trial match against current unmatched transactions.
- **REC-036**: Suggested matches shall be presented to the user in
  the "Match" column with a confidence indicator (high/medium/low)
  and the matched rule/pattern.
- **REC-037**: The user must explicitly confirm each suggestion before
  it is committed. Bulk-confirm is allowed for high-confidence matches.
- **REC-038**: Suggestion quality shall improve deterministically based
  on confirmed history. Specifically:
  - **REC-038a**: Each confirmed suggestion increments the matched
    rule's `use_count`. The confidence scoring algorithm (A-0036 §
    Confidence Scoring) adds +0.10 per historical occurrence (capped
    at +0.20). Therefore a rule confirmed twice scores 0.20 higher
    than one confirmed once.
  - **REC-038b**: Each rejected suggestion is logged with the reason.
    If a rule accumulates 3+ rejections without intervening
    confirmations, its `confidence` floor is reduced by 0.10 (minimum
    0.00), effectively suppressing it from future suggestions.
  - **REC-038c**: When a user corrects a smart-extracted meaningful
    name (REC-028a token reclassification), the correction is stored
    as a `name_override` on the rule. Future extractions for the same
    raw description pattern use the override instead of re-extracting.
  - **REC-038d**: Testable metric: for any tenant with ≥50 confirmed
    allocations, the system shall achieve ≥70% suggestion acceptance
    rate (confirmed / (confirmed + rejected)) on the next batch of
    unmatched transactions. This is measured via the `smart_recon_stats`
    view (to be created in the reporting schema).

# Out of Scope

- Full AI/ML model training (v2+ — smart reconciliation uses
  pattern matching + historical heuristics, not a trained model)
- Inter-company reconciliation
- Foreign currency reconciliation (v2)
- Automated bank statement download (handled by bank feeds R-0020)

# Actors / Users

- Bookkeeper — primary user, performs daily reconciliation
- Business Owner — reviews reconciliation status, approves batches
- Accountant — reviews reconciliation as part of period close

# Preconditions

- At least one bank account in the chart of accounts
- Bank transactions imported (via bank feed or file import)

# UX / UI Behaviour

- **REC-039**: Transaction list uses the full page width with no
  right-side panel.
- **REC-040**: Expanded rows push content down (accordion style),
  only one row expanded at a time.
- **REC-041**: Sort state persists within the session (not across
  page loads).
- **REC-042**: Filter tabs show real-time counts from the current
  dataset.
- **REC-043**: Smart reconciliation toggle shows a loading indicator
  while analysing, then updates the Match column with suggestions.

# Security / Permissions

| Capability | Minimum Role | Notes |
|---|---|---|
| View reconciliation page | Viewer | Read-only — no actions available |
| Allocate / categorise a transaction | Bookkeeper | Transitions line to `proposed` |
| Approve an allocation | Bookkeeper | Transitions line to `approved`, creates and locks GL |
| **Reverse an approved allocation** | **Accountant** | Creates reversal GL entries — higher bar because it undoes finalised work |
| Create a rule | Bookkeeper | Manual rules from allocations |
| Edit a rule | Bookkeeper | Modify pattern, account, tax code |
| **Delete a rule** | **Accountant** | Permanent removal — Bookkeeper can disable but not delete |
| Enable smart reconciliation | Bookkeeper | Toggle on/off |
| Bulk-confirm smart suggestions | Bookkeeper | High-confidence only (≥0.85) |
| **Edit smart token overrides** | **Bookkeeper** | Correct meaningful-name extraction |
| **Bulk-approve allocations** | **Bookkeeper** | Batch approve up to 50 |
