# Summary

A complete redesign of the bank reconciliation workflow to match or
exceed competitor capabilities (MYOB scores 7/10; current Ledgius
scores 2/10). The new design introduces a full-width transaction
list with in-situ expansion, split allocations, sortable columns,
bank account labelling, rule management, and an AI-assisted smart
reconciliation engine that suggests matches from historical patterns.

# Problem / Business Context

Bank reconciliation is the highest-frequency bookkeeping task — every
bank transaction must be matched to a ledger entry or categorised.
The current Ledgius reconciliation page has critical UX issues:

1. Transaction details are compressed into 3 rows on the left, wasting
   horizontal space
2. No inline expansion — users must navigate to a separate detail panel
3. Numbers are bolded (visually heavy)
4. The right-side detail panel is unnecessary when details can expand
   in-situ
5. Columns are not sortable
6. Bank accounts lack descriptive labels
7. No split allocation capability
8. No rule-based auto-matching
9. No smart reconciliation / AI-assisted suggestions

Competitors (MYOB, Xero) provide inline expansion, split allocations,
rule creation from transactions, and smart/auto-matching. Ledgius must
match and exceed this standard.

# Scope

## Transaction List (UX Overhaul)

- **REC-001**: Transaction lines shall span the full page width and
  only wrap text if the browser window is narrower than the content.
- **REC-002**: Each transaction line shall be a single row displaying:
  Date, Bank Transaction (description), Withdrawal ($), Deposit ($),
  Match status, Tax code.
- **REC-003**: Numbers shall NOT be bold — use `font-normal tabular-nums`.
- **REC-004**: All columns shall be sortable ascending/descending by
  clicking the column header. Sort indicator (▲/▼) shall be visible.
- **REC-005**: Clicking a transaction row shall expand it in-situ
  (below the row, pushing subsequent rows down) revealing three
  collapsible sections: Rule/Match, Actions, Audit.

## In-Situ Expansion

- **REC-006**: The expanded area shall contain three sections:
  - **Rule/Match** (open by default): shows matched rule details or
    "No rule matched" with an option to search/create a rule.
  - **Actions**: the available operations for this transaction:
    - Match to existing record (invoice, bill, journal)
    - Select categories (create new GL entry with category/account)
    - Transfer money (between bank accounts)
  - **Audit** (collapsed by default): shows who allocated, when,
    revision history, any rule applied.
- **REC-007**: The right-side detail panel shall be removed. All
  detail is in the expanded row area.

## Bank Account Header

- **REC-008**: The bank account selector shall display the account
  name with a descriptive label (e.g. "Nursery Bank Account" with
  the bank logo/icon and BSB/account number if available).
- **REC-009**: The header shall show: bank balance, book balance,
  and the difference (variance) prominently.
- **REC-010**: The header shall show when the bank data was last
  updated (from bank feed or last import).

## Filter Tabs

- **REC-011**: Transaction list shall have filter tabs with counts:
  "All transactions", "Not matched (N)", "Matched".
- **REC-012**: Date range filter shall be available alongside the tabs.
- **REC-013**: A search field shall filter transactions by description
  text (case-insensitive, partial match).

## Batch Operations

- **REC-014**: Each transaction row shall have a checkbox for
  multi-select.
- **REC-015**: A batch action bar shall appear when transactions are
  selected: "Approve selected" / "Categorise selected" /
  "Clear selection".
- **REC-016**: Maximum batch size: 50 transactions per operation.

## Split Allocations

- **REC-017**: When categorising a transaction, the user shall be able
  to split the amount across multiple GL accounts (split allocation).
- **REC-018**: Each split line shall have: Category (account), Amount
  ($), Amount (%), Line description, Tax code.
- **REC-019**: The split view shall show Subtotal and Out of Balance
  amounts. The total of all split lines must equal the transaction
  amount before saving.
- **REC-020**: Splits shall support 2–20 lines.

## Rule Management

- **REC-021**: Users shall be able to create a rule from any
  allocation. The "Create rule" button shall appear alongside
  Save/Cancel in the expanded row.
- **REC-022**: A rule shall define: match pattern (text pattern on
  transaction description), default category/account, default tax code,
  default contact, and whether to auto-apply or suggest only.
- **REC-023**: Rules shall be listed and managed in a Rules section
  accessible from the reconciliation page.
- **REC-024**: When a new transaction matches an existing rule, the
  rule shall be auto-applied (if the rule is set to auto-apply) or
  shown as a suggestion in the Match column.

## Smart Reconciliation (Assisted Workflow)

- **REC-025**: The reconciliation page shall offer a "Smart
  Reconciliation" toggle (off by default). When enabled, the system
  analyses unmatched transactions and proposes likely matches.
- **REC-026**: Smart reconciliation is an **assisted workflow, not
  automatic commit**. All suggested matches require explicit human
  confirmation before being applied.

### 8.a Meaningful Name Extraction

- **REC-027**: The system shall extract a "meaningful name value"
  from each bank transaction description by identifying:
  - Business / supplier / merchant name
  - Product or service description
  - Stable merchant identifier embedded in the text

- **REC-028**: The extraction shall ignore or de-prioritise noise:
  - Transaction identifiers / settlement numbers
  - Dates embedded in descriptions
  - Formatting artifacts
  - Abbreviated card numbers (purely incidental)

### 8.a.i Card Suffix Handling

- **REC-029**: Transaction descriptions containing abbreviated card
  values beginning with `~` (e.g. `~774924`) shall NOT treat those
  values as the primary merchant identity.
- **REC-030**: The `~xxxxxx` suffix MAY be used as a secondary
  linking signal — if the same suffix appears across multiple
  transactions, it helps group transactions made on the same card.
- **REC-031**: Card suffix signals shall support confidence scoring
  but shall NOT override merchant/business-name detection.

### 8.b Historical Allocation Inference

- **REC-032**: Smart reconciliation shall examine past transaction
  allocations and identify candidate rules when there is strong
  historical similarity.
- **REC-033**: High-confidence suggestions shall be generated when
  prior transactions share:
  - The same or highly similar meaningful business name
  - The same amount (where applicable)
  - The same allocation outcome (category, account, tax code)
  - The same or very similar pattern over time
- **REC-034**: The system shall place particular weight on:
  - Previous manual allocations (user-confirmed intent)
  - Previous confirmed matches
  - Repeated exact-amount transactions with the same merchant

### 8.c Suggest, Preview, Confirm

- **REC-035**: Smart reconciliation shall generate suggested rules
  and perform a trial match against current unmatched transactions.
- **REC-036**: Suggested matches shall be presented to the user in
  the "Match" column with a confidence indicator (high/medium/low)
  and the matched rule/pattern.
- **REC-037**: The user must explicitly confirm each suggestion before
  it is committed. Bulk-confirm is allowed for high-confidence matches.
- **REC-038**: Suggestions shall improve over time based on confirmed
  history — each confirmation strengthens the underlying pattern.

# Out of Scope

- Full AI/ML model training (v2+ — smart reconciliation uses
  pattern matching + historical heuristics, not a trained model)
- Inter-company reconciliation
- Foreign currency reconciliation (v2)
- Automated bank statement download (handled by bank feeds R-0020)

# Actors / Users

- Bookkeeper — primary user, performs daily reconciliation
- Business Owner — reviews reconciliation status, approves batches
- Accountant — reviews reconciliation as part of period close

# Preconditions

- At least one bank account in the chart of accounts
- Bank transactions imported (via bank feed or file import)

# UX / UI Behaviour

- **REC-039**: Transaction list uses the full page width with no
  right-side panel.
- **REC-040**: Expanded rows push content down (accordion style),
  only one row expanded at a time.
- **REC-041**: Sort state persists within the session (not across
  page loads).
- **REC-042**: Filter tabs show real-time counts from the current
  dataset.
- **REC-043**: Smart reconciliation toggle shows a loading indicator
  while analysing, then updates the Match column with suggestions.

# Security / Permissions

- Minimum role: Bookkeeper (reconcile transactions)
- Smart reconciliation available to Bookkeeper and above
- Rule creation available to Bookkeeper and above
- Batch approval available to Bookkeeper and above

# Acceptance Criteria

- [ ] Full-width transaction list with single-row display
- [ ] In-situ expansion with Rule/Match, Actions, Audit sections
- [ ] All columns sortable asc/desc with sort indicator
- [ ] Bank account header with name, balance, variance, last updated
- [ ] Filter tabs: All / Not matched (N) / Matched
- [ ] Split allocations (2-20 lines) with subtotal + out of balance
- [ ] Create rule from any allocation
- [ ] Smart reconciliation toggle with confidence-scored suggestions
- [ ] Meaningful name extraction from transaction descriptions
- [ ] Historical pattern matching from past manual allocations
- [ ] All smart suggestions require human confirmation
- [ ] No bold numbers anywhere on the page
