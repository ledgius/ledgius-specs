# Summary

Technical architecture for the bank reconciliation workflow, covering the screen layout model, candidate matching engine, action mode processing, exception handling, audit event emission, keyboard-driven UX, and backend API contracts.

# Requirement Link

- R-0019 (Reconciliation Workflow Requirements)

# Technical Context

Bank reconciliation is a high-throughput, evidence-driven workflow. The architecture must support: (1) fast queue loading with incremental rendering, (2) candidate matching with multi-signal evidence scoring, (3) six distinct action modes with pre-commit verification, (4) exception surfacing without blocking the main workflow, and (5) full audit trail emission for every decision. The design must allow a bookkeeper to clear hundreds of lines per session while preserving defensible evidence for audit.

# Proposed Design

## Screen Region Architecture

The reconciliation screen is divided into four functional regions:

```
+------------------+---------------------------+
| Bank Line Queue  | Decision Workspace        |
| (scrollable      | (action-mode specific     |
|  filtered list)  |  content)                 |
|                  +---------------------------+
|                  | Candidate Transactions /  |
|                  | Split Editor / Create Form|
+------------------+---------------------------+
| Exceptions Rail (collapsible bottom bar)     |
+----------------------------------------------+
```

- **Bank Line Queue**: Left panel, scrollable, shows imported lines with status indicators. Supports keyboard navigation (J/K).
- **Decision Workspace**: Centre/right panel, renders the active action mode content. Changes based on selected action (match candidates, split editor, create form, etc.).
- **Exceptions Rail**: Bottom bar, collapsible, surfaces unresolved items that need attention.

## Candidate Matching Engine

The matching engine scores candidates using multi-signal evidence:

| Signal | Weight | Description |
|--------|--------|-------------|
| Exact amount match | High | Bank line amount equals transaction amount exactly |
| Date within tolerance | Medium | Transaction date within configurable window (default 3 days) |
| Reference similarity | Medium | String similarity between bank reference and transaction reference |
| Contact correlation | Medium | Bank line description matches a known vendor/customer name |
| Previous behaviour | Low | Historical pattern of similar matches for this bank account |
| Clearing account path | Low | Expected account relationship exists |

Candidates are bucketed into confidence levels for display:
- **Exact**: Amount match + date match + reference match
- **Strong**: Amount match + at least one other strong signal
- **Possible**: Amount match only, or partial signal overlap
- **Weak**: Partial amount or tangential signals only

Evidence chips are rendered on each candidate card showing which signals contributed.

## Action Mode Processing

Each action mode follows a common lifecycle:

```
Select bank line -> Choose action mode -> Configure action -> Review pre-commit summary -> Commit -> Audit event emitted
```

### Match Mode
1. System fetches candidates via `/reconciliation/lines/:id/candidates`
2. User selects one or more candidates
3. System computes: bank amount vs selected total, residual delta
4. Pre-commit summary displayed
5. On commit: link created between bank line and transaction(s), bank line status updated

### Split Mode
1. User opens split editor (S key)
2. Allocation grid with: account, amount, tax code, optional entity link, notes
3. Remaining amount indicator updates in real-time
4. Invariant enforced: sum of allocations == bank line amount
5. On commit: individual transactions or journal lines created per allocation

### Create Mode
1. User opens create form (C key)
2. Form pre-filled from bank line data (amount, date, reference)
3. User selects transaction subtype (receipt, payment, expense, journal)
4. On commit: new transaction created and linked to bank line

### Transfer Mode
1. User specifies both sides (from account, to account)
2. System checks for existing opposite-side bank line
3. Duplicate transfer guard: warns if similar transfer exists
4. On commit: transfer transaction created linking both accounts

### Defer Mode
1. User selects reason category from dropdown
2. Optional note and expected review date
3. On commit: bank line status set to deferred, appears in exception queue

### Exclude Mode
1. Permission check (elevated role required)
2. User selects reason code from approved list
3. On commit: bank line marked as excluded with audit event

## Exception Model

Exceptions are categorised and prioritised:

| Category | Priority Factors |
|----------|-----------------|
| No candidate found | Age, amount |
| Multiple plausible candidates | Ambiguity, amount |
| Duplicate risk | Financial risk |
| Amount mismatch | Delta size |
| Stale deferred item | Days since deferral |
| Period lock conflict | Workflow blockage |
| Already reconciled conflict | Data integrity |

Each exception displays: short explanation, severity badge, next recommended action, linked bank line, time waiting.

## Audit Event Model

Every reconciliation decision emits a structured audit event:

```json
{
  "bank_line_id": 12345,
  "action": "match",
  "actor": "user@example.com",
  "timestamp": "2026-04-09T10:30:00Z",
  "confidence": "exact",
  "evidence_summary": ["exact_amount", "date_within_1_day", "reference_match"],
  "linked_transaction_ids": [678],
  "split_details": null,
  "defer_reason": null,
  "exclude_reason": null
}
```

# Data Flow

```
Bank line imported (via import pipeline or upload)
  -> Appears in reconciliation queue
  -> User selects line
  -> Candidate engine fetches and scores matches
  -> User chooses action mode
  -> Action configured and pre-commit summary shown
  -> User confirms commit
  -> Transaction created/linked in database
  -> Bank line status updated
  -> Audit event emitted
  -> Queue advances to next line
```

# API / Interface Changes

| Method | Path | Purpose |
|--------|------|---------|
| GET | /reconciliation/queue | Fetch reconciliation queue (paginated, filtered) |
| GET | /reconciliation/lines/:id/candidates | Fetch candidate matches with evidence |
| POST | /reconciliation/lines/:id/match | Commit match action |
| POST | /reconciliation/lines/:id/split | Commit split action |
| POST | /reconciliation/lines/:id/create | Commit create-from-line action |
| POST | /reconciliation/lines/:id/transfer | Commit transfer action |
| POST | /reconciliation/lines/:id/defer | Defer line |
| POST | /reconciliation/lines/:id/exclude | Exclude line |
| GET | /reconciliation/lines/:id/audit | Fetch audit trail for line |

# Storage / Schema Changes

New tables to be defined during implementation:
- `reconciliation_match` -- links bank line to ledger transaction(s)
- `reconciliation_audit_event` -- structured audit trail per decision
- Extensions to existing `bank_transaction` table: reconciliation status, deferred reason, exclude reason

# Background Processing

No background processing for the core workflow. Candidate scoring is synchronous per line selection. A nightly job may be considered for re-scoring stale deferred items and updating exception priorities.

# Risks / Trade-offs

- **Candidate scoring performance**: For bank accounts with thousands of unreconciled transactions, scoring all candidates per line selection may be slow. Mitigation: pre-filter candidates by date range and amount tolerance before full scoring.
- **Evidence weight tuning**: Initial weights are heuristic. May need adjustment based on real-world usage. Mitigation: weights are configurable constants, not hardcoded in scoring logic.
- **Bulk accept safety**: Allowing bulk accept of exact matches trades thoroughness for speed. Mitigation: strict criteria (exact amount + exact reference + no conflict flags) and individual audit events.

# Alternatives Considered

1. **Automatic reconciliation without human review**: Rejected because accounting reconciliation requires defensible evidence. Auto-matching is offered as a suggestion, not an automatic action.
2. **Separate pages for each action mode**: Rejected because the single-screen workflow with keyboard navigation is significantly faster for high-volume reconciliation.
3. **Background candidate pre-computation**: Considered for performance but rejected for v1 because it adds staleness risk (candidates change as other lines are reconciled). May revisit if performance requires it.

# Test Strategy

- Unit tests for candidate scoring engine with various evidence combinations
- Unit tests for confidence bucketing logic
- Integration tests for each action mode commit path
- Integration tests for audit event emission
- End-to-end test: import bank statement, reconcile lines via all action modes, verify audit trail
- Performance test: queue loading with 1000+ unreconciled lines

# Rollout / Migration Notes

New feature, additive only. Requires bank_transaction table extensions (reconciliation status fields). No migration of existing data required -- existing bank transactions default to unreconciled status.
