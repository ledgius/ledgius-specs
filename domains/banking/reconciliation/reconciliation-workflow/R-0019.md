# Summary

End-to-end bank reconciliation workflow that establishes defensible relationships between imported bank statement evidence and ledger transactions. Supports matching, splitting, creating transactions, recording transfers, deferring, and excluding bank lines, with full audit trail and evidence-based candidate scoring.

# Problem / Business Context

Bank reconciliation is one of the highest-value accounting workflows. Its objective is not merely to clear bank lines but to establish a defensible, auditable relationship between imported bank statement evidence, ledger transactions, cash movement truth, and audit history. A good reconciliation workflow lets users move quickly on low-risk exact matches while preserving evidence and reducing ambiguity on complex items.

# Scope

- Bank line queue with priority-based ordering and quick filters
- Six action modes: Match, Split, Create Transaction, Transfer, Defer, Exclude
- Evidence-based candidate matching with confidence buckets (Exact, Strong, Possible, Weak)
- Pre-commit verification summary for every action
- Exception handling for ambiguous, stale, and conflicting items
- Duplicate detection across imports and matches
- Keyboard-driven workflow for high throughput
- Bulk accept for exact matches with no conflict flags
- Full audit trail for every reconciliation decision
- Reconciliation summary metrics and reporting

# Out of Scope

- Bank feed integration (automatic statement import from banks)
- Foreign currency reconciliation in v1 (flagged as future)
- AI/ML-based categorisation (candidate scoring uses rule-based evidence)
- Multi-entity consolidated reconciliation

# Actors / Users

- Bookkeeper -- clears large volumes of low-risk lines, investigates ambiguous lines
- Reviewer / Accountant -- audits why a line was matched, finds deferred/problem items, inspects split and creation logic
- Business Owner / Approver -- reviews unusual cash movements, approves or questions exceptions

# Preconditions

- Bank statement lines have been imported (via data import pipeline or manual upload)
- Chart of Accounts is configured with bank accounts
- User is authenticated with at least Bookkeeper role

# Functional Requirements

## Queue Management

- **RECON-001**: System shall display a bank line queue showing: date, amount, description/reference, source bank account, age since import, and risk/ambiguity indicator
- **RECON-002**: Default queue ordering shall prioritise: oldest unreconciled, high-value, ambiguous, stale deferred
- **RECON-003**: System shall provide quick filters: unmatched, deferred, high risk, duplicates suspected, recent imports, bank account, amount range, date range
- **RECON-004**: After resolving a line, the UI shall advance to the next logical line without full page reload

## Matching

- **RECON-005**: System shall display ranked candidate transactions with evidence for each (exact amount, date tolerance, reference similarity, contact correlation, previous behaviour pattern, clearing account path)
- **RECON-006**: Each candidate card shall show: transaction type, entity reference, amount, date, contact, state/status, evidence chips, confidence explanation
- **RECON-007**: Confidence buckets shall be: Exact, Strong, Possible, Weak -- emphasising reasons over raw probability
- **RECON-008**: Exact one-to-one matches may be accepted with a single click in expert mode
- **RECON-009**: Non-exact matches shall require explicit evidence review before confirmation
- **RECON-010**: System shall block matching to already-reconciled transactions

## Splitting

- **RECON-011**: Split editor shall provide: allocation grid, remaining amount indicator, tax code per line, optional linked source entities, notes/reason field
- **RECON-012**: Sum of split allocation lines must equal bank line amount before commit
- **RECON-013**: System shall block split commit when total mismatch, invalid account/tax code, or incompatible linked entity exists

## Create Transaction

- **RECON-014**: Create workflow shall support subtypes: payment receipt, bill payment, expense/cash purchase, transfer, journal-backed cash adjustment
- **RECON-015**: Create form shall be pre-filled from bank line data: amount, date, reference, inferred contact, inferred account suggestions
- **RECON-016**: Verification summary shall show: transaction type, ledger impact, tax treatment, related entity linking, post-commit reconciliation state

## Transfer

- **RECON-017**: Both sides of a transfer must be explicitly specified
- **RECON-018**: System shall strongly guard against duplicate transfers
- **RECON-019**: Where another side of a transfer already exists, system shall show transfer pairing evidence

## Defer

- **RECON-020**: Deferral shall require: reason category, optional note, optional expected review timing
- **RECON-021**: Deferred items shall remain in exception queues, display age since deferral, and be easily filterable

## Exclude

- **RECON-022**: Exclusion shall only be allowed if business policy defines valid exclusions (duplicate feed items, non-bookkeeping informational lines)
- **RECON-023**: Exclusion shall require: capability permission, explicit reason code, audit event

## Pre-Commit Verification

- **RECON-024**: Every action shall show a pre-commit summary before saving: bank amount, selected transaction total, residual delta, reconciliation status after commit, warnings if non-exact
- **RECON-025**: Commit shall be confirmed via explicit user action (Ctrl/Cmd+Enter or button click)

## Exception Handling

- **RECON-026**: System shall surface exceptions: no candidate found, multiple plausible candidates, duplicate risk, amount mismatch, stale deferred item, period lock conflict, already reconciled conflict
- **RECON-027**: Exceptions shall be prioritised by: financial risk, age, ambiguity, workflow blockage
- **RECON-028**: Each exception item shall show: short explanation, severity, next recommended action, linked bank line, time waiting unresolved

## Duplicate Detection

- **RECON-029**: System shall detect: same amount/date/reference imported more than once, bank line matched to transaction already reconciled elsewhere, created transaction similar to existing unreconciled one
- **RECON-030**: Duplicates shall be surfaced as warnings with linked possible duplicates and explicit user resolution path, not silently blocked

## Audit Trail

- **RECON-031**: Every reconciliation decision shall emit an audit event with: bank line ID, action chosen, actor, timestamp, confidence/evidence summary, linked transaction IDs, split details if relevant, defer/exclude reason if relevant

## Keyboard Workflow

- **RECON-032**: Keyboard shortcuts: J (next bank line), K (previous), A (accept top suggestion), S (open split mode), C (open create mode), D (defer), X (exclude), E (open audit/evidence), Ctrl/Cmd+Enter (commit)
- **RECON-033**: Keyboard shortcuts shall be disabled while editing text fields

## Bulk Actions

- **RECON-034**: Bulk accept shall be allowed only for exact amount matches with exact date/reference or extremely strong evidence and no duplicate/conflict flags
- **RECON-035**: Bulk actions shall generate individual audit events for each line

## Edge Cases

- **RECON-036**: System shall handle partial payments against invoices/bills with residual outstanding amount tracking and future matching path
- **RECON-037**: System shall surface overpayment/underpayment with options for credit, adjustment, or separate handling
- **RECON-038**: If a candidate action would post into a closed period, system shall surface the conflict with explanation and permitted resolution path

# Data / Entities / Fields

Primary entities: `BankLine` (imported statement line), `ReconciliationMatch` (link between bank line and ledger transaction), `ReconciliationAuditEvent` (audit trail)

Detailed schema to be defined in A-0019.

# UX / UI Behaviour

- **RECON-039**: Reconciliation screen shall have four regions: Bank Line Queue (left), Decision Workspace (centre), Candidate Transactions (right), Exceptions Rail (bottom or sidebar)
- **RECON-040**: Queue loads incrementally; selecting a bank line shall feel instant; candidate rendering shall not freeze on large histories
- **RECON-041**: Next-item progression shall avoid full-page refetch

# Validation Rules

- Split allocation total must equal bank line amount (with tolerance < $0.01)
- Tax code on each split line must be a valid Ledgius tax code
- Transfer must specify different source and destination accounts
- Exclude reason code must be from approved list
- Cannot match to an already-reconciled transaction

# Error Handling

- Candidate scoring failures log the error and return empty candidates with an exception flag
- Commit failures roll back and display full error detail in feedback console
- Period lock conflicts display the locked period and suggest alternative posting dates

# Security / Permissions

- Minimum role: Bookkeeper (match, split, create, defer)
- Exclude action requires elevated permission
- All reconciliation decisions are audit-logged with actor identity

# Acceptance Criteria

- [ ] Users can clear exact matches quickly without sacrificing evidence
- [ ] Ambiguous lines expose enough evidence for confident decisions
- [ ] Every action has a visible pre-commit summary
- [ ] Deferred and risky items remain visible and searchable
- [ ] All reconciliation decisions are auditable
- [ ] Keyboard workflow supports full reconciliation without mouse
- [ ] Bulk accept works for exact matches with audit trail
- [ ] Split editor enforces amount balance invariant

# Open Questions

1. **Foreign currency**: How should FX differences be treated during reconciliation? Separate FX gain/loss account?
2. **Closed period resolution**: Should the system allow temporary period unlock for reconciliation, or force date adjustment?
3. **Candidate scoring weights**: Should evidence weights be configurable per tenant, or fixed?
