# Summary

End-to-end bank reconciliation workflow that establishes defensible, auditable relationships between imported bank statement evidence and ledger transactions. Supports a staged multi-pass matching pipeline, one-to-many and many-to-one composition, keyboard-first three-pane workstation UX, controlled bulk actions, full exception queue management, and complete audit trail with per-decision explainability.

# Problem / Business Context

Bank reconciliation is one of the highest-value accounting workflows. Its objective is not merely to clear bank lines but to establish a defensible, auditable relationship between imported bank statement evidence, ledger transactions, cash movement truth, and audit history.

The core design principle is: auto-match the obvious with deterministic rules, queue only the ambiguous items for human review, explain every match decision, and learn from approved human actions — but never hide the reasoning.

A good reconciliation workflow lets users move quickly on low-risk exact matches while preserving evidence and reducing ambiguity on complex items. Controllers should have a control surface, not just a data-entry tool.

# Scope

- Canonical normalised bank transaction model (raw + normalised fields)
- Five-pass staged matching pipeline with deterministic and probabilistic passes
- Six-level matching hierarchy from machine IDs down to free-text signals
- Seven canonical transaction pattern families including one-to-many and many-to-many
- Weighted scoring model with hard and soft controls
- Text normalisation and vendor alias dictionary
- Approved-precedent memory (visible, explainable learning)
- Three-pane keyboard-first workstation UX
- Priority-based exception queue with structured reason codes
- Bulk actions with mandatory preview confirmation
- Per-decision explainability on every row
- Close-readiness dashboard and metrics
- Full audit trail for every reconciliation decision
- Internal transfers, processor payouts, bank fees, returned items, and FX as first-class transaction types

# Out of Scope

- Bank feed integration (automatic statement import from banks) — covered by R-0017
- Multi-entity consolidated reconciliation
- Silent auto-posting of material transactions without user review
- Opaque AI compositions that replace accounting policy or bypass controls

# Actors / Users

- **Bookkeeper** — clears large volumes of low-risk lines, investigates ambiguous lines
- **Reviewer / Accountant** — audits why a line was matched, finds deferred/problem items, inspects split and creation logic
- **Business Owner / Approver** — reviews unusual cash movements, approves or questions exceptions
- **Controller** — monitors close-readiness dashboard, manages exception burn-down

# Preconditions

- Bank statement lines have been imported (via data import pipeline R-0017 or manual upload)
- Chart of Accounts is configured with bank accounts
- User is authenticated with at least Bookkeeper role

# Section 1: Data Model — Canonical Normalised Transaction

Never reconcile raw text directly. Every imported bank line must be normalised into a canonical record before any matching pass runs.

## Required Canonical Fields

| Field | Description |
|---|---|
| `bank_account_id` | Owning bank account in the chart of accounts |
| `statement_id` | Source statement batch identifier |
| `booking_date` | Date the bank booked the transaction |
| `value_date` | Value date for interest/settlement purposes |
| `amount` | Absolute amount |
| `currency` | ISO 4217 currency code |
| `debit_credit_sign` | Explicit sign: DEBIT or CREDIT |
| `bank_transaction_code` | Structured bank transaction type code (ISO 20022 or bank-specific) |
| `counterparty_name` | Raw counterparty name from bank |
| `reference` | Payment reference field |
| `remittance_info` | Remittance information / structured reference |
| `end_to_end_id` | End-to-end transaction identifier (e.g. BECS, SWIFT) |
| `check_number` | Check/cheque number if applicable |
| `channel` | Payment channel (BPAY, EFT, card, ATM, direct debit, etc.) |
| `source_type` | Import origin (bank feed, OFX, CSV, manual) |
| `raw_description` | Original unmodified bank memo line |
| `normalized_description` | Normalised tokens after text processing (see Section 6) |
| `confidence_metadata` | JSON blob of normalisation confidence, extracted tokens, regex matches |
| `duplicate_fingerprint` | Deterministic hash used for duplicate detection |

## Functional Requirements

- **RECON-001**: The system shall store both raw and normalised data for every imported bank transaction. The raw description is retained as a fallback signal, not replaced.
- **RECON-002**: The `duplicate_fingerprint` shall be computed deterministically from amount, date, counterparty, and reference so that re-imports of the same statement detect duplicates without manual review.
- **RECON-003**: Normalisation (Section 6) shall run at import time and store results in `normalized_description` and `confidence_metadata` before any matching pass executes.

## Acceptance Criteria

- [ ] Every bank transaction stores both `raw_description` and `normalized_description`
- [ ] The raw description is preserved unchanged and is visible in the detail pane
- [ ] `duplicate_fingerprint` catches re-imported duplicates before they enter the match queue

# Section 2: Staged Matching Pipeline (5 Passes)

Matching runs in sequential passes. Items matched in an earlier pass skip all later passes. Every match record stores which pass produced it.

## Pass Definitions

### Pass 1 — Exact Deterministic

Criteria (any one sufficient):
- Exact `end_to_end_id` or `bank_transaction_code` match to a ledger transaction
- Exact `check_number` match
- Exact payment reference matches invoice number, receipt number, or payroll batch ID
- Exact `external_bank_txn_id` match
- Exact amount + same booking date + exact normalised counterparty name

Auto-postable: Yes (subject to hard controls in Section 14)

### Pass 2 — Constrained Tolerance

Criteria:
- Exact amount + date within ±1–3 days (configurable tolerance per account)
- Exact amount + normalised reference similarity above high threshold (configurable)
- Exact amount + same counterparty fingerprint + date within ±3 days

Auto-postable: Only if tolerance policy permits; otherwise queued for review

### Pass 3 — One-to-Many and Many-to-One Composition

Handles:
- Bank deposit = sum of multiple receipts/invoices
- Bank withdrawal = component fees + payment + tax
- Batch payment = many individual ledger lines within a settlement window
- Marketplace/processor payout netting

Composition candidates are surfaced to the user for confirmation before posting (see Section 15 for processor payout workflow).

Auto-postable: No. Always requires user confirmation.

### Pass 4 — Rule-Based Generation

Handles:
- Bank fee line → auto-create expense transaction from template
- Interest credit → auto-create income transaction from template
- Internal transfer between owned accounts → create transfer pair
- Card settlement → link to card clearing account

Auto-postable: Yes for known fee/interest templates with configured accounts. Requires explicit template configuration.

### Pass 5 — Probabilistic Suggestion Only

Uses scoring model (Section 5) and approved-precedent memory (Section 7) to rank candidates. Results are surfaced as suggestions only. Never auto-posted unless confidence is in the highest band AND policy explicitly permits it.

Auto-postable: Never by default. Configurable opt-in for narrow use cases with audit logging.

### Pipeline status rules

When the pipeline completes processing a bank transaction, it MUST set the reconciliation status according to these rules:

| Condition | Status | Rationale |
|-----------|--------|-----------|
| Pass 1-2 match found, auto-approved | `auto_matched` | High-confidence deterministic match |
| Pass 1-2 match found, not auto-approved | `needs_review` | Deterministic but below auto-approve threshold |
| Pass 3 composition found | `needs_review` | Compositions always need human confirmation |
| Pass 4 rule match, auto-approved | `auto_matched` | Known recurring pattern |
| Pass 5 candidates found | `needs_review` | Probabilistic — always needs human review |
| Pass 5 no candidates found | `needs_review` | No match exists yet — normal for fresh imports |
| Ambiguous (multiple equally strong candidates) | `needs_review` | Needs human decision, not an exception |

**Critical rule**: `exception` status is reserved for genuine operational problems — NOT for the absence of matching ledger entries. A fresh bank import with no invoices/bills entered yet is a normal state, not an exception. The bookkeeper will create ledger entries and re-run matching, or the system will auto-create entries from bank patterns (see Section 4b).

- **RECON-013**: The pipeline shall NEVER set `exception` status solely because no candidate ledger entries exist.
- **RECON-014**: The `missing_ledger` exception shall only be raised when the system has positive evidence that a match SHOULD exist (e.g. a known recurring pattern that previously matched but no longer does).

### Section 4b: Auto-creation of ledger entries from bank transactions

When no matching ledger entry exists, the system may auto-create journal entries from bank transaction patterns:

1. **Pattern recognition**: the system recognises bank descriptions (e.g. "WOOLWORTHS" → Expense, "SQUARE" → Revenue) using the approved-precedent memory (Section 7) and bank rules (Pass 4).
2. **Auto-creation**: a journal entry is created with the matched account, flagged with `source: "bank_auto"` and `status: "draft"`.
3. **Verification required**: auto-created entries have a visible `auto-created — please verify` badge. They are NOT posted until the bookkeeper confirms.
4. **Learning**: when the bookkeeper confirms an auto-created entry, the pattern is reinforced in precedent memory. Future occurrences of the same pattern get higher confidence.
5. **Never auto-post**: auto-created entries remain in draft until explicitly approved. The system suggests, the human decides.

- **RECON-015**: Auto-created journal entries shall be flagged with `source = 'bank_auto'` and `status = 'draft'`.
- **RECON-016**: Auto-created entries shall display a "Created from bank transaction — please verify" indicator.
- **RECON-017**: On confirmation, the system shall update the precedent memory to reinforce the pattern.
- **RECON-018**: Auto-creation shall only occur for patterns with at least 2 prior approvals (to prevent spurious entries on first occurrence).

## Functional Requirements

- **RECON-010**: Each pass shall run in strict order (1 through 5). Items resolved in pass N are excluded from passes N+1 through 5.
- **RECON-011**: Every match record shall store the pass number that produced it, the rule or model version used, and the score components.
- **RECON-012**: Pass 5 suggestions shall be visually distinguished from deterministic matches (Passes 1–2) and rule-generated entries (Pass 4).

## Acceptance Criteria

- [ ] Exact deterministic matches (Pass 1) are resolved without user intervention (subject to hard controls)
- [ ] Pass 3 compositions are always presented for user confirmation before posting
- [ ] Every match record includes pass number and evidence payload
- [ ] Pass 5 suggestions are never silently posted
- [ ] Pipeline never sets `exception` for missing ledger entries — uses `needs_review` instead
- [ ] Auto-created journal entries are flagged as draft with `bank_auto` source

# Section 3: Matching Hierarchy (6 Levels)

Evidence is evaluated in strict priority order. Lower-level signals only break ties when higher-level signals are absent or tied.

| Level | Signal Type | Examples |
|---|---|---|
| 1 | Unique machine identifiers | Bank transaction ID, end-to-end ID (BECS/SWIFT), check number |
| 2 | Structured business references | Invoice number, receipt number, payroll batch ID, purchase order |
| 3 | Strong numeric anchors | Exact amount + currency + sign + tax split |
| 4 | Temporal anchors | Booking date, value date, settlement window, weekday/month-end patterns |
| 5 | Counterparty signals | Normalised name, bank account fingerprint, known alias mappings |
| 6 | Free-text pattern signals | Tokenised memo, regex-extracted codes, fuzzy similarity score |

## Functional Requirements

- **RECON-013**: Text similarity (Level 6) shall never be the sole basis for auto-posting. It is a tiebreaker and suggestion signal only.
- **RECON-014**: Machine IDs (Level 1) and structured business references (Level 2) shall always take priority over amount-only matches.
- **RECON-015**: The evidence payload returned with every candidate shall identify which levels contributed to the score and at what weight.

## Acceptance Criteria

- [ ] A match on a machine ID (Level 1) overrides any amount+text match
- [ ] Evidence payloads identify the contributing levels for every candidate
- [ ] Text-only matches are never auto-posted

# Section 4: Canonical Transaction Pattern Families

The system handles seven distinct pattern families as first-class objects, not edge cases.

## Pattern Definitions

### 1. One-to-One Exact
Single bank line maps to single ledger transaction. Amount, date, reference all match. The baseline case.

### 2. One-to-Many Aggregate
Single bank deposit or withdrawal maps to multiple ledger transactions. Examples:
- Batched deposit: multiple invoices received as one bank credit
- Marketplace payout: Stripe/Square net settlement covering many sales
- POS end-of-day settlement
- Payroll funding: single debit covers many employee payments

### 3. Many-to-One Split
Multiple bank lines map to a single ledger transaction. Examples:
- Single AP bill paid in partial instalments
- Multi-tranche loan drawdown

### 4. Many-to-Many Grouped Window
Multiple bank lines match to multiple ledger transactions within a settlement window. Examples:
- Payment processor clearing over a rolling N-day window
- Treasury sweeps and cash pooling
- Intercompany netting settlements

### 5. Recurring Pattern Recognition
The bank line matches a known recurring template based on counterparty, amount band, and period. Examples:
- Monthly subscriptions (exact amount, exact counterparty, ~28–31 day cadence)
- Rent and lease payments
- Loan repayments (principal + interest split)
- Periodic bank fees

### 6. Generated Counterpart
No existing ledger transaction exists. The bank line triggers automatic creation from a configured template. Examples:
- Bank interest credit → Interest Income
- Monthly account fee → Bank Charges Expense
- FX conversion gain/loss
- Rounding differences

### 7. Reversal / Correction
Bank line is the reversal or correction of a prior transaction. The system links the original transaction, the reversal, and any associated fee or recovery. Examples:
- NSF/dishonour return
- Chargeback
- Failed direct debit payout
- Returned item

## Functional Requirements

- **RECON-016**: The system shall handle all seven pattern families. Patterns 2, 3, and 4 shall be first-class workflows, not workarounds.
- **RECON-017**: Pattern 7 (reversal) shall create a lifecycle link between the original transaction, the reversal bank line, and any fee lines — visible as a linked chain in the detail pane.
- **RECON-018**: Pattern 5 (recurring) shall surface the prior approval history as evidence in the candidate card.

## Acceptance Criteria

- [ ] One-to-many aggregate matching can be completed without leaving the reconciliation screen
- [ ] Reversal bank lines display the linked original transaction in the detail pane
- [ ] Recurring pattern candidates show the count of prior matching approvals

# Section 5: Scoring Model

Every candidate match receives a weighted score. Hard controls can reject candidates regardless of score.

## Score Components

| Signal | Points |
|---|---|
| Transaction ID exact match | +100 |
| Check number exact match | +95 |
| Invoice/receipt reference exact match | +90 |
| Exact amount match | +50 |
| Date within 1 day | +20 |
| Date within 3 days | +10 |
| Same counterparty fingerprint | +30 |
| Strong memo similarity (normalised token overlap) | +20 |
| Known recurring template match | +15 |
| Approved-precedent memory hit (Section 7) | +10 |

## Hard Rejects (Score Irrelevant)

Any candidate is immediately rejected if:
- Currency mismatch between bank line and ledger transaction
- Sign mismatch (debit bank line matched to a credit ledger transaction or vice versa)
- Candidate transaction is already reconciled to another bank line
- Candidate belongs to a different entity or bank account context
- Amount delta exceeds configured materiality tolerance for the account
- Multiple candidates have equally high scores with no tiebreaker available (surfaced for human selection)

## Confidence Bands

| Band | Score Range | Default Action |
|---|---|---|
| Exact | 140+ | Auto-post if Pass 1 and hard controls pass |
| Strong | 100–139 | Queue for one-click confirmation |
| Possible | 60–99 | Queue with evidence for review |
| Weak | < 60 | Surface as suggestion only |

## Functional Requirements

- **RECON-019**: Every candidate presented in the UI shall include a score breakdown and a human-readable explanation payload (see Section 12).
- **RECON-020**: Hard reject conditions shall be evaluated before score computation and logged with the reason code.
- **RECON-021**: Confidence bands shall be configurable per bank account and entity (thresholds, not signal weights).

## Acceptance Criteria

- [ ] Every suggested match returns a score breakdown and explanation payload
- [ ] Hard reject conditions block candidates even at high scores
- [ ] Confidence band thresholds are configurable per account

# Section 6: Text Normalisation

Normalisation is applied deterministically at import time. Both original and normalised forms are stored.

## Normalisation Steps (Applied in Order)

1. Uppercase the entire string
2. Remove punctuation (retain spaces, digits, letters)
3. Collapse multiple whitespace to single space
4. Trim leading and trailing whitespace
5. Map known abbreviations to canonical forms (e.g. `PYMT` → `PAYMENT`, `INV` → `INVOICE`, `CHQ` → `CHEQUE`)
6. Strip bank-specific prefixes and suffixes (e.g. `DIRECT DEBIT - `, ` - REF`)
7. Apply vendor alias dictionary (e.g. `AMZN MKTP` → `AMAZON`, `SQ *` → `SQUARE`)
8. Apply regex extraction to identify structured codes:
   - Invoice numbers: `INV[-\s]?\d+`, `INVOICE\s+\d+`
   - Check numbers: `CHQ\s*\d+`, `CHECK\s*\d+`
   - Order numbers: `ORD[-\s]?\d+`, `PO\s*\d+`
   - Tax reference: `ABN\s*\d+`, `GST\s*REF`
   - Customer/account codes: configurable per tenant
   - Settlement IDs: `SETTLE[-\s]?\w+`, `BATCH\s*\w+`
9. Store extracted tokens in `confidence_metadata` alongside the normalised string

## Functional Requirements

- **RECON-022**: Normalisation shall be deterministic: identical raw input always produces identical normalised output.
- **RECON-023**: Both `raw_description` and `normalized_description` shall be stored and both shall be visible in the detail inspector pane.
- **RECON-024**: The vendor alias dictionary shall be tenant-configurable and editable from the reconciliation workstation (see Section 8, right pane actions).
- **RECON-025**: Extracted token sets shall be stored in `confidence_metadata` for use by the scoring model without re-parsing at match time.

## Acceptance Criteria

- [ ] Normalisation is deterministic and reproducible across re-imports
- [ ] Both raw and normalised descriptions are visible in the UI
- [ ] Vendor alias additions from the reconciliation screen persist and apply to future imports

# Section 7: Approved-Precedent Memory

The system maintains a memory of human-approved reconciliation decisions and uses that history to increase confidence scores on future similar items.

## Memory Types

| Type | Description | Example |
|---|---|---|
| Remembered classification | A specific normalised memo pattern was posted to a specific account | `BANK FEE MONTHLY` → Bank Charges Expense |
| Remembered match rule | A counterparty + amount band was matched to a specific transaction type | Monthly direct debit from `PROPTECH PTY LTD` → Rent Expense |
| Remembered counterparty mapping | A bank counterparty alias maps to a known contact | `AMZN MKTP AU` → Amazon Australia (contact ID) |

## Functional Requirements

- **RECON-026**: Each approved action shall write a precedent record containing: normalised memo tokens, counterparty fingerprint, amount band, account posted to, tax code applied, decision type, and approver identity.
- **RECON-027**: Future matches that trigger a precedent hit shall surface the hit as a named evidence item: "Matches 12 prior approvals: BANK FEE MONTHLY → Bank Charges Expense".
- **RECON-028**: Precedent memory shall be visible and manageable — users can review, disable, or delete precedent rules from an administration view.
- **RECON-029**: Precedent memory shall never silently override hard controls (Section 14). A high-precedent match that violates a hard control is still rejected.

## Acceptance Criteria

- [ ] Approved actions write a precedent record
- [ ] Future matches that hit a precedent show the match count and description as evidence
- [ ] Precedent rules are visible and manageable by administrators
- [ ] Precedent memory cannot override hard controls

# Section 8: Three-Pane Workstation UX

The reconciliation workstation is divided into three persistent panes. No navigation away from this screen is required to complete a reconciliation action.

## Left Pane — Bank Statement Queue

Displays: date, amount (with currency), payee / reference, status chip, confidence chip, match-type chip, anomaly icon (duplicate, high-value, aged, reversal).

Sort options (default: highest business risk first):
- Unmatched high-value
- Ambiguous (multiple candidates)
- Aged unreconciled (> N days)
- Possible duplicates
- Likely transfers
- Likely fees / interest
- Stale exceptions
- Chronological (explicit user selection)

Filter chips: Bank account, date range, amount range, status, confidence band, transaction type.

## Centre Pane — Best Match Candidates

Displays ranked candidates for the selected bank line:
- Transaction type icon, entity reference, amount, date, contact name
- Evidence chips per candidate (e.g. "Exact amount", "INV-10483 matched", "Date +1 day")
- Confidence band label and score
- One-click accept, split allocation, exclude, or skip actions

For composition matches (Pass 3): shows the proposed set of transactions with individual amounts and the summed total vs. bank line amount.

## Right Pane — Detail and Action Inspector

Displays:
- Raw description (original bank text)
- Normalised description and extracted tokens
- Confidence metadata JSON (expandable)
- Historical matches for this counterparty (last N approvals)
- Approved-precedent hits with match count

Action buttons available without leaving the screen:
- Create adjustment / bank fee / FX difference / write-off
- Create internal transfer
- Add vendor alias
- Add or edit match rule
- Escalate to reviewer (with comment)
- Snooze / set revisit date
- Mark as exception with structured reason code

## Functional Requirements

- **RECON-030**: All common reconciliation actions (accept, split, create, transfer, defer, exclude, escalate) shall be completable without navigating away from the workstation.
- **RECON-031**: The left pane queue shall update instantly (optimistic UI) when a line is resolved. The next logical line shall be auto-selected.
- **RECON-032**: Candidate sets for the next 10–20 bank lines shall be preloaded to eliminate perceptible latency when advancing through the queue.
- **RECON-033**: No full-page refresh shall occur during normal reconciliation workflow. Row resolution shall animate smoothly.

## Acceptance Criteria

- [ ] The moment the user has to leave the workstation screen to create a journal entry, the UX has failed
- [ ] All three panes are visible and functional simultaneously
- [ ] Resolving a line auto-advances to the next without page reload
- [ ] Candidate sets are preloaded for the next items in queue

# Section 10: Sort and Priority

The default queue sort is exception burn-down priority, not chronological order.

## Default Sort Algorithm

Priority score = f(unreconciled age, amount, confidence band, exception status, duplicate risk flag)

High-risk items (large amounts, long aged, duplicate candidates, stale exceptions) surface first.

## Sort Options

Available from a sort dropdown, each with secondary sort by date descending:

1. **Highest Risk First** (default) — composite risk score
2. **Unmatched High-Value** — unmatched items sorted by absolute amount descending
3. **Ambiguous** — items with 2+ candidates at similar confidence
4. **Aged Unreconciled** — oldest unmatched items first (> 7 / 14 / 30 days, configurable)
5. **Possible Duplicates** — flagged duplicate candidates
6. **Likely Transfers** — probable internal transfer pairs
7. **Likely Fees / Interest** — probable bank fee and interest lines
8. **Stale Exceptions** — exceptions not updated in > N days
9. **Chronological** — booking date ascending

## Functional Requirements

- **RECON-037**: The default sort shall be "Highest Risk First" on first load of the workstation.
- **RECON-038**: The selected sort shall persist within the session but reset to default on new session.
- **RECON-039**: The queue shall display a count and total unreconciled amount for the current filter+sort combination.

## Acceptance Criteria

- [ ] Default queue order surfaces the highest business risk items first
- [ ] All nine sort options are available from the sort control
- [ ] Queue displays count and total amount for the current view

# Section 11: Bulk Actions (Controlled)

Bulk actions are permitted only for low-risk, deterministic scenarios and always require a preview confirmation before execution.

## Permitted Bulk Actions

| Action | Eligibility Criteria |
|---|---|
| Bulk accept exact matches | Pass 1 (exact deterministic) only. No duplicate flag. No conflict flag. No hard rejects. |
| Auto-classify transfer pairs | Both sides identified as owned-account transfers within ±1 day and exact amount. Requires user confirmation. |
| Apply alias mapping | Apply a new vendor alias to all unresolved lines matching the raw counterparty. Requires user confirmation. |

## Preview Requirement

Every bulk action shall show a preview summary before execution:
- Count of lines affected
- Total amount
- Action description (e.g. "Accept 47 exact matches totalling $143,280.00")
- Warning count (any lines with secondary flags)

Execution requires explicit confirmation (button click or `Ctrl/Cmd+Enter`).

## Functional Requirements

- **RECON-040**: Bulk accept shall only be available for lines in confidence band "Exact" with no duplicate, conflict, or hard-reject flags.
- **RECON-041**: Bulk actions shall generate individual audit events for each line resolved (not a single batch event).
- **RECON-042**: No bulk action shall execute without the preview confirmation step.

## Acceptance Criteria

- [ ] Bulk accept is restricted to Pass 1 exact matches with no flags
- [ ] Every bulk action requires a preview confirmation with count and total amount
- [ ] Bulk actions generate individual audit events per line

# Section 12: Explainability

Every candidate row and every resolved line must show the human-readable reason for the match or suggestion. No row may ever display only "Suggested match" without evidence.

## Explanation Payload Structure

Each candidate includes an explanation payload:

```json
{
  "summary": "Exact amount, reference INV-10483, date +1 day",
  "pass": 2,
  "score": 130,
  "confidence_band": "Strong",
  "evidence": [
    {"signal": "invoice_reference_exact", "value": "INV-10483", "points": 90},
    {"signal": "amount_exact", "value": "1250.00", "points": 50},
    {"signal": "date_within_1_day", "value": "booking_date 2026-04-11 vs ledger 2026-04-10", "points": 20},
    {"signal": "precedent_memory", "value": "12 prior approvals: Tech Solutions Pty Ltd → Consulting Income", "points": 10}
  ],
  "hard_rejects_evaluated": [],
  "pass_description": "Constrained tolerance: exact amount + reference similarity"
}
```

## Functional Requirements

- **RECON-043**: Every candidate presented in the centre pane shall display a one-line summary explanation and expandable evidence chip list.
- **RECON-044**: Resolved lines in the audit trail shall store the explanation payload at the time of approval, not regenerated retrospectively.
- **RECON-045**: Exception rows shall display a structured reason explaining why they were not auto-resolved (e.g. "Multiple candidates with equal score — human selection required").

## Example Explanations

- "Exact amount, reference INV-10483, date +1 day"
- "Likely transfer: same amount in owned account 2026-04-11, 0 days apart"
- "Recurring fee pattern matched 12 prior approvals: BANK FEE MONTHLY → Bank Charges Expense"
- "Composition: $4,800 = Receipt #1022 ($2,100) + Receipt #1031 ($1,700) + Receipt #1044 ($1,000)"
- "Reversal: links to original payment #TXN-8820 dated 2026-03-28"

## Acceptance Criteria

- [ ] No candidate row shows only "Suggested match" — every row has a summary explanation
- [ ] The explanation payload is stored in the audit trail at decision time
- [ ] Exception rows explain specifically why they were not auto-resolved

| Unreconciled value | Total amount of unresolved bank lines |

| Duplicate candidates | Count of flagged potential duplicates pending resolution |
| Created adjustments | Count and value of bank-fee and interest auto-creations |
| Auto-matched count | Lines resolved by Passes 1 and 4 (deterministic) |
| Manually matched count | Lines resolved by user action |
| Rule hit rate | Percentage of lines resolved by match rules (Passes 1–4) |
| False-positive override rate | Percentage of auto-matches subsequently overridden by user |

## Functional Requirements

- **RECON-046**: The close-readiness dashboard shall be accessible from the workstation header without navigating away from the reconciliation screen.
- **RECON-047**: Dashboard metrics shall refresh in near-real-time as lines are resolved.
- **RECON-048**: The false-positive override rate shall be tracked and surfaced to allow tuning of scoring thresholds.

## Acceptance Criteria

- [ ] Controllers can assess close-readiness without leaving the reconciliation workflow
- [ ] All metrics listed in the table are present on the dashboard
- [ ] False-positive override rate is tracked and visible

# Section 14: Hard and Soft Controls

Controls govern what actions the system may take automatically and what requires human confirmation.

## Hard Controls (Non-Negotiable)

The system shall never automatically post a match if any of the following conditions apply:

- Currency of bank line differs from currency of candidate ledger transaction
- Debit/credit sign of bank line differs from the sign of the candidate ledger transaction
- Candidate transaction is already reconciled to another bank line
- Candidate transaction belongs to a different entity or bank account context
- Amount delta between bank line and candidate exceeds the configured materiality tolerance for the account
- Multiple candidates exist with equally high scores and no deterministic tiebreaker

## Soft Controls (Require Confirmation)

The system shall require explicit user confirmation before posting if any of the following apply:

- Inferred composition (Pass 3 aggregate match)
- Match driven primarily by memo/text scoring (Level 6 signals dominant)
- New vendor alias being applied for the first time
- Weak approved-precedent (fewer than 3 prior approvals)
- Adjustment posting into a prior accounting period (material-period adjustment)
- Match to a transaction flagged with a duplicate risk indicator

## Configurable Tolerances

The following may be configured per bank account, entity, payment method, transaction class, or materiality threshold:

- Amount delta tolerance (absolute and percentage)
- Date window for constrained-tolerance matching (±1 / ±3 / ±5 days)
- Confidence band thresholds for auto-posting
- Minimum precedent count for elevated confidence
- Days before an unreconciled item is classified as aged

## Functional Requirements

- **RECON-049**: Hard controls shall be enforced at the engine level, not just the UI level. A hard control violation shall block the action and log a structured rejection event.
- **RECON-050**: Soft controls shall surface a confirmation dialog with the specific soft control triggered and a brief explanation.
- **RECON-051**: Tolerance configuration shall be accessible to administrators and shall not require code changes.

## Acceptance Criteria

- [ ] Hard controls are enforced at engine level — they cannot be bypassed by API calls
- [ ] Soft control confirmations name the specific condition requiring approval
- [ ] Tolerance settings are configurable per account without code changes

# Section 15: Real-World Transaction Types

Each transaction type has a dedicated handling path.

## Internal Transfers

Internal transfers between owned bank accounts are first-class objects, not hacks.

Detection: opposite-sign, exact-amount pair across two owned accounts within a configurable date window (default ±1 day).

Workflow: System proposes the transfer pair. User confirms. Both bank lines are resolved simultaneously and linked as a transfer pair in the audit trail.

## Processor Payouts (Stripe, Square, PayPal, etc.)

Dedicated netting workflow:
1. Identify the payout bank line (credit to bank account)
2. Retrieve the processor reconciliation file (uploaded or API-fetched)
3. Match gross sales, refunds, fees, and net payout using the processor's settlement report
4. Create the netting journal if the processor file is available, or surface the composition for manual allocation

## Bank Fees and Interest

Auto-postable from configured templates (Pass 4):
- Template defines: normalised memo pattern, debit/credit, account, tax code
- Matching bank line triggers auto-creation of the ledger transaction
- Auto-created entry is flagged as system-generated and visible in the audit trail

## Returned Items and Reversals

Lifecycle linking — every returned item creates a chain:
`Original transaction → Return bank line → Reversal ledger entry → Associated fee → Recovery (if applicable)`

All chain members are linked by a `lifecycle_group_id` and displayed as a chain in the detail pane.

## Foreign Exchange Transactions

Dedicated FX handling path:
1. Source transaction in foreign currency
2. Settlement in domestic currency
3. FX difference posted to configured FX Gain/Loss account
4. Bank conversion fee (if separate line) linked to same FX event
5. Rounding difference posted to Rounding account

FX transactions shall never be matched using amount-only matching (currency mismatch is a hard reject).

## Functional Requirements

- **RECON-052**: Internal transfer detection shall be automatic but always require user confirmation before posting.
- **RECON-053**: Returned items shall generate a `lifecycle_group_id` linking all related bank lines and ledger entries.
- **RECON-054**: FX transactions shall be handled in a dedicated FX reconciliation path with explicit posting of FX difference and fee.

## Acceptance Criteria

- [ ] Internal transfers are proposed automatically and confirmed in one action
- [ ] Returned item chains are linked and visible as a group in the detail pane
- [ ] FX differences are posted to the configured account, not hidden in rounding

# Section 19: AI Role (Constrained)

AI assistance is explicitly scoped. The boundary between assisted and autonomous is non-negotiable.

## Permitted AI Roles

- Ranking candidates within Pass 5 (probabilistic suggestion)
- Parsing and extracting tokens from bank memo text (supporting normalisation)
- Suggesting vendor aliases based on memo patterns
- Identifying recurring pattern candidates
- Anomaly detection (flagging unusual amounts, new counterparties, pattern breaks)
- Suggested categorisation for new transaction types

## Prohibited AI Roles

- Silent posting of material transactions without user review
- Opaque composition suggestions that cannot be explained in plain terms
- Replacing accounting policy decisions
- Bypassing hard or soft controls (Section 14)
- Creating audit records without an identified human approver

## Marketing and Labelling

- Deterministic matches (Passes 1–4) shall be described as "Matched" or "Auto-matched"
- Probabilistic suggestions (Pass 5) shall be described as "Suggested" with explicit evidence
- The system shall never be marketed as "AI-reconciled" for deterministic rule-based matches
- AI-assisted features shall be labelled "Assisted reconciliation"

## Functional Requirements

- **RECON-066**: AI-generated suggestions (Pass 5) shall be visually and textually distinguished from rule-based matches (Passes 1–4).
- **RECON-067**: Every AI-generated suggestion shall include the explanation payload (Section 12) — no opaque black-box suggestions.
- **RECON-068**: AI model versions used for scoring shall be stored in the audit trail (Section 18, `model_version` field).

## Acceptance Criteria

- [ ] AI suggestions are visually distinguished from deterministic matches
- [ ] No AI suggestion is auto-posted for material transactions
- [ ] Every AI suggestion includes a human-readable explanation payload

# Section 20: Ultimate Workflow — Inbox Stages and Review Modes

## Inbox Stages (Lifecycle)

Every bank line progresses through the following stages:

```
Imported → Auto-matched → Needs Review → Exception → Resolved → Closed in Period
```

| Stage | Description |
|---|---|
| Imported | Bank line received from import pipeline, not yet matched |
| Auto-matched | System has produced a deterministic match pending commit |
| Needs Review | System has produced suggestions but human decision required |
| Exception | No viable match found, or hard/soft control triggered |
| Resolved | Bank line reconciled — action committed and audit trail written |
| Closed in Period | Accounting period containing this line has been closed |

## Review Modes

Users can filter the queue into focused review modes:

| Mode | Description |
|---|---|
| Exact Matches | Pass 1 deterministic matches pending bulk confirmation |
| Ambiguous | Lines with multiple similarly-scored candidates |
| Transfers | Probable internal transfer pairs |
| Fees / Interest | Probable bank fee and interest lines |
| Aggregates / Nets | Composition candidates (Pass 3 one-to-many / many-to-many) |
| Missing Ledger | No candidate found — possible missing transaction |
| Possible Duplicates | Duplicate fingerprint flags |
| Prior-Period Carryovers | Aged exceptions from prior periods not yet resolved |

## Smart Actions per Row

Each bank line in the queue exposes contextual smart actions based on its current state and best candidate:

| Action | Description |
|---|---|
| Accept | Accept the top candidate match and commit |
| Accept + Remember | Accept and write an approved-precedent record |
| Split | Open split allocation editor |
| Mark Transfer | Identify as one side of an internal transfer |
| Create Adjustment | Create fee, interest, write-off, or rounding adjustment |
| Exclude Duplicate | Mark as a duplicate import and exclude with audit reason |
| Escalate | Assign to reviewer with structured reason and comment |
| Undo | Reverse the most recent action on this line |

## Functional Requirements

- **RECON-069**: Every bank line shall have a clearly displayed current stage (Imported through Closed in Period).
- **RECON-070**: Review modes shall be selectable from the queue filter bar and shall persist within the session.
- **RECON-071**: Smart actions shall be context-sensitive — only actions valid for the current state of the bank line and the user's role shall be shown.
- **RECON-072**: "Accept + Remember" shall write a precedent record (Section 7) and display a confirmation: "Remembered: [normalised pattern] → [account]".

## Acceptance Criteria

- [ ] All six inbox stages are visible on bank lines and in the queue
- [ ] All eight review modes are available from the filter bar
- [ ] All eight smart actions are implemented and role-gated appropriately
- [ ] "Accept + Remember" writes a precedent record and confirms to the user

# Validation Rules

- Split allocation total must equal bank line amount (tolerance < $0.01)
- Tax code on each split line must be a valid Ledgius tax code
- Transfer must specify different source and destination accounts
- Exclude reason code must be from the approved structured list
- Cannot match to an already-reconciled transaction (hard control)
- Currency mismatch is a hard reject at engine level
- Sign mismatch is a hard reject at engine level
- Amount delta must not exceed configured materiality tolerance

# Error Handling

- Candidate scoring failures log the error and return empty candidates with an exception flag — never silent
- Commit failures roll back fully and display structured error detail in the feedback console
- Period lock conflicts display the locked period and suggest alternative resolution paths
- Normalisation failures fall back to raw description with a flag indicating normalisation was not applied
- Preload failures are retried silently; if retries fail, the next-line transition degrades gracefully to on-demand load

# Security / Permissions

- Minimum role: Bookkeeper (match, split, create, defer, mark transfer)
- Exclude action requires elevated permission (Reviewer or above)
- `SUSPECTED_FRAUD` exception creation and escalation requires Reviewer role
- Period adjustment and write-off require Accountant role or above
- All reconciliation decisions are audit-logged with actor identity
- Hard controls enforced at API layer — cannot be bypassed by direct API calls
- Audit records are append-only — no role has permission to delete or overwrite them
