## Source Priority Hierarchy

1. **ATO + SBR** -- implementation rules (tax rates, reporting structures, compliance requirements)
2. **AASB** -- financial reporting correctness (accounting standards, presentation requirements)
3. **Legislation.gov.au** -- legal authority (Acts, regulations)
4. **Scenario testing (internal)** -- executable validation layer

There is no single downloadable "rule set"; these sources must be combined.

# Implementation Boundaries

- All tax calculations must be traceable to an authoritative source listed above
- Tax rates and thresholds must be effective-date aware (no hardcoded current-year values)
- BAS label calculations must match the exact formulas defined by the ATO
- GST classification must align with the four ATO categories (Standard, GST-Free, Input-Taxed, Out-of-Scope)
- PAYG withholding must use the ATO coefficient method from Schedule 1

# Modules / Components to Prefer

- `api/pkg/rules/bundles/gst_v1.0.0.yaml` -- CEL formulas for GST calculations
- `api/pkg/rules/bundles/payroll_v1.0.0.yaml` -- CEL formulas for PAYG withholding
- `api/pkg/rules/bundles/accounting_v1.0.0.yaml` -- CEL formulas for BAS-related calculations
- `api/internal/tax/` -- Go domain package for tax code management and BAS preparation
- `shopspring/decimal` -- all monetary calculations (never float64)

# Modules / Components to Avoid

- Do not use `float64` for any tax calculation -- use `shopspring/decimal` exclusively
- Do not hardcode tax rates in Go code -- use CEL formulas with effective dates
- Do not implement BAS label calculations outside the rules engine

# Non-goals

- Tax planning or advisory features
- International tax treaty calculations
- FBT, CGT, WET, or LCT calculations
- Direct ATO electronic lodgement (file generation only)
- Tax agent workflow management

# Testing Expectations

- Every BAS label formula must have a unit test with known inputs and expected outputs
- GST calculations must be tested with half-up rounding edge cases
- PAYG withholding must be tested against ATO published examples from Schedule 1
- ABN checksum validation must be tested with known valid and invalid ABNs
- Effective date boundaries must be tested (e.g. SG rate change from 11% to 11.5%)

# Provenance Expectations

- Every tax calculation rule must cite its ATO source (article ID + section ID)
- BAS label calculations must reference ATO BAS instructions
- PAYG coefficients must reference ATO Schedule 1 for the relevant financial year
- GST classification rules must reference the relevant section of the GST Act
- All authority references must be validated at build time via CI check

# Prompting Notes

- When implementing GST calculations, always use the CEL formula approach with effective dates, not hardcoded Go arithmetic
- When implementing BAS labels, follow the exact calculation chain: G1 -> G5 -> G6 -> G8 -> G9 -> 1A (sales side); G10 -> G12 -> G16 -> G17 -> G19 -> G20 -> 1B (purchases side)
- When implementing PAYG, the coefficient method is: `withholding = (a * weekly_earnings) - b` where a and b come from the tax bracket table
- Always consider the no-TFN case (47% flat rate) as a separate path
- Quarter due dates are fixed except Q4 which has an extended date for tax agents
