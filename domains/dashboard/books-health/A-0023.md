# Summary

Technical architecture for the Books Health dashboard — a full-page health check across seven financial dimensions with traffic-light indicators, expandable detail panels, and an integrated financial calendar timeline.

# Technical Context

The Books Health page aggregates data from across the accounting system into a single health assessment. Unlike the Financial Calendar (which shows events over time), Books Health answers a point-in-time question: "How clean are my books right now?"

Most data can be derived from existing tables with aggregate queries. No new tables are required — this is a read-only reporting view.

### 1.2 Status Computation Rules

Each dimension computes its traffic-light status server-side:

**Bank Reconciliation:**
```
days_since = today - max(cr_report.end_date WHERE approved = true) per account
status = CASE
  WHEN all accounts days_since <= 7 THEN 'green'
  WHEN any account days_since BETWEEN 8 AND 14 THEN 'amber'
  ELSE 'red'
END
```

**Receivables:**
```sql
-- Aged buckets from ar table
SELECT
  count(*) FILTER (WHERE duedate >= today) as current,
  count(*) FILTER (WHERE duedate BETWEEN today - 30 AND today - 1) as days_30,
  count(*) FILTER (WHERE duedate BETWEEN today - 60 AND today - 31) as days_60,
  count(*) FILTER (WHERE duedate < today - 60) as days_90
FROM ar
WHERE force_closed IS NOT TRUE AND invoice = true
```
Status: green if days_90 = 0 AND days_60 = 0; amber if days_60 > 0; red if days_90 > 0.

**Payables:**
```sql
-- Same pattern against ap table
SELECT count(*), sum(amount_bc)
FROM ap
WHERE duedate < today AND force_closed IS NOT TRUE AND invoice = true
```
Status: green if none overdue; amber if overdue < 7 days; red if overdue >= 14 days.

**Cash Position:**
```sql
-- Sum of bank account balances (from last reconciled cr_report)
-- Net cash flow = sum(receipts this month) - sum(payments this month)
```
Status: green if net_cash_flow > 0; amber if net_cash_flow between -5% and 0; red if strongly negative.

**Expense Documentation:**
```sql
-- Count transactions with/without attachments
-- For now, approximate: expenses without a memo/source reference = undocumented
SELECT count(*) as total,
  count(*) FILTER (WHERE source IS NOT NULL AND source != '') as documented
FROM acc_trans
WHERE chart_id IN (SELECT id FROM account WHERE category = 'E')
  AND transdate >= date_trunc('month', today)
```
Status: green >= 95%; amber 80-95%; red < 80%.

**GST / BAS Readiness:**
```sql
-- Count coded vs uncoded in current BAS quarter
SELECT count(*) as total,
  count(*) FILTER (WHERE chart_id IS NOT NULL) as coded
FROM acc_trans
WHERE transdate BETWEEN quarter_start AND quarter_end
```
Status: green if uncoded = 0 and prior quarter lodged; amber if uncoded < 5; red if uncoded >= 5 or BAS overdue.

**Period Close:**
Computed from presence of locked periods and completion of checklist items. Each checklist item maps to a specific query (bank reconciled = cr_report exists for month, journals posted = no draft transactions, etc.).

## 2. Frontend Architecture

### 2.1 Page Structure

```
BooksHealthPage
  ├── HealthPanelGrid (responsive CSS grid)
  │     ├── HealthPanel × 7 (collapsed card with traffic light)
  │     │     └── HealthPanelDetail (expandable detail section)
  │     └── Each panel links to its action page
  ├── Divider
  └── FinancialCalendarFull (full-width timeline, not popover)
```

### 2.2 Component Design

**HealthPanel** — reusable card component:
```typescript
interface HealthPanelProps {
  title: string
  icon: LucideIcon
  status: "green" | "amber" | "red"
  summary: string
  metric?: string
  link: string
  children?: ReactNode  // expanded detail content
}
```

Traffic-light colours:
- Green: `bg-green-50 border-green-200` with green dot
- Amber: `bg-amber-50 border-amber-200` with amber dot
- Red: `bg-red-50 border-red-200` with red pulsing dot

### 2.3 Data Fetching

```typescript
const { data } = useQuery({
  queryKey: ["books-health"],
  queryFn: () => api.get<BooksHealthResponse>("/books-health"),
  refetchInterval: 60_000,
  staleTime: 30_000,
})
```

### 2.4 Responsive Grid

- Desktop (lg+): 4 columns for first row (4 panels), 3 columns for second row (3 panels)
- Tablet (md): 2 columns
- Mobile: 1 column, panels stack vertically

### 2.5 Full Calendar Integration

The bottom section renders the same `FinancialCalendar` data but in a full-width timeline layout instead of the popover format. Reuses `useCalendarTimeline` hook and `TimelineNode` component. The view range switcher (7 Days / Month / Quarter) is available here too.

## 3. API Implementation

### 3.1 Domain Package

Create `internal/bookshealth/` with:
- `model.go` — response types
- `service.go` — aggregation queries across ar, ap, acc_trans, cr_report, account
- `handler.go` — single GET endpoint
- `routes.go` — route registration

The service reads from multiple domain tables (read-only, same pattern as the calendar service). Each health dimension is computed by a separate method for testability.

### 3.2 Query Performance

All queries use aggregate functions (COUNT, SUM, MAX) with date filters — no full table scans. Expected execution: < 100ms total for all seven dimensions on a typical SMB dataset (< 10,000 transactions).

Add indexes if needed:
- `acc_trans.transdate` — already indexed
- `ar.duedate` — consider adding if not present
- `ap.duedate` — consider adding if not present

## 5. Future Evolution

- **Sparkline trend charts**: small inline charts showing health trend over last 4 weeks
- **Automated recommendations**: "Chase INV-0042 — 67 days overdue, $2,400"
- **PDF health report**: export for accountant review
- **Configurable thresholds**: tenant can adjust green/amber/red boundaries
- **Multi-tenant accountant view**: aggregated health across all managed clients
- **Historical comparison**: "Your books are healthier than last month"

# Risks / Trade-offs

- **Multiple aggregate queries**: seven dimensions means several database queries. Mitigated by keeping them simple (COUNT/SUM with filters) and using the 60-second cache.
- **Expense documentation approximation**: without a proper receipt attachment system, we approximate using the `source` field on `acc_trans`. This is a placeholder until file attachment is implemented.
- **Cash position simplification**: the projected end-of-month is a naive calculation (current + expected AR - expected AP). Not a proper cash flow forecast — that's a future Reports feature.

# Out of Scope

- Customisable thresholds
- Trend sparklines
- PDF export
- Automated recommendations
- Multi-tenant health comparison
