# Summary

Technical architecture for the Financial Calendar feature — a per-tenant task and event timeline accessible from the application header. Covers data model, API design, event aggregation strategy, accountant-tenant collaboration model, and frontend component architecture.

# Technical Context

The financial calendar consolidates work status from multiple domain tables (bank reconciliation, payroll, AR, BAS) into a unified timeline, supplemented by manually-injected tasks from accountants. It must be lightweight (popover, not a full page), real-time (reflects current state), and non-intrusive (badge indicator only when action needed).

# Proposed Design

## 1. Data Model

### 1.1 Calendar Task Table (new)

Stores accountant-injected tasks and manually-created reminders.

```sql
CREATE TABLE calendar_task (
    id              BIGSERIAL PRIMARY KEY,
    title           TEXT NOT NULL,
    description     TEXT,
    due_date        DATE NOT NULL,
    priority        TEXT NOT NULL DEFAULT 'normal' CHECK (priority IN ('normal', 'high')),
    status          TEXT NOT NULL DEFAULT 'open' CHECK (status IN ('open', 'completed', 'cancelled')),
    link            TEXT,
    created_by      TEXT NOT NULL,
    created_by_role TEXT,
    assigned_to     TEXT,
    completed_at    TIMESTAMPTZ,
    completed_by    TEXT,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

Key design decisions:
- `created_by` / `completed_by` store email (not UUID) for display simplicity
- `created_by_role` captures whether creator is `owner`, `master_accountant`, etc.
- `link` is an optional relative URL (e.g. `/banking`, `/bas`) for navigation
- `assigned_to` is optional — unassigned tasks are visible to all tenant users

### 1.2 No Calendar Event Table

Automated events (bank reconciliation status, payroll due dates, AR reminders, BAS deadlines) are computed on-the-fly by querying existing domain tables. This avoids data duplication and ensures the timeline always reflects current state.

### 2.2 Task Management Endpoints

```
POST   /api/v1/calendar/tasks          — Create a task
GET    /api/v1/calendar/tasks           — List tasks (with ?status=open filter)

POST   /api/v1/calendar/tasks/:id/complete  — Mark task complete
DELETE /api/v1/calendar/tasks/:id       — Cancel/delete a task
```

## 3. Event Aggregation

The timeline service queries 11 domain sources, producing a unified sorted list.

### 3.1 Bank Reconciliation
Source: `cr_report` table. Overdue if latest approved report's `end_date` > 7 days ago. Shows variance from `their_total`. Link: `/banking`.

### 3.2 Payroll Pay Runs
Source: `pay_run` table. Next due computed from last posted `pay_date` + cycle interval (default 14 days fortnightly). Overdue if expected date passed. Link: `/pay-runs`.

### 3.3 Accounts Receivable (Invoices)
Source: `ar` table. Groups overdue and upcoming invoices by count and total amount. Link: `/invoices`.

### 3.4 Accounts Payable (Bills)
Source: `ap` table. Groups overdue and upcoming bills by count and total amount. Link: `/bills`.

### 3.5 BAS/GST Lodgement
Computed from AU quarterly schedule: Q1 (Jul-Sep) due 28 Oct, Q2 (Oct-Dec) due 28 Feb, Q3 (Jan-Mar) due 28 Apr, Q4 (Apr-Jun) due 28 Jul. Monthly option available for businesses with >$20m turnover. Link: `/bas`.

### 3.6 BAS Payment
Monthly BAS payment due 21st of the month after reporting month. Separate from lodgement to help track cash outflow. Link: `/bas`.

### 3.7 Super Guarantee
Quarterly SG contributions due 28th of month after quarter end: Q1→28 Oct, Q2→28 Jan, Q3→28 Apr, Q4→28 Jul. Late payments incur SG charge. Link: `/super-rates`.

### 3.8 STP Finalisation
Annual deadline: 14 July for the preceding financial year. Shows within 30 days of deadline or when overdue. Link: `/pay-runs`.

### 3.9 FBT Return
Annual deadline: 21 May (or 25 June if lodged by tax agent). Shows within 30 days or when overdue.

### 3.10 Period Close
Computed: last day of current month. Shows within 7-day window.

### 3.11 Calendar Tasks
Source: `calendar_task` table. Includes manual entries by tenant users and accountant-injected tasks. All tasks with `due_date` in window or `status = 'open'` and overdue. Link: `calendar_task.link` field.

## 4. Frontend Architecture

### 4.1 Component Structure

```
AppHeader
  └── FinancialCalendar (popover)
        ├── CalendarHeader (summary badges)
        ├── OverdueSection (pinned, red)
        ├── DayGroup[] (Today, Tomorrow, ...)
        │     └── TimelineNode[] (card per item)
        └── CalendarFooter
```

### 4.2 Data Fetching

```typescript
const [viewRange, setViewRange] = useState<"7d" | "30d" | "quarter">("7d")
const days = { "7d": 7, "30d": 30, "quarter": 90 }[viewRange]

const { data } = useQuery({
  queryKey: ["calendar", "timeline", days],
  queryFn: () => api.get<TimelineResponse>(`/calendar/timeline?days=${days}`),
  refetchInterval: 60_000,
  staleTime: 30_000,
})
```

### 4.3 View Range Switcher

The calendar header includes a pill-button toggle for three view ranges:
- **7 Days** (default) — daily operational view
- **Month** — 30-day planning view
- **Quarter** — 90-day compliance/deadline view

The `?days=` parameter is passed to the API. Overdue items are always included regardless of range.

### 4.4 Header Badges

The calendar header shows three summary badges (in this order for positivity-first design):
1. **Green "N done"** — count of completed items (positive reinforcement)
2. **Red "N overdue"** — count of overdue items (only if > 0)
3. **Amber "N today"** — count of items due today (only if > 0)

The date display in the main app header shows:
- Red pulsing dot if any items are overdue
- Amber dot if items due today (but none overdue)
- No dot if all clear

### 4.5 Item Click Behaviour

- Items with a `link` navigate to the relevant page on click
- Done items are also clickable — navigate to view the completed state
- Chevron indicator on clickable items
- All automated event items include links to their relevant pages

## 5. Future Evolution

The timeline component is designed to evolve into:
- **Audit timeline**: journal entries, state transitions, reversal chains
- **Full calendar view**: month/week grid (separate page, not popover)
- **Cross-tenant accountant dashboard**: aggregated view across all managed tenants
- **Email/SMS notifications**: triggered by overdue items or new task assignments
- **Recurring task templates**: auto-create tasks on schedules

The `TimelineNode` component and `category` enum are extensible — adding new event types requires only a new aggregation query and a new category colour.

# Out of Scope

- Recurring task templates
- Email notifications for task assignments
- Cross-tenant accountant dashboard
- Calendar sync (Google, Outlook)
- Full-page calendar grid view

# Risks / Trade-offs

- **N+1 query risk**: timeline aggregation queries multiple tables. Mitigated by simple COUNT/SUM queries with date filters, not full row loads.
- **Stale data in popover**: 60-second refetch interval means up to 1 minute delay. Acceptable for a summary view; mutations (task complete) trigger immediate refetch.
- **Mock data phase**: initial deploy uses mock data to validate UX before API investment. Risk: user expectations set before backend is ready. Mitigated by labelling as prototype.
