# Purpose

Steering notes for AI-assisted implementation of the Financial Calendar feature. This document captures design decisions, UX intent, and implementation constraints from the original design conversation.

# Design Intent

The financial calendar was conceived as a **non-intrusive, always-accessible work planner** for bookkeepers and business owners. The core insight: bookkeepers need to answer "What have I done? What's coming up? What's overdue?" without checking multiple pages. The date in the header was already being displayed — making it clickable transforms dead UI into the most valuable element on screen.

## Key Design Decisions

### Popover, not page
The calendar is a popover triggered by clicking the date, not a separate page. This keeps the user in context — they can glance at their timeline without losing their current work. The popover should feel like a quick reference, not a destination.

### Done vs upcoming split
The timeline uses a vertical line with positioned dots and cards. Completed items get green checkmarks with completion metadata (who, when). Upcoming items get category-coloured dots. Overdue items are pinned to the top with red pulsing indicators. This done/upcoming pattern mirrors how accountants think about their work — "what did I finish, what's next?"

### Accountant-tenant collaboration
A key differentiator: accountants can inject tasks that appear in their client's timeline. This replaces phone calls and emails for simple coordination ("please upload the bank statement", "BAS draft ready for review"). Tasks are tagged with "Task" badges and show the creator. This makes the calendar a lightweight collaboration channel without requiring a full messaging system.

### Hyperlinks on items
Every timeline item should link to the relevant page where the user can take action. Done items are also clickable — users may want to verify or review completed work. The chevron icon indicates clickability.

### Non-intrusive by default
Users who don't want the calendar simply don't click the date. The only visual indicator is a small dot on the date (red for overdue, amber for due-today). No modals, no notifications, no forced interactions.

# Implementation Prompts

## Prompt 1: Timeline aggregation service

When implementing the timeline aggregation in `internal/calendar/service.go`:

- Query each data source independently (bank recon, payroll, AR, BAS, tasks)
- Each source returns `[]TimelineItem` 
- Merge all items, sort by date (overdue first, then chronological)
- Use simple queries — no JOINs across domains. Each query should be a single SELECT with date filter
- BAS deadlines are computed, not stored — use the tenant's fiscal year start month (from tenant config or default July for AU)
- Pay run next-due is computed from the last posted pay run's period_end + the modal pay cycle interval from employees
- Bank recon next-due: if no approved report in 7+ days, show as overdue with variance from `their_total` vs current account balance

## Prompt 2: Calendar task CRUD

When implementing task management:

- Tasks belong to the tenant (via the tenant DB connection, not a tenant_id column)
- `created_by` stores the user's email from JWT claims
- `created_by_role` stores the role from JWT claims (useful for displaying "From accountant: Name")
- Task completion is a state transition: `open` → `completed`, recording `completed_at` and `completed_by`
- Deletion is soft by design — set `status = 'cancelled'` rather than hard delete, for audit trail
- The `link` field is a relative URL (e.g. `/banking`, `/bas`) — the frontend prepends the base path

## Prompt 3: Frontend data fetching

When wiring the frontend to the real API:

- Use a single `useQuery` hook with `queryKey: ["calendar", "timeline"]`
- Set `refetchInterval: 60_000` (1 minute) for background refresh
- On task completion mutation, use `queryClient.invalidateQueries(["calendar"])` for immediate refresh
- The `useCalendarBadge` hook should derive counts from the same query data (not a separate request)
- Keep the popover component stateless — all data comes from the hook

## Prompt 4: UX polish

When implementing the timeline UI:

- The vertical timeline line should use Tailwind colours that match the category (blue for recon, purple for payroll, amber for AR, red for BAS)
- Today's section line should use `bg-primary-200` to visually anchor "now"
- Done items should have reduced opacity (`opacity-60`) and strikethrough title
- The popover should be `max-h-[34rem]` with `overflow-y-auto` — scrollable but not overwhelming
- On mobile (hidden header date), consider adding a calendar icon button in the visible header actions

## Prompt 5: Future timeline evolution

The timeline component is designed to evolve. Keep these extension points:

- The `category` field is an extensible string enum — future categories include `audit`, `journal`, `adjustment`, `reversal`
- The `TimelineNode` component accepts any `LucideIcon` — new categories just need a new icon mapping
- The timeline can be extracted from the popover into a full-page view by lifting the data fetching to route level
- Audit/journal events will add `before`/`after` state to timeline items for side-by-side comparison
- Cross-tenant accountant dashboard will aggregate timelines from multiple tenant DBs

# Anti-Patterns to Avoid

- **Don't cache event aggregation**: timeline data should always reflect current state. No materialised views or background jobs for this feature.
- **Don't send emails on task creation**: collaboration is in-app only for now. Email notifications are a future enhancement with its own spec.
- **Don't add a full calendar grid**: the popover is a 7-day rolling timeline, not a month view. A full calendar page is a separate feature.
- **Don't hardcode BAS dates**: compute from fiscal year config. Different tenants may have different fiscal years.
- **Don't cross domain boundaries**: the calendar service queries other domain tables read-only. It never writes to cr_report, pay_run, or ar tables.
