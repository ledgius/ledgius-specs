# Summary

Defines how task completion is verified — distinguishing between tasks that can be auto-verified by the system (checking underlying data changed) and tasks that require manual completion with evidence. Uses the rules engine (A-0011) for completion policies and the action trail (from the feedback system) to capture what the user actually did during a task session.

# Business Context

A task management system that allows mindless ticking degrades into a checkbox exercise. For an accounting system where audit trail matters, completion must be meaningful:

- "Reconcile bank" should only show as done when reconciliation actually happened
- "Review BAS draft" should capture evidence of what was reviewed
- Accountant-injected tasks should show proof of what the client did

The system must balance rigour (verifiable completion) with usability (not requiring excessive documentation for routine tasks).

# Core Principle: Completion Must Be Provable

Every completed task carries evidence — either system-verified data changes or user-provided activity summaries. The audit trail can answer: "Was this task actually done, or just ticked off?"

# Acceptance Criteria

## AC-1: Two completion modes

### Auto-verified completion

For tasks tied to system activities, the system automatically detects when the underlying work is done:

| Task type | Verification rule | Evidence |
|-----------|------------------|----------|
| Bank reconciliation | New approved `cr_report` exists for the account since task created | Report ID, date, matched transaction count |
| Payroll | New `pay_run` with status `posted` since task date | Pay run ID, employee count, total amount |
| BAS lodgement | BAS report submitted for the relevant quarter | Report ID, quarter, submission date |
| Bill payment | Target bills marked paid or `force_closed` | Bill IDs, payment amounts |
| Invoice collection | Payment received on target invoices | Invoice IDs, payment amounts |
| Super guarantee | SG payment recorded for the quarter | Payment amount, quarter |

Auto-verified tasks:
- Show a "Verified" badge with green checkmark instead of a manual tick
- Display a machine-generated completion summary: "Bank reconciled through 12 Apr 2026. 47 transactions matched, $0.00 variance."
- Cannot be manually ticked — they complete when the system detects the work
- If partially done (e.g. 30 of 47 transactions matched), show progress

### Manual completion with evidence

For accountant-injected tasks and custom reminders:
- User clicks "Complete" → completion form appears
- Completion note is required (minimum 10 characters)
- Auto-generated checklist of relevant actions (see AC-3)
- Auto-attached activity summary (see AC-4)
- Optional file attachment (receipt, screenshot, document)
- Completed-by and timestamp recorded

## AC-2: Completion policies in rules engine

Completion verification rules are defined in the rules engine (CEL/OPA), not hardcoded:

- **CEL formulas** evaluate whether the underlying data meets the "done" threshold
- **OPA policies** define which tasks require which evidence types
- **GoRules decision tables** map task categories to their verification rules

This allows:
- Adding new auto-verification rules without code changes
- Tenant-level customisation of completion thresholds (future)
- Accountants defining stricter or looser evidence requirements per client

Example CEL rule for bank reconciliation:
```cel
cr_report.exists(r, r.approved && r.end_date >= task.created_at && r.chart_id == task.metadata.chart_id)
```

Example OPA policy for manual task completion:
```rego
allow_completion {
  input.completion_note != ""
  count(input.completion_note) >= 10
  input.activity_trail_count > 0
}
```

## AC-3: Completion checklist (auto-generated)

When a user completes a task, the system generates a checklist of actions relevant to that task type:

**Bank reconciliation completion checklist:**
- [ ] Bank transactions imported
- [ ] Transactions matched to ledger entries
- [ ] Reconciliation report approved
- [ ] Variance resolved (if applicable)

**BAS preparation completion checklist:**
- [ ] All transactions coded with GST treatment
- [ ] GST accounts reconciled
- [ ] BAS figures reviewed
- [ ] BAS report generated

**Payroll completion checklist:**
- [ ] Pay rates verified for current period
- [ ] Leave balances updated
- [ ] Pay run calculated and reviewed
- [ ] Pay run posted
- [ ] STP report lodged

The checklist items are derived from the completion policy rules — the rules engine determines which items are mandatory vs optional, and which can be auto-ticked based on system data.

User ticks items they've done. Auto-verified items are pre-ticked by the system. The completion form cannot be submitted until all mandatory items are ticked.

## AC-4: Activity trail capture during task session

When a user clicks a task link and navigates to the relevant page, the system captures their activity during that "task session":

- **Pages visited** (e.g. "Opened Bank Reconciliation → Reviewed 47 transactions → Approved report")
- **Key actions performed** (e.g. "Matched 42 transactions", "Created 3 manual matches", "Approved reconciliation")
- **Values entered** — summarised, not raw (e.g. "Entered 5 payment amounts totalling $12,400")
- **Buttons clicked** (e.g. "Clicked 'Approve', clicked 'Post'")

This reuses the action trail infrastructure from the feedback system (circular buffer of click/submit events) but scoped to the task session:
- Trail starts when the user clicks the task link
- Trail ends when the user returns to the timeline/calendar or explicitly completes the task
- Trail is attached to the completion record as evidence

The activity summary is displayed in the completion form so the user can review what was captured before submitting.

## AC-5: Completion record storage

Each completed task stores:

```
task_id
completed_by (email)
completed_at (timestamp)
completion_mode (auto_verified | manual)
completion_note (text, null for auto-verified)
checklist_state (JSONB — which items were ticked)
activity_trail (JSONB — captured actions during task session)
verification_evidence (JSONB — system-generated proof for auto-verified tasks)
```

This record is immutable once written — it forms part of the audit trail.

## AC-6: Viewing completion evidence

On the timeline and calendar, completed tasks show:
- Green "Verified" badge (auto-verified) or green "Completed" badge (manual)
- Click to expand and see: completion note, checklist state, activity trail summary, verification evidence
- Accountant can review client task completions to verify work quality

## AC-7: Partial completion and progress

Some tasks are large (e.g. "Reconcile 200 bank transactions"). The system should show progress:
- Auto-verified tasks show progress bar if partially done (e.g. "120 of 200 matched")
- Progress is updated on each timeline refresh
- Task is not completed until the verification rule passes fully

# Out of Scope

- AI-generated completion summaries
- Automated quality scoring of task completion
- Peer review of completed tasks
- SLA tracking (time to complete vs expected)
- Recurring task auto-creation after completion
