# Summary

Architecture for the Export Operations Console web page and its backing API. Shape: one React/Vue page at `/data-export` calling a small set of read-only JSON endpoints plus a single start-export endpoint. All data is already persisted by R-0057 (export_run / export_run_entity) and R-0040 (audit_log.metadata); this spec is primarily a **view** and a **start** — very little new backend persistence, heavy emphasis on correct query composition and role gating.

# Terms

Vocabulary for Export Operations Console, Execute panel, History panel, Run detail drawer, Run detail is defined canonically in **R-0060 §Terms**. Vocabulary for export_run, export_run_entity, channel, run_kind is defined canonically in **R-0057 §Terms**. Vocabulary for external_system, watermark, gap sweep is defined canonically in **R-0058 §Terms**. Vocabulary for audit_log, metadata is defined canonically in **R-0040 §Data / Entities / Fields**.

| **Term** | Definition. |
|---|---|
| **Run summary projection** | The composite DTO returned by the history-list endpoint — joins `export_run` with `external_system.display_name`, the `user` display name from the platform schema, and selected `audit_log.metadata` keys for entity counts and totals. Computed at query time; no materialised view. |
| **Run detail projection** | The composite DTO returned by the run-detail endpoint — `export_run` + full `export_run_entity` list + filtered `audit_log` entries (where `metadata->>'export_run_id' = :id`) + resolved user name. One DTO per drawer load. |

# Technical Context

Most of the Console's data already lives in tables created by prior specs:
- `export_run` and `export_run_entity` — R-0057 / A-0031 (ledgius-db PR #9, merged)
- `audit_log` with `metadata` JSONB — R-0040 / A-0032 (ledgius-db PR #8, merged)
- `external_system`, `external_system_connection`, `export_watermark` — A-0032 foundation

The Console is primarily composition over these existing sources plus cross-DB user lookups (platform schema). No new persistence is introduced by this spec.

Two architectural choices drive the shape:

1. **Keyset pagination over offset pagination** for the history endpoint — large tenants will accumulate thousands of runs over time, and offset pagination becomes expensive at scale.
2. **Separate list and detail projections** — the history panel doesn't need the full audit trail or per-entity rows (both potentially large). Lazy-load them only when the drawer opens.

## API endpoints

All behind `/api/v1/data-export/*` and `TenantAuth` middleware.

| Method | Path | Roles | Description |
|---|---|---|---|
| GET | `/data-export/targets` | Accountant, Business Owner | List available targets for this tenant with connection state per channel. Drives the target / channel selectors. |
| GET | `/data-export/scope-defaults` | Accountant, Business Owner | Given `system_code` + `entity_types[]`, returns pre-filled date-range start (from watermark + 1 day), max plausible end (today), and an indication of any backdated entries visible in gap-sweep. |
| POST | `/data-export/validate` | Accountant, Business Owner | Run pre-flight validation for a proposed scope. Returns validation report JSONB + an acknowledgement token (signed hash of the report). |
| POST | `/data-export/runs` | Accountant, Business Owner | Start an export. Body includes target, channel, scope, run_kind, and the validation ack token. Returns the new `export_run.id`. |
| GET | `/data-export/runs` | Accountant, Business Owner (own tenant); Platform Owner (any tenant, with filter) | History list. Supports keyset pagination via `before_started_at` + `limit` params. Filters: system, channel, run_kind, status, date_from/date_to, user_id. |
| GET | `/data-export/runs/:id` | Same as list, with row-level tenant check | Run detail projection. Audit-logs the view. |
| GET | `/data-export/runs/:id/bundle` | Same | CSV channel only. Streams the CSV bundle zip. Returns 410 Gone if bundle has been purged (post-retention). |

## Query composition — History list

```sql
SELECT
    r.id, r.tenant_id, r.external_system_code, s.display_name AS system_name,
    r.channel, r.run_kind, r.status, r.started_at, r.completed_at,
    r.scope,
    (SELECT metadata FROM audit_log
        WHERE action LIKE 'xero.export.run.%'
          AND metadata->>'export_run_id' = r.id::text
        ORDER BY created_at DESC LIMIT 1) AS summary_metadata,
    r.user_id
FROM export_run r
JOIN external_system s ON s.code = r.external_system_code
WHERE r.tenant_id = :tenant_id
  AND (:system IS NULL OR r.external_system_code = :system)
  AND (:channel IS NULL OR r.channel = :channel)
  AND (:status IS NULL OR r.status = :status)
  AND (:before_started_at IS NULL OR r.started_at < :before_started_at)
ORDER BY r.started_at DESC
LIMIT :limit;
```

User display names are resolved in a second batch query against the platform schema (similar pattern to existing `UserFromContext` + platform lookup). Avoids per-row lookups.

## Query composition — Run detail

Four queries, executed in parallel:
1. `SELECT * FROM export_run WHERE id = :id AND tenant_id = :tenant_id` (or drop tenant filter for Platform Owner)
2. `SELECT * FROM export_run_entity WHERE export_run_id = :id ORDER BY entity_type, source_id`
3. `SELECT * FROM audit_log WHERE metadata->>'export_run_id' = :id::text ORDER BY created_at ASC`
4. Resolved user display name via platform schema

Assembled into a `RunDetailProjection` DTO.

The JSONB containment query in #3 uses the **partial GIN index on `audit_log.metadata`** from V1.29 (added per R-0040 AUD-002c). Query plan verified via `EXPLAIN ANALYZE` should show an Index Scan on `idx_audit_log_metadata`.

## Starting an export

The handler:
1. Verifies the validation ack token matches the stored validation report
2. Delegates to the Xero export orchestrator (A-0031) — which creates the `export_run` row, stages the scope, and starts the pipeline asynchronously
3. Returns `{id, status}` immediately

For CSV channel, the orchestrator runs to completion synchronously (no long-lived DB tx; writes one tx per stage); for API channel, writing is per-entity streaming and the UI polls.

## Real-time progress

For in-flight runs the drawer polls `GET /data-export/runs/:id` at most every 5 seconds. The endpoint returns the same run detail projection — the drawer diffs and updates. Polling stops automatically on terminal status. No WebSocket in v1.

Rate limit: per (user, run_id) polling is capped at 1 request per 4 seconds to prevent client accidents.

### Progress payload shape (R-0060 FR-031 through FR-033)

The run detail projection returned during polling carries a `progress` field keyed by entity type:

```json
{
  "id": "...",
  "status": "writing",
  "progress": {
    "by_entity_type": [
      {"entity_type": "invoice", "processed": 42, "total": 120, "state": "in_progress"},
      {"entity_type": "bill",    "processed": 35, "total": 35,  "state": "completed"},
      {"entity_type": "credit_note", "processed": 0, "total": 8, "state": "pending"}
    ],
    "aggregate_percent": 59
  }
}
```

`by_entity_type` is the primary indicator; `aggregate_percent` is derivable and secondary. Totals come from the pre-flight scope; `processed` counts live successes/failures from `export_run_entity`. `state` is one of `pending`, `in_progress`, `completed` (all target entities have terminal status), derived from the counts.

## CSV bundle storage and retention (R-0060 FR-027 through FR-030)

Bundles are written to **tenant-scoped object storage** via the existing `pkg/storage` abstraction. Storage keys incorporate both tenant id and run id:

```
<bucket>/data-export/<tenant_id>/<export_run_id>.zip
```

Access is tenant-scoped by construction — bundles are never addressable across tenants. The bucket is configured with server-side encryption; no additional per-tenant keys.

Retention is 90 days from `export_run.completed_at`. A background reaper job (runs daily, single-leader via existing scheduler infrastructure) deletes expired bundles and stamps `export_run.bundle_purged_at = <deletion timestamp>` on the row. The `export_run` table gains two columns:

```sql
-- Future tenant migration (when Console ships — NOT part of this spec's PR scope)
ALTER TABLE export_run
    ADD COLUMN bundle_storage_key TEXT,           -- NULL for api channel; set for csv when write completes
    ADD COLUMN bundle_purged_at   TIMESTAMPTZ;    -- NULL while retained; set when reaper purges

COMMENT ON COLUMN export_run.bundle_storage_key IS
    'Object storage key for the generated CSV bundle. Populated on csv-channel write completion; NULL for api channel.';
COMMENT ON COLUMN export_run.bundle_purged_at IS
    'Timestamp the bundle was deleted by the 90-day retention reaper. NULL while the bundle is available. Once set, download endpoint returns 410 Gone.';
```

The download endpoint `GET /data-export/runs/:id/bundle`:
1. Looks up `export_run.bundle_purged_at`; if non-null returns **HTTP 410 Gone** with body `{"error":"bundle_purged","purged_at":"<ISO8601>","retention_days":90}`.
2. Otherwise streams the object from storage with `Content-Type: application/zip` and `Content-Disposition: attachment; filename="xero-export-<tenantId>-<runId>.zip"`.
3. Records a `data_export.bundle.downloaded` audit entry with the metadata key `bundle_storage_key`.

The run detail drawer reads both `bundle_storage_key` (presence = bundle exists) and `bundle_purged_at` (presence = bundle purged) to render lifecycle state per FR-029.

## Symmetry with future Import Operations Console (R-0060 FR-034, FR-035)

The architecture is shaped to be replicated for imports without duplication of patterns:

- **Data model**: the future `import_run` / `import_run_entity` tables will mirror `export_run` / `export_run_entity` shape (composite audit via `audit_log.metadata`, per-entity outcome rows, run lifecycle states).
- **Web app**: the `<Execute panel>`, `<History panel>`, `<RunDetailDrawer>`, filter controls, and pagination hook shall live in a shared component layer (`src/domains/data-ops/shared/`). The Export and Import pages import them.
- **API**: handler/service factoring uses a thin Direction-parameterised layer so adding the Import endpoints at `/data-import/*` is primarily config, not new handler code.

When the task-runsheet navigation overhaul lands (parked in project memory pending dedicated UX work), both Export and Import consoles shall be reachable from a parent "Data" surface; sidebar entries become temporary pending that overhaul.

## Role gating

All routes behind `TenantAuth` first. Secondary gating:

- Accountant + Business Owner: `RequireFeature("data_export")` + automatic `tenant_id` scope on every query
- Platform Owner: `RequirePlatformOwner` + explicit `tenant_id` filter in query params (if omitted, results span all tenants)
- Region Manager: `RequireRegionManager` + region-scoped tenant subquery (`tenant_id IN (SELECT id FROM tenant WHERE region_id = :region)`)

# Consequences / Trade-offs

**Cross-DB user lookup adds a request dependency.** Every history list response requires both tenant DB (for runs) and platform DB (for user names). Batched once per response; acceptable.

**Polling over WebSocket is simpler but less efficient.** v1 trades some server overhead for implementation simplicity. A long-lived tenant with many users watching in-flight exports could see noticeable poll traffic; if that becomes material, SSE is the minimal upgrade path.

**Keyset pagination means no "jump to page N".** Users navigate forward and backward one page at a time. For an exports history where most interest is in the recent past, this is fine. Offset pagination could be added later for a "deep archive" view if needed.

**No in-console edit of anything.** Runs are immutable records. Users who want to re-export retry by starting a new run, not by editing history. This simplifies the UX significantly.

**Console sidebar placement is temporary.** Acknowledged in the spec, captured in memory (see `project_nav_task_runsheet_vision.md` in project notes). Future nav overhaul will fold this into a task-runsheet home.

# Alternatives Considered

- **Separate "Start export" and "History" pages**: rejected — users alternate between these actions often enough that one combined page is smoother.
- **Materialised view for run summaries**: rejected — query cost is manageable at v1 scale. Revisit if the history-list latency degrades.
- **WebSocket-based progress from day one**: rejected — polling is simpler and sufficient for v1 volumes. WebSocket upgrade path is straightforward later.
- **Putting the Console inside the Xero-specific area**: rejected — the spec is explicitly cross-system, and a Xero-only home would force duplication when MYOB export lands.

# Migration / Rollout

1. Ship API endpoints with integration-level handler tests (role gating, query composition, audit emission) — no UI yet. Endpoint is exercisable via API test, behind a feature flag.
2. Ship minimal web app page consuming the endpoints — history panel first (read-only value), then execute panel.
3. Progressive enhancement: bundle download, payments run kind, cross-tenant Platform Owner view.

Rollback: disabling the feature flag hides the sidebar entry and returns 403 on the endpoints. Data is unaffected.

# Open Questions

- Should the history panel group by target system (separate sub-sections for Xero, MYOB, etc.) or remain a flat list? Proposed: flat list with system column + filter. Grouping risks making the page feel "noisier" as more systems land.
- "Initiated by" shows user display name — but what about system-initiated runs (future scheduled exports)? Proposed: show `Scheduled (system)` with a scheduler icon when `user_id` corresponds to the system user. v1 has no scheduled runs so this is a forward-compatibility note.
- For the Execute panel's scope defaults — what happens when the selected system has zero prior runs AND zero watermark? Proposed: defaults to financial-year start per R-0058 EXS-009 and surfaces an info banner "This is your first export to <system>. All eligible records since <date> will be included."
