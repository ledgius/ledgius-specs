# Summary

A single, first-class page in the Ledgius web app where a user can **execute** a new data export to any supported external accounting system (Xero, MYOB, future) and **review the history** of past runs across all systems and channels. One UI, every target. Each historical run is inspectable with full detail — who initiated it, when, target system, channel (CSV bundle or direct API), scope and date range exported, outcome, entity counts, per-entity success/failure, and full audit trail — so operators, accountants, and support engineers have a single pane of glass for everything Ledgius exports.

Temporary navigation placement: a new top-level left sidebar item `Data Export`. The sidebar home is acknowledged as temporary pending a broader web-app navigation overhaul (task-runsheet concept, parked for future session) — but first-class placement is the right v1 home.

# Terms

| **Term** | Definition. |
|---|---|
| **Export Operations Console** | The web page addressed by this spec — `/data-export` in the web app. Combines execute and history in one page. |
| **Execute panel** | The top section of the page where the user configures and launches a new export run — target system, channel, scope, date range, validation ack, start button. |
| **History panel** | The table section showing recent export runs for the tenant, with filtering, sorting, and pagination. |
| **Run detail drawer** | A side panel or nested page that opens when the user clicks a row in the history panel, showing the full detail of a single export run (outcome, per-entity rows, audit trail, actions). |
| **Run detail** | The complete record of a past run: metadata, scope, outcome summary, per-entity outcomes, audit trail. Assembled by joining `export_run`, `export_run_entity`, `audit_log` (filtered by metadata.export_run_id), and platform-schema user lookup. |

Vocabulary used here but defined elsewhere:
- `export_run`, `export_run_entity`, `export watermark`, `gap sweep`, `run kind`, `channel` — defined canonically in **R-0057 §Terms** and **R-0058 §Terms**.
- `external_system`, `external_system_connection` — defined canonically in **R-0058 §Terms**.
- `audit_log`, `metadata`, role model (Platform Owner / Accountant / Business Owner / Region Manager) — defined canonically in **R-0040** and **R-0052**.

# Problem / Business Context

The Xero export work (R-0057) introduces a meaningful new capability. Without a dedicated operator-facing surface, users have no way to:

- Start an export without the developer reaching into the API via curl or a backend admin tool
- Understand what's been exported historically — did my last Xero push succeed? Did it cover March? Who ran it?
- Diagnose a partial or failed run — which invoices didn't land? Why?
- Know when a re-export is safe — is there an in-flight run? What's the current watermark?

These questions already have answers in the data (`export_run`, `export_run_entity`, `audit_log.metadata`, `export_watermark`). The Console makes them first-class.

A second motivation: **the Console is cross-system by design**, so it proves the "one engine, N targets" principle that R-0058 committed to. Xero shows up today, MYOB appears when its export spec lands, and no UI work is required for that to happen — MYOB runs simply appear alongside Xero runs filtered by target system.

A third motivation: **audit and compliance posture**. Australian accounting regulations (Privacy Act 1988, ATO record-keeping) require a business to show who moved financial data where and when. A single auditable surface for exports is not a nice-to-have — it's table stakes for a production accounting system.

# Scope

## In scope for v1

- Single web page at `/data-export`, sidebar-linked, first-class
- Execute panel (target + channel + scope + validation + start)
- History panel (tabular list of runs, filter/sort/paginate)
- Run detail drawer (metadata, outcome, per-entity table, audit trail, actions limited to "download CSV bundle" for CSV runs)
- Real-time progress indication for in-flight runs via polling (no WebSocket / SSE in v1)
- Role-gated access: Accountant / Business Owner for own-tenant runs; Platform Owner for cross-tenant read-only view
- Links from per-entity rows to the corresponding ledger entity (invoice, bill, etc.) so "what didn't land?" is actionable
- Links from run detail into the relevant audit log entries

## Deferred to v2

- Re-export failed entities (orchestrator capability; UI placeholder button only in v1)
- WebSocket / SSE push-based progress
- Scheduled / recurring exports
- Export templates (save a scope + channel + target for reuse)
- Per-entity drill-down into the source record diff (what Xero received vs what Ledgius has now)

## Out of scope

- Import operations console (separate, analogous page — future spec, symmetric design)
- Cross-tenant run aggregation / analytics (data pipeline concern, not a UI surface)
- Editing a run's outcome (runs are immutable; the UI is read-only except for "start new" and "download")

# Actors / Users

- **Bookkeeper** — primary day-to-day user of the Console. Executes exports on the business's behalf, monitors progress, investigates partial runs, and downloads CSV bundles for submission to external systems. A Bookkeeper may be any of: a partner of the business, the business owner themselves performing bookkeeping work, an employee of the business, or an external bookkeeper contracted to do the bookkeeping for the company. Regardless of employment relationship, the functional access is the same.
- **Accountant** — professional (often external) user. Uses the Console primarily at period end and during compliance work (BAS preparation, year-end reviews) when moving data out to a firm's preferred target system. Same operational capabilities as Bookkeeper plus the compliance/advisory context that guides *when* to export.
- **Business Owner** — runs exports on demand; reviews recent export history for reassurance. Functional access identical to Bookkeeper when acting in that capacity.
- **Region Manager** — read-only access scoped to tenants within their region. Reviews export activity across multiple tenants for oversight; cannot initiate exports on a tenant's behalf from this page.
- **Platform Owner** — cross-tenant read-only access for support. Filters by tenant, target system, status. Cannot execute new runs on a tenant's behalf from this page (that's an explicit "support-initiated export" flow, not v1).
- **Support Engineer** — uses the run detail drawer as the primary diagnostic surface when a customer reports "my export didn't include X." Access via Platform Owner or Region Manager role; no dedicated support-only role in v1.

# Functional Requirements

## Execute panel

- **OPS-001**: The Execute panel shall list available target systems populated from the `external_system` registry (R-0058) filtered to those with `active = TRUE` and at least one `supports_export_*` capability flag set. The list is the single source of truth — no hardcoded system labels in the UI.
- **OPS-002**: Selecting a target system shall show available channels — `CSV bundle` always shown if `supports_export_csv = TRUE`; `Direct API push` shown if `supports_export_api = TRUE` AND an active `external_system_connection` exists for this tenant + system. If API is shown but no active connection, a "Connect" CTA shall appear in place of the channel radio option.
- **OPS-003**: The Scope picker shall allow multi-selecting entity types filtered to those declared for the selected (target, direction) mapping configs (e.g. Xero v1: accounts, contacts, invoices, bills, credit notes).
- **OPS-004**: The Date range picker shall pre-fill `from` using the current `export_watermark` for the selected (target, entity_type) + 1 day, per R-0058 EXS-008. When multiple entity types are selected, the pre-fill uses the *earliest* watermark across the chosen types (so no entity is silently skipped).
- **OPS-005**: When no prior run exists for any chosen entity, the Scope picker shall default to the tenant's current financial-year start per R-0058 EXS-009.
- **OPS-006**: The user may override date ranges. Override does NOT disable the gap sweep.
- **OPS-007**: Before "Start export" is enabled, the user must click "Run pre-flight check." Pre-flight produces a validation report (errors + warnings per R-0057 FR-003) rendered inline. Errors block; warnings require explicit acknowledgement via a checkbox.
- **OPS-008**: "Start export" triggers the run in the chosen channel. For CSV channel, the page navigates to the run detail drawer and offers the CSV bundle for download when writing completes. For API channel, the page shows live progress.
- **OPS-009**: Payments runs (R-0057 FR-013) are presented as a distinct `run_kind` option, available only for API channel, and scope is automatically narrowed to payments.

## History panel

- **OPS-010**: The History panel shall list export runs for the current tenant, ordered by `started_at` DESC.
- **OPS-011**: Columns: Started at, Target system, Channel, Run kind, Scope summary (entity types + date range from `audit_log.metadata`), Status badge, Entity counts (success / failed / skipped from metadata), Initiated by (user display name), Duration (completed_at − started_at), Actions column.
- **OPS-012**: Filters: Target system (multi-select from registry), Channel (csv / api), Run kind (documents / payments), Status (single or multi), Date range (started_at), Initiated by (user picker).
- **OPS-013**: Status badges: `completed` (green), `partial` (amber), `failed` (red), `paused` (grey with pause icon), in-flight states (`validating`, `awaiting_ack`, `mapping`, `writing`) in blue with animated indicator.
- **OPS-014**: Default page size 20 runs; user may load more. Server-side pagination (keyset on `started_at DESC`).
- **OPS-015**: Clicking a row opens the run detail drawer.
- **OPS-016**: For Platform Owner role, a Tenant column is visible and a Tenant filter is available. Accountant / Business Owner never see other tenants.

## Run detail drawer

- **OPS-017**: The drawer shall show: run UUID, started_at, completed_at (if terminal), duration, initiated by, target system, channel, run_kind, full scope (entity types + dates), status, validation report snapshot.
- **OPS-018**: An Outcome summary section shall show per-entity-type counts (pending, success, failed, skipped) and headline figures pulled from `audit_log.metadata` (totals by category, date range covered).
- **OPS-019**: A Per-entity table shall list every `export_run_entity` row for the run, with columns: entity_type, source_id (link to the entity in the ledger — invoice, bill, contact, etc.), external_id (if success), status, error (if failed), written_at.
- **OPS-020**: An Audit trail section shall show every `audit_log` entry where `metadata->>'export_run_id'` equals the current run id, in chronological order. Each entry renders action + timestamp + user + expandable metadata.
- **OPS-021**: Actions section: "Download CSV bundle" (CSV channel, terminal state, only while the bundle is retained), "Copy validation report" (copies the JSONB to clipboard), "Re-export failed entities" (disabled in v1, v2 placeholder).
- **OPS-022**: For in-flight runs, the drawer shall poll the backend at most every 5 seconds for status updates. Polling stops when a terminal status is reached.

## Role-based access

- **OPS-023**: Bookkeeper, Accountant, and Business Owner see runs and may initiate exports for their own tenant only. Access enforced by `RequireFeature("data_export")` middleware + tenant-scoped query. Functional access is identical across these three roles; role distinctions exist for audit, reporting, and UI context but not for permission gating on this feature.
- **OPS-024**: Platform Owner has read-only access across all tenants, enforced by `RequirePlatformOwner` middleware. Cannot initiate exports on a tenant's behalf via this page.
- **OPS-025**: Region Manager (R-0052) has read-only access scoped to their region. Shares the same page as Platform Owner but with the tenant filter pre-scoped.
- **OPS-026**: Every export run records the initiating user's role in `audit_log.metadata` (under `metadata.initiator_role`) so post-hoc analysis can distinguish, for example, Bookkeeper-initiated runs from Accountant-initiated runs without requiring a second lookup.

## CSV bundle storage and retention

- **OPS-027**: CSV bundles produced by the CSV channel shall be persisted to **tenant-scoped object storage** (not the tenant database). The storage path incorporates the tenant id and run id so bundles cannot be addressed across tenants.
- **OPS-028**: Bundles shall be retained for **90 days** from the run's `completed_at` timestamp. After the retention window, the bundle is purged from object storage and the `export_run` row is flagged as `bundle_purged = TRUE`.
- **OPS-029**: The run detail drawer shall surface bundle lifecycle explicitly:
  - Within retention: a "Download CSV bundle" link with the text suffix "(retained until YYYY-MM-DD)"
  - After purge: the link is replaced with disabled text "Bundle purged on YYYY-MM-DD (retention 90 days)"
- **OPS-030**: Attempting to download a purged bundle via `GET /data-export/runs/:id/bundle` returns HTTP 410 Gone with a structured error body naming the purge date.

## Progress indication for in-flight runs

- **OPS-031**: For in-flight runs, the run detail drawer shall display progress as **entity-type-level counts** — one line per scoped entity type showing `processed / total` (e.g. "Invoices: 42 / 120", "Bills: 18 / 35"). This is the primary progress view because tenants reason in entity counts.
- **OPS-032**: Aggregate percent complete shall NOT be a primary indicator. It may be computed and shown secondarily (e.g. as an overall progress bar), but the entity-type-level breakdown is always visible.
- **OPS-033**: Completed entity types shall show a checkmark and final counts; not-yet-started entity types shall show `0 / <total>` with the total derived from pre-flight scope.

## Relationship to future Import Operations Console

- **OPS-034**: The Export Operations Console shall be designed symmetrically with the future Import Operations Console. The two share:
  - Data model shape (a `<direction>_run` table with per-entity outcome, audit-log-backed summary)
  - UI shape (Execute panel + History panel + Run detail drawer)
  - Role gating patterns (Bookkeeper/Accountant/Business Owner own-tenant; Platform Owner + Region Manager cross-tenant)
  - Sidebar placement strategy (both are temporary sidebar items pending the task-runsheet navigation overhaul)
- **OPS-035**: When the Import Operations Console ships (future spec), both consoles shall be factored under a shared component layer in the web app so changes to the common patterns (filter controls, run detail drawer, pagination) apply to both simultaneously. Once the task-runsheet nav overhaul lands, both Export and Import consoles shall be reachable from a parent "Data" surface.

# Non-functional Requirements

- **Performance**: History panel initial load ≤ 500 ms for tenants with ≤ 10k runs. Run detail drawer load ≤ 300 ms for runs with ≤ 10k entities.
- **Observability**: Every Console API call logs tenant, user, role, target_system filter, page cursor. Panel-level metrics: runs-listed counts, export-started counts, run-detail-view counts.
- **Accessibility**: Keyboard navigable, screen-reader friendly tables and badges. Colours not the only carrier of status (text label accompanies badge colour).
- **Security**: Every API endpoint behind `TenantAuth` + role middleware. Platform Owner surface behind an additional role gate, not merely a UI hide.
- **Auditability of the console itself**: Opening the run detail drawer writes a read-event audit record (`action = 'data_export.run.viewed'`) so access to export detail is itself traceable for compliance.

# Acceptance Criteria

- [ ] A newly registered external system (e.g. MYOB once its export spec lands) appears in the target selector without any web-app or API code change.
- [ ] A Bookkeeper, Accountant, or Business Owner user can each initiate an export end-to-end — pick target, channel, scope, ack validation, start, observe progress, review outcome — without leaving the Console. All three roles reach identical functional outcomes.
- [ ] A run initiated by a Bookkeeper records `metadata.initiator_role = 'bookkeeper'` on every `xero.export.run.*` audit entry; the same holds for Accountant and Business Owner with their respective role codes.
- [ ] Running a second export to the same target pre-fills the date-range start at `last_exported_transaction_date + 1 day` for each selected entity.
- [ ] A partial run shows in the history with amber status badge; clicking into it reveals the specific entities that failed with their Xero error messages.
- [ ] A Platform Owner can filter history by tenant and read a run detail without being able to initiate a new run from the Console.
- [ ] The run detail drawer's audit trail section shows every metadata-correlated `audit_log` entry, including watermark advances (`export.watermark.advanced`) and per-entity writes (`xero.export.entity.*`).
- [ ] Attempting to view a run belonging to a different tenant as Bookkeeper / Accountant / Business Owner returns 403, not 404 — role violation is distinct from missing resource.
- [ ] Opening a run detail drawer emits a `data_export.run.viewed` audit entry.
- [ ] The History panel respects keyset pagination — no skipped or duplicated runs across page turns.
