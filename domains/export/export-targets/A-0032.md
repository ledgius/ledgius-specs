# Summary

Cross-cutting architecture for strongly-typed external-system identity, shared per-tenant connection envelope, per-(tenant, external_system, entity_type) export watermark, and gap-sweep orchestration. Consumed by A-0031 (Xero export) first; future MYOB and other target specs inherit the same shape without modification. Designed to be forward-compatible with an import-side retrofit.

# Terms

Vocabulary for external system, integration direction, external-system connection, export watermark, gap sweep, and successful external ID is defined canonically in **R-0058 §Terms**. Vocabulary for audit_log, metadata, tx-on-context is defined canonically in **R-0040 §Data / Entities / Fields** and **A-0010**.

| **Term** | Definition. |
|---|---|
| **Registry parity check** | Boot-time assertion that every `ExternalSystem` Go constant maps to an active registry row, and every active registry row maps to a Go constant. Implementation in the `externalsystem` package init path. |
| **Credential envelope** | The system-agnostic columns on `external_system_connection` (status, connected_at, last_refreshed_at, scopes). Contrast with the **credential payload**, which is system-specific. |
| **Terminal-state run** | An export run whose status has moved to `completed`, `partial`, or `failed` and will not advance further. Watermark updates fire only on `completed` or `partial` (with at least one successful entity). |

# Requirement Link

- R-0058 (External Systems Registry and Incremental Export)

# Technical Context

Ledgius's existing import pipeline (A-0017) represents `source_system` as free-form strings (`xero`, `myob`, `csv`). This works because there are few consumers and the values are set in one place, but it does not scale to the cross-cutting needs described in R-0058. At the same time, the import pipeline is stable and in production: changing its schema and code requires coordinated migration and parser updates. This architecture therefore introduces the registry cleanly without touching import code, and leaves a well-defined retrofit path for a follow-up spec.

The registry-plus-typed-constants pattern is consistent with how R-0052 identity roles are modelled (enumerable population codes with guards at code boundaries) and with how compliance decisions (R-0032) depend on typed jurisdiction codes.

# Proposed Design

## Package layout

```
ledgius-api/
  internal/externalsystem/
    model.go              # ExternalSystem typed string, typed constants
    registry.go           # DB-backed registry, parity check
    repository.go         # CRUD for registry + connection
    service.go
    handler.go
    routes.go
  internal/exporter/
    watermark/
      model.go            # ExportWatermark, update logic
      repository.go
      service.go          # Read-current, advance-with-audit
    gapsweep/
      sweeper.go          # Query "inside range with no successful external_id"
```

The `externalsystem` package is consumed by both `exporter/xero` (A-0031) and any future `exporter/myob` or `importer/*` packages that adopt the registry.

## Go typed constants

```go
package externalsystem

type Code string

const (
    CodeXero Code = "xero"
    CodeMYOB Code = "myob"
)

func (c Code) Validate() error {
    switch c {
    case CodeXero, CodeMYOB:
        return nil
    default:
        return fmt.Errorf("unknown external system code: %q", c)
    }
}

func All() []Code { return []Code{CodeXero, CodeMYOB} }
```

`Code` is a distinct type — not a plain `string`. Any API boundary that accepts an external system identifier accepts `externalsystem.Code`, not `string`. A request payload carrying a free-form string goes through `externalsystem.Parse(s string) (Code, error)` at the handler edge, failing loud on unknown values.

## Registry parity check at boot

`registry.Check(ctx, db)` runs in `cmd/server/main.go` alongside other startup invariants. It asserts:

1. Every `Code` returned by `All()` has exactly one `external_system` row with `active=true`.
2. Every `external_system` row with `active=true` has a matching `Code`.

Both failure modes abort boot with a structured error pointing at the offending code or row. Inactive registry rows are permitted to exist without a Go constant (useful for deprecated systems — history preserved, not startable).

## Schema

```sql
-- ledgius-db/migrations/tenant/V1.NN__external_systems.sql
CREATE TABLE external_system (
    code TEXT PRIMARY KEY,                      -- 'xero', 'myob', ...
    display_name TEXT NOT NULL,
    country TEXT NOT NULL,                      -- ISO 3166-1 alpha-2
    supports_import_csv BOOLEAN NOT NULL DEFAULT FALSE,
    supports_import_api BOOLEAN NOT NULL DEFAULT FALSE,
    supports_export_csv BOOLEAN NOT NULL DEFAULT FALSE,
    supports_export_api BOOLEAN NOT NULL DEFAULT FALSE,
    active BOOLEAN NOT NULL DEFAULT TRUE,
    introduced_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    CONSTRAINT external_system_code_format CHECK (code ~ '^[a-z][a-z0-9_]*$')
);
COMMENT ON TABLE external_system IS
    'Registry of peer accounting systems Ledgius can interoperate with (import or export). '
    'Codes are stable lowercase snake_case. Startup parity check enforces alignment with Go typed constants.';

-- Seed v1
INSERT INTO external_system (code, display_name, country,
    supports_import_csv, supports_import_api,
    supports_export_csv, supports_export_api)
VALUES
    ('xero', 'Xero', 'AU', TRUE, FALSE, TRUE, TRUE),
    ('myob', 'MYOB', 'AU', TRUE, FALSE, FALSE, FALSE)
ON CONFLICT (code) DO NOTHING;
-- Note: Xero import_csv=true matches today's importer reading MYOB/Xero CSVs.
-- Xero import_api=false because current pipeline is file-based.
-- MYOB export_* flags remain false until the MYOB export spec lands.

CREATE TABLE external_system_connection (
    id UUID PRIMARY KEY,
    tenant_id UUID NOT NULL,
    external_system_code TEXT NOT NULL REFERENCES external_system(code),
    status TEXT NOT NULL CHECK (status IN ('active', 'disconnected', 'expired')),
    credential_payload_enc BYTEA NOT NULL,      -- encrypted, system-specific shape
    credential_schema_version TEXT NOT NULL,    -- e.g. 'xero.oauth2.v1'
    scopes TEXT,
    connected_at TIMESTAMPTZ NOT NULL,
    last_refreshed_at TIMESTAMPTZ,
    disconnected_at TIMESTAMPTZ,
    UNIQUE (tenant_id, external_system_code, status) DEFERRABLE INITIALLY DEFERRED
);
-- DEFERRABLE lets a replace-credentials flow update the old row to 'disconnected'
-- and insert a new 'active' row in a single tx without constraint trip.
CREATE INDEX ON external_system_connection (tenant_id, external_system_code);

CREATE TABLE export_watermark (
    tenant_id UUID NOT NULL,
    external_system_code TEXT NOT NULL REFERENCES external_system(code),
    entity_type TEXT NOT NULL,                      -- 'invoice','bill','credit_note','account','contact'

    last_exported_source_id UUID NOT NULL,          -- position: Ledgius record id of the last successful export
    last_exported_transaction_date DATE NOT NULL,   -- redundancy: that record's own transaction date (UI preset + gap sweep bound + crosscheck)
    last_export_run_id UUID NOT NULL,               -- correlation to export_run and audit_log
    last_advanced_at TIMESTAMPTZ NOT NULL,          -- server time the watermark advanced (not the record's timestamp)

    PRIMARY KEY (tenant_id, external_system_code, entity_type)
);
COMMENT ON TABLE export_watermark IS
    'Cursor-plus-date watermark per (tenant, external system, entity type). '
    'last_exported_source_id is the primary cursor — unambiguous when multiple records share a date. '
    'last_exported_transaction_date is derived from that record and stored for UI narrative, gap-sweep bounding, and crosscheck redundancy. '
    'Updated atomically with run terminal-state audit entry.';
```

Credential payload shape is validated by the target-specific service (e.g. `exporter/xero/apiclient` knows how to unmarshal `xero.oauth2.v1`). Schema version in the payload lets the credential format evolve without table churn.

## Watermark update — atomic binding

Inside the terminal-state DB tx (same tx that emits `xero.export.run.completed`):

```go
func (s *Service) CompleteRun(ctx context.Context, run *ExportRun, summary RunSummary) error {
    return db.Tx(ctx, func(tx *gorm.DB) error {
        if err := s.runRepo.MarkTerminal(tx, run.ID, summary.Status); err != nil { return err }

        for _, et := range summary.EntityTypes {
            // Latest successful export_run_entity for this run + type, ordered deterministically.
            tip, err := s.runEntityRepo.LatestSuccessful(tx, run.ID, et) // ORDER BY transaction_date DESC, source_id DESC LIMIT 1
            if err != nil { return err }
            if tip == nil { continue } // nothing successfully exported for this entity type in this run

            before, err := s.watermarkRepo.Get(tx, run.TenantID, run.SystemCode, et)
            if err != nil { return err }
            if err := s.watermarkRepo.Advance(tx, Watermark{
                TenantID: run.TenantID, SystemCode: run.SystemCode, EntityType: et,
                LastExportedSourceID:         tip.SourceID,
                LastExportedTransactionDate: tip.TransactionDate,
                LastExportRunID:              run.ID,
            }); err != nil { return err }
            if err := s.auditRepo.Log(tx, audit.Entry{
                Action: "export.watermark.advanced",
                EntityType: "export_watermark",
                Metadata: map[string]any{
                    "schema_version":        "export_watermark.v1",
                    "external_system":       string(run.SystemCode),
                    "entity_type":           et,
                    "from_source_id":        before.LastExportedSourceID,         // nil on first advance
                    "from_transaction_date": before.LastExportedTransactionDate,  // nil on first advance
                    "to_source_id":          tip.SourceID,
                    "to_transaction_date":   tip.TransactionDate,
                    "export_run_id":         run.ID,
                },
            }); err != nil { return err }
        }

        return s.auditRepo.Log(tx, audit.Entry{
            Action: fmt.Sprintf("xero.export.run.%s", summary.Status),
            Metadata: summary.ToMetadata(),
        })
    })
}
```

Watermark advance and run-terminal audit are inseparable by construction.

## Gap sweep

Runs after validation and before write. For each entity type in scope:

```sql
-- Pseudocode — real implementation via GORM query builder per entity type
SELECT e.id, e.transaction_date, ...
FROM invoice e
LEFT JOIN export_run_entity ere ON ere.source_id = e.id
    AND ere.entity_type = 'invoice'
    AND ere.status = 'success'
    AND ere.export_run_id IN (
        SELECT id FROM export_run
        WHERE tenant_id = :tenant_id AND external_system_code = :system_code
    )
WHERE e.tenant_id = :tenant_id
  AND e.transaction_date <= :watermark.last_exported_transaction_date
  AND ere.id IS NULL
  AND e.status IN (/* exportable statuses, e.g. approved, paid */);
```

Any row returned is a backdated entry that has not yet landed in the target. The orchestrator folds these into the run's scope alongside the user-requested range. The pre-flight validation report surfaces them under a `backdated_entries` section with counts per entity type, so the user sees them before commit.

Performance: each query is bounded by `transaction_date <= watermark_date` and backed by the existing entity-table indexes on `(tenant_id, transaction_date)`. Gap sweep cost is proportional to the size of the already-exported range, not total history.

## API surface

```
GET  /external-systems                              # list (code, display_name, country, capabilities, active)
GET  /external-systems/:code/connection             # { status, connected_at, last_refreshed_at, scopes }
POST /external-systems/:code/connection             # initiate connect flow (system-specific)
DELETE /external-systems/:code/connection           # mark disconnected (watermark preserved)
GET  /external-systems/:code/export-watermark       # { entity_type → { last_transaction_date, last_export_run_id } }
```

All route params typed via `externalsystem.Parse` at the handler; unknown codes return 404 with a structured error.

## Forward compatibility with import retrofit

The registry, connection envelope, and watermark concept are shaped to accept an import-side consumer with minimal extra schema work:

- `external_system_connection` is already direction-agnostic — when the import pipeline adopts it, the same active Xero connection row authorises both import and export for that tenant.
- An `import_watermark` table, if added, would mirror `export_watermark` in shape (PK `(tenant_id, external_system_code, entity_type)`).
- Adopting the registry in the import pipeline is a data migration on `import_batch_migration.source_system` to a FK column, plus parser code changes to use typed `externalsystem.Code` — not an architectural change.

The follow-up spec (tracked as a future R-00xx) will define the exact retrofit scope.

# Consequences / Trade-offs

**Parity check adds a boot-time dependency on the DB.** Accepted — the API already depends on the DB at boot for other reasons (R-0052 role seeding, tenant config). Failing loud on mismatch beats failing mysteriously in a request.

**Encrypted credential blob versus normalised per-system credential tables.** Blob chosen for v1 — OAuth tokens and refresh tokens fit cleanly and the shape can evolve via `credential_schema_version`. If MYOB or future systems need rich relational credential metadata (e.g. multi-file connection manifests), a dedicated normalised table is introduced alongside the blob for that system specifically; envelope columns stay shared.

**Watermark per (tenant, system, entity_type)** rather than per-(tenant, system, entity_type, fiscal_year). Simpler; fiscal-year splitting can be layered later by user selection without table redesign.

**Watermark fires on `completed` and `partial`.** A partial run with one success still advances the watermark to that success's date. Alternative: advance only on `completed`. Rejected — would mean a partial run leaves the watermark static while the user has actually moved forward for some entities, causing confusion and duplicate re-queueing on the next run. Per-entity external ID tracking makes the advance safe.

**Gap sweep surfaces backdated entries but does not auto-enable them.** User must acknowledge the pre-flight report that lists them — matches R-0057's validation-ack requirement. Alternative: silent inclusion. Rejected — principle of no silent behaviour change.

**Disconnecting preserves watermark.** A reconnection resumes where the previous connection left off, which matches user expectation ("I just re-authorised, I don't need to re-export everything").

# Alternatives Considered

- **Free-form `external_system TEXT` with a CHECK constraint listing allowed values.** Rejected — still scatters the allowed set across migration files and Go code; parity drift possible.
- **PostgreSQL native ENUM type.** Rejected — enum evolution (adding values) is awkward in Postgres and doesn't compose well with the need for capability flags and display metadata.
- **Retrofit the import pipeline in this PR.** Rejected per R-0058 scope decision — data migration and parser changes deserve their own review boundary.
- **Single `integration_watermark` table with a `direction` column for both import and export.** Attractive for symmetry. Deferred: defining export-only now avoids touching R-0017 state. The follow-up import-retrofit spec will decide one-table vs two-tables based on query patterns that emerge.

# Migration / Rollout

1. Ship `external_system` + `external_system_connection` + `export_watermark` migrations (tenant schema). Seed `xero` and `myob` rows.
2. Add `externalsystem` package with typed constants and parity check; wire the check into `cmd/server/main.go` startup.
3. A-0031 (Xero export) consumes the registry — `export_run.external_system_code` FK, `xero_connection` DDL removed in favour of `external_system_connection` with `credential_schema_version='xero.oauth2.v1'`.
4. Gap-sweep behaviour activates in the first v1 Xero export release.
5. Follow-up spec for import-side retrofit — data migration on `import_batch_migration.source_system` + parser code adoption.

# Open Questions

- Do we want to persist the `external_system` row's `display_name` in UI lookups or always resolve from the registry at request time? Registry-at-request keeps display names hot-editable; cached display names reduce round-trips.
- Should the parity check also validate capability flags against what the Go integration packages actually implement? E.g. if `exporter/xero` registers an API writer, but the registry row has `supports_export_api=false`, abort boot. Stronger but more coupling.
- When the import retrofit lands, should the two directions share a single `integration_watermark` table or stay in sibling tables? Deferred to the retrofit spec.
