# Summary

Ledgius interoperates with external accounting systems in both directions: import (R-0017) and export (R-0057). Both directions must reference the same peer system via a single canonical identifier — never via free-form strings. This spec introduces a **strongly typed external-systems registry** with capability flags per direction, a uniform **per-tenant connection envelope** shared across import and export, and **incremental-export mechanics** (watermark + gap sweep) that ensure consecutive export runs cover non-overlapping ranges while still picking up backdated entries automatically.

Scope note: the registry is defined bi-directionally from day one because renaming or re-keying it later is expensive. However, **only the export-side consumers are wired up in this spec's delivery**. The existing import pipeline (R-0017) continues to use its current free-form `source_system` strings; aligning it against the registry is a deliberate follow-up (tracked as a future requirement).

# Terms

| **Term** | Definition. |
|---|---|
| **External system** | A peer accounting platform Ledgius can import from and/or export to — e.g. Xero, MYOB, QuickBooks. Each external system has a stable registry code (lowercase snake_case: `xero`, `myob`), a country, capability flags per direction, and an `active` flag. Strongly typed in code and persisted in a registry table. |
| **Integration direction** | One of `import` (data flows external system → Ledgius) or `export` (data flows Ledgius → external system). Capabilities are declared per direction. |
| **External-system connection** | A per-tenant, per-external-system credential binding used by either direction. Exactly one active connection per (tenant, external_system). Credential shape is system-specific but the envelope (status, connected_at, last_refreshed_at, scopes) is uniform. |
| **Export watermark** | A cursor-plus-date record per (tenant, external_system, entity_type) identifying the last successfully-exported record: its Ledgius source ID (position) plus its transaction date (redundancy and UI narrative). Updated atomically with the terminal-state run audit entry. An equivalent import watermark is reserved for a future retrofit and not populated in this delivery. |
| **Gap sweep** | An orchestrator pass at the start of every export run that includes any entities within the already-watermarked date range that lack a successful external ID for this system — ensuring backdated entries are not silently left behind. |
| **Successful external ID** | The target-assigned GUID recorded on `export_run_entity` when `status='success'` for a given (external_system, source_id, entity_type). Presence of a successful external ID — not the watermark — is the authoritative "has been exported" signal. |

Vocabulary for export run, channel, pre-flight validation report, CSV bundle, balances-only scope, and transactional history scope is defined canonically in **R-0057 §Terms**. Vocabulary for audit_log, metadata, action is defined canonically in **R-0040 §Data / Entities / Fields**.

# Problem / Business Context

Three problems, one shared design:

1. **Free-form strings for "Xero" drift silently.** Import code uses `source_system = "xero"`. Export code would independently pick the same or a subtly-different string. A typo anywhere produces silent divergence across mappers, validators, audit queries, and UI filters. The only safe fix is a single source of truth for system identity, enforced at the code boundary.

2. **Per-system capabilities scatter without a registry.** "Does Xero support CSV import? Does MYOB support API export?" — today these questions are answered by ad-hoc constants scattered through parser and (future) exporter packages. Adding QuickBooks means grepping the codebase for every decision point. A registry with capability flags per direction centralises this.

3. **Sequential exports overlap or leave gaps.** Without a watermark, users manually choose date ranges every run. Either they overlap (re-exporting duplicates — idempotency protects but wastes effort) or they leave gaps (entries missed entirely). Backdated entries — an invoice dated last month entered after last month's export — are the worst case: silently skipped until a user notices.

All three problems affect import and export equally. Solving them only for export would leave the import side to drift, and retrofitting later costs double. The registry, connection envelope, and watermark concept are therefore defined bi-directionally now.

# Scope

## In scope for this spec

- Strongly typed `ExternalSystem` registry: Go typed constants + DB-seeded lookup table, with startup-time parity check
- Bi-directional capability flags (`supports_import_csv`, `supports_import_api`, `supports_export_csv`, `supports_export_api`)
- Uniform `external_system_connection` envelope (shared between import and export; per-system credential detail is system-specific)
- Per-(tenant, external_system, entity_type) export watermark with atomic update at run completion
- Gap-sweep orchestrator behaviour for every export run
- UI affordances for next-range preset and connection status
- Audit entries for every watermark advance

## Out of scope for this spec

- Retrofitting the import pipeline to use the registry (deliberate follow-up — import code keeps its current free-form `source_system` strings until a subsequent spec adds FK + data migration)
- Two-way sync / reconciliation between Ledgius and external systems (one-way per-direction only)
- Bulk re-export or "force re-export" flag
- Multi-system fan-out (one run targets exactly one external system)
- Admin UI for registering new external systems at runtime (registry changes ship via migration + code deploy)

# Actors / Users

- **Business Owner / Accountant** — picks an external system from a UI list; sees "next range" preset correctly on subsequent runs
- **Developer adding a new external system** — adds a Go constant, migrates a registry row, implements system-specific mapper/validator/writer, and deploys. No orchestrator changes.
- **Support engineer** — queries `audit_log` filtered by `metadata->>'external_system' = 'xero'` for a complete tenant history
- **Platform Owner** — approves new external systems via spec + PR flow; registry seed migrations are reviewed like any other schema change

# Functional Requirements

- **EXS-001**: The system shall maintain a registry of supported external systems. Each row: `code` (PK, lowercase snake_case), `display_name`, `country` (ISO 3166-1 alpha-2), `supports_import_csv`, `supports_import_api`, `supports_export_csv`, `supports_export_api`, `active`, `introduced_at`.
- **EXS-002**: The code space shall be constrained at the Go layer — typed constants (`ExternalSystemXero`, `ExternalSystemMYOB`, …) are the only accepted values at any API boundary. Startup-time parity check confirms every Go constant has a matching active registry row and every active registry row has a matching Go constant. Mismatch aborts boot with a structured error.
- **EXS-003**: Every export run shall record its target via a `external_system_code` FK to the registry. No free-form strings permitted in `export_run`, `audit_log.metadata`, or `external_system_connection`.
- **EXS-004**: The system shall maintain at most one active `external_system_connection` per (tenant, external_system). The envelope is direction-agnostic; the same connection row authorises both import and export for that external system. Disconnecting marks it inactive without deletion to preserve audit traceability.
- **EXS-005**: Credential payload shape is system-specific (OAuth2 tokens for Xero; file-based or API-key for MYOB; etc.) and stored encrypted in a per-system credential blob or dedicated normalised table as appropriate. The envelope columns (status, connected_at, last_refreshed_at, scopes) are uniform across systems.
- **EXS-006**: On successful terminal-state export run (`completed` or `partial` with at least one success), the system shall update `export_watermark` for each (tenant, external_system, entity_type) in scope to the position of the latest successfully-exported record, determined by ordering `export_run_entity` rows in the run by `(transaction_date DESC, source_id DESC)`. The watermark captures the record's Ledgius source ID and its transaction date together; the update rides in the same DB transaction as the terminal-state `audit_log` entry (atomic binding — see R-0057 FR-008).
- **EXS-007**: Before writing, every export run shall perform a gap sweep: query all entities in scope whose transaction date is ≤ the watermark's transaction date for (tenant, external_system, entity_type) AND whose source ID does not appear in any successful `export_run_entity` row for that external system. Swept entities are included alongside the user-requested range. Swept entities appear in the pre-flight validation report under an explicit "backdated entries" category so the user sees them before commit.
- **EXS-008**: The export initiation UI shall default the start date of the next run to the watermark's `last_exported_transaction_date + 1 day`. Users may override; overriding does not disable the gap sweep.
- **EXS-009**: When an external system has no prior successful export (no watermark row for the entity type), the UI shall default to the tenant's current financial-year start. Gap sweep has no effect in that case.
- **EXS-010**: Watermark and gap-sweep behaviour is channel-agnostic — a CSV run and an API run to the same external system advance the same watermark and share the same sweep query.
- **EXS-011**: The API permits listing external systems (`GET /external-systems`), reading connection status (`GET /external-systems/:code/connection`), and reading current export watermark (`GET /external-systems/:code/export-watermark`). All responses use registry codes only; no free-form identifiers in payloads.
- **EXS-012**: Every watermark advance emits an `export.watermark.advanced` audit entry with `metadata = {schema_version, external_system, entity_type, from_source_id, from_transaction_date, to_source_id, to_transaction_date, export_run_id}`. Missing entries indicate a bug — watermark changes without audit are forbidden by construction (single-tx invariant).

# Non-functional Requirements

- **Correctness:** atomic watermark + audit writes. No observable state where watermark has advanced but audit has not recorded it, or vice versa.
- **Observability:** registry parity check logs every registered system and capability at startup; watermark advances are structured-logged alongside the audit write.
- **Performance:** watermark lookups O(1) via primary key `(tenant_id, external_system_code, entity_type)`. Gap sweep: single indexed query per entity type, bounded by the watermarked date range.
- **Extensibility:** adding a new external system is a checklist — (1) add Go constant, (2) add registry seed migration, (3) implement system-specific integration packages, (4) parity check passes on boot. Zero changes to orchestrator, registry table, watermark table, or audit layer.
- **Forward compatibility with import retrofit:** the registry shape, connection envelope, and watermark table are built so the import side can adopt them with a data migration only (import code changes from reading free-form `source_system` strings to reading FK codes; no schema redesign).

# Acceptance Criteria

- [ ] Registering a new Go `ExternalSystem` constant without a matching active registry row aborts API startup with a clear error naming the missing code.
- [ ] Registering an active registry row without a matching Go constant aborts startup with a clear error.
- [ ] Two sequential exports to the same external system with no user-supplied date range cover non-overlapping date ranges end-to-end, with no missed entities.
- [ ] Entering an invoice dated inside an already-exported range and running the next export picks up that invoice via gap sweep without user intervention and surfaces it in the pre-flight report under "backdated entries".
- [ ] The UI preset for the next run to external system X reflects the correct next-start-date based on the last successful run for the relevant entity type.
- [ ] `audit_log` filtered by `metadata->>'external_system' = 'xero'` returns the complete Xero activity history for a tenant with no string variations.
- [ ] Disconnecting and reconnecting an external system preserves the pre-existing watermark.
- [ ] Capability flags can be toggled via migration: flipping `supports_export_api = false` on an existing system immediately disables the API channel in the UI without data loss.

# Open Questions

- Watermark granularity: (tenant, system, entity_type) is proposed. Should fiscal_year be a fourth dimension to handle year-end export conventions? Deferred pending UX input.
- Transaction-date vs created-at as the watermark axis: proposal is transaction-date plus gap sweep, because users think in transaction dates. Revisit if gap sweep cost proves material at scale.
- When the import retrofit happens (follow-up spec), do we introduce a unified "integration watermark" table (both directions in one table with a direction column) or keep `export_watermark` and add `import_watermark` as a sibling? The one-table approach is more symmetric; two tables keep each direction's queries shorter.
- Credential storage shape — per-system columns in `external_system_connection` vs credential_blob JSONB vs dedicated tables per system. A-0032 proposes the blob approach for v1 with migration room for normalised tables per system if their credentials grow complex.
