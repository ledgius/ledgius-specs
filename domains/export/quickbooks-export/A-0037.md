# Summary

Architecture for importing data from and exporting data to
QuickBooks Online (QBO). Follows the same patterns established for
Xero (A-0031) and the data import pipeline (A-0017): YAML mapping
configurations, staging pipeline, OAuth 2.0 token management, and
rate-limited API clients.

# Requirement Link

- R-0066 (QuickBooks Online Import)
- R-0067 (QuickBooks Online Export)

# Technical Context

QuickBooks Online is Intuit's cloud accounting platform. The
Australian edition supports GST, BAS lodgement, and STP Phase 2.
QBO's data exchange capabilities differ from Xero's:

| Capability | QBO | Xero | Implication |
|---|---|---|---|
| CSV export | Reports → Excel only (no direct CSV) | CSV templates per entity | QBO import must handle .xlsx |
| CSV import | Limited: COA, contacts, bank txns only | Comprehensive CSV import | QBO export has file-format gaps |
| API | REST + SQL-like queries, OAuth 2.0 | REST, OAuth 2.0 | Similar auth pattern, different query model |
| Rate limits | 500 req/min per Realm ID | 60 req/min | QBO is more generous |
| Token TTL | Access: 1hr, Refresh: 100 days | Access: 30min, Refresh: 60 days | QBO tokens last longer |
| Tax codes | Read-only via API | Read-only via API | Both require mapping, not creation |

# Proposed Design

## YAML Mapping Configurations

Following the mapping engine pattern (A-0017), QuickBooks-specific
YAML configs live under:

```
internal/importer/mappings/v1/quickbooks/
  accounts.yaml
  customers.yaml
  vendors.yaml
  invoices.yaml
  bills.yaml
  tax-codes.yaml

internal/exporter/quickbooks/mappings/v1/
  accounts.yaml
  customers.yaml
  vendors.yaml
  bank-transactions.yaml
```

Each config defines source fields → target fields with operator
chains (copy, lookup_table, default_if_null, date_format,
cel_expression) as established in A-0017.

## OAuth 2.0 Connection

Uses the same `external_connection` table and `oauth` package as
the Xero integration (A-0031):

```
external_connection
  id              SERIAL PRIMARY KEY
  tenant_id       — (from DB isolation, not a column)
  provider        TEXT NOT NULL    — 'quickbooks'
  provider_org_id TEXT             — QBO Realm ID
  org_name        TEXT             — QBO company name
  access_token    BYTEA            — AES-256-GCM encrypted
  refresh_token   BYTEA            — AES-256-GCM encrypted
  token_expires   TIMESTAMPTZ
  scopes          TEXT[]
  status          TEXT             — 'active', 'expired', 'revoked'
  connected_at    TIMESTAMPTZ
  connected_by    TEXT
```

### Token Lifecycle

```
User clicks "Connect QuickBooks"
  → Redirect to https://appcenter.intuit.com/connect/oauth2
  → User authorises
  → Callback with auth code
  → Exchange for access_token (1hr) + refresh_token (100 days)
  → Store encrypted in external_connection
  → Proactive refresh at 50 minutes
  → Refresh token rotation on each refresh
  → If refresh fails after 100 days → status = 'expired'
```

## API Client

```go
package quickbooks

type Client struct {
    httpClient  *http.Client
    baseURL     string          // https://quickbooks.api.intuit.com
    realmID     string
    tokenStore  *oauth.TokenStore
    rateLimiter *rate.Limiter   // 500/min
}

// Query executes a QBO SQL-like query with pagination.
func (c *Client) Query(ctx context.Context, entity string, where string) ([]json.RawMessage, error)

// Create posts a new entity to QBO.
func (c *Client) Create(ctx context.Context, entity string, body any) (*QBOResponse, error)

// Update puts an updated entity to QBO (requires SyncToken).
func (c *Client) Update(ctx context.Context, entity string, body any) (*QBOResponse, error)
```

### QBO API Specifics

- **Base URL**: `https://quickbooks.api.intuit.com/v3/company/{realmId}/`
- **Query syntax**: SQL-like — `SELECT * FROM Invoice WHERE TxnDate > '2026-01-01' STARTPOSITION 1 MAXRESULTS 1000`
- **Pagination**: `STARTPOSITION` (1-based) + `MAXRESULTS` (max 1000)
- **Rate limit**: 500 requests per minute per Realm ID
- **Throttle response**: HTTP 429 with `Retry-After` header
- **Content-Type**: `application/json`
- **Minor version**: `?minorversion=75` (latest as of 2026)

## Import Pipeline Integration

```
QBO File Upload / API Pull
    ↓
YAML Mapping Engine (quickbooks configs)
    ↓
Staging Tables (staging_account, staging_contact, staging_transaction)
    ↓
Validation (required fields, tax mapping, duplicates)
    ↓
Preview Grid (source vs mapped values)
    ↓
User Commit
    ↓
Ledgius Tables (account, entity, ar, ap, acc_trans)
```

File-based import handles .xlsx parsing via an Excel reader library
(e.g., `excelize`). The parsed rows feed into the same mapping
engine as CSV imports.

## Export Pipeline

```
User selects QuickBooks Online export
    ↓
Entity Selection + Date Range
    ↓
Pre-flight Validation (QBE-011)
    ↓
Validation Report + Acknowledgement
    ↓
[File channel]                    [API channel — v2]
CSV generation per entity         POST to QBO API per entity
    ↓                                 ↓
Zip bundle download               Per-entity result tracking
                                      ↓
                                  Export run summary
```

### CSV Generation

Each export entity has a writer that produces QBO-compatible CSV:

| Entity | Columns | Notes |
|---|---|---|
| Chart of Accounts | Account Name, Type, Detail Type, Account Number | Type and Detail Type must match QBO enumerations exactly |
| Customers | Name, Email, Phone, Mobile, Fax, Website, Street, City, State, Postcode, Country, Notes | Display Name must be unique |
| Vendors | Same as Customers | Display Name must be unique |
| Bank Transactions | Date, Description, Amount | 3-column format. Negative = withdrawal, positive = deposit. |

## API Channel Status

| Channel | Import (R-0066) | Export (R-0067) |
|---|---|---|
| **File (CSV/Excel)** | v1 — supported | v1 — supported (limited entities) |
| **API (OAuth 2.0)** | v2 — NOT YET SUPPORTED | v2 — NOT YET SUPPORTED |

The API channel requires:
1. Intuit Developer account with approved app
2. OAuth 2.0 app registration
3. Client ID + Secret configured as platform secrets
4. Production app review by Intuit (required for >25 connections)

These prerequisites are documented but not implemented in v1. The
connection card in the UI shows "Coming soon" for API features.

# Implementation Phases

## Phase 1: File-Based Import (R-0066 v1)
- YAML mapping configs for QBO entities
- Excel (.xlsx) file parsing
- Account type mapping (QBO → Ledgius)
- Tax code mapping (AU)
- Staging pipeline integration
- Import page UI with QuickBooks brand

## Phase 2: File-Based Export (R-0067 v1)
- CSV writers for COA, contacts, bank transactions
- QBO type/detail-type mapping (Ledgius → QBO)
- Pre-flight validation
- Zip bundle generation
- Export page UI with QuickBooks brand

## Phase 3: API Client Foundation (v2 prep)
- OAuth 2.0 connection flow
- Token management (encrypt, refresh, revoke)
- QBO API client with rate limiting
- Connection card UI

## Phase 4: API Import (R-0066 v2)
- Pull all entities via API queries
- Map to staging pipeline
- Pagination handling
- Error recovery

## Phase 5: API Export (R-0067 v2)
- Push entities via API POST
- Idempotency (track external QBO IDs)
- Per-entity error handling
- Export run tracking

# Risks / Trade-offs

- **Intuit Developer approval**: Production API access requires app
  review by Intuit, which can take weeks. v1 file-based approach
  avoids this dependency entirely.
- **QBO file import limitations**: Many entity types cannot be
  imported via file — the API is essential for a complete migration
  path. The file export clearly communicates these limitations.
- **Tax code read-only**: Cannot create tax codes in QBO via API.
  Must map to existing codes, which may not cover all Ledgius codes.
- **Detail Type enumeration**: QBO has ~100+ Detail Type values that
  must be matched exactly. A lookup table is required.
- **Display name uniqueness**: QBO enforces unique display names for
  customers and vendors. Duplicate detection is critical.
