# Summary

Ledgius must be able to export a tenant's accounting data into a
form QuickBooks Online (QBO) will accept, so that a business can
move (or replicate) its records into a QuickBooks Online
organisation. Two delivery channels are assessed: downloadable
**CSV/Excel files** matching QBO's import templates, and direct push
via the **QuickBooks Online Accounting API**. The API channel is
fully supported by Intuit and is the recommended path for
programmatic data transfer.

# Terms

| **Term** | Definition. |
|---|---|
| **QBO** | QuickBooks Online — Intuit's cloud-based accounting platform. |
| **Realm ID** | QuickBooks' unique company identifier, equivalent to a tenant ID. |
| **Detail Type** | QuickBooks' sub-classification within an account type. Required for chart of accounts import into QBO. |
| **Minor version** | QBO API versioning parameter (`minorversion=75`). Controls which fields and behaviours are available. |

Vocabulary used in this spec for export run, channel, and
pre-flight validation is defined canonically in **R-0057 §Terms**
(Xero Export). The same patterns apply here.

# Problem / Business Context

Customers leaving Ledgius for QuickBooks Online, or businesses that
need to replicate data into a QBO organisation (e.g., for an
accountant's practice management), currently have no supported
export path. Building a QuickBooks export complements the existing
Xero export (R-0057) and demonstrates that Ledgius is not a
data silo — customers can enter and exit freely.

QuickBooks Online has specific import constraints:
- **File import** is limited: QBO accepts CSV for bank transactions
  (3-column or 4-column format), chart of accounts (4 columns:
  Account Name, Type, Detail Type, Account Number), and customer/
  vendor lists. QBO does NOT support CSV import for invoices, bills,
  journal entries, or payments — these must go via the API.
- **API import** is comprehensive: the QBO Accounting API supports
  POST/PUT for all major entities (accounts, customers, vendors,
  invoices, bills, payments, journal entries, items). This is the
  primary programmatic integration path.

# Scope

## v1 — File-based export (CSV/Excel bundle)

Limited to entities QBO can import via file:

| Entity | Ledgius Source | QBO Import Format | Columns |
|---|---|---|---|
| Chart of Accounts | `account` | CSV | Account Name, Type, Detail Type, Account Number |
| Customers | `entity` + `entity_credit_account` (AR) | CSV | Name, Email, Phone, Billing Address, Notes |
| Suppliers/Vendors | `entity` + `entity_credit_account` (AP) | CSV | Name, Email, Phone, Billing Address, Notes |
| Bank Transactions | `bank_transaction` | CSV (3-col or 4-col) | Date, Description, Amount — OR — Date, Description, Credit, Debit |

### File Export Limitations

- **QBE-001**: The following entities CANNOT be imported into QBO via
  file and are therefore excluded from the CSV/Excel export:
  - Invoices (no CSV import in QBO — API only)
  - Bills (no CSV import in QBO — API only)
  - Journal entries (no CSV import in QBO — API only)
  - Payments (no CSV import in QBO — API only)
  - Credit notes (no CSV import in QBO — API only)
  - Inventory items (no CSV import in QBO — API only)

  The export UI shall clearly communicate this limitation:
  "Invoices, bills, journals, and payments require the API connection
  to transfer to QuickBooks. File export covers accounts, contacts,
  and bank transactions only."

## v2 — API-based export (OAuth 2.0)

Full entity export via the QBO Accounting API:

| Entity | Ledgius Source | QBO API Endpoint | Method |
|---|---|---|---|
| Chart of Accounts | `account` | `POST /v3/company/{realmId}/account` | Create |
| Customers | `entity` (AR) | `POST /v3/company/{realmId}/customer` | Create |
| Vendors | `entity` (AP) | `POST /v3/company/{realmId}/vendor` | Create |
| Invoices | `ar` + `invoice_line` | `POST /v3/company/{realmId}/invoice` | Create as Draft |
| Bills | `ap` + `invoice_line` | `POST /v3/company/{realmId}/bill` | Create |
| Payments (received) | `acc_trans` (receipts) | `POST /v3/company/{realmId}/payment` | Create |
| Payments (made) | `acc_trans` (payments) | `POST /v3/company/{realmId}/billpayment` | Create |
| Journal Entries | `transactions` + `acc_trans` | `POST /v3/company/{realmId}/journalentry` | Create |
| Credit Notes | `ar` (credit) | `POST /v3/company/{realmId}/creditmemo` | Create |
| Vendor Credits | `ap` (credit) | `POST /v3/company/{realmId}/vendorcredit` | Create |
| Items/Products | `parts` | `POST /v3/company/{realmId}/item` | Create |
| Tax Codes | `tax_code` | Read-only in QBO API | Cannot create — map to existing QBO tax codes |

### API Export Constraints

- **QBE-002**: QBO tax codes are **read-only** via the API. Ledgius
  cannot create new tax codes in QBO. The export must map Ledgius
  tax codes to existing QBO tax codes by name matching. Unmapped
  codes shall be flagged in the pre-flight validation report.
- **QBE-003**: QBO does not support opening balance import via API
  for individual accounts. Opening balances must be set via a journal
  entry posted to the accounts.
- **QBE-004**: Invoices created via API default to "Draft" status in
  QBO. The export shall create them as drafts unless the source
  invoice is marked as sent/approved in Ledgius.

## OAuth 2.0 Connection

- **QBE-005**: The QuickBooks connection shall use OAuth 2.0
  Authorization Code Flow, sharing the same infrastructure as the
  import connection (QBI-008 through QBI-010).
- **QBE-006**: A single OAuth connection serves both import and
  export. The connection is stored in `external_connection` with
  `provider = 'quickbooks'`.
- **QBE-007**: The connection card on the Export page shall display:
  - QuickBooks logo with brand colour (medium green, `#2CA01C`)
  - Connected organisation name and Realm ID
  - Connection status (connected / expired / not connected)
  - Last export timestamp

### API Export Not Currently Supported

- **QBE-008**: **API-based export is a v2 feature.** The initial
  release supports file-based export only. The OAuth connection
  infrastructure is shared with the import spec (R-0066) but the
  export push endpoints SHALL NOT be built in v1. The UI shall show
  "API export — coming soon" with the push button greyed out.

## Field Mapping

### Chart of Accounts (Ledgius → QBO)

- **QBE-009**: Ledgius account categories shall map to QBO types:

| Ledgius Category | QBO Account Type | QBO Detail Type (default) |
|---|---|---|
| A (Asset) — bank | Bank | Checking |
| A (Asset) — receivable | Accounts Receivable | AccountsReceivable |
| A (Asset) — other | Other Current Asset | Other Current Assets |
| A (Asset) — fixed | Fixed Asset | Furniture & Fixtures |
| L (Liability) — payable | Accounts Payable | AccountsPayable |
| L (Liability) — credit card | Credit Card | Credit Card |
| L (Liability) — current | Other Current Liability | Other Current Liabilities |
| L (Liability) — long-term | Long Term Liability | Notes Payable |
| Q (Equity) | Equity | Opening Balance Equity |
| I (Income) | Income | Sales of Product Income |
| E (Expense) — COGS | Cost of Goods Sold | Supplies & Materials - COS |
| E (Expense) — other | Expense | Other Miscellaneous Service Cost |

### Tax Code Mapping (Ledgius → QBO AU)

- **QBE-010**: Ledgius tax codes map to QBO Australian tax codes:

| Ledgius Tax Code | QBO Tax Code | Rate |
|---|---|---|
| GST | GST on Income / GST on Expenses | 10% |
| FRE | GST-Free | 0% |
| INP | Input Taxed | 0% |
| N-T | BAS Excluded | 0% |

## Pre-flight Validation

- **QBE-011**: Before any export, the system shall run validation:
  - All accounts have a valid QBO type mapping
  - All tax codes have a QBO equivalent
  - Contact names are not empty (QBO requires a display name)
  - Invoice/bill amounts are positive (QBO rejects negatives)
  - No duplicate display names for customers/vendors (QBO enforces
    uniqueness on display name)
- **QBE-012**: Validation issues shall be presented in a report with
  severity (error = blocks export, warning = proceeds with note).

## UI / UX

- **QBE-013**: The QuickBooks export option shall appear on the
  Export page with:
  - QuickBooks logo/icon
  - Brand colour: medium green (`#2CA01C`)
  - Label: "QuickBooks Online"
  - Subtitle: "Export accounts, contacts, and bank transactions.
    Invoices and bills require API connection (coming soon)."
- **QBE-014**: The export workflow:
  1. Select target system (QuickBooks Online)
  2. Select entities to export (checkboxes with entity counts)
  3. Select financial year / date range
  4. Run pre-flight validation
  5. Review validation report — acknowledge warnings
  6. Download CSV/Excel bundle (v1) or push via API (v2)
- **QBE-015**: The CSV bundle shall be a zip file containing one
  CSV per entity type, named clearly:
  - `quickbooks_chart_of_accounts.csv`
  - `quickbooks_customers.csv`
  - `quickbooks_vendors.csv`
  - `quickbooks_bank_transactions.csv`

# Out of Scope

- QuickBooks Desktop export (different product, IIF format)
- QuickBooks Self-Employed export
- Real-time sync / bi-directional integration
- Payroll data export (STP data is ATO-bound)
- Direct bank feed migration
- Creating tax codes in QBO (read-only via API)

# Actors / Users

- Bookkeeper — initiates export for client migration
- Accountant — reviews export for completeness
- Business Owner — triggers migration decision

# Preconditions

- Tenant has data in Ledgius to export
- For file export: user has access to QBO import tools
- For API export (v2): QuickBooks Online OAuth app registered,
  connection established

# Acceptance Criteria

## File Export
- [ ] QuickBooks Online appears as export target on Export page (QBE-013)
- [ ] CSV bundle downloads as zip with correct file naming (QBE-015)
- [ ] Chart of accounts CSV has correct 4-column format (QBE-009)
- [ ] Customer and vendor CSVs match QBO import format
- [ ] Bank transactions CSV in 3-col or 4-col format
- [ ] File limitation clearly communicated in UI (QBE-001)
- [ ] Pre-flight validation catches mapping issues (QBE-011)
- [ ] Validation report shown before download (QBE-012)

## API Export (v2 — not in initial release)
- [ ] OAuth 2.0 connection works (QBE-005)
- [ ] All v2 entities pushed via API (invoices, bills, journals, payments)
- [ ] Tax code mapping to existing QBO codes (QBE-010)
- [ ] Idempotent: re-export updates rather than duplicates
- [ ] API export card shows "coming soon" in v1 (QBE-008)
