# Summary

Architecture for a two-channel Xero export pipeline (CSV bundle + Xero API) that shares a single mapping + validation core. v1 ships balances-only coverage (COA, contacts, tax, invoices/bills/credit notes as drafts). v2 adds full transactional history (manual journals, inventory, bank transactions, payments) without requiring the v1 interfaces to change.

# Terms

| **Term** | Definition. |
|---|---|
| **Mapper** | The pure, I/O-free transformation layer (`exporter/xero/mapper/`) that converts Ledgius domain entities into Xero DTO shapes. Shared by both channels. |
| **Pre-flight validator** | The rule set in `exporter/xero/validator/` that streams source data and produces a `ValidationReport` before any write begins. Shared by both channels. |
| **Validation acknowledgement** | A signed hash (`ValidationAck`) of the `ValidationReport` produced at Validate stage. The orchestrator refuses to advance a run unless the supplied ack matches the stored report, preventing TOCTOU drift between validate and commit. |
| **CSV bundle** | Server-generated zip containing per-entity CSVs with Xero's exact header strings, a `manifest.json`, and a `README.md` prescribing import order in Xero. |
| **Resume cursor** | JSONB structure on `export_run.resume_cursor` recording the ordered write list position and per-entity pending set, used to resume `paused` or crashed runs without re-writing already-succeeded entities. |
| **Per-entity two-short-tx pattern** | The API-channel audit binding: Tx 1 records attempt, HTTP POST happens outside any DB tx, Tx 2 records outcome. Crash between Tx 1 and Tx 2 is recoverable via idempotency-key GET. |

Vocabulary for export run, channel, pre-flight validation report, Xero connection, external Xero ID, balances-only scope, and transactional history scope is defined canonically in **R-0057 §Terms**. Vocabulary for audit_log, action, metadata, entity_type, tx-on-context pattern is defined canonically in **R-0040 §Data / Entities / Fields** and **A-0010**.

# Technical Context

Xero's import surface is strict and well-documented. Both its self-service CSV importers and its REST API enforce the same underlying rules: exact-match enumerations (tax types, account codes, tracking, currency, branding), unique document numbers, control-account posting constraints (no journals direct to AR/AP system accounts), per-entity size limits, and draft-status landing for imported documents. The research digest at `myob_to_xero_import_reserch.md` captures these rules field-by-field and is the source of truth for field mapping tables in this spec.

The architecture therefore treats the two delivery channels as interchangeable skins over a single validation and mapping core — the rules don't change between CSV and API, only the output format. This keeps behaviour identical across channels and means v2 entity additions land in both channels simultaneously.

## Shared exporter interface

```go
type XeroExporter interface {
    Validate(ctx context.Context, scope ExportScope) (*ValidationReport, error)
    Export(ctx context.Context, scope ExportScope, ack ValidationAck) (*ExportRun, error)
}
```

Both `csvwriter.Exporter` and `apiclient.Exporter` implement this. The service layer picks the implementation based on the run's `Channel` field.

`ExportScope` describes what to export: entity set, optional date range, optional tenant filter. `ValidationAck` is a signed hash of the validation report — the handler refuses to start an export unless the user has acknowledged the exact report produced, preventing TOCTOU drift between validation and commit.

## Pipeline stages (applies to both channels)

```
Request -> Validate -> Report -> Ack -> Map -> Write (CSV bundle | API push) -> Record -> Complete
```

1. **Validate.** Stream Ledgius data through rule set, produce `ValidationReport` with severity (error / warning / info) per row. Errors block the run; warnings require user ack; info is log-only.
2. **Report.** Return report to user; persist to `export_run` row (status `validating` → `awaiting_ack`).
3. **Ack.** User reviews in UI, acknowledges warnings. Handler verifies ack hash matches the report just produced.
4. **Map.** Pull domain entities through `mapper/` package, produce Xero DTOs. Mapping is pure — no I/O, no side effects — so it's trivially testable and identical across channels.
5. **Write.**
   - CSV: `csvwriter/` builds per-entity CSVs into a zip, streams to response or object storage.
   - API: `apiclient/` POSTs each entity respecting rate limits, captures Xero-assigned GUIDs.
6. **Record.** Every entity write records outcome (success + external ID, or failure + Xero error payload) on a `export_run_entity` row. Idempotent — re-running consults existing external IDs and skips.
7. **Complete.** Run status moves to `completed`, `partial`, or `failed`. Warnings/errors surfaced in the export history UI.

## Regional defaults

`mapper/region.go` exposes `DefaultTaxType(region string) string`:

- `AU` → `BAS Excluded`
- anywhere else → `No GST`

Called by `mapper/account.go` when a source account has no tax code. Region resolves from tenant settings — never hardcoded at call sites.

## Xero field mappings (v1)

### Chart of Accounts

| Ledgius field           | Xero field     | Rule                                                                                                 |
|-------------------------|----------------|------------------------------------------------------------------------------------------------------|
| `account.code`          | `Code`         | ≤10 chars, unique, must not start with symbol (validator enforces)                                   |
| `account.name`          | `Name`         | ≤150 chars, unique; bank accounts further trimmed to fit Xero home UI                                |
| `account.type`          | `Type`         | Mapped via `mapper/account.go` typeMap (asset subtypes → ASSET etc.)                                 |
| `account.tax_code`      | `TaxType`      | Resolved via `mapper/taxrate.go`; absent → regional default                                          |
| `account.description`   | `Description`  | Passthrough, ≤4000 chars                                                                             |
| (computed opening bal)  | Conversion Bal | Optional export; written to separate `conversion-balances.csv` so Xero imports via its own template  |

### Contacts

| Ledgius field               | Xero field            | Rule                                                                       |
|-----------------------------|-----------------------|----------------------------------------------------------------------------|
| `contact.display_name`      | `Name`                | Unique within export; duplicate detection pre-flight                       |
| `contact.first_name`        | `FirstName`           | Passthrough                                                                |
| `contact.last_name`         | `LastName`            | Passthrough                                                                |
| `contact.email`             | `EmailAddress`        | Passthrough                                                                |
| `contact.is_customer`       | `IsCustomer`          | Boolean                                                                    |
| `contact.is_supplier`       | `IsSupplier`          | Boolean                                                                    |
| address fields              | `POAddress*` etc.     | Split: Xero uses distinct street / city / region / postcode / country      |
| `contact.abn`               | `TaxNumber`           | AU only                                                                    |

### Invoices (ACCREC) and Bills (ACCPAY)

| Ledgius field                   | Xero field          | Rule                                                                                              |
|---------------------------------|---------------------|---------------------------------------------------------------------------------------------------|
| `contact.display_name`          | `ContactName`       | Exact match required or Xero creates duplicate — validator checks existence at export time       |
| `document.number`               | `InvoiceNumber`     | Unique per tenant; duplicate = Xero rejects                                                       |
| `document.issue_date`           | `InvoiceDate`       | DD/MM/YYYY in CSV; ISO-8601 via API                                                               |
| `document.due_date`             | `DueDate`           | Same formatting rule                                                                              |
| `line.description`              | `Description`       | Required                                                                                         |
| `line.quantity`                 | `Quantity`          | Numeric                                                                                          |
| `line.unit_amount`              | `UnitAmount`        | Numeric                                                                                         |
| `line.account_code`             | `AccountCode`       | Must be a code also being exported or already present in Xero                                     |
| `line.tax_code`                 | `TaxType`           | Mapped via `mapper/taxrate.go`                                                                   |
| `line.tracking`                 | `TrackingName/Opt`  | v1: defer; v2 adds tracking                                                                       |
| (computed totals)               | (calculated by Xero)| Single tax treatment per run (inclusive OR exclusive — scope parameter, not per-doc)              |
| (status)                        | `Status`            | **Always `DRAFT` in v1** — keeps user in control of approval                                     |

### Credit notes

As per invoices/bills but:
- `Type` = `ACCRECCREDIT` or `ACCPAYCREDIT`
- `CreditNoteNumber` unique per tenant (separate namespace from invoices)
- Always `DRAFT`

### Tax rates

| Ledgius tax code | AU Xero tax type display name | Non-AU Xero tax type |
|------------------|-------------------------------|----------------------|
| GST 10%          | `GST on Income` / `GST on Expenses` | regional equivalent  |
| GST Free         | `GST Free Income` / `GST Free Expenses` | regional            |
| Input Taxed      | `Input Taxed`                 | regional             |
| (unmapped)       | `BAS Excluded`                | `No GST`             |

Full mapping lives in `mapper/taxrate.go` and is indexed by region code; the validator confirms every source tax code resolves before mapping runs.

## Pre-flight validator rules (v1)

Every rule produces `ValidationIssue{Severity, Entity, SourceID, Field, Message}`:

- **Error — COA:** code >10 chars, starts with symbol, non-unique, missing type
- **Error — Documents:** duplicate `InvoiceNumber` / `CreditNoteNumber` within scope, >1000 lines, missing ContactName
- **Error — Journals (v2):** debits ≠ credits, line-level account missing, posting to AR/AP system account
- **Error — Tax:** source tax code with no mapping for tenant region
- **Error — Currency:** multi-currency document where target Xero org lacks currency configured (API channel verifies via `GET /Currencies`)
- **Warning — Contacts:** fuzzy-duplicate names (Xero creates duplicates silently otherwise)
- **Warning — COA:** >500 rows (recommend splitting for CSV channel)
- **Warning — Documents:** >500 per file (CSV channel will chunk automatically)
- **Info:** counts summary per entity, default tax type applied count

## CSV bundle layout

```
xero-export-<tenantId>-<ISO8601>.zip
  manifest.json                    # { channel, scope, generated_at, entity_counts, validation_report_id }
  README.md                        # Xero import instructions, order of operations
  accounts.csv
  conversion-balances.csv          # optional, when opening balances requested
  tax-rates.csv
  contacts.csv
  invoices.csv                     # chunked as invoices-001.csv, invoices-002.csv above 500 rows
  bills.csv
  credit-notes.csv
```

README prescribes Xero import order (COA → Tax → Contacts → Invoices/Bills → Credit Notes) since dependencies between entities force this sequencing in Xero's UI.

## Xero API channel

- **Auth:** OAuth2 authorisation-code flow. `xero_connection` table in platform schema stores encrypted `access_token`, `refresh_token`, `expires_at`, `xero_tenant_id`, per Ledgius tenant.
- **Rate limiting:** `golang.org/x/time/rate` limiter at 60 req/min; global day counter in Redis capped at 5000. Hitting day limit moves run to `paused`, resumes next day.
- **Retry:** exponential backoff on 429 (`Retry-After` honoured) and 5xx (max 5 attempts). 4xx other than 429 are permanent — recorded on `export_run_entity` and the run continues.
- **Idempotency:** `Idempotency-Key` header set to `{export_run_id}:{entity_id}` on every POST. Safe to retry the run.
- **Ordering:** COA → TaxRates → Contacts → Invoices/Bills → CreditNotes. Dependencies enforced by the orchestrator, not by individual writers.
- **External IDs:** GUIDs returned by Xero persisted to `external_xero_id` + `xero_connection_id` on each Ledgius record. Re-export consults this and skips.

## Audit binding

Audit writes use the central `audit_log` table (R-0040) with the `metadata` JSONB column carrying operation-specific context (schema_version `"xero_export.v1"`, `export_run_id`, entity counts, date range, totals, external Xero IDs). `export_run` / `export_run_entity` hold *operational* state only (status, resume point, validation report, error payloads) — they are not the audit of record.

### CSV channel — atomic binding

The CSV bundle build and the run-level audit entry share a single DB transaction. Because the zip is generated server-side and streamed *after* the transaction commits, the audit state reflects reality whether the user ever downloads the file or not.

```
BEGIN
  UPDATE export_run SET status='completed', ...
  INSERT INTO audit_log (action='xero.export.run.completed',
                          metadata={schema_version:'xero_export.v1',
                                    channel:'csv', export_run_id, scope,
                                    entity_counts, totals_by_category,
                                    date_range, validation_report_id})
COMMIT
-- then stream the zip to the response
```

### API channel — per-entity two-short-transaction binding

Network calls to Xero are outside any DB transaction. The pattern is two short transactions per entity with the HTTP call between them:

```
per entity in ordered scope:
  Tx 1 (short):  audit_log(action='xero.export.entity.attempted',
                            metadata={export_run_id, entity_type, entity_id,
                                      idempotency_key})
                 export_run_entity.status = 'pending'
  (HTTP POST to Xero with Idempotency-Key — no DB tx held)
  Tx 2 (short):  on success: audit_log(action='xero.export.entity.succeeded',
                                         metadata={external_xero_id, ...})
                             export_run_entity.status = 'success', external_xero_id set
                 on failure: audit_log(action='xero.export.entity.failed',
                                         metadata={error_code, error_body, retry_count})
                             export_run_entity.status = 'failed', error set

after all entities:
  Tx (short):    audit_log(action='xero.export.run.completed' | 'partial' | 'failed',
                            metadata={entity_counts_by_outcome, totals_by_category,
                                      date_range, ...})
                 export_run.status moves to terminal state
```

Connections are never held across the HTTP call. A crash between Tx 1 and Tx 2 leaves an `attempted` audit entry with no matching outcome and a `pending` `export_run_entity` row — the resume path looks these up and consults Xero (via GET with the idempotency key's entity id) to determine whether the POST landed, then writes the delayed outcome audit entry.

Implementation uses the existing `db.ContextWithTx` pattern (A-0010) and the `audit.Repository.Log` method already exported from `internal/audit/repository.go`. No new transaction machinery.

## Persistence

Two new tenant tables, one platform table, plus a column addition to `audit_log` (see R-0040 AUD-002c):

```sql
-- ledgius-db/migrations/tenant/V1.NN__audit_log_metadata.sql
ALTER TABLE audit_log
    ADD COLUMN IF NOT EXISTS metadata JSONB;
COMMENT ON COLUMN audit_log.metadata IS
    'Operation-specific context (schema_version + free-form keys per action). '
    'Used for import/export run summaries, correlation ids, and other facts '
    'that do not belong in before_json/after_json entity snapshots.';

-- ledgius-db/migrations/tenant/V1.NN__xero_export.sql
CREATE TABLE export_run (
    id UUID PRIMARY KEY,
    tenant_id UUID NOT NULL,
    user_id UUID NOT NULL,
    channel TEXT NOT NULL CHECK (channel IN ('csv', 'api')),
    scope JSONB NOT NULL,
    status TEXT NOT NULL CHECK (status IN (
        'validating','awaiting_ack','mapping','writing',
        'completed','partial','failed','paused'
    )),
    started_at TIMESTAMPTZ NOT NULL,
    completed_at TIMESTAMPTZ,
    validation_report JSONB,
    resume_cursor JSONB,             -- where to pick up after paused/crashed
    CONSTRAINT export_run_tenant_fk FOREIGN KEY (tenant_id) REFERENCES tenant(id)
);
COMMENT ON TABLE export_run IS
    'Operational state for a Xero export run (status, resume point, validation). '
    'Audit of record is audit_log, not this table.';

CREATE TABLE export_run_entity (
    id UUID PRIMARY KEY,
    export_run_id UUID NOT NULL REFERENCES export_run(id) ON DELETE CASCADE,
    entity_type TEXT NOT NULL,      -- 'account','contact','invoice',...
    source_id UUID NOT NULL,        -- Ledgius entity ID
    external_xero_id TEXT,          -- populated on API channel success
    status TEXT NOT NULL CHECK (status IN ('pending','success','failed','skipped')),
    error TEXT,
    written_at TIMESTAMPTZ
);
CREATE INDEX ON export_run_entity (export_run_id, entity_type);
CREATE INDEX ON export_run_entity (source_id);
```

```sql
-- ledgius-db/migrations/platform/V1.NN__xero_connection.sql
CREATE TABLE xero_connection (
    id UUID PRIMARY KEY,
    tenant_id UUID NOT NULL UNIQUE,
    xero_tenant_id TEXT NOT NULL,           -- Xero's org GUID
    access_token_enc BYTEA NOT NULL,
    refresh_token_enc BYTEA NOT NULL,
    expires_at TIMESTAMPTZ NOT NULL,
    scopes TEXT NOT NULL,
    connected_at TIMESTAMPTZ NOT NULL,
    last_refreshed_at TIMESTAMPTZ
);
```

Tokens encrypted with the tenant-KMS key already used for bank feed credentials.

Note what is *not* in `export_run`: entity counts, totals by category, date range, error counts. These are operation-specific audit facts and live in `audit_log.metadata` on the `xero.export.run.*` entries — queryable centrally alongside every other tenant activity.

## Testing strategy

- **Unit:** mapper functions are pure → table-driven tests per entity type covering edge cases (unicode names, max-length strings, missing tax code, region fallback)
- **Validator:** table-driven — each rule has success + failure fixtures
- **CSV writer:** golden-file tests — produce bundle from fixture data, compare byte-for-byte to committed golden zip
- **API client:** `httptest` server returning Xero-shaped responses; covers happy path, 429 backoff, 4xx per-row failure, token refresh
- **Integration:** end-to-end test against Xero's [demo company sandbox](https://developer.xero.com) behind a build tag — not run in CI by default

## v2 preview — how the v1 architecture accommodates it

The v1 architecture is shaped so v2 entities slot in without changing the core:

- **Manual journals, inventory, bank transactions, payments** → add `mapper/journal.go` + `mapper/item.go` etc.; add corresponding `csvwriter/` + `apiclient/` writers. Orchestrator's ordered writer list grows; nothing else changes.
- **Debits-equal-credits rule** for journals → already defined as a validator category; activates when journal entities appear in scope.
- **Payments** → API-only in v2, so `csvwriter/` has no corresponding writer. Orchestrator's writer list is `map[entity]Writer` per channel, so missing-writer-for-channel is a first-class scope validation error.
- **Historical balances** → new entity type that writes to Xero's conversion balances endpoint; same mapping/validation shape.
- **Date-range scoping** → v1 `ExportScope` already carries optional `DateRange`, unused in v1; v2 transactional entities read it.

No v1 interface change is expected for v2.

# Consequences / Trade-offs

**Two channels doubles write-path code.** Mitigated by keeping mapping + validation shared — channels only differ in serialization.

**Pre-flight validation is expensive on large tenants.** Accepted: one expensive validation beats many broken exports. Streaming implementation keeps memory bounded; caching a validation report between identical scopes is a later optimisation.

**OAuth adds ops complexity.** Accepted: required for the API channel, re-usable for any future Xero-adjacent feature (e.g. pull-style verification).

**Draft-only landing in v1 means the user must still approve in Xero.** Matches Xero's own self-service behaviour and avoids writing unintended approved documents — a safer default even if slightly more manual.

**Idempotency via external_xero_id means re-exports are safe but deletes in Xero aren't detected.** If a user deletes an invoice in Xero and re-runs export, the invoice is NOT recreated (we skip because we remember the external ID). Acceptable for v1; v2 could add a "force re-export" flag that clears the external ID for a scope.

**API channel audit is per-entity, not run-atomic.** A mid-flight crash leaves some entities with `attempted` audit entries and no matching outcome; the resume path detects this via idempotency key GETs and emits the delayed outcome entry. This is strictly better than pretending atomicity exists across a DB + remote API boundary, and the audit log remains a complete record once resume completes.

# Alternatives Considered

- **CSV only (defer API):** rejected — the brief explicitly requires both, and API is the only channel for payments in v2.
- **API only (defer CSV):** rejected — blocks customers whose target Xero org isn't on the same account/subscription.
- **Third-party Go SDK (`xerogolang`):** considered. Rejected for v1 in favour of raw `net/http` because the SDK is thin and the REST surface is small; writing the client directly matches the existing `internal/bankfeed` style and gives tighter control over rate limiting and idempotency headers.
- **Write-ahead log for export state:** considered for crash recovery mid-run. Rejected: `export_run_entity` status column + idempotent re-run is simpler and achieves the same resumability.
- **Single-transaction export with full rollback on any failure:** rejected. Xero's API is per-request — there's no distributed transaction to wrap around it. Partial success with explicit reporting is more honest than pretending atomicity exists.

# Migration / Rollout

1. Ship `export_run` + `export_run_entity` migrations (tenant schema) and `xero_connection` (platform schema) first — cheap, unblocks backend work.
2. Land `mapper/` + `validator/` packages next; both are pure and testable in isolation.
3. CSV channel ships first (no OAuth dependency) — gated behind a feature flag per tenant.
4. API channel ships second — OAuth setup flow in the web app, then push functionality.
5. v2 entities rolled in one at a time, each behind its own sub-flag, so failures in (say) bank transactions don't regress the v1 surface.

# Open Questions

- Is there a case for a "dry-run API push" mode that validates against the connected Xero org (confirming account codes, tax rates, contact existence remotely) without writing? This catches issues the local validator can't see. Likely v2 candidate.
- Should the CSV bundle be uploadable back into a different Ledgius tenant for cloning purposes? (Out-of-scope here but could drive a shared serialization format.)
- How do we surface partial-success runs in the UI so users know which entities to follow up on? Needs UX input before v1 ships.
