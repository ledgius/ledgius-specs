# Summary

Ledgius must be able to export a tenant's accounting data into a form Xero will accept, so that a business can move (or replicate) its records into a Xero organisation. Two delivery channels are required: downloadable **CSV bundles** that match Xero's import templates, and direct push via the **Xero API** for tenants that have connected a Xero org via OAuth2. Both channels share identical field mapping and validation so the resulting Xero state is the same regardless of channel.

# Terms

| **Term** | Definition. |
|---|---|
| **Export run** | A single initiated attempt to move a defined scope of tenant data into Xero via one channel. Has operational state in `export_run` and audit history in `audit_log` (action prefix `xero.export.`). |
| **Channel** | The delivery mechanism for an export run. Exactly one of `csv` (zip bundle download) or `api` (direct OAuth2-authorised REST push to a connected Xero org). |
| **Pre-flight validation report** | The set of `ValidationIssue` records produced before any write, covering schema violations, duplicate keys, missing tax mappings, control-account conflicts, and size breaches. Acknowledgement is required before a run may advance beyond `awaiting_ack`. |
| **Xero connection** | A per-tenant OAuth2 link to a Xero organisation, persisted in the `xero_connection` platform table with encrypted tokens. Required for the `api` channel. |
| **External Xero ID** | The GUID Xero returns on a successful POST. Persisted against the Ledgius source record so re-exports are idempotent. |
| **Balances-only scope** | The v1 entity set: Chart of Accounts (with optional opening balances), tax rates, contacts, invoices, bills, and credit notes — matching Xero's own self-service MYOB-conversion coverage. |
| **Transactional history scope** | The v2 entity set added on top of balances-only: manual journals, inventory items, bank transactions, payments, and historical balances. |

Vocabulary used in this spec for audit and transaction management (audit_log, action, metadata, entity_type, tx-on-context) is defined canonically in **R-0040 §Data / Entities / Fields** and **A-0010**.

# Problem / Business Context

Customers migrating off Ledgius, replicating to a parent org's Xero file, or running a parallel Xero instance for reporting purposes currently have no supported path to move data out. This is a go/no-go criterion for larger customers: without a credible exit path, onboarding stalls. Building both channels matters because:

- **CSV bundle** works for every customer regardless of Xero auth state, gives the accountant full control to review/edit before import, and is the only channel usable when the target Xero org is on a different account or subscription.
- **Xero API** is faster, preserves line-level accuracy, avoids user-facing CSV size limits, and can push payments and other entities that have no documented CSV import.

Xero's own import surface is strict (template-driven schemas, exact-match enumerations, per-entity size ceilings, control-account posting rules, draft-only landing for documents). Any export that ignores those rules produces a partial or broken Xero state, which is a worse outcome than no export at all. The feature must therefore be conservative: validate aggressively up front, report every row that will fail before anything commits, and make "partial success" visible rather than silent.

# Scope

## v1 — Balances-only export

Mirrors Xero's own self-service "Convert to Xero from MYOB" coverage — the smallest, best-documented surface that still produces a usable Xero org.

- Chart of Accounts (with optional opening balances)
- Tax rates (Ledgius tax codes → Xero tax rate display names)
- Contacts (customers and suppliers)
- Invoices (ACCREC) — exported as Drafts
- Bills (ACCPAY) — exported as Drafts
- Credit notes — exported as Drafts
- Two delivery channels: CSV bundle download + direct Xero API push
- AU region default: accounts without a tax code default to `BAS Excluded`
- Pre-flight validator that surfaces every rejection cause before any bytes leave Ledgius
- Export run audit record: who, when, scope, channel, entity counts, warnings, external Xero IDs

## v2 — Full transactional history (follows v1 immediately)

- Manual journals
- Inventory items
- Bank transactions / statement lines
- Payments (API only — no documented CSV import)
- Historical balances / prior-year journals

## Out of scope (both versions)

- User/permission migration (Xero has no import surface for users; configured separately in target org)
- File attachments on invoices/bills (Xero's import templates don't accept attachments; a post-export upload flow is a separate feature)
- Recurring templates, scheduled payments, branding themes
- Two-way sync (this is one-directional export, not replication)
- Reversing a completed export from Xero (rollback lives in Xero's tooling, not Ledgius)

# Actors / Users

- **Business Owner** — initiates an export, chooses scope and channel, downloads CSVs or authorises Xero API push
- **Accountant / Bookkeeper** — reviews pre-flight report, resolves validation warnings, performs post-export reconciliation in Xero (Trial Balance parity)
- **Platform Owner** — monitors export run history for failure patterns, tenant support

# Functional Requirements

1. User can start an export run scoped to: all entities, a subset of entities, or a date range (v2).
2. User chooses delivery channel per run: **CSV bundle** or **Xero API** (API option disabled until a Xero org is connected to the tenant).
3. Every run produces a **pre-flight validation report** listing: schema violations, duplicate keys, missing tax mappings, control-account conflicts, size-limit breaches. User must acknowledge warnings before the run commits.
4. CSV bundle output is a single zip containing one CSV per entity type, with Xero's exact column headers, plus a `manifest.json` listing row counts and the validation report.
5. Xero API channel uses OAuth2 (authorisation code + refresh flow), stores tokens per-tenant, and respects Xero rate limits (60/min, 5000/day) with retry and backoff.
6. Each exported entity carries a Xero-assigned external ID stored against the Ledgius record (enables idempotent re-export and future v2 payment linking).
7. Documents (invoices, bills, credit notes) land in Xero as **Draft** in v1 — matches Xero's own self-service import behaviour and keeps the user in control of approval.
8. Export runs are fully audited via the central `audit_log` (R-0040): every run emits `xero.export.run.*` entries whose `metadata` JSONB carries the operational summary (tenant, user, channel, scope, entity counts, totals by category, date range, validation report id, external Xero ids). The `export_run` table holds only operational state (status, resume cursor, validation report body) — not the audit of record.
9. Re-running an export against the same Xero org is safe: entities already exported (by external ID) are skipped, not duplicated.
10. Failed API calls are recorded per row with the exact Xero error; the run completes with partial success rather than aborting, unless the failure is structural (auth expired, rate limit exhausted).
11. CSV bundles shall contain a plain-text `README.txt` (not markdown, not PDF) prescribing the Xero import order and any per-file notes. Plain text maximises portability and removes any dependency on markdown renderers or PDF viewers for the accountant receiving the bundle.
12. When a connecting user's Xero OAuth identity has access to more than one Xero organisation, the API-channel connect flow shall present a selector and require the user to pick exactly one organisation. The chosen `xero_tenant_id` is stored on the `external_system_connection` row. Subsequent exports on that connection target only the chosen org. Switching org requires disconnecting and reconnecting (preserves watermark — see R-0058 EXS-007).
13. v2 payments shall be pushed in a dedicated payments-only export run after the relevant invoices/bills already exist in Xero (either via a prior v1 document run or already present). Payments are never bundled into the same run as their parent documents. Rationale: cleaner run boundaries, simpler retry semantics, clearer audit narrative per run.

# Non-functional Requirements

- **Correctness over speed:** pre-flight validation runs before any byte is written or pushed. Zero silent rejections.
- **Observability:** structured logs at each stage (validation, mapping, write); metrics on run count, entity counts, error rate.
- **Security:** Xero OAuth tokens encrypted at rest; CSV bundles generated server-side and streamed, never cached unencrypted. Export run log records PII scope so deletion requests can be honoured.
- **Performance:** validator must handle 100k contacts / 500k transactions per tenant without loading everything into memory (stream from DB, chunk output).
- **Regional correctness:** tax mapping resolves via tenant region (AU default `BAS Excluded`, non-AU `No GST`) — never hardcode.

# Acceptance Criteria

- A new Xero organisation populated via a v1 export passes Xero's own Trial Balance check: debits equal credits, AR/AP control accounts reconcile to contact balances, no rows rejected silently.
- A tenant with a deliberately-seeded set of validation problems (duplicate invoice numbers, missing tax codes, COA codes >10 chars, >1000-line invoices) receives a pre-flight report naming every problem with the exact offending row and no data is written to Xero.
- CSV and API channels, run against the same tenant and scope, produce equivalent Xero state when both succeed (same entity counts, same balances).
- Re-running an export immediately after success produces zero new Xero records (idempotency).
- Export run history surfaces failures with enough context for a support engineer to resolve without reproducing the run.

# Dependencies

- OAuth2 client credentials from Xero Developer portal (tenant-level app registration)
- Xero API v2.0 (Accounting endpoints: Accounts, Contacts, Invoices, CreditNotes, TaxRates; v2 adds ManualJournals, Items, BankTransactions, Payments)
- Ledgius tax-code taxonomy (required for Xero tax-rate mapping — must be stable before v1 ships)
- `export_run` audit table (new, `ledgius-db/migrations/tenant/`)
- `xero_connection` table in platform schema for per-tenant OAuth tokens (v1 API channel only)
