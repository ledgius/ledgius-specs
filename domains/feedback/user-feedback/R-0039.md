# Summary

Collect and manage user feedback (bugs, suggestions, praise, questions) with contextual page information, action trails, browser info, and optional screenshots. Supports voting, commenting, status management, lightweight pulse signals (thumbs up/down), email notifications to admins, and a management dashboard with page-level heat maps.

# Problem / Business Context

During active development and beta usage, collecting structured user feedback with context (which page, what actions led to the issue, browser details) is essential for prioritising improvements. A built-in feedback system avoids the friction of external tools and captures rich contextual data automatically.

# Scope

- Feedback creation with contextual data (page route, feature area, action trail, browser info, screenshot)
- Feedback listing with optional page filter, sorted by vote count
- Feedback detail with comments
- Voting (one vote per user per feedback item)
- Commenting on feedback items
- Status management (open, acknowledged, in_progress, resolved, wont_fix) -- admin only
- Feedback deletion -- admin only
- Pulse signals (thumbs up/down per page, lightweight sentiment tracking)
- Email notifications to platform admins on new feedback (with screenshot attachment)
- Confirmation email to submitting user
- Management dashboard summary with page-level heat map

# Out of Scope

- Public feedback board (all feedback is internal)
- Feedback categories/tags beyond feedback_type
- Feedback assignment to specific team members
- Integration with external issue trackers (Jira, Linear)
- Feedback search/full-text search
- Feedback analytics beyond the summary dashboard

# Actors / Users

- Any authenticated user -- submits feedback, votes, comments, records pulse signals
- Platform admin -- manages feedback status, deletes items, views dashboard summary

# Preconditions

- User is authenticated
- Platform admin role is configured for admin-only endpoints

# Functional Requirements

- **FB-001**: System shall store feedback with: user_id, user_name, page_route, feature_area, feedback_type (bug/suggestion/praise/question), title, body, status (default open), vote_count (default 1), action_trail (JSONB), browser_info (JSONB), optional screenshot_data (base64), created_at, updated_at
- **FB-002**: System shall list feedback sorted by vote_count descending then created_at descending, with optional page_route filter
- **FB-003**: System shall return feedback detail with all comments loaded (ordered by created_at ascending)
- **FB-004**: System shall support one-vote-per-user-per-item voting, incrementing vote_count on the feedback record
- **FB-005**: System shall support adding comments to feedback items with user attribution
- **FB-006**: System shall support status transitions: open, acknowledged, in_progress, resolved, wont_fix -- admin only
- **FB-007**: System shall support permanent deletion of feedback items -- admin only
- **FB-008**: System shall support pulse signals (up/down) per page route with session_id tracking
- **FB-009**: On feedback creation, system shall send email notification to platform admins with feedback details and screenshot attachment (non-blocking)
- **FB-010**: On feedback creation, system shall send confirmation email to the submitting user (if authenticated)
- **FB-011**: System shall provide a management dashboard summary: total open, total resolved, by type, recent count (last 7 days), and page-level heat map (bugs, suggestions, praise, questions, open count, pulse up/down per page)
- **FB-012**: Admin endpoints shall be protected by RequirePlatformAdmin middleware

# Validation Rules

- Page route is required
- Feedback type must be one of: bug, suggestion, praise, question
- Title is required
- Comment body is required
- Pulse signal must be one of: up, down
- Status must be one of: open, acknowledged, in_progress, resolved, wont_fix

# Error Handling

- Already voted returns 400: "already voted"
- Invalid feedback ID returns 400
- Feedback not found returns 404
- Invalid feedback_type or signal returns validation error

# Security / Permissions

- User-facing endpoints: any authenticated user
- Admin endpoints (summary, status update, delete): RequirePlatformAdmin middleware
