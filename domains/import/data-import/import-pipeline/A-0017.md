# Summary

Architecture for a seven-stage, all-or-nothing data import pipeline that moves accounting data from external systems (Xero, MYOB, CSV) into Ledgius through staging tables with human checkpoints, auto-suggest mappings, balance verification, and transactional commit with rollback safety.

**2026-04-17 amendment:** the field-level translation from decoded source rows to Ledgius domain entities is now performed by the **mapping engine** (R-0059 / A-0033), the same engine that drives exports. Source-specific Go parsers remain responsible for decoding proprietary text formats (MYOB AO / CeeData / Xero CSV) into `map[string]any` rows; YAML configs under `internal/importer/mappings/v1/{source}/` then drive the domain-model mapping. This eliminates the earlier approach (A-0018) of spreading import field mapping across three rule engines (CEL / OPA / go-rules) with source-specific bundle layouts.

# Technical Context

Ledgius must support migration from Xero and MYOB, the two dominant Australian SMB accounting platforms, plus generic CSV. Each source system has different export formats, account type structures (MYOB uses 8 category types; Xero uses different account type names), and tax code conventions. The pipeline uses staging tables to decouple the import process from the live ledger, allowing validation and human review before any data touches production tables.

# Proposed Design

## Pipeline Stage Architecture

The pipeline progresses through seven discrete stages, each with its own status value stored on `import_batch_migration`:

```
Upload (uploaded) -> Analyse (analysed) -> Map Accounts (mapping) -> Map Contacts (contacts_mapped) -> Preview (previewing) -> Commit (committing -> committed) -> Verify (verified)
```

Each stage:
- Reads from staging tables and the live ledger
- Writes results back to staging tables and batch metadata
- Reports progress, warnings, and errors via `stage_progress` JSONB
- Requires explicit user approval at human checkpoints (stages 2, 3, 4, 5, 7)

## Staging Table Design

Three staging tables hold parsed data through the pipeline lifecycle:

**import_staging_account**: Holds source accounts with mapping state. Each row tracks `source_code`, `source_name`, `source_type`, and resolution via `mapped_to_id` (existing account) or null (create new). `mapping_status` tracks: pending, auto, manual, skip. `confidence` stores auto-match score (0.00-1.00).

**import_staging_contact**: Holds source customers/vendors with duplicate detection state. `duplicate_of` references a potential existing match. `mapping_status` tracks resolution.

**import_staging_transaction**: Holds source transactions with resolved mappings. `mapped_account_id` and `mapped_contact_id` are populated during the mapping stages by joining through the staging account/contact tables.

## Auto-Suggest Algorithm

Account mapping confidence scoring combines three signals:
- Code exact match: 0.50 weight
- Name similarity (fuzzy string match): 0.30 weight
- Same category (account type match): 0.20 weight

Contact duplicate detection uses:
- ABN exact match (highest confidence)
- Name similarity > 0.85 combined with same entity class

## Commit Transaction Design

Stage 6 executes a single database transaction with strict insert ordering to preserve referential integrity:

1. Chart of Accounts (+ tax codes)
2. Contacts (customers, vendors)
3. Open items (outstanding invoices/bills)
4. Transactions (invoices, bills, payments, journals)
5. Journal lines

Every inserted row is stamped with the `import_batch_id`. If any insert fails, the entire transaction rolls back and the batch status is set to `failed` with the error captured.

## Reversal Design

Committed batches can be reversed by:
1. Querying all records by `import_batch_id`
2. Creating reversal journal entries for all transactions
3. Soft-deleting imported contacts and accounts (mark inactive)
4. Recording the reversal as an audit event

# Data Flow

```
User uploads CSV/OFX
  -> Parser extracts rows into staging tables
  -> Analyser validates rows, detects patterns
  -> Account mapper joins staging accounts to Ledgius COA
  -> Contact mapper joins staging contacts to Ledgius entities
  -> Preview engine applies all mappings, computes totals
  -> Commit engine inserts in single transaction
  -> Verifier checks trial balance and control accounts
```

# Storage / Schema Changes

Three new staging tables and one batch tracking table:
- `import_batch_migration` -- batch metadata, status, counts, verification totals
- `import_staging_account` -- staged source accounts with mapping state
- `import_staging_contact` -- staged source contacts with duplicate detection
- `import_staging_transaction` -- staged source transactions with resolved mappings

All tables reference `import_batch_migration(id)` as foreign key. See R-0017 for full field specifications.

**2026-04-17 amendment — domain-model completeness migrations:**

Domain tables that imported records populate (`entity_credit_account` / `contact` / `location` / `ar` / `ap` / `invoice`) gain the fields R-0017 FR-024 / FR-025 / FR-026 require. Landing migrations:

- `V1.33__contact_email_address_phones.sql` — contact email + structured postal/shipping addresses (via the existing `location` table wired up properly) + phones (via the existing `contact` table wired up properly).
- `V1.34__invoice_line_account_tax_fks.sql` — `invoice.account_id` FK → `account(id)` and `invoice.tax_id` FK → `tax_code(id)`. Both NOT NULL on new rows; legacy rows backfilled (see migration for strategy) then made NOT NULL in a follow-up once data is clean.
- `V1.35__ar_ap_document_type.sql` — `document_type` column on `ar` and `ap` with CHECK constraint (`invoice`, `credit_note`, etc.); backfill from `is_return` (true → `credit_note`, false → `invoice`); `is_return` retained for backwards compatibility during transition.

# Background Processing

No background processing required. All stages are synchronous and user-driven. The commit stage may take several seconds for large batches but executes within a single HTTP request wrapped in a database transaction.

# Risks / Trade-offs

- **Large batch performance**: A single transaction wrapping thousands of inserts may lock tables and cause timeout for very large imports. Mitigation: set appropriate statement timeouts and consider batch-within-transaction chunking if needed.
- **Staging table cleanup**: Abandoned import batches leave orphaned staging rows. Mitigation: periodic cleanup job for batches older than 30 days in non-terminal states.
- **Source format drift**: Xero and MYOB may change their export formats. Mitigation: source-specific parsers are isolated and versioned under `api/pkg/rules/bundles/import/`.

# Alternatives Considered

1. **Direct API integration with Xero/MYOB**: Rejected because it requires OAuth token management, API rate limits, and ongoing maintenance as vendor APIs change. File-based import is simpler, more reliable, and user-controlled.
2. **Background job-based pipeline**: Rejected for v1 because the pipeline is inherently interactive (human checkpoints). Background processing adds complexity without benefit when the user is actively driving the process.
3. **Import without staging tables**: Rejected because it provides no opportunity for review, mapping, or rollback before data enters the live ledger.

# Test Strategy

- Unit tests for each source-specific parser (Xero, MYOB, generic CSV)
- Unit tests for auto-suggest confidence scoring
- Integration tests for each pipeline stage transition
- Integration test for commit + rollback on simulated failure
- Integration test for post-commit verification checks
- Parity tests verifying imported data matches source totals
