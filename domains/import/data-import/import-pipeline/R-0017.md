# Summary

A safe, auditable, multi-stage migration pipeline for importing accounting data from external systems (Xero, MYOB, CSV) into Ledgius. Designed so that a non-technical user can drive the process with full visibility, human checkpoints at every decision point, and zero risk of partial imports.

# Problem / Business Context

Migrating accounting data between systems is one of the highest-risk operations a business undertakes. Common failures include partial imports that corrupt the ledger, chart of accounts mismatches, lost transaction history, unmapped tax codes causing GST/BAS errors, and no post-migration verification. Ledgius must provide a controlled, all-or-nothing import pipeline that addresses every known migration pitfall from Xero, MYOB, and generic CSV sources.

# Scope

- Multi-stage import pipeline with seven discrete stages (Upload, Analyse, Map Accounts, Map Contacts, Preview, Commit, Verify)
- Human checkpoints at five of seven stages requiring explicit user approval before advancing
- Support for Xero, MYOB, and generic CSV source formats
- Account mapping with auto-suggest and tax code mapping
- Contact mapping with duplicate detection
- Transaction preview with balance verification
- Single-transaction commit with full rollback on failure
- Post-commit verification of trial balance, AR/AP control accounts, and bank balances
- Batch provenance stamping and reversal capability

# Out of Scope

- Real-time API integration with Xero or MYOB (file-based import only)
- Payroll data import beyond basic employee records (bank details, leave balances require manual entry)
- Inventory on-hand quantity import (flagged for manual post-import entry)
- Multi-currency exchange rate lookups (historical rates must be provided in source data)

# Actors / Users

- Business Owner -- initiates and approves the import process, reviews final verification
- Bookkeeper -- performs account and contact mapping, reviews transaction preview
- Accountant -- verifies post-import trial balance and tax code mappings

# Preconditions

- User is authenticated with at least Bookkeeper role
- Chart of Accounts exists in Ledgius (Australian COA or custom)
- Tax codes are configured (GST, FRE, INP, EXP, N-T, CAP)

# Functional Requirements

- **IMP-PIP-001**: System shall support a seven-stage import pipeline: Upload & Parse, Analyse & Validate, Map Accounts, Map Contacts, Preview Transactions, Commit, Verify
- **IMP-PIP-002**: Each stage shall have a visible status and the user shall not be able to skip stages
- **IMP-PIP-003**: Human checkpoints shall be required at stages 2 (Analyse), 3 (Map Accounts), 4 (Map Contacts), 5 (Preview), and 7 (Verify)
- **IMP-PIP-004**: Stage 1 (Upload) shall parse CSV/OFX files, validate format, required columns, and data types, and report row count, date range, and parse errors
- **IMP-PIP-005**: Stage 2 (Analyse) shall scan for duplicate accounts/contacts, identify unmappable fields, categorise accounts by type, and detect currency, date format, and tax code patterns
- **IMP-PIP-006**: Stage 2 shall flag inactive contacts, zero-balance accounts, stale transactions, and unreconciled bank accounts in source data
- **IMP-PIP-007**: Stage 3 (Map Accounts) shall present source chart of accounts alongside Ledgius chart with auto-suggested mappings by code similarity, name match, and category
- **IMP-PIP-008**: Stage 3 shall include a dedicated tax code mapping sub-step showing source tax codes with rates alongside Ledgius codes (GST, FRE, INP, EXP, N-T, CAP)
- **IMP-PIP-009**: Stage 3 shall allow user to map to existing account OR create new accounts inline
- **IMP-PIP-010**: Stage 4 (Map Contacts) shall present source customers/vendors with auto-detected duplicates (by name, ABN, email) and allow mapping or creation
- **IMP-PIP-011**: Stage 4 shall validate contact record completeness and flag missing ABNs, duplicate names, and formatting errors
- **IMP-PIP-012**: Stage 5 (Preview) shall apply all mappings to transactions, calculate totals by account and period, and show before/after comparison with source vs target totals
- **IMP-PIP-013**: Stage 5 shall verify that total debits equal total credits before allowing commit
- **IMP-PIP-014**: Stage 6 (Commit) shall execute all inserts in a single database transaction in strict order: accounts, contacts, open items, transactions, journal lines
- **IMP-PIP-015**: Stage 6 shall stamp every imported record with `import_batch_id` for traceability
- **IMP-PIP-016**: If any insert fails during commit, the entire batch shall roll back with status `failed` and error captured
- **IMP-PIP-017**: Stage 7 (Verify) shall check: trial balance still balances, AR/AP control accounts match imported receivable/payable totals, account balances match source data
- **IMP-PIP-018**: Stage 7 shall be mandatory -- user cannot mark import as complete until verification passes
- **IMP-PIP-019**: A committed import batch shall be reversible (soft delete with audit trail and reversal entries)
- **IMP-PIP-020**: System shall detect and warn on inventory items missing on-hand quantities
- **IMP-PIP-021**: System shall detect and warn on payroll data requiring manual post-import steps (bank details, leave balances, YTD figures, super fund details)
- **IMP-PIP-022**: System shall generate a post-import checklist for manual steps when payroll or inventory data is detected
- **IMP-PIP-023**: User shall be able to choose full transaction history vs opening balances only

# UX / UI Behaviour

- **IMP-PIP-024**: Import page shall display a horizontal stage indicator (wizard stepper) showing all seven stages with current progress
- **IMP-PIP-025**: Each stage shall show stage-specific content (file uploads, mapping tables, preview data), progress indicators, warnings/errors, and an "Approve & Continue" button
- **IMP-PIP-026**: "Approve & Continue" button shall be disabled until all stage requirements are met
- **IMP-PIP-027**: User shall be able to navigate back to previous stages to review or adjust
- **IMP-PIP-028**: Commit stage shall show a final confirmation dialog with total accounts, contacts, transactions, source vs target totals, and balance verification before executing
- **IMP-PIP-029**: Account mapping stage shall show confidence scores for auto-suggested mappings (code + name + category match)

# Validation Rules

- Source files must be valid CSV or OFX format with required columns present
- Account mappings must be complete (no unmapped accounts) before proceeding to contacts
- Contact mappings must be reviewed before proceeding to preview
- Total debits must equal total credits before commit is allowed
- Tax code mappings must be valid Ledgius tax codes

# Error Handling

- Parse errors at upload stage report row number, column, and expected format
- Mapping conflicts (duplicate target account) display clear resolution options
- Commit failure performs full rollback, sets status to `failed`, and captures full error detail
- Post-commit verification failures display specific mismatches with expected vs actual values

# Security / Permissions

- Minimum role: Bookkeeper (initiate and perform import)
- Import batch records include `imported_by` for audit attribution
- All imported records carry `import_batch_id` for provenance tracking
