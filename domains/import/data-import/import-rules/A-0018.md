# Summary

Architecture for declarative import rules using Ledgius's existing three-engine rules infrastructure (CEL, OPA Rego, go-rules). Migration data transformation, validation, and mapping rules are defined as versioned files per source system, editable without recompiling, and executed through the existing `pkg/rules/` orchestrator.

# Requirement Link

- R-0017 (Data Import Pipeline Requirements)
- A-0017 (Data Import Pipeline Architecture)

# Technical Context

The import pipeline (A-0017) requires data transformation (date parsing, amount normalisation), field mapping (account type translation, tax code mapping), validation (row-level completeness, duplicate detection), and policy enforcement (access control, import limits). Rather than embedding this logic in Go code, all rules are expressed declaratively using the three rule engines already in the Ledgius stack, coordinated through the existing `pkg/rules/` orchestrator.

# Proposed Design

## Engine Assignment by Concern

Each rule engine handles the concern it is best suited for:

### go-rules (JSON Decision Tables) -- Field Mapping & Type Translation

Handles lookup-table style mappings where input values map deterministically to output values:
- Account type translation (MYOB 8 category types -> Ledgius A/L/Q/I/E categories)
- Account type translation (Xero account types -> Ledgius categories)
- Tax code mapping (Xero tax types -> Ledgius GST/FRE/INP/EXP/N-T/CAP)
- Tax code mapping (MYOB tax codes -> Ledgius codes)
- Contact type mapping (source category -> Ledgius entity_class)
- Transaction type mapping (source type -> Ledgius transaction type)
- Currency code normalisation

Each mapping is a JSON decision table file under `api/pkg/rules/bundles/import/{source}/`.

### OPA Rego -- Validation & Policy Rules

Handles conditional validation, data quality checks, and conformance policies:
- Row-level validation: account requires code + name + type; contact requires name
- Duplicate detection: ABN match or name similarity > 0.85 with same entity class
- Data completeness warnings: missing ABN (GST reporting), missing email (invoicing)
- Transaction validation: non-zero amount, valid date, mapped account
- Balance verification: source debits/credits match target debits/credits within tolerance
- Import access policy: who can import, batch size limits

All validation rules are in `api/pkg/rules/bundles/import/import_validation.rego`.

### CEL Expressions -- Field Transformations

Handles value conversion, date formatting, amount calculations, and string manipulation:
- Date format conversion (DD/MM/YYYY or MM/DD/YYYY -> ISO 8601)
- Amount sign normalisation (credits as negative, debits as positive)
- GST extraction from inclusive amounts (amount / 11.0, half-up rounding to 2dp)
- Account code padding to minimum 4 digits
- Name normalisation (whitespace trimming)
- Confidence scoring for account matching: `(code_exact * 0.5) + (name_similarity * 0.3) + (same_category * 0.2)`

All transformation formulas are in `api/pkg/rules/bundles/import_transforms_v1.0.0.yaml`.

## Rules File Structure

```
api/pkg/rules/bundles/import/
  xero/
    xero_account_mapping.json        go-rules: account type translation
    xero_tax_mapping.json            go-rules: tax code mapping
    xero_contact_mapping.json        go-rules: contact type mapping
    xero_transaction_mapping.json    go-rules: transaction type mapping
    xero_field_mapping.yaml          column name -> Ledgius field mapping
  myob/
    myob_account_mapping.json        go-rules: account type translation
    myob_tax_mapping.json            go-rules: tax code mapping
    myob_contact_mapping.json        go-rules: contact type mapping
    myob_transaction_mapping.json    go-rules: transaction type mapping
    myob_field_mapping.yaml          column name -> Ledgius field mapping
  import_validation.rego             OPA: validation + duplicate detection
  import_policy.rego                 OPA: access control + limits
  import_transforms_v1.0.0.yaml      CEL: value transformations
```

## Integration with Pipeline Stages

| Pipeline Stage | Engine | Rules Used |
|----------------|--------|------------|
| 1. Upload & Parse | CEL | Date format conversion, field normalisation |
| 2. Analyse & Validate | OPA | Row validation, completeness checks, warnings |
| 3. Map Accounts | go-rules + CEL | Type translation table + confidence scoring |
| 4. Map Contacts | OPA + CEL | Duplicate detection policy + name similarity |
| 5. Preview | OPA + CEL | Transaction validation + balance verification |
| 6. Commit | -- | No rules: pure data insertion in transaction |
| 7. Verify | OPA | Post-commit conformance checks |

## Orchestrator Integration

The import service calls rules through the existing orchestrator interface:

```go
resp, err := orchestrator.Execute(ctx, rules.Request{
    Operation:     "import_validate_account",
    Jurisdiction:  "AU",
    EffectiveDate: time.Now(),
    Input: map[string]any{
        "source_code": row.SourceCode,
        "source_name": row.SourceName,
        "source_type": row.SourceType,
    },
})
```

The orchestrator returns results from all three engines:
- `resp.DecisionResult` -- go-rules output (recommended mapping)
- `resp.PolicyResult` -- OPA output (valid/invalid + warnings)
- `resp.ComputedValues` -- CEL output (confidence score, transformed values)

# Affected Components

- `api/pkg/rules/bundles/import/` -- new rules directory with source-specific subdirectories
- `api/pkg/rules/orchestrator.go` -- existing orchestrator, no changes needed
- `api/internal/importer/` -- import service calls orchestrator per pipeline stage

# Data Flow

```
Staging row
  -> Orchestrator.Execute(operation, jurisdiction, input)
    -> go-rules: lookup recommended mapping (account type, tax code)
    -> OPA: validate row (completeness, duplicates, policy)
    -> CEL: transform values (dates, amounts, confidence)
  -> Combined result returned to import service
  -> Import service updates staging row with mapping + validation state
```

# API / Interface Changes

No new API endpoints. Rules are consumed internally by the import pipeline service. The orchestrator interface is unchanged.

# Storage / Schema Changes

No schema changes. Rules are stored as files in the `bundles/` directory, loaded at startup. Staging tables (defined in A-0017) store the results of rule execution.

# Background Processing

None. Rule execution is synchronous within each pipeline stage request.

# Risks / Trade-offs

- **Rule maintenance burden**: Each new source system requires a full set of mapping rules. Mitigation: the go-rules JSON format is editable by non-developers, and the file structure is consistent across sources.
- **Validation false positives**: OPA duplicate detection may flag legitimate contacts with similar names. Mitigation: duplicates are flagged as warnings, not blocks -- user makes the final decision.
- **CEL date parsing fragility**: Source systems may use unexpected date formats. Mitigation: the CEL transformation tries multiple known formats and falls back to raw string with an error flag.

# Alternatives Considered

1. **Hardcoded Go mappings**: Rejected because adding a new source system would require code changes, compilation, and deployment. Declarative rules allow adding new sources by creating files.
2. **Single rule engine**: Rejected because no single engine handles all three concerns well. go-rules excels at lookup tables, OPA at validation policies, CEL at value transformation.
3. **Database-stored rules**: Rejected because rules are part of the application logic, not tenant data. Version control provides audit trail and diff history.

# Test Strategy

- Unit tests for each go-rules decision table (verify all source type mappings produce correct Ledgius categories)
- Unit tests for OPA validation rules (valid rows pass, invalid rows produce specific error messages)
- Unit tests for CEL transformations (date parsing, amount normalisation, confidence scoring)
- Integration tests verifying orchestrator correctly dispatches to all three engines for import operations
- End-to-end tests running sample Xero and MYOB exports through the full pipeline

# Rollout / Migration Notes

Rules files are deployed as part of the application. No database migration required. New source system support is added by creating a new subdirectory under `bundles/import/` with the appropriate mapping files.
