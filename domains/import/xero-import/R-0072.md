# Summary

Ledgius must be able to import a tenant's accounting data from
Xero, so that a business migrating from Xero to Ledgius can bring
their existing records across. Two delivery channels are required:
**file-based import** (CSV files exported from Xero) and
**API-based import** (direct pull via the Xero Accounting API
using OAuth 2.0). Both channels feed into the existing Ledgius
import staging pipeline (R-0017) with Xero-specific mapping
configurations.

v1 delivers file-based CSV import. v2 adds API-based pull.

# Terms

| **Term** | Definition |
|---|---|
| **Xero** | Cloud-based accounting platform by Xero Ltd. The Australian edition includes GST, BAS, and STP compliance. |
| **ACCREC** | Xero's internal type code for a sales invoice (Accounts Receivable). |
| **ACCPAY** | Xero's internal type code for a purchase bill (Accounts Payable). |
| **ACCRECCREDIT** | Xero's type code for a sales credit note (against a customer). |
| **ACCPAYCREDIT** | Xero's type code for a purchase credit note (against a supplier). |
| **TaxType** | Xero's display name for a tax rate (e.g. "GST on Income", "BAS Excluded"). Mapped to Ledgius tax codes. |

Vocabulary used in this spec for staging, validation, and commit
(import batch, staging table, commit phase) is defined canonically
in **R-0017 §Terms**.

# Problem / Business Context

Xero is the dominant cloud accounting platform in the Australian
and New Zealand SMB market. Businesses migrating to Ledgius from
Xero need a supported path to bring their data across without
manual re-entry. Xero Australia includes GST, BAS lodgement, and
STP Phase 2 compliance — so imported data must map correctly to
Ledgius' Australian tax model.

Xero supports CSV export for chart of accounts (Settings > Chart
of Accounts > Export) and transaction reports (Accounting >
Reports > Detailed Account Transactions > Export). Contacts,
invoices, bills, and credit notes can be exported via Xero's CSV
import/export templates or via the Xero Accounting API.

# Scope

## Supported Import Entities

### v1 — Core data (file-based import)

| Entity | Xero Source | Ledgius Target | Notes |
|---|---|---|---|
| Chart of Accounts | Settings > Chart of Accounts > Export | `account` | **Already implemented.** Columns: Code, Name, Type, TaxType, Description, Balance. |
| Transactions | Reports > Detailed Account Transactions > Export | `transactions` + `acc_trans` | **Already implemented.** Columns: Date, Reference, Contact, Description, AccountCode, TaxType, Debit, Credit. |
| Contacts | Contacts > Export (CSV template) | `entity` + `entity_credit_account` | Name, email, ABN, postal address. IsCustomer/IsSupplier flags determine AR/AP treatment. |
| Invoices | CSV export (Xero import template format) | `ar` + `invoice_line` | Type ACCREC. One row per line item, grouped by InvoiceNumber. Date format DD/MM/YYYY. |
| Bills | CSV export (Xero import template format) | `ap` + `invoice_line` | Type ACCPAY. One row per line item, grouped by InvoiceNumber (bill number). |
| Credit Notes | CSV export (Xero import template format) | `ar` or `ap` (credit type) | Type ACCRECCREDIT (sales) or ACCPAYCREDIT (purchase). One row per line item. |

### v2 — Extended data (API-based import)

| Entity | Xero API Endpoint | Ledgius Target | Notes |
|---|---|---|---|
| Manual Journals | `GET /api.xro/2.0/ManualJournals` | `transactions` + `acc_trans` | Double-entry journals with narration, date, line items. |
| Payments | `GET /api.xro/2.0/Payments` | `acc_trans` | Receipts and bill payments. |
| Items | `GET /api.xro/2.0/Items` | `parts` | Inventory and service items with price, tax code. |
| Bank Transactions | `GET /api.xro/2.0/BankTransactions` | `bank_transaction` | Spend/receive money transactions. |

## File-Based Import (CSV)

- **XRI-001**: The import page shall include "Xero" as a source
  system option with Xero brand colour (blue, `#13B5EA`).
- **XRI-002**: The user shall upload CSV files exported from Xero
  or created using Xero's CSV import template format. The system
  shall auto-detect the entity type by examining column headers.
- **XRI-003**: Each entity type shall have a dedicated YAML mapping
  configuration under `internal/importer/mappings/v1/xero/`.
- **XRI-004**: The mapping engine shall handle Xero-specific field
  names and data formats:
  - Date format: `DD/MM/YYYY` (AU locale)
  - Currency: AUD by default for AU tenants
  - Tax types: Xero display names mapped to Ledgius codes
  - Account types: Xero AccountType enum mapped to Ledgius categories

### Tax Type Mapping

- **XRI-009**: Xero Australian tax display names shall map to
  Ledgius tax codes:

| Xero Tax Type | Ledgius Tax Code | Rate | BAS Label |
|---|---|---|---|
| GST on Income | GST | 10% | G1 (Total sales) |
| GST on Expenses | GST | 10% | G11 (Capital acquisitions) |
| GST on Capital | GST | 10% | G10 (Capital acquisitions) |
| GST Free Income | FRE | 0% | G3 (GST-free sales) |
| GST Free Expenses | FRE | 0% | G3 |
| Input Taxed | INP | 0% | G4 (Input taxed sales) |
| BAS Excluded | N-T | 0% | Not reported on BAS |
| GST on Imports | IMP | 10% | G15 |

### Status Mapping

- **XRI-010**: Xero document statuses shall map to Ledgius statuses:

| Xero Status | Ledgius Status | Notes |
|---|---|---|
| DRAFT | draft | Editable, not posted to ledger |
| SUBMITTED | draft | Treated as draft (Xero approval workflow) |
| AUTHORISED | posted | Posted to ledger, not yet paid |
| PAID | paid | Fully settled |
| VOIDED | voided | Cancelled, compensating entry created |

## Staging and Validation

- **XRI-011**: All imported data enters the staging pipeline
  (R-0017) before committing to the ledger. No direct writes.
- **XRI-012**: Validation shall check:
  - Required fields present (contact name, document number, date)
  - Tax type mapping exists (unmapped types flagged as warnings)
  - Duplicate detection (by document number, contact name+ABN)
  - Amount formats (handle negative values for credit notes)
  - Date parsing (DD/MM/YYYY)
  - Contact references: invoices/bills referencing contacts not
    in staging or not yet committed are flagged
- **XRI-013**: The user shall review staged data with a preview
  grid showing source values alongside mapped Ledgius values,
  with warnings highlighted.

### Import Order and Dependency Resolution

- **XRI-014a**: Import processes entities in dependency order:
  accounts, tax codes, contacts, invoices, bills, credit notes.
  Each stage commits before the next begins.
- **XRI-014b**: Dedup keys: accounts by code+name, contacts by
  ABN+name, invoices by invoice number, bills by bill number,
  credit notes by credit note number.
- **XRI-014c**: Blocking dependency detection — if an invoice
  references an uncommitted contact, or a bill references an
  uncommitted account, the system shall flag the dependency.

## API-Based Import (OAuth 2.0)

- **XRI-015**: **API-based import is a v2 feature.** The initial
  release supports file-based import only. The OAuth connection
  infrastructure already exists (R-0064). The UI shall show
  "API import — coming soon" with the connection card greyed out.

# Out of Scope

- Real-time sync / bi-directional integration
- Payroll data import (STP data is ATO-bound, not portable)
- Bank feed migration (bank feeds are per-provider, not exportable)
- Xero Practice Manager data
- Xero Projects / Expenses data

# Actors / Users

- Bookkeeper — performs the import, reviews staged data
- Accountant — may initiate import for client migration
- Business Owner — triggers migration decision

# Preconditions

- Tenant has a Ledgius account with chart of accounts initialised
- User has exported data from Xero (for file import)
- For API import (v2): Xero OAuth app registered, tokens configured
