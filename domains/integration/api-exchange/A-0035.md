# Summary

Architecture for OAuth 2.0 integration with Xero and MYOB AccountRight
APIs, enabling direct push/pull of accounting data alongside the existing
file-based import/export channels. Built on the existing mapping engine
(A-0033) and import staging pipeline (A-0017) — the API channel is a
different transport, not a different data model.

# Requirement Link

- R-0064 (API-Based Data Exchange)

# Technical Context

Both Xero and MYOB use OAuth 2.0 authorization code flow with short-lived
access tokens and longer-lived refresh tokens. MYOB's access token TTL
(20 minutes) is particularly aggressive, requiring proactive refresh.
The architecture must handle token lifecycle transparently so the user
never sees an auth error during a push/pull operation.

## OAuth Flow

### Connect (one-time per tenant × system)

```
User clicks "Connect to Xero"
  → GET /connect/xero
  → 302 redirect to Xero login with state=JWT(tenant_id, nonce)
  → User logs in, authorises
  → Xero redirects to /connect/xero/callback?code=xxx&state=yyy
  → Backend exchanges code for tokens
  → GET /connections to select tenant/org (Xero multi-org)
  → Tokens encrypted + stored in external_connection
  → 302 redirect to /export with success flash
```

### Token Refresh (automatic)

```
Before every API call:
  1. Load external_connection for this tenant × system
  2. If token_expiry < now + 2min:
     a. POST to vendor token endpoint with refresh_token
     b. Store new access_token + refresh_token + expiry
  3. Use (possibly refreshed) access_token for the API call
```

### Disconnect

```
DELETE /connect/xero
  → POST to Xero revocation endpoint with token
  → DELETE external_connection row
  → 200 OK
```

## API Client Design

Each vendor client wraps `http.Client` with:

1. **Auto-refresh**: checks token expiry before each call
2. **Rate limiting**: token bucket (Xero: 60/min; MYOB: per response headers)
3. **Pagination**: automatic page-through for GET endpoints
4. **Error mapping**: vendor-specific error → Ledgius error type
5. **Retry**: 429/5xx with exponential backoff (max 3 retries)

```go
type XeroClient struct {
    tokenStore  *oauth.TokenStore
    tenantID    string
    httpClient  *http.Client
    rateLimiter *rate.Limiter
}

func (c *XeroClient) GetAccounts(ctx context.Context) ([]mapping.SourceRow, error)
func (c *XeroClient) PutAccounts(ctx context.Context, rows []mapping.TargetRow) error
func (c *XeroClient) GetContacts(ctx context.Context) ([]mapping.SourceRow, error)
// ... etc
```

## Push Flow (Export via API)

```
POST /export/xero/run { channel: "api" }
  → Create export_run (channel=api)
  → Fetch from Ledgius DB (same DBProvider as file export)
  → Map via mapping engine (same YAML configs)
  → Instead of writing CSV/tab files:
    → For each entity batch:
      → XeroClient.PutAccounts(targetRows)
      → Rate-limit between batches
      → Record success/failure per entity
  → Update export_run to completed
  → SSE stream reports progress per entity
```

## Pull Flow (Import via API)

```
POST /import/batches { source_system: "xero_api" }
  → Create import_batch
  → XeroClient.GetAccounts() → []mapping.SourceRow
  → Transform via mapping engine (xero-import YAML configs)
  → Feed into existing MapToStagingAccounts()
  → Insert into import_staging_account
  → Continue normal pipeline: analyse → map → preview → commit
```

# Affected Components

- `internal/oauth/` — new package: OAuth handler, token store, token
  encryption, auto-refresh middleware
- `internal/apiclient/xero/` — new: Xero REST client with rate limiting
- `internal/apiclient/myob/` — new: MYOB REST client with rate limiting
- `internal/exporter/xero/runner/` — extended: support channel='api'
  alongside channel='csv'
- `internal/exporter/myob/runner/` — extended: support channel='api'
- `internal/importer/service.go` — extended: support source_system
  'xero_api' and 'myob_api' that pull via API client
- `cmd/server/main.go` — wire OAuth routes + API client factories

# Storage / Schema Changes

New table in tenant database:

```sql
CREATE TABLE external_connection (
    id                SERIAL PRIMARY KEY,
    tenant_id         TEXT NOT NULL,
    system_code       TEXT NOT NULL,        -- 'xero', 'myob'
    access_token_enc  BYTEA NOT NULL,       -- AES-256-GCM encrypted
    refresh_token_enc BYTEA NOT NULL,
    token_expiry      TIMESTAMPTZ NOT NULL,
    org_id            TEXT,                 -- Xero tenant_id, MYOB company file GUID
    org_name          TEXT,
    status            TEXT NOT NULL DEFAULT 'active', -- active, expired, revoked
    connected_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    connected_by      TEXT,
    updated_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE (tenant_id, system_code)
);
```

Platform-level configuration (environment variables):

```
XERO_CLIENT_ID=...
XERO_CLIENT_SECRET=...
XERO_REDIRECT_URI=https://api.ledgius.com/v1/connect/xero/callback
MYOB_CLIENT_ID=...
MYOB_CLIENT_SECRET=...
MYOB_REDIRECT_URI=https://api.ledgius.com/v1/connect/myob/callback
OAUTH_ENCRYPTION_KEY=...  # 32-byte key for AES-256-GCM token encryption
```

# Risks / Trade-offs

- **Token expiry race**: MYOB's 20-minute access token can expire
  mid-batch on large exports. Mitigation: refresh proactively (2min
  before expiry), retry on 401 with fresh token.
- **Rate limiting**: Xero's 60/min limit constrains large pushes.
  Mitigation: batch entities, backoff on 429, report progress via SSE.
- **Vendor API changes**: Both vendors evolve their APIs. Mitigation:
  version-pinned endpoints, vendored catalogues in ledgius-specs,
  integration tests against sandbox accounts.
- **Token security**: OAuth tokens are high-value credentials.
  Mitigation: encrypted at rest, never logged, never returned to
  frontend, rotated on refresh.

# Alternatives Considered

1. **SDK-based integration**: Use Xero's official Go SDK (xerogolang).
   Rejected because the SDK is unmaintained (last commit 2021) and
   doesn't support OAuth 2.0 well. A thin REST client with `net/http`
   is simpler and more maintainable.

2. **Webhook-based sync**: Xero supports webhooks for change
   notifications. Rejected for v1 — adds infrastructure complexity
   (public endpoint, signature verification, delivery guarantees).
   Can be added later as an enhancement.

3. **Third-party integration platform**: Use Zapier/Make/Workato.
   Rejected — adds dependency, cost, and latency. Direct API is
   faster and gives full control over data mapping.

# Implementation Phases

## Phase 1: OAuth Foundation
- `external_connection` migration
- `internal/oauth/` package (handler, token store, encryption)
- Connect/disconnect endpoints for Xero
- Connection status API
- Web UI: connection cards in Data mode

## Phase 2: Xero API Client + Pull
- `internal/apiclient/xero/` — GET endpoints with pagination
- Pull from Xero → import staging pipeline
- Import page "Pull from Xero" button

## Phase 3: Xero API Push
- XeroClient PUT endpoints
- Export page "Push to Xero" alongside file export
- SSE progress for API push

## Phase 4: MYOB OAuth + Client
- Connect/disconnect for MYOB (post-March 2025 flow)
- `internal/apiclient/myob/` — GET + POST endpoints
- Pull from MYOB + Push to MYOB
