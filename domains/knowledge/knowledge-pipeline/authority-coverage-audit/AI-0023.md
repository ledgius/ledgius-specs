# Summary

AI guidance for maintaining and improving authority coverage across all rules in the Ledgius rules engine. Provides the current audit baseline (147 rules, 80% coverage), coverage breakdown by engine, the required knowledge articles list, and instructions for classifying rules by authority provenance and closing coverage gaps.

# Authoritative Sources

The audit covers all rules defined in `api/pkg/rules/bundles/`:
- CEL bundles: `gst_v1.0.0.yaml`, `payroll_v1.0.0.yaml`, `accounting_v1.0.0.yaml`
- OPA Rego policies: `transaction_validation.rego`, `access_control.rego`, `account_validation.rego`, `double_entry.rego`
- Import transforms: `import_transforms_v1.0.0.yaml`
- Import validation: `import_validation.rego`
- go-rules JSON: Account mappings (Xero, MYOB), Tax mappings (Xero, MYOB), Field mappings (Xero, MYOB)

Authority sources referenced by existing rules:
- ATO GST Act (Division 9, s9-70)
- ATO BAS instructions
- ATO PAYG Schedule 1
- ATO SG Act
- AASB 107 (Cash Flow Statements)
- IFRS (general)
- General accounting principles (double-entry bookkeeping)
- Ledgius internal policies (POL-009, POL-011, POL-013, POL-016, POL-017, POL-022)
- Ledgius schema definitions (LEDGIUS-SCHEMA-001)

## Coverage Gaps

The 29 rules with no authority specified are concentrated in:
- Import transforms CEL (6 rules): date normalisation, amount normalisation, code padding, confidence scoring, span calculation, balance difference
- Import validation Rego (16 rules): account/contact/transaction validation, duplicate detection, batch verification
- CEL bundles (4 rules): scattered across GST, payroll, and accounting bundles -- these already have `Description` and `Authority` string fields but 4 formulas have only partial or empty authority

Import rules (transforms + validation) account for 22 of 29 uncovered rules.

# Modules / Components to Prefer

- `api/pkg/rules/bundles/` -- all rule definition files that need authority metadata
- `docs/authority/articles/` -- knowledge article YAML files
- `api/pkg/rules/cel/engine.go` -- `FormulaSpec.AuthorityRefs` field for forward links
- `api/pkg/rules/opa/engine.go` -- `PolicyResult.PolicyRefs` for OPA authority references
- Knowledge article `implemented_by` section for reverse links

# Modules / Components to Avoid

- Do not add authority metadata to Go service code -- authority belongs on rule definitions (CEL/OPA/go-rules), not on the Go functions that call the rules engine
- Do not create ad-hoc authority strings -- use the structured `AuthorityRef` type with article ID + section ID + relationship

# Non-goals

- This audit does not evaluate the correctness of rules, only whether they cite an authority
- This audit does not evaluate the quality or currency of cited authorities
- This audit does not cover rules that may exist in Go service code but have not yet been migrated to the rules engine

# Testing Expectations

- Build-time CI check shall validate all bidirectional links (rule -> article, article -> rule)
- New rules added to bundles must include `authority` or `authority_refs` fields -- CI should warn on missing authority
- Knowledge article YAML files must pass schema validation

# Provenance Expectations

- Every rule must be classifiable into one of: `external_regulatory`, `external_standard`, `industry_practice`, `internal_policy`
- External rules must link to a downloaded/ingested source document (article with `source_url` and `source_hash`)
- Internal rules must link to a Ledgius policy article that explains the rule's purpose and rationale
- The `no authority specified` count should be driven to zero

## Classification Guide

When classifying a rule:
1. Check if the rule cites an ATO, AASB, Fair Work, or legislation reference -> `external_regulatory` or `external_standard`
2. Check if the rule implements a Ledgius business decision (POL-nnn) -> `internal_policy`
3. Check if the rule enforces a schema constraint (account categories, entity classes) -> `internal_policy` (schema)
4. Check if the rule follows industry convention without a specific regulatory mandate -> `industry_practice`
5. If none of the above, search ATO and legislation sites for the underlying requirement. If found, classify as external. If not found, classify as `internal_policy` and generate a Ledgius policy article.

# Prompting Notes

- When adding new rules to any bundle, always include an `authority` field citing the source
- When the authority is ATO, use the structured `authority_refs` format: `{article: "ATO-...", section: "...", relationship: "implements"}`
- When the authority is internal, create or reference a Ledgius policy article under `docs/authority/articles/`
- Import rules are the biggest gap -- prioritise adding `LEDGIUS-IMPORT-POLICY` article and linking all 22 uncovered import rules
- The 6 internal policy articles (LEDGIUS-TXN-POLICY, LEDGIUS-ACCESS-CONTROL, etc.) can be created immediately without external research

## Recommended Remediation Order

1. **Immediate**: Add `authority` fields to `import_transforms_v1.0.0.yaml` (8 formulas, currently blank)
2. **Immediate**: Add `policy_refs` to `import_validation.rego` rules (19 rules, currently absent)
3. **Phase 1**: Create the 6 internal Ledgius policy articles (no external research needed)
4. **Phase 2**: Search and ingest the 4 high-priority ATO articles (PAYG, SG, GST Act, BAS)
5. **Phase 3**: Complete the remaining 4 external articles (TFN, AASB 107, double-entry, general principles)
6. **Ongoing**: Add `review_date` to all time-bounded articles (PAYG schedules, SG rates)
