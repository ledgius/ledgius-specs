# Summary

Production architecture for a knowledge compiler with governance that ingests authoritative accounting and tax materials, normalises them into a canonical knowledge model, synthesises reviewable executable artifacts (CEL, OPA Rego, go-rules JSON), and publishes only validated, schema-aligned, conflict-free, temporally-correct, human-approved outputs. This is not a scraper or a free-form LLM rule generator -- it is a governed pipeline with provenance, schema binding, and mandatory human review.

# Requirement Link

- R-0008 (Knowledge Pipeline Requirements)

# Technical Context

Australian authoritative sources (ATO rulings, GST Act, Fair Work Act, AASB standards, SBR taxonomy) are heterogeneous in format (PDF, HTML, XML/XSD/XBRL, CSV) and access method (direct download, server-rendered HTML, JavaScript-rendered content). The ingestion pipeline must handle all these formats while maintaining provenance, citation-level addressability, and schema-bound artifact generation. The existing Ledgius rules engine (CEL + OPA + go-rules, coordinated via `pkg/rules/` orchestrator) is the target execution platform for all generated artifacts.

The design is based on a v1 file-based knowledge store (`docs/authority/articles/*.yaml`) for near-term implementation, with a PostgreSQL-backed knowledge catalog for the full production pipeline.

# Proposed Design

## Non-Negotiable Design Invariants

1. Every artifact must reference at least one source passage, bind to one approved schema version, map to one or more approved taxonomy nodes, have effective dates, have a review state, pass all validators, and have no unresolved active conflicts.
2. No artifact may be used at runtime unless: `status = published`, `execution_ready = true`, current time is within effective window, it is not retired, and it is not superseded by an active successor.
3. No free-text domain labelling is permitted for production artifacts.
4. No production artifact may be synthesised from source material with unresolved provenance.
5. Compilation success is not semantic correctness.
6. Ambiguous runtime selection is a hard failure, not a warning.
7. Every field name used by an artifact must resolve to an approved field in the bound schema version.

## Production Pipeline Stages

```
Source Registry
  -> Discover
  -> Acquire (tiered: Direct -> Colly -> chromedp)
  -> Canonicalise
  -> Segment Passages
  -> Extract Statements
  -> Bind to Schema + Taxonomy
  -> Synthesise Artifacts (CEL / Rego / go-rules)
  -> Validate + Detect Conflicts
  -> Generate Tests
  -> Human Review
  -> Publish Bundle
  -> Runtime Catalog / Resolver
```

## Tiered Acquisition Strategy

The system uses three acquisition modes because Australian authoritative sources are heterogeneous:

**Tier 1 -- Direct Structured Fetch (preferred)**: For XML, XSD, XBRL taxonomy files, downloadable PDFs, CSV, ZIP packages. Essential for SBR because its taxonomy is exposed as downloadable XSD/XML resources.

**Tier 2 -- HTML + Colly**: For server-rendered guidance pages, index pages, browse pages, ATO help/index content, legislation landing pages.

**Tier 3 -- chromedp fallback**: Only when initial HTML lacks target content, navigation requires JavaScript, or important content is rendered only after browser execution.

### Source-Specific Adapter Strategies

**ATO**: Use known browse pages, quick access patterns, hand-curated seed sets, category-specific paths. Do not rely on broad site crawl -- the ATO Legal Database may restrict crawler-style access.

**SBR**: Treat as a structured source. Use taxonomy directory listings, readme files, XSD/XML parsing, version detection by directory naming. Ingest both definitional and report taxonomies.

**Legislation**: Use document lifecycle metadata, text/download links, known document identifiers, point-in-time awareness where available.

Each source family has an explicit adapter; no generic crawler-only mode is sufficient.

## Canonical Data Model

### Source and Passage Model

- `source_system`: Authority source registry with name, authority level, base URL, access mode, crawl policy
- `source_document`: Individual document with canonical URL, document family/type, jurisdiction, authority rank, effective dates
- `source_document_version`: Versioned snapshots with raw blob reference, normalised text, structured JSON, content hash, HTTP etag
- `source_passage`: Citation-addressable segments (1 paragraph / 1 table row group / 1 bullet block / 1 legislative subsection cluster)

### Taxonomy Model

Controlled, hierarchical taxonomy with aliases:
- `taxonomy_node`: Hierarchical tree with canonical keys (e.g. `tax.gst.classification.taxable_supply`, `tax.payg.withholding`, `reporting.bas.label.1A`)
- `taxonomy_alias`: Alternative names mapping to canonical nodes
- `artifact_taxonomy`: Join table linking artifacts to taxonomy nodes

LLM may propose new nodes, aliases, parent assignments, and duplicate merge candidates, but only reviewed taxonomy mutations become active.

### Schema Registry

- `entity_schema`: Versioned entity models (Transaction, Invoice, BASReport, TaxTreatment, etc.) with JSON schema
- `schema_field`: Field-level metadata with path, type, nullability, enum reference
- `enumeration`: Controlled value lists

Every field path used in CEL, Rego, or go-rules must exist in `schema_field` for the bound schema version.

### Knowledge Statement Model

Statements bridge source material to executable artifacts:
- `knowledge_statement`: Structured extraction with rule_type (classification, eligibility, calculation, reporting_mapping, validation, orchestration_hint), conditions, outcomes, exceptions, assumptions, explicit/inferred flag, semantic confidence, legal risk level
- `statement_citation`: Links statements to source passages with support type

Inferred statements (`explicit_or_inferred = inferred`) cannot be published without elevated review.

### Artifact Model

- `artifact`: Common catalog with type (CEL, Rego, go-rules), unique key, taxonomy binding, schema binding, version, jurisdiction, effective dates, review status, execution readiness, semantic confidence, legal risk level, supersession references
- `artifact_citation`: Links artifacts to source passages
- `artifact_conflict`: Detected conflicts between overlapping artifacts

## Binding Validation

Family-specific validators prevent schema/artifact mismatch:

**CEL Binding Validator**: Checks referenced variables exist, field paths exist in schema, operators are type-compatible, return type matches declaration.

**Rego Binding Validator**: Checks input/document schema exists, field references are valid, JSON schema-based type validation passes, package path is canonical.

**go-rules JSON Validator**: Checks action names are valid, referenced field paths exist, sub-artifacts exist, sequencing graph is acyclic, accumulation targets are legal.

No artifact can be `execution_ready` without passing its family-specific binder.

## Conflict Detection

Detects: overlapping applicability with contradictory output, same taxonomy + same schema + same effective window with different result, duplicate semantics with different authority, reporting label double-count risk, mutually inconsistent validation outcomes.

Conflicting artifacts may not both be active unless an explicit reviewed precedence rule exists.

## Runtime Resolver

Deterministic resolution algorithm:
1. Filter by artifact_type
2. Filter by taxonomy node or subtree
3. Filter by exact schema compatibility
4. Filter by jurisdiction
5. Filter by effective window
6. Remove retired artifacts
7. Remove artifacts superseded by active replacements
8. Sort by: higher authority, more specific taxonomy, narrower schema match, explicit priority, higher version
9. If 0 matches: return not found
10. If >1 materially equivalent matches: raise ambiguity (hard failure)
11. Return selected artifact and provenance summary

## Publication Bundles

Published artifacts are distributed as versioned bundles containing: artifact manifest, schema manifest, taxonomy manifest, provenance manifest, citations, source hashes, changelog, integrity checksum. Example bundles: `gst-core`, `bas-mapping`, `payg-withholding`, `entity-schemas`.

## Human Review Model

Review roles: accounting/tax reviewer, engineering reviewer, optional legal reviewer for high-risk or inferred items.

Review UI shows: exact source passage text, source URLs and document versions, plain-language statement, generated CEL/Rego/go-rules, schema binding details, taxonomy placement, semantic confidence, risk level, generated tests, previous active version diff.

Decision options: approve, approve with edits, reject, request regeneration, mark ambiguous, escalate.

No machine-generated artifact transitions directly from generated to published.

## Orchestration

**River** (PostgreSQL-backed job queue) for pipeline orchestration. Jobs include: discover, fetch, parse, canonicalise, extract passages, extract statements, propose taxonomy, synthesise artifacts (per engine), validate, detect conflicts, generate tests, open review tasks, publish bundle, refresh source, diff versions, retire superseded.

Workers are idempotent, safe on retry, with explicit state transitions, poison-document quarantine, and structured logging.

Temporal is deferred unless workflow orchestration becomes the dominant complexity.

# Data Flow

```
Source Registry defines authoritative sources
  -> Discovery job finds new/updated documents
  -> Acquisition job fetches content (Direct/Colly/chromedp)
  -> Canonicalisation normalises to clean text + structured JSON
  -> Passage segmentation creates citation-addressable units
  -> Statement extraction (LLM-assisted) creates structured knowledge statements
  -> Schema + taxonomy binding validates and assigns
  -> Artifact synthesis generates CEL/Rego/go-rules
  -> Validation chain: syntax, schema binding, metadata, citations, type compatibility
  -> Conflict detection checks for overlapping contradictory artifacts
  -> Test generation creates positive/negative/boundary scenarios
  -> Human review task opened with full provenance
  -> On approval: artifact published into versioned bundle
  -> Runtime resolver serves published artifacts to rules engine
```

# Background Processing

River jobs for all pipeline stages. Workers are idempotent with retry budgets and dead-letter/quarantine states. Adapter health monitoring tracks per-source fetch success, parse failure rates, and browser fallback rates.

# Risks / Trade-offs

- **LLM extraction quality**: LLM-assisted statement extraction may produce syntactically valid but semantically incorrect rules. Mitigation: mandatory human review, explicit/inferred distinction, elevated review for high-risk inferred statements.
- **Source access fragility**: ATO or other sources may change access patterns. Mitigation: tiered acquisition, hand-curated seed URLs, adapter-specific strategies, no dependence on unrestricted crawling.
- **Schema drift**: Schemas evolve over time and artifacts may reference obsolete fields. Mitigation: schema registry with versioning, binding validators reject artifacts referencing missing fields.
- **Taxonomy complexity**: Taxonomy may grow unwieldy with LLM-proposed nodes. Mitigation: all taxonomy mutations require review, canonical keys are immutable once published.
- **Pipeline complexity**: Full pipeline has many stages and dependencies. Mitigation: phased implementation starting with file-based store, upgrading to full pipeline as complexity is proven necessary.

# Alternatives Considered

1. **Manual rule authoring only**: Rejected because the volume of authoritative sources (ATO, legislation, AASB) makes manual maintenance unsustainable.
2. **Unstructured LLM generation**: Rejected because free-form LLM output without schema binding, provenance, or review is unsafe for accounting and tax systems.
3. **Temporal for orchestration**: Deferred to future phase. River (PostgreSQL-backed) is sufficient for current pipeline complexity and simpler to deploy.
4. **Browser-only acquisition**: Rejected as too expensive and brittle. Tiered approach (Direct > Colly > chromedp) is more robust and efficient.

# Test Strategy

- Check A: Can SBR be ingested without scraping? (taxonomy enumeration, XSD/XML parsing, concept materialisation)
- Check B: Can ATO be ingested without full-site crawl? (curated seeds, known patterns, category browse)
- Check C: Can CEL artifacts bind to schemas? (invalid field path rejection, type mismatch rejection)
- Check D: Can Rego align to schemas? (policy compilation, OPA schema checking, metadata annotations)
- Check E: Can runtime reliably select the right artifact? (deterministic resolution, ambiguity produces failure)
- Check F: Can rule catalog evolve without chaos? (taxonomy review, alias tracking, canonical key stability)
- Generated test cases for every published artifact: positive path, negative path, threshold edges, date-effective scenarios, schema compatibility, ambiguity, supersession, label/report mapping totals

# Rollout / Migration Notes

## Implementation Phases

**Phase 1**: Source registry, direct fetch + Colly, SBR adapter, ATO adapter using curated seeds, canonical text, passage segmentation, schema registry, CEL generation, manual review, artifact catalog.

**Phase 2**: Rego generation, go-rules JSON generation, taxonomy control, conflict detection, runtime resolver, bundle publication.

**Phase 3**: chromedp fallback, source diffing and supersession automation, advanced ambiguity handling, richer reviewer tooling.

**Phase 4**: Revisit Temporal only if workflow orchestration becomes dominant complexity.

Near-term: file-based YAML articles and manifests deployed as part of the application. No database migration for Phase 1.
