# Summary

Ingest, cleanse, store, and link authoritative source materials (legislation, ATO rulings, accounting standards) to the executable rules engine (CEL, OPA Rego, go-rules). Surface these as human-readable policy explanations in the UI help panel, contextual to the page the user is viewing. Every rule in the system must be traceable to an authority -- either an external regulatory source or an internal Ledgius policy document.

# Problem / Business Context

Users of an accounting system should never have to guess why a calculation works a certain way or what legislation drives a validation rule. Auditors need to trace any calculation back to its regulatory authority to verify compliance. Without explicit authority linking, rules become opaque business logic that cannot be independently verified. Australian tax compliance requires traceability to ATO rulings, the GST Act, Fair Work Act, AASB standards, and STP Phase 2 reporting requirements.

# Product Vision — Help Capability Tiers

Competing accounting products ship traditional help: users treat it as a "last resort", scan for a small bit of information, and exit. That level of help is acceptable but undifferentiated. Ledgius aims beyond it.

The knowledge pipeline is the substrate for a multi-tier capability arc. Each tier builds on the prior and is tracked as its own task series:

| Tier | Name | User experience | Status |
|---|---|---|---|
| 1 | **Reference** | Static, jurisdiction-scoped articles resolved by domain + page (T-0038) | Shipped |
| 2 | **Contextual** | Articles vary by tenant registrations, active period, business setup (T-0039) | Shipped |
| 3 | **Primary + Related ranking** | Page-bound "primary" article separated from cross-references | Shipped |
| 4 | **Operational status** | Live checklist / counters showing what's done vs pending on the page (T-0040, planned) | Planned |
| 5 | **Guided workflow** | System actively tells the user where they are in a multi-step flow (period close, BAS lodgement, onboarding) and what to do next — holds the user's hand through the work | Vision |
| 6 | **Anticipatory** | Surfaces the right help *before* the user asks — triggered by behaviour signals (hesitation, repeated errors, first-time feature use) | Vision |

**Why this matters:** traditional help is a safety net; guided help is a multiplier. A bookkeeper who is *told* their BAS is 4 of 5 checklist items complete, with the remaining item linked from the help panel, finishes faster and with less error than one who has to check progress manually and navigate search. The guided + anticipatory tiers are the product differentiator — not the reference content itself.

**How this shapes work:** every help-system feature ask — tooltip, onboarding, info panel enhancement, notification — must be evaluated as a step along this arc, not an isolated feature. New requirements should name which tier they advance. Tier-5 / Tier-6 work needs an explicit spec in its own time; this section exists so the direction is durable across the many individual tasks that incrementally build toward it.

# Scope

- Knowledge article management with structured, addressable sections and provenance metadata
- Bidirectional linking between rules (CEL, OPA, go-rules) and knowledge articles
- Authority classification (external_regulatory, external_standard, industry_practice, internal_policy)
- Authority discovery against known authoritative domains
- Staleness detection for time-bounded authorities
- Page-level policy resolution via domain-tag join with manifest overrides
- UI help panel Policies tab showing contextual authority content
- Build-time link validation (CI check)

# Out of Scope

- Automated ingestion pipeline (covered in A-0020)
- NZ, UK, or other jurisdiction content (schema supports it, content deferred)
- AI/ML-based authority discovery (manual and targeted search only for now)
- Real-time ATO feed integration

# Actors / Users

- Bookkeeper -- views policy explanations in help panel while processing transactions
- Accountant -- verifies regulatory authority for calculations and BAS figures
- Business Owner -- understands why rules exist (e.g. credit note caps) via plain English policies
- Auditor -- traces calculations back to regulatory authority for compliance verification

# Preconditions

- Rules engine is operational (CEL, OPA, go-rules bundles deployed)
- Help panel UI exists with single-tab display

# Functional Requirements

## Knowledge Article Management

- **KNW-PIP-001**: System shall store authoritative source materials as structured knowledge articles with stable, addressable section IDs
- **KNW-PIP-002**: Each knowledge article shall record provenance metadata: source URL, source file path, issuer, jurisdiction, retrieval date, and content hash (SHA-256)
- **KNW-PIP-003**: Each article section shall have an effective date range (start, optional end) for temporal applicability
- **KNW-PIP-004**: System shall store cleansed, presentation-ready article content suitable for direct display in the UI help panel
- **KNW-PIP-005**: System shall support storing raw source documents (PDF, HTML) alongside cleansed articles for audit reference

## Bidirectional Linking

- **KNW-PIP-006**: Every CEL formula, OPA policy, and go-rules decision shall carry forward links (`authority_refs`) to the knowledge article(s) and section(s) it implements. Structure: article ID + section ID + relationship type
- **KNW-PIP-007**: Every knowledge article shall carry reverse links (`implemented_by`) identifying which rules implement each section. Structure: engine + bundle/policy + formula/rule name + section ID
- **KNW-PIP-008**: System shall enforce referential integrity -- a rule cannot reference a non-existent article or section, and vice versa
- **KNW-PIP-009**: System shall support multiple relationship types between rules and articles: `implements`, `derived_from`, `constrained_by`

## Authority Classification

- **KNW-PIP-010**: Every rule shall be classified by authority provenance: `external_regulatory`, `external_standard`, `industry_practice`, or `internal_policy`
- **KNW-PIP-011**: Rules classified as `internal_policy` shall have a corresponding Ledgius policy document generated as their authority article
- **KNW-PIP-012**: Rules classified as `external_regulatory` or `external_standard` shall link to a downloaded/ingested source document

## Authority Discovery

- **KNW-PIP-013**: System shall support discovery of authoritative sources for rules that lack authority links, using targeted search against known domains (ato.gov.au, legislation.gov.au, fairwork.gov.au, aasb.gov.au), with wider search as fallback
- **KNW-PIP-014**: When no external authority can be found for a rule, the default provenance shall be "Ledgius" (internal policy), and a policy document shall be generated

## Staleness Detection

- **KNW-PIP-015**: System shall detect when a source document has changed by comparing content hashes on re-retrieval
- **KNW-PIP-016**: Articles linked to time-bounded authorities (FY-specific PAYG schedules, annual SG rates) shall have review dates that trigger staleness warnings

## Page-Level Policy Resolution

- **KNW-PIP-017**: System shall resolve relevant articles for a page using a domain-tag join: pages declare their domains, articles declare their domains, intersection determines inclusion. Optional manifest overrides for edge cases.
- **KNW-PIP-018**: The API shall serve resolved policy content for a given page route, including article sections with their implementing rules attached

## UI -- Help Panel Policies Tab

- **KNW-PIP-019**: The help panel sidebar shall include a "Policies" tab alongside the existing "Help" tab
- **KNW-PIP-020**: The Policies tab shall display all knowledge articles relevant to the current page, grouped by article
- **KNW-PIP-021**: Each article section shall show: cleansed source text, implementing rule(s), authority issuer, and effective date range
- **KNW-PIP-022**: The Policies tab shall visually distinguish external authorities (badge: "ATO", "Fair Work", "AASB") from internal policies (badge: "Ledgius")
- **KNW-PIP-023**: Article sections shall be expandable/collapsible to manage information density

## Non-Functional Requirements

- **KNW-PIP-024**: Knowledge article loading shall complete in under 200ms for any page route
- **KNW-PIP-025**: All knowledge articles and authority links shall be stored as version-controlled YAML files (not in the database) -- these are application source, not tenant data
- **KNW-PIP-026**: The system shall validate all bidirectional links at build time (CI check) to prevent broken references
- **KNW-PIP-027**: Knowledge articles shall be jurisdiction-extensible -- the same structure must support AU, NZ, UK, and other jurisdictions

# Data / Entities / Fields

Primary entity: `KnowledgeArticle` (file-based, YAML)

| Field | Type | Notes |
|-------|------|-------|
| id | string | Stable identifier, e.g. "ATO-PAYG-SCHED1" |
| title | string | Human-readable title |
| jurisdiction | string | "AU", "NZ", "UK", etc. |
| issuer | string | "Australian Taxation Office", "Ledgius", etc. |
| authority_type | string | external_regulatory, external_standard, industry_practice, internal_policy |
| source_url | string (nullable) | Original URL of source document |
| source_file | string (nullable) | Path to raw source file |
| source_hash | string (nullable) | SHA-256 of raw source content |
| retrieved_at | date (nullable) | When source was last downloaded |
| effective | object | { start: date, end: date (nullable) } |
| review_date | date (nullable) | When article should be re-verified |
| domains | array of string | Domain tags for page resolution |
| sections | array of ArticleSection | Addressable content sections |
| implemented_by | array of RuleLink | Reverse links to implementing rules |

Related entities: `ArticleSection` (id, heading, body, effective), `AuthorityRef` (article, section, relationship), `PageManifest` (route, articles, context)

# UX / UI Behaviour

- Help panel gains a second tab ("Policies") alongside existing "Help" tab
- Policies tab content updates automatically per page via `usePagePolicies` hook
- Each article shows: title, authority badge (colour-coded by type), effective dates
- Each section is collapsible with heading, body text, and "Enforced by" rule references
- Authority badge colours: blue (external regulatory), purple (external standard), green (industry practice), gray (internal policy)

# Validation Rules

- Article ID must be unique across all articles
- Section ID must be unique within an article
- All `authority_refs` on rules must reference existing article IDs and section IDs
- All `implemented_by` on articles must reference existing rules
- Effective start date is required on all articles
- Domain tags must be from the approved domain taxonomy

# Error Handling

- Broken article/rule links detected at build time produce CI failure with specific broken reference listed
- Missing articles for a page route return empty Policies tab (not an error)
- Staleness warnings appear in audit report, not as runtime errors

# Security / Permissions

- Knowledge articles are read-only in the UI (no tenant editing)
- API endpoints are public (no auth required for policy transparency)
- Raw source documents stored in version control (not exposed via API)

# Open Questions

1. **Ingestion quality**: LLM-assisted extraction from PDFs may introduce errors. Human review of all ingested articles before linking is recommended.
2. **Staleness lag**: ATO publishes new schedules annually (July). Articles must be refreshed promptly. review_date field + CI warning provides mitigation.
3. **Volume**: How many articles are needed for initial release? Recommendation: start with ~10 core articles covering PAYG, GST, SG, double-entry, and internal policies.
4. **Internal policy generation**: For the ~107 internal rules, clear policy documents must be written. This is a content authoring task, not a code task.
5. **Jurisdiction expansion**: The schema supports it, but NZ/UK content is out of scope for initial release.
