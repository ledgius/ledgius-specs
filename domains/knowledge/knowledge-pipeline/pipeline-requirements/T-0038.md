# Summary

R-0008 is the authoritative spec for page help + policy content. It mandates that articles are **cleansed, presentation-ready** (§KNW-PIP-004), **API-served by page route** (§KNW-PIP-018), **jurisdiction-extensible** (§KNW-PIP-027), and **stored as version-controlled YAML** outside the web-app bundle (§KNW-PIP-025).

In practice, the fixed-assets and liabilities features have been landing help content as:

- YAML files bundled with the web app at `src/locales/en-AU/help/**/*.yaml`
- An inline `pageHelpContent` map in `src/hooks/usePageHelp.ts`

That tier contradicts R-0008. It locks help text to a single jurisdiction (AU), a single locale (en-AU), a web-app release cycle, and bypasses the review/citation discipline that already applies to the policy side of the same panel.

This task retires that split. Every existing help section becomes a knowledge article with `authority_type='internal_policy'`, `issuer='Ledgius'`, `jurisdiction='AU'`, `locale='en-AU'`, and appropriate domain tags — served through the existing `GET /api/v1/knowledge/articles` endpoint. The web-app becomes a dumb renderer with one resolved-content hook.

No new architecture. This is a **migration**, not a redesign.

# Terms

Vocabulary used in this task (knowledge article, article section, authority type, domain-tag join, page manifest, page override, authority directory) is defined canonically in **R-0008 §Terms and §Data / Entities / Fields**. New terms introduced here:

| **Term** | Definition. |
|---|---|
| **Help section** | An `ArticleSection` within a knowledge article whose parent article has `authority_type='internal_policy'` and `issuer='Ledgius'`. Renders in the same help panel as external authority sections; distinguishable by the "Ledgius" badge per R-0008 §KNW-PIP-022. |
| **Resolved page content** | The combined set of knowledge articles returned by `GET /api/v1/knowledge/articles?page=&domains=`, covering both internal help sections (this task) and external authority articles (already shipped). |
| **Authority store ETag** | A single SHA-256 over the in-memory store's canonical serialisation, computed at store-load time. Served in the `ETag` response header on the resolver endpoints so clients can short-circuit repeat requests with `If-None-Match`. |

# Problem

Two related failures of the current split:

1. **Spec contradiction.** R-0008 §KNW-PIP-004 explicitly says article content is suitable for direct display in the help panel. The help panel has a "Policies" tab already wired to `/api/v1/knowledge/articles` via `usePagePolicies`. But there's a second, parallel "Help" tab fed by frontend YAML and an inline map — same panel, same users, different content pipeline, different review discipline.

2. **Jurisdiction + locale coupling.** Help content today hard-codes "AU" everywhere — s8-1 ITAA, AASB 116 §68, CAP tax code, ATO instant write-off thresholds. When the first NZ or UK tenant lands, we either (a) fork `src/locales/`, (b) bundle all jurisdictions into every web-app build, or (c) migrate late under pressure. R-0008 §KNW-PIP-027 already says articles are jurisdiction-extensible — we just aren't using it for help.

3. **Release-cycle coupling.** An accountant updating help wording, or the ATO announcing a new instant-write-off threshold, should not require a TypeScript PR and a web-app deploy. The policy tab content updates without a web-app deploy today; the help tab content shouldn't be different.

4. **Project-level CLAUDE.md contradicts R-0008.** `ledgius-web-app/CLAUDE.md` currently says *"Help YAML files live in `src/locales/en-AU/help/`"*. That guidance contradicts §KNW-PIP-004 / §KNW-PIP-018 / §KNW-PIP-025 and is actively misleading.

# Scope

- **Inventory** existing frontend help content: every `*.yaml` under `src/locales/en-AU/help/` and every entry in the `pageHelpContent` map in `src/hooks/usePageHelp.ts`, mapped to its host page route and the page's `usePagePolicies` domain tags.
- **Translate to articles**: one article per logical "page help context" (usually one-to-one with a page, sometimes multiple when a page has multiple contexts). Each YAML section becomes an `ArticleSection`. Authority type `internal_policy`, issuer `Ledgius`, jurisdiction `AU`, locale `en-AU`.
- **Publish to the authority directory**: write the article YAMLs to the location the `policy.Store` loads from (`$AUTHORITY_DIR/articles/`, default `../docs/authority/articles/`). Page overrides manifest at `$AUTHORITY_DIR/manifests/overrides.yaml` if the domain-tag join doesn't already resolve correctly.
- **Add ETag + cache headers** to the resolver endpoints per the caching requirement flagged during this task's design.
- **Retire the frontend YAML tier**: delete `src/locales/en-AU/help/**/*.yaml`, the YAML loader `src/lib/help/loader.ts`, and the `pageHelpContent` map. Delete the `usePageHelp` legacy signature that accepts inline help.
- **Unify the help panel hook**: one hook per page — `usePagePolicies(domains)` (name retained for continuity) returns both internal help sections and external authority articles. The help-panel component renders them side-by-side, with internal sections grouped under a "Guide" heading (for familiarity) and external authority articles under the "Policies" section per R-0008 §KNW-PIP-019–022.
- **Update `ledgius-web-app/CLAUDE.md`** to remove the YAML-lives-in-locales guidance and point authors to the authority directory.

# Out of Scope

- **Ingestion pipeline work** — ingesting new external authority content is ongoing in `ledgius-knowledge-ingestion` and governed by A-0020. This task is authored content only.
- **Multi-locale content** — this task ships with `locale=en-AU` only. The schema supports other locales; populating them is a separate effort.
- **Multi-jurisdiction content** — same. Schema supports it; actual NZ/UK content is out of scope here (deferred per R-0008 §Out of Scope).
- **Authoring UI** — a platform-admin CRUD interface for editing articles in-app. Authors continue to edit YAML in git-version-controlled files per §KNW-PIP-025. An authoring UI is a future task.
- **Per-user locale preferences** — persisted user locale selection is a future task. The API accepts `locale` as a query parameter; default resolution is `en-AU` until overridden.
- **Rich glossary integration** changes — the existing `{{term:...}}` tooltip system stays. Glossary stays in its current location (tenant-scoped or shared depending on current wiring) and continues to work with article body markup. Unification with the article pipeline is a separate task if ever.

# Preconditions

- `internal/knowledge/policy/` store + handler are operational (they are — running on master).
- `AUTHORITY_DIR` environment variable is set (or the default path is reachable from the API binary's working directory).
- The knowledge article YAML schema is stable at the current shape (id, title, jurisdiction, issuer, authority_type, source_url?, effective, domains, sections) per R-0008 §Data.

# Functional Requirements

## Content migration

- **KHM-001**: Every frontend YAML file under `src/locales/en-AU/help/**/*.yaml` shall be migrated to an authority-article YAML under `$AUTHORITY_DIR/articles/internal/`. Suggested file naming: `ledgius-{domain}-{page-slug}.yaml` (e.g. `ledgius-assets-register.yaml`, `ledgius-loans-payout.yaml`).
- **KHM-002**: Every entry in the `pageHelpContent` map in `src/hooks/usePageHelp.ts` shall be migrated to an authority-article YAML with the same sections. If a `pageHelpContent` entry duplicates an existing YAML file, the YAML version wins (it's the newer content).
- **KHM-003**: Each migrated article shall carry the following metadata:
  - `authority_type: internal_policy`
  - `issuer: Ledgius`
  - `jurisdiction: AU`
  - `domains: [...]` — exactly the tags the page uses in `usePagePolicies([...])` (e.g. for the Asset Register: `["account", "tax", "assets"]`)
  - `effective: { start: "2026-04-01" }` — start date matches the date the content first shipped; no end date for active articles.
  - `review_date` — set to 12 months after start for the first cut; accountants review annually.
- **KHM-004**: Article ID shall be a stable, human-readable slug per §KNW-PIP-001: `LEDGIUS-{DOMAIN}-{PAGE_SLUG}` uppercased, hyphens in slug (e.g. `LEDGIUS-ASSETS-REGISTER`, `LEDGIUS-LOANS-PAYOUT`).
- **KHM-005**: Section IDs within an article shall be stable, human-readable, derived from heading text (e.g. `what-is-this`, `four-correction-paths-aasb-108`).
- **KHM-006**: Section bodies shall preserve the existing `{{term:...}}`, `{{good:...}}`, `{{warn:...}}`, `{{bad:...}}` markup — the renderer already handles these.

## Page resolution + manifest

- **KHM-007**: Where the domain-tag join (page domains ∩ article domains) correctly resolves the right articles for a page, no manifest override is needed. This is expected to cover the majority of pages.
- **KHM-008**: Where a page needs an article the domain-tag join wouldn't pick up (e.g. an article is relevant to exactly one page, not a whole domain), an entry in `$AUTHORITY_DIR/manifests/overrides.yaml` shall pin the article to the route per §KNW-PIP-017.
- **KHM-009**: The manifest shall NOT be used as the primary resolution mechanism — domains are the primary join per §KNW-PIP-017. Overrides are exceptions.

## API caching (additive change)

- **KHM-010**: `GET /api/v1/knowledge/articles?page=&domains=` and `GET /api/v1/knowledge/articles/:id` shall emit an `ETag` response header. The ETag is a SHA-256 over the in-memory store's canonical serialisation, computed once at store-load time.
- **KHM-011**: Both endpoints shall honour `If-None-Match`: if the incoming header matches the current ETag, respond `304 Not Modified` with no body.
- **KHM-012**: Both endpoints shall emit `Cache-Control: public, max-age=300, must-revalidate`. Content changes rarely; aggressive browser cache with revalidation strikes the right balance.
- **KHM-013**: When the authority store reloads (today: at API boot; future: on SIGHUP or file-watch), the ETag recomputes. The reload mechanism is out of scope for this task — boot-time load is sufficient.
- **KHM-014**: `GET /api/v1/knowledge/audit` does NOT emit ETag — it's an operator tool, not a client-cacheable resource.

## Frontend unification

- **KHM-015**: `usePageHelp(pageHelpContent.xxx)` and its inline-help fallback are retired. The only remaining hook is `usePagePolicies(domains: string[])` which now returns the resolved page content (internal help + external authority, combined).
- **KHM-016**: Every page currently calling `usePageHelp(...)` shall be migrated to call only `usePagePolicies(domains)`. Pages not currently calling `usePagePolicies` gain that call with an appropriate domain list (mirrors the domain tags used on the migrated articles).
- **KHM-017**: The help panel component shall render the resolved articles with the two-tier visual treatment from R-0008 §KNW-PIP-022: internal-policy sections shown first under a "Guide" heading with a gray "Ledgius" badge; external-authority sections shown under a "Policies" heading with colour-coded badges (blue external_regulatory, purple external_standard, green industry_practice).
- **KHM-018**: The frontend shall send `If-None-Match` on repeat requests; TanStack Query's ETag support or a light wrapper in `src/shared/lib/api.ts` handles this.
- **KHM-019**: The `src/locales/en-AU/help/` directory and `src/lib/help/loader.ts` shall be deleted after migration. The `pageHelpContent` map in `src/hooks/usePageHelp.ts` shall be deleted. The `usePageHelp` function shall be deleted (its callers all migrate to `usePagePolicies`).
- **KHM-020**: The glossary (`src/locales/en-AU/glossary.yaml`) stays in place for now. `{{term:...}}` tooltip resolution in article bodies continues to work. Migration of the glossary into the authority pipeline is a follow-up.

## Documentation

- **KHM-021**: `ledgius-web-app/CLAUDE.md` shall be updated to remove the guidance that help YAML lives in `src/locales/en-AU/help/`. Replacement guidance shall point to `$AUTHORITY_DIR/articles/internal/` in the API repo (or the relocated docs repo, depending on where the team settles), cite R-0008 §KNW-PIP-004 / §KNW-PIP-018 / §KNW-PIP-025, and name the canonical hook (`usePagePolicies`).
- **KHM-022**: `ledgius-api/CLAUDE.md` shall be updated (if it mentions help/policy content) to confirm articles are loaded from `AUTHORITY_DIR` and served via the existing `internal/knowledge/policy/` package.

# Non-Functional Requirements

- **KHM-030**: Resolver response time shall remain under the 200ms p99 budget from §KNW-PIP-024 — the in-memory store already guarantees this; caching makes it tighter.
- **KHM-031**: Content volume after migration (article count + section count) shall not exceed 2× the current bundle size to avoid regressing store load time or memory footprint. Estimated count: ~20–30 articles with ~5 sections each based on current frontend inventory.
- **KHM-032**: Build-time validation (§KNW-PIP-026) shall continue to catch broken article/rule links across both internal-policy and external-authority articles.

# Data / Entities / Fields

No new entities. This task uses the existing R-0008 schema:

- `KnowledgeArticle` (YAML file) with `authority_type='internal_policy'` and `issuer='Ledgius'`.
- `ArticleSection` embedded within the article file.
- `Override` in the manifests file for any per-route overrides.

# UX / UI Behaviour

- Help panel content visually unchanged to existing users for AU / en-AU. Same markup, same tooltips, same section expand/collapse behaviour.
- New: the "Ledgius" authority badge (gray) appears on internal-policy sections per R-0008 §KNW-PIP-022 — users see at a glance that the content is Ledgius-authored guidance vs external regulatory.

# Validation Rules

- Article YAML shall pass R-0008's existing validation (ID uniqueness, section ID uniqueness within article, effective start required, domain tags from approved taxonomy).
- Every migrated article's domain tags shall be a subset of the approved taxonomy — no new domain tags introduced by this task without adding them to the taxonomy definition.
- The retired `src/locales/en-AU/help/` directory shall not be re-added by a future PR (CI check: presence of the directory fails the build).

# Error Handling

- Missing articles for a page route return empty resolved content per §KNW-PIP-025 — not an error. Users see an empty help panel, not an error banner.
- Malformed article YAML at boot aborts the server start with a clear error. Same behaviour as today.
- `If-None-Match` with an unknown ETag is treated as a cache miss and returns the full response.

# Security / Permissions

- Articles remain read-only in the UI per §KNW-PIP-029. No editing endpoints.
- Resolver endpoints remain public (no auth) per §KNW-PIP-029 — policy transparency is part of the contract.

# Implementation Plan

## Phase 1 — Inventory

1. Script a dump of every `*.yaml` under `ledgius-web-app/src/locales/en-AU/help/` with its path, page route (from the `page:` field), and section headings.
2. Dump every entry in `pageHelpContent` in `ledgius-web-app/src/hooks/usePageHelp.ts` — key, title, section headings.
3. Cross-reference against `usePagePolicies` calls per page to capture the domain-tag list.
4. Output `domains/knowledge/knowledge-pipeline/pipeline-requirements/migration-inventory.md` (or similar) with the full mapping. This is the seed list for phase 2.

## Phase 2 — Article authoring

For each page in the inventory, write an article YAML under `$AUTHORITY_DIR/articles/internal/`:

```yaml
id: LEDGIUS-ASSETS-REGISTER
title: Asset Register
jurisdiction: AU
issuer: Ledgius
authority_type: internal_policy
effective:
  start: 2026-04-01
review_date: 2027-04-01
domains: [account, tax, assets]
sections:
  - id: what-is-this
    heading: What is this?
    body: |
      Your fixed asset register — every capital item your business owns.
      ...
  - id: how-depreciation-works
    heading: How depreciation works
    body: |
      ...
```

Merge YAML + `pageHelpContent` content for the same page — if the YAML has more sections, use the YAML. If `pageHelpContent` has sections YAML doesn't, merge them in.

## Phase 3 — API caching

Small addition to `internal/knowledge/policy/`:

- Compute a store-level ETag at `NewStore` time: sort articles by ID, serialise to JSON, SHA-256.
- Expose `(s *Store) ETag() string`.
- Handler reads the ETag into every response; honours `If-None-Match`.
- `Cache-Control` header set.

No DB migration. No data model change.

## Phase 4 — Frontend retirement

1. Update `usePagePolicies` to return the resolved page shape (it already calls the right endpoint — just update the return type to cover internal + external articles uniformly).
2. Delete `usePageHelp` (hook + `pageHelpContent` map).
3. Update every page currently calling `usePageHelp(pageHelpContent.xxx)` to call only `usePagePolicies([...])` with the correct domain list.
4. Delete `src/locales/en-AU/help/` and `src/lib/help/loader.ts`.
5. Update the help-panel rendering component to group by `authority_type` and show the appropriate badge.
6. Add TanStack Query's `If-None-Match` support (or extend `src/shared/lib/api.ts`).
7. Run `tsc -b && vite build` — clean.

## Phase 5 — Documentation

- Update `ledgius-web-app/CLAUDE.md` per KHM-021.
- Update `ledgius-api/CLAUDE.md` per KHM-022 if needed.
- Add a section to `ledgius-specs/README.md` (or `AI-prompt.md`) explicitly saying: help content is a knowledge article; never add YAML help files to `ledgius-web-app/src/locales/`.

## Phase 6 — CI check

Simple check to prevent regression: CI fails if `ledgius-web-app/src/locales/en-AU/help/` exists or if `pageHelpContent` is reintroduced in `src/hooks/usePageHelp.ts`.

# Open Questions

1. **Location of internal article YAMLs** — `ledgius-api/docs/authority/articles/internal/` is one option. `ledgius-knowledge-ingestion/docs/` is another. A new top-level `accounting/docs/authority/` that both repos read from is a third. The current `AUTHORITY_DIR` default is `../docs/authority/` — i.e. a repo-relative path above the api binary's cwd. The migration should land in whichever location the team settles; current preference (pending review) is `ledgius-api/docs/authority/articles/internal/` for locality with the code that serves it.
2. **Manifest override policy** — confirm whether the domain-tag join resolves pages correctly for the current inventory, or whether several pages need manifest overrides. Decided during Phase 1 inventory.
3. **ETag granularity** — store-level vs per-page. This task ships store-level (simpler, one hash, one check). Per-page ETags are a future optimisation if store-level invalidates too often in practice.
4. **Locale fallback** — if a future tenant asks for `fr-CA` and no `fr-CA` article exists for a page, should the resolver fall back to `en-AU`? Recommended yes, per spec §KNW-PIP-027 intent, but this is a follow-up detail — not blocking for the AU-only migration this task ships.
