# Summary

Extend the knowledge pipeline so article content is **contextual** to
three axes:

1. **Domain / page** — already solved by the domain-tag join and route
   overrides per R-0008 §KNW-PIP-017 and shipped in T-0038.
2. **Time** — article content reflects the user's **active period**
   (the quarter / FY they're currently viewing), not the server clock.
   The active period is set globally via a top-nav selector and pipes
   into every knowledge request.
3. **Business setup** — sections conditionally appear based on the
   tenant's registrations (GST / BAS cadence, STP, payroll cadence,
   entity type). A tenant with no GST registration does not see BAS
   sections; a quarterly-BAS tenant does not see monthly-BAS content.

Adds two additive schema fields (`conditions` on articles + sections),
a template engine for body interpolation, a tenant-context lookup, and
per-request ETag covering the active-period + tenant dimensions.

No existing data invalidated. Articles without template variables or
conditions behave identically to today (pure-string body, always
included). New content opts in progressively.

# Terms

Vocabulary from R-0008 §Terms applies. New here:

| **Term** | Definition. |
|---|---|
| **Active period** | The FY quarter (or financial year if nothing finer-grained is selected) the user is currently viewing. Sourced from a global top-nav selector and passed into every knowledge-resolver request. Distinct from **current period** (server clock's current quarter) — users can rewind to look at prior quarters while reconciling or auditing. |
| **Render context** | The combined object the template engine sees when evaluating article bodies + section conditions: `{tenant, fy, bas, stp, payroll, now}` where most fields derive from `(active period, tenant config, server clock, jurisdiction FY calendar)`. |
| **Active-period selector** | Top-nav UI control. Format `< Q3 >`. `<` moves earlier. `>` moves later but is **disabled** when `active_period == current_period` — users cannot view the future. Stored in React context + persisted to `localStorage` key `ledgius-active-period`. |
| **Condition expression** | A CEL boolean expression evaluated against the render context. Attached to an `ArticleSection` or whole `Article` to gate inclusion. e.g. `tenant.bas_cadence == "quarterly"`. |
| **Per-request ETag** | Cache validator covering the store state + the caller's render context (tenant id + active period + domains + page). Replaces T-0038's store-wide ETag for the `/articles` resolver endpoint. Single-article GET by id keeps the store-wide ETag since those articles are rendered without tenant context. |
| **Info Panel** | The page-top widget that shows a concise "what-to-do" instruction set for the current page. Distinct from the full help panel (F1): the Info Panel is "what + bare-minimum how" (no "why"); the help panel covers "what + why + how". Sourced from the help article's `info_panel` field so both surfaces stay aligned. Tenant-admin togglable via `tenant_settings.info_panels_enabled`. Operational status (counts, checklist done/pending) is **not** authored here — that is computed live by T-0040. |
| **Authoring rules** | The contract that governs what copy belongs in `info_panel` vs in full-help `sections`. Codified in §Authoring Rules so reviewers can reject articles that bleed "why" into the info panel or bury the "how" inside prose-only sections. |

# Problem

T-0038 shipped articles that are identical for every tenant regardless
of time or business setup. Article text like "Quarterly is the most
common small-business cadence" is generic by necessity — there's no
substitution, no conditional sections.

For policy-sensitive pages this is actively misleading:

- A BAS page showing "Next lodgement due 28 Oct" to a tenant whose
  current BAS period closed three months ago is wrong.
- A BAS page shown to a tenant not registered for GST at all is
  noise — they don't lodge BAS.
- A payroll / STP page showing the generic weekly/fortnightly/monthly
  options helps nobody — they want to see *their* cadence and *their*
  next due date.
- An audit page during period-close review needs to know which period
  the user is auditing — not "today's" period.

The knowledge pipeline is the right home for this (it already handles
domain and jurisdiction dimensions). We extend it with time and
business-setup awareness.

# Scope

- Template interpolation in article/section bodies via a small,
  scoped set of variables (see §Render Context below).
- Section-level and article-level `conditions` field — a list of CEL
  boolean expressions evaluated against the render context. All must
  evaluate `true` for the section/article to be included.
- A tenant-config lookup feeding the render context with GST/BAS/STP/
  payroll registrations. Reads existing tenant settings — no new
  persistent data unless existing tables are missing the flags.
- An active-period selector in the app's top nav. Global React
  context + `localStorage` persistence. Passes period into every
  knowledge request.
- Jurisdiction-aware FY calendar helper so AU FY (Jul–Jun) and future
  NZ/UK FYs can be supported from the same code.
- Per-request ETag on the `/articles` resolver endpoint covering
  active period + tenant context dimensions.
- `info_panel` field on the article schema so one authored source
  feeds both the page-top Info Panel widget and the full help panel.
- `tenant_settings.info_panels_enabled` toggle letting experienced
  tenants hide info panels site-wide.
- Authoring rules that divide "what / how" (info panel) from
  "what / why / how" (full help sections).
- Rewrite of the 8 articles shipped in T-0038 so they exercise
  templating + conditions + `info_panel` — no net-new content going
  out flat while this task is in flight.

# Out of Scope

- Quarterly P&L / Balance Sheet reports per the active period. Reports
  already accept date filters; the active-period selector is purely a
  read-side filter on the help/policy layer and on the specific pages
  that currently carry their own period filter.
- Multi-locale content. T-0039 is still `locale=en-AU` only; locale
  comes later when the schema includes it properly.
- Period-close workflows driven by the selector (e.g. "lock Q1"
  actions). Those belong to a separate period-close task.
- Authoring UI for managing conditions / templates. Authors continue
  to edit YAML per R-0008 §KNW-PIP-025.
- Template loop constructs / partials / includes. The first cut of
  the template engine is variable substitution + simple conditionals
  only. Anything more complex gets factored into CEL conditions on
  separate sections.
- **Operational status / live counters / checklist done-pending state**
  — "47 of 51 lines reconciled", "2 approvals pending", "BAS Q1
  marked reviewed". That data is computed from the tenant DB, has
  different cache semantics (refresh on mutation), and belongs in
  T-0040's status endpoint. T-0039's `info_panel.quick_links[*].
  condition` can reference counters exposed via the render context,
  but those counters are populated by T-0040 when it lands; until
  then any such condition simply evaluates false.

# Preconditions

- T-0038 is merged (it is — PR #31 / PR #61 / PR #57 all merged).
- `internal/knowledge/policy/` store + handler are the implementation
  site (no new package needed).
- Tenant config surfaces: GST registration flag, BAS cadence, STP

# Functional Requirements

## Schema extensions

- **KCR-001**: `ArticleSection` shall gain an optional `conditions`
  field — a list of CEL boolean expression strings. All must evaluate
  true for the section to be included. Absent `conditions` means
  always include (backward-compatible).
- **KCR-002**: `Article` shall gain an optional `conditions` field
  with identical semantics — if any article-level condition evaluates
  false, the whole article is excluded from the resolved page.
- **KCR-003**: Section bodies shall support `{{variable.path}}`
  template substitution against the render context. Undefined
  variables render as the empty string and log a warning (not an
  error — content cannot break the page).
- **KCR-004**: `Article` shall gain an optional `info_panel` field
  with the shape below. When present it feeds both the page-top
  `<InfoPanel>` widget and an optional summary block at the top of
  the full help panel, keeping the two surfaces in lock-step:

  ```yaml
  info_panel:
    # Required when info_panel is present. Ordered concise steps —
    # the "what + bare-minimum how". No background / rationale.
    steps:
      - "Reconcile all bank accounts for the active period"
      - "Review GST classifications on unusual transactions"
      - "Click **Lodge** to submit via SBR"
    # Optional. Quick-link chips that route to a related page.
    quick_links:
      - label: "Review unreconciled lines"
        route: "/bank-reconciliation"
        # Optional CEL. If present and evaluates false, link hidden.
        condition: "reconciliation.unmatched_count > 0"
  ```

  - `steps` entries support the same `{{variable.path}}` templating
    and `{{term:...}}` / `{{good:...}}` / `{{warn:...}}` / `{{bad:...}}`
    markup as `sections[].body`.
  - `quick_links[].condition` is a single CEL boolean string. Same
    render context, same failure semantics as KCR-050/053 (failed
    compile aborts store load; runtime error treats as false).
  - `info_panel` respects article-level `conditions` — if the article
    as a whole is filtered out, the info panel goes with it.
  - Absence of `info_panel` is valid: the page renders without the
    widget. List / index pages (e.g. `/contacts`) typically omit it.

## Render context

- **KCR-010**: The render context is a struct built per-request from:
  - Server clock (`now` in the tenant timezone).
  - Tenant config (GST, BAS, STP, payroll registrations).
  - Jurisdiction FY calendar (AU defaults to 1 Jul – 30 Jun).
  - Active period (from request query parameter).
- **KCR-011**: Variables available in templates + conditions:

  ```
  now                        — server time in tenant TZ
  tenant.id                  — UUID
  tenant.timezone
  tenant.jurisdiction        — "AU"
  tenant.entity_type         — "sole_trader" | "company" | "partnership" | "trust" | "other"
  tenant.gst_registered      — bool
  tenant.bas_cadence         — "quarterly" | "monthly" | "yearly" | "none"
  tenant.stp_enabled         — bool
  tenant.payroll_cadence     — "weekly" | "fortnightly" | "monthly" | "none"

  fy.current.label           — "FY 2026-27"
  fy.current.start           — 2026-07-01
  fy.current.end             — 2027-06-30

  fy.active.label            — label of the period the user is viewing
  fy.active.start            — start date
  fy.active.end              — end date

  bas.cadence                — same as tenant.bas_cadence (convenience alias)
  bas.period_label           — e.g. "Q1 FY2026-27" for the active period
  bas.period_start           — start date of active period's BAS period
  bas.period_end             — end date
  bas.next_due               — next BAS lodgement due date (derived from
                                the active period's end + ATO due-date
                                rules: quarterly → 28th of month following)
  bas.is_current             — true iff active_period is the current
                                quarter at server time

  stp.cadence                — alias for tenant.payroll_cadence when
                                stp_enabled; else nil
  stp.next_due               — next STP finalisation deadline

  payroll.cadence            — alias for tenant.payroll_cadence
  payroll.next_pay_date      — next scheduled pay date given the active
                                period
  ```

- **KCR-012**: Date fields render as ISO `YYYY-MM-DD` by default. A
  template helper `{{fmt.date variable "d mmm yyyy"}}` formats
  differently (e.g. "28 Oct 2026") where prose flow needs it.
- **KCR-013**: When `tenant.gst_registered == false`, `bas.*` fields
  render as descriptive null labels (e.g. "not applicable — not
  registered for GST") rather than missing keys. This lets an article
  optionally describe the null state inline rather than needing a
  separate conditional section.

## Active-period propagation

- **KCR-020**: `GET /api/v1/knowledge/articles` accepts an optional
  `period` query parameter. Format: `{YYYY}-Q{N}` (e.g. `2026-Q1` for
  Jul-Sep 2026) or `{YYYY}` (the FY starting 1 Jul of that year) or
  `{YYYY-MM-DD}:{YYYY-MM-DD}` (explicit range, advanced).
- **KCR-021**: Missing `period` → default to current quarter at
  request time. Malformed `period` → 400 with a clear error message.
- **KCR-022**: Invalid period (future beyond current quarter) → 400
  with a clear error message. Users cannot see the future.
- **KCR-023**: Frontend sends `period` on every knowledge request
  driven by the active-period selector. The selector is in the
  top-nav and persists via `localStorage[ledgius-active-period]`.
  Default: current quarter.

## Top-nav active-period selector

- **KCR-030**: Top-nav shows a centred control: `<  Q3 FY2026-27  >`.
  `<` button moves active period one quarter earlier. `>` button
  moves one quarter later but is disabled when
  `active_period == current_period`.
- **KCR-031**: Clicking the label opens a small popover with
  period presets: "Current", "Previous Q", "Previous FY", plus a
  date-range picker for custom ranges. Presets resolve to canonical
  period values (`2026-Q1`, etc.).
- **KCR-032**: The selector's active period is broadcast via a
  React context so any page that cares (help panel, BAS page, period-
  close pages) consumes it without prop-drilling. A hook
  `useActivePeriod()` returns `{period, setPeriod, isCurrent}`.
- **KCR-033**: Changing the active period invalidates cached
  knowledge queries (`queryClient.invalidateQueries(["knowledge"])`).

## Per-request ETag

- **KCR-040**: The `/articles` resolver endpoint's ETag covers:
  `SHA-256(store_etag + tenant_id + active_period + sorted_domains +
  page)`. Any dimension changing → new ETag → client revalidates.
- **KCR-041**: `If-None-Match` behaviour unchanged — match returns
  304, mismatch returns 200 with fresh content + new ETag.
- **KCR-042**: `GET /articles/:id` keeps the store-wide ETag from
  T-0038. Single-article fetch doesn't apply the render context
  because the UI path for this fetch is always admin/audit use and
  returns raw article data for debug/inspection.
- **KCR-043**: `Cache-Control: private, max-age=300, must-revalidate`
  (changed from `public` because tenant-specific content cannot be
  cached by shared proxies).

## Rendering rules

- **KCR-050**: Articles with an article-level condition that evaluates
  false are excluded from the resolved page entirely.
- **KCR-051**: Sections with a section-level condition that evaluates
  false are excluded from the resolved article. If all sections in an
  article are excluded, the article itself is still returned (with
  empty `sections`) so the UI can choose whether to render a header
  even when no sections apply.
- **KCR-052**: Bodies are rendered after condition filtering. No
  template substitution happens on an excluded section — keeps
  rendering cheap.
- **KCR-053**: Template errors (missing variables that don't have
  null-labels, malformed helpers) do NOT fail the request — they
  render as empty strings and log a warning. Content robustness beats
  strictness; a broken help panel on a critical workflow is worse
  than a misleading blank line.

## Tenant config

- **KCR-060**: The render context reads tenant config from existing
  tenant-settings tables where possible. If a field is missing a
  backing column, this task adds a minimal schema migration on
  `tenant_settings` covering: `gst_registered`, `bas_cadence`,
  `stp_enabled`, `payroll_cadence`, `entity_type`,
  `info_panels_enabled`. Migration is additive-only with sensible
  defaults — existing tenants get `gst_registered=true,
  bas_cadence='quarterly', stp_enabled=true,
  payroll_cadence='monthly', entity_type='company',
  info_panels_enabled=true` as defaults pending tenant review (audit
  log records this as a default-fill, not an assertion).
- **KCR-061**: A subsequent task covers a settings UI letting the
  tenant admin correct the defaults, including a toggle for
  `info_panels_enabled`. The toggle is exposed per-tenant (not
  per-user) — the admin decides whether the tenancy favours the
  guided-workflow or minimalist surface.

## Info Panel rendering

- **KCR-090**: `GET /api/v1/knowledge/articles` response includes the
  article's `info_panel` (rendered through the same render context
  as `sections[].body`) whenever present. No separate endpoint.
- **KCR-091**: When `tenant.info_panels_enabled == false`, the
  resolver MAY still include `info_panel` in the response payload
  (so the frontend can toggle on/off without a refetch), but the
  frontend MUST NOT render the `<InfoPanel>` widget on any page.
  Rationale: cache granularity — the same ETag can serve a tenant
  whose toggle flips without forcing a round-trip. The toggle is a
  UI-only decision.
- **KCR-092**: `info_panel.quick_links[].condition` evaluates against
  the same render context used for section conditions. A link whose
  condition evaluates false is omitted from the payload entirely
  (not returned as "hidden") so naive clients render only visible
  links by default.
- **KCR-093**: The full help panel MAY render the same `info_panel.
  steps` as a "Quick steps" summary block at the top of the article
  body. This is a frontend rendering choice — the payload is the
  same.

# Non-Functional Requirements

- **KCR-070**: Resolver response time shall remain under the 200ms
  p99 budget from §KNW-PIP-024. Template evaluation + CEL conditions
  add microseconds per section — well within budget.
- **KCR-071**: Condition evaluation uses CEL's existing in-process
  engine — no external service, no per-request network.
- **KCR-072**: Per-request ETag computation is a single SHA-256 over
  a short string; sub-millisecond.
- **KCR-073**: Tenant config is cached per-tenant in memory for the
  duration of a request (no repeated lookups) and, beyond that, uses
  the existing TTL cache used by the feature/role permissions
  middleware from A-0039 so hot-path tenants don't hit the DB every
  request.

# Authoring Rules

These rules govern what goes into `info_panel` vs `sections`. They are
**binding on reviewers** — a PR adding or modifying a help article
should be rejected if it violates them. The rules preserve the split
the user approved: info panel is "what to do", full help is "what /
why / how".

## `info_panel.steps`

- **Content**: imperative, operational steps the user takes on this
  page to accomplish the page's primary outcome.
- **Voice**: second person, imperative. "Reconcile all bank accounts
  for the active period."
- **Length**: each step ≤ 100 characters in its rendered form
  (post-template). The widget is compact.
- **Count**: typically 3–6 steps. Above 8 is a signal the page has
  too many top-level outcomes.
- **Forbidden**:
  - Rationale or "why" ("Because AASB requires..."). Move to a
    section.
  - Conceptual definitions ("A BAS is a quarterly activity
    statement..."). Move to an `overview` section.
  - Error recovery prose ("If reconciliation fails, check..."). Move
    to a `troubleshooting` section.
  - Marketing copy / value propositions. They don't belong anywhere.

## `info_panel.quick_links`

- **Content**: navigation to a closely related page that accomplishes
  a step or resolves a common blocker.
- **Label**: verb-first, ≤ 30 characters. "Review unreconciled lines",
  "Fix missing GST codes".
- **Condition**: use CEL whenever the link only makes sense given
  operational state ("unmatched lines exist", "pending approvals
  exist"). Fallback behaviour — when T-0040's counters aren't yet
  wired, the condition simply evaluates false and the link is
  omitted.

## `sections[*]`

- **Content**: the full help content — concept definitions,
  rationale, procedure narrative, policy references, worked
  examples, troubleshooting.
- **Voice**: third person, descriptive. Written for a user who wants
  to understand the domain, not just execute a step.
- **Structure**: each section addresses one aspect (`overview`,
  `policies`, `procedure`, `troubleshooting`, `rejection`, etc.).
- **Forbidden**:
  - Step lists that duplicate `info_panel.steps`. Reference the info
    panel instead ("See the steps above."). Single source of truth.

## Style markup

- `{{term:...}}` — glossary term tooltip.
- `{{good:...}}`, `{{warn:...}}`, `{{bad:...}}` — semantic emphasis.
- Available in both `info_panel.steps` and `sections[*].body`.
- Template variables (`{{fy.active.label}}` etc.) evaluated first,
  then markup rendering.

## Review checklist (for PR authors + reviewers)

- [ ] If the page has a primary outcome, the article has an
      `info_panel` with 3–6 steps.
- [ ] No step contains rationale (no "because", "to comply with",
      "this matters because").
- [ ] No section restates the same step list verbatim.
- [ ] Quick-link conditions reference only counters exposed by the
      render context (not hallucinated fields).
- [ ] Template variables used in steps match the section-body
      templating style.

# Validation Rules

- CEL conditions that fail to compile at article load abort the store
  load with a clear error — authors must fix the YAML before
  deployment. Runtime failures (null pointer in variable access) log
  a warning and treat the condition as false.
- Periods before the tenant's `tenant.created_at` or beyond
  `current_period` are rejected at the resolver with 400.
- Templates rendering infinite recursion (user-authored mistakes) are
  prevented by the template engine's pipeline depth limit.
- `info_panel.steps` MUST be non-empty when `info_panel` is present.
- `info_panel.quick_links[].route` MUST start with `/` (internal
  routes only — no external links in the Info Panel widget).

# Error Handling

- Invalid `period` parameter → 400 with descriptive error.
- Future period → 400 "cannot view future periods".
- CEL condition runtime error → log warning, treat as false, section
  excluded.
- Template substitution miss → log warning, empty string, continue.

# Security / Permissions

- Resolver endpoints remain accessible to authenticated users only —
  tenant context requires a JWT claim.
- No new permissions. Audit logs do NOT record individual
  active-period changes (too noisy); the selector state is a UI
  preference, not a financial event.

   before serialising the response.

1. Parse `?period=...` on `GET /api/v1/knowledge/articles`.
2. Build render context per request.
3. Filter articles by article-level conditions.
4. Filter sections by section-level conditions.
5. Render bodies.

7. Emit headers per KCR-043.

# Open Questions

1. **FY calendar pluggability** — AU FY is simple (Jul–Jun). Do we
   design the helper now to accept jurisdictions from the first
   commit, or ship AU-only and refactor later? Preference: ship
   AU-only with a jurisdiction parameter already wired through, so
   NZ/UK additions are a content change only.
2. **Tenant-settings audit** — the migration in KCR-060 defaults
   existing tenants to sensible values. Should the default-fill be
   advertised in-app as a to-do for tenant admins? Yes, recommended
   — a banner on the business-settings page noting "Please verify
   your business registrations".
3. **Custom FY end dates** — some companies run substituted
   accounting periods (e.g. 31 March end for stat reporting).
   `tenant_settings.fy_end_date` is a future extension; this task
   assumes standard FY per jurisdiction.
4. **Popover dependency** — does the codebase already have a
   Popover primitive, or does this task introduce one? Check before
   implementation.
