# Summary

Add a per-page **operational status** layer on top of the knowledge
pipeline. The status layer answers: "Where am I up to on this page,
right now?" â€” a structured checklist of done / pending / blocked
items plus a small set of live counters, scoped to the active period
and tenant.

Status comes from the **tenant database** (not the authored YAML),
runs through page-bound **checkers**, and surfaces both:

1. **Directly** via a new endpoint `GET /api/v1/pages/:route/status`
   for the frontend to render in the InfoPanel and dedicated status
   regions.
2. **Indirectly** by extending T-0039's render context with a
   `status` namespace, so info-panel `quick_links[].condition`
   expressions can show / hide links based on live state (e.g. a
   "Resolve unmatched lines" link that only appears when
   `status.bank.unmatched_count > 0`).

This is the foundational step toward Tier 4 ("Operational status") of
R-0008's Help Capability Tiers â€” a prerequisite for the Tier 5
Guided-Workflow vision.

# Terms

Vocabulary from R-0008 Â§Terms and T-0039 Â§Terms applies. New here:

| **Term** | Definition. |
|---|---|
| **Page status** | The structured operational state of a page for the current tenant + active period: a checklist + counters + optional next-action hints. Not authored content; computed from tenant data at request time. |
| **Checklist item** | A single named operational outcome (e.g. "Bank reconciliation complete for this period"). State: `done` / `pending` / `blocked` / `not_applicable`. Carries an optional drill-link to the page that resolves it. |
| **Status counter** | A named numeric metric (e.g. `bank.unmatched_count`, `approvals.pending_count`). Surfaced in the response and in T-0039 render context. |
| **Status checker** | A registered Go function bound to one or more page routes. Given a tenant DB connection + active period, returns the page's checklist + counter set. Pure function over the DB snapshot it sees. |
| **Status registry** | The in-process map of page route â†’ checker. Built at API boot from package-init registrations so each domain owns its own checker without touching a central switch statement. |
| **Status freshness** | Wall-clock age of a status response. Since status reflects live tenant data, freshness matters more than for T-0039 reference content. The endpoint advertises a short cache TTL with revalidation, and mutations on relevant tables bust the cache via a per-tenant version counter. |
| **Status snapshot** | The exact response payload returned for one (tenant, route, period) tuple at one moment. The ETag pins a snapshot so clients can revalidate cheaply. |

# Problem

T-0038 + T-0039 give every page an article and an InfoPanel with
"what to do" steps. What they don't give the user is "where am I up
to right now". A bookkeeper opening `/bas` sees the BAS workflow
described in prose; she still has to click into the bank
reconciliation page, the approvals queue, and the BAS draft to figure
out what's done and what's left.

Competing accounting products treat help as a passive reference
shelf. Ledgius' R-0008 Help Capability Tiers commit to going further
â€” Tier 4 says the system tells the user what's done and what's
pending. Tier 5 builds on Tier 4 to actively guide; Tier 6 anticipates
the next action. Tier 4 is the foundation. Without it, Tier 5/6 are
hand-waving.

The functional gap T-0040 closes:

- A page-top widget that shows "BAS Q3 FY 2025-26: 4 of 7 steps done"
  with the remaining steps as quick-link chips â€” pulling from the
  same data sources the user would otherwise visit one-by-one.
- Quick-link chips on the InfoPanel that appear only when the
  underlying state warrants them (T-0039 Â§KCR-092 already supports
  CEL-conditioned visibility, but the conditions can only reference
  authored data â€” `tenant.bas_cadence` etc. â€” not live counters).
- A single endpoint backend / agents / future automations can
  consume to know the operational state of any page without
  scraping individual domain endpoints.

# Scope

- New endpoint `GET /api/v1/pages/:route/status?period=&domains=` per
  KCS-020.
- A status registry pattern in `internal/knowledge/status/` (new
  package) so domain teams register a checker for their pages
  without touching a central wiring file.
- Initial checker set covering 6 page routes that the user feedback
  shows are status-relevant: `/bas`, `/bank-reconciliation`,
  `/approvals`, `/assets/depreciation`, `/loans`, `/gl`.
- Extension of T-0039's render context with a top-level `status`
  namespace â€” counters from the page's checker land there so
  `info_panel.quick_links[].condition` CEL expressions can read
  them.
- Cache validator (per-request ETag) covering tenant + route +
  period + a per-tenant mutation version counter.
- A small `<PageStatus>` frontend component rendered above the
  InfoPanel, showing the checklist + counters.
- A per-tenant "status revision" counter incremented by a small set
  of cross-cutting mutation hooks (journal post, bank reconcile,
  approval decision, BAS lodgement, asset disposal) so cached
  responses bust on a real change.

# Out of Scope

- Cross-page roll-up ("you have 5 things to do across the whole
  app") â€” that's a dashboard / decision-queue concern, not a page-
  status one. Reuse status data via a separate aggregator if the
  dashboard needs it.
- Automatic completion of checklist items via system actions â€”
  T-0040 only reports state, never mutates.
- Tier 5 guided sequencing ("now do this, then this") â€” the
  checklist is unordered. Sequencing is a Tier 5 feature.
- Persistent user-overridden state ("I've manually marked this
  done") â€” checklist items derive from data only, not from user
  toggles. A future task can layer manual overrides if needed.
- Push notifications when state changes â€” pull-only on this layer.
  Push (websockets / SSE) is a separate transport concern.
- Anticipatory triggering ("you've spent 30 seconds on this page,
  here's a hint") â€” Tier 6.

# Functional Requirements

## Endpoint

- **KCS-001**: System shall expose `GET /api/v1/pages/:route/status`
  returning the page's status snapshot. `:route` is the URL-encoded
  page path (e.g. `%2Fbas`). Same auth as the rest of `/api/v1` â€”
  TenantAuth middleware required.
- **KCS-002**: Endpoint accepts the same `period` query parameter
  semantics as T-0039 KCR-020. Missing â†’ current quarter; future â†’
  400 reject.
- **KCS-003**: Endpoint accepts an optional `domains` parameter for
  forward-compat with the resolver pattern, but the primary keying
  is the route.
- **KCS-004**: Response shape:

  ```json
  {
    "route": "/bas",
    "period": "2025-Q3",
    "title": "Q3 FY 2025-26 BAS",
    "summary": "4 of 7 steps complete",
    "checklist": [
      {
        "id": "reconcile-bank",
        "label": "Reconcile all bank accounts",
        "state": "done",
        "evidence": "All 3 accounts reconciled to 31 Mar 2026",
        "drill_link": "/bank-reconciliation"
      },
      {
        "id": "review-tax-codes",
        "label": "Review unusual tax-code uses",
        "state": "pending",
        "evidence": "12 transactions with non-standard codes",
        "drill_link": "/tax-codes"
      }
    ],
    "counters": {
      "bank.unmatched_count": 0,
      "approvals.pending_count": 2,
      "bas.adjustments_count": 1
    },
    "next_action": {
      "label": "Review pending approvals",
      "drill_link": "/approvals"
    }
  }
  ```

  All fields except `route`, `period`, `checklist`, and `counters`
  are optional. Empty checklists are valid (a page with no checker
  registered or no items applicable to the active period).

## Status registry

- **KCS-010**: Domain packages register checkers via package-init
  calls to a status registry, mirroring how mapping operators self-
  register today. Example:

  ```go
  // internal/banking/status.go
  func init() {
      status.Register("/bank-reconciliation", reconciliationCheck)
  }
  ```

  No central wiring file. The status package's `init` walks
  registrations into a `map[string]CheckerFunc` consumed by the
  handler.
- **KCS-011**: A checker has the signature
  `func(ctx context.Context, tdb *gorm.DB, period fycal.Period) (*PageStatus, error)`.
  `tdb` is the tenant DB (already scoped by middleware). `period` is
  the resolved active period. The checker is responsible for using
  `period` to filter its queries.
- **KCS-012**: Multiple routes can share a checker (e.g. both `/gl`
  and `/gl/:id` may register the same approvals-queue counter). The
  registry supports multi-route registration.
- **KCS-013**: A page with no registered checker returns `{checklist:
  [], counters: {}, summary: "No status configured"}` with HTTP 200
  â€” never 404. Frontend can choose to render nothing.

## Initial checker set

- **KCS-020**: This task ships checkers for six page routes:

  | Route | Checker outputs |
  |---|---|
  | `/bas` | reconcile-bank, review-tax-codes, review-adjustments, lodge; counters: `bank.unmatched_count`, `bas.adjustments_count`, `approvals.pending_count` |
  | `/bank-reconciliation` | per-account reconcile-status entries; counters: `bank.unmatched_count`, `bank.unreconciled_lines_count` |
  | `/approvals` | one item per pending approval (capped at top 10); counter: `approvals.pending_count` |
  | `/assets/depreciation` | last-run-period, run-due, count-of-active-assets; counters: `depreciation.assets_active_count`, `depreciation.last_run_period` |
  | `/loans` | per-loan upcoming-payment-due items (capped); counters: `loans.upcoming_payments_count`, `loans.active_count` |
  | `/gl` | counter `gl.draft_journals_count`, `gl.posted_this_period_count`; checklist: at most one item ("Year-end close pending") near FY end |

  Other pages can land checkers in follow-up tasks. The frontend
  should not assume a checker exists for any specific route.
- **KCS-021**: Each shipped checker must be covered by a unit test
  using a fixture tenant DB. Tests assert: known input state â†’
  expected checklist + counters.

## T-0039 integration

- **KCS-030**: Status counters are exposed in the T-0039 render
  context under a top-level `status` namespace. Example after
  `/bank-reconciliation` checker runs:

  ```
  status.bank.unmatched_count   = 4
  status.approvals.pending_count = 2
  ```

  CEL conditions in `info_panel.quick_links[].condition` can read
  these. Example:

  ```yaml
  quick_links:
    - label: "Resolve 4 unmatched lines"
      route: "/bank-reconciliation"
      condition: "status.bank.unmatched_count > 0"
  ```

- **KCS-031**: The resolver and the status endpoint share the
  per-request status fetch. To avoid running a checker twice on a
  request that hits both endpoints, the handler caches the most
  recent (tenant, route, period) status snapshot in the request
  context for downstream consumers.
- **KCS-032**: When a CEL condition references a counter that the
  current page's checker does not produce, the counter resolves to
  `0`. The condition then evaluates against zero rather than
  failing. Per KCR-053 the request never breaks.

## Caching

- **KCS-040**: Endpoint emits a per-request ETag covering
  `SHA-256(tenant_id + route + period + tenant_status_revision)`.
  `tenant_status_revision` is a monotonically-increasing counter
  per tenant, persisted on the tenant row, incremented by the
  mutation hooks in KCS-050.
- **KCS-041**: `Cache-Control: private, max-age=30, must-revalidate`.
  Shorter than T-0039's 300s because status is intended to feel
  near-live; revalidation against the ETag is the cheap check
  (304 with no body) so the budget covers the worst case where the
  client refetches every 30 seconds.
- **KCS-042**: `If-None-Match` round-trip same as T-0039 â€” match
  â†’ 304, miss â†’ fresh response.

## Mutation hooks

- **KCS-050**: A small set of cross-cutting writes increment the
  tenant's `status_revision` counter:
  - Journal post / approval / rejection
  - Bank statement import / reconciliation decision / unmatch
  - Asset acquisition / disposal / depreciation run
  - Loan payment / payout
  - BAS lodgement
  - Approval queue add / decide

  Other writes (settings, contact CRUD, product CRUD) do not bust
  status â€” their effect on operational state is indirect at best.

- **KCS-051**: The hook is a tiny helper invoked from the existing
  service layer at the same point that an audit row is written.
  Wrapping it as middleware was rejected because the hook needs to
  fire only after a successful commit â€” easier to call inline at
  the bottom of each service method that already passes audit-row
  semantics.

- **KCS-052**: `tenant_status_revision` is `BIGINT NOT NULL DEFAULT
  0` on the `tenants` table (platform DB). Migration is additive
  per the Phase 5 plan below.

# Non-Functional Requirements

- **KCS-060**: Status checker for any single page shall return in
  under 100ms p99 against a real-tenant DB with up to 100,000
  journal rows. Reuses indexes already present.
- **KCS-061**: Status endpoint p99 shall be under 200ms (matches
  T-0039 KCR-070 budget). Caching + index reuse keep this slack.
- **KCS-062**: A single tenant DB query budget per checker is 5
  queries. Above 5 the checker is asked to either denormalise into
  a materialised view or split into a cheaper subset.
- **KCS-063**: Status endpoint never holds a write lock and never
  participates in a transaction.

# Data / Entities / Fields

No new entities. Schema extensions only:

- `tenants.status_revision: BIGINT NOT NULL DEFAULT 0` (platform DB,
  additive migration).
- New Go types in `internal/knowledge/status/`:
  - `PageStatus { Title, Summary string; Checklist []ChecklistItem; Counters map[string]int64; NextAction *Action }`
  - `ChecklistItem { ID, Label, State, Evidence string; DrillLink string }`
  - `Action { Label, DrillLink string }`
  - `CheckerFunc func(ctx, *gorm.DB, fycal.Period) (*PageStatus, error)`

`State` is one of: `done`, `pending`, `blocked`, `not_applicable`.

# UX / UI Behaviour

- **KCS-070**: Frontend renders a `<PageStatus>` widget at the top
  of the page (above the existing `<ArticleInfoPanel>` from T-0039).
  Shows the page title, summary, and the checklist as a compact
  stack of rows with state icons.
- **KCS-071**: Each checklist item with `drill_link` is clickable
  and navigates the user to the linked page (with the active
  period preserved via the existing `useActivePeriod` context).
- **KCS-072**: When `next_action` is present it renders as a
  primary-styled chip, visually distinct from the checklist.
- **KCS-073**: The widget collapses gracefully â€” empty checklist
  hides the widget entirely. No "no status available" placeholder.
- **KCS-074**: Tenant `info_panels_enabled = false` (T-0039
  KCR-091) does NOT suppress `<PageStatus>`. The status widget is
  considered functional UI, not pedagogical UI; the toggle
  controls only the InfoPanel's instructional content.
- **KCS-075**: When the active period changes via the top-nav
  selector, the status widget re-fetches alongside the article
  payload (both share `["knowledge", ..., period]` query keys after
  T-0039 wiring).

# Validation Rules

- Period parameter validation matches T-0039 KCR-020..KCR-022 byte
  for byte (parse + future-period reject).
- A checker that returns a `ChecklistItem` with an unknown `State`
  fails the response with a 500 + log; the contract is enumerated
  for a reason.
- Counter keys must match `[a-z_]+(\.[a-z_]+)*` so the CEL
  evaluator can dereference them as nested maps. Validated at
  registry-add time (`status.Register` panics on a bad key during
  init).

# Error Handling

- A checker returning an error is logged but the request still
  responds 200 with `{"checklist": [], "counters": {}, "summary":
  "Status temporarily unavailable"}`. KCS-053 spirit: page UI
  cannot break because the status backend hiccupped.
- Tenant DB unreachable â†’ 503 with retry-after, not 200. Status
  isn't optional when the tenant DB itself is down â€” degrading
  silently misleads the user.

# Security / Permissions

- Endpoint requires the same role as the page it describes. A user
  who can't read `/bas` should not see status for `/bas`. Reuse the
  `RoleFunctionGuard` middleware: register `/api/v1/pages/:route/
  status` against the same feature/function as the page itself.
- Counter values that disclose financial figures (e.g. dollar
  amounts) are intentionally NOT in scope here â€” counters are
  occurrence counts only. Dollar amounts surface via the
  `summary` text where the checker has phrased them carefully.

# Open Questions

1. **Counter naming convention** â€” KCS-020's matrix uses
   `domain.metric` (e.g. `bank.unmatched_count`). Confirm this is
   the convention rather than a flat `bank_unmatched_count` style.
   The dotted form maps cleanly into CEL nested maps and reads
   well in YAML conditions; recommended.
2. **Checklist ordering** â€” items are unordered today (per the
   "no sequencing" out-of-scope point). The frontend sorts by
   state (pending â†’ done) for visual rhythm. Confirm this is the
   right default vs. checker-controlled order.
3. **Drill-link for inactive periods** â€” when a status item links
   to a sub-page (e.g. `/bank-reconciliation`), the active period
   should follow the user. T-0039's `useActivePeriod` already does
   this; just want to confirm we don't want a "drill into the BAS
   period" semantic that overrides the user's selector.
4. **Cap on dynamic checklist items** â€” `/approvals` could
   theoretically return one item per pending approval. Capping at
   10 (with a "and N more" footer) keeps the widget compact;
   confirm the cap isn't too aggressive.
5. **Materialised views** â€” KCS-062's 5-query budget is generous
   for the initial six pages. If a future checker hits the budget
   ceiling, do we tilt toward a materialised view (per-tenant) or
   a denormalised counter table? Worth deciding before the second
   tranche of checkers.
