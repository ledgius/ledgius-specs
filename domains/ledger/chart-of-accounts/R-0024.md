# Summary

Manage the chart of accounts and account headings that form the foundation of the double-entry ledger. Accounts are categorised by type (Asset, Liability, Equity, Income, Expense), grouped under headings, and linked to functional categories (AR, AP, IC) for use across the system.

# Problem / Business Context

Every accounting system requires a well-structured chart of accounts. Ledgius must support the Australian chart of accounts structure while allowing customisation. Accounts must be linkable to functional roles (AR, AP, inventory control) so that invoices, bills, receipts, and payments can automatically post to the correct GL accounts.

# Scope

- Account CRUD (create, read, update, soft-delete via obsolete flag)
- Account heading CRUD (create, read)
- Account-to-functional-link mapping (AR, AP, IC, etc.)
- Activity tracking per account (has_activity flag)
- Audit stamping on all mutations

# Out of Scope

- Account consolidation across multiple entities
- Account budgets and budget-vs-actual reporting
- Automatic account number generation
- Account hierarchies deeper than heading > account

# Actors / Users

- Business Owner -- configures chart of accounts during setup
- Bookkeeper -- creates accounts for new categories as needed
- Accountant -- reviews and restructures chart of accounts

# Preconditions

- User is authenticated
- Database is initialised with at least the Australian COA seed data

# Functional Requirements

- **COA-001**: System shall store accounts with: accno (unique), description, category (A/L/Q/I/E), heading reference, contra flag, tax flag, obsolete flag, and GIFI accno
- **COA-002**: System shall store account headings with: accno (unique), description, category, and optional parent heading reference
- **COA-003**: System shall support account links mapping accounts to functional categories (AR, AP, IC, etc.) via the `account_link` table
- **COA-004**: Account creation shall occur within a database transaction that atomically creates the account and its links
- **COA-005**: Account update shall occur within a database transaction that replaces all existing links with the new set
- **COA-006**: Account deletion shall soft-delete by marking the account as obsolete (not physical deletion)
- **COA-007**: System shall compute a `has_activity` flag per account indicating whether the account has any GL entries
- **COA-008**: Category field shall be validated to one of: A (Asset), L (Liability), Q (Equity), I (Income), E (Expense)
- **COA-009**: All account and heading mutations shall be audit-stamped (created_by, updated_by, timestamps)
- **COA-010**: System shall enforce unique account numbers across the chart

|--------|------|-------------|
| GET | /headings | List all account headings |
| GET | /headings/:id | Get a single heading |
| POST | /headings | Create a new heading |
| GET | /accounts | List all accounts with activity flags |
| GET | /accounts/:id | Get a single account with links |
| POST | /accounts | Create a new account with links |
| PUT | /accounts/:id | Update an account and its links |
| DELETE | /accounts/:id | Mark an account as obsolete |

# Validation Rules

- Account number (accno) is required and must be unique
- Description is required on create
- Category must be one of: A, L, Q, I, E
- Heading ID is required on create and must reference an existing heading

# Error Handling

- Duplicate accno returns a database constraint violation error
- Invalid category value returns a 400 with allowed values
- Non-existent account ID returns 404
- Non-existent heading ID returns 404
