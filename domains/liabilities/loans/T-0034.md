# Summary

Foundation task for the loans feature — delivers a complete, usable entry point for loan management. After this task merges, a user can:

- Originate a loan (with the correct GL postings, audit trail, and optional establishment-fee journal).
- See it on the register with header totals.
- Drill into the per-loan detail page with its projected schedule and (initially empty) payment + accrual history.
- Edit non-posting fields with a proper audit row.

Subsequent tasks layer on the rest of the lifecycle: Record Payment (T-0035), Payout (T-0036), periodic interest accrual (T-0037).

The previous draft of this task deferred origination to T-0035. That was wrong: a foundation task that only ships reads leaves the feature unusable until T-0035 lands, and masks the quality of the GL contract behind seed-only data. Origination is the core business action for creating a loan — it belongs here.

# Phase 1 — Schema (ledgius-db)

## T-0034.1: loan_register

`V1.YY__loan_register.sql` (next version at commit time):

Columns:

- `id uuid pk default gen_random_uuid()`
- `name text not null`
- `description text`
- `lender_eca_id int references entity_credit_account(id) on delete restrict`
- `original_amount numeric(20,2) not null check (original_amount > 0)`
- `interest_rate_annual_pct numeric(6,4) not null check (interest_rate_annual_pct >= 0)`
- `compounding_frequency text not null check (compounding_frequency in ('daily','monthly','quarterly','annually','none'))`
- `repayment_frequency text not null check (repayment_frequency in ('weekly','fortnightly','monthly','quarterly','annually','none'))`
- `repayment_amount numeric(20,2) not null check (repayment_amount >= 0)`
- `start_date date not null`
- `maturity_date date`
- `use_of_funds text not null check (use_of_funds in ('business','mixed','private'))`
- `business_use_pct numeric(5,2) check (business_use_pct is null or (business_use_pct >= 0 and business_use_pct <= 100))`
- `current_balance numeric(20,2) not null` — **no DB default**; service must set it explicitly on INSERT to `original_amount`.
- `accrued_interest_balance numeric(20,2) not null default 0` — zero at origination is correct (no time has passed yet).
- `liability_account_id int not null references account(id) on delete restrict`
- `accrued_interest_account_id int not null references account(id) on delete restrict`
- `interest_expense_account_id int not null references account(id) on delete restrict`
- `status text not null check (status in ('draft','active','paid_out','written_off','archived'))` — **no DB default**; service must set it explicitly on INSERT.
- `estimate_changes jsonb not null default '[]'::jsonb`
- `correction_id uuid`
- `origination_transaction_id int references transactions(id) on delete restrict`
- `payout_transaction_id int references transactions(id) on delete restrict`
- `payout_date date`
- `written_off_transaction_id int references transactions(id) on delete restrict`
- `written_off_date date`
- `created_at timestamptz not null default now()`
- `updated_at timestamptz not null default now()`

Constraints:

- Check `maturity_date is null or maturity_date > start_date`.
- Check `payout_date is null or payout_date >= start_date`.
- Check `written_off_date is null or written_off_date >= start_date`.
- Check `(use_of_funds = 'mixed') = (business_use_pct is not null)`.
- Check `(status = 'paid_out') = (payout_transaction_id is not null and payout_date is not null)`.
- Check `(status = 'written_off') = (written_off_transaction_id is not null and written_off_date is not null)`.

Indexes: `(status)`, `(lender_eca_id)`, `(maturity_date)`, `(correction_id) where correction_id is not null`.

**Why no default for `status` or `current_balance`**: the reviewer feedback on PR #29 correctly flagged that DB defaults can mask a missing service command. Origination is the *only* legitimate way a loan row comes into existence. Forcing the service to set `status='active'` and `current_balance=original_amount` explicitly in the same transaction as the GL posting eliminates the "what if someone INSERTs a row directly" class of bug.

Comments on every column pointing at the R-0063 LN-### references.

## T-0034.4: OriginateLoan command

Core business action. Delivers the complete origination flow per A-0044 §3 and A-0045 §2.

```go
type OriginateLoanCmd struct {
    TenantID                       uuid.UUID
    ActorUserID                    uuid.UUID

    Name                           string
    Description                    string
    LenderECAID                    int
    OriginalAmount                 decimal.Decimal
    InterestRateAnnualPct          decimal.Decimal
    CompoundingFrequency           string
    RepaymentFrequency             string
    RepaymentAmount                decimal.Decimal
    StartDate                      time.Time
    MaturityDate                   *time.Time
    UseOfFunds                     string  // business | mixed | private
    BusinessUsePct                 *decimal.Decimal

    // Account overrides (optional — defaults resolved from tenant COA if nil):
    LiabilityAccountID             *int
    AccruedInterestAccountID       *int
    InterestExpenseAccountID       *int

    // Receiving side — exactly one of these is used:
    ReceivingAccountID             int       // bank (cash proceeds) or capital account (asset-funded drawdown)

    // Optional establishment-fee journal:
    EstablishmentFee               *decimal.Decimal
    EstablishmentFeeBankAccountID  *int
}

func (s *Service) OriginateLoan(ctx context.Context, cmd OriginateLoanCmd) (*Loan, error)
```

Implementation:

1. **Validate** per R-0063 §Validation Rules: `original_amount > 0`; `interest_rate_annual_pct >= 0`; `maturity > start`; `(use_of_funds='mixed') ⇔ (business_use_pct is not null)`; `business_use_pct` in [0,100]; required receiving account resolvable.
2. **Period-lock check** (LN-016 / A-0044 I9): reject with `ErrPeriodLocked` if `start_date` falls in a locked or BAS-lodged period.
3. **Resolve missing accounts** from tenant COA defaults (liability, accrued interest payable, interest expense).
4. **Begin tx** via `db.RunInTx`.
5. **INSERT `loan_register`** with:
   - `status = 'active'`
   - `current_balance = original_amount`
   - `accrued_interest_balance = 0`
   - Resolved account IDs
   - `origination_transaction_id = null` (filled in step 7).
6. **Post the origination journal** via `journal.Post(ctx, journal.LoanOriginationCmd{...})` applying A-0045 §2.1:
   - Dr `receiving_account_id` (bank or capital) for `original_amount`.
   - Cr `liability_account_id` for `original_amount`.
   - Both rows tagged `loan_id`, the Cr tagged `use_of_funds`.
7. **Update the loan row** setting `origination_transaction_id` = posted parent transaction id (same tx).
8. **Optional establishment fee**: if `establishment_fee > 0`, post a sibling transaction (`type='loan_establishment_fee'`, `source_transaction_id={origination_id}`) per A-0045 §2.2: Dr Loan Establishment Fee (expense), Cr `establishment_fee_bank_account_id`.
9. **Write audit row** via `audit.Write(entity='loan', entity_id=loan.id, action='originated', payload={original_amount, lender_eca_id, use_of_funds, establishment_fee})` — same tx.
10. **COMMIT**. Return the loan with book-value-style derived fields populated.

Errors:

- `ErrInvalidLoanInput` — validation failure with field-level detail.
- `ErrAccountResolution` — required account couldn't be resolved.
- `ErrPeriodLocked` — LN-016.
- `ErrIdempotencyConflict` — duplicate origination key (retry of a partially-succeeded call).

## T-0034.5: journal service extension

Add to `internal/journal/`:

- `LoanOriginationCmd` — takes loan + principal + receiving account. Produces rows per A-0045 §2.1 with `loan_id` + `use_of_funds` tags; returns `transactions.id`.
- `LoanEstablishmentFeeCmd` — takes fee amount + fee bank + reference to parent origination. Produces rows per A-0045 §2.2; returns sibling `transactions.id`.

Idempotency keys per A-0045 §9: `loan:originate:{loan_id}` and `loan:establishment_fee:{loan_id}:{origination_txn_id}`.

## T-0034.6: endpoints

See `openapi-loans.yaml` for shapes.

- `POST /api/v1/loans` — origination. `loans:create`. Returns 201 with the created loan.
- `GET  /api/v1/loans` — list with filters (`status`, `lender_id`, `maturity_within_months`, `q`, `include_paid_out`) + header totals (count_active, total_principal_balance, total_periodic_obligation, total_accrued_interest) + pagination.
- `GET  /api/v1/loans/:id` — single loan with lender (joined entity).
- `PATCH /api/v1/loans/:id` — non-posting-field edit (name, description, use_of_funds, business_use_pct). Writes audit `edited`. (Non-posting edit is OK here because it doesn't touch GL — corrections to rate/term go through `ChangeLoanEstimate` in a later task.)
- `GET  /api/v1/loans/:id/activity` — audit timeline, newest first.
- `GET  /api/v1/loans/:id/schedule` — **projected** repayment schedule computed from rate + amount + frequency + current balance. Read-only derivation; no DB write.

Record-payment / accrual / payout / write-off endpoints are owned by T-0035/T-0036/T-0037 but the service layer method stubs land here so the package compiles.

## T-0034.8: tests

- **Unit tests** on the calculator for the schedule (straight-line amortisation, interest-only, zero-rate) and filter/totals SQL combinations.
- **Service tests** for `OriginateLoan`:
  - Happy path: validates GL rows match A-0045 §2.1 table exactly, loan row has correct `status`, `current_balance`, `origination_transaction_id`; audit row written with payload.
  - Establishment-fee variant: verifies sibling parent exists and references the origination.
  - Asset-funded drawdown: verifies the receiving account can be a capital account (category A), not just a bank.
  - Validation rejections: missing required fields, invalid `use_of_funds`, `business_use_pct` on non-mixed loan, `maturity <= start`, `original_amount <= 0`.
  - Period-lock rejection: `start_date` in a locked period → `ErrPeriodLocked`.
  - Idempotency: calling OriginateLoan twice with the same idempotency key produces one loan.
  - **Ledger invariant**: `sum(Dr) == sum(Cr)` for every posted transaction.
- **Service tests** for `EditNonPosting`: audit row written with minimal diff; empty diff produces no audit row; posting-impacting fields rejected.
- **Handler tests** via `httptest` — happy path 201; 403 on missing permission; 402 on missing `loans` feature; 400 on bad input.
