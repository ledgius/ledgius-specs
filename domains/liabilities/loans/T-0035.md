# Summary

Deliver loan origination and the payment-recording flow end-to-end. Backend: `POST /api/v1/loans` (originate) and `POST /api/v1/loans/:id/payments` (record payment). Frontend: replace the `LoanPaymentsPage` placeholder with a working list + a Record Payment form launched per-loan from the Detail page or with a direct `?loanId=` deep link.

Depends on T-0034 (schema + register/detail). Shares accrual primitives with T-0037 — if T-0037 lands first, reuse its catch-up-accrual helper; otherwise implement a thin version here that T-0037 generalises.

# Branch

`feat/loans-origination-payment` — off the same base T-0034 lands on (rebase after T-0034 merges).

# Phase 2 — Backend

## T-0035.2: originate command

Add to `internal/loan/service.go`:

```go
type OriginateLoanCmd struct {
    TenantID                       uuid.UUID
    ActorUserID                    uuid.UUID
    Name                           string
    Description                    string
    LenderECAID                    int
    OriginalAmount                 decimal.Decimal
    InterestRateAnnualPct          decimal.Decimal
    CompoundingFrequency           string
    RepaymentFrequency             string
    RepaymentAmount                decimal.Decimal
    StartDate                      time.Time
    MaturityDate                   *time.Time
    UseOfFunds                     string  // business | mixed | private
    BusinessUsePct                 *decimal.Decimal
    LiabilityAccountID             *int
    AccruedInterestAccountID       *int
    InterestExpenseAccountID       *int
    ReceivingAccountID             int  // bank or capital account
    EstablishmentFee               *decimal.Decimal
    EstablishmentFeeBankAccountID  *int
}

func (s *Service) OriginateLoan(ctx context.Context, cmd OriginateLoanCmd) (*Loan, error)
```

Implementation outline:

1. Validate per LN-### + R-0063 §Validation Rules: amount > 0; rate >= 0; maturity > start; `(use_of_funds='mixed') = (business_use_pct is not null)`; business_use_pct in [0,100]; receiving account exists.
2. **Period-lock check** (LN-016 / A-0044 I9): reject if `start_date` falls in a locked or BAS-lodged period.
3. Resolve missing accounts from tenant defaults.
4. Begin tx.
5. INSERT `loan_register` with status=`active`.
6. Call `journal.Post(ctx, journal.LoanOriginationCmd{...})` per A-0045 §2.1.
7. If `establishment_fee > 0`, call `journal.Post(ctx, journal.LoanEstablishmentFeeCmd{...})` per A-0045 §2.2 — sibling parent linked via `source_transaction_id`.
8. Set `loan.origination_transaction_id` = posted transaction id.
9. `audit.Write(entity='loan', action='originated', payload={original_amount, lender_eca_id, use_of_funds})`.
10. COMMIT. Return loan.

## T-0035.3: record-payment command

```go
type RecordPaymentCmd struct {
    TenantID         uuid.UUID
    ActorUserID      uuid.UUID
    LoanID           uuid.UUID
    PaymentDate      time.Time
    PrincipalAmount  decimal.Decimal
    InterestAmount   decimal.Decimal
    BankAccountID    int
    Notes            string
}

func (s *Service) RecordPayment(ctx context.Context, cmd RecordPaymentCmd) (*LoanPayment, error)
```

Implementation outline:

1. Load loan. Validate state ∈ {Active}, payment_date >= max(start_date, last_accrual_period_end), period not locked.
2. Determine basis: `tenant_setting.loan_interest_basis` ∈ {accrual, cash}.
3. Validate split: `principal + interest > 0`; `interest >= 0`; `principal <= loan.current_balance`; if `principal > current_balance + accrued`, reject `ErrOverpayment`.
4. Begin tx.
5. **Accrual basis only**: if `interest_amount > loan.accrued_interest_balance`, post a catch-up accrual covering `(last_accrual_end, payment_date]` to bring accrued up to `interest_amount`. Use the shared accrual helper (from T-0037 or local).
6. Call `journal.Post(ctx, journal.LoanPaymentCmd{...})` — accrual-basis or cash-basis variant per A-0045 §4.
7. Update `loan.current_balance -= principal_amount`; if accrual-basis, `loan.accrued_interest_balance -= interest_amount`.
8. INSERT `loan_payment` with `transaction_id`, `balance_after = loan.current_balance`.
9. `audit.Write(action='payment_recorded', payload={principal, interest, total, balance_after})`.
10. COMMIT.

## T-0035.4: reverse-payment command

`POST /api/v1/loans/:id/payments/:payment_id/reverse` — posts inverse rows under a new `loan_payment_reversal` parent; restores balances; sets `loan_payment.reversed_transaction_id`. Audit `payment_reversed`.

## T-0035.5: handlers + routes

- `POST /api/v1/loans` → `handler.Originate` (`loans:create`).
- `POST /api/v1/loans/:id/payments` → `handler.RecordPayment` (`loans:record_payment`).
- `POST /api/v1/loans/:id/payments/:payment_id/reverse` → `handler.ReversePayment` (`loans:record_payment`).
- `GET  /api/v1/loans/:id/payments` → list payments for a loan (already in T-0034 if sequenced first; otherwise add here).
- `GET  /api/v1/loans/payments` → all payments across all loans (powers the LoanPaymentsPage list).

Add the corresponding entries to `DefaultRoleFunctionMap`.

## T-0035.6: tests

- Originate: cash + asset-funded modes; with + without establishment fee; mixed-use loan with business_use_pct.
- Originate rejects: missing required fields, invalid use_of_funds, period locked.
- Record payment: accrual-basis with exact-accrued + over-accrued (catch-up triggered) + under-accrued cases; cash-basis variant.
- Record payment: principal-only payment (interest=0); interest-only payment (principal=0); over-payment rejected.
- Reverse payment: balances restored exactly.
- Audit rows written for every command with correct payload.
- Ledger invariant `sum(Dr) == sum(Cr)` per case.
- Period lock rejection per command.
