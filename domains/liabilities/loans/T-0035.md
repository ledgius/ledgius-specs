# Summary

Deliver the payment lifecycle chapter for loans: record a payment with the correct principal / interest split, and reverse a recorded payment. Depends on T-0034 (schema + origination already in place — this task gives users a way to pay down active loans).

This task **does not** cover origination (moved into T-0034 after PR #29 review), accruals (T-0037), or payout (T-0036).

The shared accrual helper that this task's catch-up logic needs is owned by T-0037; if T-0037 lands first, reuse it. If this task lands first, implement a thin local version that T-0037 generalises.

# Branch

`feat/loans-record-payment` — off the same base T-0034 lands on (rebase after T-0034 merges).

# Phase 1 — Schema additions

## T-0035.1: loan_payment

`V1.YY__loan_payment.sql`:

- `id bigserial pk`
- `loan_id uuid not null references loan_register(id) on delete restrict`
- `payment_date date not null`
- `principal_amount numeric(20,2) not null check (principal_amount >= 0)`
- `interest_amount numeric(20,2) not null check (interest_amount >= 0)`
- `total_amount numeric(20,2) not null` — check `total_amount = principal_amount + interest_amount`
- `balance_after numeric(20,2) not null check (balance_after >= 0)`
- `bank_account_id int not null references account(id) on delete restrict`
- `transaction_id int not null references transactions(id) on delete restrict`
- `reversed_transaction_id int references transactions(id) on delete restrict`
- `notes text`
- `created_at timestamptz not null default now()`
- `updated_at timestamptz not null default now()`

Indexes: `(loan_id, payment_date desc)`, `(reversed_transaction_id) where reversed_transaction_id is not null`.

# Phase 2 — Backend

## T-0035.2: RecordPayment command

```go
type RecordPaymentCmd struct {
    TenantID         uuid.UUID
    ActorUserID      uuid.UUID

    LoanID           uuid.UUID
    PaymentDate      time.Time
    PrincipalAmount  decimal.Decimal
    InterestAmount   decimal.Decimal
    BankAccountID    int
    Notes            string
}

func (s *Service) RecordPayment(ctx context.Context, cmd RecordPaymentCmd) (*LoanPayment, error)
```

Implementation:

1. Load loan. Validate `status='active'`; `payment_date >= max(start_date, last_accrual_period_end)`; period not locked.
2. Determine basis: `tenant_setting.loan_interest_basis` ∈ {`accrual`, `cash`}. Default is `accrual`.
3. Validate split: `principal + interest > 0`; `interest >= 0`; if `principal > loan.current_balance`, reject with `ErrOverpayment`; if `interest > loan.accrued_interest_balance + projected_catchup`, reject.
4. Begin tx.
5. **Accrual basis only**: if `interest_amount > loan.accrued_interest_balance`, post a catch-up accrual covering `(last_accrual_end_or_origination, payment_date]` via the shared helper — bringing `accrued_interest_balance` up to at least `interest_amount`. The catch-up is its own `transactions` row of `type='interest_accrual_adjustment'`, linked via `source_transaction_id` to the payment.
6. Call `journal.Post(ctx, journal.LoanPaymentCmd{...})` applying A-0045 §4.1 (accrual) or §4.2 (cash). Rows tagged `loan_id`, `payment=true`; cash-basis tags `use_of_funds` on the interest-expense row.
7. Update loan: `current_balance -= principal_amount`; accrual-basis `accrued_interest_balance -= interest_amount`.
8. INSERT `loan_payment` with `transaction_id`, `balance_after = loan.current_balance`.
9. `audit.Write(action='payment_recorded', payload={principal, interest, total, balance_after, bank_account_id, basis})`.
10. COMMIT.

Idempotency key: `loan:payment:{loan_id}:{payment_date}:{seq}` where `seq` is a monotonic counter for payments on the same date.

## T-0035.3: ReversePayment command

```go
type ReversePaymentCmd struct {
    TenantID     uuid.UUID
    ActorUserID  uuid.UUID
    PaymentID    int64
    Reason       string
}

func (s *Service) ReversePayment(ctx context.Context, cmd ReversePaymentCmd) (*LoanPayment, error)
```

Implementation:

1. Load payment. Validate not already reversed. Load loan. Period not locked.
2. Begin tx.
3. Call `journal.Post(ctx, journal.LoanPaymentReversalCmd{source_transaction_id: payment.transaction_id})`.
4. Restore balances: `loan.current_balance += payment.principal_amount`; accrual-basis `loan.accrued_interest_balance += payment.interest_amount`.
5. Set `payment.reversed_transaction_id` = reversal parent.
6. `audit.Write(action='payment_reversed', payload={reason, principal_restored, interest_restored})`.
7. COMMIT.

## T-0035.4: journal service extensions

- `LoanPaymentCmd` — produces A-0045 §4.1 or §4.2 rows based on basis.
- `LoanPaymentReversalCmd` — exact inverses of the source payment's rows.
- `LoanAccrualAdjustmentCmd` (if T-0037 hasn't landed yet) — thin helper posting A-0045 §3 shape for a single loan between two arbitrary dates.

## T-0035.5: endpoints

See `openapi-loans.yaml`.

- `POST /api/v1/loans/:id/payments` — `loans:record_payment`. Returns 201 with the recorded payment.
- `POST /api/v1/loans/:id/payments/:payment_id/reverse` — `loans:record_payment`. Returns 200 with the updated payment.
- `GET  /api/v1/loans/:id/payments` — list payments for a loan, paginated.
- `GET  /api/v1/loans/payments` — global list across all loans (powers LoanPaymentsPage).

Role-permission map additions:

```go
{"/api/v1/loans", "POST",  "loans", "record_payment"},  // path-prefix match — narrowed by method + payload check in handler? Actually the existing map uses prefix; adjust to method-scoped:
```

Because `POST /loans` (origination) and `POST /loans/:id/payments` need different functions, the role-function guard must match on exact prefix + method. Verify the middleware supports this; if not, introduce a narrower mapping that checks `/payments` in the path suffix.

## T-0035.6: tests

- Accrual-basis happy paths: interest = 0, interest ≤ accrued (no catch-up), interest > accrued (catch-up triggered — verify two parent transactions), interest_only payment (principal=0), principal_only payment (interest=0).
- Cash-basis variant: verify direct-to-expense posting, no accrual movement, `use_of_funds` tag on the interest-expense row.
- Reject: `principal > current_balance`, negative amounts, payment_date before origination, payment_date in locked period.
- ReversePayment happy path: balances restored exactly, linked txn ids correct.
- ReversePayment on already-reversed payment → error.
- Audit rows written.
- Ledger invariant per case.
