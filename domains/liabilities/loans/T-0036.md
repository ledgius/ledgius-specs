# Summary

Deliver loan payout (early settlement) end-to-end. Backend: `POST /api/v1/loans/:id/payout` that atomically posts any pro-rata accrual, the payment clearing principal + accrued, the optional termination-fee journal, and transitions the loan to `paid_out`. Preview endpoint shows the journal before commit. Frontend: replace the `LoanPayoutPage` placeholder with a working form, pre-filled when navigated from the loan detail page via `?loanId=`.

Depends on T-0034 (schema + register/detail) and T-0035 (origination, so there is something to pay out). Reuses the shared accrual helper introduced by T-0037 (or T-0035, whichever lands first).

# Phase 1 — Backend

## T-0036.1: payout command

```go
type PayoutLoanCmd struct {
    TenantID                       uuid.UUID
    ActorUserID                    uuid.UUID
    LoanID                         uuid.UUID
    PayoutDate                     time.Time
    PrincipalClearance             decimal.Decimal  // typically loan.current_balance
    InterestClearance              decimal.Decimal  // typically accrued + pro-rata
    TerminationFee                 *decimal.Decimal
    BankAccountID                  int
    TerminationFeeBankAccountID    *int
}

func (s *Service) PayoutLoan(ctx context.Context, cmd PayoutLoanCmd) (*Loan, error)
```

Implementation outline:

1. Load loan. Validate state ∈ {Active}, payout_date >= start_date, period not locked.
2. Validate clearance amounts: `principal_clearance == loan.current_balance` (default; user override only allowed for partial payouts which we don't support in v1 — reject otherwise); `interest_clearance >= loan.accrued_interest_balance`.
3. Begin tx.
4. **Accrual basis only**: if `payout_date > last_accrual_period_end`, post a pro-rata accrual covering `(last_accrual_end, payout_date]` using the shared accrual helper.
5. Verify post-accrual `interest_clearance == loan.accrued_interest_balance` exactly. If user-supplied amount differs, reject — the form should pre-compute and pass through unchanged.
6. Call `journal.Post(ctx, journal.LoanPayoutCmd{...})` which produces the rows in A-0045 §5.2 under a `loan_payout` parent.
7. If `termination_fee > 0`, call `journal.Post(ctx, journal.LoanTerminationFeeCmd{...})` per A-0045 §5.3 — sibling parent linked via `source_transaction_id`.
8. Update loan: `status='paid_out'`, `payout_date`, `payout_transaction_id`, `current_balance=0`, `accrued_interest_balance=0`.
9. `audit.Write(action='paid_out', payload={principal_cleared, interest_cleared, fee})`.
10. COMMIT.

## T-0036.2: preview endpoint

`POST /api/v1/loans/:id/payout/preview` — accepts the same body but does NOT post. Computes the would-be journal (pro-rata accrual + payout + fee), returns:

```json
{
  "current_balance": "12345.67",
  "accrued_interest_to_payout_date": "456.78",
  "termination_fee": "100.00",
  "total_cash_outlay": "12902.45",
  "estimated_savings_vs_remaining_term": "789.10",
  "rows": [...]  // JournalRow[]
}
```

The savings figure is an information-only computation (sum of remaining scheduled interest on the projected schedule less the early termination fee) — it does not appear in the journal.

## T-0036.3: handlers + routes

- `POST /api/v1/loans/:id/payout` (`loans:payout`).
- `POST /api/v1/loans/:id/payout/preview` (`loans:view`).

## T-0036.4: tests

- Payout on accrual-basis with last accrual same as payout_date (no pro-rata).
- Payout on accrual-basis with payout_date mid-period (pro-rata posted).
- Payout on cash-basis (skips pro-rata; interest goes direct to expense in the payout journal).
- Payout with + without termination fee.
- Reject when payout_date < start_date or in locked period.
- Reject when principal_clearance != current_balance.
- Audit row.
- Ledger invariant per case.
