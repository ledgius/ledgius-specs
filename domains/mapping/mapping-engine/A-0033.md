# Summary

Architecture for a bidirectional, YAML-configured, operator-based mapping engine. Operators are typed Go function pointers registered at boot. Configs declare per-operator context-layer requirements so the engine plans only the buffers and pre-passes actually needed. Single-row streaming is the default; back-window buffer, cross-entity lookup, and aggregate pre-passes are opt-in. A platform-owner web page renders configs side-by-side.

# Terms

Vocabulary for mapping config, operator, operator registry, context layer, mapping config governance, and platform-owner mapping viewer is defined canonically in **R-0059 §Terms**. Vocabulary for CEL, rules.Service is defined canonically in **A-0011**. Vocabulary for external_system, integration direction is defined canonically in **R-0058 §Terms**.

| **Term** | Definition. |
|---|---|
| **Mapping plan** | The engine's execution plan computed from a config at run-start. Lists required buffers, pre-passes, and lookup sources derived from the union of operator context declarations. Plans are cached per (config_id, version). |
| **LookupSource** | An interface providing batched, keyed access to a related entity set for Layer 3 lookups. Two built-in implementations: `InMemoryLookup` (eager-load for small bounded sets like tax codes) and `DBLookup` (batched query for large sets like historical invoices). |
| **Pre-pass** | A streaming iteration over the source dataset that runs once before the main mapping pipeline, producing aggregate snapshots. Invoked only when Layer 4 operators are declared. |

# Technical Context

Ledgius already has a rules-engine foundation (A-0011) using cel-go, OPA, and GoRules. Mapping is the next domain that fits the rules-engine principle but hasn't yet been expressed as rules. A-0031 (Xero export) is the first concrete consumer; R-0058 / A-0032 provides the external systems registry that configs will key against.

Key constraints shaping the design:
- Throughput target: ≥ 50k operator invocations/sec. Rules out reflection-heavy or interpretation-heavy designs. Favours typed function pointers, pre-registered at boot, invoked via direct function-pointer call.
- Memory bound: configurable by declaration only. A config with zero back-window requirements must maintain zero buffers.
- Streaming default: most operators are row-local and don't need context beyond the current row. Opting into more context must be explicit.
- Auditability: every mapping config is versioned, the viewer is a first-class surface, and historical runs must be replayable against the exact config version they ran under.

## Operator interface

All operators implement a single typed interface. The engine never sees `any` — operators receive typed inputs and produce typed outputs.

```go
package mapping

// OperatorContext carries per-row metadata plus any higher-layer context
// the operator has declared via its config.
type OperatorContext struct {
    RowIndex     int
    SourceRowID  string
    BackWindow   []RowSnapshot     // nil unless Layer 1 declared
    Lookup       LookupContext     // nil unless Layer 3 declared
    Aggregates   AggregateSnapshot // zero value unless Layer 4 declared
    CEL          *rules.Evaluator  // non-nil for operators that embed CEL
}

// Operator is a typed transformation: given the current source row and
// context, produce a target value. Params is a map of YAML-configured
// arguments specific to this operator.
type Operator interface {
    Name() string
    ContextRequirements() ContextRequirements
    Apply(ctx OperatorContext, source any, params map[string]any) (target any, err error)
}

// ContextRequirements declares what the engine must make available.
// The engine plans buffers and pre-passes from the union across all
// operators used in a config.
type ContextRequirements struct {
    BackWindowSize int                   // 0 = no back window
    LookupEntities []LookupRequirement   // empty = no lookups
    Aggregates     []AggregateRequirement
    CEL            bool
}
```

Operators register themselves at package-init time:

```go
package operators

func init() {
    mapping.RegisterOperator(&copyOperator{})
    mapping.RegisterOperator(&upperOperator{})
    mapping.RegisterOperator(&resolveTaxRateOperator{})
    // ...
}
```

## Mapping config YAML shape

```yaml
schema_version: mapping.xero-export.v1
external_system: xero
direction: export
entity: invoice

fields:
  - target: ContactName
    operators:
      - name: copy
        source: contact.display_name

  - target: InvoiceNumber
    operators:
      - name: copy
        source: document_number

  - target: InvoiceDate
    operators:
      - name: date_format
        source: issue_date
        params:
          format: "DD/MM/YYYY"

  - target: TaxType
    operators:
      - name: resolve_tax_rate
        source: tax_code
        params:
          region: AU
          default_if_null: BAS_Excluded
        context:
          lookup:
            - entity: tax_code
              by: id
              cache: in_memory

  - target: LineDescription
    operators:
      - name: copy
        source: line_description
      - name: default_if_null
        params:
          fallback: "Line item"

  - target: ConsolidatedLineTotal
    operators:
      - name: cel_expression
        params:
          expression: "lookback.sum('amount') + source.amount"
        context:
          back_window: 50
          cel: true
```

## Schema validation

A JSON-schema artifact (`internal/mapping/configs/schema.json`) defines the structure above. At boot:

1. For each YAML file under `internal/mapping/configs/*/`:
   - Parse YAML to Go structs
   - Validate against the schema (required fields, allowed enum values, types)
   - Resolve every `operators[].name` against the registry
   - Validate declared `context` requirements are supported by the engine
2. Aggregate all loaded configs into an in-memory registry keyed by `(external_system, direction, entity, schema_version)`
3. Log every registered config and its required context layers
4. Any failure aborts boot

## Engine execution

`Engine.Run(ctx, configID, sourceIterator)` returns `TargetIterator` that streams:

1. **Plan.** Look up cached plan for configID (or compute on first use). Plan contains: required buffers, required pre-passes, required lookup sources.
2. **Pre-pass** (if any Layer 4 aggregates are required): single streaming iteration over source, computing each aggregate. Snapshots are frozen for the main pass.
3. **Main pass:** iterate source. For each row:
   - If Layer 1 is required, shift the back-window buffer (discard oldest, append current).
   - Build an `OperatorContext` with the relevant layers wired up.
   - For each field in config order:
     - For each operator in the field's chain:
       - Resolve `source` reference on the current row
       - Invoke `operator.Apply(ctx, source, params)`
       - Use the result as the source for the next operator in chain (operators are composable)
     - Assign final chain result to the target field
   - Yield the assembled target row

Main-pass memory: back-window (N rows), one target row under construction, one source row. Bounded.

## Cross-entity lookup batching

When a config declares Layer 3 lookups, the engine does **not** issue a query per row. Instead:

1. Pre-pass (already walking the data for aggregates, or a dedicated pass if aggregates aren't declared) collects the set of lookup keys needed.
2. Engine calls `LookupSource.BulkFetch(keys)` once.
3. Result populates an in-memory map passed to operators via `LookupContext`.
4. Operators read via `ctx.Lookup.Get(entity, key)` which is O(1) map access.

For truly streaming scenarios where a pre-pass is unacceptable, `DBLookup` offers a fallback `FetchOnDemand` mode with batched reads (fetch next 100 keys when the cache misses). Default is pre-pass.

## CEL integration

`cel_expression` operator wraps a CEL program compiled at config load. At apply time it invokes `rules.Service.Evaluate(program, env)` where `env` is populated with:
- `source` (current row fields)
- `lookback` (BackWindow helper — `.sum()`, `.max()`, `.count_by(field)`)
- `lookup` (LookupContext accessor)
- `aggregates` (pre-computed snapshots)

CEL's sandboxed-by-default behaviour gives us safety (no I/O, bounded runtime). Compile-time errors in CEL expressions abort boot (via schema validation at load).

## Platform-owner mapping viewer API

```
GET  /api/v1/mapping/configs                       # list all loaded configs (code, system, direction, entity, schema_version)
GET  /api/v1/mapping/configs/:id                   # single config — full rendered view (source | operators | target rows)
GET  /api/v1/mapping/operators                     # registered operator catalog — name, param schema, context requirements
```

All routes require Platform Owner role.

## Governance for config changes

Since configs are embedded in `ledgius-api` under `internal/mapping/configs/`, config changes go through the standard PR flow:
- Developer edits YAML
- PR runs tests (including any mapping-engine integration tests that exercise the config)
- Reviewer approves
- Merge + deploy

No runtime edit path in v1. If future customer-specific overrides are required, they'd live in a tenant-scoped override layer, not as runtime edits to the base config.

## Audit-replay via schema_version

When an export run kicks off, the engine records the exact `schema_version` of the config used in the export_run row (and audit_log metadata). For historical replay or diagnostic purposes, the same config version is resolvable because older versions are kept in the `configs/` tree until deprecated.

Deprecation rule: a config version may be removed only after all export_run rows referencing it have been archived or deleted. A boot-time check warns if config references are orphaned.

# Consequences / Trade-offs

**Throughput cost of YAML vs native Go:** typed-function-pointer dispatch is ~1.5× native-call overhead. For 2M invocations that's ~50ms extra per run — negligible at real scale. Offset by pre-compiled CEL and batched lookups.

**Debuggability dips before it improves:** a Go stacktrace hits the mapping line directly; an engine log says "operator `resolve_tax_rate` failed at row 4382, source=`AU-GST-OUT`". Structured logs and the viewer make this more informative than a naked stacktrace eventually, but there's a ramp-up.

**Escape hatch via CEL:** complex transforms use CEL rather than Go extension points. This concentrates complexity in CEL expressions (bounded, sandboxed) rather than spreading custom Go across target packages. Tradeoff: CEL is less familiar than Go.

**Governance via PR flow, not runtime UI:** in v1, config edits require PRs. Less flexible than a live editor but safer: changes go through code review and deploy together. A runtime editor + approval workflow is a future enhancement, not v1 scope.

**No Layer 5 (whole dataset):** aggregate snapshots handle "need totals" without giving up streaming. If a real use case genuinely needs the full dataset, the engine will refuse the config and the requirement gets escalated — likely surfacing a better design.

# Alternatives Considered

- **Reflection-based mapper** (read source struct via reflection, write target fields via reflection): rejected — throughput unacceptable at 2M invocations, type safety only at runtime, obscures errors.
- **Protobuf/gRPC-style generated code from `.proto` mapping files**: rejected — generation tooling is heavy; the domain is small enough that YAML + operators is simpler. Revisit only if config count grows significantly.
- **Every transform in CEL, no Go operators**: rejected — simple field copies become verbose CEL, throughput is lower than pre-registered Go operators, and CEL alone can't express all needed transforms (e.g. regex with named captures, locale-aware decimal format). Hybrid wins.
- **Per-tenant mapping overrides in v1**: deferred — explodes test surface. Platform-owner-global scope for v1.

# Migration / Rollout

1. Ship engine core + built-in operators + schema + empty config set — cheap, unblocks consumer work.
2. Ship platform-owner viewer (empty state — no configs to view yet).
3. First consumer: R-0057 Xero export. Phase C of the export plan populates `configs/xero-export-v1/*.yaml`. The engine is exercised end-to-end.
4. Second consumer: future MYOB export. Validates that the engine is genuinely target-agnostic.
5. Later: config hot-reload, runtime editor with approval workflow, per-tenant overrides.

Rollback plan: if the engine proves infeasible, fall back to hardcoded Go mappers per target. This would rewrite Phase C but leave R-0058 external systems and R-0060 operations console intact.

# Open Questions

- Is there a case for letting operators skip ahead (emit multiple target rows from one source row, or none)? Keeps engine simple if we say "one source row → one target row, always." Complex grouping is handled by back-window + a sentinel row at end of group. Proposed: keep 1:1 mapping at engine level, let operators express grouping via back-window.
- Should we support mapping configs in formats other than YAML (JSON, TOML)? No v1 value. YAML is human-readable and diff-friendly — stick with it.
- Config file layout on disk: one file per entity type, or one file per config with multiple entities? Proposed: one file per entity type (e.g. `xero-export-v1/invoice.yaml`). Easier to review, smaller diffs.
