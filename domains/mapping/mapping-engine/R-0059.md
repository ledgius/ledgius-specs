# Summary

A reusable, YAML-configured mapping engine that translates records between Ledgius domain entities and external accounting systems (Xero, MYOB, QuickBooks, future) in either direction. Each source-field → target-field mapping is an invocation of a **registered operator** (typed Go function pointer); the engine iterates the mapping configuration and executes operators. Configurations are persisted as YAML, validated at boot, and rendered side-by-side in a platform-owner UI showing every field's source, operator(s), and target — so mappings are transparent, auditable, and editable without code changes.

# Terms

| **Term** | Definition. |
|---|---|
| **Mapping config** | A YAML file declaring how records of one entity type (e.g. Ledgius invoice) translate to another (e.g. Xero invoice). Identified by `schema_version` (e.g. `mapping.xero-export.v1`) so versions coexist for audit-replay of historical runs. |
| **Operator** | A typed, registered Go function that transforms one or more source values into one target value. Example built-ins: `copy`, `upper`, `lower`, `trim`, `regex_replace`, `lookup_table`, `default_if_null`, `decimal_format`, `date_format`, `resolve_tax_rate`. Target-system-specific operators live alongside target packages (e.g. `xero_account_code`). |
| **Operator registry** | The in-memory catalogue of all registered operators, populated at API boot. The YAML loader validates every operator name referenced in config against the registry before allowing the config to be used. |
| **Context layer** | The level of information an operator sees beyond the current source row. Layers: (0) single row; (1) back-window buffer; (2) forward-window buffer; (3) cross-entity lookup; (4) aggregates from a pre-pass; (5) whole dataset. Operators opt into higher layers by declaration; the engine plans buffers and pre-passes accordingly. |
| **Mapping config governance** | For v1: platform-owner edit-only, globally-scoped per (external_system, direction). Per-tenant overrides are out of scope. |
| **Platform-owner mapping viewer** | Read-only web page (v1) rendering a mapping config side-by-side: source field, operator chain with params, target field. One row per field. Scoped to platform-owner role. |

Vocabulary used here but defined elsewhere:
- `external_system`, `integration direction`, capability flags — defined canonically in **R-0058 §Terms**
- Rules engine, CEL, OPA, GoRules — defined canonically in **A-0011**

# Problem / Business Context

Translation between accounting systems is business logic — the rules for translating a Ledgius account code into a Xero account code, a Ledgius tax code into a Xero tax rate display name, a Ledgius contact record into a Xero contact, etc. Per **A-0011** and the foundational rules-engine principle documented in this project, business logic must not be hardcoded in Go. The mapping layer has so far escaped that principle only because it wasn't yet being built; this spec corrects that before the first consumer (R-0057 Xero export) lands.

Three concrete problems a hardcoded Go mapper would create:

1. **Opacity.** A platform-owner or support engineer cannot answer "how does Ledgius.Contact.AddressLine1 map to Xero.POAddressLine1?" without reading Go source. This is especially painful when customers report "this field came across wrong" — diagnosis requires a code dive.
2. **Rewrite cost for every new target.** MYOB export, QuickBooks export, and any future target would each be a fresh Go mapper package. The shape of the problem is identical across targets; the content differs. That is exactly what a data-driven engine is for.
3. **No governance surface.** A hardcoded mapper cannot be previewed, diffed, or reviewed as a first-class artifact. YAML configs can: they diff cleanly in PRs, render in a viewer, and carry their own `schema_version` for audit-replay when a mapping rule changes.

The engine is not a speculative investment — it replaces work we would otherwise duplicate per target system.

# Scope

## In scope for v1

- Operator registry with typed Go function signatures
- YAML config loader with JSON-schema validation at boot
- Streaming engine for the default single-row context (Layer 0)
- Opt-in Layer 1 (back-window buffer) for grouping-type transforms
- Opt-in Layer 3 (cross-entity lookup) with pluggable `LookupSource` implementations (in-memory cache for small sets, batched DB query for large sets)
- Opt-in Layer 4 (aggregates) via pre-pass
- Built-in operator library: `copy`, `upper`, `lower`, `trim`, `regex_replace`, `lookup_table`, `default_if_null`, `decimal_format`, `date_format`, `resolve_tax_rate`
- Optional CEL expression hook for non-trivial transforms, using the existing `rules.Service`
- Platform-owner mapping viewer web page (read-only v1): side-by-side source | operator | target rendering
- Config schema versioning (`mapping.<system>-<direction>.<version>`)

## Deferred (not this spec, not v1)

- Layer 2 (forward-window buffer) — adds latency every operator pays; defer until a real mapping demands it
- Layer 5 (whole-dataset access) — forces batch mode; support as escape hatch only when proven necessary
- Per-tenant mapping overrides
- Mapping viewer edit mode (v1 is read-only; edit comes later with diff/approve flow)
- Live config hot-reload (v1 loads at boot; SIGHUP or endpoint-triggered reload is a later enhancement)

## Out of scope

- Cross-system reconciliation (one-way translation only)
- Automatic config generation from remote API schemas (future; someday)
- Runtime admin API for adding operators at runtime (operators are always Go code, registered at boot)

# Actors / Users

- **Platform Owner** — primary editor of mapping configs. Reviews mapping viewer to understand and verify how data translates between systems.
- **Support Engineer** — reads the mapping viewer when diagnosing "why did this field come across wrong?" reports from customers. Never edits.
- **Developer adding a new external system** — writes YAML configs and (occasionally) target-system-specific operators. Follows the extension checklist.
- **Developer modifying an existing mapping** — edits YAML, runs tests, raises PR. No Go change for most edits.

# Functional Requirements

- **MAP-001**: The engine shall load mapping configs from YAML files at API boot. Schema validation against a JSON-schema artefact must succeed, or boot aborts with a structured error naming the offending file and rule.
- **MAP-002**: Every operator referenced in a config must exist in the operator registry at boot time. Unknown operator names abort boot.
- **MAP-003**: Each mapping config shall declare `schema_version` (e.g. `mapping.xero-export.v1`). Multiple versions of the same (system, direction) may coexist; the engine selects the version requested by the caller (for current runs) or frozen with a historical run (for audit replay).
- **MAP-004**: Operators shall declare their context-layer requirements in the config. The engine shall plan buffers and pre-passes based on the union of declared requirements across all operators in a run.
- **MAP-005**: Layer 1 (back-window) operators shall receive a bounded ring buffer of at most N prior source records, where N is declared in the config. Memory cost is bounded by declaration.
- **MAP-006**: Layer 3 (cross-entity lookup) operators shall receive a typed `LookupContext` providing keyed access to related entities. The engine shall batch lookup keys across rows where the backing `LookupSource` supports batching (e.g. DB-backed sources).
- **MAP-007**: Layer 4 (aggregates) operators shall receive pre-computed aggregate snapshots (counts, sums, min, max, percentile as declared). Aggregates are computed in a single pre-pass before the main streaming pipeline begins.
- **MAP-008**: The engine shall NOT support Layer 5 (whole dataset) in v1. A config that declares it shall fail schema validation at boot.
- **MAP-009**: The engine shall provide hooks for CEL expressions within operator invocations, delegating to the existing `rules.Service` (A-0011). CEL expressions must be bounded — compile-time-checked, timeout-protected, and unable to perform arbitrary I/O.
- **MAP-010**: Every mapping execution shall emit structured logs at config-load, run-start, per-row-error, and run-end — with sufficient context (config schema version, operator name, row index, field name) to diagnose failures without reproducing the run.
- **MAP-011**: Mapping configs are platform-owner-editable only. The viewer API shall enforce the role requirement.
- **MAP-012**: The platform-owner mapping viewer web page shall render a config as a table with one row per target field: source column(s), operator chain with params, target column. v1 is read-only. The page shall expose the config's `schema_version` and file path for traceability.
- **MAP-013**: The engine shall be bidirectional by design — import configs (external → Ledgius) and export configs (Ledgius → external) share the same engine, operator registry, and viewer. Only the direction of data flow differs.
- **MAP-014**: Errors produced by an operator at a specific row shall be attributable to (config file, operator name, source row id, target field). The engine shall surface these errors to callers with full context so callers (e.g. export orchestrator) can report per-row failures.
- **MAP-015**: Adding a new operator to the registry shall not require changes to the engine core — only a new file implementing the typed operator interface plus registration during package init.

# Non-functional Requirements

- **Correctness**: schema validation at boot catches every misconfiguration. No config can reach runtime that hasn't passed validation.
- **Observability**: every operator invocation is traceable via structured logs; operator latency and error rates are exposed as metrics tagged with config schema version.
- **Performance**: for a typical export of 100k invoice lines × ~20 fields (~2M operator invocations), throughput must be ≥ 50k invocations/second on reference hardware. Layer 0 streaming, no reflection, pre-registered typed function pointers.
- **Extensibility**: the extension checklist for a new external system is — (1) write YAML configs, (2) implement target-specific operators (none required for simple field-copy cases), (3) register operators, (4) add config files to the load path. No changes to engine core.
- **Determinism**: the same input + same config + same lookup context must produce bit-identical output across runs. This is required for audit-replay.
- **Safety**: CEL expressions (Layer 4.5 or embedded in operator params) are sandboxed — no arbitrary I/O, compile-time-checked, timeout-protected, reference-only access to the row and context.

# Acceptance Criteria

- [ ] A new operator can be added by adding a Go file with a typed implementation + init registration; no changes to engine core needed.
- [ ] A new external system's export mapping can be added as YAML + per-system operator Go files; no changes to engine core needed.
- [ ] Boot aborts with a clear structured error when a YAML config references an unknown operator.
- [ ] Boot aborts with a clear structured error when a YAML config fails JSON-schema validation.
- [ ] A mapping config with back-window requirements correctly feeds the declared buffer to the operator, and the operator produces the expected grouped output.
- [ ] A cross-entity lookup in a mapping config resolves correctly with batched queries (verified via query log).
- [ ] An aggregate-declaring operator receives correct pre-computed values.
- [ ] The platform-owner mapping viewer renders every field in a config with source, operator chain, and target — no fields missing, no gibberish.
- [ ] Running the same export twice against an unchanged source dataset with an unchanged config produces byte-identical target output.
- [ ] An unknown operator name in YAML produces a failure at boot, not at run time.
- [ ] CEL expressions within operator params can access the current source row and declared lookup context, and reject arbitrary I/O attempts (the rules.Service sandbox).

# Open Questions

- Should operators be able to emit multiple target fields from one source field (fan-out), or stay one-target-per-invocation? Keeping one-to-one keeps the viewer simple; fan-out handles "split address into line1/line2/suburb/postcode" more cleanly. Proposal: operators are one-to-one; multi-field outputs are expressed as multiple operator invocations sharing the same source.
- Should mapping configs live in `ledgius-db` (as seed data under `fixtures/mappings/`), in `ledgius-api` (embedded via `//go:embed`), or in their own repo? Proposal: embedded in `ledgius-api` under `internal/mapping/configs/` — deployed with the API, version-controlled alongside operators, no cross-repo coordination.
- Should the viewer support rendering two versions side-by-side (diff view) for when a `v1 → v2` schema change lands? Out of scope for v1 but the UI shape should accommodate it.
