# Terms

- **Leave application**: An employee-initiated request to take leave on a specified date range, against
  a specified leave type, with optional notes. Before approval, it is a *pending request*; after approval
  it becomes a *scheduled leave block* that the next pay run pre-fills into the employee's leave-entry row
  (see R-0055).
- **Approver**: The user (typically the business owner or a delegated manager) who sees pending requests,
  approves or rejects them. Approvals are an audit event.

# Summary

Mobile workflow for employees to request leave from their phone, and for the employer to approve from
either the mobile or web app. Approved leave auto-populates the next pay run's leave entry (R-0055) so
the bookkeeper doesn't re-key it.

# Problem / Business Context

Today an employee who wants to take leave must tell their manager by whatever channel (email, Slack,
verbal), and the manager must remember to tell the bookkeeper, who then remembers to record it in the
next pay run's hours. This is lossy, undocumented, and easy to forget — leading to pay mistakes (full
ordinary hours paid when leave was taken, no corresponding leave-balance deduction) and STP Phase 2
category misreporting.

An in-app leave-application workflow creates a durable record, routes it to an approver, and flows
directly into payroll so there's no re-keying and no missed leave.

# Scope

- Mobile (React Native) screen: *Apply for Leave* — pick type, date range, optional note, submit
- Mobile screen: *My Leave Requests* — see pending/approved/rejected history, see current leave balances
- Web admin screen: *Leave Requests* — approver sees pending requests, can approve or reject with note
- Web: approved leave appears pre-filled on the next pay run processing page (R-0055 leave-entry rows)
- Push notifications: employee gets notified when request is approved/rejected; approver gets notified of
  new requests
- API endpoints: `GET/POST /leave-requests`, `POST /leave-requests/:id/approve`, `POST /leave-requests/:id/reject`

# Out of Scope

- Leave calendar visualisation for the whole team (future)
- Complex multi-level approval chains (single-approver only in v1)
- Integration with shift-rostering systems
- Auto-approval based on rules (all requests require manual approval in v1)

# Actors / Users

- **Employee** — submits leave request from mobile; views status and balance
- **Approver** (business owner or delegated manager) — approves/rejects from web or mobile
- **Bookkeeper** — sees approved requests auto-populated when processing pay run

# Preconditions

- R-0003 operational — employees have leave balances
- R-0055 operational — pay run processing supports per-employee leave entry
- Employees have mobile app accounts linked to their employee record
- Each employee record has a designated approver (employer_user_id)

# Functional Requirements

- **LEA-APP-001**: Employee shall submit a leave request specifying leave_type, start_date, end_date,
  and optional note from the mobile app
- **LEA-APP-002**: System shall validate the request against current leave balance; if insufficient balance,
  request is still submittable but displays a warning to both employee and approver
- **LEA-APP-003**: Approver shall see pending requests on the web admin screen and on mobile
- **LEA-APP-004**: Approver shall approve or reject each request with an optional reason note
- **LEA-APP-005**: System shall notify the employee via push notification on approval/rejection
- **LEA-APP-006**: On approval, system shall create a `leave_booking` record spanning the date range
- **LEA-APP-007**: When processing a pay run whose period overlaps an approved `leave_booking`, the
  system shall pre-fill the employee's leave-entry row with the overlap hours (bookkeeper can still edit)
- **LEA-APP-008**: Audit log shall capture: request submitted, approved/rejected (with approver user,
  timestamp, reason note)

# UX / UI Behaviour

**Mobile — Apply for Leave**:
- Leave type picker (segmented control or picker)
- Date range picker (from/to dates, single-day supported)
- Hours computed and displayed — editable for part-day leave
- Current leave balance shown prominently with remaining-after indicator
- Submit button disabled until required fields complete

**Web — Leave Requests admin**:
- List view: pending at top, then approved (collapsed), then rejected (collapsed)
- Row shows: employee, leave type, dates, hours, balance-before/after, requested-at, note
- Actions per row: Approve / Reject (reject prompts for reason)
- Filter by status, employee

**Web — Pay Run processing pre-fill**:
- When a pay run period overlaps `LeaveBooking.start_date..end_date` for a selected employee, the
  leave-entry row is automatically added with the overlap hours
- UI indicates "from leave request LR-####" so bookkeeper sees why it's there
- Bookkeeper can adjust or delete; the `leave_booking.consumed` flag is set on post

# Validation Rules

- start_date ≤ end_date
- Date range cannot overlap another approved leave request for the same employee
- Leave type must be one of the configured types
- Rejections require a note; approvals optional

# API Surface

- `GET /leave-requests` — current user sees own (employee) or all-for-tenant (approver)
- `POST /leave-requests` — submit new request (employee)
- `POST /leave-requests/:id/approve` — approver action
- `POST /leave-requests/:id/reject` — approver action
- `POST /leave-requests/:id/cancel` — employee can cancel pending own
- `GET /leave-bookings?period_start=&period_end=&employee_id=` — used by pay-run pre-fill logic

# Security / Permissions

- Employees can only see and create their own leave requests
- Approvers see all requests for employees they're designated approver for
- Cancellation: employee can cancel their own *pending* requests; only approvers can cancel approved
  bookings before they're consumed

# Success Criteria

- > 80% of leave events for active employees flow through this workflow (vs ad-hoc email/verbal)
  within 3 months of rollout
- Zero pay-run mistakes where leave was taken but not recorded, for employees using the workflow
- STP Phase 2 leave-category reporting 100% accurate for leave captured via this workflow

# Open Questions

- Q1: Should mobile support approval (approver on phone) as a v1 feature, or web-only initially?
  **Recommendation**: web-only for v1 — approvers are typically at a desk; mobile approval in v2.
- Q2: Can an employee be their own approver? **Recommendation**: no — force delegation. Owner-operators
  with no staff don't need this workflow (they just enter leave directly at pay-run time per R-0055).
- Q3: What happens to approved leave if the employee leaves before taking it? **Recommendation**: on
  termination, cancel unused future bookings; any consumed (already taken) leave stays as-is.
