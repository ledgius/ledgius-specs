# Summary

Technical architecture for the mileage and vehicle logbook feature. Covers database schema, API design, GPS tracking implementation on mobile (expo-location), trip detection algorithm, ATO logbook period management, PDF report generation, and expense ledger integration.

# Database Schema

CREATE TABLE vehicle (

    make                TEXT,

    year                INT,

COMMENT ON TABLE trip_log IS 'Individual trip records. Each trip is either business or personal. GPS route stored as JSONB array of {lat, lng, ts} points.';

CREATE INDEX idx_trip_log_type ON trip_log(trip_type);
CREATE INDEX idx_trip_log_period ON trip_log(logbook_period_id) WHERE logbook_period_id IS NOT NULL;

CREATE TABLE logbook_period (
    id                  SERIAL PRIMARY KEY,

COMMENT ON COLUMN logbook_period.business_use_pct IS 'Calculated: total_business_km / total_km * 100. Applied to vehicle running costs.';
COMMENT ON COLUMN logbook_period.valid_until IS 'ATO allows logbook to be used for up to 5 years. Default: start_date + 5 years.';

# GPS Auto-Tracking (Mobile)

## Technology

- **Package**: `expo-location` with `TaskManager` for background location updates
- **Permission**: `ACCESS_FINE_LOCATION` (Android), `Always` or `When In Use` (iOS)
- **Background mode**: `expo-task-manager` registered background task for location updates

## Trip Detection Algorithm

```
State machine: IDLE → MOVING → TRIP_ACTIVE → TRIP_ENDING → IDLE

IDLE:
  - Monitor significant location changes (distanceFilter: 100m)
  - When speed > 10 km/h for 2 consecutive readings → transition to MOVING

MOVING:
  - Switch to fine tracking (distanceFilter: 50m, timeInterval: 30s)
  - Start recording GPS breadcrumbs [{lat, lng, timestamp}]
  - Record trip start time and location
  - After 500m of movement → transition to TRIP_ACTIVE

TRIP_ACTIVE:
  - Continue fine tracking
  - Accumulate distance from GPS path segments (haversine formula)
  - Update "active trip" banner in app UI

TRIP_ENDING:
  - When speed < 5 km/h for > 3 minutes → start end timer
  - If movement resumes within 5 minutes → back to TRIP_ACTIVE (traffic stop / red light)
  - If stationary for > 5 minutes → trip complete
  - Save trip to local storage
  - Transition to IDLE

Battery management:
  - Normal: distanceFilter 50m, timeInterval 30s
  - Battery < 30%: distanceFilter 200m, timeInterval 60s
  - Battery < 15%: pause GPS, show notification "GPS tracking paused — low battery"
```

## Distance Calculation

```typescript
// Haversine formula between consecutive GPS points
function haversineDistance(lat1, lng1, lat2, lng2): number {
  const R = 6371 // Earth radius in km
  const dLat = toRad(lat2 - lat1)
  const dLng = toRad(lng2 - lng1)
  const a = sin(dLat/2)² + cos(lat1) * cos(lat2) * sin(dLng/2)²
  return R * 2 * atan2(sqrt(a), sqrt(1-a))
}

// Total trip distance = sum of haversine segments
// Round to 1 decimal place for display, full precision for storage
```

# Auto-Classification Logic

```
Default rules (configurable per user):
1. Monday–Friday, 6:00 AM – 7:00 PM → "business"
2. Saturday–Sunday → "personal"
3. Public holidays (AU) → "personal"
4. Trip from saved "Home" location to saved "Office" location → "business" (commute)
5. Trip starting or ending at a saved "Client" location → "business"
6. All other trips → "business" during work hours, "personal" outside

Users can override any trip and adjust the default rules.
```

# Logbook Period Management

```
Start logbook period:
  1. Validate no active period exists for this vehicle
  2. Set start_date = today, end_date = today + 83 days (84 day period = 12 weeks)
  3. Set valid_until = start_date + 5 years (ATO maximum)
  4. All trips logged for this vehicle during the period are linked via logbook_period_id

During period:
  - Dashboard shows: days elapsed, days remaining, daily trip count heatmap
  - Alert if any day has zero trips logged (ATO requires completeness)
  - Running totals: business km, personal km, current business-use %

Complete period:
  1. Validate all 84 days have at least one trip logged (warn if gaps)
  2. Calculate: total_business_km, total_personal_km, total_km
  3. business_use_pct = total_business_km / total_km * 100
  4. Lock the period (no further edits to trips within it)
  5. Generate logbook PDF

Apply to expenses:
  - Business-use % applied to all vehicle running costs (fuel, rego, insurance, repairs)
  - E.g. if business_use_pct = 72%, then 72% of a $150 service bill is deductible
  - Create/update expense journal entries with the adjusted amounts
```

# Expense Integration

```
Two paths for expense creation:

1. Logbook method:
   - When vehicle running costs are recorded (fuel, rego, insurance, service):
     expense_amount = cost × business_use_pct
   - GL entry: debit Motor Vehicle Expenses, credit Bank/AP
   - GST credit claimed on the business portion

2. Cents-per-km method:
   - At end of FY (or on demand):
     deduction = min(total_business_km, 5000) × rate
   - GL entry: debit Motor Vehicle Expenses, credit Owner's Equity/Drawings
   - No GST credit (cents-per-km is an all-inclusive rate)
```

# Risks / Trade-offs

- **GPS battery drain**: background location tracking can drain battery. Mitigated by adaptive frequency (reduce when battery low) and clear user opt-in.
- **GPS accuracy**: urban canyons, tunnels, and multi-storey car parks reduce GPS accuracy. Distance calculation from odometer readings is more reliable — GPS is a convenience, not the source of truth.
- **Background task limits**: iOS restricts background location updates. `expo-location` handles this with `startLocationUpdatesAsync` + `TaskManager`, but Apple may reject apps that track location without clear justification. The mileage/logbook use case is well-established and accepted.
- **12-week completeness**: the ATO requires every trip (business AND personal) to be logged during the logbook period. Users may forget. Mitigated by daily push notification reminders during an active logbook period.
- **Privacy**: GPS route data is sensitive. Stored on-device first, uploaded only on user action. Clear privacy notice on opt-in. GPS data is never shared with third parties or used for any purpose other than the user's own mileage records.

# Future: Google Maps Route Display (Phase 9)

Display recorded GPS routes on an interactive map in the trip detail view.

**Technology:**
- `react-native-maps` — Google Maps on Android, Apple Maps on iOS
- Requires Google Maps Platform API key (Maps SDK for Android + iOS)
- Google Maps pricing: first $200/month free (~28,000 map loads), then pay-per-use

**Scope:**
- Trip detail screen: show recorded GPS breadcrumbs as a polyline on the map
- Start/end markers with location labels
- Route replay animation (optional)
- Does NOT use Google Maps for navigation or route calculation — only for display of already-recorded GPS paths

**Why deferred:**
- GPS auto-tracking works without maps — raw coordinates are sufficient for distance calculation and ATO compliance
- Adding Google Maps introduces an external dependency and API key cost
- The core value (accurate km tracking for tax deductions) doesn't require a visual map

# Future: Saved Locations & Geofence Classification (Phase 10)

Learn and store frequently visited locations to auto-fill from/to fields and auto-classify trips.

**Scope:**
- Save locations: "Home", "Office", "Client: Acme", "Supplier: Bunnings"
- Auto-detect: when a trip starts/ends near a saved location, auto-fill the label
- Geofence classification: trips to/from saved client locations auto-classified as "business"
- Commute detection: Home ↔ Office trips flagged as commute (business or personal per user config)

**Why deferred:**
- Manual entry and time-based auto-classification cover the majority of use cases
- Geofencing adds battery overhead and privacy complexity
- Requires enough trip history to learn patterns reliably
