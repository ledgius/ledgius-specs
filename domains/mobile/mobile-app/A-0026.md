# Summary

Technical architecture for the Ledgius iOS and Android mobile apps built with React Native / Expo. Covers app structure, offline-first data layer, API integration, Stripe payment flows, camera/OCR pipeline, and deployment.

# Tech Stack

```
React Native 0.79+ (via Expo SDK 53+)
TypeScript
Expo Router (file-based navigation)
Expo Camera (receipt capture)
Expo SQLite (offline storage)
TanStack Query (API data fetching, same as web app)
Stripe React Native SDK (payments)
Nativewind (Tailwind CSS for React Native — matches web app styles)
```

// Tables mirroring API resources

## Sync queue

```typescript
interface SyncQueueItem {
  id: string
  entity_type: "quote" | "invoice" | "contact" | "receipt_capture"
  operation: "create" | "update" | "delete"
  payload: string  // JSON
  created_at: string
  attempts: number
  last_error?: string
  status: "pending" | "syncing" | "failed"
}
```

Sync runs:
- On app foreground
- On network state change (offline → online)
- On manual pull-to-refresh
- Every 60 seconds while online

Conflict resolution:
- Server wins for single-value fields
- Timestamps compared — most recent wins
- Unsynced local changes shown with "pending sync" badge

# Stripe Payment Architecture

## Stripe Connect model

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│   Customer   │────▶│    Stripe    │────▶│  Ledgius User │
│  (pays)      │     │  (processes) │     │  (receives)   │
└──────────────┘     └──────────────┘     └──────────────┘
```

### Option A: Stripe Connect Express (default)
- Ledgius is the platform
- User onboards via Stripe Connect Express (hosted onboarding)
- Payments flow: Customer → Stripe → Ledgius platform → User's bank
- Ledgius takes application fee (e.g. 0.5%)
- User doesn't manage Stripe directly

### Option B: Stripe Connect Standard
- User connects their existing Stripe account
- OAuth flow: user authorises Ledgius to create charges on their behalf
- Payments go directly to user's Stripe account
- No Ledgius platform fee (user pays Stripe fees directly)

## Payment link flow

```
1. User creates invoice on mobile
2. Taps "Send with payment link"
3. API generates Stripe Payment Link for the invoice amount
4. Invoice email includes the payment link + QR code
5. Customer clicks link → Stripe Checkout page
6. Customer pays → Stripe webhook → API updates invoice status
7. Push notification to mobile: "Invoice #INV-0042 paid — $373.92"
8. Receipt auto-generated and sent to customer
```

## QR code payment

```
1. User opens invoice on mobile
2. Taps "Show QR Code"
3. QR code displayed on screen (encodes payment link URL)
4. Customer scans with their phone camera
5. Opens Stripe Checkout in their browser
6. Same webhook flow as payment link
```

# Receipt Capture Pipeline

```
Camera → Auto-crop → Compress → Upload → OCR → Store

1. Expo Camera captures photo
2. On-device: auto-detect edges, crop, straighten (expo-image-manipulator)
3. Compress to JPEG (quality 80, max 2000px width)
4. Upload to API: POST /receipt-captures with base64 image
5. API stores image in cloud storage (Fly.io Tigris/S3)
6. OCR extraction (on-device via ML Kit or server-side via Google Vision API):
   - Merchant name
   - Date
   - Total amount
   - GST amount
   - ABN (if visible)
7. Extracted data stored alongside the image
8. Available in web app for expense matching
```

-- Quotes (extends the transaction model)
CREATE TABLE quote (
    id              SERIAL PRIMARY KEY,
    quote_number    TEXT NOT NULL,
    entity_credit_account INT NOT NULL,
    quote_date      DATE NOT NULL DEFAULT CURRENT_DATE,
    valid_until     DATE,
    status          TEXT NOT NULL DEFAULT 'draft'

CREATE TABLE quote_line (
    id              SERIAL PRIMARY KEY,

    description     TEXT NOT NULL,
    quantity        NUMERIC NOT NULL DEFAULT 1,
    unit_price      NUMERIC NOT NULL,
    tax_code_id     INT,
    line_total      NUMERIC NOT NULL,
    sort_order      INT NOT NULL DEFAULT 0

-- Receipt captures
CREATE TABLE receipt_capture (
    id              SERIAL PRIMARY KEY,
    image_url       TEXT NOT NULL,
    thumbnail_url   TEXT,
    ocr_merchant    TEXT,
    ocr_date        DATE,
    ocr_amount      NUMERIC,
    ocr_gst         NUMERIC,
    ocr_abn         TEXT,
    category        TEXT,
    matched_transaction_id INT,
    status          TEXT NOT NULL DEFAULT 'unmatched'
        CHECK (status IN ('unmatched', 'matched', 'ignored')),
    captured_by     TEXT NOT NULL,
    captured_at     TIMESTAMPTZ NOT NULL DEFAULT now(),
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);

# Risks / Trade-offs

- **React Native performance**: camera and NFC may need native modules. Mitigate with Expo's prebuild for native access.
- **Offline complexity**: sync conflicts are hard. Start with simple "server wins" and evolve.
- **OCR accuracy**: on-device ML Kit is fast but less accurate than cloud APIs. Start with cloud, move to on-device for cost.
- **Stripe Connect compliance**: requires careful handling of KYC, settlements, disputes. Stripe handles most of this.
- **Two app stores**: review processes differ. Budget 1-2 weeks for initial approval.
