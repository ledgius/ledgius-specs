# Summary

Technical architecture for the Ledgius iOS and Android mobile apps built with React Native / Expo. Covers app structure, offline-first data layer, API integration, Stripe payment flows, camera/OCR pipeline, and deployment.

# Tech Stack

```
React Native (via Expo SDK 54)
TypeScript
Expo Router (file-based navigation)
expo-image-picker (receipt capture — launches native camera, reliable in Expo Go)
expo-file-system (read images as base64 for upload)
expo-secure-store (encrypted JWT token storage)
expo-navigation-bar (Android system nav bar theming)
expo-status-bar (status bar theming)
TanStack Query (API data fetching, same as web app)
Stripe React Native SDK (payments — planned Phase 6)
StyleSheet (React Native built-in — not Nativewind)
```

**Note**: The app uses React Native `StyleSheet.create()` with a shared `constants/Colors.ts` design system (brand colours, spacing, radius, fontSize). This was chosen over Nativewind for simplicity and to avoid a Tailwind runtime dependency on mobile.

## Sync queue

```typescript
interface SyncQueueItem {
  id: string
  entity_type: "quote" | "invoice" | "contact" | "receipt_capture"
  operation: "create" | "update" | "delete"
  payload: string  // JSON
  created_at: string
  attempts: number
  last_error?: string
  status: "pending" | "syncing" | "failed"
}
```

Sync runs:
- On app foreground
- On network state change (offline → online)
- On manual pull-to-refresh
- Every 60 seconds while online

Conflict resolution:
- Server wins for single-value fields
- Timestamps compared — most recent wins
- Unsynced local changes shown with "pending sync" badge

# Stripe Payment Architecture

## Stripe Connect model

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│   Customer   │────▶│    Stripe    │────▶│  Ledgius User │
│  (pays)      │     │  (processes) │     │  (receives)   │
└──────────────┘     └──────────────┘     └──────────────┘
```

### Option A: Stripe Connect Express (default)
- Ledgius is the platform
- User onboards via Stripe Connect Express (hosted onboarding)
- Payments flow: Customer → Stripe → Ledgius platform → User's bank
- Ledgius takes application fee (e.g. 0.5%)
- User doesn't manage Stripe directly

### Option B: Stripe Connect Standard
- User connects their existing Stripe account
- OAuth flow: user authorises Ledgius to create charges on their behalf
- Payments go directly to user's Stripe account
- No Ledgius platform fee (user pays Stripe fees directly)

## Payment link flow

```
1. User creates invoice on mobile
2. Taps "Send with payment link"
3. API generates Stripe Payment Link for the invoice amount
4. Invoice email includes the payment link + QR code
5. Customer clicks link → Stripe Checkout page
6. Customer pays → Stripe webhook → API updates invoice status
7. Push notification to mobile: "Invoice #INV-0042 paid — $373.92"
8. Receipt auto-generated and sent to customer
```

## QR code payment

```
1. User opens invoice on mobile
2. Taps "Show QR Code"
3. QR code displayed on screen (encodes payment link URL)
4. Customer scans with their phone camera
5. Opens Stripe Checkout in their browser
6. Same webhook flow as payment link
```

# Receipt Capture Pipeline

```
Camera/Gallery → Upload (base64) → Compress → Save to disk → OCR → Generate PDF → Store metadata
```

## Detailed flow

1. **Capture**: `expo-image-picker` launches native camera (or gallery picker). Returns local URI.
2. **Read**: `expo-file-system` reads the image as base64 from the local URI.
3. **Upload**: `POST /captures` with `{ image_data: "<base64>", file_name, mime_type }`. Response returns immediately with `status: "uploaded"`.
4. **Server-side compression**: Image resized to max 1200px wide, re-encoded as JPEG quality 70 using `CatmullRom` (bicubic) interpolation via `golang.org/x/image/draw`. Typical reduction: 2-3MB → 100-200KB.
5. **Thumbnail**: 150px wide JPEG at quality 40 (~10KB), stored as base64 in DB for fast list rendering.
6. **Save to disk**: Compressed image saved to Fly volume at `/data/{tenant_slug}/{fy}/receipts/{id}.jpg`.
7. **OCR (async goroutine)**: Status transitions to `processing`. Tesseract OCR extracts text. Heuristic parser identifies:
   - **Vendor name**: first non-numeric, non-date, non-boilerplate line (skips ABN, TAX, GST prefixes)
   - **Total amount**: regex matches `TOTAL` lines first, falls back to largest `$X.XX` on any line
   - **Date**: Australian DD/MM/YYYY, also ISO YYYY-MM-DD
   - **Raw text**: full OCR output preserved for debugging
8. **PDF generation**: `go-pdf/fpdf` creates a branded PDF:
   - Page 1: Ledgius header, extracted details table (vendor/amount/date), embedded receipt image
   - Page 2: Full OCR raw text (searchable in any PDF viewer)
   - Saved to `/data/{tenant_slug}/{fy}/receipts/{id}.pdf`
9. **DB update**: OCR fields + file paths written to `captured_receipt` row. Status → `ready`.
10. **Web app**: "Captured Receipts" page under Banking. Inline editing for vendor/amount/date corrections.

## OCR Technology Decision

**Chosen: Tesseract OCR** (open-source, Apache 2.0)
- Repository: `github.com/tesseract-ocr/tesseract` (62k+ GitHub stars)
- Go wrapper: `github.com/otiai10/gosseract/v2`
- Runs locally inside the API Docker container — zero external API calls, zero cost
- English language data (`tesseract-ocr-eng`) bundled in the container (~15MB)
- Accuracy: very good for printed text (receipts, invoices). Adequate for receipt capture use case.
- Trade-off: less accurate than Google Cloud Vision for handwriting or damaged documents, but free and private.

**Local development dependency:**
- macOS: `brew install tesseract` (installs to `/opt/homebrew`)
- Requires CGO flags for Go compilation on macOS: `CGO_CFLAGS="-I$(brew --prefix tesseract)/include -I$(brew --prefix leptonica)/include"` and `CGO_LDFLAGS="-L$(brew --prefix tesseract)/lib -L$(brew --prefix leptonica)/lib"`
- These flags are configured in `ledgius-infra/Makefile` for `make build` and `make dev`
- Docker (Fly.io): `libtesseract-dev` + `libleptonica-dev` in builder stage, `tesseract-ocr` + `tesseract-ocr-eng` in runtime stage — standard Debian paths, no custom flags needed

**Rejected alternatives:**
- Google Cloud Vision API — excellent accuracy but requires API key + billing account + per-request cost
- On-device ML Kit — fast but React Native integration is complex and accuracy is lower
- Claude Vision API — excellent for structured extraction but adds external dependency + cost

## File Storage Architecture

```
/data/                              — Fly volume mount point (1GB per machine)
  {tenant_slug}/                    — Tenant directory (e.g. acme_cons_df59a/)
    fy26/                           — Fiscal year (Australian Jul-Jun)
      receipts/
        42.jpg                      — Compressed receipt image
        42.pdf                      — Generated PDF (image + OCR text)
        43.jpg
        43.pdf
    fy27/
      receipts/
        ...
```

**Design decisions:**
- **Fly volume** over S3/Tigris: simpler, no additional service, persistent across deploys. Can migrate to object storage later if volume size becomes a constraint.
- **FY partitioning**: keeps directories bounded, aligns with BAS/tax year, natural archival boundary.
- **Thumbnail in DB**: 10KB base64 in the `captured_receipt` row avoids a disk read per row when rendering lists. Full image read from disk only on detail view.
- **Not in DB**: full images and PDFs are NOT stored in PostgreSQL. Only file paths are stored. This avoids bloating the DB with binary data.

## Tenant DB — V1.22: Captured Receipts

```sql
CREATE TABLE captured_receipt (
    id                  SERIAL PRIMARY KEY,
    image_data          TEXT,                           -- legacy nullable; new uploads use image_path
    image_path          TEXT,                           -- e.g. /data/acme_cons_df59a/fy26/receipts/42.jpg
    pdf_path            TEXT,                           -- e.g. /data/acme_cons_df59a/fy26/receipts/42.pdf
    thumbnail_data      TEXT,                           -- base64 ~10KB, stays in DB for list perf
    file_name           TEXT NOT NULL DEFAULT 'receipt.jpg',
    mime_type           TEXT NOT NULL DEFAULT 'image/jpeg',
    file_size_bytes     INT NOT NULL DEFAULT 0,

    -- OCR-extracted fields
    ocr_vendor          TEXT,
    ocr_amount          NUMERIC,
    ocr_date            DATE,
    ocr_currency        TEXT DEFAULT 'AUD',
    ocr_description     TEXT,
    ocr_raw_text        TEXT,

    -- User-corrected fields (override OCR)
    vendor              TEXT,
    amount              NUMERIC,
    receipt_date        DATE,
    currency            TEXT DEFAULT 'AUD',
    description         TEXT,
    category            TEXT,

    -- Linking
    bank_transaction_id INT,
    expense_account_id  INT,

    status              TEXT NOT NULL DEFAULT 'uploaded'
        CHECK (status IN ('uploaded', 'processing', 'ready', 'matched', 'archived')),
    notes               TEXT,
    uploaded_by         TEXT,
    created_at          TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at          TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

# UX Theme

The mobile app uses a consistent dark teal (`#0B6E99` — `brand.primaryDeep`) theme:

- **Status bar**: dark teal background, white text/icons (`StatusBar style="light"`)
- **Header bar**: dark teal background, white title, white back arrow
- **Tab bar**: dark teal background, white active icons, semi-transparent inactive icons, subtle white highlight on active tab
- **Android system nav bar**: dark teal background, light button style (white back/home/recents)
- **Login screen**: full dark teal background with white form elements

This ensures all white system UI elements (clock, battery, signal, navigation buttons) are visible against the dark background.

# Risks / Trade-offs

- **React Native performance**: camera via `expo-image-picker` (native camera app) is reliable. Direct camera view (`expo-camera`) crashes in Expo Go on some Android devices — avoided.
- **Android autofill overlay**: green background on login inputs. Mitigated by wrapping `TextInput` in a `View` with the visible background and making the input itself transparent.
- **Offline complexity**: sync conflicts are hard. Start with simple "server wins" and evolve. Currently requires connectivity.
- **OCR accuracy**: Tesseract is good for printed receipts but weaker on handwriting or low-contrast thermal prints. Adequate for the core use case. Can upgrade to cloud API later if needed.
- **Stripe Connect compliance**: requires careful handling of KYC, settlements, disputes. Stripe handles most of this.
- **Two app stores**: review processes differ. Budget 1-2 weeks for initial approval.
- **Fly volume limitation**: volumes are per-machine, not shared. With 2 machines, each has its own volume. Receipts uploaded to machine A aren't visible to machine B. Acceptable for now with `min_machines_running = 1`. Future: migrate to Tigris/S3 for shared storage.
