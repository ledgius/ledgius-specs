# Summary

Native iOS and Android mobile apps for Ledgius providing field/on-the-go functionality: create quotes, create and send invoices, accept payments, capture receipt photos, and manage contacts. The apps connect to the existing Ledgius API (`api.ledgius.com`) and support Stripe payment processing — either via Ledgius's master Stripe account (Stripe Connect) or the user's own Stripe account.

# Business Context

Tradespeople, sole traders, and field-based small business owners need to:
- Quote a job on-site and send it to the customer immediately
- Invoice the customer as soon as the job is done
- Accept payment on the spot (tap-to-pay, payment link, or QR code)
- Photograph receipts for expenses before they lose them
- Add new customers/suppliers while in the field

These are high-frequency, time-sensitive actions that can't wait until the bookkeeper is at a desk. The mobile app is NOT a full accounting app — it's a field companion for the web app.

# Core Principle: Field-First, Sync Later

The mobile app prioritises speed and reliability in the field:
- **Offline-capable**: quotes, invoices, and receipt captures work without internet
- **Sync when connected**: data syncs to the API when connectivity is available
- **Minimal input**: large buttons, smart defaults, auto-fill from recent data
- **Camera-first**: receipt capture is one tap from the home screen

# Acceptance Criteria

## AC-1: Create Quotes ✅ Implemented

Create, edit, and send quotes from the mobile device:

- **Quick create**: customer selector (modal with search), line items (description + quantity + price + tax code chip selector), notes, quote date, valid-until date
- **Edit**: draft quotes can be fully edited — customer, dates, line items, notes. API enforces draft-only editing.
- **Tax**: GST auto-calculated per line item. Tax codes resolved by name (e.g. "GST", "FRE") — the API maps to `tax_code_id` via lookup.
- **Totals**: subtotal, GST, total displayed in real-time (calculated client-side)
- **Send with email override**: slide-up modal with email input — pre-fills contact email, user can override to send to a different address. API delivers branded HTML email via Resend with line items table, totals, and notes.
- **Status**: draft → sent → accepted → rejected → expired → converted
- **Convert**: one-tap convert accepted quote to invoice (creates AR transaction with GL journal entries)
- **Quote numbers**: auto-generated sequential format `QTE-0001`
- **Offline**: create and save locally, send when connected (planned — currently requires connectivity)

## AC-2: Create Invoices

Create, send, and track invoices:

- **Quick create**: same flow as quotes but creates an invoice (AR transaction)
- **From quote**: convert an accepted quote to an invoice with one tap
- **Due date**: auto-calculated from customer payment terms
- **Invoice number**: auto-generated, sequential per tenant
- **Send**: email invoice to customer as branded PDF with payment link
- **Payment link**: embedded Stripe payment link in the email and on the invoice PDF
- **Status tracking**: draft → sent → viewed → paid → overdue
- **Push notifications**: notify when customer views or pays the invoice
- **Offline**: create and save locally, send when connected

## AC-3: Provide Receipt for Payments

Generate and send payment receipts:

- **Auto-generated**: when a payment is recorded (manually or via Stripe), a receipt is created automatically
- **PDF receipt**: branded receipt with payment details, GST breakdown, business ABN
- **Send**: email or SMS receipt to customer
- **Share**: native share sheet (AirDrop, Messages, WhatsApp, email)
- **Print**: support AirPrint / Bluetooth receipt printers for tradies
- **Receipt number**: auto-generated, sequential

## AC-4: Capture Receipts ✅ Implemented

Photograph expense receipts, OCR-extract data, generate searchable PDFs:

- **Camera capture**: dedicated Capture tab with camera button. Uses `expo-image-picker` (launches native camera app — reliable across all devices including Expo Go).
- **Gallery import**: secondary button to pick existing photos from the device gallery.
- **Image compression**: uploaded images resized to max 1200px wide, re-encoded as JPEG quality 70 using `CatmullRom` interpolation. Typical 2-3MB phone photo → 100-200KB.
- **Auto-upload**: photos upload to API immediately after capture with progress indicator per photo. Base64 JSON payload.
- **OCR extraction**: server-side via **Tesseract OCR** (open-source, `github.com/tesseract-ocr/tesseract`, Apache 2.0 licence). Runs locally in the API container — zero external API calls, zero cost. Extracts:
  - **Vendor name**: first meaningful non-numeric, non-date line
  - **Total amount**: TOTAL line detection, fallback to largest dollar amount
  - **Date**: Australian DD/MM/YYYY format, also ISO YYYY-MM-DD
  - **Raw OCR text**: full text output preserved for debugging and re-extraction
- **PDF generation**: each receipt auto-generates a branded PDF containing:
  - Page 1: Ledgius header, extracted details (vendor/amount/date), embedded receipt image
  - Page 2: Full OCR raw text (searchable, archivable)
  - Generated using `go-pdf/fpdf` (pure Go, no external dependencies)
- **Status lifecycle**: uploaded → processing (OCR running) → ready → matched → archived
- **User corrections**: vendor, amount, date, category, notes editable on web app via inline editing
- **Web app view**: "Captured Receipts" page under Banking in sidebar. Data table with thumbnail preview, inline edit, status badges, search. Click row to expand preview panel with full image and OCR text.
- **Bulk capture**: capture multiple receipts in sequence — each uploads independently.
- **Delete**: long-press on mobile or delete button on web app. Removes DB record + files from disk.

### File Storage Architecture

- **Location**: Fly.io persistent volume mounted at `/data`
- **Path structure**: `/data/{tenant_slug}/{fy_year}/receipts/{id}.jpg` and `{id}.pdf`
- **FY partitioning**: Australian fiscal year (July–June). E.g. `fy26` for July 2025 – June 2026.
- **Tenant isolation**: each tenant's files are in a separate directory keyed by their slug
- **Thumbnail**: 150px wide JPEG at quality 40 (~10KB), stored as base64 in DB for fast list-view rendering
- **Original image**: compressed JPEG on disk, not stored in DB
- **Generated PDF**: on disk alongside the image

### Tenant Slug Algorithm

Readable, filesystem-safe, unique directory names derived from business name + email:

1. Lowercase the business name
2. Strip non-alphanumeric characters (keep spaces)
3. Remove common suffixes: Pty Ltd, Inc, Limited, Corp, etc.
4. Take first 4 characters of up to 3 remaining words
5. Append 5-character SHA-256 hash of the owner's email

**Examples**:
| Business Name | Email | Slug |
|---|---|---|
| Acme Construction Pty Ltd | admin@acme.com.au | `acme_cons_df59a` |
| Joe's Plumbing | joe@plumb.com | `joes_plum_193b9` |
| XYZ | info@xyz.com.au | `xyz_74228` |

Database name: `ledgius_t_{slug}` (e.g. `ledgius_t_acme_cons_df59a`)
Data directory: `/data/{slug}/fy26/receipts/`

## AC-5: Create Contact ✅ Implemented

Add customers and suppliers from the field:

- **Quick create**: name (required), email, phone, ABN — all in a card-based form with collapsible address section (line 1, line 2, city/suburb, state, postcode)
- **Type**: customer or vendor toggle at top of form (default: customer)
- **Address**: optional, toggled via "Add address" link — supports Australian format (state max 3 chars, postcode max 4 digits)
- **Sync**: immediately syncs to API on save, available in web app
- **From phone contacts**: import from device contact book (planned)
- **Business card scan**: OCR from a business card photo (planned)

## AC-6: Direct Payment via Stripe

Accept payments on mobile devices:

### Option A: Ledgius Stripe Connect (default)
- Ledgius operates as a Stripe Connect platform
- User's customers pay via Stripe — funds flow through Ledgius's Stripe account then settle to the user's bank
- Ledgius takes a small platform fee (e.g. 0.5% on top of Stripe's standard fees)
- User doesn't need their own Stripe account — zero setup friction
- Settlement: daily or weekly to user's linked bank account

### Option B: User's Own Stripe Account
- User connects their existing Stripe account via OAuth (Stripe Connect Standard)
- Payments go directly to their Stripe account
- No Ledgius platform fee
- User manages their own Stripe dashboard for disputes, refunds, etc.

### Payment methods:
- **Payment link**: generated per invoice, shareable via SMS/email/QR code
- **QR code**: customer scans, pays on their phone
- **Tap-to-pay**: iPhone Tap to Pay / Android NFC for in-person card payments (future — requires Stripe Terminal SDK)
- **Manual entry**: record a cash/bank transfer payment manually

### Payment recording:
- When Stripe processes a payment: webhook updates invoice status → auto-generates receipt → pushes notification to mobile app
- Payment creates the corresponding journal entry in the ledger (receipt → debit bank, credit AR)

# Tech Stack Considerations

## Cross-platform approach options:

| Approach | Pros | Cons |
|----------|------|------|
| **React Native** | Share code with web app (React), large ecosystem, Expo for fast dev | Performance for camera/NFC, native feel |
| **Flutter** | Excellent performance, single codebase, great camera/NFC support | Different language (Dart), no code sharing with web |
| **Native (Swift + Kotlin)** | Best performance, best UX, full platform access | Two codebases to maintain, slower development |

**Recommendation**: React Native with Expo — maximises code and knowledge sharing with the existing React web app. Camera and Stripe SDK have mature React Native packages.

## API integration:
- Same API: `api.ledgius.com`
- Same auth: JWT tokens stored in Expo SecureStore (encrypted on-device), auto-refresh on 401
- Existing endpoints: `/invoices`, `/contacts`, `/receipts`, etc.
- New endpoints implemented:
  - `GET/POST /quotes`, `GET/PUT /quotes/:id`, `POST /quotes/:id/send`, `POST /quotes/:id/convert`
  - `GET/POST /captures`, `GET/PUT/DELETE /captures/:id`
- Pending: `/payments/stripe-connect`, `/payments/create-payment-link`

## Offline support:
- Local SQLite database for offline-first data
- Sync queue: pending creates/updates queued and sent when online
- Conflict resolution: server wins for most fields, merge for line items

# Out of Scope (V1)

- Full accounting operations (journals, reconciliation, BAS — desktop only)
- Inventory management
- Payroll
- Bank feeds connection management
- Multi-currency on mobile
- Apple Pay / Google Pay (as payment acceptance — only Stripe for V1)
- Tap-to-pay NFC (requires Stripe Terminal integration — V2)
- Business card OCR scanning
- Expense claim workflows
- Team management / multi-user on mobile
