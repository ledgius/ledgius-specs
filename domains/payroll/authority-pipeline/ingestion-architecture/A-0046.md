# Summary

Production architecture for ingesting the **AU workplace-relations authority surface** — Fair Work Act 2009, NES, 122 modern awards, the FWC Modern Awards Pay Database, registered EBAs, and FWO guidance — into the existing knowledge-pipeline machinery (A-0020) and synthesising governed, reviewable executable artifacts (CEL + OPA Rego + go-rules) under a strict **per-award file isolation** discipline.

This architecture **reuses** A-0020's pipeline stages (Discover → Acquire → Canonicalise → Segment → Extract → Bind → Synthesise → Validate → Generate Tests → Human Review → Publish) and **extends** A-0020's source registry, schema registry, and taxonomy with payroll-specific adapters and entities. It does not introduce a parallel pipeline.

# Requirement Link

- R-0073 (Payroll Authority Knowledge Pipeline Requirements)

# Technical Context

The AU workplace-relations surface differs from the tax/accounting surface in three ways that drive the architectural extensions:

1. **Volume + heterogeneity**: 122 awards, each ~30–80 pages of PDF, plus an HTML representation, plus an entry in the structured Modern Awards Pay Database. Tax law is large but concentrated; awards are wide and shallow.
2. **Per-award structural variation**: every award has its own classification ladder, its own clause numbering scheme, and its own allowance taxonomy. There is no single canonical schema across the 122 awards — each must be modelled clause-by-clause.
3. **Layered authority**: NES floor → award → EBA → IFA. The runtime resolver must compose four ordered layers, with strict precedence and BOOT validation. Tax/accounting has comparable layering but it's smaller (statute → ATO ruling → industry practice).

The existing rules orchestrator (`pkg/rules/`) already supports CEL, OPA Rego, and go-rules; this architecture extends bundle loading + authority introspection to scope by `region/domain` (e.g. `au/payroll`).

# Proposed Design

## Non-Negotiable Design Invariants

The seven invariants of A-0020 §Non-Negotiable Design Invariants apply unchanged. This architecture adds three payroll-specific ones:

1. **Per-award file isolation** — each modern award gets its own set of bundle files (CEL YAML, Rego policy, go-rules JSON). No bundle file may contain rules from more than one award. This isolation guarantees that a single award's annual variation never invalidates an unrelated award's bundle and that runtime selection of award rules is unambiguous.
2. **Layer precedence is explicit** — every payroll-rule resolution at runtime traces NES → award → EBA → IFA in order; the chosen layer is recorded on the resulting pay-run line for audit. No silent layer collapses.
3. **BOOT-passing is recorded, never inferred** — the system never decides on its own that an EBA or IFA passes the Better Off Overall Test; the operator declares it (with audit trail of who and when). The system enforces NES-floor invariants but does not assess BOOT.

## Pipeline Stages — Payroll Specialisation

```
Source Registry (au/payroll)
  -> Discover (FWC award index, FWO NES index, FWA legislation crawl)
  -> Acquire (Tier-1 PDFs + Tier-2 HTML + Tier-1 Pay Database structured data)
  -> Canonicalise (per-source HTML/PDF normalisers)
  -> Segment Passages (clause-aware segmentation)
  -> Extract Statements (LLM-assisted, classification-aware)
  -> Bind to Schema + Taxonomy (per-award classification ladders + cross-award penalty/allowance/leave taxonomies)
  -> Synthesise Artifacts (per-award: CEL + Rego + go-rules)
  -> Validate (per-award and cross-award conflict detection)
  -> Generate Tests (synthetic pay-run scenarios per award)
  -> Human Review (mandatory; no auto-publish)
  -> Publish Bundle (one award per file set)
  -> Runtime Catalog / Resolver (NES floor → award → EBA → IFA layered application)
```

## Tiered Acquisition — Source-Specific Adapters

### Fair Work Act 2009 — `legislation.gov.au`

- **Tier 1 (preferred)**: Direct download of the consolidated PDF or the in-force XML where available.
- **Section-level addressability**: Act sections (e.g. s61 NES, s140 modern awards, s172 EBAs, s202 IFAs) are first-class passages. The Act is the long-tail authority; most rule citations point to award/NES articles, but Act sections are the foundational layer.

### National Employment Standards — Fair Work Ombudsman

- **Tier 2 (HTML + Colly)**: FWO publishes plain-English NES guidance at known stable URLs (one page per entitlement). HTML-extract once, retain the full HTML for audit.
- **11 articles** (one per minimum entitlement) — see R-0073 §PAY-AUTH-004 for the list. Articles cite the corresponding Fair Work Act section (e.g. NES annual leave article cites FWA Part 2-2 Division 6).

### Modern Awards — Fair Work Commission

- **Tier 1 (preferred)**: PDF download for canonical/audit retention.
- **Tier 2 (extraction source)**: HTML at `fwc.gov.au/document-search?type=modern-awards`. Each award has a stable URL of form `https://www.fwc.gov.au/document-search?...&award_code=MA000003`.
- **Clause-aware segmentation**: each award's `<h2>`/`<h3>` clause headings drive passage boundaries. Award clause numbers (e.g. clause 21.1 — Saturday penalty rate) are stable identifiers that must be preserved.
- **Variation tracking**: FWC publishes variation determinations on a per-award basis. Each variation produces a new dated bundle version (`MA000003_hospitality_v2026.7.1.yaml` → `MA000003_hospitality_v2027.7.1.yaml`); old versions remain loaded for historical pay-run reproducibility.

### Modern Awards Pay Database — Fair Work Commission

- **Tier 1 (structured)**: Treat the Pay Database as a first-class structured source. Wage rates, allowance amounts, and penalty multipliers are exposed as queryable data — these populate CEL formula constants without requiring PDF/HTML extraction.
- This adapter is what unlocks reliable annual updates: when FWC pushes the new wage data on 1 July each year, the Pay Database adapter detects the change, updates the affected awards' CEL constants, and produces draft PRs for review.

### Find My Award Tool — Fair Work Ombudsman

- **Out of scope for v1 ingestion** — the FMA tool is a decision tree, not authoritative content. The decision tree may be modelled as an OPA policy in a future task to power an in-app award-suggestion feature, but v1 leaves award selection as an operator declaration.

### EBA Register — Fair Work Commission

- **Tier 2 (HTML)** for register lookup. EBA approval decisions are HTML-rendered with downloadable PDF attachments (the actual approved agreement).
- EBAs are **not** ingested as authority articles — they are tenant-scoped data. The pipeline assists tenants by providing a structured EBA-upload flow that extracts metadata (FWC reference, base award, operative date, nominal expiry) from the FWC decision page; the operator then manually maps varied clauses to override-rule structures.

### Knowledge Statement Model — Payroll Specialisation

A-0020's `knowledge_statement` model is reused with the following `rule_type` extensions:

- `award_applicability` — does award X cover this tenant + role?
- `classification_match` — given a role description, which classification level fits?
- `penalty_eligibility` — does this shift's date/time/role qualify for penalty rate Y?
- `allowance_eligibility` — does this employee qualify for allowance Z?
- `leave_accrual_calculation` — what is the leave accrued in this period?

Statements with `explicit_or_inferred = inferred` retain the elevated review requirement from A-0020.

## Per-Award File Layout — Bundle Discipline

**Every modern award produces its own self-contained set of bundle files. No co-mingling.**

```
pkg/rules/bundles/au/payroll/
  ├── nes_v<semver>.yaml                              # Cross-award NES floor (CEL)
  ├── nes.rego                                        # Cross-award NES eligibility (Rego)
  ├── nes_decisions.json                              # NES decision tables (go-rules)
  │
  ├── MA000002_clerks_v2026.7.1.yaml                  # Clerks — Private Sector (CEL)
  ├── MA000002_clerks.rego                            # Clerks — Private Sector (Rego)
  ├── MA000002_clerks_decisions.json                  # Clerks — Private Sector (go-rules)

  ├── MA000003_hospitality.rego                       # Hospitality (Rego)
  ├── MA000003_hospitality_decisions.json             # Hospitality (go-rules)
```

### File-naming convention

| Component | Format | Example |
|---|---|---|
| FWC code | Uppercase, no spaces | `MA000003` |
| Short slug | Lowercase, snake_case, derived from award title | `hospitality`, `general_retail`, `construction_general` |
| Semver | `v<year>.<month>.<minor>` aligned to FWC variation date | `v2026.7.1` (1 July 2026 variation, first patch) |
| Suffix | Engine + extension | `_v<semver>.yaml` (CEL) / `.rego` (Rego) / `_decisions.json` (go-rules) |

### Why per-award separation is non-negotiable

- **Independence of update**: FWC varies one award at a time. A change to MA000003 must never require touching MA000020's bundle.
- **Reproducibility**: a pay run for a 2025-12 period must load the award bundle as it stood at 2025-12, not the 2026-07 update. Per-award file separation + dated semver makes loading the historical version trivial; co-mingled bundles would force keeping every entire monolithic bundle indefinitely.
- **Reviewability**: a reviewer assessing the next variation of MA000003 should see only that award's change — no diff noise from other awards. Per-award separation localises the diff to a single set of three files.
- **Selective loading**: if a tenant uses only MA000003, the runtime can choose to load only that bundle plus NES; smaller memory footprint, faster startup.

### Cross-award shared utilities

A small shared bundle (`au/payroll/_shared.cel`, `_shared.rego`) holds primitives used by every award — date helpers, ordinary-hour-bucket arithmetic, rate-rounding modes. The leading underscore signals "shared, not an award" so the loader knows not to treat it as an award bundle. The shared bundle is loaded first; award bundles import its symbols.

## Runtime Resolver — Layered Application

```
function resolveLayeredRule(employee, payRunLine):
    # Layer 1 — NES floor
    nesRule = nesBundle.lookup(payRunLine.kind)

    # Layer 2 — Award (if employee classified)
    awardRule = nil
    if employee.classification.award_code:
        awardBundle = loadAwardBundle(employee.classification.award_code, payRunLine.period)
        awardRule = awardBundle.lookup(payRunLine.kind, employee.classification.level)

    # Layer 3 — EBA (if tenant on an EBA covering the employee)
    ebaRule = nil
    if tenant.eba and tenant.eba.covers(employee):
        ebaRule = tenant.eba.overrides.lookup(payRunLine.kind, employee.classification.level)

    # Layer 4 — IFA (if employee has an active IFA varying this rule)
    ifaRule = nil
    if employee.ifa and employee.ifa.varies(payRunLine.kind):
        ifaRule = employee.ifa.overrides.lookup(payRunLine.kind)

    # Compose: highest applicable layer wins; NES is the floor that must always be met
    chosen = ifaRule or ebaRule or awardRule or nesRule
    enforceNESFloor(chosen, nesRule, payRunLine)

    payRunLine.authority_refs = [chosen.authority_refs..., nesRule.authority_refs...]
    return chosen
```

The resolver records the chosen layer + each layer's `authority_refs` on every pay-run line so the audit trail is preserved.

## Validation + Conflict Detection

In addition to A-0020's existing validations:

- **Per-award internal consistency** — every classification listed in the award's classification ladder has a base rate; every penalty kind referenced has a definition; every allowance referenced has an eligibility predicate.
- **NES floor enforcement** — every award's annual leave, personal/carer's leave, and public-holiday treatment is greater than or equal to the NES floor. If an award appears to give less than the NES floor, the bundle is held in draft pending human review (likely an extraction error).
- **Cross-award orthogonality** — no two awards may publish CEL formulas with the same `id` (file-isolation discipline enforces this at filesystem level; the validator double-checks at load time).
- **EBA BOOT pre-flight** — when an EBA override gives less than the underlying award on any single dimension, the validator surfaces it for the operator to confirm BOOT-passing was assessed across the full agreement.

## Test Generation — Per-Award Scenarios

The existing test-generation framework (A-0020) is extended with award-aware scenario generation:

- For each award, generate canonical scenarios: ordinary weekday shift, Saturday shift, Sunday shift, public holiday shift, RDO accrual, leave accrual, casual loading.
- For each penalty kind in the award, generate at least one shift that triggers it.
- For each allowance in the award, generate at least one scenario where the allowance is owed.
- Expected values are populated from the FWC Pay Database (Tier-1 structured source) — this is what makes annual-update validation tractable: the new wage data drives both the formula constants AND the expected test values.

# Alternatives Considered

## Single monolithic `payroll_au.yaml` bundle

**Rejected.** Co-mingling all awards in one file makes annual updates a high-risk operation (a wrong edit cascades), removes the diff-locality property reviewers depend on, and forces every tenant to load every award even if they use one. The user-stated requirement for per-award files reinforces this rejection.

## One bundle per (award × variation date) snapshot file

**Considered.** Would replace `MA000003_hospitality_v2026.7.1.yaml` with `MA000003_hospitality.yaml` containing every variation snapshot inline. **Rejected** — Git already provides the historical snapshot via blob history; storing snapshots inline duplicates that and inflates the bundle. The semver suffix on the file name is the right granularity: each variation is a new file; old files stay loaded until cleanup criteria are met (no pay-run from that period is still expected to be replayed).

## Award rules expressed entirely in OPA Rego

**Rejected.** Penalty-rate calculation, allowance arithmetic, and leave-accrual formulas are calculations, not policies — CEL is the right tool. OPA is reserved for eligibility decisions (does this employee qualify for this allowance? does this award cover this tenant?). Mixing the two engines is correct; collapsing into one engine would make calculations awkward to express and test.

## Embedding award text directly in CEL `description` fields

**Rejected.** Award clause text is the authoritative source — it lives in the knowledge article (per R-0008). CEL artifacts link to articles by `authority_refs`, they don't duplicate the article body. Duplicating creates two-source-of-truth drift the moment the next variation publishes.

# Risks + Mitigations

| Risk | Impact | Mitigation |
|---|---|---|
| Award PDF/HTML extraction fidelity errors silently produce wrong rules | High — underpayment | Mandatory human review per A-0020 §Pipeline Stage; canonical Pay Database as expected-value source for tests |
| Annual-update lag (FWC publishes 1 July; tenants run pay before bundle updated) | Medium — recoverable via back-pay | Pay Database adapter polls weekly; draft-PR auto-generation gives reviewers a head-start; rollout target: bundles published by 7 July each year |
| EBA varied-clause misinterpretation (operator declares BOOT, system trusts) | Medium — operator-mediated | Make BOOT-passing declaration prominent + audit-trailed; defer auto-BOOT-assessment as a v2 feature |
| Award bundle size explosion as we ingest all 122 | Low — performance | Per-award file separation enables selective loading; only awards declared by at least one tenant are loaded into a given runtime |
| Variation-date semver drift (operator publishes wrong date in filename) | Low — caught at validation | CI validates semver matches the article's `latest_variation_date` field |

# Performance Targets

- Article + bundle load time: ≤200ms per page resolve (parity with R-0008 §KNW-PIP-024).
- Per-award bundle size: ≤500 KB CEL YAML, ≤200 KB Rego, ≤200 KB go-rules JSON. Awards exceeding these likely indicate over-extraction; review trigger.
- Pay-run dry-run latency: ≤1s for a 50-employee tenant on a single award.

# Migration / Rollout

1. **Phase 1 (T-0041)**: NES articles + 5 awards + cross-award shared utilities + EBA/IFA model + runtime resolver + help-panel surfacing. Feature-flagged off by default; tenants opt in.
2. **Phase 2**: Scale to ~30 awards covering the most-common-tenant industries; off-by-default flag remains until coverage threshold reached.
3. **Phase 3**: Full coverage of all 122 awards. Flag flipped on by default; tenants without a declared award land on a "Choose your award" onboarding step.
4. **Phase 4 (post-MVP)**: Long-service-leave portability schemes (per-state); auto-BOOT-pre-flight tooling for EBA upload.
