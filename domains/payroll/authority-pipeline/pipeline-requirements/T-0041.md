# Summary

Land Phase-1 of R-0073 / A-0046: ingest the NES + 5 high-volume modern awards, publish them as knowledge articles + per-award rule bundles (CEL + OPA Rego + go-rules), implement the EBA + IFA tenant data model, build the layered runtime resolver (NES → award → EBA → IFA), and wire the help panel for payroll pages. Feature-flagged off by default for tenants until coverage scales in Phase 2.

This is a ground-up implementation task. It establishes the pattern for the remaining ~117 awards rather than racing to ingest all 122 at once.

# Initial Award Set (5)

Chosen for high tenant-share + diverse clause shapes (covers the long tail of award structural variation):

| FWC code | Title | Why included |
|---|---|---|
| MA000002 | Clerks — Private Sector Award | Covers most office/admin tenants; relatively simple penalty structure |
| MA000003 | Hospitality Industry (General) Award | Complex penalty rates (Saturday, Sunday, public holiday, late-night, split shifts); broad allowance set; high underpayment risk |
| MA000004 | General Retail Industry Award | High tenant-share; Sunday penalty rates were a recent FWC variation hot-spot |
| MA000020 | Building & Construction General On-site Award | RDO cycle; tool/leading-hand/site allowances; long-service-leave portability hooks |
| MA000065 | Professional Employees Award | Annualised salary clauses (clause 17); minimal penalty structure but complex classification ladder |

Choosing these five exercises: per-award classification ladders, time-of-day penalty matrices, allowance eligibility rules, RDO accrual, annualised salary, casual loading, and Sunday-penalty variation handling.

## Order of work — vertical slices

Each slice is independently shippable + reviewable.

**Slice 1: NES floor (no awards yet)**

- Author 11 NES articles under `docs/authority/articles/au/payroll/fwo/nes-*.yaml`.
- Author the parent Fair Work Act 2009 article under `docs/authority/articles/au/payroll/fwa/fair-work-act-2009.yaml` with section-level passages for the NES-related sections (Part 2-2 of the Act).
- Synthesise `pkg/rules/bundles/au/payroll/nes_v2026.7.1.yaml` (CEL: leave-accrual rates, public-holiday-pay floor) + `nes.rego` (eligibility predicates).
- Wire help-panel surfacing on `/employees`, `/employees/:id`, `/pay-runs`, `/pay-runs/:id` to pick up the NES articles via `usePagePolicies(["payroll", "nes"])`.
- **Ship gate**: NES articles render in help panel; `nes.rego` loads at startup; existing payroll calculations continue to pass without behavioural change.

**Slice 2: First award end-to-end (MA000002 Clerks)**

Smallest of the five. Used to validate the per-award file pattern + extraction prompts + synthesis output before scaling.

- Article: `docs/authority/articles/au/payroll/fwc/MA000002_clerks.yaml` with clause-level passages.
- Bundle files (per A-0046 §Per-Award File Layout):
  - `pkg/rules/bundles/au/payroll/MA000002_clerks_v2026.7.1.yaml`
  - `pkg/rules/bundles/au/payroll/MA000002_clerks.rego`
  - `pkg/rules/bundles/au/payroll/MA000002_clerks_decisions.json`
- Cross-award shared utilities: `pkg/rules/bundles/au/payroll/_shared.cel`, `_shared.rego`.
- Schema-registry rows for `EmployeeClassification`, `WorkPattern`, `PayItem`, `PenaltyRate`, `AllowanceType`, `LeaveAccrual`.
- Bundle loader: extend `pkg/rules` to load by region/domain (`au/payroll/*`); add per-award selective load.
- Layered resolver: implement NES → award resolution (EBA + IFA layers stubbed; covered in Slice 4).
- Help-panel: declare `MA000002` as primary on payroll pages when the tenant has declared this award.
- **Ship gate**: A test tenant with award MA000002 + one Level-2 employee + a Saturday shift produces the correct pay-run output (compared against manual calc + FWC Pay Database wage value).

**Slice 3: Remaining 4 awards in parallel**

MA000003, MA000004, MA000020, MA000065 — each adds three bundle files + one article. The pattern from Slice 2 is exercised four more times to surface any per-award structural quirks before scaling further.

- One PR per award keeps reviewability tight; can run in parallel.
- Each award's PR ships its own end-to-end test scenario.
- AI-0023 audit extended: per-award coverage % reported.

**Slice 4: EBA model**

- Schema entity `EBADeclaration` + DB migration (tenant-scoped table).
- API endpoints: `POST /api/v1/payroll/eba` (upload + metadata extraction), `GET /api/v1/payroll/eba/:id`, `DELETE /api/v1/payroll/eba/:id`.
- Override-bundle loader: tenant-scoped EBA-derived rules layer over the base award at runtime.
- BOOT-passing declaration capture: required tick + signature audit at upload.
- Nominal-expiry tracking: warn 60 days before; refuse silent application after expiry.
- Tenant-onboarding flow: "Are you on an EBA?" branch.
- **Ship gate**: A test tenant on MA000003 with a synthetic EBA varying the Sunday penalty rate produces the EBA's value (not the award's) in pay-run output, with `authority_refs` recording both layers.

**Slice 5: IFA model**

- Schema entity `IFADeclaration` + DB migration (employee-scoped table).
- API endpoints: `POST /api/v1/payroll/employees/:id/ifa`, `DELETE /api/v1/payroll/employees/:id/ifa/:ifaId`.
- Per-employee override layer at pay-calculation time; integrates into the existing layered resolver.
- BOOT-passing declaration + 13-week-termination tracking per FWA s203(6).
- Employee-detail UI: capture form + active/terminated badge.
- **Ship gate**: A test employee with an IFA varying ordinary-hours pattern produces IFA-driven hours in pay-run output.

**Slice 6: Determination trace + replayable pay runs**

Added in response to PR #37 reviewer feedback. Stage 5 of the runtime architecture (per A-0046 §Runtime Architecture).

- Schema entity `DeterminationTrace` + DB migration (per pay-run-line, immutable).
- Stage 5 emitter wired into the existing pay-run path: every line writes a trace at calculation time.
- Trace contents: resolved instrument chain + formulas evaluated + input values + comparator outcomes + exceptions applied + authority refs + bundle versions + timestamp.
- Replay endpoint: `POST /api/v1/payroll/pay-runs/:id/replay` re-runs a historical pay-run from its traces and asserts identical output (within float tolerance).
- **Ship gate**: replay test on a slice-3 pay-run output produces identical result; trace is dumpable as JSON for audit handoff.

**Slice 7: Comparator engine + annualised-wage true-up**

Added in response to PR #37 reviewer feedback. Stage 3 of the runtime architecture.

- Schema entity `EmploymentPayArrangement` + DB migration; supports arrangement_type ∈ {hourly, salary_contract, annualised_wage_award, annualised_wage_agreement, high_income_guarantee, commission_incentive, commission_only, piece_rate, mixed}.
- Comparator implementations: NES floor, award shadow calc, BOOT-style operator-asserted, salary-offset, high-income-guarantee.
- For annualised-wage-award + annualised-wage-agreement employees: every pay period runs the shadow award calc + records the cumulative comparison; reconciliation close blocked when shadow exceeds salary paid.
- HIG enforcement: tenant declares HIG with a written guarantee + earnings above the threshold; employee then opts out of award coverage but NES + agreement (if any) still apply.
- **Ship gate**: a test tenant with one annualised-wage-award employee whose Sunday work pushes shadow above salary produces a top-up line at reconciliation; salary-contract employee with offset clauses correctly nets entitlements.

**Slice 8: Exception ledger + variable pay plans**

Added in response to PR #37 reviewer feedback. Stage 4 of the runtime architecture.

- Schema entities `PayrollException` + `VariablePayPlan` + DB migrations.
- API endpoints: `POST /api/v1/payroll/employees/:id/exceptions` (advance / leave-in-advance / overpayment-recovery / manual adjustment / backpay), `POST /api/v1/payroll/variable-pay-plans` (commission / bonus / incentive / piece-rate / recoverable draw).
- Approval workflow: every exception requires an approver record + reason + (where applicable) employee acknowledgement.
- Repayment scheduling for advances + overpayments — automatic deduction from future pay runs until satisfied; cannot be silently bypassed.
- Variable pay PAYG/SG/STP classification: every variable-pay component carries the correct withholding treatment (regular vs additional payment), OTE-eligibility, and STP Phase 2 income-type / payment-classification metadata before posting.
- **Ship gate**: end-to-end exception scenarios pass — wage advance + multi-period recovery; leave-in-advance + termination-pay treatment; manual adjustment with required acknowledgement; bonus payment with correct PAYG additional-payment withholding; commission-only employee blocked from setup unless instrument permits.

**Slice 9: Contextual coverage gate + LSL capability flag + employment-contract layer**

Added in response to PR #37 reviewer feedback.

- Replace bundle-level 80% gate with per-pay-run-context gate per PAY-AUTH-032: enumerate pay-relevant components for the specific employee + period, block (or warn-with-explicit-acknowledge) if any component lacks an authority-bound rule.
- LSL capability flag: surface a compliance-limitation warning during onboarding, leave-liability reporting, and termination final-pay calculation — never silently produce a final-pay output that omits LSL.
- Employment-contract layer: contract additions stack on top of the resolved instrument chain in Stage 2. Additions can only improve on minimums; any reductive clause is held invalid + flagged.
- **Ship gate**: a test pay run with a missing-allowance scenario triggers the contextual gate; an employee without LSL configured surfaces the limitation warning; a contract clause attempting to reduce annual leave below NES is rejected.

**Slice 10: Coverage audit + polish**

- AI-0023 extended: per-award coverage % (engineering-quality metric, not a runtime gate).
- CI validation: every payroll bundle artifact carries `authority_refs` to a published article.
- Help-panel badges: FWC orange / FWO teal / FWA dark blue per A-0046.
- "Post-nominal-expiry" badge on EBAs past their nominal expiry, surfaced on every pay-run line + tenant onboarding.
- Documentation pass on `ledgius-knowledge-ingestion/CLAUDE.md` + `ledgius-api/CLAUDE.md` describing the pipeline contract for future award additions.
- **Ship gate**: feature flag remains off by default; turning it on for a test tenant produces correct end-to-end behaviour across all five stages of the runtime architecture.

# Out of Scope (explicit)

- Long-service-leave portability schemes (BUSSQ / CoINVEST / etc.) — separate task.
- Auto-classification (suggesting the right classification level for a role description) — operator declares.
- Auto-BOOT-pre-flight (assessing whether an EBA passes BOOT) — operator declares per A-0046.
- Find-My-Award decision tree as an in-app feature — links out to FWO tool.
- The remaining ~117 modern awards — Phase 2 + Phase 3 per A-0046 rollout plan.
- NZ / UK payroll authority — schema is jurisdiction-extensible per R-0008 §KNW-PIP-027; populating non-AU content is a separate effort.
- Migrating the existing payroll `pkg/rules/bundles/payroll_v1.0.0.yaml` (PAYG / SG formulas) — that is the **federal-statutory** layer and is unchanged by this task. The new `au/payroll/*` bundles are the **workplace-relations** layer that compose with the existing federal layer.

# Open Questions

1. **Source-extraction primary path** — HTML or PDF for awards? A-0046 recommends HTML primary, PDF retained for audit. T-0041 confirms this in Slice 2 with MA000002; if HTML extraction is unreliable, fall back to PDF as primary for subsequent awards.
2. **EBA storage** — A-0046 recommends tenant-scoped DB rows (not source-controlled article files). T-0041 implements (b). Open: where does the approved EBA PDF live? Recommend: object storage (existing tenant-document store) referenced by URL on the `EBADeclaration` row.
3. **Pay Database adapter scope** — full crawl of the Pay Database (every award + every classification + every effective date), or pull-on-demand for the awards we ingest? Recommend: pull-on-demand for Phase 1; full crawl when scaling to all 122 awards.
4. **Feature-flag granularity** — global flag or per-tenant flag? Recommend: per-tenant for Phase 1, global default-on once Phase 3 coverage is reached.
5. **Test scenario authoring** — manually authored or generated from the Pay Database? Recommend: hybrid — generate skeletons from the Pay Database, then manual-author the assertion logic. Pay Database gives wage values; the assertion logic still needs to encode "what the operator expects to happen on a Saturday-night shift", which is human authorship.
