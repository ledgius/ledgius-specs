# Summary

Store and manage employee records for payroll processing, including personal details, employment terms, tax declarations, superannuation, and bank details. This is the foundation entity that all other payroll features depend on.

# Problem / Business Context

Australian businesses must maintain accurate employee records to comply with the Privacy Act 1988, Fair Work Act 2009, ATO tax file declaration requirements, and Superannuation Guarantee obligations. Ledgius must provide a secure, auditable employee management system that supports payroll processing.

# Scope

- Employee CRUD (create, read, update, deactivate)
- Personal details, employment terms, tax declarations, superannuation, bank details
- Employee list and detail pages following Ledgius UX rules
- Audit trail for all employee changes
- Integration with pay run processing, PAYG withholding, and super guarantee

# Out of Scope

- Onboarding workflows (document collection, e-signatures)
- Rostering and scheduling
- Performance management
- Expense claims
- Direct bank feed integration for pay disbursement

# Actors / Users

- Business Owner — adds/manages employees
- Bookkeeper — views and edits employee details for pay processing
- Accountant — verifies tax and super settings for compliance

# Preconditions

- User is authenticated with at least Bookkeeper role
- Chart of Accounts is configured (payroll expense accounts exist)

# Functional Requirements

- **PAY-EMP-001**: System shall store employee personal details (name, DOB, address, email, phone)
- **PAY-EMP-002**: System shall store employment details (type: full_time/part_time/casual/contractor, start date, end date, pay cycle)
- **PAY-EMP-003**: System shall store tax declaration details (TFN status, tax-free threshold, residency, HELP debt, SFSS debt). TFN itself stored encrypted; only `tfn_provided` boolean exposed.
- **PAY-EMP-004**: System shall store superannuation details (fund name, ABN, USI, member number)
- **PAY-EMP-005**: System shall store bank details (BSB, account number, account name). Required for pay disbursement.
- **PAY-EMP-006**: System shall support pay cycles: weekly, fortnightly, monthly
- **PAY-EMP-007**: System shall track employee active/inactive status
- **PAY-EMP-008**: System shall support termination with end date recording
- **PAY-EMP-009**: System shall store employee classification/award code (Fair Work award linkage)
- **PAY-EMP-010**: System shall store multiple pay rates per employee (base, overtime, etc.)
- **PAY-EMP-014**: System shall prevent deletion of employees with pay history (use termination/deactivation instead)
- **PAY-EMP-015**: TFN shall be encrypted at rest using application-level encryption
- **PAY-EMP-018**: Employee changes shall be audit-logged

# UX / UI Behaviour

- **PAY-EMP-011**: Employee list page shows: name, start date, employment type, pay cycle, status
- **PAY-EMP-012**: Employee detail page is editable with Save/Discard following Ledgius UX rules (buttons in header, disabled until dirty)
- **PAY-EMP-013**: Employee detail page shows Activity (audit trail) section as the last section
- **PAY-EMP-017**: TFN masked in all UI displays (show last 3 digits only)

# Validation Rules

- First name and last name are required
- Start date is required
- Employment type must be one of: full_time, part_time, casual, contractor
- Pay cycle must be one of: weekly, fortnightly, monthly
- Residency status must be one of: resident, non_resident, working_holiday

# Error Handling

- Attempt to delete an employee with pay history returns a clear error directing the user to deactivate instead
- Invalid employment type or pay cycle returns validation error with allowed values

# Security / Permissions

- Minimum role: Bookkeeper (view and edit)
- TFN encrypted at rest (Privacy Act 1988 compliance)
- TFN masked in UI (last 3 digits only)
- All changes audit-logged

# Acceptance Criteria

- [x] Employee list page shows name, start date, type, pay cycle, status
- [x] Clicking a row navigates to detail page
- [x] Detail page allows editing with dirty tracking
- [x] New employee form creates record
- [ ] TFN is encrypted at rest
- [ ] PUT endpoint updates employee
- [ ] Terminated employees hidden from active list by default
- [ ] Multiple pay rates per employee supported in UI
- [ ] Award code / classification field available

# Open Questions

1. **TFN storage**: Need to implement encryption before production. Current field exists but stores plaintext.
2. **Award complexity**: Fair Work awards are complex — may need a rules engine for rate determination.
3. **Multiple bank accounts**: Some employees split pay across accounts. Not yet supported.
