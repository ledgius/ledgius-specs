# Terms

- **Dry run**: A preview computation of a pay run that shows the gross / PAYG / super / net per employee
  without committing the payroll journal, creating `pay_run_line` rows, or queueing STP. The user reviews
  the preview and explicitly clicks *Post* to commit.
- **Leave entry**: The split of an employee's pay-period hours between *ordinary hours* and one or more
  *leave categories* (annual, personal/carer's, long service, unpaid). Each leave category draws down the
  employee's leave balance and is reported separately under STP Phase 2 leave disaggregation.

# Summary

The current Pay Run processing flow jumps straight from "select employees + type hours" to a posted journal
and generated payslips. There is no preview step and no affordance for entering leave taken during the period.
This requirement adds (a) a dry-run preview that displays the full calculation without committing, and (b)
a per-employee leave-entry row in the processing UI that splits `hours_worked` across ordinary and leave
categories and draws the corresponding balances down on post.

# Problem / Business Context

Two gaps in the current pay-run flow:

1. **No preview before commit.** Users click *Process Pay Run* and the system immediately posts the payroll
   journal, generates `pay_run_line` rows, and (when configured) queues STP. Any mistake — a miskeyed hours
   value, a wrong payment date, an employee accidentally selected — creates an immutable journal entry that
   can only be fixed with a reversal journal. Several classes of error (a 40-hour vs 4-hour typo in particular)
   are easy to make and expensive to reverse.

2. **Leave cannot be entered at run-time.** R-0003 defines leave accrual/balance tracking but does not specify
   how leave *taken* enters a pay run. In the current UI there's a single "Hours" field per employee. If an
   employee took 2 days of annual leave during the period, there's no way to record it — the whole period is
   treated as ordinary hours, leave balances never deplete, and the STP Phase 2 submission mis-categorises the
   payment (ATO non-compliance).

This requirement closes both gaps without touching the payroll engine's math — just the UI flow and the
`pay_run_line` leave fields that R-0003 already anticipated.

# Scope

- Add a two-phase *Preview → Post* flow to the Pay Run processing page
- Add per-employee leave-entry rows (annual / personal / long service / unpaid) next to the ordinary-hours input
- Show live leave-balance draw-down in the preview so the user can spot over-allocation before post
- Populate `pay_run_line.leave_accrued` and `pay_run_line.leave_taken` JSONB fields (already in schema per R-0003)
- Discard-on-unload: leaving the page during preview discards the draft, not posts it

# Out of Scope

- Leave application workflows (employee submits request, manager approves) — see R-0056 for the mobile flow
- Leave calendar / rostering integration
- Leave balance reporting UI (already covered on the Employee detail page per R-0003)
- Editing a posted pay run — posted runs remain immutable; corrections are reversal journals

# Actors / Users

- **Bookkeeper** — processes pay runs, enters leave, reviews preview, posts
- **Business Owner** — occasionally processes runs for small teams
- **Accountant** — reviews preview before bookkeeper posts on high-value runs

# Preconditions

- R-0003 (Leave Accrual & Tracking) functional — employee leave balances exist per leave type
- Pay Run entity and engine operational (R-0004)
- Employees have employment_type (full_time / part_time / casual) set — drives leave eligibility

# Functional Requirements

- **PAY-DRY-001**: System shall compute a pay run preview without persisting `pay_run_line` rows,
  without posting the payroll journal, and without queueing STP
- **PAY-DRY-002**: Preview shall show per-employee gross, PAYG withholding, super, net pay, and leave draw-down
- **PAY-DRY-003**: Preview shall show pay-run totals (gross / PAYG / super / net) matching the posted-run totals
- **PAY-DRY-004**: User shall post the preview via an explicit *Post Pay Run* action; no implicit posting
- **PAY-DRY-005**: User shall discard the preview via *Cancel* or by navigating away; discarded previews
  leave no ledger artefact
- **PAY-LEA-ENT-001**: Each employee row in the processing UI shall support entry of ordinary hours
  AND zero-or-more leave-hour entries (one per leave type)
- **PAY-LEA-ENT-002**: System shall warn (but not block) when requested leave exceeds current balance,
  allowing negative balances only when R-0003 policy permits
- **PAY-LEA-ENT-003**: On post, system shall deduct leave-taken hours from the employee's leave balance
  per leave type, in the same transaction as the payroll journal post
- **PAY-LEA-ENT-004**: `pay_run_line.leave_taken` JSONB shall record the {leave_type: hours} map for each
  employee, driving STP Phase 2 leave disaggregation
- **PAY-LEA-ENT-005**: Payslip output shall separately show ordinary hours and each leave category with
  its hours, rate, and amount

# UX / UI Behaviour

- *Process Pay Run* button renders the preview inline (no navigation)
- Each employee row shows: checkbox · name · type · ordinary hours · "+ Leave" button
- Clicking "+ Leave" adds a sub-row with a leave-type `<select>` and hours input
- Leave balance shown next to the leave-type label (live updates as hours typed)
- Over-allocation shows an amber warning inline; post still allowed if R-0003 policy permits negative balances
- Preview summary card shows totals + *Cancel* and *Post Pay Run* buttons
- Post triggers a confirmation dialog summarising totals and employee count

# Validation Rules

- Ordinary + leave hours per employee must be ≥ 0
- Leave type on a leave row must be one of: annual, personal, long_service, unpaid
- Cannot mix paid and unpaid leave in the same row; use separate rows

# API Surface

- `POST /pay-runs/preview` — returns `PayRunPreview` without persisting
- `POST /pay-runs` — existing endpoint, now accepts `leave_taken` per employee and deducts balances in-tx

# Success Criteria

- 100% of pay-run mistakes caught at preview stage (validated by user study: 10 bookkeepers, measured
  preview-correction rate vs post-correction rate)
- Leave categorisation correct on all STP Phase 2 submissions (ATO accepts, no rejection due to
  category mismatch)
- Zero regression on current pay-run processing time end-to-end

# Open Questions

- Q1: Should preview persist as a draft pay_run row (status='draft') that can be reopened from a list?
  Or purely in-memory on the client? Drafts survive browser close but cost a table row and state machine;
  in-memory is simpler but loses work on refresh. **Recommendation: in-memory for v1; drafts in v2 if needed.**
- Q2: Should leave balance draw-down occur at preview time (reserve) or only at post? Reserving at
  preview prevents two bookkeepers allocating the same leave twice but complicates cancel. **Recommendation: draw down only at post, under advisory lock.**
