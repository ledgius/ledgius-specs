# Summary

Submit payroll data to the ATO via Single Touch Payroll (STP) Phase 2 after every pay event, update event, and end-of-financial-year (EOFY) finalisation. The payload is the ATO PAYEVNT.0004 message: payee-level Year-To-Date (YTD) totals with disaggregated earnings, allowances, deductions, super, and a derived 6-character Tax Treatment Code; payer-level identifiers including BMS Vendor / BMS Name, Submission ID, and Run Date/Time Stamp. Transmission is via an SBR intermediary (decision deferred to A-0005 — MessageXchange or Ozedi) so Ledgius does not have to self-certify direct ebMS3/AS4 to the SBR gateway in v1. The product must pass the ATO Extended Conformance Testing (ECT) suite against the EVTE environment before being whitelisted for production.

# Terms

| Term | Definition |
|---|---|
| **STP Phase 2** | The ATO's Single Touch Payroll Phase 2 reporting regime, mandatory since 2022-01-01, requiring electronic per-pay-event reporting of disaggregated payroll data using the PAYEVNT.0004 message structure. Replaces end-of-year payment summaries with continuous YTD reporting accessible to employees via myGov. |
| **Pay Event** | An STP submission of type `submit` reporting a finalised pay run to the ATO. Carries per-employee YTD totals as of the pay run's payment date. |
| **Update Event** | An STP submission of type `update` correcting previously reported YTD totals without an associated new pay run. Used for retrospective corrections (e.g. classification fix, missed allowance). |
| **Finalisation Event** | The EOFY STP submission that marks an employee as "tax ready" in myGov, releasing their income statement to the employee. Distinct from a pay event. |
| **Full File Replacement (FFR)** | An ATO submission action that supersedes a prior submission in its entirety. Used when a previously-sent pay event is materially wrong and an Update would not adequately correct the record. Limited to the most recent submission per employer per period. |
| **Submission ID** | A globally-unique identifier the BMS assigns to each PAYEVNT submission. Persists across retries; an FFR or Update references the original Submission ID. |
| **BMS ID** | The Business Management Software identifier (per employer per BMS). Identifies the originating system to the ATO. Combined with the Payroll ID (per-employee) to detect transfers between BMSes. |
| **Run Date/Time Stamp** | PAYEVNT71. The wall-clock UTC timestamp at which the BMS generated the payload. Distinct from the payment date; used by the ATO to order conflicting submissions. |
| **YTD Totals** | Per-employee Year-To-Date amounts within an Australian financial year (1 July – 30 June). STP Phase 2 reports YTD, not per-pay deltas, so YTD must be a continuously-maintained aggregate that reconciles to the sum of finalised pay-run components. |
| **Tax Treatment Code (TTC)** | A 6-character ATO code derived from the employee's tax declaration that summarises how PAYG withholding is calculated. Categories cover regular, actor/performer, horticulturalist, seniors, working-holiday-maker, foreign-resident, no-TFN, and death-beneficiary scenarios, with sub-codes for tax-free-threshold, study/training support loan, and Medicare-Levy variation flags. |
| **Income Type** | Per-employee classification controlling tax-treatment scope. One of: SAW (Salary and Wages), CHP (Closely Held Payee), IAA (Inbound Assignee to Australia), WHM (Working Holiday Maker), SWP (Seasonal Worker Programme), LAB (Labour Hire), VOL (Voluntary Agreement). |
| **Employment Basis** | Per-employee employment classification reported to the ATO. One of: F (Full-time), P (Part-time), C (Casual), L (Labour-hire), V (Voluntary-agreement), D (Death-beneficiary), N (Non-employee). Distinct from the internal `employment_type` enum used elsewhere in Ledgius. |
| **Disaggregated Gross** | The breakdown of an employee's earnings into ATO-required components: Gross (residual ordinary earnings), Overtime, Bonuses & Commissions, Directors' Fees, Paid Leave (with type sub-codes), Allowances (by ATO type code), Lump Sums (A/B/D/E/W), and Salary-Sacrifice deductions. The sum reported must not over-state Gross. |
| **ECT** | Extended Conformance Testing. The ATO's mandatory product-certification suite that exercises a Digital Service Provider's (DSP's) build against representative test cases in the EVTE environment before production whitelisting. ECT submissions are marked by setting the Payer email to `ECTTEST@ect.ato` and the payee's Address Line 2 to `TestCaseNN.xx`. |
| **EVTE** | External Vendor Test Environment. The ATO's pre-production endpoint for DSP testing, distinct from production SBR. The same `BMS Vendor` / `BMS Name` values must be used in EVTE and Production. |
| **SBR Intermediary** | A licensed third-party service that operates the SBR-certified ebMS3/AS4 transport on behalf of a DSP (e.g. MessageXchange, Ozedi). Allows Ledgius to submit JSON / SOAP wrappers without operating its own ebMS3 stack. Architecture choice deferred to A-0005. |

# Problem / Business Context

STP Phase 2 has been mandatory for all Australian employers since 2022-01-01. Every pay event must be reported to the ATO electronically; failure to report correctly and on time results in penalties under the Tax Administration Act. Employees access their income statements via myGov instead of receiving end-of-year payment summaries from the employer. The Phase 2 regime adds disaggregated earnings, allowance categorisation, salary-sacrifice separation, derived Tax Treatment Codes, and per-employee employment basis / income type — none of which Ledgius currently captures or reports.

Before Ledgius can transmit a single PAYEVNT.0004 message in production it must (a) build the payload generator, (b) integrate with an SBR intermediary, (c) be whitelisted in EVTE, and (d) pass the ATO Extended Conformance Testing suite. ECT is a process gate, not a feature — but its acceptance criteria materially shape the data the BMS must capture and how it derives Tax Treatment Codes. This requirement is therefore written to be ECT-pass-ready: every mandatory PAYEVNT.0004 field has a corresponding requirement and every ECT focus area (per the *STP Phase 2 Extended Conformance Testing Guide v2.1*) has an acceptance criterion.

# Scope

- Pay event generation after pay run finalisation (PAYEVNT.0004 payload, per-employee YTD).
- Update event generation for retrospective correction of YTD totals.
- Full File Replacement (FFR) of the most recent submission per employer per period.
- EOFY finalisation event marking employees as "tax ready" in myGov.
- Per-employee Tax Treatment Code (TTC) derivation across all ATO categories.
- Per-employee Income Type and Employment Basis capture and reporting.
- TFN (mod-11) and ABN (mod-89) checksum validation at capture and at payload generation.
- Disaggregated earnings: gross, overtime, bonus & commissions, directors' fees, paid leave (typed), salary sacrifice (type S/O), lump sums A / B / D / E / W, ETP, death-benefit, foreign-income.
- Allowance disaggregation by ATO type code (CD, AD, LD, MD, RD, TD, OD plus the Phase 2 additions JD, KN, ND, QN, T1, U1, V1, WG, WL).
- Deduction disaggregation: union fees (F), workplace giving (W), child support garnishee (G), child support deduction (D).
- Super: separately-reported Super Guarantee (SG) and Salary Sacrifice (RESC) per employee per YTD.
- Country Code reporting for IAA and WHM income types.
- Employer (Payer) section: ABN, Branch, BMS Vendor, BMS Name, BMS ID, Contact, address.
- Run Date/Time Stamp (PAYEVNT71), Submission ID, BMS ID, Payroll ID, Previous Payroll ID generation and persistence.
- Period Start/End dates per pay run.
- Submission queue with idempotent retry, exponential backoff, and operator-visible status.
- ATO response capture: receipt ID, success / warning / error, with full response payload stored.
- Missed report tracking: report calendar (pay frequency × period dates) with detection of skipped periods.
- Prior-year transition: import of YTD opening balances when migrating from another BMS mid-year.
- Payroll ID transfer reporting (BMS / Payroll ID change → previous identifiers reported on next submission).
- Web UI: submission list, submission detail (payload + ATO response), resubmit / FFR / EOFY actions, employer config page.
- ATO Extended Conformance Testing (ECT) execution against EVTE for all mandatory test cases.

# Out of Scope

- Direct SBR / ebMS3 / AS4 self-certification (an SBR intermediary is used in v1; A-0005 captures the path to direct certification as a future option).
- Multi-ABN-per-tenant employer reporting (one ABN per tenant in v1).
- Employee self-service access to income statements (handled by myGov, not by Ledgius).
- Historical STP migration of pay events generated by a prior BMS (only opening YTD balances are imported; pre-Ledgius pay events are not re-emitted).
- STP Phase 1 format (Phase 2 only).
- Tax Practitioner declarations workflow (submitter is the employer, not a registered agent in v1).
- Automatic ATO error-code → user-action remediation (v1 surfaces the error; correction is human-driven).

# Actors / Users

- **Business Owner** — reviews STP submission status; triggers EOFY finalisation; signs off on prior-year-transition opening balances.
- **Bookkeeper** — finalises pay runs (which trigger STP); monitors submission responses; resubmits on failure.
- **Accountant** — verifies YTD reconciliation; reviews EOFY finalisation; handles ATO queries and update events.
- **Platform Admin (Ledgius)** — registers and maintains the BMS Vendor / BMS Name / Product ID / EVTE credentials; promotes the build through ECT.
- **ATO** — receives, validates, and responds to STP submissions via the SBR intermediary.

# Preconditions

- Pay run processing (R-0004) is operational and produces finalised, GL-posted pay runs with disaggregated pay components.
- Employee records (R-0002) include personal details, tax declaration fields (tax-free-threshold, study/training support loan, Medicare-Levy variation, foreign-resident, no-TFN status), TFN, employment basis, income type, country code, and per-employee Payroll ID.
- Super (R-0006) and PAYG (R-0007) are calculated correctly per pay run and produce the breakdowns required for YTD aggregation.
- Employer (Payer) configuration is populated: ABN, Branch, contact, address, declared BMS Vendor and BMS Name.
- The BMS ID is generated and stored once per employer per BMS.
- An SBR intermediary account is provisioned and credentials are stored in the platform secrets store.
- Ledgius is registered with the ATO as a DSP (Product ID issued; EVTE access granted).

# Functional Requirements

## Submission Lifecycle

- **PAY-STP-001**: System shall generate a PAYEVNT.0004 pay event after each pay run finalisation, before the pay run is treated as user-visibly complete in the UI.
- **PAY-STP-002**: System shall report YTD totals (not per-pay deltas) in every submission. YTD totals shall be a continuously-maintained per-employee per-financial-year aggregate.
- **PAY-STP-003**: System shall implement the lifecycle: `pending → submitted → accepted | warning | rejected | error`, with persisted state transitions for audit.
- **PAY-STP-004**: System shall persist the ATO response (receipt ID, status, full response payload) and link it to the originating submission.
- **PAY-STP-005**: System shall support resubmission of failed or rejected events (transport-level retry preserves Submission ID; payload-level correction generates an Update Event).
- **PAY-STP-006**: System shall generate Update Events that correct previously-reported YTD totals without an associated new pay run.
- **PAY-STP-007**: System shall generate an EOFY Finalisation Event per employee per financial year, marking the employee tax-ready in myGov.
- **PAY-STP-007a**: System shall generate Full File Replacement submissions to supersede the most recent submission per employer per period when an Update Event would be insufficient (e.g. wrong payee included).

## Identifier and Header Controls (ECT focus area: SBR technical errors + Vendor details)

- **PAY-STP-008**: System shall populate `BMS Vendor` and `BMS Name` consistently across EVTE and Production submissions, matching the values declared on the DSP registration. The values shall be configuration, not derived from runtime state.
- **PAY-STP-008a**: System shall populate the Product ID matching the value issued at DSP whitelisting.
- **PAY-STP-009**: System shall generate a globally-unique Submission ID per submission and persist it. Retries of the same submission reuse the original Submission ID; an Update Event or FFR references the prior Submission ID.
- **PAY-STP-010**: System shall populate the Run Date/Time Stamp (PAYEVNT71) at payload generation time (UTC), not at transmission time.
- **PAY-STP-011**: System shall maintain a per-employer-per-BMS BMS ID and per-employee Payroll ID; on Payroll ID change the prior Payroll ID shall be reported in the next submission for the affected employee.
- **PAY-STP-012**: System shall produce well-formed XML/JSON payloads (per the SBR intermediary's expected envelope) with correct CMN.ATO.GEN.001036 record delimiters, validated against the schema before transmission.

## Payee Identification and Tax Profile

- **PAY-STP-013**: System shall validate TFN using the ATO mod-11 algorithm at capture and reject malformed TFNs. The 28-day no-TFN allowance for casuals is tracked; on day 29 the system shall flag the employee for tax-treatment-code re-derivation.
- **PAY-STP-014**: System shall validate ABN using the ATO mod-89 algorithm at capture (employer ABN, super-fund ABN, contractor ABN under VOL income type) and reject malformed ABNs.
- **PAY-STP-015**: System shall capture per-employee Income Type (SAW, CHP, IAA, WHM, SWP, LAB, VOL) and Employment Basis (F, P, C, L, V, D, N) and report both on every submission.
- **PAY-STP-016**: System shall capture and report Country Code for employees with Income Type IAA or WHM.
- **PAY-STP-017**: System shall derive a 6-character Tax Treatment Code (TTC) per employee per pay event from the tax declaration fields, including: tax-free-threshold claim, study and training support loan (SSL/HELP combined indicator), Medicare-Levy variation (reduction, exemption, surcharge), spouse and dependants count, foreign-resident status, no-TFN status (with 28-day shift), seniors-and-pensioners offset, working-holiday-maker registration status, and actor/horticulturalist variant. The derivation rules shall be authored as a decision table (per A-0005) so that new ATO variants can be added without code change.

## Disaggregated Earnings (ECT focus area: Disaggregated Gross + Allowance disaggregation)

- **PAY-STP-018**: System shall report Gross (residual ordinary earnings) per employee per YTD, calculated such that the sum of the disaggregated components does not over-state Gross.
- **PAY-STP-019**: System shall report Overtime, Bonuses & Commissions, and Directors' Fees as separate YTD line items per employee.
- **PAY-STP-020**: System shall report Paid Leave per ATO leave-type sub-code (Cash-out leave in service `C`, Unused leave on termination `U`, Paid Parental leave `P`, Workers' Compensation `W`, Ancillary `A`, Other paid leave `O`).
- **PAY-STP-021**: System shall report Allowances disaggregated by ATO allowance type code, supporting at minimum: CD (cents-per-km), AD (award transport), LD (laundry), MD (overtime meal), RD (domestic/overseas travel), TD (tool), OD (other). System shall additionally support the Phase 2 codes: KN (non-deductible), QN (qualifications/certificates), T1 (tasks), and OD-with-description for residual.
- **PAY-STP-022**: System shall report Lump Sum payments separately by category: Lump Sum A (with R / T type indicator), B, D, E, and W.
- **PAY-STP-023**: System shall report Employment Termination Payments (ETP) by ETP type code.
- **PAY-STP-024**: System shall report Death Benefit ETP separately by recipient-relationship sub-code.
- **PAY-STP-025**: System shall report Foreign Income (assignment country, foreign tax paid, exempt vs assessable) for IAA / foreign-source scenarios.

## Deductions and Super

- **PAY-STP-026**: System shall report Deductions disaggregated as: F (union/professional association fees), W (workplace giving), G (child-support garnishee), D (child-support deduction).
- **PAY-STP-027**: System shall report Super Guarantee (SG) YTD and Reportable Employer Super Contribution / Salary-Sacrifice (RESC, type S `salary sacrifice` and O `other reportable`) YTD as separate line items per employee.

## Period and Calendar Controls (ECT focus area: Phase 1 carryover data quality)

- **PAY-STP-028**: System shall record Pay/Update Date (PAYEVNT69), Period Start, and Period End per submission, derived from the pay run.
- **PAY-STP-029**: System shall maintain a report calendar per employer (pay frequency + expected period dates) and detect missed periods. Missed periods shall surface as a UI warning and a finalisable backlog action.
- **PAY-STP-030**: System shall continuously reconcile YTD totals against the sum of finalised pay-run components for the same financial year. A divergence shall block submission and surface a reconciliation report.

## Prior-Year Transition

- **PAY-STP-031**: System shall accept opening YTD balances per employee per component when a tenant migrates to Ledgius mid-financial-year, persisted as an immutable opening-balance record stamped with the prior BMS ID and prior Payroll ID where known. Subsequent YTD aggregation builds on the opening balance.
- **PAY-STP-032**: System shall report the prior BMS ID and prior Payroll ID on the first submission for each transitioned employee so the ATO can stitch YTDs across BMSes.

## Employer (Payer) Section

- **PAY-STP-033**: System shall include employer ABN, Branch, contact name, contact email, contact phone, postal address, and BMS identifiers in every submission.
- **PAY-STP-034**: ECT submissions shall set the Payer email to `ECTTEST@ect.ato` and the payee Address Line 2 to the relevant `TestCaseNN.xx` marker. The ECT mode is a tenant-level toggle, never enabled in production.

## Transmission

- **PAY-STP-035**: System shall transmit submissions via the configured SBR intermediary (MessageXchange or Ozedi per A-0005 selection) and pull the business response.
- **PAY-STP-036**: System shall maintain a submission queue with idempotent retry, exponential backoff (initial 30 s, max 60 min), and a maximum-attempts ceiling after which the submission requires operator action.

## Non-Functional

- **PAY-STP-037**: STP payload generation shall complete in under 5 seconds for 50 employees, under 30 seconds for 1,000 employees.
- **PAY-STP-038**: All STP submissions, ATO responses, and YTD opening balances shall be immutable audit records (append-only).
- **PAY-STP-039**: Sensitive payload fields (TFN, address, full name) shall be encrypted at rest and access-logged.

# Validation Rules

- Pay run must be finalised before STP generation.
- Employer ABN must be valid (mod-89) and Branch in `[001..099]`.
- Employer contact email and phone must be populated.
- BMS Vendor and BMS Name must match the values declared at DSP registration (config-time check; mismatch blocks transmission).
- Each employee included in a submission must have: TFN (or active 28-day no-TFN window) with valid mod-11 if supplied, Income Type, Employment Basis, derived Tax Treatment Code, and (where applicable) Country Code.
- YTD reconciliation must succeed (no divergence beyond rounding tolerance) before submission.
- Update Events must reference a prior accepted submission for the same employee + financial year.
- FFR is only available against the most recent submission per employer per period.
- EOFY finalisation only available for the current or most recent financial year.
- ECT-mode payloads must carry the `ECTTEST@ect.ato` payer email and the `TestCaseNN.xx` payee Address Line 2 marker.

# Error Handling

- **SBR transmission failure** — submission stays `pending`; queued for retry with exponential backoff up to the attempt ceiling. Attempt-history row records each transport-level failure.
- **ATO rejection (CMN.ATO.GEN.xxx error code)** — submission moves to `rejected`; full response stored; user directed to the submission detail page to inspect the error and choose retry / update / FFR.
- **Schema validation failure** at payload generation — submission never leaves `pending`; the validation report is attached to the submission row; pay run remains finalised but flagged "STP not lodged".
- **YTD reconciliation mismatch** — submission is blocked at generation; reconciliation report shows the components that diverge; user resolves at the pay-run or component level.
- **Missing employer configuration** — submission is blocked with a clear error pointing to the employer config page.
- **Network partition during transmission** — attempt timeouts after 60 s; queued for retry. The submission row stays `submitted` until the response is captured (or a manual operator action resolves it).
- **Intermediary credential failure** — surfaces a critical alert; pauses the submission queue tenant-wide until credentials are refreshed.

# Security / Permissions

- Minimum role for viewing submissions and triggering resubmission: Bookkeeper.
- Update Event creation: Accountant.
- FFR: Accountant or Business Owner (gated by confirmation modal).
- EOFY finalisation: Business Owner or Accountant.
- Employer config (BMS Vendor / BMS Name / Branch): Business Owner only; changes are immutably audited.
- All STP submissions, ATO responses, and YTD opening balances are immutable audit records — append-only, no UPDATE / DELETE paths exposed.
- TFN, address, and full name in payloads are encrypted at rest; access to raw payloads is access-logged with actor + timestamp.
- ECT-mode toggle is platform-admin-only; cannot be enabled on a production-whitelisted tenant.

# Acceptance Criteria

The criteria are split into product-coverage criteria (must-pass for go-live) and ECT-aligned test-case criteria (must-pass against EVTE before production whitelisting). The `TestCaseNN.xx` markers follow the ATO ECT Sample Test Cases catalogue convention (Payer email = `ECTTEST@ect.ato`, Payee Address Line 2 = `TestCaseNN.xx`); exact case numbering is reconciled against the catalogue version current at ECT entry.

## Product Coverage

- [ ] Pay event generated from a finalised pay run; valid PAYEVNT.0004 schema; persisted with Submission ID, Run Date/Time Stamp, BMS ID.
- [ ] YTD totals accumulated correctly across pay runs within a financial year and continuously reconcilable to pay-run component sums.
- [ ] Per-employee Tax Treatment Code derived from tax declaration fields and re-derived on declaration change or 28-day no-TFN expiry.
- [ ] TFN mod-11 and ABN mod-89 validators reject malformed values at capture and at payload generation.

# Open Questions

1. **Intermediary selection** (deferred to A-0005 §Decision Matrix). MessageXchange and Ozedi are both shortlisted; the matrix scores price, throughput, sandbox quality, support, and direct-SBR migration path.
2. **Direct SBR certification timing** — A future-state requirement (post-v1) once volume justifies operating the ebMS3/AS4 stack in-house. Out of scope for this requirement.
3. **YTD reconciliation tolerance** — proposed `$0.01` per component due to per-pay rounding; to be confirmed once the pay-run rounding standard (A-0048) is exercised against ATO test cases.
4. **CHP quarterly reporting workflow** — Closely-Held Payees may report quarterly instead of per-pay; UI affordances for the quarterly cadence to be specified during implementation.
5. **Correction window for finalisation** — the ATO permits re-finalisation up to 14 July; unclear whether re-finalisation after that date requires a manual ATO contact rather than an STP transmission. To verify with the ATO ECT contact before go-live.
