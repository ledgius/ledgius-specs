# Summary

Submit payroll data to the ATO in real-time after each pay run via Single Touch Payroll (STP) Phase 2. Generate STP pay events with disaggregated earnings, PAYG withholding, superannuation, leave balances, and employment conditions. Maintain Year-To-Date (YTD) totals and support the full submission lifecycle including corrections and EOFY finalisation.

# Problem / Business Context

STP Phase 2 has been mandatory for all Australian employers since 1 January 2022. Every pay event must be reported to the ATO electronically, replacing end-of-year payment summaries. Employees access their income statements via myGov. STP Phase 2 requires disaggregated earnings (salary/wages, overtime, bonuses, categorised allowances), super broken out by SG and salary sacrifice, and per-employee employment conditions. Failure to report correctly and on time results in ATO penalties.

# Scope

- STP pay event generation after each pay run finalisation
- YTD total accumulation and maintenance per employee per financial year
- Disaggregated earnings reporting (salary/wages, overtime, bonuses, allowances by ATO type)
- PAYG withholding, super guarantee, and salary sacrifice reporting
- Leave balance reporting (annual, personal, long service)
- Employment basis and conditions reporting
- Employer section (ABN, branch, contact)
- Submission lifecycle (generate, transmit, response handling)
- ATO response storage (success/warning/error with receipt ID)
- Resubmission of failed or rejected events
- Update events (corrections to previously submitted data)
- EOFY finalisation event
- Submission queue with retry capability
- Immutable audit trail for all submissions

# Out of Scope

- Direct SBR gateway certification (use intermediary provider for initial release)
- Employee self-service access to income statements (handled by myGov)
- Historical STP migration (pre-Ledgius pay events)
- STP Phase 1 format (Phase 2 only)
- Multi-ABN employer reporting

# Actors / Users

- Business Owner -- reviews STP submission status, triggers EOFY finalisation
- Bookkeeper -- finalises pay runs triggering STP, monitors submission responses
- Accountant -- verifies YTD totals, reviews EOFY finalisation, handles ATO queries
- ATO -- receives and processes STP submissions

# Preconditions

- Pay run processing is operational and produces finalised pay runs
- Employee records include employment type, income type, tax profile, and super details
- Employer ABN, branch number, and contact details are configured
- SBR intermediary provider is configured for transmission
- PAYG and super amounts are calculated correctly in pay runs

# Functional Requirements

## Submission Lifecycle

- **PAY-STP-001**: System shall generate an STP pay event after each pay run finalisation
- **PAY-STP-002**: System shall report YTD totals (not per-pay amounts) in each submission; YTD totals must be maintained as a running aggregate
- **PAY-STP-003**: System shall support the STP submission lifecycle: generate -> transmit -> response
- **PAY-STP-004**: System shall store ATO response (success/warning/error) with receipt ID
- **PAY-STP-005**: System shall support resubmission of failed or rejected events
- **PAY-STP-006**: System shall support update events (corrections to previously submitted data)
- **PAY-STP-007**: System shall support EOFY finalisation event that marks employees as "tax ready" for myGov access

## Payload Content (STP Phase 2)

- **PAY-STP-008**: System shall report income type per employee (SAW, CHP, IAA, WHM, SWP, LAB, VOL)
- **PAY-STP-009**: System shall disaggregate earnings into salary/wages, overtime, bonuses, and categorised allowances (TRAVEL, MEAL, TOOL, LAUNDRY, etc. per ATO allowance type codes)
- **PAY-STP-010**: System shall report PAYG withholding amount per employee (YTD)
- **PAY-STP-011**: System shall report superannuation guarantee and salary sacrifice separately per employee (YTD)
- **PAY-STP-012**: System shall report deductions (union fees, child support, workplace giving) per employee where applicable
- **PAY-STP-013**: System shall report leave balances (annual, personal, long service) per employee
- **PAY-STP-014**: System shall report employment basis and conditions per employee (full-time, part-time, casual, labour hire)

## Employer Section

- **PAY-STP-015**: System shall include employer ABN, branch number, and contact details in every STP submission

## Transmission

- **PAY-STP-016**: System shall transmit STP submissions via SBR gateway (directly or via intermediary provider)
- **PAY-STP-017**: System shall maintain a submission queue with retry capability for failed transmissions

## Non-Functional

- **PAY-STP-018**: STP payload generation shall complete in under 5 seconds for 50 employees
- **PAY-STP-019**: YTD totals shall be continuously reconcilable against pay run history
- **PAY-STP-020**: All STP submissions shall be immutable audit records

# UX / UI Behaviour

- STP submission status page showing history of all submissions with status indicators
- Each submission row shows: date, event type, financial year, status, receipt ID
- Clickable rows to view full submission detail (payload, response)
- EOFY finalisation action available at end of financial year
- Failed submissions show resubmit action
- Status pills: pending (gray), submitted (blue), accepted (green), rejected (red), error (red)

# Validation Rules

- Pay run must be finalised before STP generation
- Employer ABN and contact details must be configured
- Each employee must have an income type assigned
- YTD totals must reconcile with pay run history before submission
- EOFY finalisation only available for the current or most recent financial year

# Error Handling

- SBR transmission failure queues the submission for retry with exponential backoff
- ATO rejection response stored with full error detail; user directed to review and correct
- YTD reconciliation mismatch raises a warning with discrepancy details before allowing submission
- Missing employer configuration prevents submission with clear error message

# Security / Permissions

- Minimum role: Bookkeeper (view submissions, trigger resubmission)
- EOFY finalisation requires Business Owner or Accountant role
- All STP submissions are immutable audit records
- STP payloads contain sensitive employee data -- access logged

# Open Questions

1. **SBR certification**: Direct SBR integration requires ATO software certification -- a significant undertaking. Recommendation is to use an SBR intermediary provider (e.g. MessageXchange, Ozedi) for initial release, with direct certification as a future option.
2. **YTD drift**: If YTD totals diverge from pay run history sum, the STP submission will be incorrect. Need continuous reconciliation checks and a correction mechanism.
3. **Phase 2 pay component mapping**: Disaggregated earnings and allowance categorisation require pay components to be properly classified in the pay run engine. This is a dependency on R-0004.
4. **Corrections workflow**: Supporting update events and full file replacement adds complexity but is required for compliance.
5. **EOFY timing**: The finalisation event has a specific deadline (14 July following the end of financial year). Employees cannot access their income statement in myGov until their employer marks them as "tax ready".
