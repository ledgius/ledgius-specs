# Summary

Provide centralised authentication (login, JWT issuance, token refresh) and role-based access control (RBAC) for the Ledgius multi-tenant platform. Users authenticate once, receive JWT tokens scoped to a tenant and role, and can switch between tenants they have membership in. Tenant provisioning manages the full lifecycle of business organisations including database creation, migration, suspension, and cancellation.

# Problem / Business Context

A multi-tenant accounting platform must isolate business data between organisations while allowing users to belong to multiple organisations (e.g. a bookkeeper managing several clients). Authentication must be secure (bcrypt, JWT with rotation), and authorisation must enforce role-based boundaries so that viewers cannot modify data and only owners can manage tenant settings. Tenant provisioning must automate database-per-tenant isolation for data security and compliance.

# Scope

- User authentication via email/password with bcrypt verification
- JWT access token issuance with tenant and role claims
- Refresh token rotation (single-use tokens with SHA-256 hashing)
- Tenant switching for multi-tenant users
- Current user profile retrieval with tenant memberships
- Five-tier RBAC: owner, master_accountant, accountant, bookkeeper, viewer
- Multi-tenant data model: Tenant, User, TenantMembership, RefreshToken
- Tenant provisioning lifecycle: create, suspend, cancel, destroy
- Automated tenant database creation and migration execution
- Platform admin flag for super-user access

# Out of Scope

- OAuth2 / SSO / social login providers
- Multi-factor authentication (MFA)
- Password reset / forgot password flow
- Email verification
- User registration (self-service signup)
- Session management (cookie-based)
- Rate limiting on auth endpoints
- Invitation workflow for adding users to tenants
- Tenant billing and subscription management

# Actors / Users

- Platform Admin -- manages tenants and users across the platform, provisions new tenants
- Business Owner -- authenticates and manages their tenant, owns tenant membership
- Accountant / Bookkeeper -- authenticates and accesses tenant(s) they are members of
- Viewer -- read-only access to tenant data

# Preconditions

- Platform database is provisioned with users, tenants, tenant_memberships, and refresh_tokens tables
- JWT secret is configured in the application environment
- At least one user account exists (created via provisioning or seed)

# Functional Requirements

## Authentication

- **AUTH-001**: System shall authenticate users via email and password using bcrypt comparison
- **AUTH-002**: System shall reject login for users with non-active status, returning the account status in the error
- **AUTH-003**: On successful login, system shall return an access token (JWT), a refresh token, token expiry, user profile, and list of tenant memberships
- **AUTH-004**: If the user has exactly one tenant membership, the access token shall be automatically scoped to that tenant and role
- **AUTH-005**: If the user has multiple tenant memberships, the access token shall have empty tenant/role claims, requiring a subsequent SwitchTenant call

## Token Management

- **AUTH-006**: Access tokens shall be signed with HMAC-SHA256 and include claims: issuer ("ledgius"), subject (user ID), issued-at, expires-at, user ID, tenant ID, and role
- **AUTH-007**: Refresh tokens shall be cryptographically random (32 bytes), stored as SHA-256 hashes, and have configurable TTL
- **AUTH-008**: Token refresh shall consume the old refresh token (rotation) and issue a new access/refresh pair
- **AUTH-009**: Expired refresh tokens shall be rejected (checked via `expires_at > now()`)
- **AUTH-010**: System shall support bulk deletion of all refresh tokens for a user (for logout-everywhere)

## Tenant Switching

- **AUTH-011**: Authenticated users shall be able to switch to any tenant they have membership in via `POST /auth/switch-tenant`
- **AUTH-012**: Tenant switching shall verify the tenant is in active status before issuing new tokens
- **AUTH-013**: Tenant switching shall return a fresh access/refresh token pair scoped to the new tenant

## Current User

- **AUTH-014**: `GET /auth/me` shall return the authenticated user's profile, current tenant ID, current role, and list of all tenant memberships with preloaded tenant details

## Role-Based Access Control

- **AUTH-015**: System shall support five ordered roles: owner > master_accountant > accountant > bookkeeper > viewer
- **AUTH-016**: Role is embedded in the JWT claims and available to all downstream middleware and handlers via `ClaimsFromGinContext`
- **AUTH-017**: Platform admin flag (`is_platform_admin`) is separate from tenant roles and grants cross-tenant capabilities

## Tenant Provisioning

- **AUTH-018**: System shall provision new tenants with automated database creation, migration execution, and owner membership assignment
- **AUTH-019**: Tenant slug must match pattern `^[a-z0-9][a-z0-9-]{1,48}[a-z0-9]$` (3-50 chars, lowercase alphanumeric with hyphens)
- **AUTH-020**: Tenant database name shall be derived from slug: `ledgius_t_<slug_with_underscores>`
- **AUTH-021**: If database creation or migration fails, tenant status shall be set to cancelled (rollback)
- **AUTH-022**: System shall support tenant suspension (revokes access, evicts from connection cache)
- **AUTH-023**: System shall support tenant cancellation (marks for future deletion, evicts from connection cache)
- **AUTH-024**: System shall support tenant database destruction only when tenant status is cancelled (irreversible)
- **AUTH-025**: Migrations shall be executed in filename order from the configured migrations directory

## Tenant Configuration

- **AUTH-026**: Tenant shall store Australian business context: ABN, legal name, default currency (AUD), fiscal year start month (default July), timezone (default Australia/Sydney), GST registration status and date

# API Endpoints

| Method | Path | Auth | Description |
|--------|------|------|-------------|
| POST | `/auth/login` | Public | Authenticate with email + password, receive JWT tokens |
| POST | `/auth/refresh` | Public | Exchange refresh token for new token pair (rotation) |
| POST | `/auth/switch-tenant` | Protected | Switch to a different tenant, receive new JWT tokens |
| GET | `/auth/me` | Protected | Get current user profile, tenant, role, and memberships |

# Validation Rules

- Login requires non-empty email (valid format) and password
- Refresh requires non-empty refresh_token
- Switch tenant requires non-empty tenant_id
- Tenant slug must be 3-50 characters, lowercase alphanumeric with hyphens, must start and end with alphanumeric
- Tenant provisioning requires slug, display_name, and owner_user_id

# Error Handling

- Invalid credentials return 401 Unauthorized with generic "invalid credentials" message (no user enumeration)
- Inactive account returns 401 with account status
- Missing or invalid JWT returns 401 Unauthorized
- Tenant switching without membership returns 403 Forbidden
- Inactive tenant on switch returns 403 with tenant status
- Invalid request body returns 400 Bad Request
- Database destruction on non-cancelled tenant returns error with current status

# Security / Permissions

- Passwords hashed with bcrypt at default cost
- JWT signed with HMAC-SHA256
- Refresh tokens are cryptographically random, stored as SHA-256 hashes (raw token never persisted)
- Refresh token rotation prevents replay attacks
- Password hash and token hash fields excluded from JSON serialization (`json:"-"`)
- Database connection details (db_name, db_host, db_port) excluded from JSON serialization
- Tenant database destruction requires cancelled status as a safety gate
- SQL identifiers are quoted to prevent injection in DDL statements

# Acceptance Criteria

- [x] Users can log in with email and password
- [x] JWT access tokens contain user ID, tenant ID, and role claims
- [x] Single-tenant users are auto-scoped on login
- [x] Multi-tenant users receive unscoped token and can switch tenant
- [x] Refresh token rotation consumes old token and issues new pair
- [x] /auth/me returns user profile with all tenant memberships
- [x] Five RBAC roles defined and embedded in JWT claims
- [x] ClaimsFromGinContext helper extracts claims for downstream handlers
- [x] Tenant provisioning creates database, runs migrations, assigns owner
- [x] Tenant suspension and cancellation evict from connection cache
- [x] Database destruction gated on cancelled status

# Open Questions

1. **Role enforcement middleware**: Roles are embedded in JWT claims but no middleware enforces minimum role per endpoint. Each handler must check roles manually or a generic role-guard middleware is needed.
2. **Token revocation**: There is no mechanism to revoke access tokens before expiry. Only refresh tokens can be deleted. Short access TTLs mitigate this.
3. **Logout endpoint**: No explicit logout endpoint exists. Client-side token deletion is the current approach; server-side `DeleteRefreshTokensForUser` is available but not exposed via API.
4. **User management endpoints**: No CRUD endpoints for users exist beyond CreateUser in the repository. Admin user management is not yet exposed.
5. **Invitation flow**: Adding users to tenants requires direct database/repository calls. No API endpoint for inviting users to a tenant.
