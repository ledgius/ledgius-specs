# Terms

Vocabulary used in this spec (population, role, active role, scope, app
surface, etc.) is defined canonically in **R-0052 §Terms** — refer there
for definitions before reading further.

# Summary

Schema, JWT claim shape, middleware pipeline, and migration approach for the
four-population identity model and universal audit fields defined in R-0052.

# Authoritative References

- **R-0040** (Audit Trail) — existing audit log shape extended here
- **R-0041** (Auth/RBAC, superseded) — existing five-tier intra-tenant role granularity preserved within the customer population
- **JWT spec (RFC 7519)** — claim shape

# Database Schema (Platform DB)

All identity-model tables live in the **platform** database. The tenant DB
already holds R-0040's `audit_log`; the new audit fields are added there.

## Migration: `101_identity_roles.sql` (platform)

```sql
-- Spec references: R-0052, A-0028.

-- =============================================================================
-- 1. Regions — geographic / market scopes for region_manager + platform_owner
-- =============================================================================

CREATE TABLE IF NOT EXISTS regions (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    code            TEXT NOT NULL UNIQUE,            -- e.g. "AU-NSW", "AU", "AP"
    display_name    TEXT NOT NULL,
    jurisdiction    TEXT,                            -- ATO/ASIC jurisdiction code where applicable
    parent_region_id UUID NULL REFERENCES regions(id),
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);

COMMENT ON TABLE regions IS
    'Geographic / market regions used to scope region_manager and platform_owner ' ||
    'population assignments. Hierarchical via parent_region_id.';

-- =============================================================================
-- 2. Population assignments — user → population + scope mapping
-- =============================================================================

CREATE TABLE IF NOT EXISTS population_assignments (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id         UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    population      TEXT NOT NULL
        CHECK (population IN ('customer', 'accountant', 'region_manager', 'platform_owner')),
    -- Scope fields — exactly one applies per population:
    tenant_id       UUID NULL REFERENCES tenants(id) ON DELETE CASCADE,  -- customer (1) | accountant (N)
    region_id       UUID NULL REFERENCES regions(id) ON DELETE CASCADE,  -- region_manager (1) | platform_owner (N) | NULL+global=true=platform_owner global
    is_global       BOOLEAN NOT NULL DEFAULT false,                       -- platform_owner only — '*' scope
    -- Intra-tenant role granularity for customer population only (per R-0041):
    intra_role      TEXT NULL
        CHECK (intra_role IS NULL OR intra_role IN ('owner', 'master_accountant', 'accountant', 'bookkeeper', 'viewer')),
    granted_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    granted_by      UUID NULL REFERENCES users(id),
    revoked_at      TIMESTAMPTZ NULL,
    revoked_by      UUID NULL REFERENCES users(id),
    -- Invariants enforced by partial-unique indexes below.
    CHECK (
        (population = 'customer'        AND tenant_id IS NOT NULL AND intra_role IS NOT NULL AND region_id IS NULL AND is_global = false)
     OR (population = 'accountant'      AND tenant_id IS NOT NULL AND region_id IS NULL AND is_global = false)
     OR (population = 'region_manager'  AND region_id IS NOT NULL AND tenant_id IS NULL AND is_global = false)
     OR (population = 'platform_owner'  AND tenant_id IS NULL AND (region_id IS NOT NULL OR is_global = true))
    )
);

COMMENT ON TABLE population_assignments IS
    'Per-user population + scope assignment. A user may hold many assignments ' ||
    '(e.g. accountant across N tenants + customer of their own firm). ' ||
    'Active assignment at session time is encoded in the JWT.';

-- A user can only hold one customer assignment per tenant.
CREATE UNIQUE INDEX IF NOT EXISTS idx_pop_customer_unique
    ON population_assignments(user_id, tenant_id)
    WHERE population = 'customer' AND revoked_at IS NULL;

-- A user can only hold one accountant assignment per tenant.
CREATE UNIQUE INDEX IF NOT EXISTS idx_pop_accountant_unique
    ON population_assignments(user_id, tenant_id)
    WHERE population = 'accountant' AND revoked_at IS NULL;

-- A user can only hold one region_manager assignment per region.
CREATE UNIQUE INDEX IF NOT EXISTS idx_pop_region_mgr_unique
    ON population_assignments(user_id, region_id)
    WHERE population = 'region_manager' AND revoked_at IS NULL;

-- A user can only hold one platform_owner global assignment.
CREATE UNIQUE INDEX IF NOT EXISTS idx_pop_platform_global_unique
    ON population_assignments(user_id)
    WHERE population = 'platform_owner' AND is_global = true AND revoked_at IS NULL;

CREATE INDEX IF NOT EXISTS idx_pop_user ON population_assignments(user_id) WHERE revoked_at IS NULL;
CREATE INDEX IF NOT EXISTS idx_pop_tenant ON population_assignments(tenant_id) WHERE revoked_at IS NULL;
CREATE INDEX IF NOT EXISTS idx_pop_region ON population_assignments(region_id) WHERE revoked_at IS NULL;
```

## Migration: tenant DB extension to `audit_log`

```sql
-- Spec references: R-0052 (IDR-014), A-0028, R-0040.

ALTER TABLE audit_log
    ADD COLUMN IF NOT EXISTS active_role          TEXT,
    ADD COLUMN IF NOT EXISTS active_role_scope_id TEXT,
    ADD COLUMN IF NOT EXISTS app_surface          TEXT,
    ADD COLUMN IF NOT EXISTS request_id           TEXT,
    ADD COLUMN IF NOT EXISTS actor_label          TEXT;

DO $$
BEGIN
    IF NOT EXISTS (
        SELECT 1 FROM information_schema.constraint_column_usage
        WHERE table_name = 'audit_log' AND constraint_name = 'audit_log_app_surface_check'
    ) THEN
        ALTER TABLE audit_log
            ADD CONSTRAINT audit_log_app_surface_check
            CHECK (app_surface IS NULL OR app_surface IN ('web', 'mobile-ios', 'mobile-android', 'api', 'system'));
    END IF;
    IF NOT EXISTS (
        SELECT 1 FROM information_schema.constraint_column_usage
        WHERE table_name = 'audit_log' AND constraint_name = 'audit_log_active_role_check'
    ) THEN
        ALTER TABLE audit_log
            ADD CONSTRAINT audit_log_active_role_check
            CHECK (active_role IS NULL OR active_role IN ('customer', 'accountant', 'region_manager', 'platform_owner', 'system'));
    END IF;
END $$;

COMMENT ON COLUMN audit_log.active_role IS
    'The user population the actor was operating as at action time. ' ||
    'See R-0052 IDR-014.';
COMMENT ON COLUMN audit_log.active_role_scope_id IS
    'tenant_id / region_id / "*" corresponding to the active_role — not a foreign ' ||
    'key because the scope vocabulary depends on the population.';
COMMENT ON COLUMN audit_log.app_surface IS
    'Originating client surface — web / mobile-ios / mobile-android / api / system. ' ||
    'Inferred from X-Client-App header.';
COMMENT ON COLUMN audit_log.request_id IS
    'HTTP request correlation id — pairs an audit row with structured logs.';
COMMENT ON COLUMN audit_log.actor_label IS
    'Optional human-readable label for system actors (e.g. "bank-feed-scheduler", ' ||
    '"stp-finalisation-job"). NULL for human users.';

-- Append-only enforcement: forbid UPDATE / DELETE on audit_log.
CREATE OR REPLACE FUNCTION audit_log_immutable() RETURNS trigger LANGUAGE plpgsql AS $$
BEGIN
    RAISE EXCEPTION 'audit_log is append-only';
END $$;

DROP TRIGGER IF EXISTS audit_log_no_update ON audit_log;
CREATE TRIGGER audit_log_no_update BEFORE UPDATE ON audit_log
    FOR EACH ROW EXECUTE FUNCTION audit_log_immutable();

DROP TRIGGER IF EXISTS audit_log_no_delete ON audit_log;
CREATE TRIGGER audit_log_no_delete BEFORE DELETE ON audit_log
    FOR EACH ROW EXECUTE FUNCTION audit_log_immutable();
```

# Backend Architecture (`ledgius-api`)

## JWT claim shape

The access token payload extends the existing R-0041 claims with active-role
fields:

```json
{
  "sub": "<user-uuid>",
  "email": "...",
  "tenant_id": "<uuid-or-null>",     // active tenant — populated for customer + accountant
  "active_role": "customer",          // population identifier
  "active_role_scope_id": "<id-or-*>", // tenant_id / region_id / *
  "available_roles": [                // for the role-switch UI; not used for authorisation
    {"population": "customer", "tenant_id": "..."},
    {"population": "accountant", "tenant_id": "..."}
  ],
  "iat": ..., "exp": ..., "jti": "..."
}
```

Token rotation on role switch: a `POST /auth/switch-role` endpoint mints a
new access token with the requested `active_role` + `active_role_scope_id`,
provided the user holds a matching un-revoked population assignment.

## Middleware pipeline

```
middleware.RequestID                  // assigns X-Request-ID
middleware.Recovery                   // panic recovery
middleware.Logger                     // structured slog
middleware.AuthRequired               // JWT verification (R-0041 existing)
middleware.LoadActiveRole(platformDB) // NEW — validates active_role assignment is still valid
middleware.LoadAppSurface             // NEW — reads X-Client-App, defaults to "api"
middleware.AuditContext               // NEW — populates audit context for downstream writes
middleware.LoadFeatures               // existing (R-0041, used by RequireFeature)
middleware.TenantAuth                 // existing
```

Each handler that performs a state-changing operation MUST resolve audit
context from the request context and pass it into the audit write. A new
helper `audit.WriteFromCtx(ctx, action, entity, before, after)` reads the
five universal fields automatically.

## Mobile-surface enforcement

`middleware.LoadAppSurface` reads `X-Client-App`. When the value is
`mobile-ios` or `mobile-android`, a follow-up check in the auth middleware
asserts that the active_role is `customer` — otherwise reject with 403
`mobile-not-permitted-for-role`. The mobile login endpoint applies the same
check at credential verification time so users see an actionable error
rather than being authenticated and then rejected at first request.

## Cross-tenant operation

When `active_role = accountant`, the active tenant is the `tenant_id`
claim. `TenantAuth` middleware loads the appropriate tenant DB connection.
Switching tenants (within the accountant's allocated set) issues a new
access token via `POST /auth/switch-tenant` — the same path as for a
multi-tenant customer user.

# Frontend Architecture (`ledgius-web-app`)

- **PopulationPicker** component: shown after login when the user has ≥2
  populations available. Stores chosen role in localStorage as a session
  preference; auto-selects on next login if still valid.
- **Header role badge**: shows current active role; clicking opens an
  inline switcher.
- **Routing guards**: `RequireRole(...populations)` HOC for routes that
  are population-specific (e.g. plan admin pages require `platform_owner`).

# Mobile Architecture (`ledgius-mobile-app`)

- Login flow: standard email/password → POST `/auth/login` → check response
  for `available_populations`. If `customer` is present, proceed with
  customer-only token. Otherwise show "This account isn't enabled for the
  mobile app — please use the web app at app.ledgius.com".
- App never sends a non-`customer` `active_role` claim — server enforces
  but client never even attempts.

# Migration Strategy

1. Apply platform migration `101_identity_roles.sql` (idempotent).
2. Apply tenant audit-log migration to every tenant DB as part of the
   standard migrate sweep.
3. Backfill: for every existing user in `tenant_memberships`, insert a
   `population_assignments` row with `population='customer'`,
   `tenant_id=<their tenant>`, `intra_role=<their existing role>`. Mark
   the script as a one-time migration, version-pinned.
4. Backfill `audit_log` historical rows: `active_role='customer'`,
   `app_surface='web'`, `actor_label='backfill-r0052'` for any row where
   the new fields are NULL.
5. Roll out new middleware + JWT shape behind a feature flag
   (`identity_v2`); old tokens issued before the rollout still validate
   for the access-token TTL window. After the TTL window, remove the
   compatibility shim.
6. Mark R-0041 as Superseded; do not delete (specs are append-only history).

# Risks / Trade-offs

| Risk | Mitigation |
|---|---|
| JWT bloat from `available_roles` list | Cap at top-N + a "more available" flag; full list via API |
| audit_log table size after backfill | Run backfill in batches with pacing; non-blocking online |
| Multi-population user picks wrong role and accidentally posts as customer when meaning to act as accountant | Visual treatment of role badge; banner "operating as <role>"; audit row makes it discoverable post-hoc |
| Region taxonomy churn (e.g. ATO redefines jurisdictions) | `parent_region_id` allows hierarchical reorg; assignments migrate with the region row |
| Platform owner accidentally exposed to tenant business data | Hard-rule in handlers: `active_role=platform_owner` is NEVER allowed on tenant-business endpoints (account, journal, invoice, etc.). Enforced via a deny-list middleware on the platform-owner role |
