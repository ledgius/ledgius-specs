# Terms

This spec introduces vocabulary that is referenced by R-0053, A-0028,
A-0029 and T-0027. Definitions here are the canonical source.

| Term | Definition |
|---|---|
| **Population** | A top-level category of user with a distinct relationship to the platform (e.g. customer, accountant, region manager, platform owner). A population is *not* a role — it is the broad class a user belongs to. The list is closed under this spec but designed to be extensible (see "Future populations" note below). |
| **Role** | Within a population, the fine-grained rank that determines what actions the user may take (e.g. within the `customer` population: `owner`, `master_accountant`, `accountant`, `bookkeeper`, `viewer`). Role exists *inside* a population, never across populations. |
| **Population assignment** | The persistent record binding a single user to a single population, with the appropriate scope (tenant, region, or global). A user may hold many population assignments simultaneously. |
| **Scope** | The bound on a population assignment — `tenant_id` for `customer` and `accountant`, `region_id` for `region_manager`, `region_id` or `*` for `platform_owner`. The scope answers "this assignment applies to *what*?" |
| **Active role** | The single (population, scope) pair the user has selected for the current session. Encoded in the JWT (`active_role` + `active_role_scope_id`). All authorisation and audit attribution flows from the active role. The legacy name "role" in the JWT claim is preserved for compatibility but the value carries the population identifier. |
| **App surface** | The originating client kind for a request: `web`, `mobile-ios`, `mobile-android`, `api`, `system`. Captured on every audit row so identical API calls from different clients are distinguishable post-hoc. |
| **Customer user** | A member of the `customer` population — operates a single tenant's books. Mobile access permitted. |
| **Accountant** (population) | A member of the `accountant` population — cross-tenant access to an allocated set of tenants for review and correcting journals. Mobile access not permitted. Distinct from the *intra-tenant role* of the same name within the customer population — disambiguation is by always pairing role with population in code. |
| **Region manager** | A member of the `region_manager` population — administers accountant role assignments within a region. Cannot post journals or read tenant business data directly. |
| **Platform owner** | A member of the `platform_owner` population — administers pricing plans, features, and regional plan assignments (see R-0053). Cannot post journals or read tenant business data directly. |
| **Region** | A geographic / market scope used by `region_manager` and `platform_owner` populations and by R-0053's plan-region assignments. Hierarchical (`parent_region_id`). |

> **Future populations**: the schema accommodates new populations (e.g.
> `support_read`, `auditor_external`, `partner_integrator`,
> `system_observer`) by extending the CHECK list on
> `population_assignments.population` plus the scope-shape rules. New
> populations should be added by amending IDR-001 and IDR-003 in this
> spec, and the corresponding CHECK constraints in A-0028.

# Summary

Defines the four-population user model for Ledgius (standard customer users,
accountants/tax agents, region managers, platform owners), the cross-tenant
scoping rules, the surface restrictions (mobile is customer-only), and the
universal audit-logging contract that captures who/when/what/app/role on
every state-changing operation across the entire system.

This spec supersedes R-0041 ("Authentication and Role-Based Access Control")
which described an intra-tenant five-tier role model. The customer-user
population retains the intra-tenant role granularity from R-0041 verbatim;
this spec wraps it inside the broader four-population model and adds the
cross-tenant populations, plus the audit-surface fields.

# Problem / Business Context

Ledgius serves four distinct user populations with materially different access
patterns:

1. **End customers** running their own books inside one tenant
2. **Accountants and tax agents** reviewing many tenants' books and posting
   correcting journals
3. **Region managers** administering accountants within a geographic / market
   region
4. **Platform owners** running the Ledgius business itself: pricing plans,
   feature toggles, region/jurisdiction assignment

The previous five-tier RBAC model (owner / master_accountant / accountant /
bookkeeper / viewer) only described *intra-tenant* role granularity. It said
nothing about cross-tenant accountants, regional management, or platform
administration — three populations that are now first-class.

Concurrently, regulatory and dispute-resolution needs require that *every*
operation across the system be traceable not just to a user but to the
**role context** the user was operating under at action time, and the **app
surface** that originated the action. A cross-role user (e.g. an accountant
who is also a customer of their own firm's books) takes very different
actions in different role contexts, and audit must distinguish them.

# Scope

- The four user populations and how they are modelled (single users + role
  assignments at different scopes)
- Cross-tenant role assignments (one accountant ↔ many tenants)
- Region-scoped management roles
- Platform-owner role with plan/feature administration powers
- Surface restrictions (mobile is customer-only at this stage)
- Active-role-at-action-time semantics
- Universal audit fields (who / when / what / app / role) on every audit row
- Token claims that carry active-role + active-app
- API request middleware that captures and validates the active-role + active-app
- Migration path from R-0041's intra-tenant model

# Out of Scope

- Self-service signup and payment collection (R-0048 owns this)
- The implementation of pricing plan administration (R-0053 owns this)
- Multi-factor authentication, password reset, email verification (future)
- OAuth/SSO providers beyond what R-0041 already documents
- Audit log retention/archival policy
- Tamper-proof cryptographic chaining of audit entries
- Real-time audit event streaming

# Actors / Users

The four populations:

| Population | Tenant scope | Surfaces | Primary purpose |
|---|---|---|---|
| Customer user | Single tenant | web, mobile-ios, mobile-android | Operate their tenant's books |
| Accountant / tax agent | Many tenants (allocated) | web | Review + post correcting journals |
| Region manager | Region (covers many accountants and their tenants) | web | Manage accountants in a region |
| Platform owner | All tenants in their assigned regions | web | Plan/feature administration |

# Functional Requirements

## Population definitions

- **IDR-001**: System SHALL recognise four user populations: `customer`, `accountant`, `region_manager`, `platform_owner`. These are the top-level discriminators on every user account.
- **IDR-002**: A single `users` row MAY have multiple population assignments simultaneously (e.g. an accountant who is also a customer of their own firm's tenant). Each assignment is distinct and audited separately.
- **IDR-003**: Each population assignment MUST carry its own scope record:
    - `customer` → exactly one `tenant_id`
    - `accountant` → set of `tenant_id` values (the allocated tenants)
    - `region_manager` → exactly one `region_id`
    - `platform_owner` → set of `region_id` values, or `*` for global
- **IDR-004**: Within the `customer` population, the intra-tenant role granularity from R-0041 is retained: `owner`, `master_accountant`, `accountant`, `bookkeeper`, `viewer`. (The intra-tenant `accountant` role within the `customer` population is distinct from the cross-tenant `accountant` *population* — naming is preserved for backward compatibility but the disambiguation must be enforced in code by always pairing role with population.)

## Active role at action time

- **IDR-005**: When a user has multiple population assignments, the user MUST select an *active role* at session start (login or tenant-switch). The active role determines which scopes apply to subsequent requests.
- **IDR-006**: The active role MUST be encoded in the JWT access token as `active_role` (population identifier) and, where applicable, `active_role_scope_id` (tenant_id, region_id, or `*`).
- **IDR-007**: Switching active role within a session MUST issue a new access token reflecting the change. The previous token remains valid until natural expiry but the audit log will show the role transition as a discrete event.

## Surface restrictions

- **IDR-008**: Only the `customer` population MAY authenticate from mobile-app surfaces. The mobile login endpoint MUST reject any user whose only available populations are accountant / region_manager / platform_owner. Such users MAY still have a customer assignment that mobile login resolves to.
- **IDR-009**: The active app surface MUST be inferred from the request context (header `X-Client-App` set by every official client) and stored on every audit row. Accepted values: `web`, `mobile-ios`, `mobile-android`, `api`, `system`.
- **IDR-010**: System SHALL reject requests where the active role + active app combination is forbidden by IDR-008 (e.g. `active_role=platform_owner` + `app=mobile-ios`).

## Cross-tenant operation

- **IDR-011**: An `accountant` operating against a tenant MUST set the active tenant explicitly (via tenant-switch); subsequent reads and writes are scoped to that tenant. The audit row MUST record both the accountant's user_id and the active tenant_id.
- **IDR-012**: A `region_manager` MAY administer accountant role assignments within their region. They MAY NOT post journals or read tenant data directly.
- **IDR-013**: A `platform_owner` MAY administer plans, features, and regional assignments. They MAY NOT post journals or read tenant business data directly. (Read access for support purposes, if any, lives in a separate `support_read` role to be defined later.)

## Universal audit fields

- **IDR-014**: Every audit log entry (per R-0040) MUST capture:
    - `user_id` — acting user identity (UUID)
    - `created_at` — TIMESTAMPTZ
    - `action` — verb (create / update / delete / post / login / role_switch / ...)
    - `entity_type` + `entity_id` — what changed
    - `before_json` + `after_json` — change snapshot
    - `ip_address` — source IP
    - `tenant_id` — the active tenant context (NULL for cross-tenant or platform actions)
    - **`active_role`** (NEW) — population identifier in effect at action time
    - **`active_role_scope_id`** (NEW) — tenant_id / region_id / `*` corresponding to active_role
    - **`app_surface`** (NEW) — `web` / `mobile-ios` / `mobile-android` / `api` / `system`
    - `request_id` — correlation id for cross-row tracing
- **IDR-015**: System operations (background jobs, scheduled tasks, webhooks) MUST audit with `active_role='system'` and `app_surface='system'`. The originating job/handler name SHOULD be captured in `actor_label`.
- **IDR-016**: System SHALL NOT accept any state-changing API request that lacks the audit context required by IDR-014. Middleware MUST populate the missing fields or reject the request with 500 (developer error — unaudited code path).

## Migration from R-0041

- **IDR-017**: All existing users in the implemented R-0041 model MUST be migrated to the `customer` population, with their existing tenant + role preserved as the customer-population intra-tenant role. The migration is non-destructive.
- **IDR-018**: A migration script SHALL backfill `active_role='customer'` and `app_surface='web'` for any historical audit rows that lack the new fields, with a note in `actor_label='backfill-r0052'`.

# Non-Functional Requirements

- **IDR-NFR-001**: Active-role lookup MUST add no more than 5ms of middleware latency at p95.
- **IDR-NFR-002**: Audit writes MUST NOT block the originating request transaction (write to a separate connection / async flush is acceptable provided durability is guaranteed within 1s).
- **IDR-NFR-003**: The audit log MUST remain append-only. Updates and deletes are forbidden at the database level (CHECK constraint or trigger).

# Data / Entities / Fields

See A-0028 for the canonical schema. High-level new entities introduced by
this spec:

- `regions` — geographic / market regions
- `population_assignments` — per-user population + scope mapping (replaces a
  flat `role` column on users)
- `audit_log` extended with `active_role`, `active_role_scope_id`,
  `app_surface`, `request_id`, `actor_label`

# UX / UI Behaviour

- **Login flow**: when a user has multiple populations available, present a
  population-picker after credentials are accepted. Single-population users
  skip the picker.
- **Role-switch UI**: in the header for users with ≥2 populations, allow
  switching active role inline. Switching forces a token refresh.
- **Mobile login**: presents a single tenant-picker; if the authenticated
  user has no `customer` population, return "This account is not enabled for
  the mobile app" rather than logging them in with reduced scope.
- **Audit log viewer**: every entry shows the active_role + app_surface as
  pill badges next to the user name.

# Validation Rules

- A user with zero population assignments cannot log in — return 403 with
  `account-not-provisioned` error code.
- `active_role_scope_id` is mandatory for `customer` and `region_manager`
  populations; optional (`*` allowed) for `accountant` (allows multi-tenant
  view) and `platform_owner` (allows global).
- Switching active role for a population the user does not hold returns 403.

# Error Handling

- Missing `X-Client-App` header on a state-changing API call: the API
  defaults to `app_surface=api` (third-party / internal tooling). Official
  clients are expected to set the header.
- Inconsistent active_role between JWT and request context (e.g. someone
  forging a JWT): middleware rejects with 401 and audits a `token_tamper`
  event.

# Security / Permissions

- Population assignments are platform-DB managed; only `platform_owner` and
  (for accountant/customer assignments within their region) `region_manager`
  may modify them.
- Switching active role MUST be re-authenticated for `platform_owner`
  population — additional MFA check is a future enhancement (out of scope).
- Audit log read endpoints respect the role filter: a customer sees only
  their tenant's actions; an accountant sees their allocated tenants; a
  region manager sees their region; a platform owner sees all.

# Acceptance Criteria

- [ ] Four populations recognised in tokens, middleware, and audit
- [ ] Cross-tenant accountant can switch active tenant and post a
      correcting journal that audits with `active_role=accountant` +
      `tenant_id=<the-tenant>` + `active_role_scope_id=<the-tenant>`
- [ ] Region manager can administer accountant assignments within their
      region; cannot post journals
- [ ] Platform owner can administer plans/features (depends on R-0053);
      cannot read tenant business data
- [ ] Mobile login rejects accountant-only / region-manager-only /
      platform-owner-only users; accepts users with a `customer`
      population
- [ ] Every audit row populated for at least 30 days carries `active_role`,
      `active_role_scope_id`, `app_surface`, `request_id`
- [ ] R-0041 marked Superseded by R-0052 with no functional regression
      for existing customer users

# Open Questions

1. **Region taxonomy**: do regions correspond to ATO jurisdictions
   (states/territories), to market segments (SMB/midmarket), or both?
   R-0053 will need this to assign plans by region.
2. **Backfill of historical audit rows** (IDR-018): is `active_role=customer`
   + `app_surface=web` a safe assumption for all pre-migration rows? Need
   to confirm there's no historical accountant access prior to this spec.
3. **Audit table size impact**: adding 4 new columns to `audit_log` is
   non-trivial on a large table. Migration plan must be online (no rewrite)
   or scheduled during maintenance window.
4. **Accountant read-everything model**: an accountant allocated to tenants
   A and B can switch between them — but can they query "all journals across
   all my tenants" in one go? Useful for batch review, but cross-tenant
   reads require care to avoid data leakage. Default: no, must switch
   tenant. Confirm before implementation.
