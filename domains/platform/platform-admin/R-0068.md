# Summary

A platform administration console accessible to users with the
`platform_owner` role, surfaced as a dedicated sidebar mode in the
Ledgius web app (app.ledgius.com). Covers the full tenant lifecycle:
sign-up review, tenant provisioning, user management, pricing plan
configuration, feature gating, database operations, and a
hierarchical tenant directory view.

# Terms

| **Term** | Definition. |
|---|---|
| **Platform DB** | The shared `ledgius_platform` database holding tenants, platform users, pricing plans, and feature gates. Not tenant-specific. |
| **Tenant DB** | An isolated PostgreSQL database per tenant, created from the slug algorithm. Contains all accounting data for that tenant. |
| **Tenant slug** | A URL-safe, lowercase identifier derived from the business name (e.g., `maiya-nail-studio`). Used as the database name and filesystem directory name. |
| **Feature gate** | A named capability (e.g., `bank_feeds`, `payroll`, `api_export`) that can be enabled/disabled per tenant based on their pricing plan. |
| **Test tenant** | A tenant flagged as `is_test = true`. Not billed via Stripe. Used for internal testing, demos, and onboarding preparation. |
| **Real tenant** | A tenant flagged as `is_test = false`. Has an active Stripe subscription. Represents a paying customer. |
| **PITR** | Point-in-Time Recovery — PostgreSQL's ability to restore a database to any moment using WAL (Write-Ahead Log) archives. |

## Lifecycle State Machines

### Sign-up States

```
pending_review → accepted → provisioning_started → provisioning_failed
                                                  → provisioned
pending_review → rejected
```

| From | To | Trigger |
|---|---|---|
| `pending_review` | `accepted` | Admin accepts sign-up (PA-002) |
| `pending_review` | `rejected` | Admin rejects sign-up (PA-002) |
| `accepted` | `provisioning_started` | Provisioning worker picks up accepted sign-up |
| `provisioning_started` | `provisioned` | All provisioning steps complete successfully |
| `provisioning_started` | `provisioning_failed` | Any provisioning step fails (retryable) |

### Tenant States

```
pending → provisioning → provisioning_failed
                        → active → trial → suspended → restore_in_progress
                                                      → deletion_pending → deleted
```

| From | To | Trigger |
|---|---|---|
| `pending` | `provisioning` | Provisioning pipeline starts |
| `provisioning` | `provisioning_failed` | Database/directory creation or migration failure |
| `provisioning` | `active` | Provisioning completes, tenant ready |
| `active` | `trial` | Trial period assigned (based on plan) |
| `trial` | `active` | Trial expires and billing starts |
| `active` | `suspended` | Payment failure or admin action |
| `suspended` | `active` | Payment resolved or admin restores |
| `active` | `restore_in_progress` | Backup restore initiated |
| `restore_in_progress` | `active` | Restore completes successfully |
| `active` | `deletion_pending` | Admin initiates deletion (PA-050) |
| `deletion_pending` | `deleted` | Deletion confirmed and executed |

# Problem / Business Context

Ledgius is a multi-tenant SaaS platform. As tenants sign up, the
platform needs administrative tooling to:
- Review and approve new sign-ups (manual for now, automated later)
- Provision tenant databases and filesystem directories
- Manage users across platform and tenant boundaries
- Configure pricing plans that control feature access
- Operate on tenant databases (backup, restore, delete)
- View the tenant portfolio in a geographic hierarchy

Without this tooling, every tenant operation requires direct database
access and manual SQL — error-prone, unaudited, and unsustainable
beyond a handful of tenants.

# Scope

## 1. Sign-up Review (PA-001 through PA-005)

- **PA-001**: The platform admin console shall display a queue of
  pending sign-ups with: business name, contact name, email, phone,
  ABN, business type, location (state/town), requested plan,
  sign-up timestamp.
- **PA-002**: Each sign-up can be **accepted** (proceeds to tenant
  provisioning) or **rejected** (with reason — email sent to
  applicant).
- **PA-003**: The admin shall be able to view the full sign-up
  details including any notes, referral source, and validation
  checks. Validation fields stored on the sign-up record:
  - `abn_validation_status` — result of ABN lookup (`pending`,
    `valid`, `invalid`, `not_checked`)
  - `email_verification_status` — email verification state
    (`pending`, `verified`, `bounced`, `not_sent`)
  - `risk_score` — numeric risk score (0-100) from automated
    checks (v2), displayed but not actioned in v1
  - `review_notes` — free-text notes added by the reviewing
    admin during the review process
- **PA-004**: Accepted sign-ups transition to tenant provisioning
  (PA-010). Rejected sign-ups are retained for audit with the
  rejection reason and admin who rejected.
- **PA-005**: In future this process will be automated with
  risk-scoring and auto-approval. The manual queue is the v1
  implementation. The UI shall be designed to accommodate both
  workflows.

## 2. Tenant Provisioning (PA-010 through PA-016)

- **PA-010**: Tenant provisioning creates or completes the tenant
  record in the platform DB. The `ledgius.com/signup` process
  creates a partial record; the admin completes it.
- **PA-010a**: All provisioning steps are **async operations**
  orchestrated via the `platform_operation` table. Each step
  (DB creation, migration, directory creation, user seeding) is
  recorded as a discrete operation with status tracking
  (`pending`, `in_progress`, `completed`, `failed`). Failed
  operations are retryable without re-running completed steps.
- **PA-010b**: Provisioning is executed by a **provisioning worker**
  process that holds elevated database credentials (CREATEDB,
  superuser). The web API does NOT hold these credentials — it
  writes operation requests to `platform_operation`, and the
  worker picks them up. This privilege separation ensures the
  web-facing API cannot be exploited to create or drop databases.
- **PA-011**: The admin shall be able to set: business name,
  trading name, ABN, business type, address (street, city, state,
  postcode), contact details, timezone (default: Australia/Sydney),
  financial year start (default: July).
- **PA-012**: A **tenant slug** is auto-generated from the business
  name using the slug algorithm:
  - Maximum 63 characters (PostgreSQL database name limit).
  - ASCII transliteration of unicode characters (e.g.,
    "Muller" from "Muller", "cafe" from "cafe").
  - Lowercase, hyphens replace spaces/separators, no special chars.
  - Reserved-word blocklist: `postgres`, `template0`, `template1`,
    `ledgius_platform`, `admin`, `platform`, `system`, `test`,
    `null`, `undefined`.
  - Uniqueness enforced — append `-2`, `-3`, etc. if slug exists.
  - **Immutable after provisioning** — the slug cannot be changed
    once the tenant database has been created, as it is the
    database name and directory name.
  - The admin can override the slug before provisioning only.
- **PA-013**: **Tenant database creation** — the system creates a
  new PostgreSQL database using the tenant slug as the database
  name. Flyway migrations (V1.00 through latest) are applied
  automatically. Expected provisioning time: 20-30 seconds.
- **PA-014**: **Tenant directory creation** — a filesystem directory
  is created on the data volume using the tenant slug
  (`/data/{tenant-slug}/`). Used for tenant-specific file storage
  (exports, uploads, backups).
- **PA-015**: The provisioning process shall display a progress
  indicator showing: creating DB → applying migrations → creating
  directory → seeding defaults → complete.
- **PA-016**: Each tenant record carries an `is_test` flag:
  - `is_test = true` — test tenant, no Stripe billing applied.
    Used for internal testing, demos, onboarding preparation.
  - `is_test = false` — real tenant, Stripe subscription required.
    Represents a paying customer.

## 3. User Management (PA-020 through PA-027)

- **PA-020**: The admin shall be able to create platform-level users
  with roles: `platform_owner`, `accountant`, `region_manager`.
- **PA-021**: The admin shall be able to create tenant-level users
  with roles: `business_owner`, `bookkeeper`, `viewer`.
- **PA-022**: The **platform DB is the source of truth** for all
  user records. Tenant DB user rows are **synced projections**
  maintained by a retryable, idempotent sync job. When a user is
  created or updated in the platform DB, the sync job propagates
  the change to each assigned tenant DB. If a sync fails, it is
  retried with exponential backoff. The tenant DB user table
  includes a `synced_at` timestamp and `platform_user_id` foreign
  reference to detect and repair drift.
- **PA-023**: The admin shall be able to add additional users to a
  tenant (e.g., adding an accountant to a client's tenant).
- **PA-024**: User listing shows: name, email, role, assigned
  tenants, last login, status (active/disabled).
- **PA-025**: **Accountant assignment** — the admin can assign a
  tenant to an accountant user. This sets the
  `accountant_user_id` on the tenant record. An accountant can
  have multiple tenants assigned.
- **PA-026**: The admin shall be able to disable/enable user
  accounts without deleting them.
- **PA-027**: All user management operations are audit-logged with
  who, when, and what changed.

## 4. Pricing Plans and Feature Gates (PA-030 through PA-039)

- **PA-030**: The admin shall be able to create and edit pricing
  plans. Each plan defines:
  - Plan name (e.g., "Starter", "Professional", "Enterprise")
  - Plan code (slug, unique)
  - Monthly subscription price (AUD, inc GST)
  - Yearly subscription price (AUD, inc GST)
  - Monthly discount percentage + effective period (from/until)
  - Yearly discount percentage + effective period (from/until)
  - Trial period (days, default 30)
  - Effective from date and until date (plan validity period)
- **PA-030a**: Each plan shall also carry **display metadata** for
  rendering pricing cards on the public website (ledgius.com):
  - Tagline (short description, e.g., "For sole traders getting started")
  - Feature bullet points (human-readable, ordered for display)
  - Highlight flag (marks the "recommended" plan)
  - Badge text (e.g., "Most Popular", "Best Value")
  - CTA button text (e.g., "Start Free Trial", "Get Started")
  - Display order (integer, controls card positioning)
  - Inherits-from plan (optional FK, renders "Everything in X plus:")
  The pricing plan API shall expose a public (unauthenticated)
  endpoint `GET /pricing/plans` that returns plan data suitable
  for rendering pricing cards without exposing internal feature
  gate identifiers.
- **PA-030c**: The pricing plans admin page shall include a
  **"Preview"** tab that renders plan cards exactly as the public
  pricing page displays them (R-0069 PP-050). This enables the
  admin to verify content, ordering, and inheritance presentation
  before it goes live. The preview fetches from the same regional
  pricing API used by the public website.
- **PA-030b**: **Plan versioning** — a pricing plan is split into
  two concepts:
  - `pricing_plan` — stable identity (name, code, display
    metadata). Does not change when commercial terms are updated.
  - `pricing_plan_version` — dated commercial terms for a plan
    (prices, discounts, trial period, effective dates). A new
    version is created when pricing changes; existing subscribers
    remain on their assigned version until explicitly migrated.
  Features are split into three categories:
  - **Capabilities** — boolean flags (e.g., `bank_feeds = true`).
    The feature is either available or not.
  - **Entitlements** — typed limits (e.g., `max_users = 5`,
    `max_transactions = 1000`). Enforced at runtime.
  - **Marketing metadata** — display-only data (tagline, bullets,
    badge, CTA) that lives on the plan, not the version.
- **PA-031**: Each plan version has a **feature set** — a list of
  capabilities (boolean) and entitlements (typed limits) that are
  enabled for that version. Feature gates are real identifiers
  understood by the API and web app for allow/disallow decisions.
- **PA-032**: Feature gate identifiers include (not exhaustive):
  - `bank_feeds` — live bank feed connections
  - `bank_reconciliation` — bank reconciliation module
  - `payroll` — payroll and STP
  - `super_guarantee` — super obligations tracking
  - `bas_lodgement` — BAS lodgement from within app
  - `api_import` — API-based data import (Xero/MYOB/QuickBooks)
  - `api_export` — API-based data export
  - `multi_currency` — foreign currency support
  - `inventory` — inventory management
  - `fixed_assets` — fixed asset register + depreciation
  - `smart_reconciliation` — AI-assisted bank reconciliation
  - `document_ocr` — receipt/invoice OCR capture
  - `custom_reports` — custom report builder
  - `api_access` — external API access for integrations
  - `max_users` — maximum number of users (integer value)
  - `max_transactions` — monthly transaction limit (integer)
- **PA-033**: The admin assigns a pricing plan to a tenant. The
  tenant's enabled features are derived from their plan's feature
  set.
- **PA-034**: Feature gates are checked at both API (middleware)
  and UI (conditional rendering) layers. The API returns a
  `403 Feature Not Available` when a gated endpoint is accessed
  by a tenant without that feature.
- **PA-035**: The admin can override individual feature gates per
  tenant (e.g., enable `payroll` for a Starter plan tenant as a
  special arrangement). Overrides are tracked with reason and
  expiry date.
- **PA-036**: Plan pricing changes do not retroactively affect
  existing subscribers. The effective date controls when new
  pricing takes effect.
- **PA-037**: Trial period: when a tenant is provisioned, the trial
  end date is set based on the plan's trial period. During trial,
  all plan features are available. After trial, billing begins.
- **PA-038**: Stripe integration: real tenants (`is_test = false`)
  have a Stripe customer ID and subscription ID on their tenant
  record. The admin can view subscription status but billing
  management is via Stripe dashboard (v1).
- **PA-039**: The admin can change a tenant's plan. The change
  takes effect immediately — feature gates update, billing
  adjusts at next cycle.

## 5. Database Operations (PA-040 through PA-046)

- **PA-040**: The admin shall be able to trigger a **manual backup**
  of any tenant database. The backup is a `pg_dump` compressed
  archive stored in the tenant's data directory.
- **PA-041**: The admin shall be able to **restore from backup** —
  selecting from a list of available backups with timestamps.
  Restore uses a **fresh target database** approach:
  1. Place tenant in `restore_in_progress` state (read-only).
  2. Create a new database from the backup (e.g., `{slug}_restore`).
  3. Validate the restored database (schema version, row counts,
     integrity checks).
  4. Switch routing: rename old DB to `{slug}_pre_restore`,
     rename restored DB to `{slug}`.
  5. Keep old database for a configurable retention period
     (default 7 days) before dropping.
  Restore requires confirmation ("Type tenant name to confirm").
- **PA-042**: **PITR / WAL support** — the system shall support
  point-in-time recovery using PostgreSQL WAL archives. The admin
  specifies a target timestamp and the system restores to that
  point.
- **PA-042a**: **PITR hosting validation** — before offering PITR
  as a restore option, the system shall validate that the hosting
  environment supports WAL archiving (e.g., Fly.io Postgres
  clusters with WAL-G configured). If WAL archiving is not
  available, PITR options are hidden and the admin is informed
  that only file-based backups are available.
- **PA-043**: **Google Drive backup** — the admin can zip a backup
  and send it to a configured Google Drive folder. Requires
  Google Drive API credentials configured at the platform level.
- **PA-044**: Backup history shows: timestamp, size, type (manual/
  scheduled/PITR), storage location, retention status.
- **PA-045**: **Scheduled backups** — the admin can configure a
  backup schedule per tenant (e.g., daily at 2am AEST). The
  schedule runs as a background cron job.
- **PA-046**: Backup retention policy: keep last N backups, or
  keep backups for N days. Configurable per tenant.

## 6. Tenant Deletion (PA-050 through PA-053)

- **PA-050**: Tenant deletion is a **heavily guarded** operation.
  The admin must:
  1. Acknowledge warnings about data loss
  2. Type the tenant name to confirm
  3. Provide a deletion reason
  4. A final backup is automatically created before deletion
- **PA-051**: Deletion removes: the tenant database, the tenant
  directory, all tenant user associations, and marks the tenant
  record as `status = 'deleted'` in the platform DB (soft delete
  for audit trail).
- **PA-052**: Deleted tenants cannot be restored via the UI. The
  final backup can be used for manual recovery if needed.
- **PA-053**: All deletion operations are audit-logged with full
  context: who, when, reason, backup reference.

## 7. Tenant Directory View (PA-060 through PA-065)

- **PA-060**: The tenant listing shall be displayed in a **tree
  hierarchy**: Australia → State → Town → Tenant Name.
- **PA-061**: Each tenant node shows:
  - Business name and trading name
  - Tenant status (active/trial/suspended/deleted)
  - `is_test` flag (test or real)
  - Under the tenant:
    - **Payment status** — current, overdue, trial, not billed (test)
    - **Pricing plan** — plan name, trial end date if applicable
    - **Users** — count and list of assigned users with roles
    - **Feedback** — count of unresolved feedback items (sourced
      from the tenant DB `feedback` table)
    - **Issues** — count of open issues/support tickets (sourced
      from the platform DB `support_ticket` table)
- **PA-062**: The tree is collapsible/expandable. Clicking a tenant
  navigates to the tenant detail page.
- **PA-063**: Search and filter: search by business name, ABN,
  email. Filter by state, plan, status, test/real.
- **PA-064**: Summary statistics at the top: total tenants,
  active, trial, test, total MRR (monthly recurring revenue).
- **PA-065**: The admin can quick-action from the tree: assign
  accountant, change plan, disable tenant.

## 8. Sidebar Mode (PA-070)

- **PA-070**: A new sidebar mode "Platform" shall appear for users
  with `platform_owner` role only. Contains menu items:
  - Sign-ups
  - Tenants
  - Plans
  - Users
  - Operations
  - Settings

## 9. Multi-Tenant Access (PA-080 through PA-085)

- **PA-080**: Users with `platform_owner` and `accountant` roles
  have access to multiple tenants. After login, these users see a
  **tenant selector** — a dropdown or switcher that lets them
  choose which tenant context to operate in.
- **PA-081**: The JWT token shall include:
  - `role` — the user's platform role
  - `authorised_tenants` — a claim containing either a UUID array
    of permitted tenant IDs, or the string `"*"` for platform
    owners who can access all tenants.
  The client sends an `X-Tenant-ID` HTTP header on every request
  to indicate the active tenant context. The API validates that
  the `X-Tenant-ID` value is present in the JWT's
  `authorised_tenants` claim (or that the claim is `"*"`). If
  validation fails, the API returns `403 Tenant Access Denied`.
- **PA-082**: The tenant selector shall be visible in the app
  header for multi-tenant users. Single-tenant users (e.g.,
  `business_owner`) do not see it — they are locked to their
  tenant.
- **PA-083**: Switching tenants changes the database context for
  all subsequent API requests. The switch is instant — no
  re-authentication needed. The UI refreshes to show the selected
  tenant's data.
- **PA-084**: The tenant selector shows: tenant name, business
  type, and a test/real badge. Recent tenants appear at the top
  for quick access.
- **PA-085**: **Security**: a user can only access tenants they
  are explicitly assigned to (via `platform_user_tenant` mapping)
  or all tenants if they are a `platform_owner`. The API rejects
  requests where the JWT's tenant list does not include the
  requested tenant ID — returning `403 Tenant Access Denied`.

## 10. Platform Audit Model (PA-090 through PA-093)

- **PA-090**: All platform administration operations shall be
  recorded in a `platform_audit_event` table in the platform DB.
  Every auditable action produces an immutable event row.
- **PA-091**: Each audit event captures:
  - `event_type` — the operation performed (e.g.,
    `signup.accepted`, `tenant.provisioned`, `user.created`,
    `plan.changed`, `backup.triggered`, `tenant.deleted`,
    `feature_override.set`)
  - `actor_id` — UUID of the user who performed the action
  - `actor_email` — email of the acting user (denormalised for
    query convenience)
  - `target_type` — the entity type affected (`tenant`, `user`,
    `plan`, `signup`, `backup`)
  - `target_id` — the ID of the affected entity
  - `tenant_id` — the tenant context (nullable for
    platform-level operations)
  - `details` — JSONB payload with operation-specific data
    (before/after values, reason, metadata)
  - `ip_address` — source IP of the request
  - `user_agent` — request user-agent string
  - `created_at` — immutable timestamp
- **PA-092**: Audit events are **append-only** — no updates or
  deletes. The table has no UPDATE or DELETE grants for the
  application role.
- **PA-093**: The admin console shall provide an audit log viewer
  with filtering by event type, actor, target, tenant, and date
  range. Audit entries are displayed in reverse chronological
  order with expandable detail panels.

# Out of Scope

- Automated sign-up approval (v2 — risk scoring)
- Stripe billing management UI (v1 uses Stripe dashboard)
- Multi-region deployment (v1 is Australia-only)
- Tenant data migration between databases
- White-label / custom branding per tenant

# Actors / Users

- Platform Owner — full access to all admin functions
- Region Manager — view tenants in their region (future)
