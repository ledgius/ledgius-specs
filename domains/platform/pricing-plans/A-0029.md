# Terms

Vocabulary used in this spec (population, role, platform owner, region,
etc.) is defined canonically in **R-0052 §Terms**.

# Summary

Schema, API surface, and caching design for the platform-owner pricing-plan
administration system defined in R-0053. The catalogue replaces ad-hoc
env-var feature gating (e.g. R-0049's `BANK_FEED_DEFAULT_MAX_CONNECTIONS`)
with a database-driven, versioned, region-scoped model.

# Database Schema (Platform DB)

## Migration: `102_pricing_plans.sql`

```sql
-- Spec references: R-0053, A-0029.

-- =============================================================================
-- 1. Features catalogue + attributes
-- =============================================================================

CREATE TABLE IF NOT EXISTS features (
    slug         TEXT PRIMARY KEY,
    display_name TEXT NOT NULL,
    description  TEXT,
    category     TEXT,
    created_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at   TIMESTAMPTZ NOT NULL DEFAULT now()
);

COMMENT ON TABLE features IS
    'Universe of toggleable features. The slug is the authoritative ' ||
    'identifier referenced by code via middleware.RequireFeature(slug).';

-- Default attributes attached to a feature (e.g. bank_feeds.max_connections=3).
-- Plan versions may override these per plan (see plan_version_features).
CREATE TABLE IF NOT EXISTS feature_attributes (
    feature_slug TEXT NOT NULL REFERENCES features(slug) ON DELETE CASCADE,
    key          TEXT NOT NULL,
    value        TEXT NOT NULL,
    value_type   TEXT NOT NULL DEFAULT 'string'
        CHECK (value_type IN ('string', 'int', 'bool', 'json')),
    PRIMARY KEY (feature_slug, key)
);

COMMENT ON TABLE feature_attributes IS
    'Default attribute values for a feature. Overridable per plan version.';

-- =============================================================================
-- 2. Plans (slug + display) and versions (the priced/scheduled offerings)
-- =============================================================================

CREATE TABLE IF NOT EXISTS plans (
    slug         TEXT PRIMARY KEY,
    display_name TEXT NOT NULL,
    description  TEXT,
    created_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at   TIMESTAMPTZ NOT NULL DEFAULT now()
);

COMMENT ON TABLE plans IS
    'Stable product slug (e.g. "essential", "business"). Versioned in plan_versions.';

CREATE TABLE IF NOT EXISTS plan_versions (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    plan_slug       TEXT NOT NULL REFERENCES plans(slug) ON DELETE CASCADE,
    version         INT NOT NULL,
    region_id       UUID NULL REFERENCES regions(id) ON DELETE SET NULL,  -- NULL = global
    display_name    TEXT NOT NULL,                                         -- copy of plans.display_name at version time
    description     TEXT,
    billing_cycle   TEXT NOT NULL CHECK (billing_cycle IN ('monthly', 'annual', 'lifetime')),
    price           NUMERIC(10,2) NOT NULL,
    currency        TEXT NOT NULL DEFAULT 'AUD',
    effective_from  TIMESTAMPTZ NULL,    -- NULL while in draft
    effective_to    TIMESTAMPTZ NULL,    -- NULL = open-ended
    status          TEXT NOT NULL DEFAULT 'draft' CHECK (status IN ('draft', 'published', 'retired')),
    metadata        JSONB NOT NULL DEFAULT '{}'::jsonb,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    created_by      UUID NULL REFERENCES users(id),
    published_at    TIMESTAMPTZ NULL,
    published_by    UUID NULL REFERENCES users(id),
    UNIQUE (plan_slug, version, region_id)
);

COMMENT ON TABLE plan_versions IS
    'A single version of a plan, optionally region-scoped. Immutable after publish ' ||
    '(enforced by trigger). Resolution: tenant''s region → plan_slug → version with ' ||
    'effective_from <= now < effective_to.';

CREATE INDEX IF NOT EXISTS idx_plan_versions_resolve
    ON plan_versions(plan_slug, region_id, effective_from DESC)
    WHERE status = 'published';

CREATE INDEX IF NOT EXISTS idx_plan_versions_effective
    ON plan_versions(effective_from, effective_to)
    WHERE status = 'published';

-- Mapping: which features each plan version includes, with optional
-- per-version attribute overrides.
CREATE TABLE IF NOT EXISTS plan_version_features (
    plan_version_id UUID NOT NULL REFERENCES plan_versions(id) ON DELETE CASCADE,
    feature_slug    TEXT NOT NULL REFERENCES features(slug) ON DELETE RESTRICT,
    enabled         BOOLEAN NOT NULL DEFAULT true,
    PRIMARY KEY (plan_version_id, feature_slug)
);

CREATE TABLE IF NOT EXISTS plan_version_feature_attributes (
    plan_version_id UUID NOT NULL REFERENCES plan_versions(id) ON DELETE CASCADE,
    feature_slug    TEXT NOT NULL,
    key             TEXT NOT NULL,
    value           TEXT NOT NULL,
    value_type      TEXT NOT NULL DEFAULT 'string'
        CHECK (value_type IN ('string', 'int', 'bool', 'json')),
    PRIMARY KEY (plan_version_id, feature_slug, key),
    FOREIGN KEY (plan_version_id, feature_slug)
        REFERENCES plan_version_features(plan_version_id, feature_slug) ON DELETE CASCADE
);

COMMENT ON TABLE plan_version_feature_attributes IS
    'Per-plan-version override of a feature attribute. ' ||
    'When absent, the default from feature_attributes applies.';

-- =============================================================================
-- 3. Tenant ↔ plan version assignment
-- =============================================================================

CREATE TABLE IF NOT EXISTS tenant_plan_assignments (
    tenant_id        UUID PRIMARY KEY REFERENCES tenants(id) ON DELETE CASCADE,
    plan_version_id  UUID NOT NULL REFERENCES plan_versions(id) ON DELETE RESTRICT,
    assigned_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    assigned_by      UUID NULL REFERENCES users(id),
    notes            TEXT
);

COMMENT ON TABLE tenant_plan_assignments IS
    'Current plan-version assignment for each tenant. History of changes ' ||
    'lives in audit_log per R-0052 IDR-014.';

-- =============================================================================
-- 4. Immutability of published plan versions (trigger)
-- =============================================================================

CREATE OR REPLACE FUNCTION plan_version_immutable() RETURNS trigger LANGUAGE plpgsql AS $$
BEGIN
    IF OLD.status = 'published' AND (
        NEW.price          IS DISTINCT FROM OLD.price        OR
        NEW.billing_cycle  IS DISTINCT FROM OLD.billing_cycle OR
        NEW.currency       IS DISTINCT FROM OLD.currency      OR
        NEW.effective_from IS DISTINCT FROM OLD.effective_from
    ) THEN
        RAISE EXCEPTION 'plan_version % is published and cannot be edited; create a new version', OLD.id;
    END IF;
    RETURN NEW;
END $$;

DROP TRIGGER IF EXISTS plan_version_immutable_check ON plan_versions;
CREATE TRIGGER plan_version_immutable_check BEFORE UPDATE ON plan_versions
    FOR EACH ROW EXECUTE FUNCTION plan_version_immutable();
```

# Plan Resolution

Each request carries a tenant context (per R-0041 / R-0052). Resolving the
tenant's effective feature set:

```
1. Look up tenant_plan_assignments → plan_version_id.
2. (Cached per-request after first lookup; cached cross-request via in-process LRU with 30s TTL.)
3. Load plan_version_features + plan_version_feature_attributes for that version.
4. Merge with feature_attributes defaults (override wins).
5. Return FeatureSet { slug → enabled, slug.attribute → value }.
```

The existing `middleware.LoadFeatures` (R-0041) is updated to read from
this catalogue rather than the legacy `plan_features` table. The middleware
contract for handlers (`RequireFeature(slug)`, `FeatureAttribute(slug, key)`)
is preserved — call sites need no change beyond importing the new helpers.

## Cache invalidation

Three options, in order of preference:

1. **Postgres LISTEN/NOTIFY** on `pricing_catalogue_changed` channel.
   Plan publish + tenant migration emit NOTIFY; every API instance has a
   listener that flushes the LRU. Lowest latency, no extra infrastructure.
2. **Short TTL only** (30s). Simpler but worst-case staleness window of 30s.
3. **Per-request only**. Always re-fetches from DB. Safest but adds load.

Recommendation: option 1 + option 2 as defence in depth (TTL is the
backstop if NOTIFY is missed for any reason).

# API Architecture (`ledgius-api`)

## New domain: `internal/pricing/`

```
internal/pricing/
  model.go             — Plan, PlanVersion, Feature, TenantAssignment
  repository.go        — Catalogue queries (used by middleware + admin)
  resolver.go          — TenantPlan resolver with cache + LISTEN/NOTIFY
  admin_service.go     — Plan/feature/assignment mutations (platform_owner only)
  admin_handler.go     — HTTP handlers for /api/v1/admin/plans/*
  routes.go            — Route registration
```

## Updates to existing middleware

`pkg/middleware/feature_gate.go` (R-0041 implementation):

- `LoadFeatures` now reads from `internal/pricing.Resolver` (which encapsulates
  the catalogue lookup + cache). The legacy `plan_features` table is left
  in place but unused; a follow-up migration can drop it once telemetry
  confirms zero readers.
- New helper `FeatureAttribute(ctx, slug, key) (value string, ok bool)`
  for code paths that need a parameterised limit (e.g. bankfeed's
  per-tenant connection cap).

## Bank feeds integration (replaces R-0049 interim cap)

`internal/bankfeed/service.go` is updated to read the per-tenant cap from
the catalogue:

```go
// Old (R-0049 interim):
//   service.WithConnectionCap(cfg.BankFeed.DefaultMaxConnections)
// New (post-R-0053):
maxStr, ok := middleware.FeatureAttribute(ctx, "bank_feeds", "max_connections")
maxConns, _ := strconv.Atoi(maxStr)
if !ok { maxConns = 0 }  // unlimited if not specified
```

# Frontend Architecture (`ledgius-web-app`)

Admin-only pages mounted at `/admin/plans`, `/admin/features`,
`/admin/regions`, gated by `RequireRole("platform_owner")` (R-0052).

- **PlansListPage** — table of plan slugs × regions, with current
  effective version cells linking to detail
- **PlanVersionDetailPage** — read view + "Create new version from this"
- **PlanVersionEditPage** (draft only) — feature-toggle grid with
  attribute-override fields per feature
- **PublishPlanVersionDialog** — preview of effective_to truncation +
  affected tenant count
- **TenantMigrationPage** — bulk select tenants, choose target plan
  version, see feature delta per tenant, confirm

# Migration Strategy

1. Apply `102_pricing_plans.sql` (idempotent).
2. Seed the catalogue: insert one row per existing feature slug
   (currently: `bank_feeds`, plus whatever R-0041 / R-0049 reference).
3. Seed initial plans: `starter`, `essential`, `business` per current
   marketing copy. Region: `AU` (or NULL → global).
4. Seed `plan_version_features` per plan version per the current
   marketing matrix.
5. Backfill `tenant_plan_assignments` from the legacy `tenants.plan_id`
   column (R-0041 era).
6. Roll out `pricing.Resolver` behind a feature flag (`pricing_v2`); when
   stable, switch `LoadFeatures` to read from it exclusively.
7. Drop the legacy `plan_features` table in a follow-up migration after
   30 days of clean telemetry.

# Risks / Trade-offs

| Risk | Mitigation |
|---|---|
| Catalogue-driven resolution adds latency | LRU cache + LISTEN/NOTIFY invalidation; budget 5ms p95 (PLN-NFR-001) |
| Plan downgrade silently breaks tenant workflows | `plan-downgrade-notice` UI banner for 30 days post-change |
| Admin mistakes (price typo, wrong region) | All mutations audited; bulk migration has 24h rollback |
| Stripe ↔ catalogue divergence | R-0048 follow-up: publish-plan-version triggers Stripe sync; R-0053 stops at the catalogue itself |
| Region taxonomy depends on R-0052 having seeded `regions` | Document seed order in T-0027 |
