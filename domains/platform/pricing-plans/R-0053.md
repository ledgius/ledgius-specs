# Terms

Vocabulary used in this spec (population, role, platform owner, region,
etc.) is defined canonically in **R-0052 §Terms**.

# Summary

The platform-owner-facing surface for administering pricing plans:
versioning, effective dates, feature inclusion/exclusion, plan metadata,
and assignment to regions / jurisdictions. This is the back-office
counterpart to R-0048 (which handles the customer-facing signup flow that
applies an already-defined plan).

# Problem / Business Context

Ledgius needs platform owners to manage commercial offerings without
engineering involvement:

- Adjust plan pricing in line with market changes
- Add or remove features from a plan (e.g. enable bank feeds for Essential+)
- Run a new plan version while keeping existing customers on the prior
  version (grandfathering)
- Offer different plans in different regions (AU vs future expansion)
- Schedule plan changes to take effect on a specific date
- Hold plan metadata (display name, marketing copy, support tier) without
  redeploying

Today (post-R-0049 simple env-var feature gate), feature enablement is
hardcoded. This spec replaces that with a managed catalogue.

# Scope

- Feature catalogue (the universe of toggleable features and their slugs)
- Plan model: name, version, currency, price, billing cycle, region
- Plan-feature mapping: which features each plan version includes
- Effective-date scheduling for plan versions
- Region/jurisdiction assignment of plans
- Platform-owner administration UI (list / edit / version / publish)
- Audit trail of every plan / feature change (per R-0052 IDR-014)

# Out of Scope

- Self-service signup / payment collection (R-0048 owns the user-facing flow)
- Stripe product-catalogue sync (touched but designed in detail under R-0048)
- Per-tenant overrides / discounts / promotional codes (future)
- Usage-metered billing (future)

# Actors / Users

- **Platform owner** (per R-0052) — manages plans, features, regional
  assignments. Sole population permitted to access these endpoints.
- **Customer user / accountant** — read-only consumer of "what's enabled
  on my plan?" queries via the existing feature-flag middleware.

# Preconditions

- R-0052 (Identity, Roles, and Audit Surface) implemented
- `platform_owner` population assignment exists for at least one user
- Feature-flag middleware (R-0041, retained under R-0052) reads from this
  catalogue rather than env vars

# Functional Requirements

## Feature catalogue

- **PLN-001**: System SHALL maintain a catalogue of features as `(slug, display_name, description, category)` rows. Slugs are the authoritative identifier used by code's `RequireFeature(...)` middleware.
- **PLN-002**: A feature MAY have one or more numeric or text `feature_attributes` (e.g. `bank_feeds.max_connections=3`) that travel with it and are queryable by code that needs a parameterised limit.
- **PLN-003**: Removing a feature from the catalogue requires explicit confirmation and SHALL fail if any active plan version references it.

## Plan model + versioning

- **PLN-004**: A plan is identified by `(slug, version)`. Slugs are stable across versions (e.g. `essential`); versions are monotonic integers.
- **PLN-005**: A plan version SHALL carry: display_name, description, billing_cycle (`monthly` | `annual` | `lifetime`), price (decimal), currency, effective_from, effective_to (nullable), region_id (nullable — global if NULL), and the feature set + feature_attribute overrides.
- **PLN-006**: Plan versions are immutable after publication. Editing a published plan creates a new draft version; publishing the draft sets its effective_from and (optionally) updates the prior version's effective_to to align.
- **PLN-007**: A draft plan version SHALL NOT be visible to customer-facing surfaces or feature-flag middleware until published.
- **PLN-008**: System SHALL guarantee non-overlapping (region, slug) effective-date ranges. Publishing a new version automatically truncates the prior version's effective_to to `(new.effective_from - epsilon)`.

## Region assignment

- **PLN-009**: A plan version MAY be scoped to a region (per R-0052's `regions` table). NULL region means global.
- **PLN-010**: When a tenant queries its plan, the resolution order is: (1) plan version explicitly scoped to the tenant's region, effective on the lookup date; (2) plan version scoped to a parent region in the region hierarchy; (3) plan version with NULL region (global). First match wins.

## Tenant assignment

- **PLN-011**: Tenants reference a plan via `(plan_slug, version)` rather than a single plan_id, enabling grandfathered tenants to remain on an older version while new signups land on the latest.
- **PLN-012**: A platform owner MAY migrate a tenant from one version to another (with audit). Bulk migration tools MUST allow rollback within a 24h window.
- **PLN-013**: Plan changes that reduce a tenant's feature set MUST surface in the tenant UI as a "plan downgrade — features removed" notice for at least 30 days.

## Audit

- **PLN-014**: Every plan / feature / assignment change SHALL audit per R-0052 IDR-014, with `active_role=platform_owner`, the changed entity's before/after JSON, and the platform owner's user_id.

## API surface (platform-owner only)

- **PLN-015**: System SHALL expose the following endpoints, all gated to `active_role=platform_owner`:
    - `GET    /api/v1/admin/plans`
    - `POST   /api/v1/admin/plans` (create new draft version)
    - `GET    /api/v1/admin/plans/:slug/:version`
    - `PUT    /api/v1/admin/plans/:slug/:version` (only on draft)
    - `POST   /api/v1/admin/plans/:slug/:version/publish`
    - `GET    /api/v1/admin/features`
    - `POST   /api/v1/admin/features` (add new feature slug)
    - `PUT    /api/v1/admin/features/:slug`
    - `GET    /api/v1/admin/regions`
    - `POST   /api/v1/admin/regions`
    - `POST   /api/v1/admin/tenants/:id/plan` (migrate tenant)

# Non-Functional Requirements

- **PLN-NFR-001**: Plan resolution for a tenant on the request hot path MUST add no more than 5ms p95 (cached after first resolve per request).
- **PLN-NFR-002**: Plan + feature catalogue cache invalidation MUST propagate within 30 seconds across all running API instances (NOTIFY/LISTEN or short TTL).
- **PLN-NFR-003**: Audit writes for plan changes MUST capture the full before/after JSON of the plan version, not just the diff, so a single audit row replays the historical state.

# Data / Entities / Fields

See A-0029 for canonical schema. New entities:

- `features` (catalogue)
- `feature_attributes` (slug → key/value attributes)
- `plans` (slug + display info)
- `plan_versions` (slug + version + region + dates + price + feature set)
- `plan_version_features` (mapping with optional attribute overrides)
- `tenant_plan_assignments` (tenant → plan_slug + version + assigned_at)

# UX / UI Behaviour

Platform-owner admin UI (`/admin/plans`):

- **Plans list**: every plan slug with its currently-effective version per
  region, plus a count of tenants on each version.
- **Plan version editor**: form for draft versions, pre-filled from prior
  version. Publish CTA opens a confirmation showing the calculated
  effective_to truncation on the prior version and the count of tenants
  who'll see the change.
- **Feature catalogue**: add/edit features. Removing a feature only
  enabled when zero plan versions reference it.
- **Region tree**: hierarchical view of regions with plan coverage
  visualised.
- **Tenant migration**: bulk select tenants → new plan slug + version →
  preview affected features per tenant → confirm.

# Validation Rules

- Effective_from cannot be in the past (admin can override with explicit
  `back_date=true` flag, audited).
- Two published plan versions for the same `(slug, region)` cannot share
  an effective date.
- A draft version cannot be published if the `effective_from` is later
  than any subsequent draft awaiting publication for the same
  `(slug, region)` — must publish in chronological order.

# Error Handling

- Attempting to edit a published version: 409 Conflict with body
  `{error: "version-immutable", create_new_version: true}`.
- Removing a feature that's still referenced: 409 Conflict listing the
  blocking plan versions.
- Tenant migration that would orphan a Stripe subscription mid-cycle:
  409 with proration preview; admin must confirm.

# Security / Permissions

- All `/admin/plans/*`, `/admin/features/*`, `/admin/regions/*` endpoints
  require `active_role=platform_owner`.
- `POST /admin/tenants/:id/plan` additionally requires the platform owner
  to have a region assignment that covers the tenant's region (or be
  global).
- All mutations audited per R-0052 IDR-014.

# Acceptance Criteria

- [ ] Platform owner can create a new draft plan version, edit it,
      publish it, and see it apply to new signups in the matching region
- [ ] Existing tenants stay on their assigned version until explicitly
      migrated
- [ ] Bulk tenant migration shows feature delta preview before commit
- [ ] Removing a feature from a published plan is rejected (must create
      a new version)
- [ ] Region-scoped plan resolves correctly via the parent-region
      fallback chain
- [ ] R-0049 `bank_feeds` env-var cap is replaced by the
      `bank_feeds.max_connections` feature attribute, read by the
      bankfeed service via the catalogue lookup
- [ ] Every plan-administration endpoint returns 403 for active_role !=
      platform_owner
- [ ] Plan change audit row shows full before/after JSON, replayable

# Open Questions

1. **Stripe sync model**: do Ledgius plan versions map 1:1 onto Stripe
   products + prices? If so, publishing a plan version triggers a
   Stripe API call. Need to define the failure mode (retry queue?
   block publish?).
2. **Currency model**: AU starts AUD-only, but multi-currency support
   for international expansion is foreseeable. Plan version carries
   currency now; do we also need price-per-currency variants per
   version?
3. **Promotional pricing**: 50% off first 3 months, etc. — do these
   live as a separate `promotion` entity layered on plan version, or
   inside the plan version itself? Recommendation: separate entity.
4. **Plan downgrades**: when a tenant's plan loses a feature, what
   happens to data already created under that feature (e.g. existing
   bank feed connections if `bank_feeds` is removed)? Default:
   read-only access retained, no new actions allowed.
