# Summary

Technical architecture for user signup, Stripe payment integration, trial lifecycle management, and marketing site user portal.

# Implementation Phases

## Phase 1: Database Schema (ledgius-db)

### Migration V1.03: Stripe and trial columns

```sql
-- Stripe integration on tenants
ALTER TABLE tenants ADD COLUMN IF NOT EXISTS stripe_customer_id TEXT;
ALTER TABLE tenants ADD COLUMN IF NOT EXISTS stripe_subscription_id TEXT;
ALTER TABLE tenants ADD COLUMN IF NOT EXISTS payment_status TEXT DEFAULT 'none'
    CHECK (payment_status IN ('none', 'trial', 'active', 'past_due', 'cancelled', 'suspended'));

-- Trial cancellation feedback
CREATE TABLE IF NOT EXISTS trial_cancellation_feedback (
    id          SERIAL PRIMARY KEY,
    tenant_id   UUID NOT NULL REFERENCES tenants(id),
    user_id     UUID NOT NULL REFERENCES users(id),
    reason      TEXT NOT NULL CHECK (reason IN (
        'too_expensive', 'missing_features', 'not_ready',
        'switched_competitor', 'too_complex', 'other')),
    details     TEXT,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

## Phase 2: API — Signup & Payment Endpoints (ledgius-api)

### New domain: `internal/signup/`

- `model.go` — SignupCmd, TrialStatus, SubscriptionStatus
- `service.go` — CreateAccount (user + tenant + membership + trial), StripeCheckout, HandleWebhook, CancelTrial
- `handler.go` — HTTP handlers
- `routes.go` — Public routes (signup, webhook) + protected routes (subscription, cancel)

### Endpoints

```
POST /api/v1/signup                    — Create user + tenant + redirect URL for Stripe
POST /api/v1/payments/webhook          — Stripe webhook handler (no auth, signature verified)
GET  /api/v1/payments/subscription     — Current subscription status (authenticated)
POST /api/v1/payments/cancel           — Cancel subscription/trial with feedback (authenticated)
POST /api/v1/payments/create-portal    — Stripe Customer Portal URL for subscription management
```

### Stripe Integration

```go
// Dependencies
import "github.com/stripe/stripe-go/v82"

// Checkout session creation
params := &stripe.CheckoutSessionParams{
    CustomerEmail: &email,
    Mode:          stripe.String("subscription"),
    LineItems: []*stripe.CheckoutSessionLineItemParams{{
        Price:    stripe.String(plan.StripePriceID),
        Quantity: stripe.Int64(1),
    }},
    SubscriptionData: &stripe.CheckoutSessionSubscriptionDataParams{
        TrialPeriodDays: stripe.Int64(14),
    },
    SuccessURL: stripe.String("https://app.ledgius.com?session_id={CHECKOUT_SESSION_ID}"),
    CancelURL:  stripe.String("https://ledgius.com/signup?cancelled=true"),
}
```

### Webhook event handling

```go
switch event.Type {
case "checkout.session.completed":
    // Update tenant with stripe_customer_id, stripe_subscription_id
    // Set payment_status = "trial"
case "customer.subscription.updated":
    // Update payment_status based on subscription.status
case "invoice.payment_succeeded":
    // Set payment_status = "active"
case "invoice.payment_failed":
    // Set payment_status = "past_due", send warning email
case "customer.subscription.deleted":
    // Set payment_status = "cancelled", suspend tenant
}
```

### Trial email reminders

Cron job or scheduled task that runs daily:
1. Query tenants WHERE `payment_status = 'trial'` AND `trial_ends_at` is approaching
2. Send emails at day 7, day 12, day 14 via Resend
3. Day 14: Stripe automatically charges (trial_period_days handles this)

## Phase 3: Marketing Site — Signup & Portal (ledgius-website)

### New pages

- `/signup` — Signup form with plan pre-selection
- `/portal` — Logged-in user dashboard (plan status, news, app link)
- `/portal/subscription` — Subscription management (via Stripe Customer Portal redirect)

### Auth on marketing site

The marketing site needs lightweight auth:
- Store JWT in localStorage after signup/login
- Login form on marketing site calls same `/api/v1/auth/login` endpoint
- Auth state shown in navbar: user icon + dropdown

### Signup form flow

```
1. User fills form → POST /api/v1/signup
2. API creates user + tenant, returns Stripe Checkout URL
3. Frontend redirects to Stripe Checkout
4. Stripe redirects back to app.ledgius.com on success
5. User is now logged in with trial active
```

## Phase 4: App — Trial UI (ledgius-web-app)

### Trial countdown in header

When `payment_status = "trial"`:
- Show amber badge: "Trial: 8 days remaining"
- Link to subscription management
- Dismiss temporarily but reappears next session

### Upgrade prompts

When accessing a feature not in the current plan:
- Feature gate middleware returns 403
- Show upgrade prompt: "This feature requires the Professional plan. Upgrade now?"

# Security Considerations

- Stripe webhook endpoint verifies signature using `STRIPE_WEBHOOK_SECRET`
- Signup endpoint is rate-limited (5 per minute per IP)
- Password hashed with bcrypt (existing pattern)
- Stripe Customer Portal used for sensitive payment operations (PCI compliant)
- Trial cancellation feedback stored but never exposes payment details

# Testing Strategy

1. Use Stripe test mode for development
2. Test webhook handling with Stripe CLI: `stripe listen --forward-to localhost:8090/api/v1/payments/webhook`
3. Test trial expiry with short trial periods (1 day) in test mode
4. Verify tenant provisioning creates all tables and seed data
5. Verify feature gate blocks correctly when trial expires
