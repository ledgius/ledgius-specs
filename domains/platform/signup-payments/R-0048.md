# Summary

End-to-end user signup, payment processing, trial management, and portal navigation for the Ledgius platform. Users sign up on ledgius.com, choose a plan, provide payment details via Stripe, and are provisioned with a tenant database. Trial users get 14 days free with reminders before billing begins.

# Business Context

Ledgius needs a self-service signup flow to acquire customers without manual onboarding. The flow must:
- Convert marketing website visitors into paying customers
- Support a free trial period to reduce signup friction
- Handle payment collection via Stripe (credit card, bank debit)
- Provision tenant databases automatically
- Manage the trial-to-paid transition with email reminders
- Allow logged-in users on the marketing site to navigate to the app

# Acceptance Criteria

## AC-1: Signup form on ledgius.com

The marketing website (`ledgius.com`) has a signup form accessible from:
- "Get Started" CTA buttons throughout the landing page
- `/signup` route

### Sign up with SSO (primary method)

Two SSO buttons above the manual form:
- **"Continue with Google"** — Google OAuth 2.0 (covers Gmail + Google Workspace users)
- **"Continue with Microsoft"** — Microsoft OAuth 2.0 / MSAL (covers Outlook + M365 users)

SSO flow:
1. User clicks SSO button → redirected to Google/Microsoft consent screen
2. On return, API receives OAuth token, extracts email + name
3. If email not in platform DB → create user + show business name form (minimal: business name + ABN optional + plan selection)
4. If email already exists → log in and redirect to app
5. Password not required for SSO users — auth is via OAuth token exchange

### Sign up with email (fallback)

Below the SSO buttons, a divider "— or sign up with email —" and the manual form:
- Full name
- Email address
- Password (with strength indicator, min 8 chars)
- Business name
- ABN (optional)
- Selected plan (pre-selected from pricing page click, changeable)

### On successful signup (both SSO and email):
1. Creates a `user` record in the platform DB (with `auth_provider` field: "google", "microsoft", or "email")
2. Creates a `tenant` record with a sanitised slug from business name
3. Creates a `tenant_membership` with role `owner`
4. Sets `trial_ends_at` to 14 days from now
5. Redirects to Stripe Checkout for payment details (even for trial — card captured but not charged)

## AC-1b: SSO login on app.ledgius.com

The app login page (`app.ledgius.com/login`) also shows the same SSO buttons:
- "Continue with Google"
- "Continue with Microsoft"
- Divider "— or sign in with email —"
- Email + password form (existing)

SSO login flow:
1. User clicks SSO button → OAuth consent screen
2. On return, API validates OAuth token, finds matching user by email
3. Issues JWT access + refresh tokens (same as email login)
4. If no user found → redirect to marketing site signup page

## AC-2: Stripe payment integration

Payment processing uses Stripe:
- **Stripe Checkout** for initial card capture during signup
- **Stripe Subscriptions** for recurring billing
- **Stripe Webhooks** for async event handling

API endpoints:
- `POST /api/v1/payments/create-checkout-session` — creates Stripe Checkout session with trial period
- `POST /api/v1/payments/webhook` — receives Stripe events
- `GET /api/v1/payments/subscription` — returns current subscription status (authenticated)
- `POST /api/v1/payments/cancel` — cancels subscription (authenticated)

Stripe events to handle:
- `checkout.session.completed` — activate tenant, store Stripe customer/subscription IDs
- `invoice.payment_succeeded` — extend subscription period
- `invoice.payment_failed` — mark tenant as payment_failed, send email
- `customer.subscription.deleted` — suspend tenant

## AC-3: Trial management

- Trial period: 14 days from signup
- During trial: full access to all features of the selected plan
- Trial countdown visible in the app header (e.g. "Trial: 8 days remaining")
- Email reminders sent via Resend:
  - Day 7: "Your trial is halfway through — here's what you've accomplished"
  - Day 12: "Your trial ends in 2 days — your card will be charged $XX/month"
  - Day 14: "Your trial has ended — billing has started"
- Each reminder email includes:
  - Link to cancel trial / downgrade
  - Feedback form if cancelling (reason dropdown + free text)

## AC-4: Trial cancellation with feedback

When a user cancels during trial:
- Show a cancellation form: reason (too expensive, missing features, not ready, switched to competitor, other) + free text
- Store feedback in `trial_cancellation_feedback` table
- Suspend the tenant (don't delete — data preserved for 90 days)
- Send confirmation email: "Your trial has been cancelled. Your data will be preserved for 90 days."
- No charge applied

## AC-5: Tenant database provisioning

On successful signup:
1. Create a new PostgreSQL database named `ledgius_t_{sanitised_slug}`
2. Run all Flyway migrations against the new database
3. Seed with AU chart of accounts and default data
4. Update tenant record with `db_name`, `db_host`, `db_port`
5. Tenant is immediately accessible at `app.ledgius.com`

For trial MVP: use the existing shared database with tenant isolation via `tenant_id` context. Dedicated databases are a Phase 2 enhancement.

## AC-6: Marketing site user portal

Logged-in users on `ledgius.com` see:
- User icon in the top-right nav bar (replacing "Login" / "Get Started")
- Dropdown menu:
  - "Go to Ledgius App" → navigates to `app.ledgius.com` (with auth token handoff or separate login)
  - "Manage Subscription" → subscription management page
  - "Sign Out"
- Dashboard section showing:
  - Current plan and billing status
  - Trial countdown (if applicable)
  - Recent news / product updates
  - Quick links to help / documentation

## AC-7: Plan selection and upgrade

- Pricing page on marketing site shows plans from the API (`GET /api/v1/pricing`)
- Clicking "Get Started" on a plan pre-selects it in the signup form
- Logged-in users can upgrade/downgrade via "Manage Subscription"
- Plan changes take effect at the next billing cycle
- Downgrade warnings: "You will lose access to [features] when downgrading"

## AC-8: Database schema additions

### Platform DB additions:

```sql
-- SSO auth provider on users
ALTER TABLE users ADD COLUMN auth_provider TEXT DEFAULT 'email'
    CHECK (auth_provider IN ('email', 'google', 'microsoft'));
ALTER TABLE users ADD COLUMN oauth_id TEXT;
COMMENT ON COLUMN users.auth_provider IS 'How the user authenticates: email (password), google (OAuth), microsoft (OAuth)';
COMMENT ON COLUMN users.oauth_id IS 'External OAuth subject ID from Google/Microsoft. Null for email users.';

-- Stripe integration columns on tenants
ALTER TABLE tenants ADD COLUMN stripe_customer_id TEXT;
ALTER TABLE tenants ADD COLUMN stripe_subscription_id TEXT;
ALTER TABLE tenants ADD COLUMN payment_status TEXT DEFAULT 'none'
    CHECK (payment_status IN ('none', 'trial', 'active', 'past_due', 'cancelled', 'suspended'));

-- Trial cancellation feedback
CREATE TABLE trial_cancellation_feedback (
    id          SERIAL PRIMARY KEY,
    tenant_id   UUID NOT NULL REFERENCES tenants(id),
    user_id     UUID NOT NULL REFERENCES users(id),
    reason      TEXT NOT NULL,
    details     TEXT,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

# Out of Scope (Current Release)

- Dedicated per-tenant databases (use shared DB with context isolation)
- Annual billing toggle (monthly only for MVP)
- Coupon/discount codes
- Team invitations during signup (owner-only for now)
- Apple Sign In (add later if mobile app launched)
- LinkedIn / Facebook SSO (wrong audience for B2B accounting)
- SAML / enterprise SSO (future, for large accounting firms)
- Mobile app deep links
