# Summary

Architecture for the tenant provisioning pipeline. Two execution modes share one provisioning library:

1. **Local dev mode** — shell script + Make targets in `ledgius-db`. Fast iteration; no platform-DB schema changes required to run.
2. **Production mode** — Go binary embedded in the `ledgius-api` admin endpoints, invoked from the signup approval flow + the deploy migration step.

Both modes implement the same five-step provisioning operation against the same `tenant_provisioning_log` audit trail.

# Requirement Link

- R-0074 (Tenant Database Provisioning Pipeline Requirements)

# Technical Context

The existing infrastructure already provides most of the building blocks:

- **`flyway.sh`** — applies versioned + repeatable migrations to a target DB, idempotent via the `flyway_schema_history` table.
- **`seed/main.go`** — seed-runner registers tenants in the platform DB.
- **`platform.Repository`** — Go API for reading/writing `tenants` rows.
- **`migrate/main.go --all-tenants`** — runs migrations against every active tenant (assumes DBs exist).

The missing pieces are: the **CREATE DATABASE** step, the **audit-trail recording**, the **signup-approval integration**, and the **single-tenant scoping** (the existing `--all-tenants` runner is bulk-only).

# Proposed Design

## Five-Step Provisioning Operation

Both modes execute the same operation, scoped to one tenant. Bulk mode loops over tenants.

```
function provisionTenant(ctx, tenant, actor):
    log = startLog(tenant.id, actor)

    # Step 1 — Validate
    if tenant.db_name in RESERVED_NAMES:
        return logFailure(log, "reserved DB name")
    if not validIdentifier(tenant.db_name):
        return logFailure(log, "invalid DB identifier")

    # Step 2 — CREATE DATABASE (idempotent)
    if dbExists(tenant.db_name):
        log.note("database already exists — skipping CREATE")
    else:
        try:
            createDatabase(tenant.db_name, owner=DEFAULT_OWNER)
            log.note("database created")
        except err:
            return logFailure(log, "create database failed: " + err)

    # Step 3 — Apply versioned migrations
    versionedResult = flyway.applyVersioned(tenant.db_name)
    if versionedResult.failed:
        return logFailure(log, "versioned migration failed: " + versionedResult.error)
    log.recordMigrations(versionedResult.applied)

    # Step 4 — Apply repeatable migrations
    repeatableResult = flyway.applyRepeatable(tenant.db_name)
    if repeatableResult.failed:
        # Repeatable failures are non-fatal — the tenant is usable.
        log.note("repeatable seed failed (non-fatal): " + repeatableResult.error)
        log.outcome = "partial"
    else:
        log.recordSeeds(repeatableResult.applied)
        log.outcome = "success"

    # Step 5 — Mark tenant as provisioned
    if log.outcome in ("success", "partial"):
        updateTenant(tenant.id, provisioned_at=NOW)

    return logComplete(log)
```

## Local Dev Mode — `provision-tenants.sh`

A shell script + Make targets in `ledgius-db`. Already shipped 2026-04-25 as a stop-gap.

```
ledgius-db/
  scripts/
    provision-tenants.sh           # shell implementation of the five-step op
  Makefile                         # provision-tenants, provision-tenants-dry targets
```

Characteristics:
- Reads platform DB directly (no Go binary build step).
- Idempotent: skips DB creation when DB exists; lets Flyway skip already-applied migrations.
- Tolerates the pre-existing `R__12` seed bug (treats repeatable failure as non-fatal when versioned succeeded).
- No write to `tenant_provisioning_log` (the table doesn't yet exist in the platform DB; deferred until T-0042 lands the migration).

When the platform-DB migration adds `tenant_provisioning_log`, the shell script gains an `INSERT` step at start + an `UPDATE` step at end so dev-mode runs are also audited.

## Production Mode — Go Service in `ledgius-api`

A Go service in `internal/platform/provisioning/` exposing two entry points:

```
package provisioning

type Service struct {
    platformRepo *platform.Repository
    flywayRunner FlywayRunner
    db           *sql.DB  // admin connection with CREATEDB
    logger       *slog.Logger
}

// ProvisionOne provisions a single tenant. Used by signup-approval flow.
func (s *Service) ProvisionOne(ctx context.Context, tenantID, actor string) error

// ProvisionAll provisions every tenant whose db doesn't exist or has
// pending migrations. Used by deploy pipeline.
func (s *Service) ProvisionAll(ctx context.Context, actor string) (Summary, error)
```

The Go service uses the same Flyway library bindings as the shell script (or shells out to `flyway.sh` — TBD; recommend shelling out for v1 to avoid duplicating the runner logic).

## Signup Approval Integration (Path 2)

R-0068's signup approval flow currently:
1. Validates the signup request.
2. Creates the `tenants` row with `status='pending'`.
3. Returns approval response.

Updated flow (this spec):
1. Validates the signup request.
2. Creates the `tenants` row with `status='pending'` + `created_via='signup_self_serve'`.
3. **Calls `Service.ProvisionOne(tenantID, actor=adminUserID)`** within the same approval transaction (or in a follow-up step before the tenant is exposed to the user). On success → `status='active'` + `provisioned_at=now`. On failure → tenant stays `pending`, error surfaced to the admin, no user notification sent.
4. Returns approval response (success only after provisioning success).

## Admin-Assisted Create Integration (Path 3)

Per R-0074 §Admin-Assisted Tenant Creation. Sits alongside Path 2; reuses the provisioning + audit + invitation infrastructure but with a different entry point and a richer intent payload.

```go
// AssistedCreate provisions a tenant the platform admin is onboarding
// directly (VIP / beta / hand-held / pre-signup migration). Returns the
// created tenant + status of provisioning + invite.
func (s *Service) AssistedCreate(ctx context.Context, payload AssistedCreatePayload, actor string) (*AssistedCreateResult, error)

type AssistedCreatePayload struct {
    // Identity
    DisplayName  string
    Slug         string  // auto from DisplayName if empty
    BusinessType string
    LegalName    string
    ABN          string
    Country      string
    // Billing address
    BillingState, BillingCity, BillingPostcode, BillingLine1 string
    // Initial owner (required)
    OwnerEmail       string
    OwnerDisplayName string
    // Plan + billing
    PlanID            string  // any active plan including comp/beta
    BillingMode       string  // paid_stripe / paid_offline / comp / beta / pending
    BillingNote       string  // free-text for offline/comp/beta
    // Assisted-create context (required)
    Reason             string
    TicketURL          string
    CustomerCategory   string  // vip / beta / migration / manual_signup / other
    WelcomeEmailMode   string  // send_now / defer
}
```

Service orchestration (single transaction where possible; provisioning happens after commit):

```
1. Validate payload (slug uniqueness, plan exists, owner email well-formed).
2. INSERT tenants row with status='pending', created_via='admin_assisted',
   billing_mode=payload.BillingMode, billing_note=payload.BillingNote.
3. Create or look up user by OwnerEmail; INSERT tenant_membership(role='owner').
4. Issue email invitation via existing R-0068 invite flow UNLESS WelcomeEmailMode='defer'.
5. Apply plan to tenant (existing plan-assignment endpoint logic).
6. INSERT tenant_assisted_create_log row capturing actor + reason + ticket + category + welcome-email mode + billing-mode-chosen.
7. Commit transaction.
8. Invoke Service.ProvisionOne(tenantID, actor) — synchronous; transitions tenant to active on success.
9. On provisioning failure: tenant stays pending; assisted-create log row stays (the create context is still recorded, audit-wise); return failure with form-state-preservation hints to UI.
```

The admin endpoint:

```
POST /api/v1/platform/tenants/assisted-create
  Authorization: Bearer <platform-admin JWT>
  Body: AssistedCreatePayload
  Response: AssistedCreateResult { tenant, provisioning_outcome, invite_status }
```

Restricted to `is_platform_admin = true`. Audit trail combines `tenant_provisioning_log` (the provisioning step's row) + `tenant_assisted_create_log` (the assisted-create context). Both rows are queryable from the platform admin UI for the tenant detail page.

### Why not reuse the existing signup-approval path?

Path 2 expects a customer-submitted `signup_request` row with the customer's own intent + Stripe SetupIntent + email confirmation. Path 3 doesn't have a customer-submitted record — the admin is the originator. Re-using Path 2 by having the admin "submit a synthetic signup request and approve it themselves" was considered and rejected: it pollutes the signup-request table with admin-originated records, conflates two actor models, and forces the admin to provide Stripe SetupIntent metadata that's irrelevant for comp / beta tenants. A separate endpoint with a separate audit table keeps the two intent models cleanly distinguishable.

## Deploy Pipeline Integration

The existing `fly-migrate` step (per ledgius-infra `Makefile`) runs platform + tenant migrations. A new step inserted between platform-migration and tenant-migration:

```
make fly-deploy-api:
    fly-migrate-platform
    fly-provision-all-tenants     # NEW — runs Service.ProvisionAll
    fly-deploy-image
```

`fly-provision-all-tenants` opens a fly-proxy tunnel to the API, invokes the platform-admin endpoint `POST /api/v1/platform/provisioning/run-all`, awaits completion, asserts no failures.

## Audit Trail Schema

```sql
-- migrations/platform/V1.21__tenant_provisioning_log.sql
CREATE TABLE tenant_provisioning_log (
    id                  uuid PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id           uuid NOT NULL REFERENCES tenants(id),
    action              text NOT NULL CHECK (action IN ('create_db', 'apply_migrations', 'apply_seeds', 'repair')),
    actor               text NOT NULL,
    started_at          timestamptz NOT NULL DEFAULT now(),
    completed_at        timestamptz,
    outcome             text NOT NULL CHECK (outcome IN ('in_progress', 'success', 'partial', 'failure')),
    error_message       text,
    migrations_applied  int DEFAULT 0,
    seeds_applied       int DEFAULT 0
);

CREATE INDEX idx_tenant_provisioning_log_tenant_started
    ON tenant_provisioning_log (tenant_id, started_at DESC);

ALTER TABLE tenants
    ADD COLUMN provisioned_at timestamptz;
```

## Idempotency Guarantees

| Re-run scenario | Behaviour |
|---|---|
| Tenant DB doesn't exist | CREATE DATABASE + run migrations |
| Tenant DB exists, all migrations applied | No-op (Flyway sees nothing to apply) |
| Tenant DB exists, partial migrations | Flyway resumes from where it left off |
| Tenant DB exists, repeatable failed last run | Re-attempts repeatable; previous failure cleared |
| Tenant DB exists, schema corrupted | Out of scope — operator escalation, no auto-repair |

## Failure Recovery

- **CREATE DATABASE fails** — log, exit non-fatally for bulk mode (continue to next tenant), fatally for single-tenant mode.
- **Versioned migration fails** — fatal in both modes. DB is in an undefined state; operator must investigate. The platform admin UI surfaces a "provisioning failed" badge on the tenant row pulling from `tenant_provisioning_log`.
- **Repeatable migration fails** — non-fatal. Tenant marked `outcome='partial'` in the log; usable but with seed gap.
- **Network partition during provisioning** — Flyway commits per migration; the next run resumes. The audit log row stays `in_progress` until completion is recorded; an external process (out of scope for this architecture, future) reaps stale `in_progress` rows.

# Alternatives Considered

## "Provision on first request" lazy mode

When the API receives a request for an un-provisioned tenant, provision it on the fly.

**Rejected.** Provisioning takes ~30s; a user request that triggers provisioning would time out. Surfacing "your tenant is being set up, refresh in a minute" is a poor onboarding experience. Eager provisioning at approval time is unambiguously better.

## Single Go binary, no shell script

Skip the shell stop-gap; ship the Go binary first.

**Rejected.** The shell script is already shipped (2026-04-25) and unblocks local dev today. The Go binary is the right long-term home but doesn't need to land before R-0068's signup-approval integration; the integration brings the Go binary in naturally. Both can coexist (shell for local dev, Go for production).

## Per-tenant database hosts from day one

Place each tenant's DB on a dedicated Postgres host or cluster.

**Rejected for v1.** Adds significant infra complexity for marginal benefit at current scale. The schema (`tenants.db_host`, `tenants.db_port`) already supports it; we can opt into per-tenant placement later by setting non-null host/port on a tenant row, and the provisioning library will honour it. v1 placement is the same host as the platform DB.

# Risks + Mitigations

| Risk | Impact | Mitigation |
|---|---|---|
| CREATE DATABASE fails silently in bulk mode | Tenant remains unusable, operator unaware | Audit log + admin UI badge surface every failure |
| Versioned migration breaks an existing tenant on re-run | Tenant data corruption | Flyway forward-only + per-DB versioning isolates the change; standard pre-deploy migration testing applies |
| Pre-existing `R__12` seed bug confuses operators | Provisioning reports failure when tenant is in fact usable | Script tolerates this case explicitly + surfaces clear "non-fatal — versioned succeeded" message |
| Stale `in_progress` rows accumulate | Audit log clutter, no functional impact | Future reaper job; not blocking |
| Local dev script + production binary drift over time | Different behaviour in dev vs prod | Both shell out to the same `flyway.sh` runner; the only divergence is the audit-log INSERT step |

# Performance Targets

- Single-tenant provisioning: ≤60s end-to-end (CREATE DATABASE + ~50 versioned migrations + ~12 repeatable seeds).
- Bulk provisioning of 8 tenants: ≤8 minutes serial; ≤2 minutes if parallelised (post-MVP).
- Audit log write: ≤50ms.
