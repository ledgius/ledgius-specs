# Summary

Define the **tenant database provisioning pipeline** that takes a tenant from "registered in `ledgius_platform.tenants`" to "fully usable" — its own PostgreSQL database created, all Flyway tenant migrations applied, repeatable seed data loaded, and the tenant routable from the API's per-request DB selector.

This pipeline is the missing link between three existing or expected capabilities:

- **`make seed-load DATASET=test-tenants`** (ledgius-db `seed/main.go`) — registers test/dev/CI tenants in bulk from a YAML dataset; does not create per-tenant DBs.
- **The signup approval flow** (R-0068) — customer-initiated self-serve onboarding; approves a signup request, transitioning the platform `tenants` row from `pending` to `active`.
- **Admin-assisted tenant creation** (this spec, new) — platform-owner-initiated tenant onboarding, used for VIP / hand-held / beta-tester / older customers who do not go through the self-serve signup. Operates on the same destination state (`status='active'` + provisioned + audit row) as the signup approval flow but with a different actor + intent metadata.

All three paths share a common runtime: provisioning, migration application, audit trail, signup-of-the-initial-owner-user, plan + billing setup. This spec formalises the shared substrate plus the specific contract for the admin-assisted path.

# Problem / Business Context

Current state (verified 2026-04-25 against the local dev environment):

- 8 tenants are registered in `ledgius_platform.tenants` with `status='active'`.
- **Only one tenant database exists** — `ledgius` (used by the `looking-good` tenant via `db_name='ledgius'`). The other 7 tenants reference DB names (`crestline-mowing`, `test-farm`, etc.) that **don't exist as PostgreSQL databases**.
- A user with multi-tenant memberships (e.g. `matt@ledgius.com`, 4 memberships) can complete login + see the tenant picker, but selecting any tenant other than `looking-good` results in connection failure when the API tries to open the tenant DB.
- A user with a single membership against an unprovisioned tenant — typical of a freshly-approved signup — will fail at first request after login.

The gap is operational, not architectural: the platform DB knows about the tenant; the tenant DB simply doesn't exist. There is no automation (in dev or in production) that creates the database, applies migrations, and loads the standard chart of accounts.

A stop-gap script (`ledgius-db/scripts/provision-tenants.sh`, `make provision-tenants`) was shipped 2026-04-25 to bridge the dev gap. This spec formalises:

1. The contract that script must satisfy.
2. The production-flow integration with the signup approval workflow.
3. The schema-validation, idempotency, and audit requirements.
4. The error-recovery requirements for when provisioning fails partway.

# Scope

- **Per-tenant database creation** for tenants registered in `ledgius_platform.tenants` with `status='active'` (or transitioning to active).
- **Tenant migration application** — Flyway `migrations/tenant/V*.sql` versioned + `R__*.sql` repeatable migrations applied to every newly-created tenant DB.
- **Repeatable seed data** — chart of accounts, tax codes, currencies, PAYG rates, super rates, asset categories, instant-write-off thresholds, default report templates — applied via the existing `R__*` migrations.
- **Idempotency** — re-running provisioning against a partially-provisioned tenant must not destroy data; missing pieces are filled in.
- **Audit trail** — every provisioning action recorded with actor, timestamp, target tenant, outcome.
- **Local dev mode** — `make provision-tenants` operates against a local Postgres + Flyway runner.
- **Production mode** — same pipeline triggered from the signup approval flow (R-0068) when a platform admin approves a tenant.
- **Re-provisioning + repair** — a tenant DB that's missing tables (corrupted / partially restored / never finished) can be re-run safely.

# Out of Scope

- **Tenant DB tear-down** — handled separately (data retention + audit constraints make tenant-DB destruction a separate concern with stricter governance per LEDGIUS-DATA-RETENTION).
- **Tenant DB migration rollback** — Flyway is forward-only; rollback is out of scope here and elsewhere in Ledgius.
- **Multi-region DB placement** — initial implementation places every tenant DB on the same Postgres host as the platform DB. Multi-region is a future infra concern.
- **Tenant data import** (Xero / MYOB / QuickBooks bring-up) — covered separately by R-0017 + the importer module. This pipeline only creates an empty migrated DB; data import is a tenant action after provisioning.
- **Per-tenant data volume directories** (`/data/{slug}/`) for file uploads — referenced in `tenants.yaml` comments but addressed by infra setup, not this pipeline.

# Actors / Users

- **Platform admin** — approves a signup; expects the approved tenant to be immediately usable.
- **Developer** — runs `make provision-tenants` locally to bootstrap multi-tenant testing without manual `CREATE DATABASE` chores.
- **CI / deploy pipeline** — applies tenant migrations to every active tenant after a deploy that ships migrations.
- **Auditor** — reviews provisioning audit trail to verify when each tenant was bootstrapped and by whom.

# Preconditions

- `ledgius_platform.tenants` row exists for the target tenant with: `slug`, `db_name`, `status='active'`, `display_name`.
- Postgres host + admin credentials available (CREATE DATABASE permission).
- Flyway migrations directory present (`ledgius-db/migrations/tenant/`).
- Repeatable seed migrations present (R__01 through R__12 currently).

# Functional Requirements

## Provisioning Action

- **TPP-001**: The pipeline shall accept a tenant identifier (slug or UUID) and provision the tenant DB end-to-end: CREATE DATABASE → apply versioned migrations → apply repeatable migrations.
- **TPP-002**: The pipeline shall accept a "bulk" mode that selects every tenant with `status='active'` and provisions any whose DB doesn't yet exist.
- **TPP-003**: The pipeline shall be idempotent: re-running against an already-provisioned tenant DB must produce zero destructive operations and exit successfully.
- **TPP-004**: A partial-provision (DB created but migrations failed mid-way) shall be safely resumable on re-run; Flyway's history table provides this for migrations, and the script must not attempt CREATE DATABASE on an existing DB.
- **TPP-005**: The tenant DB name shall be sourced from `tenants.db_name`. Where `db_name` is empty, the pipeline shall use `slug` and update the `db_name` column.
- **TPP-006**: The pipeline shall not silently rename or alter an existing tenant DB.

## Migration + Seeding

- **TPP-007**: Versioned migrations (`V*.sql` in `ledgius-db/migrations/tenant/`) shall apply in version order to every newly-created tenant DB.
- **TPP-008**: Repeatable migrations (`R__*.sql` in `ledgius-db/migrations/tenant/`) shall apply after versioned migrations, on every run (per Flyway semantics — re-applied when the file content changes).
- **TPP-009**: A repeatable-migration failure that leaves all versioned migrations successfully applied shall not be treated as a fatal provisioning failure for the database itself; the tenant is usable. The pipeline shall report the partial failure clearly so the operator can investigate.
- **TPP-010**: Versioned-migration failures shall be treated as fatal — the tenant DB exists but is in an undefined state. The pipeline shall surface the failing migration + DB name + log excerpt and exit non-zero.

## Audit Trail

- **TPP-011**: Every provisioning attempt shall be recorded in `ledgius_platform.tenant_provisioning_log` with: tenant_id, action (`create_db` / `apply_migrations` / `apply_seeds` / `repair`), actor (user_id or `system`), started_at, completed_at, outcome (`success` / `partial` / `failure`), error message (if any).
- **TPP-012**: Audit records shall be retained per LEDGIUS-DATA-RETENTION (5-year baseline).
- **TPP-013**: The most recent successful full-provisioning timestamp shall be exposed on the tenant row (`tenants.provisioned_at`) so the platform admin UI can show provisioning status at a glance.

## Local Dev Mode

- **TPP-014**: A make target `make provision-tenants` shall run the bulk-mode provisioning against the local platform DB.
- **TPP-015**: A make target `make provision-tenants-dry` shall preview what would be provisioned without changing anything.
- **TPP-016**: A flag `--slug=<slug>` shall scope the run to a single tenant.

## Production Mode

- **TPP-017**: The signup approval flow (R-0068) shall invoke the provisioning pipeline as part of approving a tenant. The approval transaction shall not commit until provisioning reports success — the operator must not see "approved" while the tenant DB doesn't exist.
- **TPP-018**: Where the production deploy applies new tenant migrations, the deploy pipeline shall invoke `provision-tenants` in repair-mode against every active tenant so the new migrations land on every tenant DB before the API serves traffic referencing them.
- **TPP-019**: Production provisioning failures shall page the on-call channel (existing alerting integration) and prevent the tenant from transitioning to `status='active'`.

## Admin-Assisted Tenant Creation (Path 3)

This is the platform-owner-initiated tenant onboarding path. It sits alongside the customer-initiated signup approval flow (R-0068), shares the same runtime (provisioning + audit + welcome-email infrastructure), but differs in actor, trigger, and intent metadata. Used for VIP customers, beta testers, hand-held migrations from older systems, and any tenant the platform owner wants to onboard outside the self-serve flow.

- **TPP-030**: The platform admin UI shall expose a "+ New Tenant" action on the Tenants list page (R-0068 §PA-NNN) that opens an assisted-create form. Restricted to users with `is_platform_admin = true`.
- **TPP-031**: The form shall capture, in a single submission:
  - **Tenant identity**: display name (required), slug (auto-generated from display name, editable), business type (optional), legal name (optional), ABN (optional), country (default AU).
  - **Billing address**: state, city, postcode, line 1 (all optional but recommended).
  - **Initial owner user (required)**: email + display name. The user is **invited via email**; the invite link establishes the initial password and validates the email address per existing tenant-invitation flow (R-0068). A tenant may not be created without an initial owner user — that's a footgun and breaks the audit story for "who is responsible for this tenant?".
  - **Plan assignment (required, plan-flexible)**: the admin selects from any active plan, including comp / custom-priced / "no-billing" plans. Beta-tester comp accounts are a primary use case.
  - **Billing setup (optional)**: where the chosen plan has a non-zero price, the admin may either (a) collect a Stripe payment method via a one-time setup link emailed to the tenant, (b) record an offline billing arrangement with a free-text note, or (c) skip billing entirely (comp / beta / no-charge). The chosen approach is recorded on the tenant + audit row.
  - **Assisted-create context (required)**: actor (the admin user, captured from the JWT — not user-entered), reason / ticket reference (free-text + optional URL), intended customer category (`vip` / `beta` / `migration` / `manual_signup` / `other`), and welcome-email behaviour (`send_now` / `defer` — defer when the admin plans to walk the customer through the first login personally).
- **TPP-032**: On submit, the system shall:
  1. Validate the form (slug unique, email well-formed, plan exists).
  2. Create the `tenants` row with `status='pending'`.
  3. Create the user (or look up existing user by email) and the tenant membership (role `owner`).
  4. Issue the email invitation per the existing invite flow (R-0068), unless the admin chose `welcome_email = defer`.
  5. Apply the plan + billing-mode to the tenant (record the comp/offline/skip-billing choice on the row).
  6. Invoke the provisioning pipeline (Step 2) synchronously.
  7. On success: transition the tenant to `status='active'` + set `provisioned_at = now()`.
  8. Record a `tenant_provisioning_log` entry with `actor = <admin user id>` AND a separate `tenant_assisted_create_log` entry with the assisted-create context (reason, ticket, customer category, welcome-email behaviour, billing mode chosen).
- **TPP-033**: Provisioning failures shall hold the tenant in `pending` and surface the failure to the admin on the same page; the form shall be resubmittable for retry without re-entering the full payload (form state preserved).
- **TPP-034**: The Tenants list page shall render an `assisted-created` badge on tenants created via this path (distinct from the badge for self-serve signups), and the tenant detail page shall surface the assisted-create context (actor, reason, ticket, customer category, billing mode) for audit purposes.
- **TPP-035**: An assisted-created tenant whose initial owner has not yet accepted the email invite (within a configurable TTL, default 14 days) shall surface a warning on the tenant detail page; the admin may resend the invite, change the invited email, or remove the unprovisioned tenant if invite is never accepted.
- **TPP-036**: Comp / beta / no-billing tenants shall persist their billing-mode choice on the tenant row + audit log, and the platform billing pipeline shall not attempt to charge them. A change of plan from comp to paid is a separate audited admin action (existing plan-change endpoint with audit reason).

## Configuration

- **TPP-020**: All connection parameters (host, port, user, password, ssl mode) shall be sourced from environment / config — no hard-coded credentials in the script.
- **TPP-021**: The pipeline shall support a separate Postgres role for `CREATE DATABASE` operations vs migration application, where infra security policy requires it. Default: same role.

## Non-Functional Requirements

- **TPP-022**: Bulk-mode provisioning of N tenants shall complete in approximately N × (single-tenant time). Single-tenant provisioning is dominated by Flyway migration time (currently ~30s for the full versioned + repeatable set).
- **TPP-023**: The pipeline shall not require manual intervention for a successful happy-path provisioning. Failures may require operator attention; success shall not.

# Validation Rules

- A tenant cannot be set to `status='active'` while no successful `tenant_provisioning_log` row exists for it.
- A tenant DB name in `tenants.db_name` must be a valid Postgres database identifier (≤63 chars, lowercase, hyphens-allowed).
- The pipeline must refuse to operate on `db_name='postgres'`, `template0`, `template1`, `ledgius_platform`, or any reserved name.

# Error Handling

- **CREATE DATABASE failure** (permission denied, name in use as template, etc.) — abort with clear error; do not modify `tenants` row; log to `tenant_provisioning_log` with `outcome='failure'`.
- **Versioned migration failure** — DB exists but is in an undefined state; log to `tenant_provisioning_log`; surface to operator. Manual investigation required.
- **Repeatable migration failure** — versioned migrations succeeded; tenant is usable for the migrated tables. Log as `outcome='partial'` with a clear message identifying the failed seed; tenant remains usable.
- **Connection failure mid-run** — script must be re-runnable. The Flyway history table tracks what's been applied; the script must not attempt to re-apply versioned migrations that already succeeded.

# Security / Permissions

- Provisioning is a platform-admin-only action (R-0068 §PA-NNN — to be linked).
- The Postgres role used by the pipeline requires `CREATEDB` privilege; this role must not be reused for normal API runtime DB connections.
- Connection credentials must be sourced from secrets manager (Fly secrets in production); never logged.

# Open Questions

1. **Should provisioning be a separate Go binary or stay as a shell script?** The shell script is faster to ship; a Go binary integrates better with the existing `cmd/migrate` runner + uses the platform repository. Recommend: shell stop-gap stays for local dev; production mode is a Go binary integrated into the API server's admin endpoints.
2. **Should each tenant get its own Postgres user role?** Currently every tenant DB is owned by the `ledgius` role. Better isolation would use a per-tenant role, but adds significant infra complexity. Defer.
3. **What about tenant DB host placement for HA?** Initial implementation: same host as platform DB. Future: per-tenant or per-region placement. Schema (`tenants.db_host`, `tenants.db_port`) already supports this; pipeline must honour those fields when set.
4. **Should the pipeline support tenant DB *destruction* / unprovisioning?** Out of scope here — see LEDGIUS-DATA-RETENTION + a separate decommissioning spec.
5. **Should the pre-existing `R__12_seed_default_report_templates.sql` bug be fixed in this task?** That migration is in `migrations/tenant/` but the table only exists in `migrations/platform/`. Recommend: fix in a separate small PR independent of this task, since it's a 1-file move/delete. Document in the implementation task as a known-issue tolerance.
