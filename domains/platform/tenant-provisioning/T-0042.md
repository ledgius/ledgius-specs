# Summary

Land the tenant provisioning pipeline per R-0074 / A-0047. Five vertical slices, each independently shippable.

# Slices

## Slice 1 — Local-dev shell script (shipped 2026-04-25)

Already in place via `feat/provision-seeded-tenants` PR.

- `ledgius-db/scripts/provision-tenants.sh` — five-step provisioning op as a shell.
- `make provision-tenants`, `make provision-tenants-dry`, `--slug=<slug>` flag.
- Tolerates the `R__12_seed_default_report_templates.sql` repeatable-failure as non-fatal when versioned migrations succeed.
- **Ship gate**: 8 dev tenants provisioned end-to-end; multi-tenant login picker works against all of them.

## Slice 2 — Audit-log schema

- New platform-DB migration `migrations/platform/V1.21__tenant_provisioning_log.sql` (table + index + `tenants.provisioned_at` column).
- Extend `platform.Repository` with `RecordProvisioningAttempt`, `CompleteProvisioningAttempt`, `LatestProvisioningForTenant`, `MarkTenantProvisioned`.
- Backfill: run a one-off SQL script that inserts a `tenant_provisioning_log` row for each existing active tenant with `outcome='success'` + sets `provisioned_at = now()`.
- Update `provision-tenants.sh` to write audit rows.
- **Ship gate**: every provisioning run produces an audit row; backfill complete for the 8 existing tenants.

## Slice 3 — Repair the pre-existing `R__12` bug

Independent prep work — a 1-file change that unblocks clean provisioning runs.

- Move (or delete as duplicate of `migrations/platform/R__03_seed_default_report_templates.sql`) the misplaced `migrations/tenant/R__12_seed_default_report_templates.sql`.
- Bump Flyway repeatable file checksums so the change is picked up cleanly.
- Re-run `make provision-tenants` against an existing tenant DB; verify clean success (no partial outcome).
- **Ship gate**: a fresh provisioning run reports `outcome='success'` (not `partial`) for every tenant.

## Slice 4 — Go provisioning service

- New package `ledgius-api/internal/platform/provisioning/`:
  - `service.go` — `Service.ProvisionOne(ctx, tenantID, actor)` + `Service.ProvisionAll(ctx, actor)`.
  - `flyway.go` — wrapper that shells out to `ledgius-db/flyway.sh` (or links the runner library if extracted).
  - `validator.go` — DB-name validation + reserved-name list.
- Admin endpoint: `POST /api/v1/platform/provisioning/tenants/:id/provision` + `POST /api/v1/platform/provisioning/run-all`.
- Wire to existing platform-admin RBAC (only `is_platform_admin=true` users).
- **Ship gate**: an admin endpoint call against an unprovisioned tenant runs end-to-end + writes audit + flips `tenants.status` to active.

## Slice 5 — Signup approval + deploy pipeline integration

- R-0068 signup approval flow updated: provisioning runs as part of approve-tenant action; failures hold the tenant in `pending`.
- Platform admin UI surfaces provisioning status per tenant (badge: green `provisioned at <date>` / amber `partial` / red `failed`).
- Deploy pipeline (`ledgius-infra/Makefile`) gains `fly-provision-all-tenants` step inserted between platform migration and image deploy.
- **Ship gate**: an approved signup goes from "pending" → "active + provisioned" without manual intervention; a deploy applying new tenant migrations runs them on every active tenant before the image flips.

# Acceptance Criteria (cumulative)

- [ ] All 8 existing dev tenants have audit-log entries + `provisioned_at` set.
- [ ] `tenant_provisioning_log` schema landed.
- [ ] `R__12` placement bug fixed (or accepted as known-issue with rationale).
- [ ] Go provisioning service operational + admin endpoints live.
- [ ] Signup approval invokes provisioning + holds tenant in `pending` on failure.
- [ ] Deploy pipeline runs provisioning before image deploy.
- [ ] Platform admin UI shows provisioning status.
- [ ] Test scenario: approve a fresh signup → tenant DB exists with all migrations + standard CoA → multi-tenant login works against the new tenant.
- [ ] Test scenario: deploy a new tenant migration → bulk-provisioning applies it to all 8 dev tenants → no manual intervention.

# Out of Scope

- Tenant DB destruction / unprovisioning (see LEDGIUS-DATA-RETENTION + a future decommissioning spec).
- Per-tenant Postgres roles / per-tenant connection pooling.
- Multi-region DB placement (schema supports it; v1 places every DB on the same host).
- Auto-recovery of stale `in_progress` audit rows (future reaper job).
