# Summary

Business owners and accountants can invite users to their tenant
by email, assigning a role. The invitee receives an email with a
link to accept the invitation — either creating a new account or
linking an existing one. Invitation lifecycle is tracked with
states: `invited` → `accepted` | `expired` | `revoked`. Tenant
owners can also remove users and change roles.

# Terms

| **Term** | Definition. |
|---|---|
| **Invitation** | A record linking an email address to a tenant with a proposed role. Created by a tenant owner/accountant, fulfilled when the invitee accepts. |
| **Inviter** | The user who creates the invitation. Must be an owner or master_accountant of the tenant. |
| **Invitee** | The person being invited. May or may not have an existing Ledgius account. |
| **Accept token** | A single-use, time-limited token embedded in the invitation email link. Validates the invitee's identity. |

# Problem / Business Context

Currently, adding users to a tenant requires platform admin
intervention (PA-023). Business owners cannot self-service invite
their accountant, bookkeeper, or staff. This creates friction:

1. Owner contacts Ledgius support → admin manually adds user
2. No invitation email — user must be told their credentials separately
3. No tracking of pending invitations
4. No way for the owner to manage their own team

Competitors (Xero, MYOB, QuickBooks) all provide self-service
user invitation from within the app. This is table stakes.

# Scope

## Invitation Flow (UI-001 through UI-010)

- **UI-001**: Business owners and master accountants can invite
  users from **Settings → Users** (tenant-scoped, not platform admin).
  The form collects: email address, role (bookkeeper, accountant,
  viewer), optional display name.
- **UI-002**: On submit, the system:
  1. Creates an `invitation` record in the platform DB
  2. Generates a single-use accept token (UUID, expires in 7 days)
  3. Sends an invitation email via Resend with the accept link
  4. If the email matches an existing user, creates the membership
     immediately (status: `accepted`) and sends a notification
     instead of an invitation
- **UI-003**: The invitation email contains:
  - Tenant name (who invited you)
  - Inviter name and email
  - Assigned role
  - "Accept Invitation" button linking to
    `{FRONTEND_URL}/invite/{token}`
  - Expiry note ("This invitation expires in 7 days")
- **UI-004**: When the invitee clicks the accept link:
  - If logged in → membership created, redirect to tenant dashboard
  - If not logged in but account exists → redirect to login, then
    accept on return
  - If no account → redirect to account creation form (name,
    password), then accept
- **UI-005**: Invitation states:
  - `invited` — email sent, awaiting acceptance
  - `accepted` — invitee accepted, membership created
  - `expired` — token TTL exceeded (7 days)
  - `revoked` — owner cancelled before acceptance
- **UI-006**: The Settings → Users page shows:
  - Active users with roles (owner, master_accountant, accountant,
    bookkeeper, viewer)
  - Pending invitations with email, role, sent date, expiry
  - Actions: change role, remove user, revoke invitation, resend
    invitation

## User Management (UI-011 through UI-015)

- **UI-011**: Tenant owners can **change a user's role** from the
  Settings → Users page. Role changes take effect immediately.
- **UI-012**: Tenant owners can **remove a user** from the tenant.
  This deletes the `tenant_membership` record. The user's platform
  account is retained (they may belong to other tenants).
- **UI-013**: Only owners can remove other owners. Master accountants
  can remove accountants, bookkeepers, and viewers but not owners
  or other master accountants.
- **UI-014**: Users cannot remove themselves — they must ask another
  owner to remove them (prevents accidental lockout).
- **UI-015**: All user management operations are audit-logged.

## API Endpoints (UI-020 through UI-025)

- **UI-020**: `POST /settings/users/invite` — create invitation
  (tenant-scoped, requires owner or master_accountant role)
- **UI-021**: `GET /settings/users` — list tenant users + pending
  invitations
- **UI-022**: `PUT /settings/users/:id/role` — change user role
- **UI-023**: `DELETE /settings/users/:id` — remove user from tenant
- **UI-024**: `POST /settings/users/invitations/:id/revoke` — revoke
  pending invitation
- **UI-025**: `POST /settings/users/invitations/:id/resend` — resend
  invitation email
- **UI-026**: `GET /invite/:token` — validate invitation token
  (public, unauthenticated)
- **UI-027**: `POST /invite/:token/accept` — accept invitation
  (creates account if needed, creates membership)

# Database Changes

## New table: `invitation` (platform DB)

```sql
CREATE TABLE invitation (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id       UUID NOT NULL REFERENCES tenants(id),
    email           TEXT NOT NULL,
    role            TEXT NOT NULL DEFAULT 'viewer',
    display_name    TEXT,
    token           TEXT NOT NULL UNIQUE,
    status          TEXT NOT NULL DEFAULT 'invited'
        CHECK (status IN ('invited', 'accepted', 'expired', 'revoked')),
    invited_by      UUID NOT NULL REFERENCES users(id),
    expires_at      TIMESTAMPTZ NOT NULL,
    accepted_at     TIMESTAMPTZ,
    accepted_by     UUID REFERENCES users(id),
    revoked_at      TIMESTAMPTZ,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_invitation_token ON invitation(token);
CREATE INDEX idx_invitation_tenant ON invitation(tenant_id);
CREATE INDEX idx_invitation_email ON invitation(email);
```

# Out of Scope

- Bulk invitation (CSV upload of emails) — future
- Custom invitation message text — future
- Invitation approval workflow (admin must approve) — not needed
- SSO/SAML invitation flow — future
- Team/group roles — future

# Actors / Users

- Business owner — invites users, manages roles, removes users
- Master accountant — invites users (except owners), manages roles
- Invitee — receives email, accepts invitation, creates account
- Platform admin — can also manage via PA-023 (existing)

# Acceptance Criteria

## Invitation
- [ ] Owner can invite by email + role from Settings → Users (UI-001)
- [ ] Invitation email sent via Resend (UI-002, UI-003)
- [ ] Existing user auto-accepted with notification (UI-002)
- [ ] Accept link creates account or links existing (UI-004)
- [ ] Invitation states tracked correctly (UI-005)
- [ ] Pending invitations shown on Settings → Users (UI-006)

## User Management
- [ ] Change role (UI-011)
- [ ] Remove user (UI-012)
- [ ] Role-based removal permissions (UI-013)
- [ ] Cannot self-remove (UI-014)

## API
- [ ] All endpoints return appropriate status codes
- [ ] Token expires after 7 days (UI-005)
- [ ] Revoked/expired tokens rejected on accept (UI-027)
