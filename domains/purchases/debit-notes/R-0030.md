# Summary

Create accounts payable debit notes that fully or partially reverse an existing AP bill. Debit notes generate reversed GL entries (debit AP, credit expense, credit GST) and are validated against the original bill amount using the OPA/CEL rules engine. This is the AP equivalent of an AR credit note.

# Problem / Business Context

When goods are returned to a vendor, services are not provided, or billing errors are discovered on a vendor bill, a debit note must be issued to reduce the amount owed. The debit note must create correct reversing GL entries and must not exceed the original bill amount.

# Scope

- Debit note creation against a specific original bill
- Amount validation against original bill amount (via OPA policy)
- GST reversal calculation via CEL rules engine
- Reversed GL entries: debit AP, credit expense, credit GST (if applicable)
- Open item creation for the debit note
- Negative AP record (is_return = true)

# Out of Scope

- Debit note listing (currently only creation endpoint)
- Debit note detail retrieval endpoint
- Multi-line debit notes (currently single-amount)
- Debit note PDF generation

# Actors / Users

- Bookkeeper -- creates debit notes against vendor bills
- Business Owner -- reviews vendor adjustments
- Accountant -- verifies expense reversals

# Preconditions

- User is authenticated
- Original bill exists and belongs to the specified vendor
- Vendor has a configured AP account

# Functional Requirements

- **DN-001**: System shall create a debit note atomically: open_item + transaction + AP record (is_return=true, negative amounts) + reversed GL entries, all within a single database transaction
- **DN-002**: Debit note amount must be positive (validated on entry)
- **DN-003**: Debit note amount must not exceed the original bill amount, validated via OPA `ValidateNoteCap` policy
- **DN-004**: Original bill must exist and must belong to the specified vendor
- **DN-005**: GL entries shall reverse the original bill pattern: debit AP (positive), credit expense (negative), credit GST (negative if GST was on the original)
- **DN-006**: GST reversal amount shall be calculated via CEL rule `credit_note_gst_reversal` (amount / 1.10)
- **DN-007**: Expense and GST account IDs shall be looked up from the original bill's GL entries
- **DN-008**: AP record shall have is_return=true and negative amount_bc/amount_tc/netamount_bc/netamount_tc
- **DN-009**: Transaction record shall store the debit note number as reference and the reason as description

| Method | Path | Description |

# Validation Rules

- Vendor ID is required
- Original bill ID is required and must belong to the vendor
- Debit note number is required
- Debit date is required
- Amount must be positive
- Amount must not exceed original bill amount (OPA policy)
- Reason is required
- Currency is required

# Error Handling

- Amount <= 0 returns error: "debit note amount must be positive"
- Original bill not found returns error with bill ID
- Bill does not belong to vendor returns error with both IDs
- Amount exceeds bill amount returns error with both amounts

# Acceptance Criteria

- [x] Debit note creation atomically creates all records in a single transaction
- [x] Positive amount validation
- [x] OPA-validated amount cap against original bill
- [x] Reversed GL entries (debit AP, credit expense, credit GST)
- [x] GST reversal via CEL rules engine
- [x] Expense/GST accounts looked up from original bill GL entries
- [x] AP record stored with is_return=true and negative amounts

# Open Questions

1. **Debit note listing/detail**: Only creation is implemented; list and detail endpoints are not yet built.
2. **Multi-line debit notes**: Current implementation is single-amount; multi-line debit notes are not yet supported.
