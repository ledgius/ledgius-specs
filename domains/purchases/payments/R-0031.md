# Summary

Record payments made to vendors against open AP bills. Payments allocate funds across one or more bills and generate GL entries that debit the AP account (reducing liability) and credit the bank account (money out), with each allocation referencing the bill's open item for reconciliation.

# Problem / Business Context

When a business pays a vendor bill, the payment must be recorded with correct GL entries (debit AP, credit bank) and allocated against specific bills to track which bills are paid. Open item references enable downstream reconciliation and ageing reports.

# Scope

- Payment creation with multi-bill allocation
- GL entry generation (debit AP per allocation, credit bank)
- Open item referencing per allocation for reconciliation
- Payment listing with summary view
- Allocation amount validation (positive amounts only, via rules engine)

# Out of Scope

- Payment reversal/void
- Overpayment handling (debit balance on vendor account)
- Payment detail retrieval (no GET by ID endpoint)
- Bank feed matching/auto-allocation
- Payment PDF generation
- Batch payments

# Actors / Users

- Bookkeeper -- records vendor payments
- Business Owner -- approves and reviews payments

# Preconditions

- User is authenticated
- Vendor exists with a configured AP account
- At least one open AP bill exists for the vendor
- Bank account exists in the chart of accounts

# Functional Requirements

- **PAY-001**: System shall create a payment atomically: transaction record + GL entries (debit AP per allocation, credit bank), all within a single database transaction
- **PAY-002**: Each allocation amount must be positive, validated via rules engine `ValidatePositiveAmount`
- **PAY-003**: Each allocation must reference a valid open AP bill for the specified vendor
- **PAY-004**: Each AP debit GL entry must reference the bill's open_item_id for reconciliation
- **PAY-005**: GL entries shall follow: debit AP account (per allocation, each referencing open_item_id, positive amount), credit bank account (total amount, negative)
- **PAY-006**: System shall list all payments with summary data: trans_id, reference, payment_date, vendor_name, amount, curr, approved
- **PAY-007**: Transaction record shall store the reference number and optional notes

# Validation Rules

- Bank account ID is required
- Vendor ID is required and must have a configured AP account
- Payment date is required
- Reference is required
- Currency is required
- At least one allocation is required
- All allocation amounts must be positive
- Each allocation must reference a valid bill for the vendor

# Error Handling

- Vendor with no AP account returns error: "vendor N has no AP account configured"
- Negative allocation amount returns error: "allocation amounts must be positive"
- Bill not found for vendor returns error with bill and vendor IDs
- Open item lookup failure returns error with bill ID

# Acceptance Criteria

- [x] Payment creation atomically creates transaction and GL entries
- [x] Multi-bill allocation supported
- [x] Positive amount validation via rules engine
- [x] Each allocation references bill open_item_id
- [x] GL entries balance (sum debit AP = credit bank)
- [x] Payment list with summary data

# Open Questions

1. **Payment detail endpoint**: No GET by ID endpoint exists yet.
2. **Overpayment**: No handling for payments exceeding bill totals.
3. **Payment reversal**: No void/reversal mechanism implemented.
4. **Batch payments**: No support for paying multiple vendors in a single batch.
