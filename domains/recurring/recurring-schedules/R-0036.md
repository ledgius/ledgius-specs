# Summary

Define and manage recurring transaction schedules using RRULE-based recurrence patterns. Schedules store a JSON template of the transaction to be generated, support frequency presets (weekly, fortnightly, monthly, quarterly, annually) or custom RRULE strings, and track next due dates for automated generation.

# Problem / Business Context

Businesses have regular recurring transactions (monthly rent, weekly supplier orders, quarterly subscriptions). Rather than manually creating these each period, a schedule system allows templates to be defined once and generated automatically on the due date.

# Scope

- Schedule CRUD (create, read, list)
- RRULE-based recurrence with frequency presets and custom RRULE strings
- Next due date calculation and advancement after generation
- Due schedule listing (schedules ready for generation)
- RRULE preview (show next N occurrences for a given rule)
- Auto-deactivation when past end date
- Support for GL, AR, and AP source types

# Out of Scope

- Actual transaction generation from templates (schedule defines when, not the generation logic)
- Schedule update/edit after creation
- Schedule deletion/pause
- Email notifications for due schedules
- Approval workflow for generated transactions

# Actors / Users

- Bookkeeper -- creates recurring schedules for regular transactions
- System -- queries due schedules and advances them after generation
- Business Owner -- reviews scheduled recurring items

# Preconditions

- User is authenticated

# Functional Requirements

- **REC-001**: System shall store recurring schedules with: name, description, source_type (gl/ar/ap), template_json (JSONB), frequency label, rrule string, start_date, end_date, next_due_date, last_generated, auto_approve flag, and active flag
- **REC-002**: System shall support frequency presets: weekly (FREQ=WEEKLY;INTERVAL=1), fortnightly (FREQ=WEEKLY;INTERVAL=2), monthly (FREQ=MONTHLY;INTERVAL=1), quarterly (FREQ=MONTHLY;INTERVAL=3), annually (FREQ=YEARLY;INTERVAL=1)
- **REC-003**: System shall accept custom RRULE strings and validate them by parsing
- **REC-004**: If only a frequency preset is provided (no explicit rrule), the system shall map it to the corresponding RRULE string
- **REC-005**: If only an rrule is provided (no frequency), the system shall derive the frequency label from preset mapping or "custom"
- **REC-006**: System shall set next_due_date to start_date on creation
- **REC-007**: System shall list due schedules where next_due_date <= now and active = true
- **REC-008**: AdvanceSchedule shall: set last_generated to now, calculate next occurrence via RRULE after current next_due_date, update next_due_date, and deactivate if past end_date
- **REC-009**: System shall provide RRULE preview returning the next N occurrences from a given start date
- **REC-010**: All dates shall be truncated to date-only (no time component)

# Validation Rules

- Name is required
- Source type must be one of: gl, ar, ap
- Template JSON is required
- Either frequency or rrule must be provided
- If frequency is provided, it must match a known preset
- If rrule is provided, it must be a parseable RFC 5545 RRULE string
- Start date is required and must be valid ISO date

# Error Handling

- Neither frequency nor rrule provided returns error
- Unknown frequency preset returns error with suggestion to provide rrule
- Invalid rrule returns parse error
- No next occurrence (exhausted rule) returns error

# Acceptance Criteria

- [x] Schedule creation with frequency preset
- [x] Schedule creation with custom RRULE
- [x] Frequency label derived from RRULE if only rrule provided
- [x] Due schedule listing
- [x] Schedule advancement with next due date recalculation
- [x] Auto-deactivation past end date
- [x] RRULE preview with N occurrences
- [x] Date truncation to date-only

# Open Questions

1. **Transaction generation**: The schedule system defines when, but actual transaction creation from templates is not yet implemented.
2. **Schedule editing**: No update endpoint exists after creation.
3. **Schedule pause/resume**: No mechanism to temporarily pause a schedule.
