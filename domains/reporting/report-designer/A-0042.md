# Summary

Technical architecture for the report template designer. Covers
the visual editor (Puck), template storage (JSON), data source
abstraction, HTML rendering pipeline, PDF generation (Gotenberg),
and print optimisation.

# Terms

Vocabulary used in this spec (report template, data source, field,
component, canvas, hydration) is defined canonically in
**R-0071 §Terms**.

# Tech Stack

| Layer | Technology | Why |
|---|---|---|
| Template editor | **Puck** (`@measured/puck`) | React visual editor with drag-drop, component config, prop editing. Purpose-built for visual page building. MIT license, 5k+ stars, actively maintained. |
| Template storage | **JSONB** in PostgreSQL | Puck outputs JSON natively. JSONB enables querying, indexing, and partial updates. |
| Data hydration | **Go `html/template`** | Server-side template rendering. Secure (auto-escaping), fast, stdlib. |
| PDF generation | **Gotenberg** (primary) or **chromedp** (fallback) | Gotenberg: Docker sidecar, Chromium-based HTML→PDF, REST API. chromedp: headless Chrome from Go (no sidecar). |
| Print CSS | **`@media print`** + **`@page`** rules | Browser-native print support. Page breaks, margins, repeating headers. |
| Field catalogue | **Go structs with reflect tags** | Each data source defines its fields as a Go struct. A registry function introspects the struct and publishes the field catalogue. |

# Component Design

## 1. Puck Editor Integration

Puck is configured with custom report components:

```typescript
const puckConfig: Config = {
  components: {
    TextField: {
      label: "Text",
      fields: {
        content: { type: "textarea" },
        fontSize: { type: "number" },
        fontWeight: { type: "select", options: ["normal", "bold"] },
        alignment: { type: "select", options: ["left", "center", "right"] },
      },
      render: ({ content, fontSize, fontWeight, alignment }) => (
        <p style={{ fontSize, fontWeight, textAlign: alignment }}>{content}</p>
      ),
    },
    DataField: {
      label: "Data Field",
      fields: {
        fieldSlug: { type: "select", options: [] }, // populated from data source
        format: { type: "select", options: ["text", "currency", "date", "number"] },
        fontSize: { type: "number" },
      },
      render: ({ fieldSlug, format }) => (
        <span data-field={fieldSlug} data-format={format}>
          {`{{${fieldSlug}}}`}
        </span>
      ),
    },
    DataTable: { ... },
    Logo: { ... },
    PageHeader: { ... },
    PageFooter: { ... },
    PageNumber: { ... },
    HorizontalRule: { ... },
    Spacer: { ... },
    SubtotalRow: { ... },
  },
}
```

Puck outputs a JSON structure (`PuckData`) that captures the
complete layout. This JSON is stored in `report_templates.template_json`.

## 3. Data Source Registry

Each data source is a Go interface:

```go
type DataSource interface {
    // Slug returns the unique identifier (e.g. "profit_loss").
    Slug() string
    // FieldCatalogue returns the available fields with types.
    FieldCatalogue() []DataField
    // Query executes the data source and returns structured data.
    Query(ctx context.Context, params map[string]string) (map[string]interface{}, error)
}

type DataField struct {
    Slug     string `json:"slug"`
    Name     string `json:"name"`
    Type     string `json:"type"` // string, number, currency, date, boolean, list
    Category string `json:"category"`
}
```

Data sources are registered at startup:

```go
registry := report.NewDataSourceRegistry()
registry.Register(report.NewProfitLossSource(reportingRepo))
registry.Register(report.NewBalanceSheetSource(reportingRepo))
registry.Register(report.NewInvoiceSource(receivableRepo))
// ...
```

## 4. Rendering Pipeline

```go
func (s *ReportService) Generate(ctx context.Context, req GenerateRequest) ([]byte, error) {
    // 1. Load template.
    template, err := s.repo.GetTemplate(ctx, req.TemplateID)

    // 2. Execute data source.
    source := s.registry.Get(template.DataSource)
    data, err := source.Query(ctx, req.Params)

    // 3. Hydrate template → HTML.
    html, err := s.renderer.RenderToHTML(template.TemplateJSON, data)

    // 4. If PDF requested, convert via Gotenberg.
    if req.Format == "pdf" {
        return s.pdfService.HTMLToPDF(html, template.PageSettings)
    }

    return []byte(html), nil
}
```

## 5. PDF Generation

### Option A: Gotenberg (recommended for production)

Gotenberg runs as a Docker sidecar on Fly.io:

```yaml
# fly.toml for gotenberg
[build]
  image = "gotenberg/gotenberg:8"

[http_service]
  internal_port = 3000
  auto_stop_machines = "stop"
  auto_start_machines = true
```

API call from Go:

```go
func (s *GotenbergPDF) HTMLToPDF(html string, pageSettings PageSettings) ([]byte, error) {
    body := &bytes.Buffer{}
    writer := multipart.NewWriter(body)
    part, _ := writer.CreateFormFile("files", "index.html")
    part.Write([]byte(html))
    writer.WriteField("paperWidth", pageSettings.WidthInches())
    writer.WriteField("paperHeight", pageSettings.HeightInches())
    writer.WriteField("marginTop", pageSettings.MarginTopInches())
    // ...
    writer.Close()

    resp, err := http.Post(s.gotenbergURL+"/forms/chromium/convert/html",
        writer.FormDataContentType(), body)
    // resp.Body contains the PDF bytes.
}
```

### Option B: chromedp (fallback, no sidecar)

```go
func (s *ChromedpPDF) HTMLToPDF(html string, pageSettings PageSettings) ([]byte, error) {
    ctx, cancel := chromedp.NewContext(context.Background())
    defer cancel()

    var pdf []byte
    chromedp.Run(ctx,
        chromedp.Navigate("data:text/html,"+url.QueryEscape(html)),
        chromedp.ActionFunc(func(ctx context.Context) error {
            var err error
            pdf, _, err = page.PrintToPDF().
                WithPaperWidth(pageSettings.WidthInches()).
                WithPaperHeight(pageSettings.HeightInches()).
                Do(ctx)
            return err
        }),
    )
    return pdf, nil
}
```

## 6. Print CSS

All report HTML includes print-optimised CSS:

```css
@media print {
  body { margin: 0; padding: 0; }
  .no-print { display: none; }
  table { page-break-inside: auto; }
  tr { page-break-inside: avoid; }
  thead { display: table-header-group; } /* repeat on every page */
  tfoot { display: table-footer-group; }
}

@page {
  size: A4 portrait;
  margin: 15mm;
  @top-center { content: element(page-header); }
  @bottom-center { content: counter(page) " / " counter(pages); }
}
```

# Deployment

## Gotenberg on Fly.io

```bash
# Create the Gotenberg app
fly apps create ledgius-gotenberg --org ledgius

# Deploy
cd infra/gotenberg && fly deploy

# Set the URL on the API
fly secrets set GOTENBERG_URL="http://ledgius-gotenberg.internal:3000" --app ledgius-api
```

Gotenberg runs on the Fly private network (`*.internal`), so
the API calls it directly without public internet exposure.

# Performance Considerations

- Template JSON is typically 5-50 KB — fast to load and parse
- Data source queries are the same queries used by the existing
  reporting endpoints — no new DB load
- HTML rendering is CPU-bound but fast (Go templates, <10ms)
- PDF generation via Gotenberg: 200-500ms per page (Chromium
  startup is amortised across requests)
- Generated PDFs should be cached if the same report is requested
  multiple times with the same params (TTL: 5 min)

# Security

- Templates belong to a tenant — cross-tenant access prevented
  by tenant_id filter on all queries
- System templates (is_default = true, tenant_id = NULL) are
  read-only — users can clone but not modify
- Template JSON is sanitised — no script injection possible since
  rendering uses Go html/template (auto-escaping)
- PDF generation happens server-side — no client-side code
  execution
