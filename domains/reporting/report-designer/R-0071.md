# Summary

A visual report template designer that allows users to create,
edit, and manage report layouts using a WYSIWYG drag-and-drop
editor. Users select data fields from a palette, position them on
a canvas, configure formatting, and save reusable templates.
Reports are generated by hydrating templates with live data,
previewed in-browser, and exported as PDF or sent to print.

# Terms

| **Term** | Definition. |
|---|---|
| **Report template** | A reusable layout definition stored as JSON. Defines field positions, sizes, formatting, headers, footers, and data bindings. |
| **Data source** | A named query or report type that provides data fields (e.g., "Profit & Loss", "Invoice", "Customer Statement"). |
| **Field** | A named data element from a data source that can be placed on the template canvas (e.g., `total_revenue`, `customer_name`, `invoice_date`). |
| **Component** | A visual building block in the template editor: text field, data field, table, image, header, footer, page number, horizontal rule, etc. |
| **Canvas** | The WYSIWYG editing surface representing a page (A4/Letter). Components are positioned on the canvas via drag-and-drop. |
| **Hydration** | The process of injecting live data into a template to produce a rendered report. |

# Problem / Business Context

Currently, all reports in Ledgius are code-defined with fixed
layouts. Users cannot:
- Customise which fields appear on a report
- Reposition or resize fields
- Add their own headers, logos, or notes
- Create entirely new report layouts
- Export reports as PDF with consistent formatting
- Print reports with a print-optimised layout

Competitors (Xero, MYOB, QuickBooks) offer limited report
customisation. A visual template designer would be a
differentiator, particularly for accountants who need to produce
client-facing reports with specific layouts.

Use cases:
1. **Invoice template** — business owner customises their invoice
   layout with logo, payment terms, bank details
2. **P&L report** — accountant creates a board-ready P&L with
   custom groupings and commentary fields
3. **Customer statement** — bookkeeper designs a statement layout
   matching the business's branding
4. **BAS worksheet** — master accountant creates a pre-lodgement
   review template with specific field selections

# Scope

## Template Editor (RT-001 through RT-010)

- **RT-001**: A visual WYSIWYG editor accessible from
  **Reports → Templates**. The editor presents:
  - A **canvas** representing a page (A4 portrait default,
    configurable to A4 landscape, Letter, Legal)
  - A **component palette** on the left with draggable components
  - A **properties panel** on the right for selected component
  - A **data field picker** showing available fields from the
    selected data source

- **RT-002**: Available components:
  - **Text** — static text with formatting (bold, italic, size,
    colour, alignment)
  - **Data field** — bound to a data source field, displays the
    value at render time
  - **Table** — repeating rows bound to a list data source (e.g.,
    invoice lines, GL entries)
  - **Image** — static image (logo upload) or tenant logo
  - **Header** — page header (repeats on every page)
  - **Footer** — page footer (repeats on every page)
  - **Page number** — auto-incrementing page counter
  - **Horizontal rule** — visual separator
  - **Subtotal / Total** — calculated field (sum, count, average)
  - **Date** — current date or report period date
  - **Spacer** — empty space for layout control

- **RT-003**: Components are positioned via **drag-and-drop** on
  the canvas. Users can:
  - Drag from the palette to the canvas
  - Reposition by dragging on the canvas
  - Resize by dragging handles
  - Align via snap-to-grid (configurable grid size)
  - Set exact position/size via the properties panel (x, y, width,
    height in mm)

- **RT-004**: The **data field picker** shows fields grouped by
  category. When a data source is selected (e.g., "Profit & Loss"),
  the picker shows all available fields:
  - Revenue fields (total_revenue, revenue_by_account, ...)
  - Expense fields (total_expenses, expense_by_account, ...)
  - Summary fields (net_profit, gross_margin, ...)
  - Period fields (start_date, end_date, period_name, ...)
  - Business fields (business_name, abn, logo, ...)

- **RT-005**: Each component has configurable properties:
  - **Position**: x, y (mm from top-left)
  - **Size**: width, height (mm)
  - **Font**: family, size, weight, colour
  - **Alignment**: left, centre, right
  - **Border**: style, width, colour
  - **Padding**: top, right, bottom, left
  - **Visibility**: always, first page only, last page only
  - **Format**: number format, date format, currency format

- **RT-006**: Templates are **saved as JSON** in the platform DB.
  The JSON captures the complete layout: components, positions,
  sizes, bindings, formatting, page settings.

- **RT-007**: Templates belong to a tenant. Each tenant can create
  and manage their own templates. System-provided default templates
  are read-only and can be cloned.

- **RT-008**: Template versioning — each save creates a new version.
  Users can revert to a previous version.

- **RT-009**: Template categories:
  - **Financial reports** — P&L, Balance Sheet, Trial Balance,
    Cash Flow
  - **Customer documents** — Invoice, Quote, Credit Note,
    Customer Statement
  - **Vendor documents** — Purchase Order, Remittance Advice,
    Vendor Statement
  - **Compliance** — BAS Worksheet, PAYG Summary, STP Report
  - **Custom** — user-created templates

- **RT-010**: The editor shows a **live preview** of the template
  with sample data as the user edits. Changes are reflected
  immediately on the preview canvas.

## Data Sources (RT-011 through RT-015)

- **RT-011**: Each data source is a named query that returns
  structured data. The template binds fields to data source
  field names.

- **RT-012**: Built-in data sources:
  - `profit_loss` — P&L report data
  - `balance_sheet` — Balance sheet data
  - `trial_balance` — Trial balance data
  - `cash_flow` — Cash flow statement data
  - `invoice` — Single invoice with lines
  - `quote` — Single quote with lines
  - `customer_statement` — Customer transaction history
  - `vendor_statement` — Vendor transaction history
  - `ar_ageing` — Aged receivables
  - `ap_ageing` — Aged payables
  - `gl_detail` — General ledger detail
  - `employee_payslip` — Individual payslip
  - `bas_worksheet` — BAS preparation data
  - `bank_reconciliation` — Reconciliation summary

- **RT-013**: Each data source publishes a **field catalogue** —
  the list of available fields with types (string, number, date,
  currency, boolean, list). The template editor reads this
  catalogue to populate the field picker.

- **RT-014**: List fields (e.g., invoice lines, GL entries) are
  bound to table components. The table component defines column
  headers and row templates; the renderer repeats the row for each
  item in the list.

- **RT-015**: Custom data sources can be added in future by
  defining a query + field catalogue. The template editor
  dynamically adapts to any data source.

## Report Generation (RT-020 through RT-025)

- **RT-020**: To generate a report, the user selects:
  - A template
  - Parameters (date range, entity, filters)
  - Output format (preview, PDF, print)

- **RT-021**: The API endpoint `POST /reports/generate` accepts:
  ```json
  {
    "template_id": "uuid",
    "data_source": "profit_loss",
    "params": {
      "start_date": "2026-01-01",
      "end_date": "2026-03-31"
    },
    "format": "html" | "pdf"
  }
  ```

- **RT-022**: The generation pipeline:
  1. Load template JSON from DB
  2. Execute data source query with params → structured data
  3. Hydrate template with data → HTML document
  4. If format = "html" → return for preview
  5. If format = "pdf" → render HTML → PDF via Gotenberg/chromedp

- **RT-023**: Generated reports can be:
  - **Previewed** in-browser (HTML in iframe)
  - **Downloaded** as PDF
  - **Printed** via browser print dialog (print-optimised CSS)
  - **Emailed** as PDF attachment (future)

- **RT-024**: PDF generation uses **Gotenberg** (Chromium-based
  HTML → PDF service) or **chromedp** (headless Chrome from Go).
  The HTML is styled with print-specific CSS:
  - `@page` rules for margins, orientation, size
  - `page-break-before/after` for section control
  - Headers and footers via CSS `position: fixed` or
    `@page` margin boxes

- **RT-025**: Report generation is audited — who generated what
  report, when, with what parameters.

## Print (RT-030 through RT-033)

- **RT-030**: Print preview opens in a new browser tab with the
  HTML report styled for print. The user can inspect the layout
  before printing.

- **RT-031**: The print button triggers `window.print()` which
  opens the browser's native print dialog. The HTML uses
  `@media print` CSS to:
  - Hide non-print elements (nav, buttons)
  - Apply print margins
  - Force page breaks at section boundaries
  - Repeat table headers on each page

- **RT-032**: Users can also download the PDF and print from their
  PDF viewer for more control (especially for duplex, collation,
  paper tray selection).

- **RT-033**: Print settings are saved per template: default paper
  size, orientation, margins, colour/mono preference.

## Default Templates (RT-040 through RT-042)

- **RT-040**: The system ships with **default templates** for all
  standard report types (P&L, Balance Sheet, Invoice, etc.).
  These are read-only system templates.

- **RT-041**: Users can **clone** a default template to create
  their own customised version. The clone is fully editable.

- **RT-042**: Default templates use the tenant's branding (logo,
  business name, ABN, contact details) automatically via the
  business identity fields from the tenant record.

# Database Changes

## New tables (platform DB)

```sql
CREATE TABLE report_templates (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id       UUID REFERENCES tenants(id),  -- NULL = system template
    name            TEXT NOT NULL,
    description     TEXT,
    data_source     TEXT NOT NULL,
    category        TEXT NOT NULL DEFAULT 'custom',
    template_json   JSONB NOT NULL,
    page_size       TEXT NOT NULL DEFAULT 'A4',
    page_orientation TEXT NOT NULL DEFAULT 'portrait',
    page_margins    JSONB DEFAULT '{"top": 15, "right": 15, "bottom": 15, "left": 15}',
    is_default      BOOLEAN NOT NULL DEFAULT false,
    is_active       BOOLEAN NOT NULL DEFAULT true,
    version         INT NOT NULL DEFAULT 1,
    created_by      UUID REFERENCES users(id),
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE report_template_versions (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    template_id     UUID NOT NULL REFERENCES report_templates(id) ON DELETE CASCADE,
    version         INT NOT NULL,
    template_json   JSONB NOT NULL,
    created_by      UUID REFERENCES users(id),
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE (template_id, version)
);

CREATE TABLE report_generation_log (
    id              BIGSERIAL PRIMARY KEY,
    tenant_id       UUID NOT NULL REFERENCES tenants(id),
    template_id     UUID NOT NULL REFERENCES report_templates(id),
    user_id         UUID NOT NULL REFERENCES users(id),
    data_source     TEXT NOT NULL,
    params          JSONB,
    format          TEXT NOT NULL,
    generated_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

## New table (tenant DB)

```sql
CREATE TABLE data_source_fields (
    id              SERIAL PRIMARY KEY,
    data_source     TEXT NOT NULL,
    field_slug      TEXT NOT NULL,
    field_name      TEXT NOT NULL,
    field_type      TEXT NOT NULL DEFAULT 'string',
    category        TEXT,
    sort_order      INT NOT NULL DEFAULT 0,
    UNIQUE (data_source, field_slug)
);
```

# Out of Scope

- Scheduled/automated report generation (future)
- Report sharing/publishing (future)
- Excel/CSV export from templates (future — existing CSV export
  covers this)
- Sub-reports / embedded reports (future)
- Conditional formatting rules (future)
- Chart/graph components in templates (future)
- Multi-language report templates (future)

# Actors / Users

- **Business owner** — creates and customises templates for their
  business (invoices, statements)
- **Accountant** — designs financial report templates (P&L,
  Balance Sheet, BAS worksheets)
- **Bookkeeper** — uses existing templates to generate and print
  reports
- **Viewer** — can generate and view reports but not edit templates
