# Summary

Create accounts receivable credit notes that fully or partially reverse an existing AR invoice. Credit notes generate reversed GL entries (credit AR, debit revenue, debit GST) and are validated against the outstanding invoice amount using the OPA/CEL rules engine.

# Problem / Business Context

When goods are returned, services are not delivered, or billing errors occur, a credit note must be issued to reduce the customer's outstanding balance. The credit note must create correct reversing GL entries and must not exceed the original invoice amount. GST must also be reversed proportionally.

# Scope

- Credit note creation against a specific original invoice
- Amount validation against outstanding invoice balance (via OPA policy)
- GST reversal calculation via CEL rules engine
- Reversed GL entries: credit AR, debit revenue, debit GST (if applicable)
- Open item creation for the credit note
- Negative AR record (is_return = true)

# Out of Scope

- Credit note listing (currently only creation endpoint)
- Credit note detail retrieval endpoint
- Allocation of credit notes against other invoices
- Multi-line credit notes (currently single-amount)
- Credit note PDF generation

# Actors / Users

- Business Owner -- issues credit notes for returns or errors
- Bookkeeper -- creates credit notes against invoices
- Accountant -- reviews credit notes for compliance

# Preconditions

- User is authenticated
- Original invoice exists and belongs to the specified customer
- Customer has a configured AR account

# Functional Requirements

- **CN-001**: System shall create a credit note atomically: open_item + transaction + AR record (is_return=true, negative amounts) + reversed GL entries, all within a single database transaction
- **CN-002**: Credit note amount must be positive (validated on entry)
- **CN-003**: Credit note amount must not exceed the outstanding invoice amount, validated via OPA `ValidateNoteCap` policy
- **CN-004**: Original invoice must exist and must belong to the specified customer
- **CN-005**: GL entries shall reverse the original invoice pattern: credit AR (negative), debit revenue (positive), debit GST (positive if GST was on the original)
- **CN-006**: GST reversal amount shall be calculated via CEL rule `credit_note_gst_reversal` (amount / 1.10)
- **CN-007**: Revenue and GST account IDs shall be looked up from the original invoice's GL entries
- **CN-008**: Credit note AR record shall have is_return=true and negative amount_bc/amount_tc/netamount_bc/netamount_tc
- **CN-009**: Transaction record shall store the credit note number as reference and the reason as description

| Method | Path | Description |

# Validation Rules

- Customer ID is required
- Original invoice ID is required and must belong to the customer
- Credit note number is required
- Credit date is required
- Amount must be positive
- Amount must not exceed outstanding invoice amount (OPA policy)
- Reason is required
- Currency is required

# Error Handling

- Amount <= 0 returns error: "credit note amount must be positive"
- Original invoice not found returns error with invoice ID
- Invoice does not belong to customer returns error with both IDs
- Amount exceeds outstanding returns error with both amounts

# Acceptance Criteria

- [x] Credit note creation atomically creates all records in a single transaction
- [x] Positive amount validation
- [x] OPA-validated amount cap against outstanding invoice
- [x] Reversed GL entries (credit AR, debit revenue, debit GST)
- [x] GST reversal via CEL rules engine
- [x] Revenue/GST accounts looked up from original invoice GL entries
- [x] AR record stored with is_return=true and negative amounts

# Open Questions

1. **Credit note listing/detail**: Only creation is implemented; list and detail endpoints are not yet built.
2. **Multi-line credit notes**: Current implementation is single-amount; multi-line credit notes are not yet supported.
3. **Credit allocation**: Credit notes cannot yet be allocated against other invoices.
