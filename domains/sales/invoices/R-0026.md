# Summary

Create, view, update, and list accounts receivable (AR) invoices. Invoice creation is a complete vertical slice that atomically creates an open item, transaction record, AR record, invoice line items, and double-entry GL journal entries. Draft invoices can be updated with recalculated totals and GL entries. PDF generation is supported.

# Problem / Business Context

Sales invoices are the primary revenue document for any business. Ledgius must support creating invoices with multiple line items, automatic GST calculation using configurable tax codes, and correct double-entry GL posting (debit AR, credit revenue, credit GST collected). Draft invoices must be editable before posting.

# Scope

- Invoice creation with multi-line items and automatic GL posting
- Draft invoice updates (notes, due date, line items) with GL recalculation
- Invoice listing with summary view (customer name, amounts, status)
- Invoice detail retrieval with lines and customer name
- Customer change on draft invoices
- PDF generation for invoices
- Per-line tax code support for configurable GST rates

# Out of Scope

- Invoice email/sending
- Payment allocation (handled by Receipts domain R-0028)
- Credit notes (handled by R-0027)
- Recurring invoices (handled by R-0036)
- Invoice approval workflow (currently auto-approved on create)
- Partial payments and open item matching

# Actors / Users

- Business Owner -- creates and reviews invoices
- Bookkeeper -- creates invoices, updates drafts, generates PDFs
- Customer -- receives invoice PDF

# Preconditions

- User is authenticated
- Customer contact exists with a configured AR account (entity_credit_account)
- Revenue accounts exist in the chart of accounts
- GST Collected account exists (if tax-inclusive lines)

# Functional Requirements

- **INV-001**: System shall create an AR invoice atomically: open_item + transaction + AR record + invoice lines + acc_trans GL entries, all within a single database transaction
- **INV-002**: GL entries shall follow double-entry rules: debit AR account (gross total), credit revenue accounts (net per line), credit GST Collected account (GST per line)
- **INV-003**: GST shall be calculated per line item using the CEL rules engine, with rate looked up from tax_code if tax_code_id is provided, falling back to 10% default
- **INV-004**: Default due date shall be calculated via CEL rule `default_due_date_days` from the transaction date if not explicitly provided
- **INV-005**: System shall support updating draft (unapproved) invoices: notes, due date, and complete line item replacement with GL recalculation
- **INV-006**: Draft invoice update shall reject updates to posted (approved) invoices with a clear error message
- **INV-007**: System shall support changing the customer on a draft invoice (PATCH endpoint)
- **INV-008**: System shall list all invoices with summary data: trans_id, invnumber, transdate, duedate, customer_name, meta_number, amount_bc, curr, on_hold, approved
- **INV-009**: System shall return invoice detail with AR record, line items, and resolved customer name
- **INV-010**: System shall generate PDF representation of an invoice
- **INV-011**: Line items shall support optional product_id reference to the product catalogue

# Validation Rules

- Customer ID is required and must reference a valid entity_credit_account with an AR account
- Invoice number is required
- Transaction date is required
- Currency is required
- At least one line item is required
- Each line requires description, qty, sellprice, and account_id

# Error Handling

- Customer with no AR account returns error: "customer N has no AR account configured"
- Update to posted invoice returns error: "cannot update: invoice is already posted (approved)"
- Customer change on posted invoice returns error: "cannot change customer: invoice is already posted"
- Non-existent invoice returns 404

# Acceptance Criteria
