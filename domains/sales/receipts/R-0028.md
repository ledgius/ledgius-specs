# Summary

Record payments received from customers against open AR invoices. Receipts allocate funds across one or more invoices and generate GL entries that debit the bank account and credit the AR account, with each allocation referencing the invoice's open item for reconciliation.

# Problem / Business Context

When a customer pays an invoice, the receipt must be recorded with correct GL entries (debit bank, credit AR) and allocated against specific invoices to track which invoices are paid. Open item references enable downstream reconciliation and ageing reports.

# Scope

- Receipt creation with multi-invoice allocation
- GL entry generation (debit bank, credit AR per allocation)
- Open item referencing per allocation for reconciliation
- Receipt listing with summary view
- Allocation amount validation (positive amounts only, via rules engine)

# Out of Scope

- Receipt reversal/void
- Overpayment handling (credit balance on customer account)
- Partial receipt detail retrieval (no GET by ID endpoint)
- Bank feed matching/auto-allocation
- Receipt PDF generation

# Actors / Users

- Bookkeeper -- records customer payments
- Business Owner -- reviews payment status

# Preconditions

- User is authenticated
- Customer exists with a configured AR account
- At least one open AR invoice exists for the customer
- Bank account exists in the chart of accounts

# Functional Requirements

- **REC-001**: System shall create a receipt atomically: transaction record + GL entries (debit bank, credit AR per allocation), all within a single database transaction
- **REC-002**: Each allocation amount must be positive, validated via rules engine `ValidatePositiveAmount`
- **REC-003**: Each allocation must reference a valid open AR invoice for the specified customer
- **REC-004**: Each AR credit GL entry must reference the invoice's open_item_id for reconciliation
- **REC-005**: GL entries shall follow: debit bank account (total amount), credit AR account (per allocation, each referencing open_item_id)
- **REC-006**: System shall list all receipts with summary data: trans_id, reference, receipt_date, customer_name, amount, curr, approved
- **REC-007**: Transaction record shall store the reference number and optional notes

# Validation Rules

- Bank account ID is required
- Customer ID is required and must have a configured AR account
- Receipt date is required
- Reference is required
- Currency is required
- At least one allocation is required
- All allocation amounts must be positive
- Each allocation must reference a valid invoice for the customer

# Error Handling

- Customer with no AR account returns error: "customer N has no AR account configured"
- Negative allocation amount returns error: "allocation amounts must be positive"
- Invoice not found for customer returns error with invoice and customer IDs
- Open item lookup failure returns error with invoice ID

# Acceptance Criteria

- [x] Receipt creation atomically creates transaction and GL entries
- [x] Multi-invoice allocation supported
- [x] Positive amount validation via rules engine
- [x] Each allocation references invoice open_item_id
- [x] GL entries balance (debit bank = sum of credit AR allocations)
- [x] Receipt list with summary data

# Open Questions

1. **Receipt detail endpoint**: No GET by ID endpoint exists yet.
2. **Overpayment**: No handling for payments exceeding invoice totals.
3. **Receipt reversal**: No void/reversal mechanism implemented.
