# Summary

Provide reusable transaction templates that allow users to save and recall predefined transaction structures for quick data entry. Templates store the full transaction payload as JSON and are categorised by source type (general ledger, accounts receivable, or accounts payable).

# Problem / Business Context

Bookkeepers and accountants frequently create recurring or similar transactions -- monthly rent payments, standard sales invoices, regular journal entries. Without templates, users must re-enter the same account codes, descriptions, and line items each time. Transaction templates eliminate repetitive data entry, reduce errors, and speed up common accounting workflows.

# Scope

- Template CRUD (create, read, list, delete)
- Templates categorised by source type: gl (general ledger), ar (accounts receivable), ap (accounts payable)
- Template payload stored as flexible JSON (supports any transaction structure)
- Listing with optional source type filter
- Alphabetical ordering of template lists

# Out of Scope

- Template versioning or change history
- Template sharing between tenants
- Template categories or tags beyond source_type
- Scheduling templates for automatic execution
- Template import/export
- Template update (PUT endpoint)
- Template usage tracking or analytics

# Actors / Users

- Bookkeeper -- creates and uses templates for routine transactions
- Accountant -- creates templates for standard journal entries and adjustments
- Business Owner -- uses templates for common sales invoices or purchase orders

# Preconditions

- User is authenticated with a valid JWT
- Tenant context is established (templates are per-tenant database)

# Functional Requirements

- **TPL-001**: System shall store transaction templates with a name, optional description, source type, and JSON payload
- **TPL-002**: System shall support three source types: `gl` (general ledger), `ar` (accounts receivable), `ap` (accounts payable)
- **TPL-003**: System shall list all templates, optionally filtered by source type, ordered alphabetically by name
- **TPL-004**: System shall retrieve a single template by ID
- **TPL-005**: System shall create a new template from a name, source type, and JSON payload
- **TPL-006**: System shall delete a template by ID
- **TPL-007**: Template JSON payload shall be stored as JSONB to support flexible transaction structures without schema coupling
- **TPL-008**: System shall track which user created each template via `created_by` field

# Data / Entities / Fields

Primary entity: `Template` (table: `transaction_template`)

| Field | Type | Notes |
|-------|------|-------|
| id | int (PK) | Auto-increment |
| name | string | Required |
| description | string (nullable) | Optional description of the template's purpose |
| source_type | string | Required. One of: gl, ar, ap |
| template_json | jsonb | Required. Full transaction payload |
| created_by | string (nullable) | User ID of the creator |
| created_at | timestamp | Required |
| updated_at | timestamp | Required |

# API Endpoints

| Method | Path | Description |
|--------|------|-------------|
| GET | `/templates` | List all templates. Query params: `type` (optional, filters by source_type) |
| GET | `/templates/:id` | Get a single template by ID |
| POST | `/templates` | Create a new template. Body: `{ name, source_type, description?, template_json }` |
| DELETE | `/templates/:id` | Delete a template by ID |

# Validation Rules

- Name is required (binding: required)
- Source type is required and must be one of: `gl`, `ar`, `ap` (binding: required, oneof)
- Template JSON is required (binding: required)
- Description is optional

# Error Handling

- Invalid or missing required fields on create return 400 Bad Request with validation details
- Template not found on GET returns 404 Not Found with message "template not found"
- Delete of non-existent template completes silently (no error)
- Database errors return 500 Internal Server Error

# Security / Permissions

- All template endpoints require authentication (mounted under protected route group)
- Templates are scoped to the tenant database (multi-tenant isolation)

# Acceptance Criteria

- [x] Templates can be created with name, source_type, and JSON payload
- [x] Templates can be listed with optional source_type filter
- [x] Template list is ordered alphabetically by name
- [x] Individual template can be retrieved by ID
- [x] Templates can be deleted by ID
- [x] Source type validation enforces gl, ar, ap values

# Open Questions

1. **Update endpoint**: No PUT/PATCH endpoint exists. Users must delete and recreate to modify a template. An update endpoint would be a natural addition.
2. **Template application**: The API stores templates but does not provide an endpoint to "apply" a template (i.e. create a transaction from a template). This is currently a client-side concern.
3. **Soft delete**: Templates are hard-deleted. If templates are referenced in audit trails or transaction metadata, soft delete may be preferable.
4. **Ownership and sharing**: The `created_by` field is stored but not populated by the handler (no user extraction from JWT claims). Templates are visible to all users in the tenant.
