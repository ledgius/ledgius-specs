# Summary

Ledgius moves financial data between peer accounting systems (MYOB, Xero, and future targets) in both directions. The correctness of those transformations is a compliance obligation — not a best-effort feature. This spec establishes the **canonical fixture strategy**: a comprehensive, vendored set of representative MYOB, Xero, and Ledgius test fixtures covering every entity type, in every direction, with explicit contracts for what the test harness must prove about each. Self-consistent tests (mapper output matches mapper expectations) are necessary but not sufficient; only fixtures that trace to authoritative sources give defensible confidence that imports and exports actually conform to each peer system's format.

# Terms

| **Term** | Definition. |
|---|---|
| **Canonical fixture** | A representative data sample whose provenance traces to an authoritative source (vendor documentation, vendor OpenAPI spec, vendor-provided template, or a pinned real-world export). Carries a provenance header naming the source URL or document, the date captured, and the pinned version/SHA. |
| **Reference fixture** | A canonical fixture representing "what the source/target system's format actually looks like." Lives alongside the spec and is shared across all test suites. Example: a vendored Xero CSV invoice template. |
| **Test-flow fixture** | A fixture pairing an input (source system record) with a golden expected output (target system record). Used by a specific test suite to assert conformance. Example: a MYOB contact card paired with the Ledgius entity shape the importer should produce. |
| **Round-trip fixture** | A test-flow fixture chain covering two transformations end-to-end — import from system A to Ledgius, then export from Ledgius to system B. A and B may be different peers (`MYOB → Ledgius → Xero`) or the same peer (`Xero → Ledgius → Xero` — a Ledgius-fidelity check). Used to prove no data loss or drift across the composite transformation. Every shipped `(A, B)` pairing has its own round-trip fixture set; absent pairings are tracked as deferred coverage until the corresponding import or export ships. |
| **Provenance header** | A mandatory YAML/comment block at the top of every fixture file stating source URL, capture date, pinned version, and any intentional modifications from the original. |
| **Fidelity contract** | A named, enforced correctness guarantee the test harness proves for a given fixture — e.g. "import produces Ledgius shape matching golden expectation field-for-field," or "export output passes OpenAPI schema validation." |

Vocabulary used here but defined elsewhere:
- `external_system`, `capability flags`, `direction` — defined canonically in **R-0058 §Terms**

# Problem / Business Context

Ledgius import and export test suites as they stand today verify **self-consistency**: the code agrees with the code. That catches regressions but cannot catch **format drift** — the cases where Ledgius's notion of "what Xero wants" or "what MYOB produces" has silently diverged from reality. Three concrete risks:

1. **Vendor format updates land silently.** Xero's Accounting API ships new OpenAPI versions regularly; MYOB periodically changes its AO/CeeData export layout. Without fixtures tied to pinned vendor versions, a format change goes unnoticed until a customer reports a rejected import or a malformed export.
2. **Transcription errors compound.** Field-mapping tables in specs (e.g. A-0031) are human-curated from third-party descriptions of vendor formats. Without fixtures sourced directly from vendor material, a wrong field name propagates from spec → YAML config → test assertion, and every layer agrees the result is correct.
3. **Compliance defensibility.** For regulated accounting data (ATO record-keeping, TPB tax-agent obligations — see R-0060 compliance context), "our tests pass" is weaker evidence than "our tests pass against the vendor's published OpenAPI schema at version X, SHA Y, with round-trip fidelity demonstrated on fixtures vetted by a CPA." The latter is what an auditor or a customer-due-diligence reviewer will ask for.

The fix is structural: stop treating fixtures as ad-hoc test data and start treating them as first-class, vendored artefacts with their own governance.

# Scope

## In scope

- Three-system coverage: **MYOB** (source/target), **Ledgius** (middle), **Xero** (source/target) — with symmetry so that import and export share fixture sources where possible
- Both directions per external system that supports them. At the time this spec lands, the directional support matrix is:
  - **MYOB → Ledgius** (import): supported today via R-0017
  - **Xero → Ledgius** (import): supported today via R-0017 (Xero CSV path)
  - **Ledgius → Xero** (export): in flight per R-0057, first real export target
  - **Ledgius → MYOB** (export): not yet specced; reserved for a future export spec
- Six entity types across the v1 + v2 export surface: **contacts, invoices, bills, credit notes, payments, bank transactions**
- Three fixture categories per (system, entity): **happy path**, **edge cases**, **rejection cases**
- Four fidelity contracts the test harness must enforce: **import conformance**, **export conformance**, **round-trip fidelity**, **rejection handling**
- **Bi-directional round-trip coverage**: every `(import source A, export target B)` pair where both A-import and B-export are shipped shall have at least one happy-path round-trip fixture. Currently that means `MYOB → Ledgius → Xero` and `Xero → Ledgius → Xero` (a Ledgius-fidelity check that exercises only one direction of Ledgius-involvement per peer system). Future pairings (`Xero → Ledgius → MYOB`, `MYOB → Ledgius → MYOB`) are tracked as **deferred coverage** — not silent omissions — until the corresponding export lands.
- Vendored machine-readable vendor specs (Xero OpenAPI at pinned SHA; similar for any future target that publishes one)
- Directory structure across `ledgius-specs`, `ledgius-api`, and `ledgius-db` so fixtures live alongside the concern that owns them
- Provenance header convention for every fixture
- Governance for fixture lifecycle — who adds, who approves, how version bumps happen

## Out of scope

- Synthetic load-testing datasets (different concern; covered by performance testing strategy if/when needed)
- Customer-specific fixtures used for ad-hoc support investigations (those go through existing support tooling, not this spec)
- Cross-region variants beyond AU (NZ, UK, US fixtures land when those regions are supported per the relevant export/import spec)
- Non-accounting domains (payroll, banking reconciliation) — their fixture needs are addressed in their own domain specs
- Automatic vendor-format drift detection (future enhancement; v1 requires manual review when a vendor ships a new spec version)

# Actors / Users

- **Developer adding a new import or export feature** — writes new fixtures following the taxonomy, runs the contract harness locally, submits fixtures alongside the feature PR
- **Reviewer / CPA consultant** — occasionally vets a representative fixture against their professional knowledge of the peer system's behaviour (one-off, not every PR)
- **Support engineer** — references canonical fixtures when investigating a customer report ("is this the Xero CSV shape we expect?")
- **Auditor / due-diligence reviewer** — reads the vendored vendor specs + fixture provenance headers + round-trip test reports as evidence of format conformance
- **Platform owner** — approves version bumps on vendored vendor specs (major jumps require awareness; minor patches do not)

# Functional Requirements

## Fixture taxonomy and location

- **FIX-001**: Canonical reference fixtures — files representing "what the source/target system's format actually looks like" — shall live in `ledgius-specs/domains/{domain}/fixtures/` so they are visible alongside the specs that describe them, and so non-engineers (reviewers, CPAs, auditors) can browse them without clone-access to API code.
- **FIX-002**: Test-flow fixtures — input + golden expected output pairs used by specific test suites — shall live alongside the code they test. The directory structure supports every direction currently shipped, with parallel directories reserved for future directions so the shape stays consistent:
  - `ledgius-api/internal/importer/testdata/myob-to-ledgius/{happy,edge-cases,rejection-cases}/` — active
  - `ledgius-api/internal/importer/testdata/xero-to-ledgius/{happy,edge-cases,rejection-cases}/` — active (R-0017 supports Xero CSV)
  - `ledgius-api/internal/exporter/xero/testdata/ledgius-to-xero/{happy,edge-cases,rejection-cases}/` — active (R-0057)
  - `ledgius-api/internal/exporter/myob/testdata/ledgius-to-myob/{happy,edge-cases,rejection-cases}/` — reserved for when Ledgius→MYOB export is specced
  - `ledgius-api/tests/roundtrip/{happy,edge-cases}/` for end-to-end flows, with subdirectories per `(source, target)` pairing — `myob-to-xero/`, `xero-to-xero/`, and `xero-to-myob/` (latter reserved for future)
- **FIX-003**: Ledgius internal representative entities — showing what each Ledgius domain entity looks like in canonical form — shall live in `ledgius-specs/domains/{domain}/fixtures/entities/` so import and export tests share one source of truth for the middle shape.

## Provenance — every fixture file

- **FIX-004**: Every canonical reference fixture shall carry a provenance header stating: source URL or document, capture date in ISO-8601, pinned vendor version or commit SHA, any intentional modifications from the original (e.g. anonymisation of real ABNs), and the licensing status that permits redistribution.
- **FIX-005**: When a fixture is modified from its original source (e.g. anonymisation of customer data, currency normalisation for test determinism), the modification and its rationale shall be recorded in the provenance header. Silent edits are forbidden.
- **FIX-006**: Vendored vendor specifications (e.g. Xero OpenAPI) shall be committed at a pinned SHA with the source repo URL and commit SHA captured in the file header. Updates to the pinned SHA shall go through a PR titled `chore(fixtures): bump {vendor} spec to {new-version}` so version drift is visible in git history.

## Fidelity contracts — what the harness proves

- **FIX-007**: The **import conformance** contract: every canonical source fixture (MYOB, Xero, or any future import source) shall, when fed through the Ledgius importer, produce a Ledgius entity graph matching the golden expected output. Fields excluded from comparison (auto-generated UUIDs, created_at timestamps) shall be declared explicitly in each test's exclusion list.
- **FIX-008**: The **export conformance** contract: every Ledgius canonical fixture shall, when fed through the target-system exporter, produce output that (a) validates against the vendored target-system spec (e.g. Xero OpenAPI schema) for the relevant entity, and (b) matches the golden expected CSV/JSON byte-for-byte after normalisation (line endings, whitespace trim). The spec validation is the authoritative check; the byte-for-byte check catches regressions in the specific Ledgius → target mapping configs.
- **FIX-009**: The **round-trip fidelity** contract: every round-trip fixture — for any `(import source A, export target B)` pair where both directions are shipped — shall prove that Ledgius acts as a lossless conduit across the `A → Ledgius → B` chain. Preserved invariants: total debits and credits, line-level arithmetic, contact identity, document reference and date, tax classification. Any intentional data loss (e.g. source fields that have no target equivalent) shall be explicitly listed in the fixture's expected-loss declaration; anything else counts as a test failure. Initial shipped pairings: `MYOB → Ledgius → Xero` and (when supplier side lands) `Xero → Ledgius → MYOB`.
- **FIX-010**: The **rejection handling** contract: every rejection-case fixture shall, when fed through its respective pipeline, be rejected with a specific, row-attributed error message naming the offending field and the rule violated. Silent acceptance or a generic "import failed" error counts as a test failure.
- **FIX-022**: The **Ledgius-fidelity round-trip** contract: for every peer system that Ledgius both imports from AND exports to, at least one happy-path `Peer → Ledgius → Peer` fixture shall prove that a losslessly-imported payload re-exported back to the same system produces a payload with the same invariants as FIX-009 (totals, identities, references, tax classification). Catches bugs that cancel out across a single direction (e.g. an import error compensated by a symmetric export error) but fail to cancel when both sides run against real vendor format. Active once Ledgius→Xero ships: `Xero → Ledgius → Xero`. Future: `MYOB → Ledgius → MYOB`.

## Coverage

- **FIX-011**: Each entity type (contacts, invoices, bills, credit notes, payments, bank transactions) shall have at minimum one **happy path** fixture per (source system, direction) combination that the product claims to support.
- **FIX-012**: Edge-case fixtures shall cover at minimum: multi-line documents, partial payments against documents, foreign currency transactions, missing optional fields, fields at maximum declared length, zero-value transactions, and contacts with structured addresses including both postal and shipping variants where the target format supports them.
- **FIX-013**: Rejection-case fixtures shall cover at minimum: duplicate document numbers within a batch, malformed dates, amount values in wrong decimal precision, references to non-existent accounts/contacts, control-account posting attempts, document line counts above the vendor's declared ceiling, and data exceeding declared length limits.
- **FIX-014**: Every new entity type added to the export or import surface shall ship with its complete fixture set (happy / edge / rejection) in the same PR that introduces the code — no retroactive "fixtures will follow" debt.

## Vendor spec vendoring

- **FIX-015**: Xero Accounting API OpenAPI spec shall be vendored at `ledgius-specs/domains/export/fixtures/xero/openapi/xero_accounting_v{N}.yaml` from the public `XeroAPI/Xero-OpenAPI` repository, pinned to a specific commit SHA.
- **FIX-016**: Xero CSV import templates (the four v1 templates: contacts, invoices, bills, credit notes) shall live at `ledgius-specs/domains/export/fixtures/xero/csv-templates/` once the user supplies them from a Xero demo account. Until supplied, the repository shall contain a `README.md` in that directory explaining the gap and referencing the publicly-quoted header sets in use as interim fixtures.
- **FIX-017**: MYOB export format fixtures shall be extended from the existing `tests/testdata/myob/` set to cover every entity type (the current set covers chart of accounts, AO export, CeeData export; coverage gaps for contacts, credit notes, payments, bank transactions shall be filled).
- **FIX-018**: Any future target system with a published OpenAPI or JSON schema (QuickBooks, etc.) shall follow the same vendoring pattern as Xero — vendored spec at pinned SHA, CSV templates in a parallel directory, provenance headers on every file.

## Governance

- **FIX-019**: Fixture additions and modifications go through the standard PR flow. Fixtures carry governance expectations equal to production code — reviewers verify provenance headers, fidelity contract coverage, and representativeness.
- **FIX-020**: Major vendor spec version bumps (e.g. Xero Accounting API v12 → v13) shall be reviewed by someone with accounting-domain knowledge, not just by any available reviewer. The PR title shall flag this explicitly.
- **FIX-021**: Removal of a fixture shall include the rationale in the commit message. Removing a fixture without a replacement is discouraged and requires justification (e.g. "source format deprecated by vendor").

# Non-functional Requirements

- **Transparency:** fixture provenance is human-readable. Any engineer, reviewer, or auditor can open a fixture file and immediately see where it came from, when it was captured, and what version of the vendor format it represents.
- **Independence from Claude memory:** all fixtures are persisted in the repository. No fixture exists only in conversational context; nothing relies on cross-session memory.
- **Stability:** vendored vendor specs use pinned SHAs so a CI run in six months resolves the same artefact as a CI run today.
- **Coverage visibility:** the test harness surfaces which entity × direction × category combinations have fixtures and which are missing, so gaps are visible without reading the test files.
- **Review ergonomics:** non-engineer reviewers (CPAs, auditors) can read fixture files without Go or CSV-tooling knowledge. YAML is preferred for Ledgius representative entities; native vendor formats (CSV, proprietary text) for vendor-reference fixtures.

# Acceptance Criteria

- [ ] Every entity type currently claimed as supported by v1 export (COA, contacts, invoices, bills, credit notes, tax rates) has at least one happy-path fixture in each of the three relevant locations (MYOB source, Ledgius representative, Xero expected output).
- [ ] Xero Accounting OpenAPI spec is vendored at a pinned SHA with a provenance header naming the source repo and commit.
- [ ] The contract test harness runs under `go test ./...` and reports per-fidelity-contract pass/fail counts.
- [ ] A deliberately malformed MYOB fixture (e.g. duplicate contact name with mismatched ABN) is rejected by the importer with an error message naming the offending row.
- [ ] A deliberately malformed Ledgius fixture (e.g. invoice with an account code >10 chars) is rejected by the Xero pre-flight validator with an error message naming the offending field.
- [ ] For each shipped `(import source, export target)` pair, at least one happy-path round-trip fixture verifies total debits, total credits, every document number, and every contact identity survives both transformations without modification. Initial pairings: `MYOB → Ledgius → Xero` (active once Ledgius→Xero ships), `Xero → Ledgius → Xero` (active the same time via FIX-022). Deferred pairings (`Xero → Ledgius → MYOB`, `MYOB → Ledgius → MYOB`) are listed as tracked gaps in the fixtures README until the Ledgius→MYOB export is specced.
- [ ] The harness surfaces directional coverage explicitly — listing every `(source, target)` pair the product claims to support and whether a round-trip fixture exists. Missing pairings appear in the output as "deferred," not as silently absent.
- [ ] A non-engineer reviewer (CPA consultant) can open any fixture file and trace its provenance to an authoritative vendor source within one click.
- [ ] The Xero CSV templates directory either contains the four v1 templates (when user-supplied) or a README explaining the gap and naming the interim header source.
- [ ] Adding a new external system (e.g. QuickBooks) follows the same directory structure and provenance conventions with no ambiguity.

# Open Questions

- **Cross-repo fixture discovery.** Should the test harness in `ledgius-api` load canonical fixtures from `ledgius-specs` via a git submodule, a go-embed into a checked-in mirror, or an out-of-band symlink? Proposal: checked-in mirror refreshed via `make sync-fixtures`. Revisit if the dual-repo overhead becomes painful.
- **Anonymisation policy.** Some MYOB fixtures currently contain real-looking ABNs and names. What level of anonymisation is required before broader-than-team access? Proposal: treat every committed fixture as "anyone with repo access can see it"; anything sensitive lives outside the repo and is referenced via an off-repo path that CI can resolve in controlled environments.
- **Performance.** Running every fidelity contract on every CI build has a cost. Should some contracts (especially round-trip) run only on nightly or on release-candidate branches? Proposal: defer until cost is measured; run everything on every PR in v1.
