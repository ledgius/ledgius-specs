# Summary

This architecture defines the phase-1 implementation structure for manual banking import.

# Requirement Link

This document satisfies `R-0001`.

# Technical Context

The system uses a Go backend and a web portal frontend. The initial implementation should remain simple, synchronous, and easy to reason about.

# Proposed Design

- frontend uploads the selected file to a dedicated API endpoint
- backend validates file type and request metadata
- backend parses the file into an internal import model
- backend validates rows and performs duplicate detection
- backend persists accepted rows and returns an import result payload

# Package / Service Boundaries

- web UI handles file selection and result presentation
- API handler handles request validation and orchestration
- import service handles parsing, validation, duplicate detection, and result construction
- persistence layer stores accepted rows and import batch metadata

# Data Flow

1. User selects file.
2. Web portal submits multipart request.
3. Go endpoint receives request.
4. Import service parses and validates rows.
5. Duplicate detection runs before persistence.
6. Accepted rows are stored.
7. Result summary is returned to the UI.

# API / Interface Changes

Add a dedicated authenticated endpoint for manual transaction import.

# Storage / Schema Changes

Store:

- import batch record
- source file metadata
- accepted imported transaction rows
- duplicate detection evidence when practical

# Validation Placement

Validation belongs in the import service, not scattered across the handler and persistence layers.

# Background Processing

No background processing in phase 1.

# Risks

- large files may stress synchronous request handling
- duplicate detection may be too weak if not well defined
- format variation may increase parser complexity

# Alternatives Considered

- asynchronous job processing: rejected for phase 1 to keep implementation simple
- direct persistence from handler: rejected to preserve clean orchestration and testability

# Test Strategy

- parser unit tests
- validation rule unit tests
- duplicate detection unit tests
- endpoint integration tests

# Rollout / Migration Notes

Roll out behind a permission gate if needed.
