# Summary

Provide a web-based manual import capability for bank transaction files so finance staff can load external data into the reconciliation workflow.

# Problem / Business Context

Finance staff receive transaction files from banks and third-party systems. They need a controlled method to import those files into Ledgius without relying on direct integrations for every institution.

# Scope

This requirement governs manual import of banking transaction data through the web portal.

# Out of Scope

- direct bank feed integrations
- automatic invoice matching
- asynchronous bulk import processing in phase 1
- background fraud detection

# Actors / Users

- Finance Administrator
- Finance Officer

# Preconditions

- user is authenticated
- user has permission to import banking data
- target reconciliation account is already configured

# Functional Requirements

1. The user must be able to select a supported data file from the web portal.
2. The system must upload the file to a Go HTTP endpoint for processing.
3. The system must parse supported file formats and extract transaction rows.
4. The system must validate imported rows before persistence.
5. The system must prevent obvious duplicate imports using the defined duplicate detection rule.
6. The system must present the import result to the user, including accepted rows and rejected rows.

# Data / Entities / Fields

Each imported transaction row must support capture of:

- account identifier
- transaction date
- transaction reference
- transaction description
- debit amount
- credit amount
- balance when provided
- source file name
- import batch identifier

# UX / UI Behaviour

- The portal must present a file picker and import action.
- The user must receive a clear success, partial-success, or failure outcome.
- Validation failures must be visible to the user with row-level detail where practical.

# Validation Rules

- Unsupported file formats must be rejected before parsing.
- Required fields must be validated before persistence.
- Invalid monetary values must be rejected.
- Duplicate detection must occur before rows are persisted.

# Error Handling

- Malformed rows must be reported, not silently dropped.
- If the file cannot be parsed, the user must receive a clear parse failure result.
- The system must not partially persist invalid rows without clearly reporting the outcome.

# Security / Permissions

Only authorized users with banking import permission may perform imports.

# Acceptance Criteria

- A permitted user can upload a supported file through the web portal.
- The file is processed by the Go endpoint.
- Valid rows are identified correctly.
- Invalid rows are reported clearly.
- Duplicate imports are detected according to the defined rule.
- The user receives a result summary including accepted and rejected rows.

# Open Questions

- Which file formats are supported in phase 1: CSV only, or CSV plus OFX/QIF?
- Should duplicate detection include source file checksum in addition to row-level matching?
